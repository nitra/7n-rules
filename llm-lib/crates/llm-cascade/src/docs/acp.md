## Огляд

`AcpAgentKind`, `command`, `one_shot_acp`, `one_shot_acp_with_tier` визначають ACP-бекенд для запуску вже залогіненого локального agent CLI через особисту підписку, без API-ключів. Комунікація відбувається через stdio/JSON-RPC, а якщо CLI не авторизований підпискою, виклик провалюється одразу без retry, щоб викликач сам вирішував, чи падати далі на [`crate::local_cloud`]. За певних помилок API повертає порожнє значення замість винятку.

## Поведінка

AcpAgentKind визначає, до якого ACP-клієнта звертатися, а command повертає базову команду запуску саме для цього вибору; далі ці дані переходять у one_shot_acp або one_shot_acp_with_tier, які створюють ACP-сесію в явному робочому каталозі викликачa, надсилають один prompt і збирають повну текстову відповідь до завершення ходу.

one_shot_acp працює як простий шлях без додаткового резолвінгу: використовує вже готову ACP-спеку, зберігає той самий idle timeout, auto-approve дозволів і progress-логування, що й інші transport-потоки, і повертає результат одного діалогу; за проблем із відсутнім агентом, неавторизованою підпискою або падінням процесу він завершується provider-помилкою, а в окремих збоях може дати порожній результат.

one_shot_acp_with_tier додає перед цим крок вибору preset за Tier і застосовує його однаково для всіх ACP-агентів: для одних це спавн-налаштування, для інших — post-session конфігурація, але зовнішня поведінка лишається тією самою, як у one_shot_acp; якщо preset не валідний або агент недоступний, результат теж іде через provider-помилку або порожнє значення там, де це передбачено.

Спільний стан між AcpAgentKind, command, one_shot_acp і one_shot_acp_with_tier — лише вибір офіційного ACP-бриджа та узгоджена схема запуску через локально залогінений CLI без API-ключів; якщо CLI не авторизований підпискою, виклик не маскує цю проблему і віддає її назовні без додаткового retry.

## Публічний API

- AcpAgentKind — Ціль ACP-підключення. Список відкритий — нові агенти додаються нових варіантом і `spec()`, протокол той самий для всіх.
- command — Команда спавну для цього агента (базовий argv, без env/extra-args префіксів — ті додає [`transport::build_acp_args`]).
- one_shot_acp — Один виклик через ACP: спавнить агента, відкриває сесію в `cwd`, надсилає `prompt`, читає повний текст відповіді до кінця ходу.  `cwd` — явний, а не [`std::env::current_dir`]: викликач (напр. napi-міст в один процес із Node) має свій власний робочий каталог проєкту, який не обов'язково збігається з cwd самого хост-процесу.  Тонкий фасад над публічним [`session`]-API (задача T2): створює сесію з дефолтних [`SessionOptions`] (`PermissionMode::AutoApprove`, без post-session-config-кроку — той потрібен лише Pi-тіру, T3), надсилає один `prompt`, акумулює текстові `AgentMessageChunk`-події з потоку подій до кінця ходу. Поведінка не змінюється — той самий idle-timeout, auto-approve дозволів і progress-логування, що й раніше (спільна операційна броня [`transport`], тепер розділена з session-режимом).  # Errors [`CascadeError::Provider`] — агент не встановлений, не залогінений підпискою, чи процес завершився з помилкою.
- one_shot_acp_with_tier — Той самий один виклик через ACP, що й [`one_shot_acp`], але з явним [`Tier`] (задача T3, рішення К: тір, ніколи конкретна модель) — резолвиться в [`presets::TierPreset`] цього `agent`-а і застосовується одноманітно незалежно від того, чи kind бере тір через env на спавні (Codex), extra-arg на спавні (Cursor) чи post-session протокольний виклик (Pi, рішення З.1): `env`/`extra_args` пресету йдуть у той самий [`transport::spec_for`], що й раніше, `post_session_config` — у [`SessionOptions`] session-шару. Стара сигнатура [`one_shot_acp`] не чіпається — цей фасад лише додає крок резолвінгу перед тим самим `run_one_shot`.  # Errors [`CascadeError::Provider`] — агент не встановлений, не залогінений підпискою, невалідний `configId`/`value` (Pi) чи процес завершився з помилкою.
