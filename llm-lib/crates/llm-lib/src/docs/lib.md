## Огляд

`LlmError` описує єдиний шлях відмови для каскадного доступу до LLM у контракті, що відповідає `@7n/llm-lib` (`model-tiers.mjs`) і розширює його через `genai` та ACP-бекенди. Тут `AcpAgentKind`, `one_shot_acp`, `LocalCloud` і `Tier::Max` зібрані в спільну поведінку: кожен `one_shot_*` робить рівно одну спробу, а драбину ескалації — наприклад, від `Cursor` чи `Codex` до `LocalCloud` — будує викликач.

## Поведінка

1. `LlmError` уніфікує провал каскадного доступу до LLM в один бізнес-сигнал замість окремих помилок кожного бекенда.
2. `NoModelConfigured` означає, що для потрібного тиру не знайдено жодної придатної конфігурації моделі через `N_LOCAL_*` або `N_CLOUD_*`, тож запит не має куди ескалювати.
3. `InvalidModelSpec` означає, що переданий spec моделі не був розпізнаний як валідна конфігурація провайдера та моделі.
4. `Provider` означає технічний збій на боці виклику: мережа, handshake ACP або запуск процесу; деталі провайдера не розкладаються на окремі типи, а зводяться в один текстовий сигнал.
5. У разі помилки модуль повертає саме `LlmError` замість приховування причини або автоматичного повтору, щоб драбину ескалації будував викликач на рівні бізнес-логіки.

## Публічний API

- LlmError — Помилка каскаду. Навмисно плоска — деталі провайдера/ACP-агента вже в тексті, без вкладеної типізації для кожного backend-у.
