## Огляд

`BatchConfig`, `BatchItem`, `BatchResult`, `BatchProgress` і `submit` задають спільний контракт для batch-обробки в OpenAI-сумісному потоці `submit → progress → results`.

У v1 це лише емуляція batch API через `crate::local_cloud` (Тип 2a): той самий інтерфейс використовується і тут, і для справжнього серверного batch-режиму `/v1/batches` у v2.

Окрема помилка одного item або одного чанка не валить увесь batch: вона потрапляє у відповідний `BatchResult::outcome`, а решта items продовжують обробку.

## Поведінка

BatchConfig задає межі чанкованого прогону: скільки BatchItem обробляється в одному проході та скільки з них можуть виконуватися одночасно в межах цього чанка.

BatchItem несе вхідні дані одного запиту й стабільний custom_id, за яким результат пізніше повертається до викликачa.

submit бере послідовність BatchItem, ділить її на чанки за BatchConfig, виконує чанки по черзі, а всередині кожного чанка запускає обробку конкурентно в межах заданого ліміту. Результати повертаються в тому ж порядку, що й вхідні items, навіть якщо фактичне завершення було іншим.

submit працює fail-safe: помилка одного BatchItem не зупиняє решту batch, а перетворюється на помилковий BatchResult для цього ж custom_id. Якщо виконавець для item переривається панікою, цей item також не губиться — він повертається як помилка, а batch продовжується.

BatchProgress є єдиним спільним станом прогресу для всього прогону: кожен завершений BatchItem, успішний чи ні, збільшує completed, а total лишається незмінним до кінця batch.

BatchResult збирає фінальний вихід submit: для кожного BatchItem повертається або успішний текст, або текстова помилка; це дозволяє оминати типізовану помилку на рівні окремого item і передавати результат далі як плоский batch-вихід.

## Публічний API

- BatchConfig — Ліміти чанка/конкурентності. Дефолти — стартові з рішення Р спеки (чанк ≤35), уточнені бенч-калібруванням на omlx (див. `docs/specs/2026-07-24-batch-emulation-bench.md`).
- BatchItem — Один запит у batch — той самий `custom_id`-контракт, що й OpenAI Batch API (v2 буде говорити тим самим полем), щоб емуляція v1 і справжній сервер v2 ділили один виклик-сайт.
- BatchResult — Результат одного item. `outcome` — `Ok(text)` чи `Err(message)`; помилка **не** типізована як [`LlmError`] навмисно (той самий плоский підхід, що й у решті крейта) — рядок достатній для napi-мосту, де помилка одного item лише показується користувачу, не оброблюється програмно.
- BatchProgress — Знімок прогресу — скільки items уже має результат (успішний чи ні) з усього `total`. Монотонно зростає до `total` включно.
- submit — Емуляція batch (рішення Р, задача T6): розбиває `items` на чанки за `config.chunk_size`, у межах чанка виконує до `config.concurrency` `executor`-викликів одночасно, чанки — послідовно один за одним. `on_progress` викликається після кожного завершеного item (успіх чи помилка).  `executor` — ін'єкція виконавця одного item (у продакшн-коді — `LocalCloud::one_shot_with_spec`/`one_shot`, у юніт-тестах — фейкова async-функція без мережі; той самий підхід ін'єкції, що й `AcpSessionUpdates` у `acp::transport`). Паніка `executor`-а всередині задачі не валить `submit` — item, чия задача запанікувала, теж отримує `Err`-результат (за `custom_id`, зібраним до спавну задачі).  Порядок [`BatchResult`] у поверненому `Vec` збігається з порядком вхідних `items` — попри конкурентне виконання всередині чанка.

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
