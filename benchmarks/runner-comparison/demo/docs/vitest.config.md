# vitest.config.js

## Огляд

Файл `benchmarks/runner-comparison/demo/vitest.config.js` — це конфігураційний модуль для тестового раннера [Vitest](https://vitest.dev/). Він описує параметри запуску тестів у демо-проєкті `benchmarks/runner-comparison/demo`, який слугує підставним бенчмарком для порівняння різних тестових раннерів.

Файл написаний як ES Module (`import` / `export default`) і використовує офіційний хелпер `defineConfig` із підпакета `vitest/config`. Хелпер `defineConfig` забезпечує типову підказку (IntelliSense / IDE autocompletion) для об'єкта конфігурації, але самі значення не модифікує — він повертає переданий об'єкт без змін, виступаючи лише як identity-функція з типовою інформацією.

Конфігурація задає три речі:

1. Які файли вважаються тестами (`test.include`).
2. У якому середовищі запускати ці тести (`test.environment`).
3. Як збирати та подавати звіт про покриття коду (`test.coverage`).

## Експорти / API

Модуль має один default-експорт.

| Експорт   | Тип          | Опис                                                                                                                        |
| --------- | ------------ | --------------------------------------------------------------------------------------------------------------------------- |
| `default` | `UserConfig` | Об'єкт-конфігурація Vitest, отриманий із виклику `defineConfig({...})`. Vitest CLI читає цей експорт під час старту тестів. |

Іменованих експортів (`export const ...`, `export function ...`) у файлі немає.

### Структура default-експорту

Об'єкт, який повертає `defineConfig`, має таку форму:

```js
{
  test: {
    include: ['tests/**/*.test.mjs'],
    environment: 'node',
    coverage: {
      provider: 'v8',
      reporter: ['lcov', 'text-summary']
    }
  }
}
```

Поля об'єкта:

- `test.include` — масив glob-патернів, що визначає, які файли Vitest включить у запуск як тести.
  - Значення: `['tests/**/*.test.mjs']`.
  - Семантика: усі файли з розширенням `.test.mjs`, що лежать у директорії `tests/` (рекурсивно, на будь-якій глибині).
- `test.environment` — назва середовища виконання, у якому Vitest запускає кожен тестовий файл.
  - Значення: `'node'`.
  - Семантика: тести виконуються в чистому Node.js-середовищі без емуляції DOM (на відміну від `'jsdom'` чи `'happy-dom'`). Глобалі браузера (`window`, `document`, `navigator` тощо) у тестах недоступні.
- `test.coverage` — підоб'єкт із налаштуваннями збору покриття коду.
  - `test.coverage.provider` — рушій вимірювання покриття.
    - Значення: `'v8'`.
    - Семантика: використовується вбудоване в Node.js V8-покриття (через `node:inspector` / `--experimental-coverage`-подібний механізм). Альтернативи (наприклад, `'istanbul'`) — не задіяні.
  - `test.coverage.reporter` — масив форматів звіту про покриття.
    - Значення: `['lcov', 'text-summary']`.
    - `'lcov'` — формат, сумісний із зовнішніми інструментами (Codecov, Coveralls, SonarQube); зазвичай виводиться у директорію `coverage/` з файлом `lcov.info`.
    - `'text-summary'` — стислий текстовий підсумок (загальні відсотки statements / branches / functions / lines) у stdout.

## Функції

У файлі немає визначень власних функцій (`function`, arrow-функцій верхнього рівня тощо). Єдиний виклик функції — це звернення до імпортованого хелпера `defineConfig`.

### `defineConfig(config)` (імпортована, не визначена тут)

- **Походження**: `import { defineConfig } from 'vitest/config'`.
- **Сигнатура**: `defineConfig(config: UserConfig): UserConfig`.
- **Параметри**:
  - `config` — об'єкт-конфігурація Vitest (тип `UserConfig` із пакета `vitest`). У цьому файлі передається літерал `{ test: { include, environment, coverage } }`.
- **Що повертає**: той самий об'єкт `config`, без змін (identity по семантиці; основне призначення — типова підказка для TypeScript / IDE).
- **Side effects**: відсутні. `defineConfig` не модифікує глобальний стан, не звертається до файлової системи й не запускає тести.

## Залежності

| Модуль          | Тип                     | Що імпортується | Використання                                                  |
| --------------- | ----------------------- | --------------- | ------------------------------------------------------------- |
| `vitest/config` | npm-пакет (third-party) | `defineConfig`  | Хелпер-обгортка для типобезпечного опису конфігурації Vitest. |

Файл не має:

- внутрішніх (relative) імпортів зі свого репозиторію;
- імпортів із Node.js core-модулів;
- залежностей від змінних оточення (`process.env.*`);
- умовної логіки (if/else, тернарних виразів) — конфігурація статична.

Транзитивно для роботи файлу потрібен встановлений пакет `vitest` у `node_modules` (саме він експонує підшлях `vitest/config`).

## Потік виконання / Використання

### Сценарій 1: запуск тестів

1. Розробник (або CI-задача) запускає Vitest у директорії `benchmarks/runner-comparison/demo` командою на кшталт `vitest run` чи `bun x vitest run`.
2. Vitest CLI шукає файл конфігурації за стандартними іменами (`vitest.config.js`, `vitest.config.ts`, `vite.config.*` тощо). У цьому проєкті він знаходить саме `vitest.config.js`.
3. Node.js (або bundler, який Vitest використовує внутрішньо) виконує модуль `vitest.config.js` як ES Module:
   - відбувається імпорт `defineConfig` із `vitest/config`;
   - викликається `defineConfig({ test: { include, environment, coverage } })`;
   - повернений об'єкт експортується як `default`.
4. Vitest читає `default`-експорт і застосовує його як конфігурацію раннера:
   - збирає список тестових файлів за патерном `tests/**/*.test.mjs`;
   - готує `node`-середовище для кожного worker-процесу/потоку;
   - якщо запуск виконується з прапорцем покриття (наприклад, `vitest run --coverage`), використовує V8-провайдер і вивантажує звіти у форматах `lcov` та `text-summary`.

### Сценарій 2: запуск без покриття

Якщо команда запускається без прапорця `--coverage`, секція `test.coverage` фактично не задіюється — Vitest застосує її, лише коли збір покриття буде явно активований через CLI або env-змінні. Інші секції (`include`, `environment`) діють у будь-якому разі.

### Сценарій 3: відсутні відповідні файли

Якщо у директорії `tests/` немає жодного файлу з розширенням `.test.mjs`, Vitest відрапортує "no test files found" (поведінка раннера, а не файлу конфігурації) і завершиться з ненульовим кодом — це поведінка Vitest, не цього модуля.

### Side effects файлу `vitest.config.js`

- Імпорт-час: підвантажується модуль `vitest/config` (його ініціалізаційний код може виконатися, але це зовнішня бібліотека — не побічний ефект цього файлу).
- Рантайм: створюється один JS-об'єкт із літералу й експортується. Жодного запису у файлову систему, мережевих звернень, мутації глобалей чи `process.env` цей файл не виконує.

## Rebuild Test (контрольна відтворюваність)

Щоб відтворити логіку файлу `benchmarks/runner-comparison/demo/vitest.config.js` за цим документом, достатньо:

1. Створити ESM-файл (з `package.json` `"type": "module"` або з розширенням `.mjs`).
2. Імпортувати `defineConfig` з підпакета `vitest/config`.
3. Викликати `defineConfig` з об'єктом, що містить поле `test` із трьома підполями: `include: ['tests/**/*.test.mjs']`, `environment: 'node'`, `coverage: { provider: 'v8', reporter: ['lcov', 'text-summary'] }`.
4. Експортувати результат як `default`.

Семантично еквівалентний код:

```js
import { defineConfig } from 'vitest/config'

export default defineConfig({
  test: {
    include: ['tests/**/*.test.mjs'],
    environment: 'node',
    coverage: { provider: 'v8', reporter: ['lcov', 'text-summary'] }
  }
})
```
