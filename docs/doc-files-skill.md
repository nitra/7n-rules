# Skill `doc-files`: пофайлова поведінкова документація local-only конвеєром

## Проблема

Документація до коду бреше, щойно відстає від нього — а вручну тримати сотні md-файлів синхронними з кодом неможливо. Генерувати все хмарною моделлю — дорого: перший прогін цього репо ≈ 26M токенів (~$300–500). Генерувати слабкою локальною моделлю «в лоб» — неякісно: one-shot цілим файлом на `gemma-4-e2b` дає 47% якості проти 92% у секційної оркестрації (виміряно на тих самих файлах).

`doc-files` розв'язує обидві проблеми одночасно: **детермінована перевірка актуальності** (CRC) + **local-only конвеєр генерації**, де складність поглинає JS-оркестрація, а не розмір моделі.

## Ключові принципи

1. **Оркеструє JS, не модель.** Черга, батчинг, виклики LLM, штамп CRC — усе в CLI-команді. Агент не диспатчить субагентів і не тримає сотні файлів у контексті: масовий прогін не «заморює» сесію.
2. **Local-only** (ADR `260610-2228`). Будь-який файл генерується локальною моделлю через omlx; хмарних ескалацій немає. Якщо модель не дотягнула — дока пишеться з **degraded-маркером**, а не делегується в cloud.
3. **Детермінізм навколо LLM.** Усе, що можна зробити без токенів, робиться без токенів: екстракція фактів, анкорів, секція «Гарантії», пост-лінт, скоринг, перевірка застарілості.
4. **Дока поряд із кодом:** `<dir>/docs/<stem>.md` для кожного кодового файлу (`js`/`mjs`/`ts`/`vue`/`py`).

## Стани доки

Стан визначається **детерміновано** за frontmatter доки проти поточного джерела:

```
                ┌─ доки немає ──────────────► stale: missing
джерело ──CRC──►├─ crc ≠ frontmatter.crc ───► stale: crc-mismatch
                └─ crc збігається ─┬─ score ≥ порогу або відсутній ─► свіжа
                                   └─ score < порогу ───────────────► свіжа, degraded
```

Frontmatter — машинні метадані, єдиний виняток із правила «чистий Markdown»:

```markdown
---
docgen:
  source: npm/lib/llm.mjs
  crc: a3f1c9e0
  score: 55
  issues: short-behavior,internal-name:callPi
---
```

- `crc` — CRC32 байтів джерела на момент генерації. Не залежить від git-стану (rebase, гілки, незакомічене).
- `score`/`issues` — det-оцінка якості (пишуться, коли відомі; `issues` — лише для degraded). **Degraded ≠ застаріла**: CRC свіжий, гейт не блокує, борг видимий окремо.

## Команди CLI

| Команда | Що робить | Exit |
|---|---|---|
| `doc-files scan` | JSON-список усіх кодових файлів зі станом застарілості | 0 / 1 (корінь не існує) |
| `doc-files gen` | згенерувати доки для застарілих; `--overwrite` — для всіх; `--retry-degraded` — лише для degraded; `--from N --limit M` — порціями | 0 / 1 (помилка файлу або фейл preflight) |
| `doc-files stamp` | (пере)штампувати frontmatter у наявних доках без LLM (міграція; `score`/`issues` зберігаються) | 0 |
| `doc-files check` | детектор застарілості: `--hook` (один файл зі stdin), `--git` (Stop-гейт), `--degraded` (звіт), явні шляхи | 0 / 2 (стале знайдено) |

## Алгоритм `gen` (батч)

```
scan дерева ──► вибір цілей (stale | всі | degraded) ──► зріз --from/--limit
      │
      ▼
preflight omlx (1 крихітний виклик) ── фейл ──► одна зрозуміла зупинка, exit 1
      │ ok                                       (НЕ сотні «✗» по файлах)
      ▼
для кожного файлу послідовно (паралельність 1 — стеля 8GB):
      generateDoc(файл) ──► CRC джерела ──► stampDoc(md, source, crc, quality)
      │                                      └─ запис <dir>/docs/<stem>.md
      ▼
підсумок: ✓ OK: N  ⚠ degraded: D  ✗ Err: E   (+підказка --retry-degraded)
```

**Preflight** розрізняє три стани локального сервера, які вимагають різних дій:

- `down` — omlx не запущений → «запусти `omlx serve`»;
- `memory-guard` — модель (3.5GB) не влазить у **динамічну** стелю пам'яті зайнятої 8GB машини → «звільни пам'ять або повтори пізніше». Це властивість моменту, не дефект конвеєра;
- `auth` — на сервері ввімкнено API-ключ → «вистав `N_CURSOR_OMLX_KEY`» (зазвичай не потрібно: ключ авторезолвиться з `~/.omlx/settings.json`).

## Алгоритм генерації одного файлу (`generateDoc`)

```
файл
 │
 ├─ Stage 0  факти (0 ткн):  header-намір, експорти+JSDoc, класифіковані імпорти,
 │                           internalSymbols, маркери поведінки
 ├─ E1       анкори (0 ткн): URLs, експортовані константи, маркери помилок,
 │                           конфіг-посилання, приклади з header-коментаря
 │
 ├─ unsupported (vue/py)? ──► one-shot: 1 виклик на весь документ, score=null ──► готово
 │
 ├─ Stage 1  секційна генерація (по 1 виклику на секцію, див. таблицю)
 ├─ E2       критика→переписування для вразливих секцій (+0–4 виклики)
 ├─ Stage 2  пост-лінт (0 ткн): зрізати код-фенси, випадкові заголовки, сигнатури
 ├─ Stage 2.5 det-скоринг (0 ткн)
 ├─ E4       score < порогу? ──► best-of-2: повторний прогін з temperature 0.5,
 │                               перемагає кращий за score
 └─ результат { md, score, issues, degraded } ──► рішення приймає батч
```

### Секції: що бачить модель

Принцип мінімального вікна — **код потрапляє лише туди, де без нього не можна**:

| Секція | Вікно виклику | Викликів | Критика E2 |
|---|---|---|---|
| «Огляд» | лише факт-лист (без коду) | 1 | завжди (тут модель зриває на generic) |
| «Поведінка» | **єдина секція з кодом** + факти + анкори | 1 | ні |
| «Публічний API» | лише список експортів з описами | 0–1 (якщо >1 експорту) | якщо всі описи порожні |
| «Гарантії поведінки» | — | **0** (детермінований шаблон з маркерів) | — |

Факт-лист містить і **негативні маркери** («Кешування: НЕМАЄ — не згадуй кеш») — найдешевший спосіб відсікти типові галюцинації ще в промпті. Анкори підставляються блоком «обов'язкового включення», щоб текст не зісковзував у загальні фрази.

### E2: критика → переписування

Критик — та сама модель з окремою роллю і **закритим списком критеріїв** (generic-фрази, пропущені анкори, граматика, підзаголовки всередині секції, скопійовані сигнатури, вигадані факти). Відповідь — нумерований список проблем або `NONE`. Переписувач отримує чорнетку + список і повертає лише оновлений текст. Один цикл на секцію, без рекурсії.

### Stage 2.5: det-скоринг

Оцінка від 100 зі штрафами, кожен лишає код у `issues`:

| Перевірка | Штраф | Код |
|---|---|---|
| немає секції «Огляд» | −25 | `no-overview` |
| «Поведінка» < 60 символів | −20 | `short-behavior` |
| згадка кешу в «Гарантіях», коли файл не кешує (з розпізнаванням негації) | −20 | `cache-hallucination` |
| бектік-згадка внутрішнього символу у видимих секціях | −10 за символ | `internal-name:<sym>` |

`score < 70` (env `N_CURSOR_DOC_FILES_THRESHOLD`) → **degraded**: дока пишеться, маркер у frontmatter, перегенерація — `gen --retry-degraded` (після покращення конвеєра або на вільній машині). Помилка транспорту **не** маскується під degraded — це чесний `✗`.

## Транспорт

Єдина точка `callLlm` (`npm/lib/llm.mjs`); бекенд визначає **сам рядок моделі**, жодних env-перемикачів:

```
model-id ── omlx/<m> ──► прямий HTTP до omlx (127.0.0.1:8000, OpenAI-сумісний, text-only)
         └─ інакше ────► pi CLI (хмарні провайдери; повільніше на ~46% для локалі)
```

- Дефолтна модель: `N_CURSOR_DOCGEN_MODEL` → `resolveModel('min')` (каскад `N_LOCAL_*`/`N_CLOUD_*`) → `omlx/mlx-community--gemma-4-e2b-it-4bit` напряму.
- API-ключ omlx: параметр виклику → `N_CURSOR_OMLX_KEY` → `auth.api_key` із `~/.omlx/settings.json` (zero-config) → без заголовка.
- Wire-trace: `N_CURSOR_LLM_TRACE=<file>` → JSONL-рядок на кожен виклик (бекенд, модель, тривалість, розміри, успіх/помилка). Fail-safe: помилка запису трейсу не ламає виклик.

## Гейти і хуки

- **PostToolUse** (`check --hook`): після редагування кодового файлу нагадує, що його дока відстала (exit 2 → повідомлення агенту).
- **Stop-гейт** (`check --git`): перед завершенням задачі перевіряє змінені відносно HEAD джерела. Застарілі доки → блок (exit 2). Запобіжник: stale понад `N_CURSOR_DOC_FILES_GATE_MAX` (дефолт 50) → лише попередження — масовий перший прогін не може заблокувати роботу назавжди.
- **Degraded не блокує ніколи**: слабкість моделі — не привід зупиняти задачі. Видимість боргу — `check --degraded` (інформаційний, exit 0).

## Сканер: що документується

- Розширення: `.js .mjs .ts .vue .py`; виключаються тести (`*.test.*`, `*.spec.*`), `*.d.ts`.
- Ignore-дерева: `node_modules`, `dist`, `.git`, `coverage`, `__pycache__`, `.cursor`, `.claude`, усі теки `docs/` тощо — список у `docgen-ignore.mjs` (спільний для `doc-files` і `doc-aggregate` через re-export).
- Корінь із system-wide layout (`docs/adr` або `docs/explanation`) — file-level доки на верхньому рівні не плануються: кореневий `docs/` лишається для системної документації (як цей файл).

| `N_CURSOR_LLM_TRACE` | — | JSONL wire-trace |

## Відомі межі (чесно)

- `vue`/`py` поки йдуть one-shot цілим файлом без скорингу — найслабший шлях; лікується юніт-шаром (див. «Еволюція»).
- Det-скорер міряє **форму**, не фактичну точність змісту: `score=100` не гарантує правди. Семантичний det-проксі (фільтр доказів) з'являється у claims-конвеєрі.
- Паралельність — строго 1: 8GB пам'яті ділиться між моделлю і системою; саме тому існує preflight memory-guard.
- Складні файли (багато внутрішніх символів) на whole-file промптах якісно просідають — це головна мотивація наступного етапу.

## Еволюція: куди конвеєр іде далі

Цільова архітектура зафіксована у спеку [`docs/superpowers/specs/2026-06-10-doc-files-local-only-pipeline.md`](superpowers/specs/2026-06-10-doc-files-local-only-pipeline.md) і ADR `260610-2228`:

1. **Інкремент 1:** мовно-агностичний юніт-шар на AST (`oxc-parser`; vue — script-блок, py — stdlib `ast`) → плоскі per-export зрізи; двофазна генерація **claims→render** з детермінованим фільтром доказів; det-verify пакет (анкор-покриття, identifier-whitelist, дублікати, рендер-лінт) + точкові micro-refine; render best-of-2; few-shot; команда `bench`.
2. **Інкремент 2:** bottom-up мікро-резюме по call-graph (будь-яка складність локально — конструкцією), адаптивний self-consistency, кеш юнітів за hash (інкрементальні повторні прогони).

Емпірична база рішень: ADR `260610-0643` (omlx-бекенд і вибір self-critique напряму), `260610-1344`/`260610-1349` (межа pi/omlx, JS-owned loop), бенчі оркестрованого конвеєра v2.
