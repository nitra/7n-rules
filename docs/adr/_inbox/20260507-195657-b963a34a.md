## ADR Re-exec після self-upgrade у `@nitra/cursor` CLI

**Контекст:** `n-cursor.js` оновлює пакет `@nitra/cursor` на диску під час власного виконання (`upgradeNitraCursorToLatestAndBunInstall`), але ES-модулі вже завантажені у V8-процес і не перезавантажуються. Будь-яка нова логіка (таблиці міграцій `RULE_MIGRATIONS`, списки правил у `auto-rules.mjs`, функції `detectAutoRulesAndSkills`) залишається невидимою для поточного запуску.

**Рішення/Процедура/Факт:** У `npm/bin/n-cursor.js` додано функцію `reexecIfPackageVersionChanged(effectivePackageRoot)`, яка викликається у `runSync()` одразу після `upgradeNitraCursorToLatestAndBunInstall`, до `readConfig`. Функція: (1) виходить одразу, якщо `NITRA_CURSOR_REEXEC === '1'` (захист від нескінченного циклу); (2) виходить, якщо `effectivePackageRoot === BUNDLED_PACKAGE_ROOT` (апгрейду не було — той самий каталог); (3) порівнює `version` зі старого та нового `package.json` через існуючий `readBundledVersionAt`; (4) якщо версії різні — виконує `spawnSync(process.execPath, [newBinary, ...argv], { stdio: 'inherit', env: { ...process.env, NITRA_CURSOR_REEXEC: '1' } })` і завершує поточний процес з кодом дочірнього. Версія бампнута `1.8.201 → 1.8.202`, запис додано до `npm/CHANGELOG.md`.

**Обґрунтування:** Re-exec — єдиний надійний спосіб гарантувати, що після self-upgrade виконується саме новий код. ES-модулі в Node.js кешуються у V8 і не можуть бути перезавантажені в рамках одного процесу.

**Розглянуті альтернативи:** Dynamic `import()` зі свіжого шляху після апгрейду — не вирішує проблему, бо `n-cursor.js` вже виконується з npx-кешу, а не з `node_modules/`; потрібна саме повна заміна процесу через `spawnSync`.

**Зачіпає:** `npm/bin/n-cursor.js` (нова функція `reexecIfPackageVersionChanged`, виклик у `runSync`, новий import `spawnSync`), `npm/package.json` (версія `1.8.202`), `npm/CHANGELOG.md`.
