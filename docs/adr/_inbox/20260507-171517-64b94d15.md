## ADR Валідація GitHub Actions через conftest/Rego (PoC-міграція з JS)

**Контекст:** `npm/scripts/check-ga.mjs` містить ~1074 рядки JS для структурної валідації `.github/workflows/`. Більша частина — декларативні перевірки канонічних полів (concurrency, triggers, steps, runs-on), які виграють від Rego-декларацій порівняно з imperative JS-парсингом.

**Рішення/Процедура/Факт:**
- Створено `npm/policy/ga/clean-ga-workflows.rego` — Rego v1-порт `validateCleanGaWorkflows` (~95 рядків JS → ~110 рядків декларативного Rego). 12 правил покривають: `name`, `on.schedule.cron`, `on.workflow_dispatch`, `concurrency.group/cancel-in-progress`, `jobs.*.runs-on`, `jobs.*.permissions`, `jobs.*.steps` (порядок, uses, відсутність заборонених кроків), `save_period`.
- В `npm/scripts/lint-ga.mjs` додано `runConftestStep()` з graceful fallback: якщо `conftest` відсутній у PATH — крок пропускається з `ℹ`-повідомленням (не ламає pipeline); якщо є порушення — lint-ga падає з exit-кодом conftest.
- `npm/package.json` → `files[]` додано `"policy"`, щоб `.rego` публікувалися в npm-tarball.
- JS-перевірку `validateCleanGaWorkflows` в `check-ga.mjs` **навмисно не видалено** — side-by-side PoC до підтвердження підходу.

**Обґрунтування:** Декларативне Rego читається як таблиця вимог без helper-функцій (`getObjKey`, `isExactString`). conftest CLI є self-contained binary, вже є механізм preflight-перевірок у `lint-ga`. Fallback-логіка дозволяє поступову міграцію без розриву наявного CI.

**Розглянуті альтернативи:** Залишити все в JS (відхилено: занадто багато imperative-коду для структурних правил); повністю замінити JS на Rego одразу (відхилено: file-existence/git/binary-preflight перевірки не мігрують у conftest).

**Зачіпає:** `npm/policy/ga/clean-ga-workflows.rego` (новий), `npm/scripts/lint-ga.mjs`, `npm/package.json`

## Knowledge YAML 1.1 `on:` key парситься conftest як `input["true"]`

**Контекст:** GitHub Actions workflows використовують ключ `on:` для тригерів. При запуску `conftest test` на такому YAML-файлі виникають помилки типу "відсутній schedule/workflow_dispatch".

**Рішення/Процедура/Факт:** Conftest (через OPA) парсить YAML, де `on:` є булевим `true` за стандартом YAML 1.1. Ключ зберігається не як рядок `"on"`, а як рядок `"true"` в Rego-об'єкті `input`. Правильна адресація: `input["true"]` або через аліас `gha_on := input["true"]`. Варіанти `input.on`, `input["on"]`, `input[true]` (булевий) — усі хибні.

**Обґрунтування:** Діагностовано через `conftest parse` — виведений JSON показав `"true": { "schedule": [...] }` замість `"on": {...}`. Підтверджено мінімальними тест-полісями.

**Розглянуті альтернативи:** не обговорювалися

**Зачіпає:** Будь-який Rego-файл, що перевіряє GitHub Actions workflow через conftest; `npm/policy/ga/clean-ga-workflows.rego`
