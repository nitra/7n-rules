## ADR Пріоритет oxlint над ESLint у `@nitra/eslint-config`

**Контекст:** У `npm/index.js` кінцевий крок `getConfig(...)` пушив `oxlint.configs['flat/recommended']`, який вимикає в ESLint лише правила з *recommended-пресету* oxlint, а не ті, що реально увімкнені у `.oxlintrc.json` споживача. Через це одні й ті самі перевірки виконувались обома інструментами паралельно, порушуючи принцип «oxlint = джерело правди для overlap-правил» із `.cursor/rules/oxlint-eslint-config.mdc`.

**Рішення/Процедура/Факт:** У `npm/index.js` замінити `oxlint.configs['flat/recommended']` на виклик `buildFromOxlintConfigFile('.oxlintrc.json')` (або еквівалентного API із `eslint-plugin-oxlint`), який генерує flat-конфіг, що вимикає в ESLint точно ті правила, увімкнені у фактичному `.oxlintrc.json` споживача. Шлях резолвиться відносно `process.cwd()`. Якщо файл відсутній — fallback на `flat/recommended` із `console.warn`. Сигнатура `getConfig({ node, vue, vue2 })` не змінюється; новий параметр `oxlintConfigFile` — необов'язковий. Версія пакету bump-ається `3.9.2 → 3.10.0`, до `npm/CHANGELOG.md` додається запис `## [3.10.0]`. Додається bun-тест із тимчасовим `.oxlintrc.json`, що підтверджує: правила з нього з'являються як `'off'` у вихідному масиві `getConfig`.

**Обґрунтування:** `.oxlintrc.json` — канонічний конфіг oxlint, синхронізований із `@nitra/cursor`. Саме він визначає реальну поведінку лінтера у споживача. Використання `flat/recommended` як бази вимикання — неточне: вмикання у рекомендованому наборі ≠ вмиканню у кастомному конфізі. `eslint-plugin-oxlint` надає спеціальне API (`buildFromOxlintConfigFile`) саме для цього сценарію, що дозволяє підтримувати узгодженість автоматично при оновленні `.oxlintrc.json`, без ручних override-ів.

**Розглянуті альтернативи:** Залишити `flat/recommended` — відхилено (неповне перекриття); вручну вимикати правила у `npm/index.js` — відхилено (порушує принцип «не дублюй канон вручну»).

**Зачіпає:** `npm/index.js` (логіка `getConfig`), `npm/package.json` (bump версії), `npm/CHANGELOG.md`, нові bun-тести в `npm/*.test.mjs`; публічний API `getConfig` — зворотньо сумісний.
