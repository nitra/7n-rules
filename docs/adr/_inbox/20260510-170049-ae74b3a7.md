## ADR Порт per-document K8s правил з JavaScript у Rego-полісі

**Контекст:** `npm/scripts/check-k8s.mjs` (6580 рядків) містив усі перевірки K8s-маніфестів у JavaScript — як per-document структурні перевірки (що легко портуються в OPA/Rego), так і cross-file / filesystem / autofix логіку. Rego-полісі в `npm/policy/k8s/` існував лише в зачатку (`manifest.rego`, 132 рядки, лише 3 правила). Потреба: перенести максимум pure-document правил у Rego для можливості запуску через IDE-розширення OPA та `conftest` без виклику всього JS-скрипта.

**Рішення/Процедура/Факт:** Виконано 6 фаз портування — створено 10 нових Rego-пакетів і розширено існуючий `manifest.rego`. Усі пакети і їх `*_test.rego` розміщено у `npm/policy/k8s/`:
- `manifest/` — розширено: Deployment `resources.requests.cpu+memory`, Hasura image pin (allowlist), `topologySpreadConstraints` canonical app-label
- `gateway/` — Gateway API backendRef обов'язковий суфікс `-hl`, заборона redundant `namespace`; `HealthCheckPolicy` targetRef суфікс `-hl`
- `kustomization/` — алфавітне сортування `resources[]` і `patches[]`, конфлікт JSON6902 `remove`+`add` на тому самому path
- `svc_yaml/` і `svc_hl_yaml/` — `Service.spec.type: ClusterIP` та headless `spec.clusterIP: None`
- `base_manifest/` і `base_kustomization/` — canonical `cpu='0.02'`/`memory='128Mi'` для base-шару, обов'язковий `metadata.namespace` у `base/`
- `kustomize_managed/` — заборона `metadata.namespace` у файлах під kustomize-tree
- `hasura_configmap/` — обов'язковий `HASURA_GRAPHQL_ENABLE_REMOTE_SCHEMA_PERMISSIONS: "true"`
- `hasura_httproute/` — canonical 4-rule структура Hasura HTTPRoute
- `hpa_pdb/` — структурний gate HPA/PDB (apiVersion, `behavior`, `metrics`, `minAvailable`)

Оркестрація: `lint-rego.mjs` доповнено опційним `conftest verify -p npm/policy`; `lint-conftest.mjs` отримав 7 нових TARGETS. `.cursor/rules/n-k8s.mdc` доповнено розділом з мапою пакетів. Версія: `1.8.221 → 1.8.222`.

**Обґрунтування:** Rego-полісі дають змогу запускати перевірки без Node.js-середовища (IDE OPA extension, `conftest` у будь-якому pipeline), а також формалізують правила у декларативній мові з вбудованою системою тестів. Cross-file логіка (резолюція kustomize-tree, пари svc/svc-hl, HPA/PDB по каталогу, env-залежні межі) залишається у `check-k8s.mjs` — OPA не має доступу до файлової системи й не вміє autofix; JS лишається authoritative для цих кейсів. Поділ — «Rego як швидкий gate одиничного маніфеста, JS як cross-file оркестратор» — явно зафіксовано у коментарі `manifest.rego` і в `n-k8s.mdc`.

**Розглянуті альтернативи:** Не обговорювалися явно; обраний підхід «інкрементальний порт лише pure-document правил» виник у аналізі асистента (фаза A/B/C/D класифікація), який користувач підтвердив.

**Зачіпає:**
- `npm/policy/k8s/**/*.rego` — 11 пакетів (10 нових + розширений `manifest`), 10 `*_test.rego` файлів
- `npm/scripts/lint-rego.mjs` — новий крок `conftest verify`
- `npm/scripts/lint-conftest.mjs` — 7 нових TARGETS
- `.cursor/rules/n-k8s.mdc` — новий розділ «Швидкий gate через conftest (Rego)»
- `npm/package.json` і `npm/CHANGELOG.md` — версія `1.8.222`
- `.cspell.json` — розширений allowlist
