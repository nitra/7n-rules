Ідея зрозуміла. Це «convention over configuration» на максимумі — жодного `rule.json`, жодного `TARGETS`. Є два аспекти, і вони принципово різні.

## JS — це вже майже так

Сьогодні `discoverCheckScripts` шукає `rules/<id>/js/check.mjs` за іменем. Якщо генералізувати на «будь-який `*.mjs` у `js/`, що експортує `check()`» — ніяких перешкод. Кожен такий файл уже самодостатній: він сам знає, які файли обходити, який gating застосовувати, що вважати `fail`. Проблеми немає.

## Rego — є одне єдине питання

Rego-полісі не є самодостатніми в тому самому сенсі: `.rego` описує **правило перевірки**, але не описує **які файли проєкту годувати conftest-у**. `conftest test` вимагає явного списку файлів як аргументи. Без нього — нічого не запускається.

Є рівно два виходи:

### Варіант A — «годуй усе, нехай Rego само фільтрує»

CLI: `walkDir` → всі `.yaml`/`.json` → один `conftest test <всі файли> -p npm/rules/<id>/policy/ --all-namespaces`.

```
conftest test project/**/*.yaml project/**/*.json \
-p npm/rules/rego/policy/ \
--all-namespaces
```

Конвенція: кожне `deny` у `.rego` **обовʼязково** має `input.kind == "X"` або `input.name == "package.json"` або інший структурний гейт — інакше полісі стрілятиме на будь-який YAML.

**Плюс:** нульова конфігурація.
**Мінус:** `package.json`-полісі (`bun.package_json`, `text.cspell`) написані без `input.kind`-гейта — вони зараз перевіряють структуру JSON і впадуть на будь-якому `deployment.yaml`. Треба або дописати їм гейти, або прийняти, що деякі полісі завжди пишуться з `input.apiVersion`/`input.kind` чи аналогом.

### Варіант B — companion `target` у самому каталозі полісі

Мінімальна конвенція без `rule.json`: поряд з `<name>.rego` кладемо `<name>.target` — один рядок з glob:

```
# npm/rules/rego/policy/vscode_extensions/vscode_extensions.target
.vscode/extensions.json
```

CLI: бачить `.rego` → шукає `.target` поруч → якщо є, використовує як glob для `walkDir` або `existsSync`; якщо немає — fallback на «всі YAML/JSON» (варіант A).

**Плюс:** targeting залишається в каталозі полісі, без зовнішніх реєстрів.
**Мінус:** +1 файл на кожну полісі. Для `hc.yaml`-батчу (де список файлів обчислюваний JS-логікою) `.target` не виражає «знайди всі `hc.yaml` поруч з `Deployment`» — ці полісі все одно потребують JS.

---

**Де реально стоїть межа:**

| Тип полісі | Варіант A | Варіант B |
|---|---|---|
| `input.kind == "HTTPRoute"` — будь-який yaml | ✅ | ✅ |
| `package.json` кореневий | ⚠️ треба гейт у .rego | ✅ `.target: package.json` |
| `.vscode/settings.json` | ⚠️ треба гейт | ✅ `.target: .vscode/settings.json` |
| `hc.yaml` поруч з Deployment | ❌ не виразити | ❌ не виразити |

Рядок «не виразити» — це єдиний клас, де JS-оркестратор незамінний. Все інше вкривається або A, або B.

**Практична рекомендація:** прийняти варіант B як конвенцію для pure-Rego правил (`.rego` + `.target` поряд), варіант A як fallback, і JS лишити тільки для обчислюваних батчів типу `health_check_policy`. Це максимальне спрощення без втрати жодної реальної перевірки.
