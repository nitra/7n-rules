## ADR Міграція check-скриптів на Rego/conftest: аналіз та перший порт (workflow_common)

**Контекст:** Проєкт має набір `npm/scripts/check-*.mjs` (JS-скриптів), які дублюють перевірки, що вже частково реалізовані у Rego-поліси під `npm/policy/ga/`. Виникло питання: які скрипти ефективно перевести на `conftest`/Rego, а які лишити у JS.

**Рішення/Процедура/Факт:**
1. Проведено аналіз усіх `npm/scripts/check-*.mjs` за критерієм «чи є ядро перевірки валідацією JSON/YAML/TOML-структур без виклику зовнішніх CLI чи AST-сканів».
2. Визначено три категорії:
- **Повний порт** (ядро — field-checking): `check-bun`, `check-text`, `check-style-lint`, `check-php`, `check-npm-module`, `check-k8s`, продовження `check-ga`.
- **Гібрид** (JSON/YAML-частина → Rego; AST/FS/CLI-частина → JS): `check-js-lint`, `check-js-mssql`, `check-js-bun-db`, `check-js-run`, `check-vue`, `check-capacitor`, `check-graphql`, `check-image-compress`, `check-hasura`, `check-adr`.
- **Не для Rego**: `lint-ga.mjs`, `lint-rego.mjs`, `run-*.mjs`, `check-docker`, `check-image-avif`, `check-nginx-default-tpl`, `check-changelog`.
3. Як перший порт створено `npm/policy/ga/workflow_common/workflow_common.rego` — 4 універсальні правила для **будь-якого** `.github/workflows/*.yml`: заборона `oven-sh/setup-bun`/`actions/cache`, заборона backslash line-continuation у `run:`, перевірка порядку `checkout` перед `setup-bun-deps`, наявність секції `concurrency`.
4. Баг: `not is_object(input.concurrency)` повертає `undefined` (а не `true`) коли поля немає — виправлено через `object.get(input, "concurrency", null); _ == null`.
5. `lint-ga.mjs` підключено до `workflow_common`: тепер conftest прогонить поліс на **кожному** знайденому `*.yml` з `glob .github/workflows/`.
6. Злито два `import` (`existsSync`, `readdirSync`) в один деструктурований імпорт.

**Обґрунтування:** Rego/conftest ефективніший за JS для декларативних file-checks: декларативна логіка коротша, conftest нативно парсить YAML/JSON/TOML, і результати вбудовуються у той самий `conftest test` пайплайн, що вже використовує `lint-ga.mjs`. Гібридна стратегія (Rego для конфігів, JS для AST) дозволяє не переписувати `oxc-parser`/CLI-логіку, де виграш від Rego відсутній.

**Розглянуті альтернативи:** Лишити всі перевірки у JS — відхилено, бо вже існував прецедент (`lint_ga`, `clean_ga_workflows`, `git_ai` у Rego) і обсяг boilerplate у JS-скриптах зростає.

**Зачіпає:** `npm/policy/ga/workflow_common/workflow_common.rego` (новий файл), `npm/scripts/lint-ga.mjs` (додано glob-прогін `workflow_common` на всіх workflows).
