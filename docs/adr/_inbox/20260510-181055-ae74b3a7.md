## ADR Стратегія «Rego-first» для пер-документних перевірок

**Контекст:** У проєкті паралельно існують два способи валідації YAML-файлів: JS-функції у `npm/scripts/check-*.mjs` (запускаються через `npx @nitra/cursor check`) і Rego-полісі у `npm/policy/` (запускаються через `conftest`). Виникло питання — де реалізовувати нові правила та як уникнути дрифту між двома джерелами істини.

**Рішення/Процедура/Факт:** До `.cursor/rules/conftest.mdc` (`alwaysApply: true`) і `npm/.claude-template/npm-CLAUDE.md` (path-scoped) додано явний алгоритм вибору: **default — Rego** для пер-документних структурних перевірок (kind/apiVersion, поля, форма масивів), **JS — тільки** для cross-file resolution (kustomize-tree, парність файлів), FS-операцій, autofix/rewrite, raw-text-парсингу до YAML, HTTP-запитів. **Гібрид**: якщо задача частково per-document — пишуться обидва шари; `k8s.*/check-k8s.mjs` є референсною реалізацією. Оркестратор `lint-rego.mjs` розширено опційним `conftest verify` (graceful skip якщо `conftest` не в PATH). У `npm/CLAUDE.md` (регенерується через `npx @nitra/cursor`) з'явився розділ «Перш ніж писати `check-*.mjs`» з посиланням на алгоритм.

**Обґрунтування:** Rego-полісі є read-only (IDE-extension `tsandall.opa` дає інлайн-фідбек без запуску CLI), декларативні та легко тестовані через `conftest verify`. JS залишається авторитативним для всього, що Rego не може: side effects, FS-traversal, cross-file контекст. Кодування алгоритму в `alwaysApply`-правилі гарантує, що агент побачить вимогу до вибору без явної підказки.

**Розглянуті альтернативи:** (A) Залишити рішення на розсуд розробника — відхилено через непослідовність. (B) Повний перехід на Rego (видалити JS-предикати) — відхилено через залежність від `conftest` у PATH та накладні витрати спавну (~50-100 ms/файл) для великих k8s-дерев.

**Зачіпає:** `.cursor/rules/conftest.mdc`, `npm/.claude-template/npm-CLAUDE.md`, `npm/CLAUDE.md` (генерований), `npm/scripts/lint-rego.mjs`, `npm/scripts/lint-conftest.mjs`.

## ADR Паттерн «JS authoritative + Rego mirror» та відкрите питання дрифту

**Контекст:** Після міграції `check-k8s.mjs` та частини `check-abie.mjs` до Rego-пакетів виникло питання: чи потрібно видаляти з JS-скриптів предикати, які тепер продубльовані у Rego? Під час роботи знайдено конкретний дрифт: `health_check_policy.rego` читав `spec.config.httpHealthCheck`, тоді як JS-авторитет читав `spec.default.config.httpHealthCheck` — і розбіжність жила непоміченою.

**Рішення/Процедура/Факт:** Прийнято тримати JS-предикати як є (варіант A) без видалення. Rego-пакети є **дзеркалом** (швидкий gate, IDE-фідбек), JS-функції — **авторитетом** (source of truth для `npx @nitra/cursor check`). Виявлений дрифт у `health_check_policy.rego` виправлено: пакет переписано з повним набором перевірок (`apiVersion: networking.gke.io/v1`, `metadata.name`, `spec.default.config.type: HTTP`, `targetRef.kind: Service`). Додано 10 тестів у `health_check_policy_test.rego`.

**Обґрунтування:** Варіант B (Rego authoritative, JS спавнить `conftest`) вимагає `conftest` як обов'язкову залежність і вповільнює `check` для великих дерев. Варіант C (JS-проксі через `opa eval`) аналогічно вповільнює і ускладнює unit-тестування. Поточний дрифт-ризик можна мітигувати **golden cross-check тестами** у `*.test.mjs` — фікстура виганяється через JS-предикат і через `conftest test`, обидва вердикти мають збігатися.

**Розглянуті альтернативи:** (B) Rego authoritative + JS спавнить conftest — явно розглянуто, відкладено через залежності та продуктивність. (C) Тонкі JS-проксі через `opa eval` — явно розглянуто, відкладено. (golden cross-check тести) — запропоновано як наступний крок для варіанту A.

**Зачіпає:** `npm/policy/abie/health_check_policy/`, `npm/scripts/check-abie.mjs`, архітектура всіх rego-дзеркал у `npm/policy/`.

## Knowledge Rego `object.get` для відсутніх вкладених полів — обов'язковий патерн

**Контекст:** Під час написання юніт-тестів для rego-пакетів `k8s.*` та `abie.*` (128 тестів) виявлено поведінку OPA/conftest: вираз `not is_object(input.spec)`, коли поле `spec` взагалі відсутнє в `input`, **не виконується** (є `undefined`, а не `false`), тому `deny`-правило ніколи не спрацьовує для документів без поля.

**Рішення/Процедура/Факт:** Для перевірки відсутності або некоректності вкладеного поля потрібно використовувати `object.get(input, "spec", null)` замість прямого `input.spec`. Правильний патерн: `not is_object(object.get(input, "spec", null))`. Те ж стосується будь-якого вкладеного об'єкта на першому рівні: `input.metadata`, `input.data`, `input.spec.rules`. Виправлено у: `svc_yaml.rego`, `svc_hl_yaml.rego`, `base_manifest.rego`, `hasura_configmap.rego`, `hasura_httproute.rego`, `hpa_pdb.rego`, `kustomize_managed.rego`. До виправлення тести на відсутні поля проходили в `opa eval`, але падали в `conftest verify` — оскільки conftest суворіше до типів.

**Обґрунтування:** OPA v1 (rego.v1) з type-checking розрізняє `undefined` і `false`; `is_object(undefined)` повертає `undefined`, а не `false`, тому `not is_object(undefined)` — теж `undefined`, а не `true`. `object.get(obj, key, default)` перетворює відсутність на явний `null`, і далі `not is_object(null)` — `true`.

**Розглянуті альтернативи:** не обговорювалися.

**Зачіпає:** Усі Rego-пакети у `npm/policy/` — будь-який пакет, що перевіряє `not is_object(input.X)` на першому або вкладеному рівні.
