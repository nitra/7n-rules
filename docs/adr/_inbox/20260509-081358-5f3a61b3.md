## ADR Правило js-bun-redis: автовиявлення та програмна перевірка

**Контекст:** Проєкт вже мав аналогічну систему правил для `js-bun-db` (заміна `pg`/`mysql2` на Bun native SQL). Потрібно створити симетричне правило `js-bun-redis` — щоб автоматично виявляти використання `ioredis`/`node-redis`/`redis` та направляти команду на `import { redis } from 'bun'`.

**Рішення/Процедура/Факт:**
- `npm/scripts/utils/redis-imports.mjs` — AST-сканер (oxc-parser): ловить `import`/`require`/dynamic `import()` пакетів `ioredis`, `node-redis`, `redis`, `@redis/*` та підшляхів; не зачіпає сторонні `redis-*` (наприклад, `redis-mock`).
- `npm/scripts/check-js-bun-redis.mjs` — check-скрипт, доступний як `npx @nitra/cursor check js-bun-redis`; шанує `.n-cursor.json#ignore`.
- `npm/policy/js_bun_redis/package_json/package_json.rego` — Rego-полісі під `bun run lint-conftest`: забороняє ті самі пакети у `dependencies`; зареєстрована в `npm/scripts/lint-conftest.mjs`.
- `npm/scripts/auto-rules.mjs` — додано `js-bun-redis` до `AUTO_RULE_ORDER`; `collectDependencyKeysPresentInPackageJsonTree` розширено ключами `ioredis`/`node-redis`.
- Тести: `npm/tests/redis-imports.test.mjs` (новий) + оновлений `npm/tests/auto-rules.test.mjs` з `ALL_RULES` і кейсом авто-увімкнення — 36/36 pass.
- Версія bump `1.8.212 → 1.8.213`, запис у `npm/CHANGELOG.md`.

**Обґрунтування:** Архітектурне рішення — дворівнева перевірка (AST у check-скрипті + Rego у lint-conftest) — повністю симетрична до `js-bun-db`, щоб однотипні правила мали однотипну реалізацію і покриття. Автоувімкнення знижує бар'єр: розробник додає `ioredis` у новий сервіс → правило саме з'являється в конфіги. Ціль — одна залежність на Redis у Bun-проєктах, менше ручного керування підключеннями.

**Розглянуті альтернативи:** Не обговорювалися — підхід диктувався існуючим патерном `js-bun-db`.
