## ADR Виключення Kustomize Component з prod-overlay-перевірки HPA/PDB

**Контекст:** У `@nitra/cursor@1.8.219` введено канон HPA+PDB через `<pkg>/k8s/components/` (Kustomize Component, `kind: Component`). Функція `prodOverlayHpaPdbOverrideNeeds` в `scripts/check-k8s.mjs` трактувала будь-який `kustomization.yaml` поза `base/`/`dev/`/`*-qa/` як прод-overlay і вимагала JSON6902-патчів для `spec.minReplicas`, `spec.maxReplicas`, `spec.minAvailable`. Через це `components/kustomization.yaml` (джерело ресурсів, не overlay) хибно тригерив 45 помилок (15 пакетів × 3 поля).

**Рішення/Процедура/Факт:**
- `scripts/check-k8s.mjs` — в `prodOverlayHpaPdbOverrideNeeds` (~рядок 5240) додано ранній `return { needsHpaReplicaPatches: false, needsPdbMinAvailablePatch: false }`, якщо перший YAML-документ `kustomization.yaml` містить `kind: Component`. JSDoc функції оновлено з явним описом цього винятку.
- `mdc/k8s.mdc` — до розділу `components/` додано абзац про те, що `components/kustomization.yaml` з `kind: Component` є джерелом ресурсів для overlays, а не overlay сам по собі; `check k8s` не вимагає від нього прод-патчів.
- `tests/check-k8s-schema.test.mjs` — новий тест: `components/kustomization.yaml` з `kind: Component` → `{ needsHpaReplicaPatches: false, needsPdbMinAvailablePatch: false }`. Регресійний кейс для звичайного прод-overlay (без `kind: Component`) залишився незмінним.
- `package.json`: `1.8.219` → `1.8.220`; `CHANGELOG.md`: запис `[1.8.220] - 2026-05-09`, секція `### Fixed`.

**Обґрунтування:** Kustomize Component (`kind: Component`) — це env-нейтральне джерело ресурсів HPA/PDB, яке overlays підключають через `components:` і потім перевизначають через JSON6902-патчі. Вимагати від самого Component прод-патчів логічно неправильно: він не знає контексту оточення. Перевірка наявності `kind: Component` через `readFirstYamlObject` — мінімальний та прямолінійний критерій розмежування.

**Розглянуті альтернативи:** Не обговорювалися явно; технічне завдання однозначно вказало конкретну реалізацію (ранній `return` на основі `kind: Component`).

**Зачіпає:** `npm/scripts/check-k8s.mjs` (функція `prodOverlayHpaPdbOverrideNeeds`), `npm/mdc/k8s.mdc` (документація розділу `components/`), `npm/tests/check-k8s-schema.test.mjs` (нова фікстура), `npm/package.json`, `npm/CHANGELOG.md`.

## ADR Канонічна структура HPA + PDB через Kustomize Component (`components/`)

**Контекст:** До версії `1.8.219` в `@nitra/cursor` HPA у `base/hpa.yaml` заборонявся правилом, але PDB аналогічної заборони не мав. `base/` накопичував env-специфічні ресурси, ускладнюючи dev-середовище; патч `$patch: delete` у kustomization-overlay використовувався як механізм виключення HPA для dev — крихкий і нестандартний підхід.

**Рішення/Процедура/Факт:**
- Введено єдиний sibling-каталог `<pkg>/k8s/components/` на сервіс (фіксована назва, не `scale/`, не `hpa-component/`) з `kustomization.yaml` (`kind: Component`), `hpa.yaml`, `pdb.yaml`.
- `base/` не підключає `components/`; у dev-середовищі HPA та PDB відсутні повністю.
- Прод-overlays (`ru/`, `ua/`) додають `components: [- ../components]` та JSON6902-патчі для `/spec/minReplicas`, `/spec/maxReplicas`, `/spec/minAvailable`.
- Заборонено `base/hpa.yaml` і `base/pdb.yaml` (file-existence check у `scripts/check-k8s.mjs`).
- `scripts/check-k8s.mjs`: нова експортована функція `validateComponentsForBaseDeployment`, константа `COMPONENTS_DIR = 'components'`, приватна `verifyK8sBaseKustomizeHasNoHpaPdb`. Видалено мертвий код: `patchTextDeclaresHpaStrategicDelete`, `kustomizationDeclaresHpaStrategicDelete`, `verifyK8sBaseKustomizeHpaDeletedWhenInherited`, `verifyK8sBaseKustomizeHpaPdbNeedDeployment`, регекспи `HPA_STRATEGIC_DELETE_RE`/`HPA_KIND_LINE_RE`.
- `mdc/k8s.mdc`: переписано розділ "Deployment: topologySpreadConstraints, HPA / PDB через `components/`" — канонічні приклади `components/{kustomization,hpa,pdb}.yaml` і прод-overlay.
- `tests/check-k8s-schema.test.mjs`: видалено `describe('patchTextDeclaresHpaStrategicDelete')`, оновлено `prodOverlayHpaPdbOverrideNeeds` тести, додано `describe('validateComponentsForBaseDeployment')` з 5 кейсами.
- `package.json`: `1.8.218` → `1.8.219`; `CHANGELOG.md`: запис `[1.8.219] - 2026-05-09`.

**Обґрунтування:** Kustomize Component дозволяє зберігати HPA/PDB один раз (env-нейтрально) і підключати їх лише у потрібних overlays без `$patch: delete`-хаків. `base/` залишається чистим dev-станом. Фіксована назва `components/` (без варіантів) спрощує документацію, повідомлення про помилки і автоматичні перевірки.

**Розглянуті альтернативи:** `$patch: delete` у base-kustomization (попередній підхід — прибрано як крихкий); альтернативні назви каталогу (`scale/`, `hpa-component/`, `pdb-component/`) — відхилено на користь єдиного `components/`.

**Зачіпає:** `npm/scripts/check-k8s.mjs` (функції `validateComponentsForBaseDeployment`, `verifyK8sBaseKustomizeHasNoHpaPdb`, константа `COMPONENTS_DIR`), `npm/mdc/k8s.mdc` (документація, приклади), `npm/tests/check-k8s-schema.test.mjs`, `npm/package.json`, `npm/CHANGELOG.md`.
