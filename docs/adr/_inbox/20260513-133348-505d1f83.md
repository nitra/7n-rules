## ADR Rego-полісі для VSCode/workflow канонів замість JS-дублікатів

**Контекст:** У `npm/scripts/check-*.mjs` накопичились функції перевірки VSCode-конфігів (`.vscode/extensions.json`, `.vscode/settings.json`) та workflow-канонів, які дублювали логіку вже наявних або потенційних rego-полісів. Паралельно `docker.mdc` містив застарілий приклад `lint-docker.yml` з `oven-sh/setup-bun@v2` / `actions/cache@v5` / `bun install` безпосередньо — що суперечило `ga.workflow_common.rego`, де ці кроки заборонені на користь composite-action `./.github/actions/setup-bun-deps`.

**Рішення/Процедура/Факт:**
- Видалено JS-функції `checkVscodeStylelint`, `checkVscodeText*`, `checkVscodeNginx`, тіло `checkExtensionsRecommendation` з `check-style-lint.mjs`, `check-text.mjs`, `check-nginx-default-tpl.mjs`, `check-graphql.mjs`.
- Для **unconditional** правил (style-lint, text) rego-полісі зареєстровано глобально у TARGETS `lint-conftest.mjs` (`rule: '<name>'` + `single: '<file>'`).
- Для **conditional** правил (graphql — за наявності `gql\`…\``, nginx — за наявності `default.conf.template`) залишено JS-orchestrator із `runConftestBatch(...)` для запуску conftest лише коли умова виконана.
- Узгоджено `docker.mdc` (v1.8 → v1.9): канонічний приклад `lint-docker.yml` замінено зі старих 4 кроків на `uses: ./.github/actions/setup-bun-deps`.
- Додано нові rego-пакети: `text.vscode_extensions`, `text.vscode_settings`, `style_lint.vscode_extensions`, `style_lint.vscode_settings`, `graphql.vscode_extensions`, `nginx_default_tpl.vscode_extensions`, `nginx_default_tpl.vscode_settings`, `docker.package_json`, `docker.lint_docker_yml`, `js_run.jsconfig` (registration), `image_avif.package_json`.
- Від 183 до 267 rego-тестів через json.patch-фікстури.
- Версії: `1.9.13 → 1.9.18` за фазами.

**Обґрунтування:** JS-перевірки жорстко прив'язані до синтаксичного розбору конкретного файлу і не дають machine-readable артефактів — rego дає перевіряємі deny-правила з іменованим простором, що можна тестувати ізольовано через `conftest verify`. Conditional правила не можна реєструвати глобально у TARGETS без JS-orchestrator: `lint-conftest` фільтрує тільки за активними правилами у `.n-cursor.json`, але не за per-workspace умовами (наявність `gql`, `default.conf.template`). Коли полісі конфліктує з іншою (docker.mdc vs `ga.workflow_common`), спочатку узгоджується документація.

**Розглянуті альтернативи:**
- Залишити JS-перевірки: відхилено — дублювання логіки, немає ізольованих тестів.
- Зареєструвати conditional правила (graphql, nginx) глобально у TARGETS: відхилено — спричиняє false-positive порушень на проєктах без відповідної технології.
- Одна глобальна rego-полісі для VSCode замість per-rule: не обговорювалась, але рішення — окремі полісі per-rule для точного фільтрування через `rule:` у TARGETS.
