## ADR Декларативний targeting Rego-полісі та конвенція discovery для JS-чекерів

**Контекст:** Сьогоднішній CLI (`n-cursor.js`) запускає перевірки правил лише якщо є `rules/<id>/js/check.mjs`; цей файл одночасно містить cross-file логіку, умовні гейти та виклики `runConftestBatch`. Зростання правил (30+) виявило три проблеми: pure-Rego правила «тихо випадають» з перевірки без JS-обгортки; `check.mjs` окремих правил розростаються (abie — 1153 рядки); targeting (який файл годувати якій полісі) дублюється між `lint-conftest.mjs:TARGETS` і `runConftestBatch`-викликами у кожному `check.mjs`.

**Рішення/Процедура/Факт:**

1. **JS-discovery (конвенція):** CLI імпортує та запускає всі `rules/<id>/js/check.mjs` і `rules/<id>/js/check-*.mjs`; `*.test.mjs` ігноруються, шейред-хелпери живуть у `rules/<id>/js/utils/` або `npm/scripts/utils/`. Порядок — алфавіт. Великі `check.mjs` (як `abie`) розпилюються на `check-firebase.mjs`, `check-k8s-base.mjs` тощо. Спільний стан між файлами — module-level singleton-кеш у хелпері (`let cachedYaml = null; if (cachedYaml) return cachedYaml`), перший виклик платить за `walkDir`, наступні — кешовано безкоштовно.

2. **Rego-targeting (декларація per-policy):** Поряд із кожним `policy/<name>/<name>.rego` додається `target.json`:
```json
{ "single": "package.json", "required": true }
// або
{ "walkGlob": "**/k8s/**/base/**/*.{yaml,yml}", "required": false }
// + опційно
{ "missingMessage": "..." }
```
CLI автоматично знаходить усі `policy/**/target.json`, резолвить файли і передає у `runConftestBatch`. Без `target.json` — polісі пропускається з попередженням.

3. **`applies`-гейтинг (inline в JS):** Conditional-логіка «правило abie увімкнено в `.n-cursor.json`» або «в репо є `.rego`» лишається у відповідному `check-*.mjs` (як `isAbieRuleEnabled`, `projectHasRegoFiles` зараз). Окремий `rule.json` з `applies`-блоком **не вводиться** — для pure-Rego правил без жодного `.mjs` це питання відкладається.

4. **Glob-парсер:** обрати той, що швидше за бенчмарком для pattern типу `**/*.{yaml,yml}` (кандидати: `picomatch`, вбудований `Bun.Glob`); залежності прийнятні.

**Обґрунтування:** Конвенція `check-*.mjs` усуває монолітні 1000-рядкові файли без нового DSL; singleton-кеш дозволяє розбити файли без регресії по продуктивності `walkDir`. `target.json` per-policy — locality principle: таргет живе поряд із полісі, а не у глобальному реєстрі `TARGETS`. `applies` inline у JS прийнятний, поки кожне правило має хоч один `.mjs`; перехід на декларативний `rule.json` — окреме рішення, коли зʼявляться pure-Rego правила без `.mjs`.

**Розглянуті альтернативи:**
- `rule.json` з `applies` + `targets` (єдиний конфіг на правило) — відхилено на користь per-policy `target.json` для кращої locality;
- `conftest test .` (feed-everything, варіант A) — відхилено: потребує self-gating у кожній полісі по `input.kind`, не покриває безкіндових JSON (`.cspell.json`, `package.json`), продуктивність: ~30 повних обходів дерева на прогін;
- name-convention `policy/package_json/` → `single: "package.json"` — відхилено: конфлікт при однакових іменах з різними матчерами (root single vs workspace walk).

**Зачіпає:** `npm/bin/n-cursor.js` (`discoverCheckScripts` → `discoverCheckableRules`, `runChecks`), `npm/rules/*/js/check.mjs` (розпиляти на `check-*.mjs` + `js/utils/`), `npm/rules/*/policy/<name>/` (додати `target.json`), `npm/scripts/utils/run-conftest-batch.mjs` (викликатиметься CLI напряму, не лише з JS), `npm/scripts/lint-conftest.mjs:TARGETS` (поступово замінюється `target.json`-файлами).
