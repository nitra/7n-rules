## ADR Правило компактності npm-пакета та rego-authoritative перевірки

**Контекст:** Пакет `@nitra/cursor` публікувався з надлишковим вмістом — `devDependencies` у `npm/package.json`, rego-юніт-тести `*_test.rego` всередині `policy/` (яка входить до `files`), відсутня явна вимога `"files"` whitelist. Нові пер-документні перевірки були реалізовані у JS замість Rego, порушуючи проєктну конвенцію «default — Rego для структурних перевірок одного JSON/YAML».

**Рішення/Процедура/Факт:**
- `npm-module.mdc` (версія `1.10 → 1.11`): додано секцію «Компактний пакет» із трьома правилами — `"files"` обовʼязковий whitelist, тести/fixtures за межами опублікованих шляхів (виняток: `*_test.rego` розташовується поруч з policy за conftest-конвенцією і тому не є порушенням), лише runtime-залежності у `npm/package.json`.
- `npm/policy/npm_module/npm_package_json/npm_package_json.rego`: додано два `deny` — обовʼязковий непорожній `files` (3 варіанти порушення) і заборона `devDependencies`; новий `npm_package_json_test.rego` з 7 негативними кейсами (141/141 тестів).
- `npm/scripts/check-npm-module.mjs`: `checkPackageCompactness` видалено (перенесено у rego); лишився лише `checkNoTestsInPublishedFiles` (FS-walk + oxc-parser AST — rego не вміє). Нові допоміжні функції `globToRegex`, `findTestFrameworkImport`, `classifyPublishedFileAsTest` (з export для unit-тестів); `TEST_FRAMEWORK_MODULES` перелік з 8 модулів.
- `npm/tests/check-npm-module.test.mjs`: 14 нових тестів для glob-matcher, AST-імпортів, класифікації test-style файлів.
- `npm/package.json`: прибрано `devDependencies` (`@nitra/cursor` є у корені як `workspace:*`), прибрано тимчасовий негативний glob `!**/*_test.rego`; версія `1.9.4 → 1.9.5`.
- `eslint.config.js`: додано `.claude/worktrees/**` до `ignores`.
- `conftest.mdc` + `npm/.claude-template/npm-CLAUDE.md` / `npm/CLAUDE.md`: додано STOP-блок «перед редагуванням `check-*.mjs` — перевір, чи не rego-задача»; у списку «JS-only» явно вказано AST-парсинг JS/TS і walk дерева.

**Обґрунтування:** Максимально компактний пакет — менше `npm install` часу і розміру для кінцевих користувачів. Пер-документні структурні перевірки (`files`, `devDependencies`) природно виражаються у Rego (`deny contains msg if`) без FS-доступу; JS-чек залишено лише там, де він незамінний — AST-аналіз файлів і FS-walk. Rego-тести `*_test.rego` залишено поруч з policy (conftest-конвенція: тест у тому ж `package`), а не виносити у `npm/tests/`.

**Розглянуті альтернативи:**
- Негативний glob `!**/*_test.rego` у `files` (виключити rego-тести з tarball не переносячи їх) — реалізовано тимчасово, але відхилено на користь семантичного винятку: `*_test.rego` не є порушенням правила за проєктною конвенцією.
- Фізичне перенесення `*_test.rego` у `npm/tests/policy/` — відхилено як надмірний рефакторинг (потрібно переписати `lint-rego.mjs` / `lint-conftest.mjs`).
- Залишити `checkPackageCompactness` у JS — відхилено, ляп; структурна перевірка одного JSON — це rego-задача.
