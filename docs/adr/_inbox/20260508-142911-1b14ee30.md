## Runbook `n-taze` — Процедура оновлення версій модулів з аналізом major-змін

**Контекст:** Файл `npm/skills/taze/SKILL.md` існував як заглушка без повноцінного workflow. Потрібна була чітка покрокова процедура: від запуску оновлення до рефакторингу несумісного коду і звіту.

**Рішення/Процедура/Факт:** Скіл переписано з повним 8-кроковим workflow:
1. Бекап `package.json` / `bun.lock` у `*.taze-bak`-файли перед оновленням.
2. `bunx taze -w -r latest` + `bun install`.
3. Порівняння бекапів з оновленими файлами для виявлення major-стрибків (зміна першої значущої цифри semver).
4. Збір breaking changes: спочатку CHANGELOG/Releases репозиторію модуля, потім `diff -r` між `~/.bun/install/cache/<name>@<old>/` і `node_modules/<name>/`, або встановлення старої версії у `/tmp/taze-old`.
5. Пошук використань зміненого API в коді проекту через `rg`.
6. Автоматичний рефакторинг сумісних міграцій; TODO-коментар з посиланням на CHANGELOG для неоднозначних.
7. Прибирання `*.taze-bak`-файлів.
8. Структурований звіт: minor/patch-кількість, major-список, авторефакторинг, ручні дії, стан lint/typecheck/test.
Додатково: версію підвищено `1.8.211 → 1.8.212` і додано запис у `npm/CHANGELOG.md`.

**Обґрунтування:** Користувач хотів детерміновану процедуру, яка не просто оновлює версії, а гарантує аналіз сумісності — щоб уникнути прихованих регресій від major-стрибків. Бекап перед оновленням дає точне diff без зовнішніх залежностей. Пріоритет CHANGELOG над git-diff мінімізує шум.

**Розглянуті альтернативи:** не обговорювалися (підхід цілком продиктований користувачем у завданні).

**Зачіпає:** `npm/skills/taze/SKILL.md`, `npm/package.json` (version bump), `npm/CHANGELOG.md`.
