## ADR Rego-first для пер-документних JSON/YAML-перевірок — міграція `checkPackageCompactness`
**Контекст:** Функція `checkPackageCompactness` у `npm/scripts/check-npm-module.mjs` перевіряла наявність whitelist `files` та відсутність `devDependencies` у `npm/package.json` засобами JavaScript — всупереч проєктній конвенції з `conftest.mdc`, де default для пер-документних структурних перевірок — Rego.
**Рішення/Процедура/Факт:** `checkPackageCompactness` видалено з JS. Два deny (`files` обовʼязковий/непорожній, заборона `devDependencies`) додано у `npm/policy/npm_module/npm_package_json/npm_package_json.rego`. Написано `npm_package_json_test.rego` (happy path + 7 негативних кейсів через `json.patch`). `conftest.mdc` доповнено STOP-блоком перед `Edit` будь-якого `check-*.mjs` — вимагає спершу перевірити, чи задача не є пер-документною. Секцію `npm/CLAUDE.md` та `.claude-template/npm-CLAUDE.md` переписано у 3-пунктовий self-check із червоним прапором.
**Обґрунтування:** Пер-документні структурні deny (наявність ключа верхнього рівня, форма поля) — типовий Rego-use-case: декларативно, тестується `conftest verify`, інтегрується у `lint-rego`. JS виправданий лише тоді, коли потрібен FS-walk або AST-парсинг — `checkNoTestsInPublishedFiles` залишився у JS саме тому.
**Розглянуті альтернативи:** Лишити у JS (відхилено — порушує conftest.mdc конвенцію); гібридний підхід — deny у rego, виклик rego з JS (надмірно).
**Зачіпає:** `npm/policy/npm_module/npm_package_json/npm_package_json.rego`, новий `npm_package_json_test.rego`, `npm/scripts/check-npm-module.mjs`, `.cursor/rules/conftest.mdc`, `npm/CLAUDE.md`, `npm/.claude-template/npm-CLAUDE.md`

## ADR `*_test.rego` дозволено в опублікованому дереві — виняток conftest-конвенції
**Контекст:** Правило `npm-module.mdc` заборонило тести всередині шляхів з `files`. `*_test.rego` лежать поряд із полісі у `policy/` — так вимагає OPA/conftest (пакет і unit-тест — один каталог). Перше рішення (негативний glob `"!**/*_test.rego"` у `files`) виключало файли з tarball, але не вирішувало структурну суперечність.
**Рішення/Процедура/Факт:** `*_test.rego` прибрано з `TEST_FILE_PATTERNS` у `check-npm-module.mjs`. Негативний glob `"!**/*_test.rego"` прибрано з `npm/package.json`. Rego-юніт-тести знову потрапляють у tarball (175 файлів). `npm-module.mdc` та дзеркало `n-npm-module.mdc` доповнено: «Виняток — Rego (`*_test.rego`): conftest-конвенція вимагає unit-тест у тому ж каталозі, що й полісі».
**Обґрунтування:** Conftest шукає `*_test.rego` за патерном у тому ж `package`. Фізичне перенесення тестів у `npm/tests/policy/` вимагало б зміни `lint-rego.mjs` і `lint-conftest.mjs` — суттєвий рефактор без функціональної вигоди. Споживачі пакету отримують полісі разом із unit-тестами, що є нормою для OPA-екосистеми.
**Розглянуті альтернативи:** Перенести `*_test.rego` до `npm/tests/policy/` + адаптувати lint-скрипти (відхилено через непропорційний diff); негативний glob у `files` (відхилено — лише симптоматично).
**Зачіпає:** `npm/scripts/check-npm-module.mjs`, `npm/tests/check-npm-module.test.mjs`, `npm/package.json`, `npm/mdc/npm-module.mdc`, `.cursor/rules/n-npm-module.mdc`

## ADR Канонічний `knip-canonical.json` — existence-only перевірка з auto-create
**Контекст:** `check-js-lint.mjs` перевіряв вміст `knip.json` (чи є `graphql` в `ignoreDependencies`). Це не покривало 9 категорій false-positives, які knip генерує у типовому проєкті. Паттерн `oxlint-canonical.json` (subset-перевірка) вже існував, але для knip потрібна інша семантика.
**Рішення/Процедура/Факт:** Створено `npm/scripts/utils/knip-canonical.json` з повним baseline: `entry` (CLI-конфіги — eslint/stylelint/oxlint/jscpd/markdownlint-cli2/commitlint), `project` (`**/*.{js,mjs,...}`), `ignore` (`__fixtures__`), `ignoreDependencies` (`@nitra/cspell-dict`, `@cspell/dict-.*`, `graphql`), `ignoreBinaries` (14 CLI, що викликаються через `bunx`/`npx`). `checkKnipConfig` у `check-js-lint.mjs` тепер перевіряє **лише наявність** `knip.json`; якщо файл відсутній — копіює канон (side effect). Стару перевірку вмісту прибрано. Правило `js-lint.mdc` версія `1.20`, CHANGELOG `1.10.0`.
**Обґрунтування:** Вміст `knip.json` — проєктно-специфічний (кожен проєкт має власні `ignoreDependencies`), тому subset-перевірка дала б занадто багато false-positives. Auto-create дає нову точку відліку без зобовʼязань підтримувати синхронізацію. Паттерн аналогічний до `capture-decisions.sh` у `check-adr.mjs`.
**Розглянуті альтернативи:** Subset-перевірка (відхилено — зайві failures); повна перевірка вмісту (відхилено — надмірно обмежує проєкти-споживачі); немає авто-копії (обговорювалося, відхилено — без starter-файлу нова точка відліку не дає цінності).
**Зачіпає:** `npm/scripts/utils/knip-canonical.json` (новий), `npm/scripts/check-js-lint.mjs`, `npm/mdc/js-lint.mdc`, `.cursor/rules/n-js-lint.mdc`, `npm/package.json` (→ 1.10.0), `npm/CHANGELOG.md`
