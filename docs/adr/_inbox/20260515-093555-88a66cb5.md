## Knowledge Ланцюжок виконання правил `@nitra/cursor check` — від CLI до Rego

**Контекст:** Потрібно було зрозуміти повний шлях виконання одного правила (на прикладі `abie`) — від запуску `/n-fix` у Claude Code до виклику `conftest` з Rego-полісями.

**Рішення/Процедура/Факт:**
Ланцюжок:
1. `/n-fix` (skill) → `npx @nitra/cursor check` → `npm/bin/n-cursor.js → runChecks()`
2. `discoverCheckScripts()` сканує `npm/rules/*/js/check.mjs` — правило потрапляє в список **лише** якщо цей файл існує.
3. `discoverCheckRulesFromAgentsMd()` — фільтрує список через `AGENTS.md` проєкту (не через `.n-cursor.json`). Якщо правило не згадане в `AGENTS.md`, воно **не запускається** навіть якщо є у `.n-cursor.json`.
4. Для кожного id: `import('npm/rules/<id>/js/check.mjs').check()` — послідовно.
5. Усередині `check.mjs`: JS-обгортка → `runConftestBatch()` → `spawnSync('conftest', ['test', ...files, '-p', policyAbs, '--namespace', ns, '--output', 'json'])` → parse JSON → `fail(msg)`.
6. Rego-deny-правила у `npm/rules/<id>/policy/<name>/<name>.rego` — пакет `<id>.<name>`, gate за `input.kind`.

**Обґрунтування:** Така архітектура дозволяє: JS вирішує cross-file логіку (readdir, glob, gating), а Rego перевіряє per-document структуру. `js/check.mjs` — єдина обов'язкова точка входу; без неї CLI правило взагалі не бачить.

**Розглянуті альтернативи:** Pure-Rego без JS-обгортки — неможливо в поточній архітектурі CLI: `discoverCheckScripts()` вимагає наявності `js/check.mjs`. При відсутності файлу правило просто не потрапляє в `available` і не запускається.

**Зачіпає:** `npm/bin/n-cursor.js` (`discoverCheckScripts`, `runChecks`, `discoverCheckRulesFromAgentsMd`), `npm/scripts/utils/run-conftest-batch.mjs`, `npm/rules/*/js/check.mjs`, `npm/rules/*/policy/**/*.rego`

## ADR Мінімальна структура нового pure-Rego правила

**Контекст:** При додаванні нового правила, де вся валідація — пер-документна (JSON/YAML поля, regex, літерали) і не потребує cross-file JS-логіки, постало питання скільки JS взагалі треба писати.

**Рішення/Процедура/Факт:**
Навіть для pure-Rego правила обов'язкова структура:
```
npm/rules/<id>/
├── <id>.mdc
├── auto.md            (опційно)
├── js/
│   ├── check.mjs      ← ОБОВ'ЯЗКОВИЙ (тонкий orchestrator)
│   └── check.test.mjs
└── policy/
└── <name>/
├── <name>.rego
└── <name>_test.rego
```
`check.mjs` — мінімальний: gating (чи правило застосовне), `existsSync` для кожного target-файлу, один виклик `runConftestBatch` на кожен `policy/<name>/`. Батч файлів (наприклад всі `HealthCheckPolicy`-YAML) — один `runConftestBatch` з масивом `files`. Rego-файл: `package <id>.<name>`, `import rego.v1`, `deny contains msg if { ... }` з gate за `input.kind`.

**Обґрунтування:** `discoverCheckScripts()` в CLI обов'язково шукає `js/check.mjs` — без нього правило невидиме для `npx @nitra/cursor check`. JS лишається мінімальним orchestrator-ом (gating + file-existence + виклик `runConftestBatch`), вся документна логіка — у Rego.

**Розглянуті альтернативи:** Не розглядалися (JS-обгортка — архітектурна вимога CLI).

**Зачіпає:** `npm/bin/n-cursor.js` (`discoverCheckScripts`), `npm/scripts/utils/run-conftest-batch.mjs`, конвенція `package <id>.<name>` у Rego-файлах, `bun run lint-rego` (opa/regal/conftest verify по всіх `npm/rules/*/policy/`)
