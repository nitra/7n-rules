## ADR Розширення стандарту ci4 принципами AI-native документації

**Контекст:** Правило `npm/mdc/ci4.mdc` описувало C4-модель і Docs-as-Code, але не містило явного визначення AI-native принципів — тих, що формують «чому» за цим підходом. Компанія має внутрішній стандарт документації для AI-native розробки, який потрібно було зафіксувати в проєкті.

**Рішення/Процедура/Факт:** До обох файлів — `npm/mdc/ci4.mdc` та `.cursor/rules/n-ci4.mdc` — після секції «Markdown як джерело істини» додано шість нових секцій: **Специфікація як джерело істини (Spec-as-Source)**, **Тест на повне відтворення (Rebuild Test)**, **Ефективність токенів і чистий Markdown**, **Контекстна незалежність розділів**, **Docs-as-Code** та **Трасування як документація недетермінованої поведінки**. Обидва файли синхронізовано так, щоб `.cursor/rules/n-ci4.mdc` (alwaysApply-правило для агентів) відображав той самий зміст, що й канонічний `npm/mdc/ci4.mdc`.

**Обґрунтування:** Принципи (Spec-as-Source, Rebuild Test, Token Efficiency тощо) пояснюють *чому* C4 у Markdown є джерелом істини — без них правило виглядало як набір інструкцій без мотивації. Явне закріплення цих принципів дозволяє LLM-агенту застосовувати правило правильно навіть у нетипових ситуаціях, а не лише слідувати шаблону.

**Розглянуті альтернативи:** Не обговорювалися; підхід задано користувачем через наданий текст стандарту.

**Зачіпає:** `npm/mdc/ci4.mdc`, `.cursor/rules/n-ci4.mdc`
