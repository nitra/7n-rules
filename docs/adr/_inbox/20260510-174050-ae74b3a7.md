## ADR Стратегія «Rego-first» для пер-документних перевірок у `@nitra/cursor`

**Контекст:** Пакет `@nitra/cursor` накопичив великий `npm/scripts/check-k8s.mjs` (~6500 рядків) із ~50 експортними `*Violation`-функціями. Більшість із них — пер-документні структурні перевірки, які OPA/Rego може виконати без JavaScript. Виникла потреба визначити, що переноситься в rego-полісі (`npm/policy/`), що лишається в JS, і де межа між ними — щоб агенти й автори нових правил знали default за замовчуванням.

**Рішення/Процедура/Факт:**

*Структурне рішення (11 нових rego-пакетів):*
- Розширено `npm/policy/k8s/manifest/manifest.rego` (Deployment `resources.requests.cpu+memory`, Hasura image pin, `topologySpreadConstraints` canonical)
- Нові пакети: `k8s.gateway`, `k8s.kustomization`, `k8s.svc_yaml`, `k8s.svc_hl_yaml`, `k8s.base_manifest`, `k8s.base_kustomization`, `k8s.kustomize_managed`, `k8s.hasura_configmap`, `k8s.hasura_httproute`, `k8s.hpa_pdb`
- До кожного пакету — `*_test.rego`; 93 тести проходять через `conftest verify`
- `npm/scripts/lint-rego.mjs` розширено опційним `conftest verify`; `lint-conftest.mjs` — targets для нових пакетів

*Правило (закодовано у `.cursor/rules/conftest.mdc` + `npm/.claude-template/npm-CLAUDE.md`):*
- **Default — Rego**: будь-яка перевірка, що дивиться на **один** YAML-документ (kind, apiVersion, поля, форма масивів/мап), пишеться як rego-пакет у `npm/policy/<rule>/<name>/` + `_test.rego`
- **JS тільки коли**: cross-file resolution (kustomize-tree, парність `svc.yaml`/`svc-hl.yaml`, ConfigMap↔Deployment lookup), file-system access, autofix/rewrite (rego read-only), парсинг сирого тексту до YAML-body
- **Гібрид**: структурний gate → rego; cross-file оркестрація → JS (поточний `check-k8s.mjs` — референс)

**Обґрунтування:** Rego-полісі статично аналізуються (`opa check --strict`, `regal lint`), виконуються ізольовано, пишуться декларативно і можуть запускатись через IDE-розширення `tsandall.opa` ще до CI. JS лишається необхідним тільки там, де OPA не має доступу до FS, не може читати сусідні файли й не робить side-effects. Розмежування зменшує розмір JS-файлів і робить перевірки тестованими в isolation.

**Розглянуті альтернативи:**
- Лишити все в JS — не дає static-analysis і IDE-coverage rego-полісі
- Перенести cross-file логіку в rego через `data`-bundle (сформувати JSON із усіх файлів і передати conftest) — можливо, але різко ускладнює оркестрацію і breaking для існуючого `check-k8s.mjs`; відкладено на майбутнє

**Зачіпає:**
- `.cursor/rules/conftest.mdc` — нове decision-tree правило «rego-first»
- `npm/.claude-template/npm-CLAUDE.md` → `npm/CLAUDE.md` — розділ «Перш ніж писати `check-*.mjs`»
- `npm/policy/k8s/*` — 11 нових пакетів (22 файли `*.rego`)
- `npm/scripts/lint-rego.mjs`, `npm/scripts/lint-conftest.mjs` — нові targets
- `.cursor/rules/n-k8s.mdc` — розділ «Швидкий gate через conftest (Rego)»
- `npm/package.json` v1.8.222, `npm/CHANGELOG.md`
