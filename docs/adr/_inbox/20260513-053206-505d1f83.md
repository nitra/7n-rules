## ADR Rego-first для пер-документних перевірок `npm/package.json` — міграція `checkPackageCompactness`

**Контекст:** У `check-npm-module.mjs` була функція `checkPackageCompactness`, яка перевіряла наявність непорожнього `files`-whitelist і відсутність `devDependencies` у `npm/package.json`. Ці перевірки є пер-документними структурними deny на одному JSON-файлі — класичний Rego-кейс, але реалізований у JS (ляп попередньої сесії).

**Рішення/Процедура/Факт:** `checkPackageCompactness` видалено з `npm/scripts/check-npm-module.mjs`. Два нові deny (`files` обов'язковий і непорожній; `devDependencies` заборонені) додано до `npm/policy/npm_module/npm_package_json/npm_package_json.rego`. Створено `npm/policy/npm_module/npm_package_json/npm_package_json_test.rego` — happy path + 7 негативних кейсів через `json.patch`. В `conftest.mdc` додано STOP-блок перед будь-яким `Edit` `check-*.mjs` — з явним нагадуванням перевірити, чи логіка не є пер-документною. Аналогічне нагадування у 3-пунктовому self-check додано до `npm/.claude-template/npm-CLAUDE.md`.

**Обґрунтування:** Rego — декларативна, тестована, ізольована мова для пер-документних заборон. JS потрібен лише для cross-file / FS-walk / AST-парсингу. Ведення перевірки в JS дублює логіку без переваг і порушує принцип «одне джерело істини».

**Розглянуті альтернативи:** Залишити в JS (відхилено — не використовує переваги Rego); гібрид JS+Rego (відхилено — дублювання).

**Зачіпає:** `npm/scripts/check-npm-module.mjs`, `npm/policy/npm_module/npm_package_json/npm_package_json.rego`, `npm/policy/npm_module/npm_package_json/npm_package_json_test.rego`, `npm/mdc/npm-module.mdc`, `.cursor/rules/n-npm-module.mdc`, `npm/.claude-template/npm-CLAUDE.md`, `npm/CLAUDE.md`, `.cursor/rules/conftest.mdc`

## ADR Канонічний `knip-canonical.json` — перевірка лише наявності файлу, auto-create при відсутності

**Контекст:** `knip` регулярно генерував «Configuration hints» для `graphql`, `vue`, `@nitra/cursor`, `vite` у `knip.json` проєкту. Не було канонічного baseline для `knip.json`, аналогічного до `oxlint-canonical.json`. Стара перевірка `checkKnipConfig` в `check-js-lint.mjs` перевіряла лише `ignoreDependencies ∋ "graphql"` — вузько і не масштабується.

**Рішення/Процедура/Факт:** Створено `npm/scripts/utils/knip-canonical.json` з покриттям 9 категорій false-positives: `entry` (CLI-конфіги eslint/stylelint/oxlint/jscpd/markdownlint-cli2/`commitlint`), `project`, `ignore`, `ignoreDependencies` (`@nitra/cspell-dict`, `@cspell/dict-*`, `graphql`), `ignoreBinaries` (усі бінарники, що викликаються через `bunx`/`npx` без присутності в `devDependencies`). Семантика чека — **лише наявність файлу**: якщо `knip.json` відсутній — `checkKnipConfig` копіює `knip-canonical.json` у корінь (side effect, описано в правилі). Зміст наявного `knip.json` не перевіряється.

**Обґрунтування:** Перевірка subset-відповідності між `knip.json` і canonical зробила б правило жорстким і вимагала б синхронізації при кожному оновленні canonical. «Лише наявність + auto-create» дає zero-friction onboarding для нових проєктів і дозволяє довільні розширення у споживача.

**Розглянуті альтернативи:** Subset-перевірка полів (відхилено — жорстко, складно супроводжувати); повна перевірка вмісту (відхилено — ще жорсткіше); нічого не перевіряти (відхилено — тоді канон не застосовується автоматично).

**Зачіпає:** `npm/scripts/utils/knip-canonical.json` (новий файл), `npm/scripts/check-js-lint.mjs` (новий export `KNIP_CANONICAL_JSON_PATH`, оновлено `checkKnipConfig`), `npm/mdc/js-lint.mdc` v1.20, `.cursor/rules/n-js-lint.mdc` v1.20, `npm/CHANGELOG.md`, `npm/package.json` (1.10.0)

## ADR `bunx knip --no-config-hints` як канонічний рядок `lint-js`

**Контекст:** `bunx knip` без прапорів виводив «Configuration hints» (підказки про зайві записи в `ignoreDependencies`/`ignoreBinaries`), що шуміло у виводі і не завершувалося з ненульовим кодом — hints не є помилкою, але відволікають. Записи типу `graphql` у `ignoreDependencies` є навмисними (peer-залежність без прямого import), тому видаляти їх не можна.

**Рішення/Процедура/Факт:** Канонічний рядок скрипту `lint-js` змінено на `bunx oxlint --fix && bunx eslint --fix . && bunx jscpd . && bunx knip --no-config-hints`. Оновлено: `CANONICAL_LINT_JS` у `check-js-lint.mjs`, `canonical_lint_js` у `npm/policy/js_lint/package_json/package_json.rego`, `scripts.lint-js` у кореневому `package.json`, приклади у `js-lint.mdc` (v1.21) і `.cursor/rules/n-js-lint.mdc`, крок `run:` у `.github/workflows/lint-js.yml`.

**Обґрунтування:** `--no-config-hints` — офіційний прапор knip для придушення підказок (не помилок). Навмисні `ignoreDependencies` (`graphql` як peer, `@nitra/cspell-dict` через `.cspell.json`) є коректними з точки зору архітектури, але knip не знає про не-JS-конфіги як споживачів залежностей.

**Розглянуті альтернативи:** Видалити `graphql` з `ignoreDependencies` (відхилено — peer-залежність реально потрібна); додати `--reporter json` і фільтрувати (надмірно складно); залишити hints (відхилено — шум у CI-виводі).

**Зачіпає:** `package.json` (кореневий `scripts.lint-js`), `npm/scripts/check-js-lint.mjs`, `npm/policy/js_lint/package_json/package_json.rego`, `npm/mdc/js-lint.mdc`, `.cursor/rules/n-js-lint.mdc`, `.github/workflows/lint-js.yml`, `npm/CHANGELOG.md`, `npm/package.json` (1.10.1)
