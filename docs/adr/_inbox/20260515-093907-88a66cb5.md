## Що треба змінити і в чому справжня складність

Зміна `discoverCheckScripts` — тривіальна. Поточний фільтр:

```js
// n-cursor.js:998
.filter(e => existsSync(join(BUNDLED_RULES_DIR, e.name, 'js', 'check.mjs')))
```

Замінюється на «хоча б один із двох»:

```js
.filter(e => {
const hasJs = existsSync(join(BUNDLED_RULES_DIR, e.name, 'js', 'check.mjs'))
const hasPolicy = existsSync(join(BUNDLED_RULES_DIR, e.name, 'policy'))
return hasJs || hasPolicy
})
```

Але це дає лише список id. Далі `runChecks` [n-cursor.js:1098](npm/bin/n-cursor.js#L1098) завжди робить `import(scriptPath)`. Для policy-only правил — немає що імпортувати. Тобто потрібна **синтетична обгортка**.

### Справжня проблема: звідки знати, які файли передати conftest?

`runConftestBatch` чекає:
```js
{ policyDirRel, namespace, files: ['/abs/path/to/file'] }
```

`files` — це конкретні файли у **цільовому проєкті**. Щоб знати їх, JS-скрипти зараз роблять `walkDir` або `existsSync` вручну. Без JS це нікуди не зникає — треба декларувати це десь ще.

Варіантів два.

---

### Варіант A — `meta.json` у кожному `policy/<name>/`

```
rules/rego/policy/package_json/
├── package_json.rego
├── package_json_test.rego
└── meta.json          ← { "file": "package.json" }
```

CLI читає `meta.json`, якщо немає JS — будує синтетичну `check()`:

```js
async function runPolicyOnlyRule(id) {
const policyDir = join(BUNDLED_RULES_DIR, id, 'policy')
const subs = await readdir(policyDir, { withFileTypes: true })
for (const sub of subs.filter(e => e.isDirectory())) {
const meta = JSON.parse(readFileSync(join(sub.path, sub.name, 'meta.json'), 'utf8'))
const target = join(cwd(), meta.file)
if (!existsSync(target)) { /* fail or skip */ continue }
const violations = runConftestBatch({
policyDirRel: `${id}/${sub.name}`,
namespace: `${id}.${sub.name}`,
files: [target]
})
for (const v of violations) fail(v.message)
}
}
```

**Плюси:** zero JS у правилі. **Мінус:** нова конвенція `meta.json` скрізь, не підтримує glob (наприклад, «всі `*.yaml` у `k8s/`»), не підтримує conditional-gating («лише якщо є `.rego` у дереві»).

### Варіант B — окремий `check.json` на рівні правила

```json
// rules/rego/check.json
{
"gate": { "hasFilesMatching": "**/*.rego" },
"policies": {
"package_json":    { "file": "package.json" },
"vscode_extensions": { "file": ".vscode/extensions.json" },
"vscode_settings": { "file": ".vscode/settings.json" }
}
}
```

Все одне місце, підтримує gate-умову. Але це фактично DSL — доведеться реалізувати `hasFilesMatching`, і за 2–3 правила він стане повноцінним мінімовим рушієм. Ціна — складніший `n-cursor.js`.

---

### Що не покривається обома варіантами

| Сценарій | Потребує JS? |
|---|---|
| single file (`package.json`) | Ні |
| single file, optional (skip if missing) | Ні (прапор у meta) |
| glob (`k8s/**/*.yaml`) | Варіант B з `glob` |
| «лише якщо є `.rego` у дереві» | gate-умова |
| парність файлів (`Deployment` ↔ `hc.yaml`) | Так, JS обовʼязковий |
| kustomize-резолюція, AST | Так, JS обовʼязковий |

---

### Реальний стан у поточних правилах

З тих правил, що мають тільки `policy/` (без складного JS), `rego` — єдиний кандидат. Усі інші або мають walk-логіку, або парність, або conditional gating.

Тому **прагматична відповідь**: Варіант A (`meta.json`) покриває саме `rules/rego/` і потенційно нові прості правила, не вимагає складного DSL, і вплив на `n-cursor.js` мінімальний (~30 рядків нового `runPolicyOnlyRule`). Варіант B варто розглядати лише якщо зʼявляться правила з glob або нетривіальним gating-ом.
