## Knowledge Дублювання import-рядків вже заборонено через oxlint-canonical

**Контекст:** Виникло питання, чи потрібно додавати до `check-js.mjs` / `check-vue.mjs` перевірку того, що імпорти з одного модуля об'єднані в один рядок (наприклад, `import { mssqlClient, sql } from '#conn/mssql.js'` замість двох окремих `import`).

**Рішення/Процедура/Факт:** Правило вже присутнє в `/Users/vitalii/www/nitra/cursor/npm/scripts/utils/oxlint-canonical.json` у двох варіантах:

- `"import/no-duplicates": "deny"` (рядок 250)
- `"no-duplicate-imports": "deny"` (рядок 291)

Обидва правила обов'язкові для всіх проєктів, де `check-js-lint.mjs` перевіряє `.oxlintrc.json` на відповідність канону. Тому додавати окрему перевірку до `check-js.mjs` або `check-vue.mjs` не потрібно.

**Обґрунтування:** Oxlint перевіряє файли проєкту під час `lint-js`, і `check-js-lint.mjs` вже верифікує, що `.oxlintrc.json` містить весь канонічний набір правил. Дублювати цю логіку в окремих check-скриптах зайве.

**Розглянуті альтернативи:** Додати явну перевірку AST-рівня до `check-js.mjs` / `check-vue.mjs` — відхилено, бо oxlint-canonical вже покриває цей кейс.

**Зачіпає:** `npm/scripts/utils/oxlint-canonical.json`, `npm/scripts/check-js-lint.mjs`, `npm/scripts/check-vue.mjs`
