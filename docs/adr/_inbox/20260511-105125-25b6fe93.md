## ADR Опційний `$schema`-modeline та `file:` як заглушка заборонені

**Контекст:** Механізм автовиправлення `n-fix` додавав рядок `# yaml-language-server: $schema=file:.` до k8s-маніфестів, для яких не існує публічної схеми. Така «заглушка» не валідує нічого, але создає хибне відчуття перевіреного файлу.

**Рішення/Процедура/Факт:**
- `check-k8s.mjs` — логіка `checkK8sYamlFile`: відсутній modeline більше не фейлить, а дає `pass` із позначкою про пропуск; `$schema=file:…` тепер явна помилка.
- `k8s.mdc` (версія `1.28`) — правило уточнено: modeline обов'язковий лише коли є надійна публічна схема (yannh/datree/kustomization); без схеми — файл лишається без modeline.
- Оновлено JSDoc у `check-k8s.mjs` та `npm/CHANGELOG.md`; версія `1.9.1` → `1.9.2`.

**Обґрунтування:** `$schema=file:.` — недійсне посилання, яке не викликає жодної валідації. Правило «валідуємо тільки коли є схема» чесніше: файл або перевіряється через реальний URL, або зовсім не має modeline. `lint-k8s` (kubeconform `-ignore-missing-schemas`) залишається незалежним шаром валідації.

**Розглянуті альтернативи:** Не обговорювались; підхід «modeline тільки через URL» задано у ТЗ.

**Зачіпає:** `npm/scripts/check-k8s.mjs`, `npm/mdc/k8s.mdc`, `npm/CHANGELOG.md`, `npm/package.json`

## ADR Виправлення `pathHasK8sSegment`: relativize за коренем репо, `.github/` поза скоупом

**Контекст:** `pathHasK8sSegment` перевіряла наявність сегмента `k8s` в **абсолютному** шляху. Якщо кореневий каталог репозиторію сам містить компонент `k8s` (наприклад `/Users/…/abie/k8s/`), функція повертала `true` для будь-якого файлу у проєкті — зокрема для `.github/workflows/*.yml`. Чекер помилково вимагав перейменувати `.yml` → `.yaml` для файлів GitHub Actions, хоча `ga.mdc` наказує зворотне.

**Рішення/Процедура/Факт:**
- `pathHasK8sSegment(filePath, root?)`: додано опціональний параметр `root`; коли передано — шлях спочатку relativize-ується через `relative(root, filePath)`, і сегменти рахуються вже у відносному шляху.
- `findK8sYamlFiles` у `check-k8s.mjs`, `run-k8s.mjs`, `check-abie.mjs` — передають `root`; додано ранній `return` для `.github/` (defense-in-depth).
- `k8s.mdc` (версія `1.29`) — новий параграф «Скоп — поза `.github/`»: пояснює, що визначення k8s-скопу відносне до кореня репо, а `.github/workflows/` і `.github/actions/` належать `ga.mdc`.
- Нові тести у `check-k8s-schema.test.mjs` (worst-case `/tmp/some/k8s/…`) та `run-k8s-roots.test.mjs` (репо з кореневим іменем `k8s/`, `.github/workflows/apply-k8s.yml` не потрапляє у `findK8sRoots`).
- Версія `1.9.2` → `1.9.3`, запис у `CHANGELOG.md`.

**Обґрунтування:** Пошук сегмента за абсолютним шляхом — помилка дизайну: будь-який батьківський каталог з іменем `k8s` «заражає» всі файли проєкту. Відносний шлях від кореня репо є єдиним правильним скопом для правила, що діє «в межах k8s-піддерева проєкту».

**Розглянуті альтернативи:** Заглушити проблему раннім `return` лише у `checkK8sYamlFile` — відхилено, бо `findK8sYamlFiles` залишила б файли у списку і вони могли б дістатися `runAllK8sRego` та `validateKustomization*`.

**Зачіпає:** `npm/scripts/check-k8s.mjs`, `npm/scripts/run-k8s.mjs`, `npm/scripts/check-abie.mjs`, `npm/mdc/k8s.mdc`, `npm/tests/check-k8s-schema.test.mjs`, `npm/tests/run-k8s-roots.test.mjs`, `npm/package.json`, `npm/CHANGELOG.md`
