## ADR Зміна дефолту `resources.requests.cpu` у dev/base до `0.05`
**Контекст:** У dev-кластері pod-и займали `0.5` CPU request за дефолтом, що заважало щільному bin-packing-у навіть коли реальне споживання — мілісекунди. При цьому `HorizontalPodAutoscaler` в dev тримає `maxReplicas: 1`, тож зменшення request на масштабування не впливає.
**Рішення/Процедура/Факт:** У `npm/mdc/k8s.mdc` дефолтне значення `resources.requests.cpu` для Deployment знижено з `'0.5'` до `'0.05'`. Додано примітку про HPA-шум (`ScalingLimited` у статусі) при такому низькому порозі, і підсекцію «CPU request у прод-overlay» з прикладом Kustomize patch для підняття до реального значення. `check-k8s.mjs` не змінювався — він перевіряє лише наявність непорожнього значення.
**Обґрунтування:** Мета — щільніший packing dev-кластера. При `maxReplicas=1` HPA фізично не масштабує pod-и, тому низький request безпечний: `ScalingLimited` у dev — лише косметичний статус, не реальна проблема. Прод-значення — відповідальність команди через overlay; правило рекомендує, але не enforcement-ить.
**Розглянуті альтернативи:** env-залежна перевірка в `check-k8s.mjs` (аналогічно до HPA/PDB min/max) — відхилено, scope тільки `.mdc`; проміжний дефолт `0.1`/`100m` — відхилено на користь явного `0.05` для однозначності.
**Зачіпає:** `npm/mdc/k8s.mdc` (рядки 99–128); всі проєкти, які читають це правило і мають Deployment з дефолтним CPU request.
