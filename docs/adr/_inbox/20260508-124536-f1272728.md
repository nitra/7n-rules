## ADR Rego checks як fail-fast gate перед JS у lint-пайплайні

**Контекст:** Проєкт має два шари лінту: структурні перевірки через `conftest`/`opa` (Rego, легкі) та глибокий аналіз через JS (`check-*.mjs` — AST-скан, обхід файлового дерева, cross-file Kustomize-резолюція). Rego-перевірки були портовані з JS-скриптів, але запускалися після JS у `bun run lint`.

**Рішення/Процедура/Факт:** Переставлено порядок у скрипті `lint` у кореневому `package.json`: `lint-rego` і `lint-conftest` тепер стоять **перед** `lint-js`, `lint-k8s`, `lint-docker`, `lint-style`, `lint-image`. Ланцюжок `&&` гарантує, що JS-скрипти запускаються лише якщо структурні Rego-правила пройшли. У `check-*.mjs` залишилася логіка, яку Rego не може замінити (FS-existence, cross-file, git, семантична semver-логіка з OR-діапазонами).

**Обґрунтування:** Rego/conftest — дешеві і детерміновані (JSON-схема over parsed YAML/JSON, без FS-обходу). JS — дорожчі (AST-парсинг через `oxc-parser`, `walkDir`, `readdir` по всьому дереву). Fail-fast на Rego скорочує час лінту у разі структурних помилок і дає точніші повідомлення до запуску повного JS-скану.

**Розглянуті альтернативи:** Залишити Rego разом із JS в одному порядку без fail-fast — відкинуто, бо не дає переваги швидкого скасування при типових структурних порушеннях (`packageManager`, відсутній `lint-image`, неправильний порт у `HealthCheckPolicy` тощо).

**Зачіпає:** `package.json` (кореневий, поле `scripts.lint`), `npm/scripts/lint-conftest.mjs`, усі `npm/policy/**/*.rego`.
