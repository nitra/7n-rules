Ключове розходження — **семантика `.mdc` у цьому проєкті вже зайнята**. Розкладу конкретно.

## Чим `.mdc` особливий у цій кодовій базі

[n-cursor.js:135](npm/bin/n-cursor.js#L135) `discoverBundledRuleNames()` шукає `rules/<id>/<id>.mdc` і **копіює ці файли** в `.cursor/rules/n-<id>.mdc` цільових проєктів. `.mdc` = «вміст для агентів у проєкті користувача». Це сильна конвенція.

`policy/<name>/target.mdc` **не потрапляє** під цю логіку (шаблон `<id>.mdc` не матчить), тож технічно не зламає синхронізацію. Але для читача коду файл `.mdc` серед `.rego` буде сюрпризом — очікуваний партнер `.rego` це `_test.rego`, не `.mdc`.

## Конкретне порівняння

| | `target.json` | `target.mdc` |
|---|---|---|
| Схема і валідація | JSON Schema → `bun run lint-text` ловить помилки | YAML frontmatter, потрібен окремий валідатор |
| Проза | немає (є у `.rego`-коментарях) | можна, але дублює коментарі `.rego` |
| Читаність агентами | мінімальна — JSON без контексту | вища — Cursor/Claude бачать prose |
| Відповідність конвенції | новий тип файлу в `policy/` | конфліктує з семантикою «`.mdc` → sync у проєкт» |
| Парсинг у CLI | `JSON.parse` | потрібен frontmatter-парсер (вже є для `SKILL.md`, але простий) |
| Confusion risk | низький | помірний — особливо для нових авторів правил |

## Де `.mdc` виграє насправді

Якби per-policy маніфест містив **prose**, яка корисна агентам і яку немає де ще написати. Наприклад, для `health_check_policy` це могло б бути:

```
---
target:
walkGlob: "**/hc.yaml"
---
Ця полісі застосовується до `HealthCheckPolicy`-документів.
Якщо в каталозі з `Deployment` немає `hc.yaml` — JS-перевірка вже `fail`-нула раніше,
тож rego бачить тільки валідні файли.
```

Але цей контекст **вже є** у коментарях `.rego`. `.mdc` просто повторить його.

## Висновок

`target.json` є кращим вибором: JSON Schema, нульова семантична плутанина з `.mdc`-синком, простий парсинг. `.mdc` виправданий тільки якщо вирішити, що per-policy маніфест — це публічний документ для агентів-споживачів, які читають правило. Якщо так — тоді назвати його не `target.mdc`, а `README.mdc`, де frontmatter = machine-targeting, body = prose. Але це ускладнення без очевидного виграшу, поки проєкт не досяг стадії, де агентам треба читати окремі полісі за межами `check`-прогону.

## ADR Декларативний таргетинг Rego-полісі (`rule.json` + `target.json`)

**Контекст:** CLI `npx @nitra/cursor check` вимагає `js/check.mjs` для кожного правила, навіть якщо логіка суто декларативна (list of Rego policies + static file targets). Для правил без cross-file / обчислюваної логіки (наприклад `rules/rego/`) `check.mjs` — пустий оркестратор. Також у `lint-conftest.mjs` є жорстко прописана таблиця `TARGETS`, яка фактично є тією ж декларацією таргетів, але живе централізовано, а не поруч із правилом.

**Рішення/Процедура/Факт:**

1. **`js/check*.mjs` — конвенція без окремої декларації.** CLI (`discoverCheckableRules`) автоматично запускає всі `.mjs`-файли у `rules/<id>/js/`, крім `*.test.mjs`. Shared-код кладеться в `utils/` поза `js/`. Порядок виконання — алфавітний. Спільний стан між файлами — кеш на рівні модуля (`Map`-singleton).

2. **`rules/<id>/rule.json` — rule-level applies-гейт.** Декларує умову застосовності правила: `requireRule: <id>`, `requireFileExists: [paths]`, `requireFilesByGlob: [globs]` — усі AND-комбінуються. Без `rule.json` правило вважається завжди активним.

3. **`policy/<name>/target.json` — per-policy таргет файлів.** Декларує `{ single, required, missingMessage }` або `{ walkGlob, required }`. CLI читає ці файли сам і викликає `runConftestBatch` без участі JS. Для обчислюваних батчів (де список файлів залежить від результатів попередньої перевірки) — лишається `js/check*.mjs`.

4. **Гібридні правила (JS + policy):** CLI спершу виконує декларативні targets з `policy/*/target.json`, потім виконує JS-файли. JS відповідає **лише за обчислювані і cross-file частини** і не повторює виклики `runConftestBatch` для статично задекларованих targets. Формат `target.json` (не `.mdc`) — через JSON Schema валідацію, відсутність плутанини з `.mdc`-семантикою (синк у `.cursor/rules/` проєктів), простий `JSON.parse`.

**Обґрунтування:**

- Сьогоднішня `lint-conftest.mjs:TARGETS`-таблиця і `runConftestBatch`-виклики у JS дублюють одну й ту саму інформацію (що годувати у яку полісі). Декларація переносить таргети всередину самого правила (cohesion). `lint-conftest.mjs` після міграції читає `rule.json` + `target.json` динамічно замість жорсткої таблиці.
- Конвенція «CLI бачить усі `check*.mjs`» усуває необхідність реєструвати JS-файли вручну; розбивка великих `check.mjs` (1153 рядки у `abie`) на `check-firebase.mjs`, `check-k8s-overlays.mjs` тощо стає природньою.
- Per-policy `target.json` не потребує власного prose (`.rego`-коментарі й `_test.rego` вже задокументовують полісі); JSON Schema ловить помилки на CI; semantic confusion з `.mdc`-синком у проєкти відсутня.

**Розглянуті альтернативи:**

- **Feed-everything (`conftest test .`):** CLI годує усе дерево у кожну полісі. Відхилено: полісі на конкретні безkind-файли (`.cspell.json`, `package.json`) не вміють self-gate без `kind`; перформанс — ~30 повних обходів репозиторію замість батчу.
- **Name-convention (каталог полісі диктує файл):** `policy/package_json/` → годуй `package.json`. Відхилено: двоє правил хочуть `package_json`-полісі з різними матчерами (single vs walk), конвенція ламається.
- **`.mdc` як per-policy маніфест (`target.mdc`):** Технічно не ламає синх (шаблон `<id>.mdc` не матчить `policy/*/target.mdc`), але конфліктує з семантикою `.mdc` в проєкті, немає JSON Schema, prose дублює коментарі `.rego`. Відхилено.
- **Централізований `rule.json` без `target.json`:** Усі таргети в одному файлі правила. Відхилено: `target.json` поруч із `.rego` — кращий locality (всі метадані полісі в одному місці).

**Зачіпає:**

- `npm/bin/n-cursor.js` — `discoverCheckScripts` → `discoverCheckableRules`, `runChecks` — додається `runDeclaredTargets`, `evaluateApplies`
- `npm/scripts/lint-conftest.mjs` — `TARGETS`-таблиця замінюється на динамічне читання `rule.json` + `target.json`
- `npm/scripts/utils/evaluate-applies.mjs` — новий хелпер (три примітиви)
- `npm/schemas/rule.json`, `npm/schemas/target.json` — нові JSON Schema для валідації
- `npm/rules/rego/` — перший кандидат на міграцію: видалення `js/check.mjs`, додавання `rule.json` + `policy/*/target.json`
- `npm/rules/abie/` — гібридна міграція: три статичні targets → `target.json`, `health_check_policy` + ua-patches лишаються у `js/check-*.mjs`
