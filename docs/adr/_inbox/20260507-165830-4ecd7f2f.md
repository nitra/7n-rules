## ADR Структурне сортування `patches[]` та JSON Patch ops у `kustomization.yaml`

**Контекст:** `check-k8s.mjs` вже перевіряв сортування `resources[]` алфавітно, але `patches[]` і `patch: |-` (JSON 6902 ops) залишалися без порядку. Це призводило до дифів «сорт-шум» у ревʼю і нечитабельних overlay.

**Рішення/Процедура/Факт:**
- Додано дві pure-функції: `kustomizationPatchesSortedViolation` і `kustomizationInlinePatchOpsSortedViolation` у `npm/scripts/check-k8s.mjs`.
- `kustomizationPatchesSortedViolation` — очікує `patches[]` відсортованими за tuple `[target.kind, target.name, target.namespace]` (en, case-insensitive localeCompare).
- `kustomizationInlinePatchOpsSortedViolation` — очікує ops у `patch: |-` відсортованими за `path`, але лише якщо: (1) всі ops ∈ `{add, replace}`, (2) всі `path` є disjoint (жоден не є префіксом іншого). Якщо умова не виконана — функція повертає `null` (без помилки).
- Обидва валідатори підключені у `validateKustomizationPatchesSorted` і підключені в `check()`.
- `npm/mdc/k8s.mdc` доповнено описом нових вимог до порядку.
- Версія `1.8.199 → 1.8.200`, `CHANGELOG.md` оновлено.

**Обґрунтування:** Порядок `patches[]` — лише конвенція, але він впливає на читабельність ревʼю (без сорту кожен devops додає патч «куди зручно»). Для JSON Patch ops порядок семантично значущий (RFC 6902: `test`/`move`/`copy` утворюють залежності), тому сортування дозволено лише для безпечної підмножини (`add`/`replace`, disjoint paths).

**Розглянуті альтернативи:**
- Сортування «всього на одному рівні по алфавіту» (сортування по серіалізації YAML-елемента) — відхилено: не дає потрібного порядку (`target.kind`+`name`), ламає `initContainers[]`, `args[]`, масиви з deps.
- Розширити сорт на `images:`, `configMapGenerator:`, `replacements:` в тому ж PR — відкладено, щоб зберегти мінімальний скоп.

**Зачіпає:** `npm/scripts/check-k8s.mjs`, `npm/mdc/k8s.mdc`, `npm/tests/check-k8s-schema.test.mjs`, `npm/package.json`, `npm/CHANGELOG.md`.
