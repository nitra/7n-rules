## ADR Канонічний knip-canonical.json і перевірка наявності knip.json

**Контекст:** Проєкти-споживачі `@nitra/cursor` не мали baseline для `knip.json`, через що knip регулярно друкував «Configuration hints» (Remove from ignoreDependencies, Remove from ignoreBinaries) — зайвий шум при кожному `bun run lint`.

**Рішення/Процедура/Факт:** Створено `npm/scripts/utils/knip-canonical.json` за тією ж схемою, що й `oxlint-canonical.json`. Файл покриває 9 категорій false-positives: `entry` для CLI-конфігів (eslint, stylelint, oxlint, jscpd, markdownlint-cli2, commitlint), `project`, `ignore`, `ignoreDependencies` (`@nitra/cspell-dict`, `@cspell/dict-.*`), `ignoreBinaries` (actionlint, cspell, eslint, git-ai, jscpd, markdownlint-cli2, oxfmt, oxlint, shellcheck, uvx, v8r, zizmor). Семантика перевірки — **лише наявність** `knip.json`; якщо відсутній — `checkKnipConfig` у `check-js-lint.mjs` копіює `KNIP_CANONICAL_JSON_PATH` у `knip.json` (side-effect з описом у `js-lint.mdc`). Вміст не валідується. Канонічний `lint-js` доповнено прапором `bunx knip --no-config-hints` в усіх місцях: `check-js-lint.mjs`, `package_json.rego`, root `package.json`, `js-lint.mdc` + дзеркало, `.github/workflows/lint-js.yml`.

**Обґрунтування:** Існуюча перевірка `ignoreDependencies ∋ "graphql"` була хибно-специфічною (графікл-залежність проєкту, а не правило для всіх); натомість автокопія цілого canonical-файлу — детерміновано заповнює лакуну. `--no-config-hints` прибирає шум у `bun run lint`, не впливаючи на standalone `bunx knip`.

**Розглянуті альтернативи:** Перевіряти вміст canonical-полів через subset-порівняння (як для `.oxlintrc.json`) — відхилено, бо knip.json суто проєктний і може легально дивергувати; достатньо autofix при відсутності.

**Зачіпає:** `npm/scripts/utils/knip-canonical.json` (новий), `npm/scripts/check-js-lint.mjs`, `npm/policy/js_lint/package_json/package_json.rego`, `npm/policy/js_lint/package_json/package_json_test.rego` (новий, 16 тестів), `npm/mdc/js-lint.mdc` (v1.19→1.21), `.cursor/rules/n-js-lint.mdc`, `package.json`, `.github/workflows/lint-js.yml`

## ADR Міграція JS canonical-констант у rego: усунення подвійного джерела істини

**Контекст:** `check-js-lint.mjs` містив `CANONICAL_LINT_JS`, `isCanonicalLintJs`, `normalizeLintJsScript`, `nitraEslintConfigMeetsMinVersion` — усі вже були реалізовані в `npm/policy/js_lint/package_json/package_json.rego`, але JS-версії лишалися як нібито «активний» код з тестами. Той самий паттерн знайшовся у `check-bun.mjs::isAllowedRootDevDependency` (покривається `bun.package_json` rego).

**Рішення/Процедура/Факт:** Видалено з JS dead exports (ті, що не викликаються в `check()` і мають rego-двійника): `CANONICAL_LINT_JS`, `isCanonicalLintJs`, `normalizeLintJsScript`, `WHITESPACE_RE`, `nitraEslintConfigMeetsMinVersion` — з `check-js-lint.mjs`; `isAllowedRootDevDependency` — з `check-bun.mjs`. Видалено відповідні `describe`-блоки з `check-js-lint.test.mjs` і `check-bun.test.mjs`. Натомість покриття перенесено в rego: `npm/policy/js_lint/package_json/package_json_test.rego` (16 тестів) і `npm/policy/bun/package_json/package_json_test.rego` (12 тестів) через `json.patch`-фікстури.

**Обґрунтування:** Два місця для однієї перевірки — класичне порушення `conftest.mdc`; при кожному оновленні (як `--no-config-hints`) потрібно правити обидва синхронно. Rego authoritative для структурних перевірок `package.json` — JS лишається лише для FS/cross-file задач, які conftest не може покрити.

**Розглянуті альтернативи:** Зберегти JS-версії для «наочності» — відхилено; docstring-посилання на rego-namespace достатньо для орієнтації.

**Зачіпає:** `npm/scripts/check-js-lint.mjs`, `npm/scripts/check-bun.mjs`, `npm/tests/check-js-lint.test.mjs`, `npm/tests/check-bun.test.mjs`, `npm/policy/js_lint/package_json/package_json_test.rego` (новий), `npm/policy/bun/package_json/package_json_test.rego` (новий)

## Knowledge Stale git worktrees від Claude Code agent sessions шумлять в eslint / jscpd

**Контекст:** Claude Code при запуску агентів з `isolation: "worktree"` створює git worktree у `.claude/worktrees/<name>/` на гілці `claude/<name>`. Якщо агент не вніс змін — worktree прибирається автоматично; якщо вніс — лишається до ручного `git worktree remove`. Ці worktrees не потрапляють у `.gitignore` і не перераховані в `.eslintignore`.

**Рішення/Процедура/Факт:** ESLint лінтить файли у `.claude/worktrees/**` як звичайні джерельні файли — стара копія `check-npm-module.mjs` у worktree `gifted-snyder-ec1f74` спричиняла `e18e/prefer-static-regex` помилки. Виправлення: додати `.claude/worktrees/**` до `ignores` у `eslint.config.js`. Те саме стосується jscpd — він знаходить клони між основним деревом і worktree-копіями; конфіг `.jscpd.json` варто доповнити аналогічно.

**Обґрунтування:** Worktree — повноцінний checkout гілки; eslint/jscpd не знають, що це «тимчасовий» каталог, і сканують його разом із основним проєктом.

**Розглянуті альтернативи:** не обговорювались

**Зачіпає:** `eslint.config.js` (додано `'.claude/worktrees/**'` до `ignores`); для `.jscpd.json` — поточна конфігурація не додана, але проблема описана вище
