## Knowledge GitHub Actions: обов'язкова секція `concurrency` з `cancel-in-progress`
**Контекст:** У файлі `npm/mdc/ga.mdc` розглядається вимога до всіх GitHub Actions workflow — обов'язково містити секцію `concurrency`, щоб скасовувати попередні запуски при новому push у ту саму гілку.
**Рішення/Процедура/Факт:** Шаблон, що обговорювався:
```yaml
concurrency:
group: ${{ github.ref }}-${{ github.workflow }}
cancel-in-progress: true
```
Це стандартна практика для CI-пайплайнів: нові коміти роблять попередній запуск неактуальним, тому скасування економить хвилини runner-ів і запобігає race conditions між паралельними деплоями.
**Обґрунтування:** Розумно для більшості workflow (lint, build, test, deploy на feature-гілках) — не варто витрачати CI-квоту на застарілий код. Виняток: workflow на `main`/`release`, де кожен запуск може мати побічні ефекти (публікація, тег), — там `cancel-in-progress: false` безпечніше.
**Розглянуті альтернативи:** Не обговорювалися.
**Зачіпає:** `npm/mdc/ga.mdc`, усі `.github/workflows/*.yml` у проєкті.
