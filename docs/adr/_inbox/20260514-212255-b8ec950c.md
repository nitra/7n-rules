## ADR Правило-центрична реорганізація `npm/` у `@nitra/cursor`

**Контекст:** Пакет `@nitra/cursor` розростався: `.mdc`-правила жили в `npm/mdc/`, їхні `.rego`-полісі — в `npm/policy/`, JS-скрипти (check/run/lint) — у пласкому `npm/scripts/`, тести — у пласкому `npm/tests/`, умови автоактивації — в єдиних файлах `bin/auto-rules.md` / `bin/auto-skills.md`. Видалення одного правила вимагало хірургічних правок у шести різних місцях.

**Рішення/Процедура/Факт:**
Переїзд на правило-центричну структуру:
- `npm/mdc/` → `npm/rules/` (перейменування)
- Кожне правило отримує власний каталог `npm/rules/{rule}/` з:
- `{rule}.mdc` — текст правила
- `auto.md` — виріз умови автоактивації (замість центрального `bin/auto-rules.md`)
- `policy/` — `.rego`-полісі (переїзд з `npm/policy/{rule}/`)
- `js/` — скрипти (`check.mjs`, `run.mjs`, `lint.mjs` — без суфікса `{rule}`) і тести (`check.test.mjs`, `check-compile.test.mjs` тощо)
- `npm/policy/` зникає повністю
- `npm/skills/{skill}/` отримує ту саму схему: `auto.md` + `js/` (скрипти та тести скіла)
- `npm/scripts/` звужується до крос-правильної інфраструктури (`cli-entry.mjs`, `auto-rules.mjs`, `lint-conftest.mjs`, `sync-*.mjs`, `utils/`)
- `npm/tests/` залишається тільки для крос-правильних тестів (`integration-repo-checks`, `check-empty-trees`) і `fixtures/`
- `bin/n-cursor.js`: `BUNDLED_MDC_DIR` → `BUNDLED_RULES_DIR`, discovery `check-{rule}.mjs` → `readdir rules/ + existsSync js/check.mjs`, агрегація `auto.md` — in-memory обхід `rules/*/auto.md`
- `package.json#files`: прибрати `mdc/` і `policy/`, додати `rules/`; bun test налаштований на `**/*.test.mjs` глобально

**Обґрунтування:** Головна ціля — «видалив каталог `rules/{rule}/` → правило і всі його атрибути зникають без сліду». Пласка структура порушувала цей інваріант: артефакти правила були розсіяні по 4–6 директоріях. Міграція до правило-центричного дерева усуває цей зв'язок.

**Розглянуті альтернативи:**
- Реорганізувати тільки `npm/scripts/` під `npm/policy/`-стиль (відкинуто: вирішує лише частину проблеми, `mdc/` і `policy/` лишалися б окремо)
- Залишити центральний `auto-rules.md` як агрегат, генерований з per-rule `auto.md` build-степом (відкладено: можливий варіант реалізації, але не принципове рішення)
- Зберегти суфікс `{rule}` у назві файлу всередині каталогу, напр. `js/check-docker.mjs` (відкинуто: директорія вже несе namespace, суфікс — дублювання)
- Залишити всі скіли в `npm/skills/` без розрізання на per-skill (відкинуто: суперечить тій самій інваріанті «видалив каталог → очистив все»)

**Зачіпає:** `npm/mdc/` → `npm/rules/` (28 правил), `npm/policy/` (зникає), `npm/scripts/check-*.mjs` / `run-*.mjs` / `lint-*.mjs` (переїзд), `npm/tests/*.test.mjs` (більшість переїжджає per-rule), `npm/skills/*/auto.md` (новий файл), `npm/bin/auto-rules.md`, `npm/bin/auto-skills.md` (розрізаються), `npm/bin/n-cursor.js` (резолвери і discovery), `npm/scripts/lint-conftest.mjs` (`POLICY_DIR`), `package.json#files`, `bunfig.toml` / test-скрипт
