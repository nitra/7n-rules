## Runbook n-taze — процедура оновлення залежностей з аналізом major-змін

**Контекст:** Потрібна стандартизована процедура оновлення npm-залежностей проекту, яка не просто піднімає версії, а й автоматично виявляє breaking changes і рефакторить несумісний код до завершення.

**Рішення/Процедура/Факт:** Визначено покроковий workflow у `npm/skills/taze/SKILL.md`:

1. Зафіксувати стан (`cp package.json package.json.taze-bak`, `cp bun.lock bun.lock.taze-bak`)
2. `bunx taze -w -r latest` → `bun install`
3. Порівняти бекапи з оновленими файлами — виявити залежності зі зміною першої значущої цифри semver (major)
4. Для кожного major-оновлення зібрати breaking changes: CHANGELOG → git-diff у `node_modules` → тимчасова установка старої версії в `/tmp/taze-old`
5. Знайти використання зачеплених API в коді проекту через `rg`
6. Рефакторинг несумісних місць; якщо міграція неоднозначна — залишити TODO з посиланням на CHANGELOG
7. Прибрати тимчасові файли
8. Надати звіт: minor/patch-кількість, major-список, що зрефакторено автоматично, що потребує ручного втручання, стан lint/typecheck/test

**Обґрунтування:** Користувач хотів, щоб skill не зупинявся на механічному оновленні версій, а відразу закривав цикл: знаходив причину потенційних поломок і усував їх (або фіксував як ручну дію). Збереження бекапів перед оновленням дозволяє точно обчислити diff без покладання на зовнішні реєстри.

**Розглянуті альтернативи:** Не обговорювалися явно; неявна альтернатива — просто запустити `taze` і залишити перевірку сумісності розробнику — відхилена на користь автоматичного аналізу.

**Зачіпає:** `npm/skills/taze/SKILL.md`, проекти, де викликається skill (`package.json`, `bun.lock`, `node_modules`, вихідний код проекту)
