## ADR Обов'язковий блок `concurrency` у всіх GitHub Actions workflow

**Контекст:** Репозиторій містить кілька `.github/workflows/*.yml`-файлів без секції `concurrency`, через що при паралельних запусках (push + PR тощо) кілька екземплярів одного workflow можуть виконуватись одночасно, витрачаючи CI-квоту та потенційно створюючи race conditions.

**Рішення/Процедура/Факт:** Введено **універсальну** вимогу: кожен workflow у `.github/workflows/` обов'язково має містити
```yaml
concurrency:
group: ${{ github.ref }}-${{ github.workflow }}
cancel-in-progress: true
```
Зміни охопили: (1) додавання блоку до трьох реальних workflow (`clean-ga-workflows.yml`, `clean-merged-branch.yml`, `git-ai.yml`); (2) оновлення правила `npm/mdc/ga.mdc` з v1.6 → v1.7 із додаванням опису вимоги та канонічних прикладів; (3) нову функцію `validateConcurrencyOnRoot` / `verifyConcurrencyBlock` у `npm/scripts/check-ga.mjs` — перевірка `group === '${{ github.ref }}-${{ github.workflow }}'` і `cancel-in-progress === true` для кожного `*.yml`-файлу та в канонічних структурних валідаторах; (4) версія пакету `1.8.182 → 1.8.183`.

**Обґрунтування:** Рішення продиктоване користувачем як явна командна норма. Асистент вказав, що для deploy/release-workflow `cancel-in-progress: true` технічно ризикований (обрив посеред publish може лишити артефакт у проміжному стані), проте користувач свідомо прийняв однорідне правило без винятків, надаючи перевагу простоті та єдності стандарту над granular-підходом.

**Розглянуті альтернативи:** Диференційований підхід — `cancel-in-progress: true` лише для PR/CI-workflow, `cancel-in-progress: false` або взагалі без `concurrency` для deploy/publish/schedule. Явно обговорювався асистентом, але відхилений користувачем на користь єдиного правила.

**Зачіпає:** `.github/workflows/clean-ga-workflows.yml`, `.github/workflows/clean-merged-branch.yml`, `.github/workflows/git-ai.yml`; `npm/mdc/ga.mdc` (публічне правило пакету); `npm/scripts/check-ga.mjs` (programmatic check); `npm/package.json`, `npm/CHANGELOG.md`.
