## ADR CI4-модель `@nitra/cursor` — структура та формат документації

**Контекст:** Проєкт `@nitra/cursor` не мав архітектурної документації. Нове правило `ci4.mdc` (v1.8.221) зобов'язує кожен проєкт мати C4-опис у Markdown поряд із кодом — тому перший кандидат на застосування — сам `@nitra/cursor` (recursive case: продукт документує себе своїм же правилом).

**Рішення/Процедура/Факт:** Створено каталог `docs/ci4/` з шістьма файлами: `README.md` (конвенція якорів і ADR-лінкування), `01-context.md` (L1 — 7 категорій зовнішніх акторів), `02-containers.md` (L2 — 6 контейнерів: `cnt-rule-sync`, `cnt-check-runner`, `cnt-stop-hook`, `cnt-capture-decisions`, `cnt-gh-action`, `cnt-pkg-artifact`), `03-components.md` (L3 — 4 mermaid-діаграми + таблиці компонентів із колонкою `Tests`), `04-code.md` (L4 — 5 code-flow діаграм для всіх runtime-контейнерів), `decisions.md` (централізований cross-ref CI4 ↔ ADR). Одночасно оновлено `.markdownlint-cli2.jsonc` (дозволено `<a>` для стабільних якорів-ID) і `.cspell.json` (словник CI4-термінів).

**Обґрунтування:** Mermaid у Markdown обрано бо проєкт уже text-first (oxfmt, markdownlint, cspell) і діаграми рендеряться нативно в GitHub/Cursor/VS Code без додаткових інструментів. Каталог замість одного файлу — щоб URL кожного рівня лишався стабільним для cross-reference в ADR (лінки не ламаються при переписуванні секцій). Стабільні `<a id="...">` якорі (`cnt-*` для контейнерів, `cmp-*` для компонентів, `code-*` для L4) гарантують, що посилання в ADR не стають "dead links" при реорганізації тексту. `decisions.md` як центральна таблиця CI4↔ADR доповнює inline-секції `## Related decisions` у кожному файлі рівня — обидва підходи разом, бо централізована таблиця дає огляд, а inline-секції — локальний контекст без переходу між файлами.

**Розглянуті альтернативи:** Один файл `docs/ci4-model.md` (відхилено — нестабільні якорі при редагуванні великих секцій); Structurizr DSL (відхилено — потребує Docker/CLI для рендеру); PlantUML (відхилено — не рендериться нативно в GitHub); тільки централізований `decisions.md` без inline-секцій (відхилено — вимагає переходу між файлами для локального контексту); L4 тільки для 2 контейнерів (розширено до 5 — покрити всі runtime-контейнери).
