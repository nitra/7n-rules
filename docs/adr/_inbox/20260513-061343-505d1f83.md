## ADR Канонічний `jsconfig.json` у js-run — перевірка через `runConftestBatch` у JS-оркестраторі, а не `lint-conftest` TARGETS
**Контекст:** `npm/policy/js_run/jsconfig/jsconfig.rego` вже існувала, але не була зареєстрована у `lint-conftest` TARGETS — її планувалося запускати «від bun run lint-conftest», однак механізм filter у `lint-conftest` (`activeRules`) працює лише на рівні репозиторію і не знає, чи є даний workspace backend-пакетом з `src/`, чи Vite-frontend.
**Рішення/Процедура/Факт:** `check-js-run.mjs` тепер після визначення що пакет є backend із каталогом `src/` викликає `runConftestBatch` для `js_run.jsconfig` безпосередньо — так само, як `check-ga.mjs` викликає конфтест для GA-специфічних правил. У `lint-conftest` TARGETS ця полісі **не реєструється** щоб уникнути false-positive для Vite-пакетів.
**Обґрунтування:** Умовні polisi (ті, де застосовність залежить від per-workspace атрибутів: наявність `src/`, тип пакета, glob `**/*.rego`, наявність Tauri) потрібно запускати з JS-оркестратора, а не через глобальний `lint-conftest`. `lint-conftest` підходить лише для «безумовних» полісей (наприклад, `.markdownlint-cli2.jsonc` є завжди, `package.json` — завжди).
**Розглянуті альтернативи:** Реєстрація у `lint-conftest` з `walk`-фільтром — відхилено, бо `walk` фільтрує по імені файлу, а не по наявності `src/` у пакеті.
**Зачіпає:** `npm/scripts/check-js-run.mjs`, `npm/policy/js_run/jsconfig/`, `npm/scripts/lint-conftest.mjs`

## Knowledge `lint-conftest` — глобальний фільтр не знає per-workspace умов, тому умовні polisi мають йти через JS-оркестратор
**Контекст:** При аудиті gap між `npm/mdc/*.mdc` і `npm/policy/` виявлено, що `rego.mdc` і `tauri.mdc` — умовні правила: rego — glob `**/*.rego`, tauri — лише Tauri-проєкти. Запуск їхніх rego-полісей через `lint-conftest` TARGETS — false-positive для всіх не-rego/не-tauri репозиторіїв.
**Рішення/Процедура/Факт:** Правило-орієнтир: якщо rego-полісь застосовна не до всіх `package.json` / не до кожного репозиторію, а залежить від per-workspace чи per-project умов (тип пакета, наявний toolchain, glob на файли специфічного розширення) — полісь реєструється через `runConftestBatch` у відповідному `check-<rule>.mjs`, а не у `lint-conftest` TARGETS.
**Обґрунтування:** `lint-conftest` TARGETS мають `rule`-gate лише на рівні `.n-cursor.json#rules` (проєктно), але не per-workspace. `check-<rule>.mjs` натомість ітерує по workspace-пакетах і має власну бізнес-логіку для gate-умов.
**Розглянуті альтернативи:** Не обговорювалися.
**Зачіпає:** `npm/scripts/lint-conftest.mjs`, будь-які нові rego-полісі з умовним застосуванням

## ADR Канонічна перевірка `knip.json` — лише existence, auto-create з канону при відсутності
**Контекст:** Правило `js-lint.mdc` вимагало `knip.json` і перевіряло `ignoreDependencies ∋ "graphql"`, але knip при запуску друкував «Configuration hints» для полів, що є в `ignoreDependencies`. Потрібен канонічний baseline для нових проєктів і можливість замовчати hints.
**Рішення/Процедура/Факт:** Створено `npm/scripts/utils/knip-canonical.json` з 9 категоріями false-positive (cli-tools через `npx/bunx`, config-only deps, devDeps без прямого імпорту). `check-js-lint.mjs::checkKnipConfig` перевіряє **лише наявність** `knip.json`; якщо відсутній — копіює канон і звітує pass. Canonical `lint-js`-скрипт: `bunx knip --no-config-hints`. Стара перевірка `ignoreDependencies ∋ "graphql"` — видалена разом із мертвими JS-константами `CANONICAL_LINT_JS`/`isCanonicalLintJs`/`normalizeLintJsScript`.
**Обґрунтування:** Зміст `knip.json` залежить від проєкту, обов'язковість конкретних полів нема сенсу enforce-ити у rego (user-specific). `--no-config-hints` — єдиний спосіб прибрати informational output, що заважає читати lint-звіт.
**Розглянуті альтернативи:** Subset-перевірка (обов'язкові поля мають бути підмножиною) — відхилено користувачем: «перевіряємо тільки наявність файлу».
**Зачіпає:** `npm/scripts/check-js-lint.mjs`, `npm/scripts/utils/knip-canonical.json`, `npm/policy/js_lint/package_json/package_json.rego`, `npm/mdc/js-lint.mdc`, `.cursor/rules/n-js-lint.mdc`, `package.json` (lint-js script), `.github/workflows/lint-js.yml`

## ADR Видалення JS-дублів канону (`CANONICAL_LINT_JS`, `isAllowedRootDevDependency`) — єдине джерело правди у rego
**Контекст:** `check-js-lint.mjs` і `check-bun.mjs` містили JS-функції та константи, що раніше використовувалися для перевірок, але ці перевірки вже були мігровані у rego-полісі (`js_lint.package_json`, `bun.package_json`). JS-копії лишилися «живими» через тести — і спричиняли drift при кожному оновленні канону (вимагали правки в двох місцях).
**Рішення/Процедура/Факт:** Видалено `CANONICAL_LINT_JS`, `isCanonicalLintJs`, `normalizeLintJsScript`, `WHITESPACE_RE`, `nitraEslintConfigMeetsMinVersion` з `check-js-lint.mjs`; `isAllowedRootDevDependency` з `check-bun.mjs`. Відповідні `describe` блоки видалено з тестів. Rego-тести (через `json.patch`-фікстури) додано у `npm/policy/js_lint/package_json/package_json_test.rego` і `npm/policy/bun/package_json/package_json_test.rego`.
**Обґрунтування:** Два джерела правди для одного канону — класичний drift. Rego є авторитетним для `package.json`-перевірок; JS авторитетний лише для cross-file/FS-перевірок, яких rego не може робити.
**Розглянуті альтернативи:** Не обговорювалися.
**Зачіпає:** `npm/scripts/check-js-lint.mjs`, `npm/scripts/check-bun.mjs`, `npm/tests/check-js-lint.test.mjs`, `npm/tests/check-bun.test.mjs`, `npm/policy/js_lint/package_json/`, `npm/policy/bun/package_json/`
