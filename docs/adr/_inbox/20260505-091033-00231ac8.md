## ADR Правило n-changelog: тригер — PR, а не поточна версія

**Контекст:** У монорепо на Bun кожен workspace (крім `npm/`, покритого `n-npm-module`) потребує власного `CHANGELOG.md`. Без єдиного правила changelog-дисципліна не дотримується: розробники можуть злити гілку без фіксації змін.

**Рішення/Процедура/Факт:** Правило `n-changelog` перевіряє `CHANGELOG.md` не за поточною версією в `package.json`, а на рівні pull request. Схема: розробник веде роботу у feature-гілці → перед PR робить `version` bump у `<workspace>/package.json` і додає запис у `<workspace>/CHANGELOG.md` → CI-перевірка (`check-changelog.mjs`) блокує PR якщо changelog не містить запис для нової версії або версія не змінилась відносно `main`. Кожен workspace має окремий `CHANGELOG.md`; спільний файл на рівні кореня не ведеться.

**Обґрунтування:** Підхід «сума за пул реквест» відповідає реальному git-процесу: коміти у гілці — внутрішні деталі, `CHANGELOG.md` фіксує зміну з точки зору споживача/команди. Порівняння версії відносно `main` (через `git show origin/main:<ws>/package.json`) є точним тригером: немає bump — немає changelog. Це симетрично до поведінки `n-npm-module`, але спрацьовує саме в момент злиття, а не «завжди».

**Розглянуті альтернативи:**

- **Варіант A (завжди вимагати запис для поточної версії)** — простіший у реалізації, але зобов'язує оновлювати changelog навіть у чисто внутрішніх apps без смислового релізу.
- **PR-тригер (обраний варіант B)** — складніша перевірка (`git show HEAD` vs `origin/main`), але відповідає реальному workflow і не зайво навантажує внутрішні workspace.

**Зачіпає:** нове правило `npm/mdc/changelog.mdc` + `.cursor/rules/n-changelog.mdc`, новий скрипт `npm/scripts/check-changelog.mjs`, CI-конфіг `.github/workflows/`; потенційно `.n-cursor.json` для виключення окремих workspace (наприклад `demo/`).
