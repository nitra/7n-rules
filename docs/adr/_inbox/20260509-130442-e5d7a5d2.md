## ADR Новий скіл `n-llm-patch`: генерація self-contained промпта для іншої LLM

**Контекст:** У проєктах Nitra виникала потреба передавати задачі іншій LLM (Claude/Cursor agent у цільовому репо), але описувати контекст вручну — довго і неточно. Каркас `npm/skills/llm-patch/SKILL.md` вже існував, але був порожнім.

**Рішення/Процедура/Факт:**
- Наповнено `npm/skills/llm-patch/SKILL.md`: команда `/n-llm-patch <опис>` збирає read-only snapshot CWD (`package.json`, `find` дерево до 3 рівнів, `README.md`, релевантні конфіги, `CHANGELOG.md`) і виводить самодостатній markdown-промпт у чат.
- `'llm-patch'` додано до `ALWAYS_ON_SKILLS` у `npm/scripts/auto-skills.mjs` (вже було в `npm/bin/auto-skills.md` як `завжди`, але не в коді).
- Створено дзеркало `.cursor/skills/n-llm-patch/SKILL.md` — ідентичний файл (аналог `n-publish-telegram`).
- Версія `@nitra/cursor` bumped `1.8.217` → `1.8.218`, `npm/CHANGELOG.md` оновлено.

**Обґрунтування:** Скіл read-only за дизайном — не може зламати репо. Джерело лише CWD (не npm registry / git clone) — простіше, передбачувано, і достатньо для завдань типу «врахуй останню версію node». Повний вміст дрібних файлів включається в промпт, щоб інша LLM не потребувала доступу до репо. Шаблон виводу структурований: Завдання / Контекст / Релевантні файли / Що треба зробити / Обмеження / Як перевірити.

**Розглянуті альтернативи:** `npm pack` у `/tmp` (отримувати tarball з registry) — відхилено: інша LLM бачить інший стан пакету, ніж локальний; `git clone` — зайва складність; підтримка Gemini/GPT форматів — YAGNI.

**Зачіпає:** `npm/skills/llm-patch/SKILL.md`, `npm/scripts/auto-skills.mjs` (`ALWAYS_ON_SKILLS`), `.cursor/skills/n-llm-patch/SKILL.md` (новий), `npm/package.json` (версія), `npm/CHANGELOG.md`.
