## ADR Правило `n-changelog`: перевірка CHANGELOG у кожному workspace на рівні PR

**Контекст:** У монорепо правило `n-npm-module` вже вимагало синхронного оновлення `CHANGELOG.md` для workspace `npm/`, але для решти workspace (apps, services, тощо) аналогічний механізм був відсутній. Також логіка була прив'язана до кожного коміту, а не до PR, що ускладнювало workflow розробників.

**Рішення/Процедура/Факт:** Створено нове правило `changelog` як окремий модуль:
- `npm/scripts/check-changelog.mjs` — PR-scoped перевірка: для кожного workspace із `package.json.workspaces` (включно з `npm/`), якщо `git diff <dev> -- <ws>/` виявляє зміни, вимагає: (1) bump `version` у `<ws>/package.json` відносно `dev`; (2) наявність запису `## [version] - YYYY-MM-DD` у `<ws>/CHANGELOG.md`; (3) якщо `<ws>/package.json` оголошує масив `files` — `"CHANGELOG.md"` має бути в ньому.
- `npm/mdc/changelog.mdc` і `.cursor/rules/n-changelog.mdc` — опис правила.
- `npm/scripts/auto-rules.mjs` + `npm/bin/auto-rules.md` — `changelog` авто-вмикається за умови `[bun]` (Bun monorepo).
- З `check-npm-module.mjs` та `n-npm-module.mdc` видалено дублюючу `checkChangelog()` — `npm/` тепер повністю покривається `changelog`.
- Тести (`npm/tests/check-changelog.test.mjs`, 11 сценаріїв): skip-логіка (не git, поточна гілка = dev, немає реф dev), workspace без змін, без bump, без запису в CHANGELOG, без CHANGELOG у `files`.
- `npm/CLAUDE.md` та шаблон оновлено: CHANGELOG-інструкція тепер посилається на `n-changelog.mdc`, наголошена PR-scoped логіка.
- Базова гілка для порівняння — `dev` хардкодом (без конфіга).

**Обґрунтування:** Розробник має один раз зробити bump + запис у CHANGELOG за весь PR (не на кожен коміт). Перевірка PR-scoped через `git diff <dev> -- <ws>/` дозволяє проміжним комітам не зачіпати CHANGELOG — обов'язок виникає лише в момент відкриття/злиття PR. Правило уніфіковано для всіх workspace (включно з `npm/`), щоб не дублювати логіку в `npm-module`. Приватні workspace (без масиву `files` у `package.json`) перевіряються на наявність bump і запису, але `files`-перевірка для них пропускається (changelog — для внутрішнього використання, не для публікації пакету).

**Розглянуті альтернативи:**
- Варіант A (завжди вимагати запис для поточної версії, без git diff) — відхилено як надто жорсткий для приватних apps без версійного bump на кожен коміт.
- Варіант B (перевірка лише коли `version` відрізняється від `HEAD`) — відхилено як складніший і такий, що не ловить забутий bump.
- Використання `git merge-base <dev> HEAD` замість прямого `git diff <dev>` — обговорювалось у кінці сесії як покращення для edge-case прямих комітів у `main`; відкладено до наступної ітерації.
