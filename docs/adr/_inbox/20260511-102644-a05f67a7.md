## ADR Переведення `alwaysApply: true` → `false` + `globs` для файлово-чітких MDC-правил

**Контекст:** У пакеті `@nitra/cursor` є 26 MDC-правил у `npm/mdc/`. Більшість мали `alwaysApply: true`, тобто завантажувались у контекст Cursor при кожному запиті незалежно від активних файлів — даремно витрачаючи контекстне вікно для проєктів, де відповідні технології не використовуються.

**Рішення/Процедура/Факт:** 10 правил із чітким файловим scope переведено на `alwaysApply: false` + поле `globs`:

| Правило | `globs` |
|---|---|
| `ga.mdc` | `.github/workflows/*.yml` |
| `vue.mdc` | `**/*.vue` |
| `php.mdc` | `**/*.php` |
| `style-lint.mdc` | `**/*.{css,scss,vue}` |
| `nginx-default-tpl.mdc` | `**/default.{conf.template,tpl.conf}` |
| `image-avif.mdc` | `**/*.{png,jpg,jpeg,gif,avif,vue,html}` |
| `image-compress.mdc` | `**/*.{png,jpg,jpeg,gif,svg}` |
| `changelog.mdc` | `**/{CHANGELOG.md,package.json}` |
| `hasura.mdc` | `**/hasura/**,**/*.env` |
| `graphql.mdc` | `**/*.{vue,js,mjs,cjs,ts,tsx,jsx}` |

Дзеркала у `.cursor/rules/n-*.mdc` синхронізовано для 7 правил, активних у `.n-cursor.json`. Версія пакету `1.8.229` → `1.9.1`.

**Обґрунтування:** Поле `globs` у парі з `alwaysApply: false` активує правило лише тоді, коли в контексті є відповідний файл — економить токени і не засмічує агента нерелевантними інструкціями. Правила з широким scope (`bun`, `npm-module`, `ci4`, `text`, `js-lint`) та opt-in/контент-залежні (`abie`, `adr`, `js-bun-db`, `capacitor` тощо) лишились `alwaysApply: true`, бо їх спрацювання не прив'язане до конкретного розширення файлу.

**Розглянуті альтернативи:** Лишити всі правила `alwaysApply: true` (поточний стан) — відхилено через марну витрату контексту.

**Зачіпає:** `npm/mdc/ga.mdc`, `vue.mdc`, `php.mdc`, `style-lint.mdc`, `nginx-default-tpl.mdc`, `image-avif.mdc`, `image-compress.mdc`, `changelog.mdc`, `hasura.mdc`, `graphql.mdc`; `.cursor/rules/n-ga.mdc`, `n-vue.mdc`, `n-style-lint.mdc`, `n-changelog.mdc`, `n-image-avif.mdc`, `n-image-compress.mdc`, `n-nginx-default-tpl.mdc`; `npm/package.json`; `npm/CHANGELOG.md`

## ADR Виправлення `GATEWAY_API_GROUP_PREFIX` + Rego defense-in-depth для HPA/PDB у `k8s/base`

**Контекст:** Правило «HPA і PDB у `k8s/base` заборонені» було реалізоване у `check-k8s.mjs` (функція `validateKustomizeHpaPdbOnlyWithBaseDeployment`), але ніколи не спрацьовувало: раніше в черзі виконання падав `ReferenceError: GATEWAY_API_GROUP_PREFIX is not defined` у `indexOneK8sYamlForHasuraCanon`, а outer `try/catch` у `bin/n-cursor.js` тихо поглинав помилку й пропускав усі наступні JS-валідатори.

**Рішення/Процедура/Факт:**
1. Додано константу `const GATEWAY_API_GROUP_PREFIX = 'gateway.networking.k8s.io/'` у `npm/scripts/check-k8s.mjs` — усуває `ReferenceError` і розблоковує всю чергу cross-file JS-валідаторів.
2. У `npm/policy/k8s/base_kustomization/base_kustomization.rego` додано `deny` на `resources:` у `kustomization.yaml`, що містять `hpa.yaml`, `pdb.yaml`, `hpa.yml`, `pdb.yml` (точний basename). Rego-правило запускається батчем `runAllK8sRego` до будь-якого JS-кроку, тому є незалежним страховочним рівнем.
3. До `npm/policy/k8s/base_kustomization/base_kustomization_test.rego` додано 5 нових тестів: `hpa.yaml`/`pdb.yaml`/`hpa.yml` у `resources:` ловляться; чистий `resources:` і lookalike-назви (`myhpa.yaml`, `pdb-extra.yaml`) проходять.
4. Версію пакету піднято до `1.9.1`; `npm/CHANGELOG.md` оновлено секціями `### Fixed` та `### Added`.

**Обґрунтування:** Рекурсивний обхід `bases:`/`components:` вимагає доступу до файлової системи й залишається у JS. Rego-рівень покриває найпоширеніший локальний випадок (пряме посилання у `resources:` поточного файлу) як single-document перевірку — декларативно, ізольовано, нечутливо до краш-ланцюжків у JS.

**Розглянуті альтернативи:** Переписати всю перевірку у Rego — неможливо без filesystem-доступу для рекурсивного обходу; лишити лише JS-фікс без Rego-шару — менш надійно, якщо в майбутньому з'явиться новий `ReferenceError` у тій самій черзі.

**Зачіпає:** `npm/scripts/check-k8s.mjs`; `npm/policy/k8s/base_kustomization/base_kustomization.rego`; `npm/policy/k8s/base_kustomization/base_kustomization_test.rego`; `npm/package.json`; `npm/CHANGELOG.md`
