## ADR Заборона `await new Promise` для пауз у backend Node.js коді

**Контекст:** У backend-пакетах з'являвся паттерн `await new Promise(resolve => setTimeout(resolve, ms))` для реалізації затримок. Цей шаблон є зайвим boilerplate, бо Node.js має вбудовану промісну версію `setTimeout` в стандартній бібліотеці.

**Рішення/Процедура/Факт:**
1. У `npm/mdc/js-run.mdc` (v1.3 → v1.4) додано секцію **«Паузи через setTimeout»**: забороняється використовувати `await new Promise(resolve => setTimeout(resolve, ms))`, натомість обов'язково застосовується `import { setTimeout } from 'node:timers/promises'` + `await setTimeout(ms)`.
2. Створено AST-сканер `npm/scripts/utils/promise-settimeout-scan.mjs` на базі `oxc-parser`, що виявляє заборонений паттерн у вихідниках (розпізнає `new Promise` з коллбеком, що містить `setTimeout(resolve, ...)`).
3. Сканер інтегровано в `npm/scripts/check-js-run.mjs` — тепер `npx @nitra/cursor check js-run` фейлить, якщо в пакеті знайдено заборонений шаблон.
4. Написано 13 юніт-тестів (`npm/tests/promise-settimeout-scan.test.mjs`) та інтеграційні фікстури в `npm/tests/check-js-run-fixture.test.mjs`.
5. Версію пакета бампнуто `1.8.185 → 1.8.186`, запис додано до `npm/CHANGELOG.md`.

**Обґрунтування:** `node:timers/promises` — стандартна бібліотека Node.js, доступна без додаткових залежностей. Промісна обгортка `new Promise(...)` є зайвим шаблонним кодом, який не дає ніяких переваг. Правило дозволяє автоматично виявляти порушення через існуючий механізм `check-js-run`.

**Розглянуті альтернативи:** Зберегти лише як memory-нотатку асистента — відхилено, бо правило має бути в `.mdc` для фіксації в кодовій базі та перевірки через `check`.
