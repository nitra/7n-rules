## ADR: Rego-first пріоритет для пер-документних перевірок у `@nitra/cursor`

**Контекст:** У проєкті одночасно існували JS-предикати у `check-*.mjs` і `*.rego`-пакети у `npm/policy/` з однаковою валідаційною логікою. Без явного правила агент обирав JS за замовчуванням — що призводило до дублювання і мовчазного дрифту між JS і rego (знайдено: `spec.config` vs `spec.default.config` у `health_check_policy.rego`).

**Рішення/Процедура/Факт:** Прийнято три-рівневий алгоритм: (1) **Pure-per-document** — завжди Rego (`npm/policy/<rule>/<name>/<name>.rego` + `<name>_test.rego`); (2) **Cross-file / FS / autofix / pre-YAML text** — JS у `check-*.mjs`; (3) **Гібрид** — обидва шари: rego як швидкий gate, JS-orchestrator для cross-file. Правило закріплено у `.cursor/rules/conftest.mdc` (`alwaysApply: true`) і `npm/.claude-template/npm-CLAUDE.md` (path-scoped нагадування «Перш ніж писати `check-*.mjs`»).

**Обґрунтування:** Rego має IDE-інтеграцію (`tsandall.opa`), виконується через `conftest test` з батч-виклику (1 спавн на namespace, ~80-150ms), і є авторитативним для per-document структурних перевірок. JS необхідний лише там де Rego read-only або не має доступу до FS/мережі/стану між файлами.

**Розглянуті альтернативи:** Лишити JS authoritative (Plan A — дозволяє дрифт); Rego authoritative з JS-проксі через `opa eval` (Plan C — зайвий шар без переваг над прямим batched conftest).

**Зачіпає:** `.cursor/rules/conftest.mdc`, `npm/.claude-template/npm-CLAUDE.md`, `npm/CLAUDE.md` (регенерований), усі `npm/scripts/check-*.mjs`, `npm/policy/`.

## ADR: `runConftestBatch` — utility для batched-виклику `conftest test` з JS-скриптів

**Контекст:** Після ухвалення Rego-first підходу виникла потреба викликати `conftest test` зсередині `check-*.mjs` (Plan B). Наївний варіант — спавн на кожен файл — дає ~50-100ms на файл. Потрібен batched-механізм як у `lint-conftest.mjs`, але доступний з check-скриптів.

**Рішення/Процедура/Факт:** Створено `npm/scripts/utils/run-conftest-batch.mjs` з функцією `runConftestBatch({ policyDirRel, namespace, files, policyRoot? })`. Запускає один `conftest test <files...> -p <policyDir> --namespace <ns> --output json` і повертає структурований масив `{ filename, namespace, message }[]`. Hard-fail (exit 1) якщо `conftest` не у PATH — з install-hint.

**Обґрунтування:** Один спавн на namespace дає ~80-150ms незалежно від кількості файлів. Для типового `check-abie` (10-30 hc.yaml + base/* файли) — ~200-400ms сумарно для 4 namespaces. Без helper кожен check-скрипт мав би дублювати логіку спавну і парсингу JSON-output.

**Розглянуті альтернативи:** `opa eval` замість `conftest test` — складніший query-синтаксис, менш узгоджений з `lint-conftest.mjs`; виклик через stdin (input.json) — тільки для одиничних документів (cross-check тести використовують саме його).

**Зачіпає:** `npm/scripts/utils/run-conftest-batch.mjs` (новий), `npm/scripts/check-abie.mjs` (перший споживач), майбутні `check-*.mjs`.

## ADR: Видалення дубльованих JS-предикатів з `check-abie.mjs` (Plan B pilot)

**Контекст:** `check-abie.mjs` містив JS-копії логіки, що вже була в rego-пакетах `abie.*`. JS-версії були «авторитативними», rego — дзеркалами. Дрифт виник мовчки: `health_check_policy.rego` читав `spec.config.httpHealthCheck`, тоді як JS читав `spec.default.config.httpHealthCheck`.

**Рішення/Процедура/Факт:** Видалено JS-функції `abieBaseHttpRouteHostnamesErrors`, `deploymentDocumentHasAbieBasePreemNodeSelector`, `parseCleanMergedIgnoreBranches`, `ignoreBranchesIncludesRequired`, `validateAbieHcPolicy`, хелпери `collectAbieHostnames`, `isAllowedAbieBaseDevHostname`, `isAbiePreemTruthy`, `processBaseHttpRouteDoc`, `httpRouteHasNonEmptyHostnames`, `findHealthCheckPolicyInDocs`, константу `ABIE_REQUIRED_IGNORE_BRANCHES`. Відповідні 4 orchestrator-функції (`ensureAbieBaseDeploymentPreemNodeSelector`, `checkCleanMergedBranch`, `checkHcYamlFiles`, `ensureAbieBaseHttpRouteHostnames`) перебудовано на `runConftestBatch`. JS лишає лише: cross-file (overlay-resolution, pairing), autofix, modeline-перевірку (`validateAbieHcModeline`).

**Обґрунтування:** Єдине джерело істини — rego. Дрифт між JS і rego неможливий після видалення JS-копій. Виправлено bug у `health_check_policy.rego` (`targetRef.name` тепер exact-match `<hcp.name>-hl`, а не `endswith "-hl"`). Cross-check тест видалено — він більше не потрібен, бо JS-сторони немає, а `*_test.rego` покривають самі rego-пакети.

**Розглянуті альтернативи:** Golden cross-check тести (Plan A доповнення) — реалізовані спочатку (`cross-check-rego-abie.test.mjs`, 25 тестів), але після Plan B — видалені як зайві.

**Зачіпає:** `npm/scripts/check-abie.mjs`, `npm/tests/check-abie.test.mjs`, `npm/policy/abie/health_check_policy/health_check_policy.rego`, `npm/scripts/utils/run-conftest-batch.mjs`.

## Knowledge: Архітектура lint-pipeline — JS `check-*` і `lint-conftest` є незалежними entry-points

**Контекст:** Питання виникло під час Plan B: «чи `lint-ga` запускає спочатку всі rego, а потім JS-частину?»

**Рішення/Процедура/Факт:** Ні. `bun run lint-ga` і `npx @nitra/cursor check ga` — повністю незалежні команди. `lint-ga` запускає: actionlint → zizmor → N окремих `conftest test` (по одному на workflow-специфічний namespace) + один батч-виклик для `ga.workflow_common`. JS `check-ga.mjs` (cross-file логіка) взагалі не викликається з `lint-ga`. Аналогічно `lint-conftest` (rego проти реальних файлів репо) і `check <rule>` (JS + потенційно batched conftest зсередини) — два різних шляхи. Після Plan B `check-abie.mjs` сам викликає conftest через `runConftestBatch`, тому `npx @nitra/cursor check abie` містить і rego-gate, і JS cross-file в одному виклику.

**Обґрунтування:** `lint-ga` містить PoC-conftest-крок, що частково перекривається з `lint-conftest` TARGETS — це відомий технічний борг. При повному Plan B для ga: conftest-крок прибирається з `lint-ga`, лишаються лише actionlint/zizmor; rego-перевірка переїздить в `check-ga.mjs` через `runConftestBatch`.

**Розглянуті альтернативи:** не обговорювалися.

**Зачіпає:** `npm/scripts/lint-ga.mjs`, `npm/scripts/check-ga.mjs`, `bun run lint` (агрегатор), `npm/scripts/lint-conftest.mjs`.
