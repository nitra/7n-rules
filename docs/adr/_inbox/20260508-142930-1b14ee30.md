## ADR Архітектура workflow скілу `n-taze`

**Контекст:** Скіл `npm/skills/taze/SKILL.md` містив лише заглушку без конкретного алгоритму — не було чіткого порядку кроків для оновлення залежностей, виявлення breaking changes і рефакторингу несумісного коду.

**Рішення/Процедура/Факт:** Визначено восьмикроковий workflow: (1) зафіксувати стартовий стан через бекап `package.json` + `bun.lock`; (2) запустити `bunx taze -w -r latest` + `bun install`; (3) виявити major-стрибки порівнянням бекапу з оновленим файлом; (4) зібрати breaking changes з CHANGELOG репозиторію модуля, або через `diff` між старою версією з кешу bun (`~/.bun/install/cache/`) і новою в `node_modules/`, або через тимчасову установку старої версії; (5) перевірити використання в коді через `rg`; (6) рефакторити несумісні місця або залишити TODO з посиланням на CHANGELOG; (7) прибрати тимчасові файли; (8) надати структурований звіт. Версію пакета піднято `1.8.211 → 1.8.212`.

**Обґрунтування:** Користувач явно задав послідовність: спочатку оновлення, потім аналіз major-змін через порівняння `package.json`/`bun.lock`, далі звірка changelog або git-діф `node_modules`, пошук несумісностей у коді проекту й рефакторинг. Пріоритет джерел breaking changes (CHANGELOG → git-діф → тимчасова установка) дозволяє не завантажувати зайве, якщо є швидший шлях.

**Розглянуті альтернативи:** Не обговорювалися — користувач задав конкретний підхід.

**Зачіпає:** `npm/skills/taze/SKILL.md`, `npm/CHANGELOG.md`, `npm/package.json` (версія `1.8.212`).
