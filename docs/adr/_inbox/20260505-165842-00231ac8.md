## ADR Правило `n-changelog`: PR-scoped перевірка CHANGELOG для всіх workspace-ів монорепо

**Контекст:** У монорепо (Bun workspaces) кожен workspace мав власний `package.json` із версією, але не існувало єдиного механізму, що примушував би розробника оновлювати `CHANGELOG.md` перед зливанням гілки. Правило `n-npm-module` покривало лише workspace `npm/`; для інших workspace-ів гарантії не було.

**Рішення/Процедура/Факт:** Створено нове правило `changelog` (файли `npm/mdc/changelog.mdc`, `.cursor/rules/n-changelog.mdc`, `npm/scripts/check-changelog.mjs`, тести `npm/tests/check-changelog.test.mjs`). Логіка CHANGELOG-перевірки перенесена з `check-npm-module.mjs` до `check-changelog.mjs`, workspace `npm/` більше не є винятком. Правило реалізує дві моделі бази порівняння:
- **npm-published** — для workspace-ів, де `package.json.files` оголошено: локальна `version` порівнюється з опублікованою у реєстрі npm через `npm info <name> version`; якщо версії рівні — зміни вважаються «вже відрелізованими», перевірка пасує.
- **local-only** — для приватних/без `files` workspace-ів: база = `git merge-base dev HEAD`; якщо у поточному PR (відносно merge-base) є зміни у `<ws>/` — вимагаються bump version та запис `## [version] - YYYY-MM-DD` у `<ws>/CHANGELOG.md`.
Додаткова вимога: якщо `files` оголошено — `"CHANGELOG.md"` має бути в масиві (щоб changelog публікувався з пакетом). `npm-module.mdc` позбавлено секції `## CHANGELOG` (версія 1.8→1.9). Правило `changelog` авто-вмикається при наявності `package.json` з залежністю `[bun]` (`AUTO_RULE_DEPENDENCIES`).

**Обґрунтування:** Порівняння per-commit (vs HEAD) — зайве; розробник робить проміжні коміти на гілці. Правильний момент перевірки — перед PR в `main`. Для npm-пакетів base = `dev` або `merge-base` небезпечний: гілка `dev` у npm-проектах рідко використовується; більш надійно порівнювати з реально опублікованою версією в реєстрі — це абсолютна, не відносна база. Для приватних workspace-ів git-підхід з `merge-base` точніший за прямий `diff dev..HEAD`, бо виключає помилкові спрацювання коли `dev` відстає від `main`.

**Розглянуті альтернативи:**
- Завжди вимагати запис для поточної version (незалежно від PR, як перевірка у `n-npm-module`): відхилено — занадто жорстко для локальних коміт-циклів.
- База = `dev` хардкодом для всіх workspace-ів: прийнято на першому ітері, відхилено для npm-published workspace-ів після зауваження що `dev` у npm-проектах нетиповий.
- Одна база `main` з порівнянням через `git show HEAD:..`: не обговорювалась явно.
