## ADR Міграція GA-перевірок у Rego conftest-полісі

**Контекст:** Перевірки структури канонічних GitHub Actions workflow (`clean-ga-workflows.yml`, `clean-merged-branch.yml`) жили в `npm/scripts/check-ga.mjs` як ~213 рядків імперативного JS. Постало питання, чи можна перенести структурну валідацію у Rego/conftest, де вона стає декларативною.

**Рішення/Процедура/Факт:**
- Створено `npm/policy/ga/clean_ga_workflows/clean_ga_workflows.rego` (порт `validateCleanGaWorkflows`) та `npm/policy/ga/clean_merged_branch/clean_merged_branch.rego` (порт `validateCleanMergedBranch`) — структура директорій відповідає `package ga.<name>` (вимога regal `directory-package-mismatch`)
- З `npm/scripts/check-ga.mjs` видалено ~213 рядків JS-валідаторів (`validateCleanGaWorkflowsStep0`, `validateCleanMergedBranchStep*`, `validateCleanGaWorkflows`, `validateCleanMergedBranch`, виклики в `checkCanonicalWorkflowsMatchRule`)
- До `npm/scripts/lint-ga.mjs` додано крок `runConftestStep()` зі soft-fallback: якщо `conftest` відсутній у PATH — skip + exit 0
- `npm/package.json` → `files[]` доповнено `"policy"`, щоб `.rego` потрапляли в npm-tarball
- Створено `.regal/config.yaml` з вимкненим `idiomatic.no-defined-entrypoint` (для conftest `deny`-правила є де-факто entrypoint-ами)
- Версія бампнута до `1.8.201`

**Обґрунтування:** Conftest побудований саме під структурну валідацію YAML/JSON-документів — `deny contains msg if { ... }` читається як таблиця правил, а не як imperativ JS з `getObjKey`/`isExactString` helper-ами. Файл-існування, git-glob і binary-preflight перевірки залишено в JS, бо conftest приймає лише input-документи.

**Розглянуті альтернативи:**
- Залишити все в JS — відхилено через декларативний виграш conftest для структурних перевірок
- Один плаский `npm/policy/ga/*.rego` файл без під-пакетів — відхилено через порушення regal `directory-package-mismatch`; conftest вимагає `--namespace` при запуску без під-директорій
- Повна міграція включно з file-existence/git-glob — відхилено: conftest не може перевірити наявність файлів або запускати `git ls-files`

**Зачіпає:** `npm/policy/ga/clean_ga_workflows/clean_ga_workflows.rego` (новий), `npm/policy/ga/clean_merged_branch/clean_merged_branch.rego` (новий), `npm/scripts/check-ga.mjs` (видалено ~213 рядків), `npm/scripts/lint-ga.mjs` (новий крок `runConftestStep`), `npm/package.json` (`files`, версія), `.regal/config.yaml` (новий), `npm/CHANGELOG.md`

## Knowledge YAML 1.1 quirk: `on:` у GitHub Actions парситься conftest як `"true"`

**Контекст:** При написанні Rego-полісі для GHA workflow виявилося, що жоден з очікуваних способів звернутися до кореневого ключа `on:` не працює.

**Рішення/Процедура/Факт:** `conftest parse` показує ключ `"true"` (рядок) замість `"on"` — YAML 1.1 трактує bare `on` як булевий `true`, і conftest зберігає його як рядок `"true"`. Нічний `input.on`, `input["on"]` та `input[true]` (булевий) не дають результату. Єдиний робочий варіант — `input["true"]`. У `.rego` обертається в alias: `gha_on := input["true"]`.

**Обґрунтування:** YAML 1.1 (яким є `.github/workflows/*.yml`) визначає `on`, `yes`, `no`, `true`, `false` як булеві значення. OPA/conftest зберігають ключ як рядкове представлення цього булевого — `"true"`.

**Розглянуті альтернативи:** Не обговорювалися; виявлено емпірично через `conftest parse` під час налагодження.

**Зачіпає:** Будь-який Rego-файл, що читає `on:` з GHA workflow; `npm/policy/ga/*/` — використовують `gha_on := input["true"]`
