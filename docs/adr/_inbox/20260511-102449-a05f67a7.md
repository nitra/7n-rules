## ADR Перехід mdc-правил на `alwaysApply: false` + `globs` для файлово-чітких правил

**Контекст:** Більшість правил у `npm/mdc/` мали `alwaysApply: true`, через що Cursor завантажував усі правила в контекст завжди — навіть коли відкритий файл не стосується відповідної технології. Це марнувало контекстне вікно.

**Рішення/Процедура/Факт:** Для 10 правил із чіткою файловою прив'язкою встановлено `alwaysApply: false` + відповідний `globs`. Паралельно оновлено дзеркала у `.cursor/rules/n-*.mdc` для 7 правил, активних у `.n-cursor.json`. Версію пакета піднято до `1.8.230`, запис у `npm/CHANGELOG.md` додано.

| Правило | globs |
|---|---|
| `ga.mdc` | `.github/workflows/*.yml` |
| `vue.mdc` | `**/*.vue` |
| `php.mdc` | `**/*.php` |
| `style-lint.mdc` | `**/*.{css,scss,vue}` |
| `nginx-default-tpl.mdc` | `**/default.{conf.template,tpl.conf}` |
| `image-avif.mdc` | `**/*.{png,jpg,jpeg,gif,avif,vue,html}` |
| `image-compress.mdc` | `**/*.{png,jpg,jpeg,gif,svg}` |
| `changelog.mdc` | `**/{CHANGELOG.md,package.json}` |
| `hasura.mdc` | `**/hasura/**,**/*.env` |
| `graphql.mdc` | `**/*.{vue,js,mjs,cjs,ts,tsx,jsx}` |

Залишено `alwaysApply: true` для проєктно-широких правил (`bun`, `npm-module`, `ci4`, `text`, `js-lint`) та контент-залежних (`abie`, `adr`, `js-bun-db`, `js-bun-redis`, `js-mssql`, `js-run`, `capacitor`, `tauri`).

**Обґрунтування:** `globs` у парі з `alwaysApply: false` дозволяє Cursor підвантажувати правило лише за наявності відповідного файлу в контексті, що економить токени. `alwaysApply: true` доцільний тільки для правил, що стосуються будь-якого файлу в репо (менеджер пакетів, архітектура monorepo тощо).

**Розглянуті альтернативи:** Не обговорювалися (рішення задиктував користувач після класифікації правил асистентом).

**Зачіпає:** `npm/mdc/*.mdc` (10 файлів), `.cursor/rules/n-*.mdc` (7 файлів), `npm/package.json`, `npm/CHANGELOG.md`

## ADR Виправлення `GATEWAY_API_GROUP_PREFIX` і rego defense-in-depth для HPA/PDB у k8s base

**Контекст:** Правило «HPA/PDB не можна тримати у base» існувало в `check-k8s.mjs` (`validateKustomizeHpaPdbOnlyWithBaseDeployment`), але ніколи не спрацьовувало: функція `validateHasuraHttpRouteCanon` раніше в черзі кидала `ReferenceError: GATEWAY_API_GROUP_PREFIX is not defined`, виняток тихо ловився в `bin/n-cursor.js:1100-1102`, і всі наступні JS-валідатори скіпались.

**Рішення/Процедура/Факт:**
1. У `npm/scripts/check-k8s.mjs` оголошено константу `const GATEWAY_API_GROUP_PREFIX = 'gateway.networking.k8s.io/'` — усуває `ReferenceError` і розблоковує весь ланцюжок JS-валідаторів.
2. У `npm/policy/k8s/base_kustomization/base_kustomization.rego` додано deny-правило `base_hpa_pdb_forbidden`: якщо файл розпізнається як `k8s/base/kustomization.yaml` і його `resources:` містить шлях із суфіксом `/hpa.yaml` або `/pdb.yaml` — deny.
3. У `base_kustomization_test.rego` додано 4 нові тест-кейси; усі 10 rego-тестів і 206 bun k8s-тестів зелені.
4. `npm/CHANGELOG.md` оновлено під версією `1.9.0` (Added + Fixed).

**Обґрунтування:** Rego-deny є single-document gate і запускається до JS-кроку (`runAllK8sRego` виконується раніше за JS-валідатори), тому залишається ефективним навіть якщо JS-частина знову зламається. JS-правило (`validateKustomizeHpaPdbOnlyWithBaseDeployment`) потрібне для рекурсивного обходу `bases:`/`components:` — filesystem access у Rego неможливий.

**Розглянуті альтернативи:** Повний перенос перевірки в Rego — неможливий для рекурсивного обходу дерева файлів; повний перенос у JS без Rego — залишає вразливість до майбутніх крашів у ланцюжку.

**Зачіпає:** `npm/scripts/check-k8s.mjs`, `npm/policy/k8s/base_kustomization/base_kustomization.rego`, `npm/policy/k8s/base_kustomization/base_kustomization_test.rego`, `npm/CHANGELOG.md`
