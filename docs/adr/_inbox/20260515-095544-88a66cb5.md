## Звідки виникає дублювання (і що контракт вирішує)

Зараз те саме таргетування існує у двох місцях одночасно:

| Де живе | Для чого | Формат |
|---|---|---|
| `lint-conftest.mjs:TARGETS` | `bun run lint-conftest` (лінт самих конфігів) | масив JS-об'єктів з `rule:`, `single:`, `walk:` |
| `check.mjs` → `runConftestBatch(...)` | `npx @nitra/cursor check` (перевірка проєкту) | виклики функції всередині скрипту |

Додати нову полісі = оновити **обидва місця**, інакше вона запускається по одному шляху і мовчить по-іншому. `target.json` — це спроба зробити **одне джерело правди**, яке CLI читає і для `check`, і для `lint-conftest`.

## Три гейт-примітиви

«Гейт» — умова, яка дозволяє або пропускає запуск полісі без помилки. У поточній кодовій базі вони розкидані по JS:

### `requireRule`

**Що робить:** читає `.n-cursor.json` → `rules[]`, перевіряє що масив містить заданий рядок.

**Аналог у поточному коді:**
- В `lint-conftest.mjs`: поле `rule: 'bun'` → [lint-conftest.mjs:65-66](npm/scripts/lint-conftest.mjs#L65-L66) читає `.n-cursor.json` і фільтрує `TARGETS` за цим полем.
- В `check.mjs`: `isAbieRuleEnabled()` [abie/js/check.mjs:235](npm/rules/abie/js/check.mjs#L235) робить те саме вручну.

**Коли потрібен:** правило opt-in (наприклад `abie` — тільки для AbInBev Efes проєктів). Якщо правило universal (наприклад `bun`) — `requireRule: "bun"` і так стоїть у `lint-conftest`, бо всі правила вже фільтруються через AGENTS.md, але explicit gate у `target.json` робить поведінку явною і не залежить від AGENTS.md-фільтра.

**Ребро:** `requireRule` ≠ `requireRule === ruleId`. Теоретично полісі правила `bun` може вимагати, щоб у проєкті також був `docker`. Рідко, але можливо.

---

### `requireFileExists`

**Що робить:** `existsSync(path)` — якщо файлу немає, skip без помилки.

**Аналог у поточному коді:**
В `lint-conftest.mjs` це вбудовано в логіку резолюції таргетів — [lint-conftest.mjs:62](npm/scripts/lint-conftest.mjs#L62):
```js
if (!existsSync(path)) return null
```
Якщо файл не знайдено — таргет повертає `null` і `lint-conftest` його тихо пропускає. В `check.mjs` те саме через явні `if (!existsSync(path)) { fail(...) }` або `if (!existsSync(path)) { pass('no file — ok'); return }`.

**Де відрізняється від `single`-таргета:** `single: "bunfig.toml"` — це **що перевіряти**. `requireFileExists: "bunfig.toml"` — це **чи взагалі запускатись**. Різниця:
- `single` без гейту: конфіг відсутній → conftest падає або виходить 0, залежно від rego-правила.
- `requireFileExists`: конфіг відсутній → polisa взагалі не запускається, exit 0.

Для більшості `single`-таргетів вони збігаються — тому в `lint-conftest.mjs` немає окремого `requireFileExists`, а поведінка вбудована у резолюцію. Але для гейту на рівні **правила цілком** (а не одного таргета) — це окреме поле.

### `requireFilesByGlob` (або `requireFileInTree`)

**Що робить:** ходить деревом від `cwd` ([walkDir.mjs](npm/scripts/utils/walkDir.mjs)), шукає файли за glob-паттерном. Якщо знайдено `≥ minCount` (за замовчуванням 1) — гейт відкритий. Якщо ні — skip без помилки.

**Аналог у поточному коді:**
`rego/js/check.mjs` — `projectHasRegoFiles()` [rego/js/check.mjs:44-57](npm/rules/rego/js/check.mjs#L44-L57):
```js
await walkDir(root, ignorePaths, async rel => {
if (rel.endsWith('.rego')) { found = true; return 'stop' }
})
```
Якщо `!hasRego` → `pass('Немає *.rego — не вимагається')` і вихід 0.

**Чому це важливіше за `requireFileExists`:** `requireFilesByGlob` — це **content-detected** гейт. Правило застосовується лише якщо сам проєкт має певний тип артефактів. Не можна вказати конкретний шлях — потрібен обхід дерева.

**Параметри:**
```json
{
"type": "requireFileInTree",
"glob": "**/*.rego",
"minCount": 1      // опційно, за замовчуванням 1
}
```

## Два target-примітиви (що передавати в conftest)

Окремо від гейтів — як знаходити файли для конкретної полісі:

**`single`** — один конкретний файл відносно `cwd`:
```json
{ "type": "single", "path": ".vscode/extensions.json" }
```
Якщо файл відсутній — skip цього target (без `fail`).

**`walk`** — всі файли в дереві, що матчать glob:
```json
{ "type": "walk", "glob": "**/package.json" }
```
Це те, що зараз у `lint-conftest.mjs` як `walk: { match: rel => rel.endsWith('/package.json') }` — [lint-conftest.mjs:194](npm/scripts/lint-conftest.mjs#L194). Glob `**/*.yaml` всередині `k8s/` — це `{ "type": "walk", "glob": "k8s/**/*.yaml" }`.

## Схема `target.json`

Один файл на один rego-пакет (`policy/<name>/target.json`):

```json
{
"namespace": "rego.vscode_extensions",
"gates": [
{ "type": "requireRule", "rule": "rego" },
{ "type": "requireFileInTree", "glob": "**/*.rego" }
],
"files": { "type": "single", "path": ".vscode/extensions.json" }
}
```

```json
{
"namespace": "bun.package_json",
"gates": [
{ "type": "requireRule", "rule": "bun" }
],
"files": { "type": "single", "path": "package.json" }
}
```

```json
{
"namespace": "k8s.manifest",
"gates": [
{ "type": "requireRule", "rule": "k8s" }
],
"files": { "type": "walk", "glob": "k8s/**/*.{yaml,yml}" }
}
```

**namespace у `target.json` або в `<name>.rego`?** Зараз `package <id>.<name>` у rego і `--namespace <id>.<name>` в CLI — це одне й те саме, але rego-namespace може мати підчеркування (`abie.clean_merged_ignore_branches`), а каталог — ні. Тому `namespace` у `target.json` — explicit і авторитетний, rego-файл лише підтверджує через `package`.

## Що CLI робить з `target.json`

`discoverCheckScripts` [n-cursor.js:993](npm/bin/n-cursor.js#L993) розширюється: якщо `rules/<id>/js/check.mjs` відсутній, але є `rules/<id>/policy/*/target.json` — правило все одно потрапляє в `available`. При виконанні замість `import(check.mjs).check()` CLI сам проходить по `policy/*/target.json`:

```
for each target.json in rules/<id>/policy/*/target.json:
1. evaluate gates in order → якщо будь-який fail → skip (exit 0)
2. resolve files (single → [path] або walk → [list])
3. runConftestBatch({ policyDirRel: '<id>/<name>', namespace, files })
4. для кожного violation → fail(msg)
```

Гейти виконуються sequential, щоб уникнути walkDir без потреби (cheap gates першими: `requireRule` і `requireFileExists` — O(1), `requireFileInTree` — O(n)).

**Що з `lint-conftest.mjs:TARGETS`:** можна або залишити як є і синхронізувати з `target.json` (два джерела), або навчити `lint-conftest.mjs` читати `target.json` замість жорсткого масиву. Друге — правильно, але потребує рефакторингу `lint-conftest.mjs`.

## Що декларативний формат НЕ покриває

| Ситуація | Чому потрібен JS |
|---|---|
| `abie`: знайти `k8s/**/hc.yaml` по **deployment-dirs**, а не по всьому дереву | `collectDeploymentDirs` → динамічна вибірка батчу, неможливо виразити статичним glob |
| `bun`: cross-rule логіка — `if docker в rules → перевір скрипт` | умови між правилами всередині однієї перевірки |
| `adr`: порівняння вмісту файлу з еталоном (checksum hook file) | JS-логіка зіставлення, а не conftest |
| `rego`: `pass('Немає *.rego — tooling не вимагається')` — кастомне повідомлення при skip | декларативний skip завжди мовчазний |

Правило класифікується як **pure-declarative** якщо: один або кілька `target.json` + статичні шляхи або прості glob + стандартні гейти. Все решта потребує `check.mjs` хоча б частково.

## Сумарна відповідь на «що треба продумати один раз»

Це три окремих рішення:

1. **Схема `target.json`** — що написано вище. Один раз фіксується формат полів і семантика.
2. **Порядок gates** — cheap gates першими, `requireFileInTree` завжди останній (бо дорогий).
3. **Поведінка при skip** — skip без помилки і без `pass`-рядка, або з явним `pass('...')`. Зараз у `rego/check.mjs` є явний `pass`, у `lint-conftest` — тиха перевірка `existsSync`. Потрібно обрати одну конвенцію і дотримуватись: наприклад skip = `pass` з повідомленням виду `<id>: gate <type> не виконано — пропускаємо`.

Далі — нові правила без `check.mjs` пишуться тільки як `<id>.mdc` + `policy/<name>/{<name>.rego, target.json}`. CLI підхоплює автоматично.

## Knowledge Декларативний контракт `target.json` для Rego-only правил
**Контекст:** CLI `npx @nitra/cursor check` виявляє правила лише через `rules/<id>/js/check.mjs`, тому pure-Rego правило без JS-обгортки повністю ігнорується. Водночас таргети для `lint-conftest.mjs:TARGETS` і `runConftestBatch`-виклики в `check.mjs` дублюються вручну.
**Рішення/Процедура/Факт:** Запропонована схема `policy/<name>/target.json` — файл поряд із `.rego`, що містить: `namespace`, масив `gates` (`requireRule` / `requireFileExists` / `requireFileInTree`) і `files` (`single` або `walk`). `discoverCheckScripts()` у `n-cursor.js:993` розширюється: правило потрапляє в `available`, якщо є хоча б один `policy/*/target.json`, навіть без `js/check.mjs`. CLI сам проходить по `target.json`, виконує gates sequential (cheap-first: `requireRule` → `requireFileExists` → `requireFileInTree`) і викликає `runConftestBatch`. `lint-conftest.mjs:TARGETS` замінюється читанням `target.json` замість жорсткого масиву. Правила з динамічними батчами (`abie` з `collectDeploymentDirs`) лишають `check.mjs` і не мігрують.
**Обґрунтування:** Усуває дублювання між `lint-conftest.mjs:TARGETS` і `runConftestBatch`-викликами в `check.mjs` — одне джерело правди для таргетування полісі. Дозволяє pure-Rego правила без JS-обгортки. Гейти cheap-first мінімізують непотрібні `walkDir`-обходи.
**Розглянуті альтернативи:** Frontmatter у `<id>.mdc` (відкинуто — `.mdc` копіюється в проєкт, метадані CLI не повинні жити там); авто-discover через `input.kind` у Rego (відкинуто — feed усім YAML марнує CPU і не підходить для не-YAML цілей); залишити `check.mjs` обов'язковим (статус-кво, не усуває дублювання).
**Зачіпає:** `npm/bin/n-cursor.js` (функція `discoverCheckScripts`), `npm/scripts/lint-conftest.mjs` (заміна `TARGETS`), `npm/scripts/utils/run-conftest-batch.mjs`, усі `npm/rules/*/policy/` (нові `target.json`).
