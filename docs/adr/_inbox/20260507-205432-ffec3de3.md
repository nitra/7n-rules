## ADR Ігнорування `**/adr/**` у oxfmt, cspell та мінімум `@nitra/eslint-config` ^3.9.2

**Контекст:** У директорії `docs/adr/` зберігаються автоматично згенеровані Markdown-файли з рішеннями (ADR). Вони мають довільний стиль, власну термінологію та не потребують форматування oxfmt, спелчекінгу cspell чи ESLint-правил; запуск лінтерів на них давав хибні помилки.

**Рішення/Процедура/Факт:**

- `.oxfmtrc.json` → `ignorePatterns` доповнено глобом `**/adr/**`
- `.cspell.json` → `ignorePaths` доповнено глобом `**/adr/**`
- `npm/mdc/js-lint.mdc` (v1.16 → v1.17), `.cursor/rules/n-js-lint.mdc`, `npm/scripts/check-js-lint.mjs` — мінімум `@nitra/eslint-config` піднято з `^3.8.0` до `^3.9.2`; у `nitraEslintConfigMeetsMinVersion` оновлено порогове порівняння, JSDoc і pass/fail-повідомлення
- `npm/tests/check-js-lint.test.mjs` — кейси перебудовано під новий мінімум: `^3.9.2`/`^3.9.10`/`^3.10.0`/`^4.0.0` → ok; `^3.9.1`/`^3.8.0`/`^3.6.12` → fail
- `package.json` + `bun.lock` — встановлено `@nitra/eslint-config@3.9.2`
- `npm/CHANGELOG.md` — додано записи до `[1.8.203]`

**Обґрунтування:** Починаючи з версії 3.9.2 функція `getConfig` у `@nitra/eslint-config` вбудовує ігнорування `**/adr/**` автоматично, тому примусово додавати цей glob локально в кожному проєкті не потрібно. Підняття мінімальної версії у правилі `js-lint` гарантує, що всі проєкти, які слідують правилу, отримають ADR-ignore без ручних правок. Для oxfmt і cspell аналогічного механізму немає, тому glob додано безпосередньо в конфіги.

**Розглянуті альтернативи:** `markdownlint-cli2` — не обговорювалася (користувач обрав лише oxfmt і cspell); додавання glob у кожному проєкті вручну — відхилено як надмірно повторювальне.

**Зачіпає:** `.oxfmtrc.json`, `.cspell.json`, `npm/mdc/js-lint.mdc`, `.cursor/rules/n-js-lint.mdc`, `npm/scripts/check-js-lint.mjs`, `npm/tests/check-js-lint.test.mjs`, `package.json`, `bun.lock`, `npm/CHANGELOG.md`

## ADR Заміна `process.env.*` на `env` з `node:process` у `bin/n-cursor.js`

**Контекст:** Правило `js-run.mdc` забороняє прямий доступ `process.env.*` у CLI-скриптах — усі змінні середовища мають бути отримані через деструктуроване `env` з `node:process` (опційні) або `@nitra/check-env` (обов'язкові). Інтеграційний тест `integration-repo-checks.test.mjs` виявив порушення на рядку 1136 у `bin/n-cursor.js`: `process.env.NITRA_CURSOR_REEXEC`.

**Рішення/Процедура/Факт:**

- `npm/bin/n-cursor.js:56` — `import { cwd } from 'node:process'` розширено до `import { cwd, env } from 'node:process'`
- Два місця використання `process.env.NITRA_CURSOR_REEXEC` замінено на `env.NITRA_CURSOR_REEXEC` (перевірка re-exec guard та передача в `spawnSync`)

**Обґрунтування:** `NITRA_CURSOR_REEXEC` — внутрішня опційна змінна самого CLI (ознака повторного запуску після self-upgrade); вона не є обов'язковою для запуску, тому `node:process` достатньо, без `checkEnv`. Зміна усунула єдине падіння в інтеграційних тестах (674 pass, 1 fail — несповіднє падіння `check-ga: shellcheck в PATH`, не пов'язане з цією правкою).

**Розглянуті альтернативи:** `// @nitra/cursor ignore-next-line` — відхилено; проблема усувається правильним імпортом, а не придушенням перевірки.

**Зачіпає:** `npm/bin/n-cursor.js` (рядки 56, ~1136, ~1160)
