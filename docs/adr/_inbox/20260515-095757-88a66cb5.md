## ADR Декларативний контракт `rule.json` для pure-Rego правил без `js/check.mjs`

**Контекст:** Поточна архітектура CLI (`npm/bin/n-cursor.js`) обовʼязково потребує `rules/<id>/js/check.mjs` — `discoverCheckScripts()` фільтрує правила саме за його наявністю. Правила, у яких є тільки `policy/*.rego` без JS-оркестратора, залишаються «німими»: `check` їх не підхоплює без попередження.

**Рішення/Процедура/Факт:**
Ввести `npm/rules/<id>/rule.json` — декларативний маніфест правила — і навчити `discoverCheckScripts()` підхоплювати такий каталог навіть без `js/check.mjs`. Файл містить два блоки:

1. **`applies`** — rule-level гейт (AND-комбінація):
- `requireRule: <id|id[]>` — перевірити `.n-cursor.json:rules`; default = id самого правила; `null` = завжди увімкнено
- `requireFileExists: <path|path[]>` — `existsSync` відносно cwd
- `requireFilesByGlob: <glob|glob[]>` — `walkDir` від cwd, зупиняється на першому матчі

2. **`targets[]`** — перелік polisі з резолвером файлів:
- `single: <path>` — один файл; `required: true/false` керує `fail` vs silent-skip при відсутності
- `walkGlob: <glob>` — батч файлів через `walkDir`, матч фільтрує список для одного `spawnSync conftest`
- `conftest.combine: true` — передати `--combine` у conftest (для полісі, що розглядають кілька документів разом)

Поведінка CLI при `policy` + `js`:
- CLI прогонить **декларовані** targets (`rule.json`) сам
- `js/check.mjs` залишається лише для **обчислюваних** батчів (зібраних через `walkDir`+`collectDeploymentDirs`) і non-Rego перевірок
- JS більше не викликає `runConftestBatch` для статично-оголошених targets → дублювання відсутнє

`lint-conftest.mjs:TARGETS` стає похідною: скрипт читає всі `rule.json` динамічно замість hard-coded таблиці (рядки 96–373).

**Обґрунтування:** Прагматична межа між декларативним і імперативним: `rule.json` покриває 90% випадків (`single file`, `walkGlob`); все, що вимагає cross-file залежностей між результатами (`collectDeploymentDirs` → `hc.yaml`-батч) або autofix-логіки, лишається в `js/check.mjs`. Вводити DSL для цих випадків означало б відтворити частину Rego всередині JSON.

**Розглянуті альтернативи:**
- YAML-frontmatter у `<id>.mdc` — відхилено, бо `.mdc` копіюється у проєкти і дублює метадані там, де вони зайві
- Auto-discover через `input.kind`-гейти (feed conftest усіма YAML/JSON у дереві) — відхилено через марнування CPU і неможливість для не-YAML цілей
- Залишити обовʼязковим `js/check.mjs` скрізь — відхилено, бо для pure-Rego правил це boilerplate без реальної логіки

**Зачіпає:** `npm/bin/n-cursor.js` (`discoverCheckScripts`, `runChecks`), `npm/scripts/lint-conftest.mjs` (динамічне читання `rule.json` замість hard-coded `TARGETS`), `npm/rules/*/rule.json` (нові файли для кожного правила), `npm/schemas/rule.json` (JSON-схема для валідації), `npm/scripts/utils/run-conftest-batch.mjs` (без змін API, але CLI тепер викликає напряму)

## Knowledge Ланцюжок виконання правила `@nitra/cursor check` — від `/n-fix` до `conftest`

**Контекст:** Архітектура пакета `@nitra/cursor` не очевидна з першого погляду: точка входу `/n-fix` (LLM skill) → CLI bin → динамічний `import()` → Rego через `spawnSync`. Неочевидне місце — фільтр правил іде через `AGENTS.md`, а не `.n-cursor.json`.

**Рішення/Процедура/Факт:**
1. `/n-fix` — LLM-skill (`.cursor/skills/n-fix/SKILL.md`), перший крок якого: `npx @nitra/cursor check`
2. `npm/bin/n-cursor.js` → `runChecks([])`:
- `discoverCheckScripts()` — сканує `rules/*/js/check.mjs`
- `discoverCheckRulesFromAgentsMd(available)` — читає кореневий `AGENTS.md`, виловлює `rules/<id>.mdc`-посилання, фільтрує по `available` → **фактичний список правил для прогону**
- Для кожного: `await import('rules/<id>/js/check.mjs').check()`
3. `check.mjs` — сам перевіряє `.n-cursor.json:rules` (self-guard), потім суміш JS-перевірок і `runConftestBatch`
4. `runConftestBatch` → `spawnSync('conftest', ['test', ...files, '-p', policyAbs, '--namespace', ns, '--output', 'json'])` → парсинг `deny`-повідомлень → `fail(msg)` у репортері

Критичний інваріант: фільтрація іде через `AGENTS.md`, не `.n-cursor.json`. Якщо `AGENTS.md` не пересинхронізований після додавання правила у `.n-cursor.json` — `check` його **не запустить** без помилки.

**Обґрунтування:** Роздільність concern: `.n-cursor.json` — конфіг для синхронізатора; `AGENTS.md` — декларація активних правил для агентів і CLI. `discoverCheckRulesFromAgentsMd` гарантує, що те, що задокументовано для агентів, і те, що перевіряється CLI, — завжди одне і те саме.

**Розглянуті альтернативи:** не обговорювалися

**Зачіпає:** `npm/bin/n-cursor.js` (`discoverCheckScripts`, `discoverCheckRulesFromAgentsMd`, `runChecks`), `npm/scripts/utils/run-conftest-batch.mjs`, `npm/rules/*/js/check.mjs`, `npm/rules/*/policy/*/`
