Для **map-ключів** (наприклад `apiVersion`, `kind`, `metadata`, `spec`) — так, YAML map не має порядку, алфавітне сортування безпечне.

Але для **sequences (масивів)** — ні, порядок семантично значущий у багатьох місцях:

| Масив | Чи можна сортувати? |
|---|---|
| `patches[].target` (kind→name) | Так — ваш випадок |
| `resources:`, `images:`, `components:` | Так — рядки, unordered |
| `patch: |-` JSON6902 ops | **Ні** — RFC 6902, порядок важливий |
| `initContainers:` | **Ні** — виконуються послідовно |
| `containers[].command/args` | **Ні** — це argv |
| `containers[].env` | Так — але вже є своє правило |
| `rules:` в RBAC | Так |

Тобто «сортувати всі масиви» = «зламати `initContainers`, `command`, `args`, JSON6902 patches».

**Рекомендую точковий підхід:**

- `patches[]` — сортуємо за `kind → name` (як просив)
- `patch: |-` inline ops — **не сортуємо** (семантика RFC 6902), тільки якщо захочеш окреме правило з обмеженнями
- `images:` / `components:` / `crds:` — так, можна додати в той самий пас

Погоджуєшся з `patches[]` без `patch: |-` ops на старт?

## ADR Структурне сортування `patches[]` у Kustomize (check-k8s)

**Контекст:** У `kustomization.yaml` масив `patches:` не мав обовʼязкового порядку — однакові за суттю конфіги відрізнялися порядком елементів між авторами, що ускладнювало diff-рев'ю.

**Рішення/Процедура/Факт:** Додати до `check-k8s.mjs` нову перевірку: масив `patches[]` повинен бути відсортований спочатку за `target.kind`, потім за `target.name` (en, case-insensitive, `localeCompare`). Inline JSON6902 ops у `patch: |-` — **не сортуються**, оскільки RFC 6902 визначає семантично значущий порядок операцій. Перевірка виводить «очікуваний» порядок аналогічно до наявної `kustomizationResourcesSortedAlphabeticallyViolation`.

**Обґрунтування:** Алфавітний сорт на рівні масивів загалом небезпечний — `initContainers`, `command/args`, JSON6902 ops мають семантичний порядок. Точкова перевірка лише `patches[].target` (kind→name) безпечна, бо `target` ідентифікує ціль, а не визначає порядок застосування. Обмеження на поля `kind`+`name` (без `group`/`version`) вже закладено в існуючих правилах проєкту.

**Розглянуті альтернативи:** Сортувати всі масиви на одному рівні YAML алфавітно — відхилено через ризик зламати `initContainers`, `command`, `args`, JSON6902 `patch` ops.

**Зачіпає:** `npm/scripts/check-k8s.mjs`, `npm/mdc/k8s.mdc`, тести `npm/tests/auto-rules.test.mjs` (якщо є fixtures для kustomization).
