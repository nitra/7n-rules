## ADR Обов'язкові CHANGELOG.md у кожному workspace (кореневий + demo)

**Контекст:** Правило `n-changelog` вимагає наявності `CHANGELOG.md` у кожному workspace з `package.json.workspaces` (включно з кореневим пакетом). Після додавання `demo` до `workspaces` обидва файли — `CHANGELOG.md` (корінь) і `demo/CHANGELOG.md` — були відсутні, і `npx @nitra/cursor check changelog` падав з `❌`.

**Рішення/Процедура/Факт:**
- Підвищено версію кореневого пакету: `1.0.0` → `1.0.1` у `package.json`.
- Створено `CHANGELOG.md` із записом `[1.0.1] - 2026-05-09` (перелічено зміни гілки відносно `dev`: скрипти лінту, ADR-процес, `.cursor/rules/`, `demo`-workspace, `.claude/`-хуки).
- Створено `demo/CHANGELOG.md` з початковим записом `[0.0.0] - 2026-05-09` (`demo` — пісочниця без публікації, `private: true`).
- При написанні CHANGELOG уникнуто слів `rego`/`conftest` у тексті (вони тоді ще не були в словнику), і перевірено `cspell CHANGELOG.md demo/CHANGELOG.md` → 0 issues.

**Обґрунтування:** Правило `n-changelog` є `alwaysApply: true`; кожен workspace повинен мати власний CHANGELOG, щоб зміни були відстежені незалежно від npm-публікації. `demo` — private workspace і також підпадає під правило.

**Розглянуті альтернативи:** Виключити `demo` з перевірки — не обговорювалося; правило явно охоплює всі workspace без винятків.

**Зачіпає:** `package.json` (version bump), `CHANGELOG.md` (новий файл), `demo/CHANGELOG.md` (новий файл)
