## ADR Правило `n-changelog`: PR-scoped перевірка CHANGELOG у Bun-монорепо

**Контекст:** Існуюче правило `npm-module` вимагало оновлення `CHANGELOG.md` для workspace `npm/`, але інші workspace-и монорепо не мали аналогічного захисту. Потрібно було поширити цю вимогу на всі workspace-и проекту, включно з тим, щоб підхід відображав реальний workflow (один bump + один запис на PR, а не на кожен коміт).

**Рішення/Процедура/Факт:**

- Створено `npm/scripts/check-changelog.mjs` і `npm/mdc/changelog.mdc` / `.cursor/rules/n-changelog.mdc`.
- Алгоритм — **PR-scoped**: базова гілка `dev` хардкодом; для кожного workspace визначається `git diff <merge-base dev HEAD> -- <ws>/` (з `:(exclude)` підворкспейсів для root); якщо є зміни — вимагається bump `version` відносно `dev` і запис `## [version] - YYYY-MM-DD` у `<ws>/CHANGELOG.md`. Для workspace-ів із `package.json.files` також перевіряється, що `"CHANGELOG.md"` є у масиві `files`.
- Skip-логіка: не git-репо; поточна гілка = `dev`; refs `dev` / `origin/dev` не існують — перевірка тихо пропускається.
- Changelog-логіка **перенесена з `check-npm-module.mjs`** у `check-changelog.mjs`: `npm/` більше не є винятком, перевіряється поряд з іншими workspace-ами. З `check-npm-module.mjs` видалено `checkChangelog()` та константу `CHANGELOG_PATH`.
- `npm-module.mdc` / `n-npm-module.mdc` підняті до версії `1.9` зі вилученою секцією `## CHANGELOG`.
- `changelog` авто-вмикається за умовою `[bun]` через `AUTO_RULE_DEPENDENCIES`.
- Додано 11 тестів (`npm/tests/check-changelog.test.mjs`), включно з кейсом на `files`-перевірку; усі 563 тести пакету проходять.
- `npm/.claude-template/npm-CLAUDE.md` і `npm/CLAUDE.md` оновлені: замість «перед комітом» — «перед PR».

**Обґрунтування:** PR — природна одиниця роботи: розробник може робити скільки завгодно комітів, але обов'язок фіксувати зміни у CHANGELOG виникає рівно один раз — при відкритті PR у `dev`. Відповідно, тригером є не факт коміту, а різниця між гілкою фічі та `dev`. Базова гілка вибрана `dev` (хардкод), а не `main`, бо в реальному workflow PR і git-стратегія орієнтовані на `dev`. Об'єднання логіки з `npm-module` в одне правило усуває дублювання і дає єдину точку відповідальності.

**Розглянуті альтернативи:**

- **Перевірка відносно HEAD** (порівнювати лише останній коміт) — відхилено: не відповідає PR-семантиці, вимагала б запису на кожен коміт.
- **Вимагати запис лише для публікованих (не-private) workspace-ів** — відхилено: проекти все одно приватні; changelog для внутрішнього використання; перевіряти варто всі.
- **Конфігурований базовий ref** через `.n-cursor.json` — відкладено: більшість проектів мають `dev`; можна додати пізніше.
- **Залишити `npm/` як виняток** (перша ітерація) — переглянуто і відхилено у другій ітерації: краще мати одне правило для всіх workspace-ів.
