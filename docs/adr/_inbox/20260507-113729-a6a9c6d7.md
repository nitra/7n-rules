## ADR Розширення резолвера шляхів зображень у `check-image.mjs` для Vite/Quasar-проєктів

**Контекст:** У Vite- та Quasar-проєктах `src="/foo/bar.png"` означає `<pkgRoot>/public/foo/bar.png`, а не `<cwd>/foo/bar.png`. Голі відносні шляхи на кшталт `assets/images/x.png` (без `./`) трактуються HTML/Vue як відносні до файла-джерела. Попередня реалізація `resolveImagePath` давала хибний результат для обох форм: `/`-шляхи вказували на `<cwd>`, голі — повертали `null` і мовчки пропускалися rewrite-ом, проте AVIF для них видалявся cleanup-ом як «сирота».

**Рішення/Процедура/Факт:** Функцію `resolveImagePath` переписано: абсолютні (`/`-) шляхи тепер пробуються у порядку `<pkgRoot>/public<path>` → `<pkgRoot><path>` → `<cwd><path>` (перший, де існує `.avif`-двійник); голі відносні (містять `/`) резолвуються відносно директорії файла-джерела; bare module specifiers (без жодного `/`) ігноруються — `null`. `cleanupOrphanAvifs` розширено списком ігнорованих build-каталогів: `dist`, `build`, `android`, `ios`, `.output`, `.next`, `.nuxt`. Версія пакету: 1.8.193 → 1.8.194.

**Обґрунтування:** Vite/Quasar резолвують `/`-шляхи з `public/` — це специфіка фреймворку, а не CWD. Ігнорування голих відносних шляхів призводило до накопичення `failedRefs` і видалення валідних AVIF-файлів cleanup-ом, що ламало роботу у реальних монорепо (32 ❌ у b2b-репо).

**Розглянуті альтернативи:** (1) Запускати справжній Vite resolver для визначення шляху — надто важка залежність для lint-скрипта; (2) Вимагати лише `./`-форми у `.vue` — ламає Quasar-конвенцію і потребувало б рефакторингу в цільових проєктах.

**Зачіпає:** `npm/scripts/check-image.mjs` (`resolveImagePath`, `cleanupOrphanAvifs`), `npm/tests/check-image.test.mjs` (4 нові тести), `npm/package.json` (1.8.194), `npm/CHANGELOG.md`.
