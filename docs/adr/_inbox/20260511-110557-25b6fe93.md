## ADR `k8s.mdc`: modeline `$schema` є опційним; `file:…` заборонено як заглушка
**Контекст:** Механізм автовиправлення у `@nitra/cursor` додавав рядок `# yaml-language-server: $schema=file:.` до k8s YAML-файлів, коли публічна схема для даного `apiVersion`/`kind` не існує. Це створювало видимість валідації без реального ефекту.
**Рішення/Процедура/Факт:** У `npm/mdc/k8s.mdc` (v1.28→v1.29) та `npm/scripts/check-k8s.mjs` змінено правило: modeline `# yaml-language-server: $schema=…` обов'язковий лише для ресурсів, що мають публічну схему (yannh, datree, kustomization). За відсутності схеми файл залишається без modeline — відсутність рядка є коректним станом. Будь-який `$schema=file:…` тепер реєструється як помилка з підказкою прибрати рядок. `lint-k8s` (kubeconform з `--ignore-missing-schemas`) продовжує покривати валідацію таких файлів. Версія пакету підвищена 1.9.1 → 1.9.2.
**Обґрунтування:** `$schema=file:.` є семантично хибним (не вказує на реальну схему), вводить в оману редактор і порушує принцип «fail-fast» — помилка у конфігурації мусить бути видима, а не замаскована заглушкою.
**Розглянуті альтернативи:** Додавати коментар `# no-schema` замість `file:…` — відхилено через надлишковість; залишати `file:` як дозволений варіант — відхилено явно.
**Зачіпає:** `npm/mdc/k8s.mdc`, `npm/scripts/check-k8s.mjs` (JSDoc + `checkK8sYamlFile` + перевірка `file:` схеми), `npm/CHANGELOG.md`, `npm/package.json`

## ADR `pathHasK8sSegment`: перевірка сегмента `k8s` відносно кореня репо, а не абсолютного шляху
**Контекст:** Якщо кореневий каталог репозиторію містить сегмент `k8s` у своєму абсолютному шляху (наприклад `/Users/.../abie/k8s/`), функція `pathHasK8sSegment` повертала `true` для **всіх** файлів проєкту — включно з `.github/workflows/apply-k8s.yml`. Це спричиняло false-positive: `check-k8s.mjs` вимагав перейменувати `.yml` файли GitHub Actions на `.yaml`, конфліктуючи з правилом `ga.mdc`.
**Рішення/Процедура/Факт:** Сигнатуру `pathHasK8sSegment` у `check-k8s.mjs` та `run-k8s.mjs` змінено на `pathHasK8sSegment(filePath, root?)` — коли `root` передано, шлях relativize-ується через `path.relative(root, filePath)` перед пошуком компонента `k8s`. У `findK8sYamlFiles` (у трьох файлах) додано передачу `root` + ранній `return` для шляхів під `.github/` (defense-in-depth). У `k8s.mdc` додано параграф про те, що скоп визначається відносно кореня репо і не зачіпає `.github/`. Додано тести worst-case (`/tmp/some/k8s/…`). Версія підвищена 1.9.2 → 1.9.3.
**Обґрунтування:** Семантика «файл є k8s-маніфестом» визначається відносним розташуванням у репо (наявність каталогу `k8s/` у відносному шляху), а не абсолютним шляхом на файловій системі. Без `root`-параметра ім'я репозиторію забруднює пошук сегментів.
**Розглянуті альтернативи:** Додати `return` у `checkK8sYamlFile` для `.github/` — відхилено, бо `findK8sYamlFiles` все одно включало б workflows у списки для rego та kubeconform; виключити `.github/` у `walkDir` ignorePaths — відхилено як надто широке, `walkDir` використовується і в інших перевірках.
**Зачіпає:** `npm/scripts/check-k8s.mjs`, `npm/scripts/run-k8s.mjs`, `npm/scripts/check-abie.mjs` (функції `findK8sYamlFiles`, `findK8sRoots`, `pathHasK8sSegment`), `npm/mdc/k8s.mdc`, `npm/tests/check-k8s-schema.test.mjs`, `npm/tests/run-k8s-roots.test.mjs`, `npm/CHANGELOG.md`, `npm/package.json`
