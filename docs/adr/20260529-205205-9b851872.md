## Дизайн

### 1. Формат change-файлу (`<workspace>/.changes/<unique>.md`)

```markdown
---
type: patch | minor | major | none
---
Людський опис зміни (одним рядком або кількома).
```

Ім'я файлу — **унікальне**, пропоную схему `<timestamp>-<git-branch-slug>.md` (наприклад, `20260529-feat-auth.md`). Дві причини: читабельне, і per-branch slug дає природню гарантію відсутності колізій між worktree-агентами та людьми.

Python vs JS: однаковий формат, `release` детектить тип workspace за наявністю `package.json` (npm) або `pyproject.toml` (Python) і відповідно читає поточну версію та оновлює.

### 2. Команди `n-cursor`

| Команда | Що робить |
|---|---|
| `n-cursor change` | Інтерактивно (або з флагами) створює `<workspace>/.changes/<unique>.md` |
| `n-cursor check changelog` | **Оновлений** гейт: замість «чи є bump+CHANGELOG» → «чи є ≥1 change-файл у `.changes/`» для кожного зміненого workspace (рахує diff від origin/main) |
| `n-cursor release` | Агрегує всі `.changes/*.md` per-workspace, рахує semver-bump (max з усіх type), генерує секцію CHANGELOG, ставить нову версію у `package.json` / `pyproject.toml`, видаляє use-up файли, робить release-commit |

`n-cursor release` **не публікує в npm** — це лишається за `npm publish` у CI (як зараз). `release` тільки: bump + CHANGELOG + git-commit.

**Виглядає правильно?**
