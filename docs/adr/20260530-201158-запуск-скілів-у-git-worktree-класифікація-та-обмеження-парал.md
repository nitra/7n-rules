## ADR Запуск скілів у git worktree — класифікація та обмеження паралелізму

## Context and Problem Statement

Команда має набір скілів (`n-fix`, `n-lint`, `n-coverage-fix`, `n-taze` та ін.), прив'язаних до CWD. Виникло питання: чи доцільно запускати ці скіли у `git worktree`, який ефект це дає й чи є обмеження на паралельний запуск.

## Considered Options

* Запускати скіли лише в головному checkout (без worktree)
* Запускати скіли у worktree для ізоляції правок на окрему гілку
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Запускати скіли у worktree для ізоляції правок на окрему гілку", because worktree — повноцінний checkout зі своїм `package.json`, тож передумови всіх скілів у ньому виконуються; ізоляція правок безкоштовна. Водночас worktree **не обходить** заборону паралельного eslint/CPU-важкого прогону: усі worktree читають з того ж диска й CPU, тому CPU-важкі скіли (`n-lint`, `n-coverage-fix`, `n-fix-tests`, `n-taze`) виконуються **по одному за раз** на всій машині.

### Consequences

* Good, because правки структури, лінту, тестів, залежностей ізольовані на власній гілці й не торкаються головного checkout.
* Good, because `n-taze` виграє найбільше: worktree автоматично задовольняє його передумову «чисте робоче дерево», а dep-bump природно живе на окремій гілці.
* Good, because легкі скіли (`n-fix`, `mdc-check`) безпечно паралелити в кількох worktree одночасно (з застереженням: `mdc-check` бампає `npm/package.json` — конфлікт версій при декількох паралельних worktree).
* Bad, because CPU-важкі скіли (`n-lint`, `n-coverage-fix`, `n-fix-tests`, `n-taze`) залишаються під забороною паралельного запуску з `CLAUDE.md` — worktree не дає можливості ганяти їх одночасно.
* Bad, because `n-start-check` не виграє від worktree (нема file-ізоляції), а паралельний запуск ризикує конфліктом портів.
* Bad, because `n-llm-patch` і `n-publish-telegram` read-only, тому worktree безглуздий — нічого ізолювати, лише зайвий `bun i`.
