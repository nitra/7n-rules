## ADR Систематичне видалення нерезультативного тексту з MDC-правил

**Контекст:** MDC-правила у `npm/rules/` містили пояснювальні блоки, що не впливали на поведінку AI або check-скриптів, але збільшували розмір контексту підвантаженого в Cursor/Claude Code при кожному редагуванні файлів під `globs:`.

**Рішення/Процедура/Факт:** Проведено аудит усіх 27 файлів `npm/rules/*/*.mdc` через паралельні агенти. Із 23 файлів видалено ~314 рядків (7,6% загального обсягу) за шістьма патернами: секції `## Перевірка` з командою `npx @nitra/cursor check <id>` (18 файлів), блоки `.vscode/extensions.json` і `.vscode/settings.json` (8 файлів), інструкції встановлення CLI через `brew`/`apt-get` (5 файлів), описи того що робить зовнішній інструмент (hadolint, gitleaks, kubeconform, kubescape), активаційна мета про `disable-rules`/`alwaysApply`, дублікати, міграційний контекст старих версій, cross-references без директивного впливу. Пропущені: `vue.mdc`, `php.mdc`, `text.mdc` — містять ризикові суперечності або блоки, що можуть бути частиною канону check-скриптів.

**Обґрунтування:** `.mdc` читається AI як директивний документ: лише те, що змінює поведінку при генерації коду, має залишатися. Секція `## Перевірка` і VS Code-блоки — інформація для людини-оператора, а не для AI; `lint-<id>` і `check-<id>` запускаються через фіксовану файлову структуру `fix/`/`policy/`/`lint/`, а не через вміст `.mdc`. Видалення скорочує токен-бюджет підвантаженого правила.

**Розглянуті альтернативи:** Перенести "Перевірка" у CLAUDE.md або README — відхилено, бо не скорочує токен-бюджет. Лишити без змін — явний перевитрата контексту.

**Зачіпає:** `npm/rules/*/rego/*.mdc`, `adr`, `bun`, `capacitor`, `changelog`, `ci4`, `docker`, `ga`, `graphql`, `hasura`, `image-avif`, `image-compress`, `js-bun-db`, `js-bun-redis`, `js-lint`, `js-mssql`, `js-run`, `k8s`, `nginx-default-tpl`, `npm-module`, `security`, `style-lint`, `tauri`, `abie`

## ADR Архітектура `template/` у concern-директоріях MDC-правил

**Контекст:** Canon-фрагменти (expected content для `package.json`, `.gitleaks.toml`, workflow YAML) дублювалися між `.mdc` (AI-контекст) і `fix/<concern>/check.mjs` / `policy/<concern>/*.rego` (програмні перевірки) як окремі hardcoded рядки. Зміна канону вимагала правки в обох місцях.

**Рішення/Процедура/Факт:** Проектне рішення (не реалізовано, є spec + plan): додати `template/` до кожного concern у `policy/<concern>/template/` (canonical) або `fix/<concern>/template/` (fix-only concerns). Три файли: `snippet.json` (required fragments — leaf-рівень = exact-match), `deny.json` (forbidden paths — наявність = fail), `contains.json` (substring-перевірки для рядкових полів). Template завжди в JSON незалежно від формату target-файлу. При наявності і `policy/`, і `fix/` для того самого концерну — canonical у `policy/`, JS-check читає через relative path. Rego отримує template через `conftest --data`. У `.mdc` snippet замінюється посиланням на файл. Spec: `docs/superpowers/specs/2026-05-17-template-dir-design.md`. Plan: `docs/superpowers/plans/2026-05-17-template-dir-plan.md`.

**Обґрунтування:** Rule-level layout (один `template/` на правило) відхилено після інвентаризації: у `k8s` — 4 концерни на `*.yaml` glob, у `npm-module` — 2 концерни на `package.json`; однаковий basename у різних концернах → collision. Concern-level ізолює очікування кожного концерну незалежно від збігу цільових файлів. `conftest --data` — стандартний механізм Rego для зовнішніх даних, без зміни API.

**Розглянуті альтернативи:** Rule-level `template/` — відхилено через колізії basename серед концернів. Expectations DSL у YAML — відхилено як новий DSL що потребує навчання AI і окремого парсера. Snippet-only без `deny`/`contains` — відхилено як неповне: не покриває substring-перевірки `lint`-скрипту.

**Зачіпає:** `npm/rules/*/policy/*/template/`, `npm/rules/*/fix/*/template/`, `npm/scripts/` (template-loader утиліта, conftest runner), `npm/rules/*/*.mdc` (заміна snippet-блоків на посилання), 77 `target.json` (65 `single`, 12 `walkGlob`)
