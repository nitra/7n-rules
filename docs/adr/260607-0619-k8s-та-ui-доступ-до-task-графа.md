type: ADR
title: K8s та UI-доступ до task-графа
description: Task-граф виконується у Kubernetes, а UI читає стан через API на базі graph scan --json замість прямого монтування tasks.
---

**Status:** Accepted
**Date:** 2026-06-07

## Context and Problem Statement

Після введення файлової структури `tasks/` потрібно визначити середовище виконання задач і спосіб доступу UI до стану графа. Transcript фіксує вимогу, що задачі житимуть у Kubernetes, а UI має переглядати стан задач і цикл виконання.

## Considered Options

- Dev pod, що монтує той самий PVC, що й worker-поди.
- `kubectl port-forward` + SSH у поді.
- Tailscale / WireGuard у поді.
- UI читає стан через `graph scan --json` через REST або SSE.
- Пряме монтування `tasks/` у Zed remote.

## Decision Outcome

Chosen option: "Kubernetes worker/dev pod + UI через REST/SSE на базі `graph scan --json`", because worker-поди можуть працювати з `tasks-pvc`, dev pod дає доступ до живих файлів для розробки, а UI не потребує локального монтування `tasks/` і може читати стан через API-сервер у k8s.

### Consequences

- Good, because `tasks-pvc` одночасно доступний worker-подам і dev pod, тож Zed remote бачить живий стан файлів задач.
- Good, because UI не залежить від Zed remote і прямого доступу до файлової системи; достатньо HTTP/SSE до API-сервера.
- Bad, because transcript не містить підтверджених негативних наслідків.
- Neutral, because остаточну назву UI-проєкту transcript не підтверджує.

## More Information

Запропонована команда доступу до dev pod: `kubectl port-forward pod/n-graph-dev 2222:22`. Dev pod потребує SSH-сервера та `zed --headless`. Схема з transcript: `tasks-pvc` монтується у `n-cursor-graph` worker pods і dev pod.

UI читає стан через API, що використовує `graph scan --json`; транспорт для live updates — REST або SSE. Запропоновані назви UI: `n-graph`, `graphwatch`, `taskflow`; фінальний вибір у transcript не зафіксований.

## Update 2026-06-07

Уточнено принцип доступу розробників до k8s dev-середовища: розробники не повинні отримувати прямі `kubectl` credentials. Варіант `kubectl port-forward` + SSH у поді відхилено для сценарію розробницького доступу, бо він вимагає прав `kubectl`.

Перевага надається SSH gateway / bastion з backend-авторизацією: backend перевіряє права користувача і лише після цього дозволяє SSH-зʼєднання до dev pod. Розглянуті реалізації: Teleport або власний gateway. Остаточний вибір між ними transcript не фіксує.
