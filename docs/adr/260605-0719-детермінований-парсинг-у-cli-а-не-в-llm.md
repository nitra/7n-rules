# Детермінований парсинг у CLI, а не в LLM

**Status:** Accepted
**Date:** 2026-06-05

## Context and Problem Statement

Кілька скілів (`n-fix-tests`, `n-coverage-fix`, `n-taze` та інші) інструктували LLM-агента зчитувати і парсити великі артефакти: `COVERAGE.md` (~2.76 МБ / ~700K токенів), `package.json.taze-bak`. Агент класифікував результати — знаходив survived-блоки, визначав semver major/minor — там, де можна витратити 0 LLM-токенів. При фан-ауті (122 групи мутантів) це масштабувалось до десятків мільйонів токенів за ітерацію.

## Considered Options

- Залишити агента парсити артефакти безпосередньо (поточний підхід).
- Перенести весь детермінований парсинг у read-only CLI-екстрактор (`n-cursor <skill> index|slice|diff`), залишивши LLM лише когнітивну роботу.

## Decision Outcome

Chosen option: "read-only CLI-екстрактор", because `COVERAGE.md` (~700K токенів) перевищує практичний контекст одного агента, а semver-класифікація та парсинг JSON-блоків — повністю детерміновані операції, що не потребують мовної моделі.

Правило закріплено в `.cursor/rules/scripts.mdc` (v1.13): детермінований парсинг / лічба / класифікація у скілі — вада, яку усуває скрипт; контракт — `n-cursor <skill> <verb>` повертає компактний `index` або `slice` під один елемент.

### Consequences

- Good, because `n-cursor coverage-fix index` повертає 7.4 КБ (~2K токенів) замість 2.76 МБ (~700K) — ≈350× менше для оркестратора; субагент отримує `slice` (~10 КБ) рівно під один файл.
- Good, because `n-cursor taze diff` класифікує major-оновлення детерміновано — LLM отримує готовий список і фокусується лише на CHANGELOG та рефакторингу.
- Good, because патерн відтворюється: `worktree` і `adr-normalize` уже є еталонами; `scripts.mdc` v1.13 містить red-flags і чек-лист перед фіналізацією будь-якого скіла.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Реалізовані CLI-команди:
- `npm/scripts/coverage-fix-extract.mjs` → `n-cursor coverage-fix index|slice`
- `npm/skills/taze/js/diff.mjs` → `n-cursor taze diff`

Обидва підключені в `npm/bin/n-cursor.js` (не root-guarded, read-only).

Змінені SKILL.md: `npm/skills/coverage-fix/SKILL.md` (кроки 2, 3, 5), `npm/skills/taze/SKILL.md` (крок 3, підсумок крок 8); синхронізовані копії в `.cursor/skills/`.

Черга застосування принципу до решти скілів: lint → start-check → fix → llm-patch → docgen → publish-telegram.

## Update 2026-06-05

### `n-cursor taze diff` — semver-класифікація major без LLM

Крок 3 скіла `n-taze` раніше казав агенту вручну порівняти `package.json.taze-bak` і `package.json` по всіх воркспейсах та визначити major-зміни. Замінено на `n-cursor taze diff`, що повертає `{major:[{workspace,pkg,from,to}], minorPatch, totalChanged, comparedWorkspaces}`. Semver-класифікація — детермінований алгоритм (caret-семантика: `1.x→2.x`, `0.4.x→0.5.x`, `0.0.3→0.0.4`); LLM займається лише читанням CHANGELOG breaking-changes і рефакторингом коду. Реалізація: `npm/skills/taze/js/diff.mjs` (функції `parseVersion` з anchor-regex проти `workspace:`-префіксів, `isBreaking`, `diffPackageJson`, `collectTazeDiff`); 12/12 тестів у `npm/skills/taze/js/tests/diff.test.mjs`. Smoke: root без бекапів воркспейсів → `totalChanged:0, comparedWorkspaces:1`.

### `n-cursor start-check scan|run` — детерміноване виявлення воркспейсів і spawn

Кроки 1–4 скіла `n-start-check` давали агенту інструкції вручну читати `package.json#workspaces`, розгортати glob-патерни, запускати процеси з таймаутом через `perl -e 'alarm'` і парсити лог на маркери готовності. Замінено на: `scan` — повертає `[{workspace, name, hasStart, startCmd, type}]`; `run <ws>` — повертає `{workspace, status, exitCode, ready, firstError, logTail, sideEffects}`. `type` (`server`/`cli`) визначається регекс-класифікацією за маркерами `vite`, `next`, `listen` тощо. Spawn із таймаутом через `spawnSync({timeout})` крос-платформно. Реалізація: `npm/skills/start-check/js/check.mjs` з `spawnImpl` для тестованості; 15/15 тестів. Особливість: початковий `READY_RE` з `\b` не матчив `Local:` (двокрапка не є word-boundary) — баг виявлено і виправлено в тій же сесії.

## Update 2026-06-06

**Відмова від широкого `lint-extract`:** кореневий `lint` — `&&`-ланцюг, що зупиняється на першому фейлі (агент і так бачить один інструмент за раз); більшість тулів (trufflehog, jscpd, knip, cspell, stylelint) не мають JSON-виводу; JSON-режим oxlint/eslint вимагав би повторного прогону — подвійний lint і порушення `withLock`-ланцюга (drift-ризик). Широкий `lint-extract` визнаний шкідливим.

**`n-cursor fix --json` пропускає `ensureHkInstall`:** `--json` — діагностичний read-only режим, не мутує проєкт і не потребує git-хука. Рядок у `npm/bin/n-cursor.js`: `if (!json) ensureHkInstall(hkBin)`. Smoke-тест підтвердив чистий парсабельний JSON у stdout без артефактів hk-інсталяції.

## Update 2026-06-06

Патерн закріплено як секція «🔴 ВИСОКИЙ ПРІОРИТЕТ» у `.cursor/rules/scripts.mdc` v1.13 як обов'язковий чекліст перед фіналізацією будь-якого скіла. Version-skew увага: нові підкоманди (`coverage-fix`, `taze`, `start-check`, `fix --json`) активуються для споживачів лише після публікації `@nitra/cursor`; до релізу між SKILL.md і встановленою копією — розходження версій (3.24.0 local vs 3.20.0 installed). Скіли `worktree` та `adr-normalize` — чисті еталони патерну (без рефактора з нуля).

## Update 2026-06-06

Детальний контракт CLI-команд: `n-cursor coverage-fix index` → `[{file, mutants}]`; `slice --file` → промпт під один файл; `n-cursor taze diff` → `{major, minorPatch, totalChanged, comparedWorkspaces}`; `n-cursor start-check scan` → `[{workspace, name, hasStart, startCmd, type}]`; `run <ws>` → `{type, exitCode, timedOut, status, ready, firstError, logTail, sideEffects}`; `n-cursor fix --json` → JSON зі скоупованим per-rule виводом. Скіл `n-lint` — неприйнятний для патерну: `&&`-ланцюг ~10 тулів без єдиного JSON-формату, human-in-the-loop гейти, ризик подвійного прогону важких тулів.

## Update 2026-06-06

Конкретні реалізації патерну (сесія 38aa0305, `scripts.mdc` v1.13→v1.14):

- `n-cursor taze diff` (`npm/skills/taze/js/diff.mjs`) — semver-diff і класифікація major/minor/patch; агент отримує `{major:[{workspace,pkg,from,to}], minorPatch, totalChanged}` замість ручного порівняння `package.json.taze-bak`.
- `n-cursor start-check scan|run` (`npm/skills/start-check/js/check.mjs`) — розгортання glob-воркспейсів, крос-платформний `spawnSync({timeout})` замість `perl alarm`, regex готовності `/(ready(?![\w-])|listening|local:|started|…)/i`, `sideEffects:{newFiles,changedTracked}` для керованого відкату.
- `n-cursor fix --json` — `--json` прапорець на `runFixCommand`; пропускає `ensureHkInstall` у JSON-режимі (щоб не забруднювати stdout).
- `coverage --fix` permissionMode — `permissionMode: 'bypassPermissions'` у `query()` (`npm/scripts/coverage-fix.mjs`); без нього headless SDK-агент не редагував файли (підтверджено пробою: без режиму — файл `(missing)`, з режимом — `WORKER_OK`).

Еталони «чисті від початку» (не потребували рефакторингу): `worktree`, `adr-normalize`. Черга on-demand: `llm-patch`, `docgen filter`, `publish-telegram`.
