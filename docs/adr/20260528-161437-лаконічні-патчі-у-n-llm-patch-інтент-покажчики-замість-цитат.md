## ADR Лаконічні патчі у n-llm-patch: інтент + покажчики замість цитат + готового коду

## Context and Problem Statement
Скіл `n-llm-patch` генерував надто об'ємні промпти: вбудовував повні фрагменти вихідного коду, готові реалізації з псевдокодом, `package.json`, дамп `tree`, альтернативи та самоочевидні acceptance-checks. Приклад — патч із задачею coverage для monorepo зайняв сотні рядків, хоча цільова LLM має доступ до CWD і прочитає актуальний код сама.

## Considered Options
* Залишити поточний формат (обсяг ~200-500 рядків, вбудовані цитати коду)
* Переробити на формат "інтент + покажчики": тільки ціль, симптом/репро, `file:line` рефи, реальні обмеження і як перевірити

## Decision Outcome
Chosen option: "інтент + покажчики без цитат і готового коду", because цільова LLM має CWD і читає актуальний стан репо сама — дублювати код у промпті означає додавати шум і ризик застарілих цитат.

### Consequences
* Good, because transcript фіксує очікувану користь: промпт стає коротшим (30-100 рядків проти 200-500), менш схильним до staleness, і не нав'язує реалізацію цільовому агенту.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Файл: `npm/skills/llm-patch/SKILL.md`
- Нова секція "Що не включати у промпт (cut list)": великі цитати коду, готові імплементації, покрокові підказки з псевдокодом, повний `package.json`, `tree -L 2`, огляд відкинутих альтернатив, самоочевидні acceptance-checks.
- Цільовий обсяг тепер: 30-100 рядків; >150 рядків — сигнал, що цитується зайве.
- Шаблон скорочено до секцій: `Завдання`, `Симптом / репро`, `Точки правки`, `Що треба зробити`, `Обмеження`, `Як перевірити`.

## ADR Stryker Ignore-plugin для Vue `<script setup>` macros

## Context and Problem Statement
Stryker інструментує аргументи Vue compile-time macros (`defineProps`, `defineEmits` тощо) у `<script setup>` SFC, вставляючи `stryMutAct_9fa48(...)`. Vue compiler-sfc забороняє нестатичні вирази у цих macros і падає з помилкою `defineProps() in <script setup> cannot reference locally declared variables`. Без системного рішення кожен проєкт-споживач мав би вручну писати `// Stryker disable next-line` навколо кожного macro-виклику.

## Considered Options
* Stryker `PluginKind.Ignore`-плагін `vue-macros-ignorer.mjs`, що постачається концерном `stryker_config`
* Inline `// Stryker disable next-line` коментарі у кожному SFC вручну
* Composables-only підхід (винести логіку з macros у composables — macros лишаються порожніми)

## Decision Outcome
Chosen option: "Stryker `PluginKind.Ignore`-плагін", because він усуває проблему на рівні інфраструктури без boilerplate у кожному `.vue` файлі і без зміни архітектури компонентів споживача.

### Consequences
* Good, because transcript фіксує очікувану користь: нульова конфігурація для проєкту-споживача — концерн `stryker_config` автоматично виявляє `.vue` файли через `hasVueSfc(jsRoot)` і постачає плагін.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Новий файл плагіна: `rules/test/js/data/stryker_plugins/vue-macros-ignorer.mjs` (або аналогічний шлях у `npm/`)
- API: `declareValuePlugin(PluginKind.Ignore, 'vue-macros', new VueMacrosIgnorer())` з `@stryker-mutator/api/plugin`
- Macros у скоупі: `defineProps`, `defineEmits`, `defineModel`, `defineSlots`, `defineExpose`, `defineOptions`
- ADR задокументовано в `docs/adr/`; CHANGELOG bump до `1.29.0` (minor)

## ADR Два окремих baseline-файли для `stryker.config.mjs` (Vue vs non-Vue)

## Context and Problem Statement
Концерн `stryker_config` має idempotently постачати `stryker.config.mjs` у кожен JS-root. Vue-workspaces потребують додаткових секцій `plugins` та `ignorers` для реєстрації `vue-macros-ignorer.mjs`, але non-Vue workspaces мають діставати базовий конфіг без цих секцій.

## Considered Options
* Два окремих baseline-файли: `stryker.config.baseline.mjs` (базовий) і `stryker.config.vue.baseline.mjs` (з `plugins`/`ignorers`) — вибір залежно від `hasVueSfc(jsRoot)`
* AST-патч існуючого `stryker.config.mjs` після створення через `oxc-parser` або `acorn`

## Decision Outcome
Chosen option: "два окремих baseline-файли", because варіант з AST-патчингом потребує парсера як додаткової залежності; два файли — простіше і концерн `stryker_config` лишається idempotent без складної трансформації.

### Consequences
* Good, because transcript фіксує очікувану користь: мінімальне дублювання (один додатковий файл), пряма логіка "Vue-workspace → vue-baseline", backward-compat для non-Vue проєктів гарантована структурно.
* Bad, because зміна спільних полів (наприклад, `tempDirName`, `reporters`) тепер вимагає оновлення двох файлів одночасно.

## More Information
- `rules/test/js/data/stryker_config/stryker.config.baseline.mjs` — базовий (без `plugins`/`ignorers`)
- `rules/test/js/data/stryker_config/stryker.config.vue.baseline.mjs` — Vue-варіант із `plugins: ['@stryker-mutator/vitest-runner', './stryker-vue-macros-ignorer.mjs']` та `ignorers: ['vue-macros']`
- Detection: `hasVueSfc(jsRoot)` — рекурсивний пошук `**/*.vue` у `<jsRoot>/src/**` (skip `node_modules`, `dist`, `reports`)
- Idempotency через `ensureBaselineFile()` — існуючий механізм, не перетирає вже наявний файл
