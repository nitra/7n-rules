## ADR Нова архітектура concern-based структури правил @nitra/cursor

**Контекст:** CLI `npx @nitra/cursor check` виявляє виконувані перевірки лише за наявністю `rules/<id>/js/check.mjs`. Зростаючі правила (передусім `abie` — 1153 рядки в одному файлі) монолітні; pure-Rego правила без JS-обгортки взагалі не підхоплюються CLI; `TARGETS`-таблиця у `lint-conftest.mjs` (рядки 96–373) дублює targeting, що вже є у `check.mjs`-викликах `runConftestBatch`.

**Рішення/Процедура/Факт:**
Перейти на симетричну concern-based структуру:
```
npm/rules/<id>/
├── <id>.mdc
├── auto.md
├── js/<name>/            ← один каталог = один concern
│   ├── check.mjs | check-<x>.mjs   ← запускаються CLI (regex /^check(?:-.+)?\.mjs$/)
│   └── *.test.mjs                  ← пропускаються
├── utils/               ← shared у межах правила (не запускаються)
└── policy/<name>/
├── <name>.rego
├── <name>_test.rego
└── target.json      ← { files: { single | walkGlob, required?, missingMessage? } }
```
CLI (`discoverCheckableRules`) сканує `js/*/` та `policy/*/` незалежно; прогін concern-а: спочатку `policy/<name>/target.json` → `runConftestBatch`, потім `js/<name>/check*.mjs`. Порядок між concern-ами — алфавіт. File-targeting для Rego — через `target.json` поряд із `.rego` (picomatch для `walkGlob`). Кеш обходу файлів — module-level `Map` singleton у `resolveTargetFiles`, щоб `walkDir` не повторювався для однакових glob-патернів. Conditional-гейт — `export async function applies()` у `js/<name>/check.mjs`; CLI викликає його першим і пропускає весь concern при `false`. Shared-хелпери переносяться в `rules/<id>/utils/` (наприклад, `abie/utils/k8s-tree.mjs` із кешованим `findK8sYamlFiles`).

**Обґрунтування:**
- Симетрія `js/<name>/` ↔ `policy/<name>/` дає єдиний словник concern-ів: чи реалізація декларативна, імперативна, або гібридна — ім'я concern-а незмінне.
- `target.json` усуває централізовану `TARGETS`-таблицю (`lint-conftest.mjs`): targeting живе поряд із полісі (locality).
- picomatch — 1 пакет, нуль транзитивних залежностей; компілює glob раз, потім O(n) regex-match — оптимально для нашого walk-і-фільтрувати кейсу. Alternatives: `minimatch` (повільніший), `micromatch` (обгортка над picomatch), `fast-glob` (walker, не matcher).
- Walk-кеш усуває M×N проходів (M правил × N glob-патернів), які інакше виникли б при декількох `walkGlob` з однаковим патерном.
- Інлайн `applies()` у JS замість окремого `rule.json` — менше файлів, гейт живе в тому ж модулі, де є контекст для його обчислення.

**Розглянуті альтернативи:**
- `rule.json` на рівні правила з полями `applies` та `targets` — відкинуто на користь per-policy `target.json` + inline JS-гейту (кращий locality, менше прошарків декларації).
- Централізований `TARGETS` у `lint-conftest.mjs` — поточна модель, відкидається через дублювання з `runConftestBatch`-викликами у `check.mjs`.
- `conftest test .` (feed-everything) — відкинуто: не покриває перевірки «конкретний безкіндовий файл» (`.cspell.json`, `package.json`), N повних обходів репо замість одного, відсутність missing-file-detection.
- Name-convention `policy/package_json/ → годуй package.json` — відкинуто: конфлікт між single vs walk для однойменних полісі різних правил.
- Flat `js/check.mjs` + `js/check-*.mjs` без підкаталогів — відкинуто через асиметрію з `policy/<name>/` і неможливість тримати тести поруч з конкретним concern-ом.

**Зачіпає:** `npm/bin/n-cursor.js` (`discoverCheckScripts` → `discoverCheckableRules`, `runChecks`), `npm/scripts/lint-conftest.mjs` (TARGETS → похідна від `target.json`), `npm/scripts/utils/run-conftest-batch.mjs`, `npm/rules/*/js/`, `npm/rules/*/policy/`, JSON-schema в `npm/schemas/`, тести в `npm/tests/`.
