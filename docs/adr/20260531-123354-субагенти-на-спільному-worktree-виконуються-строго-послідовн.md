## Considered Options
* Паралельний запуск усіх субагентів одним блоком (як у ранньому виклику `Agent(...)` × 9)
* Строго послідовний запуск: один субагент завершує роботу і комітить → запускається наступний

### Consequences

## ADR Squash-merge як рекомендований спосіб завершення worktree-гілки

## Context and Problem Statement
Після виконання Spec B постало питання про стратегію злиття feature-гілки з `main`. Spec B потрапила у `main` через fast-forward merge (фонова сесія), тому в `main` опинились 9 окремих коммітів. Користувач явно зазначив, що хотів би **один коміт** і попросив закріпити це як конвенцію для всіх worktree-гілок.

## Considered Options
* Fast-forward merge — зберігає всі N комітів гілки в `main` (сталося de-facto для Spec B)
* Squash merge (`git merge --squash`) — N комітів стискаються в один
* Merge-commit (`--no-ff`) — N комітів + 1 merge-коміт зверху

## Decision Outcome
Chosen option: "Squash merge (`git merge --squash`)", because Spec B = одна логічна зміна; поетапна TDD-історія гілки не потрібна в `main`; CI-реліз агрегує зміни по `.changes/`-файлах, а не по комітах.

### Consequences
* Good, because transcript фіксує очікувану користь: чистіша лінійна історія `main` — одна фіча = один коміт.
* Bad, because поетапна TDD-/рев'ю-історія гілки не збережена в `main`; доступна лише в самій гілці до її видалення.

## More Information
- Конвенція зафіксована в `npm/rules/worktree/worktree.mdc` (розділ «Завершення гілки worktree»), коміт `41cc767` (`feat(worktree-rule): пропонувати squash-merge при завершенні гілки worktree`)
- Change-файл: `npm/.changes/1780218783124-a30f10.md` (patch bump)
- Набуде сили в проєктах-споживачах після релізу нової версії `@nitra/cursor` (дзеркало `.cursor/rules/n-worktree.mdc` синкається з опублікованого пакета)
- Для Spec B squash не застосовувався (гілка вже змерджена ff до прийняття рішення); user обрав опцію B — залишити 9 комітів як є

## ADR Конвенція назв lint-скриптів: `lint` = quick (змінені файли), `lint-ci` = повний

## Context and Problem Statement
Поточний кореневий `lint`-скрипт запускає повний ланцюг (`lint-ga && lint-js && lint-rego && lint-security && lint-style && lint-text && oxfmt`) по всьому репо. Агент (Claude) і CI потребують різних режимів: швидка перевірка лише по змінених файлах для локальної розробки та повна перевірка для CI/CD.

## Considered Options
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`lint` = швидкий (тільки змінені файли), `lint-ci` = повний (всі файли)", because це інтуїтивна конвенція: звичайне `bun run lint` дає миттєвий feedback; CI явно кличе `lint-ci` і отримує повну перевірку. Запропоновано та підтверджено користувачем.

### Consequences
* Good, because transcript фіксує очікувану користь: агент може запускати `lint` часто без накладних витрат `jscpd`/`knip`/`trufflehog`; CI не втрачає покриття.
* Bad, because Neutral, because transcript не містить підтвердження наслідку — деталі реалізації (data-driven атрибут у `meta.json`, розщеплення `lint-js`) ще в стадії brainstorming; фінальний дизайн не затверджено.

- Поточний `lint-js` = `oxlint --fix && eslint --fix . && jscpd . && knip` — містить mix quick (oxlint/eslint) і ci-only (jscpd/knip) інструментів, що ускладнює атрибут на рівні правила
- Запланований data-driven механізм: атрибут у `rules/<id>/meta.json` визначає, в який lint-режим потрапляє кожне правило
