## ADR забороняє запуск default-sync поза коренем git-репо

## Context and Problem Statement
CLI `@nitra/cursor` (бінар `npm/bin/n-cursor.js`) при запуску без підкоманди (default sync) скаффолдить `.cursor/rules/`, `.cursor/skills/`, `.claude/`, `CLAUDE.md`, `AGENTS.md`, `.n-cursor.json`, `.gitignore` і робить `bun install` у `process.cwd()`. При виклику з піддиректорії git-репо (напряму, не через `bun run start`) ці артефакти записуються не в корінь, а в піддиректорію — непомітна деструктивна помилка.

## Considered Options
* Hard-guard `assertCwdIsProjectRoot` через `git rev-parse --show-toplevel`, до першої мутації
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Hard-guard через `git rev-parse --show-toplevel`", because `bun run start` виявився безпечним (bun скидає cwd на корінь пакета автоматично), але прямий виклик `bun npm/bin/n-cursor.js` з піддиректорії успадковує cwd — що підтверджено тестом; git-toplevel прив'язує до кореня будь-якого репо-споживача, не тільки цього монорепо.

### Consequences
* Good, because guard спрацьовує до перших мутацій (`assertCwdIsProjectRoot` → `STOP` з `exit 1`), перевірено вживу з `npm/bin/`.
* Bad, because транскрипт не містить підтверджених негативних наслідків.

## More Information
- `npm/scripts/lib/assert-project-root.mjs` — новий guard-модуль; поза git-репо (toplevel невідомий) — не блокує.
- `npm/bin/n-cursor.js` — виклик guard'а для `case undefined/''` до `ensureNitraCursorInRootDevDependencies`.
- `npm/scripts/lib/tests/assert-project-root.test.mjs` — 3 тести: корінь → ok, піддиректорія → throw, поза git → ok.

## ADR жорсткий root-guard для деструктивних CLI-команд і скілів

## Context and Problem Statement
Після аналізу всіх скілів виявлено, що 7 з них (`n-taze`, `n-fix`, `n-adr-normalize`, `n-lint`, `n-fix-tests`, `n-coverage-fix`, `docgen`) деструктивно мутують файли у `cwd()`. Усі мають `worktree: true` у `meta.json`, але їхній захист — лише soft-preflight (LLM-інструкція), яку агент може пропустити. Крім того, `worktree-notice.mjs` перевіряв лише «чи ти в `.worktrees/`», але не «чи стартував з кореня», тому старт із піддиректорії основного дерева призводив до збою `cd .worktrees/<…>` з непевним станом.

## Considered Options
* Поширити `assertCwdIsProjectRoot` на деструктивні сабкоманди CLI + підсилити worktree-preflight + додати декларативний флаг `requireRoot` у `meta.json`
* Обмежитись лише CLI-гардом (без meta-флага і без зміни preflight)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Три рівні захисту: CLI hard-guard + підсилений worktree-preflight + `requireRoot` у `meta.json`", because user explicitly requested all three levels and a declarative meta attribute to make the protection status visible per skill.

### Consequences
* Good, because `ROOT_GUARDED_COMMANDS = {default-sync, fix, check, lint, coverage, change, release}` перехоплює деструктивні CLI-шляхи хардом незалежно від LLM; worktree-preflight тепер ловить старт із піддиректорії ще до `cd`; `requireRoot` у `meta.json` дає явну машино- та людиночитану ознаку захисту на рівні метаданих скіла.
* Bad, because транскрипт не містить підтверджених негативних наслідків.

## More Information
- `npm/bin/n-cursor.js` — константа `ROOT_GUARDED_COMMANDS` (Set), функція `describeRootGuardedAction` для контекстного повідомлення, виклик `assertCwdIsProjectRoot` перед `switch`.
- `npm/scripts/lib/worktree-notice.mjs` — доданий Root-assert крок (перевірка `pwd ≠ git toplevel` → `cd <toplevel>` перед `cd .worktrees/<…>`).
- `npm/scripts/lib/root-notice.mjs` — новий модуль `injectRootNotice` для in-place скілів (`worktree:false + requireRoot:true`); `syncSkills` застосовує його паралельно з `injectWorktreeNotice`.

- `npm/rules/npm-module/js/skill_meta.mjs` — валідація: `requireRoot` опційний boolean; `worktree:true + requireRoot:false` → fail.
