## ADR Gemma3:4b як оптимальна локальна модель для docgen Tier 1 на 8 GB M2

## Context and Problem Statement
Tier 1 скіла `n-docgen` покриває понад 1000 файлів і коштує ~26 M хмарних токенів за повний прогін. Розглядалася можливість перекласти цей рівень на локальну Ollama для зниження витрат до нуля.

## Considered Options
* `gemma3:4b` (3.3 GB, 100% GPU на 8 GB)
* `qwen2.5-coder:3b` (2.4 GB, 100% GPU) — найшвидша
* `gemma4:e4b` (9.6 GB — не вміщається у 8 GB)
* `qwen3:4b` — reasoning-модель, ігнорує `/no_think`, відповідає російською
* `qwen2.5:7b`, `qwen2.5-coder:7b`, `llama3.1:8b` (5.1–5.7 GB, CPU-офлоад)
* `llama3.2:3b`, `gemma2:2b`, `phi3.5` — слабка українська

## Decision Outcome
Chosen option: "`gemma3:4b`", because вона єдина в класі ≤4.2 GB, що дає 100% GPU (без свопу), ~14–21 tok/s та якість ~86–87% відносно еталона (найвища серед моделей, що вміщаються у 8 GB); усі 7B+клас впираються в CPU-офлоад і не дають переваги в якості.

### Consequences
* Good, because transcript фіксує очікувану користь: прогін 1042 файлів Tier 1 — ~14–22 год (порівняно з ~37 добами для `gemma4:e4b` через своп).
* Bad, because `qwen2.5-coder:3b` швидша (~36 tok/s, ~9 год) але якість ~77% — на 9 п.п. нижча за `gemma3:4b`, потребує обов'язкового cleanup-проходу.

## More Information
Бенчмарк: `run.py` у `/tmp/docgen-bench`, `run.py` у `/tmp/docgen-bench2` (чистий старт кожної моделі). Метрики: `ollama ps` SIZE + processor split, wall/gen tok/s, якість vs еталон по 4 рубриках (×25%). Модель завантажена на машину командою `ollama pull gemma3:4b`.

## ADR System-prompt як головний важіль якості локальної LLM для docgen

## Context and Problem Statement
Порівняння трьох способів виклику `gemma3:4b` показало несподівано великий розрив у якості. Постало питання: чи pi дає якісну перевагу сам по собі, чи різниця пояснюється іншим чинником.

## Considered Options
* Прямий `ollama /api/generate` без system-prompt
* Виклик через `pi CLI` (додає власний coding-assistant system-prompt)
* Прямий `ollama /api/chat` із явним system-prompt

## Decision Outcome
Chosen option: "прямий `ollama /api/chat` з system-prompt", because контрольований експеримент підтвердив: A (без system) — ~71%, B (pi) — ~87%, C (прямий + system) — ~85%. Різниця A↔(B,C) системна (+14–16 п.п.) і повністю пояснюється наявністю system-prompt, а не транспортом; різниця B↔C — ~2 п.п. (у межах шуму оцінювання).

### Consequences
* Good, because transcript фіксує очікувану користь: прямий виклик з system-prompt дає якість рівня pi, але без node-старту (3.8 с/файл оверхеду) і з повним контролем `num_ctx`/`num_predict` (усуває обрізання виходу, що траплялося в варіанті A).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли бенчмарку: `~/docgen-bench3/duel.py`, `~/docgen-bench3/confirm.py`. Промпт-структура: `system`-роль — заборони (stdlib/сигнатури/regex/внутрішні імена), вимоги секцій; `user`-роль — інструкція + вміст файлу. Еталонні доки: `~/docgen-bench3/etalon/{firebase_hosting,overlay-paths,k8s-tree}.md`.

## ADR pi RPC-режим для амортизації старту при масовому docgen

## Context and Problem Statement
pi CLI у spawn-режимі (окремий процес на файл) додає ~3.8–6 с фіксованого оверхеду на кожен виклик. Для 1042 файлів це ~+1.1 год зайвого часу без будь-якого виграшу у якості.

## Considered Options
* Spawn pi на кожен файл (окремий процес)
* Персистентна сесія через `pi --mode rpc` (JSON-протокол stdin/stdout, один процес на батч)

## Decision Outcome
Chosen option: "персистентна сесія `pi --mode rpc`", because RPC-режим тримає один node-процес живим і надсилає кілька промптів послідовно через stdin — старт амортизується до нуля для всіх файлів крім першого.

### Consequences
* Good, because transcript фіксує очікувану користь: перевірочний прогін `pi_recipe.py` через RPC-сесію запущено.
* Neutral, because transcript не містить підтвердження наслідку — результат прогону RPC-рецепта ще не зафіксовано (виконується у фоні на момент завершення сесії).

## More Information
Файл: `~/docgen-bench3/pi_recipe.py`. RPC-протокол описано у `docs/rpc.md` пакета `@earendil-works/pi-coding-agent`. Конфіг ollama-провайдера для pi: `~/.pi/agent/models.json` (`api: "openai-completions"`, `baseUrl: "http://localhost:11434/v1"`).
