# Програмна перевірка заборонених Prettier-артефактів у правилі `text`

**Status:** Accepted
**Date:** 2026-05-26

## Context and Problem Statement

Правило `text.mdc` забороняло Prettier лише декларативно: `.prettierignore`, `.prettierrc*`, `prettier.config.*` і виклики prettier у `package.json#scripts`. Проте `npx @nitra/cursor fix text` не виявляв програматично ані конфіг-файлів Prettier у корені проєкту, ані команд у `package.json#scripts.*`. Inline-список у `formatting.mjs` містив лише 5 файлів із неповним переліком Prettier 3.x-форматів. Rego-полісі покривала лише top-level поля та `dependencies`/`devDependencies` у `package.json`, але не `scripts`. Потрібно вибрати стратегію матчингу для `scripts.*` (substring vs. token-based regex) та місце для FS-перевірки артефактів.

## Considered Options

- Новий JS concern `forbidden-prettier.mjs` + Rego `deny` у `package_json.rego` для `scripts.*` з token-based regex
- Розширення schema `target.json` новим полем `files.forbiddenSingle` / `files.forbiddenGlob` (generic declarative механізм)
- Проста substring перевірка `contains(cmd, "prettier")` у Rego замість token-based regex

## Decision Outcome

Chosen option: "JS concern `forbidden-prettier.mjs` + Rego deny з token-based regex `(^|[\\s/\"'])prettier($|[\\s'\"@])`", because runner вже автоматично виявляє `rules/<id>/js/*.mjs`-файли — новий declarative механізм вимагав би змін схеми, runner-а і тестів runner-а; token-based regex уникає false-positive на підрядки типу `not-prettier` або `prettier-ignore-comment`.

### Consequences

- Good, because `npx @nitra/cursor fix text` тепер падає на `.prettierignore`, `.prettierrc*`, `prettier.config.*` і `package.json#scripts.*` із `prettier`; ловить `bunx prettier`, `npx prettier`, `./node_modules/.bin/prettier`, `prettier --write`, і не реагує на `not-prettier` у назвах.
- Good, because inline-список у `formatting.mjs` видалено; канонічне місце — `forbidden-prettier.mjs` з повним переліком форматів Prettier 3.x.
- Bad, because список заборонених файлів живе в `forbidden-prettier.mjs`, а не в декларативному `target.json` — якщо в майбутньому з'явиться `forbiddenGlob`, цей concern треба буде мігрувати.
