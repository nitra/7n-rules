## ADR Виправлення блокувальних тестів як передумова запуску coverage

## Context and Problem Statement
Під час виконання `/n-coverage-fix` команда `bun run coverage` падала на інтеграційному тесті `integration-repo-checks.test.mjs`, що не давала Stryker запуститись. Причиною були два непов'язані з mutation testing дефекти: неправильний рівень заголовка в `npm/CHANGELOG.md` і хибне значення `root` у тесті `auto-rules.test.mjs`.

## Considered Options
* Виправити блокувальні дефекти щоб `bun run coverage` проходила перед Stryker
* Запускати Stryker напряму (`npx @stryker-mutator/core run`) в обхід кореневого `coverage`-скрипта
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Виправити блокувальні дефекти", because обидва дефекти були реальними помилками в коді, а не тимчасовими обходами; `bun run coverage` — канонічна точка входу згідно зі скілом `n-coverage-fix`.

### Consequences
* Good, because transcript фіксує очікувану користь: після виправлень `bun run coverage` пройшла інтеграційні перевірки і Stryker зміг запуститися з incremental report.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `npm/CHANGELOG.md` рядок 6: `# [1.30.0]` → `## [1.30.0]` — заголовок версії має рівень `##`, рядок `# Changelog` залишається `#`.
- `npm/scripts/tests/auto-rules.test.mjs:282`: `root: process.cwd()` → `root: dir` — тест `AUTO_RULE_DEPENDENCIES: disable-rules vue → image-avif` посилався на робочу директорію процесу замість тимчасової `dir`.
- Перевірка: `bunx vitest run scripts/tests/auto-rules.test.mjs -t "disable-rules vue"` — 1 passed.
