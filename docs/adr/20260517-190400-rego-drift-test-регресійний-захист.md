# Drift-тест як регресійний захист шаблонного канону в rego

**Status:** Accepted
**Date:** 2026-05-17

## Context and Problem Statement

Після міграції rego-файлу до `data.template.*` ніщо не заважало майбутньому розробнику випадково повернути canonical value як inline-літерал і зламати template-патерн непомітно — рядовий `opa test` все одно пройшов би зелений.

## Considered Options

* Покладатись лише на code review
* Додавати окремий тест `test_data_template_drives_<field>` у кожен `*_test.rego`, який перевіряє, що rego дійсно читає з `data.template`, а не з inline-значення

## Decision Outcome

Chosen option: "Окремий drift-тест у кожному `*_test.rego`", because якщо rego повертається до inline-literal, тест із нестандартним значенням у `data.template` обов'язково падає — RED-фаза TDD підтверджена для кожного концерну у transcript.

### Consequences

* Good, because кожна concern-міграція проходила RED → GREEN саме завдяки цьому тесту; зламаний `LINT_TARGETS` (v1.13.10) також виявлений через аналогічний інтеграційний тест.
* Good, because transcript фіксує 354/354 `opa test` після впровадження drift-тестів у всіх GA-концернах.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Конвенція іменування: `test_data_template_drives_<field>` або `test_data_template_drives_expected`.

Перший drift-тест, що впав на RED: `npm/rules/rego/policy/package_json/package_json_test.rego:47`.

Інтеграційний варіант: `npm/rules/rego/lint/lint.test.mjs` — 3 тести: no-targets / broken-syntax / well-formed.
