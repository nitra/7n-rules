---
type: ADR
title: "fix як автономний оркестратор з приватним check та pi"
description: Команда `fix` стає публічним автономним convergence-loop, а внутрішні перевірки й LLM-tier ізолюються через `_fix-check` та `pi`.
---

**Status:** Accepted
**Date:** 2026-06-06

## Context and Problem Statement

`npx @nitra/cursor fix` виконувала лише check-only режим, а автономне виправлення було винесене в окремий `fix-run`. Одночасно `fix --json` використовувався як внутрішній transport для orchestrator і T0-auto, але залишався частиною публічного CLI. LLM-tier також був привʼязаний до Anthropic SDK і `ANTHROPIC_API_KEY`, що не відповідало provider-agnostic підходу через `pi`.

## Considered Options

- Залишити `fix` як check-only, `fix-run` як окремий orchestrator, а `fix --json` як машинний публічний API.
- Зробити `fix` автономним orchestrator, замінити `fix --json` приватною командою `_fix-check`, а LLM-tier перевести на C1 pattern через `pi`.

## Decision Outcome

Chosen option: "`fix` як автономний orchestrator, `_fix-check` як internal check, LLM-tier через `pi`", because transcript фіксує вимогу, що `npx @nitra/cursor fix` має виконувати повний convergence-loop без участі агента, а внутрішній JSON-check потрібен лише subprocess-ам `orchestrator.mjs` і `t0.mjs`.

### Consequences

- Good, because публічний API спрощується до `fix [rules] [--max-iter N]`, а `fix-run` стає deprecated alias без окремого сценарію.
- Good, because `_fix-check` із underscore явно сигналізує private-команду і замінює `fix --json` у внутрішніх викликах.
- Good, because LLM-виправлення виконуються через `pi`, де користувач сам налаштовує provider credentials.
- Bad, because transcript не містить підтверджених негативних наслідків для публічного API; зовнішні скрипти, які покладалися на `fix --json`, можуть зламатися.
- Neutral, because transcript не містить підтвердження, що C1 pattern покриє випадки, де violation output не дає повних шляхів до файлів.

## More Information

Змінені точки: `npm/bin/n-cursor.js` (`case 'fix'` → `runOrchestratorCli`, `case '_fix-check'` → `runFixCommand(args, { json: true })`, `fix-run` як deprecated alias), `npm/skills/fix/js/t0.mjs` (`fix --json` → `_fix-check`), `npm/skills/fix/js/orchestrator.mjs`, `npm/skills/fix/js/llm-worker.mjs`.

LLM-tier працює як C1 pattern: script збирає контекст → `pi -p ... --no-session --mode text --no-tools` → script парсить відповідь і записує файл. `callPi(prompt, model)` не передає `--model`, якщо model порожній, щоб використати subscription default. У transcript перевірялись команди `bun npm/bin/n-cursor.js fix js-run 2>&1` і `_fix-check` як JSON-subprocess.

## Update 2026-06-06

Додати до рішення про `fix`-оркестратор такі факти:

- Автономність скіла декларується прапорцем `"orchestrator": true` у `meta.json`; для `npm/skills/fix/meta.json` зафіксовано `{ "auto": "завжди", "worktree": true, "orchestrator": true }`.
- LLM-tier реалізується C1-патерном: script збирає контекст з `.mdc` правила і файлів із violation output, викликає `pi -p ... --no-session --mode text --no-tools`, парсить JSON-відповідь і сам записує файли.
- `pi` не отримує `--model`, коли model є порожнім рядком, щоб використати subscription default через налаштований у користувача provider.

## Update 2026-06-06

Додаткові implementation details для `fix`-оркестратора:

- Env vars для LLM-tier перейменовані на provider-neutral `N_CURSOR_FIX_MODEL` і `N_CURSOR_FIX_MODEL_HEAVY`; порожній рядок означає не передавати `--model` у `pi`.
- `extractFilePaths(output, projectRoot)` має workspace-aware regex: `[npm] skills/fix/js/llm-worker.mjs` резолвиться як `npm/skills/fix/js/llm-worker.mjs`.
- Лаконічний clean output: при відсутності порушень перший check завершується рядком на кшталт `✅ fix: 19 правил — все чисто`; деталі ітерацій показуються лише при порушеннях або LLM-ескалаціях.
