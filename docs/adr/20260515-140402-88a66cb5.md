## ADR concern-based layout і `target.json` для правил `@nitra/cursor`

**Контекст:** Монолітний `rules/<id>/js/check.mjs` (до 1153 рядків на правило) одночасно виконував FS-перевірки і руками викликав `runConftestBatch`, що призводило до тісного зв'язування JS і Rego та унеможливлювало паралельне розширення.

**Рішення/Процедура/Факт:**
- Нова структура каталогів: `rules/<id>/js/<concern>/check*.mjs` — окремий файл на концерн; `rules/<id>/policy/<concern>/target.json` — маніфест, що декларує цільові файли для Rego.
- Три інфраструктурні модулі у `npm/scripts/utils/`: `resolve-target-files.mjs` (picomatch-резолвер `single`/`walkGlob` зі shared walk-кешем), `discover-checkable-rules.mjs` (сканер дерева правил з legacy-fallback), `run-rule.mjs` (applies-gate → JS-концерни → policy-концерни).
- `npm/schemas/target.json` — JSON Schema контракту (без `conftest.combine` — YAGNI до першого юзкейса).
- `lint-conftest.mjs` переписано: вилучено hardcoded `TARGETS`-таблицю (~150 рядків), тепер discovery через `target.json`.
- `abie/js/check.mjs` (1152 рядки) розпиляно на 6 JS-концернів + 7 утиліт у `abie/utils/`.
- Мігровано 22 правила: 19 одиночних `target.json` + 9 walk-glob + 4 abie-policy.
- Виправлено два баги: kebab→snake namespace для rego-пакетів (`style-lint.x` → `style_lint.x`); picomatch масив `['pos', '!neg']` трактує `!neg` як позитивний матчер — розділено на окремі passes.

**Обґрунтування:** CLI має сам знати, які файли фідити до Rego — JS не повинен це повторювати. Concern-based layout дозволяє незалежно версіонувати, тестувати і замінювати кожну одиницю перевірки. Shared walk-кеш між концернами одного правила дає O(1) доступ на повторних концернах.

**Розглянуті альтернативи:** Залишити monolithic `check.mjs` з явними викликами `runConftestBatch` (відхилено через дублювання стану); `conftest.combine` як маніфест-прапор (відхилено YAGNI — наразі немає реального юзкейса).

**Зачіпає:** `npm/scripts/utils/` (3 нових модулі), `npm/schemas/target.json`, `npm/scripts/lint-conftest.mjs`, `npm/bin/n-cursor.js`, `rules/abie/js/` (6 концернів), `rules/abie/utils/` (7 модулів), `rules/abie/policy/` (4 `target.json`), `rules/*/policy/*/target.json` (42 нових файли).
