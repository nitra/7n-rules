## ADR Concern-based layout і `target.json` для правил `@nitra/cursor`

**Контекст:** CLI `@nitra/cursor check` перевіряв правила через один монолітний `js/check.mjs` на правило (JS-only) або через статичну таблицю `TARGETS` у `lint-conftest.mjs` (rego-only). Неможливо було ізолювати, тестувати або розширювати окремі «concern» (аспект перевірки) правила.
**Рішення/Процедура/Факт:** Введено нову дволінійну структуру правила: `policy/<concern>/target.json` (декларативний маніфест файлів для rego) + `js/<concern>/check*.mjs` (imperативна перевірка). CLI через `discoverCheckableRules` знаходить обидва типи концернів, `runRule` спершу викликає `applies()` (gate), далі — JS-концерни і rego-концерни по черзі. `resolveTargetFiles` резолвує `single`-файл або `walkGlob`-патерн з walk-кешем для уникнення подвійного обходу. Нові модулі: `npm/schemas/target.json` (JSON Schema), `npm/scripts/utils/{resolve-target-files,discover-checkable-rules,run-rule}.mjs`. Пілот на `rego`, далі мігровано 19 з ~25 правил.
**Обґрунтування:** Симетрія `js/<concern>/` ↔ `policy/<concern>/` дає єдиний словник аспектів правила незалежно від способу реалізації (JS, rego або hybrid). `applies()` як named export з `js/applies/check.mjs` дозволяє пропустити все правило без окремого `rule.json`-файлу.
**Розглянуті альтернативи:** `rule.json` per-rule з `applies`-функцією — відхилено (зайвий файл-оголошення); name-convention (фільтр файлів без вкладеної структури) — відхилено (немає ізоляції між концернами); плаский `js/check.mjs` — залишено як legacy-fallback на час міграції.
**Зачіпає:** `npm/bin/n-cursor.js` (discoverCheckScripts → discoverCheckableRules, runChecks → runRule), `npm/scripts/lint-conftest.mjs` (TARGETS стануть похідними від target.json після міграції abie), `npm/rules/*/policy/*/target.json` (нові файли в 19 правилах), `npm/schemas/target.json`, `npm/scripts/utils/{resolve-target-files,discover-checkable-rules,run-rule}.mjs`.
