## ADR Структура директорій `rules/`: принцип `fix/lint/policy`
**Контекст:** Правила `@nitra/cursor` зберігалися у монолітній директорії `rules/<id>/js/`, що не розрізняло між діагностикою для `n-cursor check` та lint-оркестраторами для `bun run lint-X`. Треба було ввести чіткий принцип розподілу.
**Рішення/Процедура/Факт:** Введено три-рівневу структуру під `rules/<id>/`: `fix/` — для JS-концернів, що живлять `npx @nitra/cursor check`; `lint/` — для JS-оркестраторів `bun run lint-X`; `policy/` — для rego-правил, що також живлять `npx @nitra/cursor check`. Всі 26 правил мігровано з `js/` → `fix/` + `lint/`. Видалено дублюючий канал `lint-conftest.mjs`.
**Обґрунтування:** Принцип «технологія реалізації визначає директорію»: JS-check у `fix/`, JS-lint-runner у `lint/`, rego — у `policy/` (третій сиблінг). Це дозволяє знаходити всі check-артефакти за шляхом без знання рядків конфігу.
**Розглянуті альтернативи:** (A) залишити `policy/` під `fix/policy/`; (B) плоско `fix/<concern>/*.rego` поруч із `check.mjs` — відхилено через конфлікт namespace і змішування технологій.
**Зачіпає:** `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/run-rule.mjs`, всі 26 `npm/rules/*/`, `npm/bin/n-cursor.js`, `knip.json`, `npm/README.md`
