## ADR Concern-Based JS + Per-Policy target.json в @nitra/cursor

**Контекст:** Кожне правило в `@nitra/cursor` мало один монолітний `rules/<id>/js/check.mjs` (abie — 1153 рядки) і дублювало targeting для rego-полісі: списки файлів жили в JS-викликах `runConftestBatch` і в hardcoded `TARGETS`-таблиці у `lint-conftest.mjs` (~280 рядків). Масштабування до нових правил ставало дедалі дорожчим: кожна нова перевірка вимагала змін у 3–4 місцях одночасно.

**Рішення/Процедура/Факт:** Введено декларативні `target.json` поруч із кожним `<concern>.rego` — маніфест описує `{ "rule": "<id>", "files": { "single" | "walkGlob": "..." } }`. CLI `check` сам резолвить файли через picomatch і запускає `conftest`, JS-код більше не дублює виклики `runConftestBatch`. Кожне правило розпиляне з flat `js/check.mjs` на `js/<concern>/check.mjs`; shared state (наприклад, обхід k8s-дерева в `abie`) вийнятий у `utils/` з module-level кешем. `lint-conftest.mjs` переписаний: hardcoded `TARGETS` замінені на `discoverCheckableRules` + читання `target.json`. Applies-гейт через `js/applies/check.mjs`: якщо `applies()` повертає `false`, CLI пропускає всі концерни правила. Виконано concern-split для всіх 25 правил у 5 batch-ах (по 3–5 правил); legacy-fallback на flat `js/check.mjs` у `discoverCheckableRules` прибрано після завершення міграції. Додатково знайдено і виправлено баг `runSync()` у `bin/n-cursor.js`: хардкодений шлях `join(effectivePackageRoot, 'mdc')` не відповідав реальному `rules/` після phase 1–4 rename — патч включено у 1.11.1.

**Обґрунтування:** Rego об'єктивно краще для per-document перевірок JSON/YAML; JS потрібен лише для FS-парності, AST-аналізу, autofix. Розділення не лише архітектурне — воно дозволяє CLI запускати тільки потрібні концерни (applies-gate) і виключає double-run при гібридних правилах. Декларативний `target.json` усуває клас помилок «оновили rego, забули таблицю».

**Розглянуті альтернативи:** Залишити монолітні `check.mjs` і лише додати `target.json` без concern-split — відхилено: не усуває дублювання targeting і ускладнює додавання нових перевірок. Генерація `TARGETS` автоматично з `target.json` без concern-split — відхилено як half-measure, не вирішує проблему масштабування монолітів.
