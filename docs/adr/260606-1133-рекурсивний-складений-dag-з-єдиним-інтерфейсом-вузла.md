## ADR Рекурсивний складений DAG з єдиним інтерфейсом вузла

## Context and Problem Statement
Система містить задачі на кількох рівнях ієрархії (корінь, проект, workflow, підзадача). Потрібно визначити, як описати структуру так, щоб батьківський вузол не залежав від того, чи його дочірній вузол є простою задачею чи цілим підграфом.

## Considered Options
* Recursive Compound DAG — вузол або атомарний, або містить власний підграф
* Flat DAG з окремим рівнем для підзадач
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Recursive Compound DAG", because батьківський вузол бачить лише `state: resolved | failed` і `outputs`, не знаючи внутрішньої структури дочірнього вузла (принцип substitutability: атомарний вузол можна розкрити у Compound без змін у батьківському графі).

### Consequences
* Good, because transcript фіксує очікувану користь: єдиний `execute(node)` рекурсивно застосовується на всіх рівнях без спеціальних гілок.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Структура файлів: `<nodeId>/meta.json`, `inputs.json`, `outputs.json`, `graph/<childId>/...`. Executor при виконанні атомарного вузла може самостійно вирішити розкрити його у Compound і записати підграф у `graph/`.

## Context and Problem Statement
Граф задач є динамічним і може модифікуватися під час виконання. Потрібен підхід до зберігання стану всіх вузлів, патчів та repair-history без центрального координатора.

## Considered Options
* Файлова система — кожен вузол сам пише свої файли
* Централізований оркестратор зберігає граф
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Файлова система — кожен вузол сам пише свої файли", because transcript явно зазначає: «граф з усіма його атрибутами пишеться у файли, кожен таск/вузол пише свій сам».

### Consequences

## ADR LLM-інженер як мета-рівень з необмеженим доступом до запису графу

## Context and Problem Statement
При помилці вузла система потребує механізму діагностики і виправлення. Питання: чи інженер є вузлом у графі, чи зовнішнім супервізором; чи може він змінювати батьківські рівні.

## Considered Options
* Інженер як мета-рівень поза графом із повним доступом (обраний варіант)
* Інженер як вузол всередині графу (рекурсивний)
* Інженер з обмеженим доступом лише до локального підграфу
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Інженер як мета-рівень поза графом із повним доступом", because transcript явно зазначає: «мета-рівень поза графом» і «може підніматись вгору і змінювати батьківський граф» без обмежень.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер бачить повний path від кореня до вузла що впав і вирішує на якому рівні ієрархії втрутитись (patch node, restructure workflow, redesign project, patch root).
* Bad, because transcript фіксує ризик: необмежений доступ дозволяє нескінченний цикл (`node_7 fails → engineer patches → node_7 fails → ...`), що вирішується через time-budget guard.

## More Information
`GraphPatch` може включати: replace node (атомарний → compound), insert nodes, rewire edges, modify inputs. При модифікації батьківського вузла через помилку дочірнього — memory записується на обидва: `child.repair_history` містить `triggered_parent_patch`, `parent.repair_history` — деталі патчу.

## ADR Граф є mutable з invalidation cascade

## Context and Problem Statement
Після того як інженер змінює вузол у графі, вузли-наступники що вже мали `state: resolved` потенційно мають застарілі `outputs`. Потрібно визначити модель консистентності.

## Considered Options
* Mutable з invalidation — змінені вузли інвалідують successors
* Append-only — нові вузли додаються, старі не змінюються
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Mutable з invalidation", because transcript явно зазначає: «mutable with invalidation».

### Consequences
* Good, because transcript фіксує очікувану користь: система може коригувати вже виконані частини графу при глибоких структурних змінах.
* Bad, because transcript фіксує наслідок: `outputs.json` отримує `{ stale: true, reason: "patch_by_engineer" }` і каскад інвалідації поширюється до листів графу, що може призвести до масового перевиконання.

## More Information
Граф зберігається як послідовність снапшотів: `DAG_v1 → [EngineerPatch] → DAG_v2 → ...`. Кожен патч логується у `patches/patch_NNN.json`.

## ADR Node-local repair_history як пам'ять інженера

## Context and Problem Statement
Інженер stateless між викликами — без зовнішньої пам'яті він повторюватиме невдалі спроби. Потрібно визначити де зберігати діагностичне знання про вузол між ітераціями ремонту.

## Considered Options
* Пам'ять зберігається на вузлі (`repair_history.json`)
* Пам'ять зберігається у самому інженері (stateful agent)
* Централізований журнал ремонтів
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Пам'ять зберігається на вузлі (`repair_history.json`)", because transcript явно зазначає: «потрібна memory на вузлі де він вносе зміни»; інженер stateless, знання — у файлах.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер при кожному виклику читає `repair_history`, не повторює вже невдалі стратегії; при escalation повний `repair_history` передається батькові.
* Bad, because Neutral, because transcript не містить підтвердження наслідку щодо розміру repair_history при довгих repair-циклах.

## More Information
Структура запису: `{ attempt, engineer_reasoning, patch_applied, result, failure_reason }`. При escalation вгору — батько отримує не лише `{ node_id, reason: "timeout" }`, а повний масив attempts дочірнього вузла.

## ADR Time-budget як convergence guard з процентним успадкуванням

## Context and Problem Statement
Без обмеження інженер може нескінченно повторювати спроби при невирішуваній помилці. Потрібно визначити механізм термінації і як бюджет розподіляється по ієрархії.

## Considered Options
* Time-budget (обмеження часом, не кількістю спроб)
* Max-attempts (обмеження кількістю спроб)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Time-budget з фіксованим процентним успадкуванням від батька", because transcript явно зазначає: «не в кількості спроб, а виділеному часі» і «фіксований відсоток від батьківського бюджету».

### Consequences
* Good, because transcript фіксує очікувану користь: чим глибше в ієрархії — тим менший бюджет; інженер може адаптувати стратегію (багато часу → складна спроба, мало часу → швидкий fix); батько при escalation отримує свій **свіжий** бюджет.
* Bad, because Neutral, because transcript не містить підтвердження наслідку щодо конкретного значення відсотку редукції.

## More Information
`repair_context.json`: `{ deadline, started_at, time_budget_sec, attempts[] }`. `deadline = parent.deadline - reduction_factor`. При timeout: `node.state = "unresolvable"`, escalate з повним `repair_history`. При timeout root-рівня — ескалація до людини.

## ADR Специфікаційно-кероване виконання атомарних вузлів через LLM-executor

## Context and Problem Statement
Вузол потребує логіки виконання. Питання: захардкодити логіку в `meta.json`, чи генерувати її динамічно при кожному виконанні.

## Considered Options
* LLM генерує виконання динамічно на основі опису у `meta.json`
* Логіка виконання захардкоджена у `meta.json`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "LLM генерує виконання динамічно", because transcript явно зазначає: «meta містить лише опис/промпт, а LLM генерує виконання динамічно».

### Consequences
* Good, because transcript фіксує очікувану користь: система повністю специфікаційна — жодного захардкодженого коду; інженер може змінювати `meta.json` (опис/обмеження) щоб змінити поведінку вузла при retry.
* Bad, because Neutral, because transcript не містить підтвердження наслідку щодо детермінованості повторного виконання з тим самим промптом.

## More Information
При retry вузол виконує повний re-read: `meta.json` (може бути новий тип/логіка після патчу інженера) і `inputs.json`. Executor і engineer — два окремих агенти з різними ролями і промптами. Executor також самостійно вирішує розкрити вузол у Compound якщо задача складна.
