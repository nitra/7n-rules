## ADR concern-based JS + per-policy target.json

**Контекст:** Сесія аналізувала поточну архітектуру `check`-скриптів у CLI `@nitra/cursor`, де кожне правило мало один монолітний `js/check.mjs`, а targeting для Rego-полісі (список файлів для `conftest`) жив або у централізованій таблиці `lint-conftest.mjs:TARGETS`, або розпорошено по JS-коду.

**Рішення/Процедура/Факт:** Затверджена нова структура каталогу правила: `rules/<id>/js/<concern>/check*.mjs` та `rules/<id>/policy/<concern>/<name>.rego` + `target.json`. CLI (`discoverCheckableRules`, `runRule`, `resolveTargetFiles`) автоматично підхоплює обидві сторони. Applies-гейт — `js/applies/check.mjs` з `export async function applies()`. Targeting для Rego — декларативний `target.json` поряд із `.rego` (`single` або `walkGlob` через picomatch). Спільний стан між concerns — кешуючий singleton у `rules/<id>/utils/`. Пілот завершено на `rules/rego/`: видалено монолітний `check.mjs`, створено три `target.json` і `js/applies/check.mjs`. Версія пакета піднята до 1.11.0.

**Обґрунтування:** Монолітний `check.mjs` ставав нечитабельним (1153 рядки в `abie`), а centralized `lint-conftest.mjs:TARGETS` дублював targeting-логіку, яка вже жила у JS. Нова структура: кожен concern самодостатній, targeting декларативний і живе поряд із `.rego`, discovery без конфігурації — конвенція замість реєстру. Симетрія `js/<name>/` ↔ `policy/<name>/` дає єдиний словник concern-ів на правило.

**Розглянуті альтернативи:** (1) `rule.json` per-rule з декларацією всіх targets — відхилено: менш локальний, всі targets в одному файлі. (2) Name-convention (ім'я policy-каталогу → файл-таргет) — відхилено: конфліктує коли два правила хочуть однойменну `package_json`-полісі з різними walk-матчерами. (3) `conftest test .` (feed everything) — відхилено: не виразити «файл має обов'язково існувати», перформанс-регресія на великих деревах. (4) Frontmatter у `<id>.mdc` — відхилено: `.mdc` копіюється у проєкти, метадані там зайві.

**Зачіпає:** `npm/bin/n-cursor.js` (`discoverCheckableRules`, `runChecks`, `runRule`), `npm/scripts/utils/` (3 нових модулі), `npm/schemas/target.json`, `npm/rules/rego/` (повністю мігровано), `npm/rules/*/js/check.mjs` (legacy-сумісність під час інкрементальної міграції решти правил).
