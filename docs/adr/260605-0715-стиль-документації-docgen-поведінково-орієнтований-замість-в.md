## Висновок для docgen

**Direct REST — кращий вибір для Tier 1**, з трьох причин:

1. **+28% часу** від pi-оверхеду на кожному файлі → на 1042 файлах це +7 годин зайвого часу.
2. **Гірша якість** через те, що pi's system-prompt і CLAUDE.md шум додають "confusion" до промпту малої моделі.
3. **Tier 1 не потребує інструментів**: читання файлу й запис доки — детерміновані операції в CLI, не в агента.

**pi через ollama має сенс** лише для Tier 2/3 (де потрібний синтез і Tools), або якщо хочеш **готовий pi-субагент** що сам читає і пише файл — але і там кращий варіант це Claude з Ollama як fallback, а не pi+Ollama.

Гібридна архітектура залишається: `docgen gen --engine ollama` (direct REST, concurrency 1) для Tier 1 → Claude для Tier 2/3.

## ADR Ollama як локальний двигун для docgen Tier 1 (економія токенів)

## Context and Problem Statement
Tier 1 скіла `docgen` коштує ~26–36 млн хмарних токенів Claude на повний прогін 1042–1442 файлів. Запит: розглянути використання локальної Ollama для Tier 1 щоб обнулити цю статтю витрат.

## Considered Options
* Локальна Ollama (direct REST API) для Tier 1, Claude для Tier 2/3.
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Локальна Ollama (direct REST) для Tier 1, Claude для Tier 2/3", because це ізолює найбільшу статтю витрат (~26 M токенів) у незалежних паралельних задачах і не впливає на якість синтезу Tier 2/3.

### Consequences
* Good, because transcript фіксує очікувану користь: Tier 1 коштує $0, Tier 2/3 залишається Claude (~2.6 M токенів).

## ADR Direct Ollama REST vs pi CLI як канал для Tier 1 docgen

## Context and Problem Statement
При використанні локальної Ollama для docgen Tier 1 постало питання: звертатися до моделі напряму через `POST http://localhost:11434/api/generate` чи через `pi CLI` з провайдером ollama.

## Considered Options
* Direct Ollama REST API (`POST /api/generate`).
* `pi --provider ollama --model gemma3:4b` (pi CLI як обгортка над Ollama OpenAI-сумісним ендпойнтом).

## Decision Outcome
Chosen option: "Direct Ollama REST API", because на однаковій моделі і промпті direct дав 71.8 с проти 91.6 с через pi (+28% overhead), і вищу якість виходу (~73% vs ~65% vs еталону), оскільки pi-фреймворк завантажує додатковий контекст (`CLAUDE.md`, сесія), що шумить для малої 4B-моделі.

### Consequences
* Good, because transcript фіксує очікувану користь: на 1042 файлах -28% часу ≈ -7 годин; відсутність фреймворкового шуму знижує кількість галюцинацій (pi видав «немає публічного API» і вигаданих споживачів).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Бенчмарк: `python3 /tmp/docgen-pi-bench/run.py`, файл `npm/rules/abie/lib/k8s-tree.mjs` (110 рядків).
pi підключається до Ollama через `~/.pi/agent/models.json` з полями `baseUrl: http://localhost:11434/v1`, `api: openai-completions`.
pi вивантажує модель після завершення виклику (`ollama ps` порожній), direct — ні (тепла модель для наступного файлу).
