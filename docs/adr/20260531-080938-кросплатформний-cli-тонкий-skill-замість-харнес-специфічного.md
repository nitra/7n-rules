## ADR Семантика `worktree remove` і `worktree prune` (H2 + H-prune-b)

## Context and Problem Statement
Обидві операції незворотні: `remove` видаляє checkout і опис, `prune` чистить розсинхрон. Потрібно визначити, чи мають вони бути безпечними за замовчуванням чи агресивними.

## Considered Options
* `remove`: H1 — без `--force` (git сам відмовиться на брудному дереві), H2 — те саме + явний `--force` для свідомого видалення
* `prune`: H-prune-a — dry-run за замовчуванням, видалення лише з `--yes`; H-prune-b — одразу прибирає осиротілі

## Decision Outcome
Chosen option: "`remove` = H2, `prune` = H-prune-b", because `remove --force` дозволяє явно викинути брудний worktree без обходу захисту; `prune` агресивний, тому що осиротілі `.md`-файли не мають цінності і їх безпечно прибирати без підтвердження.

### Consequences
* Good, because transcript фіксує очікувану користь: `remove` захищає від випадкової втрати незакомічених змін; `prune` не вимагає ручного підтвердження для типових завдань прибирання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`remove` передає `--force` далі в `git worktree remove --force`. `prune` робить `git worktree prune` + видаляє `.worktrees/*.md` без відповідного git-worktree; worktree без `.md` — лише попередження у звіт. Реалізовано в `npm/scripts/worktree-cli.mjs`.
