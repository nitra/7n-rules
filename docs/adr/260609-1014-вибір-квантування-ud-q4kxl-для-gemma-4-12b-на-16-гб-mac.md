## ADR Вибір квантування UD-Q4_K_XL для Gemma-4 12B на 16 ГБ Mac

## Context and Problem Statement
Потрібно запустити `unsloth/gemma-4-12b-it-GGUF` (11.95B параметрів) на Mac із 16 ГБ unified memory без свопу. macOS + застосунки займають ~3–5 ГБ, залишаючи ~9–10 ГБ під ваги + KV-кеш.

## Considered Options
* UD-Q4_K_XL (7.37 ГБ) — Unsloth Dynamic 2.0, нерівномірна побітова розкладка з вищою точністю критичних шарів
* Q4_K_M (7.12 ГБ) — класична k-quant схема llama.cpp з фіксованим рецептом
* Q5_K_S / Q5_K_M (8.2–8.41 ГБ) — вищий бітрейт, але залишає мало місця під KV-кеш
* Q6_K / Q8_0 (9.79–12.7 ГБ) — гарантований своп на 16 ГБ

## Decision Outcome
Chosen option: "UD-Q4_K_XL", because за майже однакового розміру (+0.25 ГБ vs Q4_K_M) дає кращу якість завдяки каліброваному побітовому розподілу по конкретній моделі.

### Consequences
* Good, because модель завантажена як `gemma4:12b` (7.5 ГБ за `ollama list`), шар `ee33ab5be8e0` — влізла без свопу.
* Bad, because потребує свіжого llama.cpp/Ollama (≥0.20.0 для IQ/dynamic-типів); на старших версіях необхідний fallback на Q4_K_M.

## More Information
Завантаження: `ollama pull hf.co/unsloth/gemma-4-12b-it-GGUF:UD-Q4_K_XL`, потім перейменовано через `ollama cp` + `ollama rm` в `gemma4:12b`. Рекомендовані параметри для 16 ГБ: `/set parameter num_ctx 8192`, `OLLAMA_KV_CACHE_TYPE=q8_0`, `sudo sysctl iogpu.wired_limit_mb=13000`.

## ADR Заміна мультимодальних Ollama-моделей Gemma E-series на text-only через прямий імпорт LM GGUF

## Context and Problem Statement
Ollama-теги `gemma4:e2b` та `gemma4:e4b` включали vision + audio capabilities (7.2 ГБ / 9.6 ГБ). Потрібні були text-only варіанти для зменшення витрат пам'яті. Спроба пересобрати модель із того самого блоба не допомогла — capability лишились незмінними.

## Considered Options
* Прямий імпорт LM-only GGUF (без `mmproj`) через `curl` + `ollama create -f Modelfile`
* Пересобрати з того самого Ollama-блоба — виявилося неможливим: capabilities визначаються метаданими GGUF
* Re-conversion із safetensors через `convert_hf_to_gguf.py` (llama.cpp) — важкий шлях з великим даунлоадом

## Decision Outcome
Chosen option: "Прямий імпорт LM-only GGUF", because в репо `unsloth/gemma-4-E2B-it-GGUF` і `unsloth/gemma-4-E4B-it-GGUF` vision/audio-проектор зберігається окремим файлом `mmproj-*.gguf`; основні quant-файли (`UD-Q4_K_XL.gguf`) є чистими LM-вагами без multimodal-ченкерів.

### Consequences
* Good, because нові `gemma4:e2b` (3.4 ГБ) та `gemma4:e4b` (4.4 ГБ) мають capabilities лише `completion + tools + thinking` без `vision` і `audio` — підтверджено `ollama show`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли завантажені через `curl` з `https://huggingface.co/unsloth/gemma-4-E2B-it-GGUF/resolve/main/gemma-4-E2B-it-UD-Q4_K_XL.gguf` та аналогічно для E4B. Modelfile-и (`/tmp/e2b-text.Modelfile`, `/tmp/e4b-text.Modelfile`) містили лише `FROM <path>` + параметри (`temperature 1`, `top_k 64`, `top_p 0.95`). Фінальні теги створено через `ollama create gemma4:e2b -f ...` та `ollama create gemma4:e4b -f ...`. Старі мультимодальні теги видалені через `ollama rm`.
