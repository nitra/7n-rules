# `check changelog`: вибір бази порівняння залежно від поточної гілки

**Status:** Accepted
**Date:** 2026-05-20

## Context and Problem Statement

`resolveBaseRef()` у `check.mjs` перебирала кандидатів у порядку `['dev', 'main']`. Якщо репо мало гілку `dev`, на `main` база ставала `merge-base(dev, HEAD)` — потенційно дуже старий спільний предок. Зміни, внесені безпосередньо в `main`, порівнювалися з надто старим станом. Крім того, `dev` може бути відсутній у репозиторії.

## Considered Options

* Залишити `['dev', 'main']` як пріоритетний список кандидатів для всіх гілок (попередня поведінка)
* На `main` порівнювати з `origin/main` (або `HEAD~1` за відсутності remote); feature-гілки — `merge-base` з `dev` якщо є, інакше з `main`

## Decision Outcome

Chosen option: "На `main` порівнювати з `origin/main` або `HEAD~1`", because зміни, що вносяться безпосередньо в `main`, мають порівнюватися з попереднім станом `main`, а не з `dev`; `dev` взагалі може бути відсутній у репо.

### Consequences

* Good, because таблиця бази порівняння стала детермінованою: `main` → `origin/main` / `HEAD~1`; `dev` → local-only пропускається; feature → `merge-base(dev, HEAD)` або `merge-base(main, HEAD)`.
* Good, because виправлено крайній випадок: якщо `origin/main` збігається з `HEAD`, перевірка коректно бачить порожній diff замість хибного fallback на `HEAD~1`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

* Змінено: `npm/rules/changelog/fix/consistency/check.mjs` — константа `BASE_BRANCH_CANDIDATES` видалена, `resolveBaseRef()` переписана з розгалуженням по `branch === 'main'`.
* `npm/rules/changelog/fix/consistency/check.test.mjs` — тест `'main після merge dev → main'` замінено на `'main синхронізований з origin/main без локальних змін → pass'`; додано ще 2 сценарії.
* `.cursor/rules/n-changelog.mdc` — оновлено секцію `### local-only`.
* `npm/CHANGELOG.md` — запис у розділі `1.13.63`.
* Верифікація: `bun ./npm/bin/n-cursor.js check changelog`.

## Update 2026-05-20

Додаткова деталь реалізації: коли `origin/main` збігається з `HEAD`, `is-ancestor` повертав `false` і хибно активував fallback на `HEAD~1`; виправлено прямим порівнянням SHA.

Логіка `resolveBaseRef()` по гілках:
* `branch === 'main'`: `origin/main` якщо SHA відомий; якщо `origin/main === HEAD` (синхронізовано) — diff порожній, перевірка проходить; без remote — `HEAD~1`.
* `branch === 'dev'`: local-only перевірка пропускається.
* feature: `merge-base(dev, HEAD)` якщо `dev` є, інакше `merge-base(main, HEAD)`.

## Update 2026-06-01

**False-positive при значно відсталому `origin/dev`:** `fix-changelog` позначав пакет `demo` як такий, що містить uncommitted relevant changes, хоча `demo/.changes/` порожній і реліз `demo@0.0.2` вже відбувся. Причина: `merge-base` обирався з `origin/dev`, який відставав від `main` на 589 комітів. `git diff --stat "$(git merge-base HEAD origin/dev)"...HEAD -- demo/` показував увесь `demo/` як нові зміни — базою слугувала точка задовго до релізу.

Діагностичні команди: `git merge-base HEAD origin/dev`, `git log "$(git merge-base HEAD origin/dev)"..HEAD --oneline | wc -l` (589 комітів), `git show-ref --verify --quiet refs/remotes/origin/dev`.

Наслідок: `fix-changelog` ненадійний якщо `origin/dev` суттєво відстає від `main`; worktree треба перестворювати від синхронізованого `main` з `git fetch` перед перевіркою.

**Stale worktree та колізія назви:** `npx @nitra/cursor worktree add "main-fix"` впало з `fatal: a branch named 'main-fix' already exists` — stale worktree з попереднього запуску. Preflight-послідовність: `git branch --list 'main-fix*' && git worktree list` → обрати вільну назву → `worktree add "main-fix2"`. Автоматичний вибір вільної назви впроваджено в CLI окремим рішенням (see `20260531-070618-n-cursor-worktree-cli.md`).
