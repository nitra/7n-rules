# Скіли у git worktree: класифікація за сумісністю та обмеження паралелізму

**Status:** Accepted
**Date:** 2026-05-30

## Context and Problem Statement

Проєкт використовує git worktrees для паралельної роботи над гілками. Набір скілів (`n-fix`, `n-lint`, `n-taze`, `n-coverage-fix`, `n-fix-tests`, `mdc-check`, `n-adr-normalize`, `n-start-check`, `n-llm-patch`, `n-publish-telegram`) прив'язаний до CWD. Виникло питання: чи доцільно запускати ці скіли у git worktree, який ефект це дає й чи є обмеження на паралельний запуск.

## Considered Options

* Запускати скіли лише в головному checkout (без worktree)
* Запускати скіли у worktree для ізоляції правок без класифікації
* Класифікувати скіли за ознаками «пише в репо» та «важкий/CPU-чутливий» і для кожної групи задати окремі правила щодо worktree

## Decision Outcome

Chosen option: "Класифікувати скіли за ознаками «пише в репо» та «важкий/CPU-чутливий»", because worktree — повноцінний checkout зі своїм `package.json`, тож передумови всіх скілів у ньому виконуються. Класифікація дозволяє чітко описати, де worktree дає ізоляцію результату (7 із 10 скілів), а де — зайвий overhead або конфлікт. Водночас worktree не обходить заборону паралельного eslint/CPU-важкого прогону: всі worktree читають з того ж диска й CPU.

### Consequences

* Good, because правки структури, лінту, тестів, залежностей ізольовані на власній гілці й не торкаються головного checkout.
* Good, because `n-taze` виграє найбільше: worktree автоматично задовольняє його передумову «чисте робоче дерево», а dep-bump природно живе на окремій гілці.
* Good, because легкі скіли (`n-fix`, `mdc-check`) безпечно паралелити в кількох worktree одночасно (застереження: `mdc-check` бампає `npm/package.json` — потенційний merge-конфлікт при декількох паралельних worktree).
* Bad, because CPU-важкі скіли (`n-lint`, `n-coverage-fix`, `n-fix-tests`, `n-taze`) залишаються під забороною паралельного запуску з `CLAUDE.md` — worktree не дозволяє ганяти їх одночасно.
* Bad, because `n-start-check` не виграє від worktree — паралельний запуск ризикує конфліктом портів.
* Bad, because `n-llm-patch` і `n-publish-telegram` read-only, тому worktree безглуздий — нічого ізолювати, лише зайвий `bun i`.
* Neutral, because скіли присутні у кожному worktree під `.pi/skills/` (симлінки до `npm/skills/`) — зміни скіла в одній гілці не впливають на інші активні worktrees.

## More Information

Класифікація скілів за доцільністю worktree:

| Категорія | Скіли |
|---|---|
| ✅ Виграють, безпечно паралелити | `n-fix`, `mdc-check`, `n-adr-normalize` |
| ✅ Виграють, лише послідовно | `n-lint`, `n-coverage-fix`, `n-fix-tests`, `n-taze` |
| ⚠️ Особливий (конфлікт портів) | `n-start-check` |
| ⛔ Безглуздо (read-only) | `n-llm-patch`, `n-publish-telegram` |

Заборона паралельного `eslint` зафіксована у `CLAUDE.md` і `.cursor/skills/n-lint/SKILL.md:94-105`. Директорія `.claude/worktrees/` позначена як захищена у `CLAUDE.md` — файли там не можна змінювати вручну.

## Update 2026-05-30

Виявлено крос-worktree gap у `withLock`: механізм зберігав стан у `node_modules/.cache/n-cursor/<key>/`, а кожен git-worktree має власний `node_modules`. Через це важкі CLI-команди, запущені паралельно у різних worktree, не серіалізувались — паралельний eslint/Stryker міг запускатись всупереч явній забороні. Запропонована стратегія виправлення: перенести `cacheDir` до `git rev-parse --git-common-dir` — спільного для всіх linked-worktree. Реалізацію зафіксовано в `docs/adr/20260531-211500-withlock-крос-worktree-серіалізація.md`.
