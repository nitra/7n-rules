**Status:** Accepted
**Date:** 2026-07-18

## Context and Problem Statement

`npm/rules/js/eslint/fix-worker.mjs` (#99) замінив один мегапромпт на всі файли concern-а окремою `runAgentFix`-сесією на файл — вирішило 100%-timeout, але не 100%-успіх. Два незалежні реальні виміри (isolated worktree, реальний `runAgentFix`, cloud-min, бюджет 120с) показали: **час сесії майже не залежить від кількості порушень у файлі** — `dispatch.test.js` (2 порушення) зайняв 96.1с, майже стільки ж, скільки `acp-kit.test.js` (21 порушення, 90.8с ізольовано). Це виключає гіпотезу "сортування файлів за складністю" (перевірено й відкинуто — ідентичний результат sorted/unsorted). Домінують фіксовані витрати: bootstrap агентної сесії, tool-виклики (`ast_facts`/`self_check`), verify-feedback цикл (до 3 раундів моделі). Наслідок: per-file батчинг масштабується щонайбільше на ~1 файл на rung, незалежно від складності — другий файл (`acp-kit.test.js`, 21 порушення) жодного разу не встиг навіть спробувати в парі з `dispatch.test.js`.

Окремо зафіксовано explicit design-межу: `types.mjs` (FixContext.recordDurableWrite) прямо забороняє durable-запис для правок наявного стороннього коду ("rollback-контракт незмінний") — lint-чистота не гарантує поведінкову коректність, на відміну від self-contained артефактів (doc-files). Обхід цього правила розглядався й відкинутий як найризикованіший варіант.

## Considered Options

Brainstorm SCAMPER, 50 ідей, згруповані в 10 кластерів:

- **A** — скоротити раунди в сесії (verify-feedback, ast_facts, self_check).
- **B** — прибрати bootstrap-накладні витрати: перевикористати одну сесію на кілька файлів.
- **C** — перенести механічні правила (require-await, no-await-expression-member) у детермінований T0-codemod.
- **D** — пропускати приречені local-тири для js/eslint (0% успіху за даними).
- **E** — батчити за rule, не за file (спекулятивно, непідтверджено).
- **F** — паралелізм замість послідовного циклу по файлах (`Promise.all`).
- **G** — тюнінг-ручки (`DEADLINE_FRACTION`, per-concern timeout override).
- **H** — точний профайлінг (turnCount/wallMs/verifyAttempts) перед подальшими змінами.
- **I** — прийняти multi-run конвергенцію без змін коду.
- **J** — обхід durable-write safety-правила (з test-suite гейтом).

## Decision Outcome

Chosen option: **послідовно H → F → B/C/D за результатами кожного кроку**, тому що:

1. **H** майже безкоштовний (телеметрія, не архітектурна зміна) і є передумовою для розумного вибору між рештою — поточне розуміння причини 90-96с ґрунтується на здогадці, не на профайлінгу.
2. **F** дає результат незалежно від того, що покаже H: паралелізм атакує "1 файл на rung" напряму, без залежності від ПРИЧИНИ накладних витрат. Низький ризик (snapshot-запис синхронний, гонок не буде), контейнерно (лише `fix-worker.mjs`).
3. **B** (найбільший потенційний виграш, якщо H підтвердить bootstrap-гіпотезу) і **C** (безпечний, зменшує обсяг роботи для LLM незалежно від решти) — наступні кроки за результатами H/F.
4. **D**/**G** — дешеві доповнення, не самостійне рішення.
5. **E**, **I**, **J** — відхилені: E повторює структуру вже спростованої гіпотези (сортування), I не є рішенням вузького місця, J суперечить явно задокументованому safety-правилу без достатньої компенсації ризику.

Кожен крок виконується окремо, з аналізом результату (реальний прогін/експеримент) перед переходом до наступного.

### Consequences

- Good, because рішення спирається на дані (два реальні виміри), а не на здогадку — сортування вже відкинуто саме тому, що вимір спростував гіпотезу.
- Good, because F і H не потребують компромісу з existing safety-правилами (recordWrite-контракт незмінний).
- Bad, because B (найбільший потенційний виграш) відкладено до підтвердження H — можливо, доведеться повернутись і переоцінити пріоритет, якщо профайлінг покаже іншу картину.
- Bad, because сесія не дає гарантії повного вирішення "1 файл на rung" — навіть після F/B/C можливо, що конвергенція до 0 порушень для найважчих файлів вимагатиме кількох прогонів `lint --full`.

## More Information

Усі 50 ідей генерації — у транскрипті сесії (не продубльовано тут за обсягом); ключові по кластерах:

- A: verify-feedback trim, ast_facts/self_check прибрати, turn-based замість time-based бюджет.
- B: перевикористання сесії між файлами (потребує зміни в `@7n/llm-lib`, не лише `fix-worker.mjs`).
- C: T0-codemod для require-await/no-await-expression-member за зразком `fix-eslint.mjs`/`fix-check.mjs`.
- D: per-concern override, старт одразу з cloud-min.
- F: `for...of` → `Promise.all`, дедлайн гейтить лише старт нових файлів.
- H: телеметрія per-call (turnCount, wallMs, verifyAttempts) у `fix-worker.mjs`.

Відкладені кластери: E (батчинг за rule — потребує окремого підтвердження гіпотези), I (multi-run — чесний fallback, не пріоритет), J (durable-write обхід — найризикованіше, суперечить документованому правилу).

Відкриті питання: чи підтвердить H гіпотезу про bootstrap-домінування; чи достатньо F+C без B для практичного покриття; per-concern timeout override (D) — механізм ще не існує в `concern.json`, потребує окремого дизайну, якщо буде обрано.
