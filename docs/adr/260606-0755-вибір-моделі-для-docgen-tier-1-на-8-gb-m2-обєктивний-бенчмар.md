## ADR Вибір моделі для docgen Tier 1 на 8 GB M2: об'єктивний бенчмарк

## Context and Problem Statement
Для локального Tier 1 docgen-конвеєра на машині з 8 GB unified memory потрібно обрати модель Ollama, яка дає найкращий баланс якості документації та швидкості за умови повного вміщення в GPU без свопу.

## Considered Options
* `gemma3:4b` (3.3 GB) — 100% GPU
* `qwen2.5-coder:3b` (2.4 GB) — 100% GPU
* `qwen2.5:7b` (5.1 GB) — 9%/91% CPU/GPU офлоад
* `qwen2.5-coder:7b` (5.1 GB) — 9%/91% CPU/GPU офлоад
* `llama3.1:8b` (5.7 GB) — 13%/87% CPU/GPU офлоад
* `llama3.2:3b`, `gemma2:2b`, `phi3.5`, `qwen3:4b` — відкинуті (мова/контроль)

## Decision Outcome
Chosen option: "`gemma3:4b`", because об'єктивний бенчмарк з чистим запуском перед кожною моделлю і ручною оцінкою виходів проти еталона показав: лише `gemma3:4b` та `qwen2.5-coder:3b` повністю вміщуються в GPU (100% GPU, без офлоаду); `gemma3:4b` виграла за якістю (~86% vs 77%), правильно формує поведінкові секції та точніше відтворює крайові деталі.

### Consequences
* Good, because transcript фіксує очікувану користь: 14 tok/s (проти 0.4 tok/s у моделей, що свопляться), повністю в GPU, відсутність memory thrashing, екстраполяція Tier 1 ~14.5 год на 1042 файли.
* Bad, because `qwen2.5-coder:3b` (~36 tok/s) вдвічі швидша — `gemma3:4b` обрана як backup якщо з'являться ресурсні обмеження і якість чернетки не критична.

## More Information
Бенчмарк: `/tmp/docgen-bench/run.py`, `/tmp/docgen-bench2/run.py`; еталон якості: `/tmp/docgen-bench2/etalon/*.md`; виміряно `ollama ps` (RAM, processor split), wall_s, gen_tps, ручна оцінка виходів за рубрикою точність/повнота/мова/формат (4×25%). Моделі що не помістились: `gemma4:e4b` 9.6 GB → 0.4 tok/s (своп), `deepseek-coder-v2:16b` → не завантажено. `qwen3:4b` дискваліфікована: ігнорує `/no_think`, генерує «думання» російською.

## ADR System-prompt — головний важіль якості локальної моделі, не транспорт

## Context and Problem Statement
Порівняння прямого Ollama-виклику з викликом через `pi` CLI показало різницю якості (~71% vs ~87%). Потрібно зрозуміти, чи перевага від транспорту `pi` або від іншого чинника — щоб ухвалити обґрунтоване рішення про архітектуру Tier 1.

## Considered Options
* Використовувати `pi` як транспорт (вбудований system-prompt, node-процес)
* Прямий `ollama /api/chat` без system-prompt
* Прямий `ollama /api/chat` з явним system-prompt

## Decision Outcome
Chosen option: "Прямий `ollama /api/chat` з явним system-prompt", because контрольований експеримент довів: прямий виклик із system-роллю (C) дав ~85% — аналогічно `pi` (~87%); різниця 2 п.п. знаходиться в межах шуму ручного оцінювання (~±3 п.п.). Варіант без system-prompt (A) дав ~71% — систематична просадка через відсутність поведінкових обмежень.

### Consequences
* Good, because transcript фіксує очікувану користь: прямий виклик дає повний контроль `num_ctx`/`num_predict` (прибирає обрізання виходу, яке підвело варіант A на `overlay-paths`), відсутній node-overhead pi (~3.8–5 с/виклик), простіша інтеграція в JS-оркестратор.
* Bad, because `pi` має власний відшліфований system-prompt; чорновий system-prompt (C) програє йому ~2 п.п. — але ця різниця ліквідується допрацюванням промпта.

## More Information
Скрипти: `~/docgen-bench3/run.py`, `~/docgen-bench3/duel.py`, `~/docgen-bench3/confirm.py`; результати: `~/docgen-bench3/results.jsonl`; 3 тестові файли: `npm/rules/abie/js/firebase_hosting.mjs`, `npm/rules/abie/lib/overlay-paths.mjs`, `npm/rules/abie/lib/k8s-tree.mjs`; виміряний pi boot overhead: ~3.8–5 с/виклик (перший виклик ~6.4 с).

## ADR JS-оркестратор як точка входу docgen (інверсія керування)

## Context and Problem Statement
One-shot підхід (модель отримує файл + один промпт) дає нестабільну якість: модель псує структуру (витікають сигнатури, stdlib, внутрішні імена, галюцинації в гарантіях). Потрібна архітектура, де ці проблеми вирішуються детерміновано, а не prompt-інженерією.

## Considered Options
* One-shot: один промпт → модель пише весь `.md` самостійно
* Секційні промпти через `pi` з `--append-system-prompt`
* JS-оркестратор (Stage 0 екстракція → Stage 1 секційні промпти → Stage 3 зборка)

## Decision Outcome
Chosen option: "JS-оркестратор як точка входу", because інверсія керування виносить факти й структуру з-під моделі: JS детерміновано знає імена експортів, stdlib-vs-internal, JSDoc, поведінкові маркери (readOnly, catchesErrors, skips, caches) — і передає їх моделі як контекст, а не сподівається, що вона їх правильно витягне. Модель лишає тільки «перефразування відомих фактів у поведінкову прозу».

### Consequences
* Good, because transcript фіксує очікувану користь: Stage 0 екстрактор `docgen-extract.mjs` витягує per-function JSDoc для всіх 6 функцій `overlay-paths` — API-секція практично готова без LLM; правильна автоматична класифікація stdlib/internal; smoke-тест показав результат без витоку stdlib і без довільних сигнатур.
* Bad, because складність вища за one-shot (3 модулі: `docgen-extract.mjs`, `docgen-prompts.mjs`, `docgen-gen.mjs`); Stage 0 прив'язаний до JS/MJS (для `.vue`/`.py` потрібні окремі екстрактори або fallback до one-shot).

## More Information
Реалізація: `.worktrees/feat-docgen-orchestrator-pi/npm/skills/docgen/js/docgen-extract.mjs`, `docgen-prompts.mjs`, `docgen-gen.mjs`; гілка `feat/docgen-orchestrator-pi`; KV-cache оптимізація: `system + код файлу` — стабільний префікс, секційні інструкції — хвіст → ollama перевикористовує KV-cache, інгест платиться один раз; персистентна сесія між різними файлами — шкідлива (накопичення контексту), всередині одного файлу — доцільна.

## ADR Назва-alias `gemma4:4b` для `batiai/gemma4-e4b:q4` як якісний варіант на 8 GB

## Context and Problem Statement
Модель `batiai/gemma4-e4b:q4` (5.3 GB, q4-квантизація) показала якість ~92% проти ~85% у `gemma3:4b` на тих самих тестових файлах. Потрібно вирішити, як інтегрувати її в робочий набір і яку назву використовувати.

## Considered Options
* Використовувати `batiai/gemma4-e4b:q4` (оригінальна назва з Ollama Hub)
* Створити канонічний alias `gemma4-e4b:q4`
* Створити канонічний alias `gemma4:4b`

## Decision Outcome
Chosen option: "`gemma4:4b` (alias)", because коротка канонічна назва без vendor-префікса зручніша у CLI та pi-конфізі; `ollama cp` ділить blob-и (0 додаткового диску); оригінал `batiai/gemma4-e4b:q4` видалено після перевірки alias.

### Consequences
* Good, because transcript фіксує очікувану користь: `pi --list-models` і `ollama list` бачать `gemma4:4b`; alias підтверджений функціональним smoke-тестом (`ollama run gemma4:4b`); `~/.pi/agent/models.json` оновлено — `gemma4:4b` стоїть першою в провайдері `ollama`.
* Bad, because модель частково офлоадиться (56%/44% CPU/GPU, RAM 6.2 GB) → ~11 tok/s (~30 год на 1042 файли) проти ~20 tok/s у `gemma3:4b` (~14.5 год); вибір між якістю-first і швидкістю-first залишається ручним (через явний `--model`).

## More Information
Команди: `ollama cp batiai/gemma4-e4b:q4 gemma4:4b`; `ollama rm batiai/gemma4-e4b:q4`; конфіг: `~/.pi/agent/models.json` (providers.ollama.models: перша `gemma4:4b`, друга `gemma3:4b`); глобальний pi default (`~/.pi/agent/settings.json`) не змінено (лишається `openai-codex / gpt-5.5`) — щоб не перемикати весь інтерактивний pi-кодинг на локальну модель.
