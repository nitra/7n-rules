## ADR n-cursor як автономний CLI-оркестратор (fix, lint, taze)

## Context and Problem Statement

Поточна архітектура n-cursor: команди `fix`, `lint`, `taze` є **перевірниками**, які виводять порушення. Вся логіка «як виправляти» міститься у SKILL.md, який читає агент-LLM: 8-крокові інструкції, ескалація тирів, convergence-loop — все виконується агентом. Це означає що кожен агент повторно синтезує однакову стратегію і може зробити помилку у будь-якому кроці. Сесія вирішила перенести всю оркестраторну логіку з SKILL.md у CLI.

## Considered Options

* Залишити SKILL.md як джерело правди (агент-оркестратор, поточна модель)
* Зробити CLI-команди автономними оркестраторами (агент викликає одну команду і отримує exit 0/1)
* MCP-сервер як проміжний шар між агентом і CLI

## Decision Outcome

Chosen option: "CLI-команди як автономні оркестратори", because агент не повинен знати «як виправляти» — він викликає `npx @nitra/cursor fix` і отримує exit 0/1; уся складність (convergence-loop, T0-auto, ескалація тирів, check-gate) інкапсульована у CLI.

### Consequences

* Good, because SKILL.md скорочується до одного рядка (`npx @nitra/cursor fix`), агент не може допустити помилку в оркестрації.
* Good, because transcript фіксує очікувану користь: T0-auto закрив 2 порушення за 4s без LLM; orchestrator правильно ескалював haiku→sonnet після 2 провалів.
* Bad, because LLM-tier потребує налаштованого `pi` з cloud-моделлю; на машинах де налаштовано лише ollama — LLM-tier недоступний (підтверджено в тесті: gemma3:4b не повертає JSON).

## ADR fix --json видалено з публічного API

## Context and Problem Statement

`n-cursor fix --json` використовувався як зовнішній інтерфейс для CI, SKILL.md і T0-auto. Після переходу на автономний оркестратор, `fix` завжди виконує convergence-loop — окремий «check-only» режим у публічному API створює плутанину про семантику команди.

## Considered Options

* Залишити `fix --json` як публічний check-only режим
* Видалити `fix --json` з публічного API, замінити внутрішньою командою `_fix-check`

## Decision Outcome

Chosen option: "видалити `fix --json`, замінити `_fix-check`", because публічний API повинен мати єдину семантику: `fix` = виправити; внутрішня реалізація (orchestrator.mjs, t0.mjs) використовує `_fix-check` з підкресленням як private command.

### Consequences

* Good, because transcript фіксує очікувану користь: `bun npm/bin/n-cursor.js _fix-check js-run` повертає структурований JSON (підтверджено `total=1 failed=0`).
* Bad, because існуючий CI та скрипти що використовували `fix --json` потребують міграції на `_fix-check`.

## ADR LLM-tier через pi (C1 pattern): script збирає контекст, LLM повертає виправлений файл

## Context and Problem Statement

Попередня реалізація `llm-worker.mjs` використовувала Anthropic SDK напряму з `ANTHROPIC_API_KEY` і tool-use (read_file/write_file/run_command). Це прив'язує оркестратор до конкретного провайдера і означає що кожен користувач мусить мати `ANTHROPIC_API_KEY`.

## Considered Options

* Anthropic SDK напряму з tool-use (поточна реалізація)
* `pi` CLI як єдина точка виходу до LLM (кожен користувач налаштовує свій провайдер у `~/.pi/`)
* MCP-сервер з file-write tools через pi

## Decision Outcome

Chosen option: "pi CLI, C1 pattern", because `pi` є абстракцією над провайдерами — кожен користувач самостійно налаштовує ключі (`ANTHROPIC_API_KEY`, openai-codex, ollama). Оркестратор не залежить від конкретного провайдера. C1 pattern: script читає файли і формує self-contained prompt → `pi -p "..." --no-session --no-tools --mode text --model <model>` → script парсить відповідь і записує файли програмно.

### Consequences

* Good, because transcript фіксує очікувану користь: flow ескалації підтверджено (haiku × 2 → sonnet); pi відповів `OK.` на prewarm тест з gemma3:4b.
* Bad, because local 4B моделі (gemma3:4b, gemma4:4b) не повертають structured JSON надійно — LLM-tier потребує cloud-модель; підтверджено в тесті: `cannot parse pi response: Okay, I understand...`.

## ADR meta.json: прапор orchestrator для декларації скіла як автономного оркестратора

## Context and Problem Statement

Скіли мають різні типи поведінки: деякі є перевірниками, деякі — інтерактивними помічниками, а деякі (після рефакторингу) — автономними оркестраторами. Без явної декларації у `meta.json` неможливо програмно розрізнити ці типи.

## Considered Options

* Немає прапора — тип скіла визначається за вмістом SKILL.md
* `"orchestrator": true` у `meta.json`

## Decision Outcome

Chosen option: `"orchestrator": true` у `meta.json`, because декларативний прапор дозволяє скриптам і агентам визначити тип скіла без парсингу SKILL.md. Перший скіл з прапором: `npm/skills/fix/meta.json` → `{ "auto": "завжди", "worktree": true, "orchestrator": true }`.

### Consequences

* Good, because transcript фіксує очікувану користь: прапор стане сигналом для подальшого рефакторингу taze та lint за тим самим патерном.
* Bad, because transcript не містить підтверджених негативних наслідків.

## ADR Порядок виконання fix → lint: послідовно, не паралельно

## Context and Problem Statement

Коли агент завершує свою роботу у worktree і хоче запустити і `fix`, і `lint`, виникає питання: запускати їх послідовно в тому самому worktree чи паралельно у двох під-worktree з подальшим merge.

## Considered Options

* Послідовно в тому самому worktree: `fix` → `lint`
* Паралельно у двох під-worktree від поточного worktree з merge назад

## Decision Outcome

Chosen option: "послідовно в тому самому worktree", because `fix` і `lint` **не незалежні**: `fix` може створити ESLint config, extensions.json, package.json — файли які `lint` потім перевіряє або яких очікує. Паралельні під-worktree гарантовано конфліктують на спільних файлах (package.json, extensions.json, configs).

### Consequences

* Good, because відсутність конфліктів merge і правильна залежність між кроками.
* Bad, because повільніше ніж паралельне виконання.

## More Information

* `npm/skills/fix/meta.json`: `{ "auto": "завжди", "worktree": true, "orchestrator": true }`
* `npm/skills/fix/js/orchestrator.mjs`: convergence-loop `_fix-check → fix-t0 → llm-worker`
* `npm/skills/fix/js/llm-worker.mjs`: C1 pattern, env vars `N_CURSOR_FIX_MODEL_HAIKU`, `N_CURSOR_FIX_MODEL_SONNET`
* `npm/skills/fix/js/t0.mjs`: оновлено `fix --json` → `_fix-check`
* `npm/bin/n-cursor.js`: `case 'fix'` → orchestrator; `case '_fix-check'` → internal; `case 'fix-run'` → deprecated alias
* Тест T0-auto: yarn.lock + tsandall.opa — закриті детерміністично без LLM за < 5s
* Тест LLM-tier: flow підтверджено (detect → T0 pass → haiku × 2 → escalate sonnet); gemma3:4b не повертає JSON → потрібна cloud-модель
