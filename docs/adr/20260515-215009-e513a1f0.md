## ADR Розмежування ролей `/n-fix` і `/n-lint` через дедуплікацію кроку 6

**Контекст:** Скіл `/n-fix` мав крок 6, де окремо запускались `lint-js`, `lint-text`, `lint-style` — частина логіки, що дублювала роботу `/n-lint`. Виникло питання: чи варто злити ці команди в одну, чи достатньо прибрати дублювання.

**Рішення/Процедура/Факт:** Дедуплікація без злиття: крок 6 у `npm/skills/fix/SKILL.md` і `.cursor/skills/n-fix/SKILL.md` замінено на єдиний виклик `bun run lint` з делегуванням до `/n-lint`; явно заборонено дублювати лінт-логіку в `/n-fix`. Cross-reference «повний набір `lint-*` (навичка `n-fix`)» у `npm/rules/style-lint/style-lint.mdc:12` переписано на «`bun run lint` (навичка `/n-lint`)». Ролі зафіксовані: `/n-fix` = «структура проєкту відповідає `.cursor/rules/`»; `/n-lint` = «код чистий згідно `bun run lint`».

**Обґрунтування:** Злиття нав'язувало б overhead дорогого `bun run lint` до рідкісної структурної операції `/n-fix`, і навпаки — ускладнювало б частий сценарій «перевір код перед комітом» зайвими структурними кроками. Критично: обидві команди запускають ESLint, а `CLAUDE.md` прямо забороняє паралельні запуски ESLint; злиття збільшувало ризик випадкового дублювання. Дедуплікація усуває конкретне місце дублювання (крок 6) і зберігає чіткі межі.

**Розглянуті альтернативи:** Повне злиття в одну команду `/n-fix` — відхилено через різні частоти виклику та ризик порушення інваріанти «один ESLint на сесію».

**Зачіпає:** `npm/skills/fix/SKILL.md`, `.cursor/skills/n-fix/SKILL.md` (синк-копія, обидва tracked у git), `npm/rules/style-lint/style-lint.mdc`, `npm/CHANGELOG.md` (секція 1.11.6).
