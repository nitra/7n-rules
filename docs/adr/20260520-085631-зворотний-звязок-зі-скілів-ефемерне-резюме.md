# Зворотний Зв'язок зі Скілів через Ефемерне Резюме

**Status:** Accepted
**Date:** 2026-05-20

## Context and Problem Statement

Скіли `@nitra/cursor` (наприклад `n-lint`, `n-fix`) виконуються в споживацьких репозиторіях і накопичують цінний сигнал про «тертя» з правилами та SKILL.md пакета — неоднозначні кроки, відсутні autofix, хибні спрацювання. Цей сигнал зникає після сесії, бо жодного каналу назад у пакет не існувало.

## Considered Options

- Persistent-інбокс: чернетки `docs/cursor-feedback/<timestamp>.md` + Stop-хук + батч-скіл `/n-feedback` → `n-llm-patch` у репо `@nitra/cursor`
- Ефемерне резюме наприкінці скілу: агент аналізує тертя у чаті, нічого не пише у файли

## Decision Outcome

Chosen option: "Ефемерне резюме наприкінці скілу", because користувач явно обрав ефемерний варіант — він простіший в реалізації (одне `alwaysApply`-правило, нуль інфраструктури) і достатній для ручного перенесення спостережень у пакет через `/n-llm-patch`.

### Consequences

- Good, because нульова інфраструктура — жодного хука, інбоксу чи фази 2; агент, що щойно виконав скіл, має найточніший контекст для аналізу тертя.
- Bad, because відсутня персистентність: без накопичення неможливо бачити частоту повторення тих самих проблем між сесіями; якщо не зреагувати одразу — резюме втрачене.

## More Information

Реалізація: `npm/rules/feedback/feedback.mdc` (`alwaysApply: true`, version 1.0), синкпропуститий як `.cursor/rules/n-feedback.mdc`; `"feedback"` додано до `.n-cursor.json`→`rules`; `AGENTS.md` і `CLAUDE.md` оновлено; версія пакета підвищена до `1.13.59`. Схема звіту тертя (файл `npm/schemas/cursor-feedback.json`): `target` (rule|skill|check), `id`, `kind` (missing-check|ambiguous-doc|false-positive|no-autofix|recurring-pattern), `evidence`, `suggestion`. Інструкція міститься в одному файлі-правилі — не в кожному `SKILL.md` окремо — щоб не порушувати обмеження jscpd на дублювання. Для cross-repo маршруту сигналу у `@nitra/cursor` (якщо пізніше додати персистентність) обрано формат `n-llm-patch`.

## Update 2026-05-20

### Паралельна сесія: підхід B+C (n-llm-patch + Stop-hook)

Окрема сесія (5b23f892) розглянула ту саму проблему і обрала комбінацію B+C: скіл `n-llm-patch` забезпечує явний self-contained handoff, а stop-hook автоматично ловить архітектурні рішення з transcript без додаткових дій. Варіант A (`alwaysApply feedback.mdc`) розглядався як опція лише для миттєвого UX у чаті.

- Good, because після завершення відповіді агента очікується один draft ADR у `docs/adr/YYYYMMDD-HHMMSS-<session>.md`.
- Bad, because якщо сесія не містить блоку `## ADR` у відповіді LLM — `capture-decisions.sh` пропускає запис (зафіксовано в логу: `response missing '## ' header`).

Задіяні файли: `docs/adr/`, `.claude/hooks/capture-decisions.sh`, `.cursor/hooks.json`, `npm/skills/n-llm-patch/SKILL.md`. Stop-хуки: таймаути 180s / 600s згідно `.cursor/hooks.json`. Normalize-хук: cooldown `ADR_NORMALIZE_MIN_INTERVAL_HOURS=6`; для ручного обходу — `/n-adr-normalize`.
