## Чому ця ідея правильна

Проблема `lint` — не в тому, що він «важкий», а в тому, що він **змішує дві різні потреби**:

1. **Агент у сесії** (реактивний): тільки що написав код — хоче швидко перевірити TypeScript/ESLint/oxfmt/oxlint, виправити авто-фіксабельне, і продовжити роботу. Потрібна відповідь за ~10-20 секунд.

2. **CI / ізольована QA-гілка** (проактивний): повна перевірка безпеки, ga, js-run, jscpd, cspell, knip, markdownlint, rego — все це можна ганяти окремо від живої роботи агента.

Зараз один скіл `n-lint` намагається бути обома одночасно → агент або страждає від довгого чекання, або пропускає важкі перевірки через їхній вес.

## Як би я розділив

**`n-lint`** (легкий, реактивний — лишається як є для агента):
- ESLint (`lint-js`) + oxlint + oxfmt
- Можливо markdownlint (якщо швидко)
- **Не** включає: lint-ga, lint-security, lint-text, jscpd, cspell, knip, lint-rego, lint-docker
- `worktree: false` (реактивний — на незакомічені зміни агента)

**`n-lint-full`** (важкий, ізольований, для CI/QA-гілки):
- Весь поточний ланцюжок: все вищесказане + lint-ga + lint-security + lint-text + jscpd + cspell + knip + lint-rego + lint-docker + lint-k8s
- `worktree: true` (генеративний — ганяє на конкретній гілці ізольовано)
- Блокований `withLock` крос-worktree (вже реалізовано)

## Питання

Ти говорив «є `bun run lint`» — це зараз єдиний ентрипоінт для всього ланцюжка. Чи маєш на увазі:

- **(E1) Два окремих скіли** з різними іменами (`n-lint` та `n-lint-full`) — агент явно вибирає, що запускати;
- **(E2) Один скіл `n-lint` зі стратегіями** — параметр `--mode quick|full` або дві окремих точки входу в одному SKILL.md;
- **(E3) Перейменувати** поточний `n-lint` → `n-lint-full`, і написати новий легкий `n-lint` — мінімальна переробка, назви зберігають очевидність?

Я схиляюсь до **E1** або **E3** — явні окремі скіли з різними іменами зрозуміліші агенту (одна інструкція = одна відповідальність), ніж параметри. Але хочу почути тебе, бо це впливає на те, скільки `SKILL.md` файлів і `meta.json` нам зрештою писати.
