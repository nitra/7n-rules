## ADR Детермінований парсинг скілів — у CLI, а не в LLM

## Context and Problem Statement

Кілька скілів репо (`n-fix-tests`, `n-coverage-fix`, `n-taze` та інші) інструктували LLM-агента самостійно зчитувати та парсити великі артефакти (`COVERAGE.md` ~2.76 МБ, `package.json.taze-bak`) і класифікувати результати (semver major/minor, знаходити `survived`-блок тощо). Це витрачало ~700K токенів на оркестратора, де можна витратити 0 LLM-токенів, і в разі фан-ауту (122 групи мутантів) масштабувалось до десятків мільйонів токенів за ітерацію.

## Considered Options

* Залишити агента парсити артефакти безпосередньо (поточний підхід)
* Перенести весь детермінований парсинг у read-only CLI-екстрактор (`n-cursor <skill> index|slice|diff`), залишивши LLM лише когнітивну роботу

## Decision Outcome

Chosen option: "read-only CLI-екстрактор", because COVERAGE.md (~700K токенів) перевищує практичний контекст одного агента, а semver-класифікація та парсинг JSON-блоків — повністю детерміновані операції, що не потребують мовної моделі.

Правило закріплено як **пріоритетна вимога** в `.cursor/rules/scripts.mdc` (v1.13): детермінований парсинг / лічба / класифікація у скілі — вада, яку усуває скрипт; контракт — `n-cursor <skill> <verb>` повертає компактний `index` або `slice` під один елемент.

### Consequences

* Good, because `n-cursor coverage-fix index` повертає 7.4 КБ (~2K токенів) замість 2.76 МБ (~700K токенів) — **≈350× менше** для оркестратора; субагент отримує `slice` (~10 КБ) рівно під один файл.
* Good, because `n-cursor taze diff` класифікує major-оновлення детерміновано (caret-семантика) — LLM отримує готовий список і фокусується на CHANGELOG та рефакторингу.
* Good, because патерн відтворюється: `worktree` і `adr-normalize` уже є еталонами; `scripts.mdc` v1.13 містить red-flags і чек-лист перед фіналізацією будь-якого скіла.
* Bad, because transcript не містить підтверджених негативних наслідків (CLI-екстрактори додають новий шар коду, але він покривається тестами і не несе семантичної складності для агента).
