## ADR Спостережувані сліди як основний сигнал для аналізу AI-сесій

## Context and Problem Statement
Команда розглядає збір reasoning-логів з AI-кодинг інструментів (Claude Code, Codex, Cursor) для виявлення точок тертя та покращення правил/скілів проєкту. Виникло питання, який сигнал є надійним і портативним через усі інструменти.

## Considered Options
* Raw reasoning / thinking-блоки (де доступні)
* Спостережувані сліди: послідовності tool-calls, помилкові виходи, retry-ланцюги, token_count

## Decision Outcome
Chosen option: "Спостережувані сліди", because reasoning недоступний або непридатний у більшості транспортів: у Codex (OpenAI-провайдер) `encrypted_content` містить зашифроване reasoning, `summary` порожній; у Cursor немає lifecycle-hooks; лише Claude Code віддає сирий `thinking`. Спостережувані сліди (`function_call` / `tool_use` / `tool_result` / `token_count`) присутні в усіх інструментах, об'єктивні й прямо мапляться на конкретні правки в `.mdc`/`SKILL.md`/allowlist.

### Consequences
* Good, because transcript фіксує очікувану користь: дев'ять сигналів (серії tool-calls, помилки, retry-ланцюги, розвідка vs виконання, permission-prompts, token-вартість, відсутнє зчитування правила, обсяг дифу, покинуті ходи) детектуються кодом без LLM і портативні через Claude Code і Codex.
* Bad, because transcript не містить підтверджених негативних наслідків; reasoning зберігається як додатковий сигнал де доступний (Claude Code).

## More Information
- `~/.claude/projects/<proj>/*.jsonl` — `tool_use` / `tool_result` / `isSidechain` / usage
- `~/.codex/sessions/РРРР/ММ/ДД/rollout-*.jsonl` — `function_call` / `function_call_output` / `token_count`
- Приклад з реальної Codex-сесії: `{"type":"reasoning","summary":[],"encrypted_content":"gAAAAABq..."}`
- `npm/skills/docgen/js/docgen-compare-pi-vs-direct.mjs` — A/B скрипт pi-cli vs прямий omlx; кореляція транспорту з доступністю reasoning не вичерпана в transcript (сесія обірвалась)

## ADR Область збору трейсів AI-сесій: Claude Code + Codex без Cursor

## Context and Problem Statement
Для побудови колектора трейсів AI-сесій потрібно визначити, з яких інструментів збирати дані, враховуючи технічні обмеження доступу до логів.

## Considered Options
* Claude Code + Codex + Cursor
* Claude Code + Codex (без Cursor)

## Decision Outcome
Chosen option: "Claude Code + Codex (без Cursor)", because Cursor зберігає чати в SQLite (`state.vscdb`) у `workspaceStorage/<hash>/` — недокументований внутрішній формат; reasoning там лише summary; lifecycle-hooks відсутні. Користувач явно виключив Cursor зі списку.

### Consequences
* Good, because transcript фіксує очікувану користь: два джерела з відкритим форматом (JSONL) покривають основні AI-робочі потоки в проєкті.
* Bad, because Cursor-сесії залишаються за межами аналізу.

## More Information
- Claude Code: `~/.claude/projects/-Users-vitalii-www-nitra-cursor/*.jsonl`
- Codex: `~/.codex/sessions/РРРР/ММ/ДД/rollout-*.jsonl`
- Cursor SQLite: `~/Library/Application Support/Cursor/User/workspaceStorage/<hash>/state.vscdb` — виключено з scope

## ADR Фільтрація Codex-сесій за `session_meta.cwd` для проєктного scope

## Context and Problem Statement
Транскрипти Codex зберігаються глобально в `~/.codex/sessions/` по даті — усі проєкти в одній ієрархії директорій. Колектор, що читав би теку без фільтрації, тягнув би дані з чужих проєктів (наприклад, k8s-сесії потрапляли б в аналіз nitra/cursor).

## Considered Options
* Читати всі файли в `~/.codex/sessions/` за датою
* Читати всі файли, фільтрувати за `session_meta.cwd` (і/або `session_meta.git`) всередині кожного файлу

## Decision Outcome
Chosen option: "Фільтрувати за `session_meta.cwd`", because `session_meta` кожного rollout-файлу містить поля `cwd`, `git` та ін. Приклад: `cwd: /Users/vitalii/www/nitra/k8s` — не збігається з `CLAUDE_PROJECT_DIR`. Лише після перевірки `cwd == цей-репо` трейс можна зараховувати до проєктного аналізу.

### Consequences
* Good, because transcript фіксує очікувану користь: запобігає крос-проєктному забрудненню статистики (конкретний випадок: gcloud-сесія з k8s не потрапить до аналізу nitra/cursor).
* Bad, because transcript не містить підтверджених негативних наслідків; парсинг кожного файлу дорожчий за фільтрацію по директорії.

## More Information
- Структура `session_meta`: `{id, timestamp, cwd, originator, cli_version, source, thread_source, model_provider, base_instructions, dynamic_tools, git}`
- Знайдено в сесії: `cwd: /Users/vitalii/www/nitra/k8s`, `originator: Codex Desktop 0.136.0-alpha.2`
- Аналогія з Claude Code: там директорія проєкту = ключ (`-Users-vitalii-www-nitra-cursor`) — фільтрація не потрібна

## ADR Зберігання проєктних трейсів поза 30-денним вікном Claude Code

## Context and Problem Statement
Claude Code автоматично видаляє транскрипти після 30 днів (`cleanupPeriodDays` не заданий → дефолт). Для накопичення тривалої історії точок тертя потрібен окремий механізм збереження.

## Considered Options
* Підняти `cleanupPeriodDays` у `settings.json`
* Колектор, що регулярно копіює релевантні зрізи в проєктно-локальне сховище поза глобальною текою

## Decision Outcome
Chosen option: "Колектор із проєктно-локальним append-логом", because дані глобальні та мають ковзне 30-денне вікно; централізований збір у проєктний append-лог (поза git або в `docs/`) переживає авто-очистку й дає відтворювану довготривалу історію незалежно від дефолтів Claude Code.

### Consequences
* Good, because transcript фіксує очікувану користь: сировина накопичується пасивно й автоматично — «вмикати нічого не треба»; колектор лише дистилює вже наявне.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `cleanupPeriodDays` у `.claude/settings.json` — не задано (дефолт 30 днів)
- Наявний прецедент: `.claude/hooks/capture-decisions.sh` (Stop-hook, append-лог) та `normalize-decisions.sh` — той самий конвеєр «зафіксувати → LLM дистилює»
- Колектор має читати `~/.claude/projects/<цей-проєкт>/` і `~/.codex/sessions/**` (з фільтром `cwd`)
