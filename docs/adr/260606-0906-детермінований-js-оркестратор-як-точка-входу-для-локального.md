## ADR Детермінований JS-оркестратор як точка входу для локального docgen

## Context and Problem Statement

У рамках `n-docgen`-скіла (Tier 1) документація генерується one-shot викликом локальної моделі (`gemma3:4b`). Бенчмарки показали, що локальна 4B-модель систематично псує вивід: витікають сигнатури, stdlib-деталі та внутрішні імена; іноді обрізається вивід (overflow `num_predict`); виникають галюцинації в секції «Гарантії». Постало питання, чи підняти якість через кращу модель, чи через оркестрацію процесу генерації.

## Considered Options

* One-shot виклик через pi CLI (модель `gemma4:4b` — більша, якісніша)
* Детермінований JS-оркестратор: Stage 0 екстракція → секційні промпти → ollama → детерм. зборка (входна точка `docgen-gen.mjs`)

## Decision Outcome

Chosen option: "Детермінований JS-оркестратор як входна точка", because бенчмарки підтвердили, що головна причина витоків — не модель, а відсутність заземлення фактами; JS-оркестратор виносить з-під моделі всі детерміновані рішення (структура, імена, маркери, порядок секцій) і залишає їй лише перефразування, що підняло `gemma3:4b` з ~80% (one-shot) до ~86% (orch v2) при порівнянному часі.

### Consequences

* Good, because transcript фіксує очікувану користь: Stage 0 (`docgen-extract.mjs`) усуває клас помилок «вигадані назви / сигнатури / витік stdlib» — нуль токенів, детерміновано.
* Good, because секційно-мінімальний контекст v2 (код тільки у «Поведінці») усунув часовий регрес v1: `overlay-paths` 310 с → 77 с, `k8s-tree` 141 с → 55 с (швидше за one-shot).
* Good, because той самий `docgen-gen.mjs` працює з будь-якою моделлю через параметр `--model`; можна перемикатись між `gemma3:4b` (швидкість, 100% GPU) і `gemma4:4b` (якість, офлоад) без зміни оркестратора.
* Bad, because оркестрована `gemma3:4b` (~86%) досі відстає від `gemma4:4b` one-shot (~92%); Stage 2 детермінований пост-лінт ще не повністю виміряний (бенч із обома моделями обривається наприкінці transcript).
