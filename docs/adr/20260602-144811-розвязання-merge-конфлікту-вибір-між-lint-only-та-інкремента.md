## ADR Розв'язання merge-конфлікту: вибір між lint-only та інкрементальним coverage у verify-гейті

## Context and Problem Statement
Незавершений merge `feat/coverage-changed-gate → main` лишив маркери конфлікту у файлах `reviewer.mjs`, `flow.mdc` і `rust/coverage/coverage.mjs`, через що `=======` парсився як JS-токен і `flow` CLI падав на будь-якій підкоманді. Конфлікт відображав два протилежні архітектурні рішення щодо `DEFAULT_GATES` у `flow verify`.

## Considered Options
* Відновити HEAD — `git checkout HEAD -- <3 файли>` (lint-only verify, coverage повністю поза turnstile)
* Дорезолвити merge — `feat` для `reviewer.mjs`/`flow.mdc`, HEAD для `rust/coverage/coverage.mjs`

## Decision Outcome
Chosen option: "Дорезолвити merge", because інкрементальний `coverage --changed` (scoped до diff) зберігає safety net і вирішує проблему швидкості, заради якої main прибрав coverage; для `rust/coverage/coverage.mjs` обрано HEAD як новіший (містить `diffPath`/`--in-diff` + `baseline: skip` — feat зрегресував би API).

### Consequences
* Good, because `flow` CLI відновлено без втрати coverage-safety net; 330 тестів по всьому сліду merge зелені.
* Bad, because `coverage --changed` залежить від Stryker-env (окремий беклог-пункт); verify знову червоніє там, де Stryker не стартує локально.

## More Information
Файли: `npm/scripts/dispatcher/lib/reviewer.mjs`, `npm/rules/flow/flow.mdc`, `npm/rules/rust/coverage/coverage.mjs`, `npm/scripts/dispatcher/lib/tests/reviewer.test.mjs`. Коміт: `c091708`. `DEFAULT_GATES` у merged-стані: `['lint', 'coverage --changed']`.

## ADR flow release: авто-інференс воркспейсу із git-diff

## Context and Problem Statement
`flow release` без явного `--ws` передавав `n-cursor change` без параметра воркспейсу, через що change-файл лягав у корінь монорепо, навіть якщо всі зміни належали підпакету (наприклад, `npm/`). Беклог-пункт #9.

## Considered Options
* Потребувати явного `--ws` від користувача
* Інферувати `--ws` із `collectChangedFilesSince(base_commit)` + `getMonorepoProjectRootDirs` з fail-soft

## Decision Outcome
Chosen option: "Інферувати `--ws` із git-diff", because `base_commit` вже зберігається у flow-стані і дозволяє визначити змінений підпакет без участі користувача; при помилці інференсу (недосяжний base, кілька воркспейсів) поведінка деградує до існуючого дефолту.

### Consequences
* Good, because transcript фіксує очікувану користь: `release` автоматично пише change-файл у правильну `<ws>/.changes/` без `--ws`-прапора.
* Bad, because transcript не містить підтверджених негативних наслідків; `--ws=value` (inline-форма) і вкладені воркспейси виявлені під час `flow review` і виправлені ще до merge.

## More Information
Функції: `matchChangedWorkspaces`, `resolveChangeWsArgs` у `npm/scripts/dispatcher/lib/commands.mjs`. Семантика: один збіг → `--ws <ws>`; кілька → exit 1 без change; нуль / помилка → pass-through (дефолт `.`). Перевірка явного `--ws`: `rest.some(a => a === '--ws' || a.startsWith('--ws='))`. 27 тестів commands. Коміт: `282332c`.

## Considered Options
* Залишити поточний промпт (лише diff)
* Дозволити й зобов'язати читати referenced-файли через Read для верифікації cross-file тверджень

## Decision Outcome
Chosen option: "Дозволити й зобов'язати читати referenced-файли через Read", because рецензент вже має технічний доступ до Read; явна інструкція перевіряти перед репортом усуває клас хибних findings; scope обмежено змінами цього diff (pre-existing баги сусідніх файлів — не репортувати).

### Consequences
* Good, because transcript фіксує очікувану користь: findings «з diff не видно» мають бути виключені; рецензент верифікує cross-file твердження, а не здогадується.

## ADR coverage-gate: запуск Stryker через локальний bin (node-shebang) замість npx

## Context and Problem Statement
`runStryker` у `npm/rules/js-lint/coverage/coverage.mjs` запускав Stryker через `npx @stryker-mutator/core`, що завантажувало core у npx-кеш без сусіднього `vitest-runner` → `plugin-discovery` падав із «Cannot find TestRunner plugin vitest», блокуючи coverage-gate у `flow verify` для всіх воркспейсів.

## Considered Options
* `npx @stryker-mutator/core` (поточний підхід — кеш без плагінів)
* `bunx` (відкинуто раніше: завжди ставить пакет у `T/bunx-<uid>-<pkg>@latest`)
* Локальний core через `process.execPath` + `node` — відкинуто: під bun `process.execPath` вказує на bun-бінарник, а не node
* Локальний core через прямий запуск `bin/stryker.js` (node-shebang)

## Decision Outcome
Chosen option: "Прямий запуск `bin/stryker.js` (node-shebang)", because bin резолвиться через `package.json#bin` (subpath exports закриті — `ERR_PACKAGE_PATH_NOT_EXPORTED`), а його `#!/usr/bin/env node`-shebang гарантує node-рантайм незалежно від того, під яким runtime (node чи bun) запущений батьківський процес.

### Consequences
* Good, because transcript фіксує очікувану користь: e2e-валідація двічі підтверджує «Starting initial test run (vitest test runner…)», exit 0; plugin-discovery знаходить `vitest-runner` як сусід core у `node_modules/@stryker-mutator/`.
* Bad, because фікс активується в `flow verify` лише після merge у main (локальний `npx @nitra/cursor` виконує main-код через симлінк `node_modules/@nitra/cursor → ./npm`).

## More Information
Функція: `resolveLocalStrykerBin` у `npm/rules/js-lint/coverage/coverage.mjs`. Резолв: `createRequire(import.meta.url).resolve('@stryker-mutator/core/package.json')` → `dirname` → `join(bin[stryker])`. Мемоізація через модульний `let localStrykerBin`. 34 тести js-lint/coverage. Коміт: `373ce42`.
