# n-cursor release: власний скрипт замість @changesets/cli чи Beachball

**Status:** Accepted
**Date:** 2026-05-30

## Context and Problem Statement

У монорепо `@nitra/cursor` агенти, що працюють у паралельних worktree, конкурентно редагували спільний `CHANGELOG.md` і `package.json`, що призводило до merge-конфліктів. Потрібен release-флоу для scope B: будь-який споживач `@nitra/cursor` (включаючи Python-проєкти) отримує автоматизовану публікацію без конфліктів у CHANGELOG/version.

## Considered Options

* `@changesets/cli` — зрілий інструмент, власний формат change-файлів, підтримує лише JS/npm
* Beachball (Microsoft) — `check`/`publish`/JSON change-файли, JS-only; bump-типи (`patch/minor/major`) не збігаються з Keep-a-Changelog категоріями (`Added/Changed/Fixed/Removed`) без кастомного рендерера
* Власний скрипт у `@nitra/cursor` — повний контроль над форматом CHANGELOG, рідна підтримка Python і npm через наявний `package-manifest.mjs`

## Decision Outcome

Chosen option: "Власний скрипт у `@nitra/cursor`", because `@changesets/cli` і Beachball підтримують лише JS/npm, тоді як споживачі монорепо включають Python-workspace (`pyproject.toml`); крім того, обидва не підтримують Keep-a-Changelog категоризацію (`Added/Changed/Fixed/Removed`) без кастомного рендерера, а репо вже має власну інфраструктуру (`package-manifest.mjs`) з підтримкою npm і Python.

Суміжні рішення цієї ж сесії:
- **Авторство change-файлів — гібрид:** агент явно пише `.changes/<unique>.md` командою `n-cursor change`; CI-fallback синтезує generic-запис із комітів якщо файл відсутній, щоб реліз ніколи не переривався.
- **Реліз у CI на `main`:** worktree-агенти лишають лише `.changes/*.md`; агрегацію (bump, CHANGELOG, тег, push) виконує `n-cursor release` у CI після merge у `main`, серіалізовано через `concurrency.group`.

### Consequences

* Good, because нуль зовнішніх залежностей у кожному споживачі; рідна підтримка Python (`pyproject.toml`) і npm (`package.json`); повний контроль над форматом CHANGELOG.
* Good, because гібридне авторство: висока якість описів (агент задає намір) + реліз ніколи не переривається через відсутній change-файл (CI-fallback).
* Good, because CI-реліз серіалізує всі worktree-прогони — конфлікт версій/CHANGELOG зникає в корені.
* Bad, because більше коду доводиться писати і підтримувати власноруч порівняно з off-the-shelf рішенням.

## More Information

- Реалізовано в `npm/rules/release/` (нова директорія).
- Python-підтримка базується на `npm/rules/changelog/lib/package-manifest.mjs`.
- Fallback: `npm/rules/release/lib/fallback.mjs` (`synthesizeChangeFromCommits`) — логує warning, повертає `{ bump: 'patch', section: 'Changed', description }` з commit-меседжів.
- CI: `.github/workflows/npm-publish.yml` — `n-cursor release` перед `JS-DevTools/npm-publish`; `permissions.contents: write` для commit-back і тегування.
- Composite-action template для scope B: `npm/github-actions/release/action.yml`.
- Beachball детально розглядався: `bump`-типи (`patch/minor/major`) не збігаються з Keep-a-Changelog категоріями (`Added/Changed/Fixed/Removed`).
- Self-release першої версії: `@nitra/cursor@1.31.0`; 65/65 тестів pass.
