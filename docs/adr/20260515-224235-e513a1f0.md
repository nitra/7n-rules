## ADR Розмежування ролей `/n-fix` і `/n-lint` та структура `rules/<rule>/{fix,lint,policy}/`

**Контекст:** Скіл `/n-fix` містив крок 6, що запускав `lint-js` / `lint-text` / `lint-style` — тобто частково дублював роботу `/n-lint`. Одночасно директорія `npm/rules/<rule>/js/` змішувала в одному місці JS-перевірки для `npx @nitra/cursor check`, CLI-оркестратори `bun run lint-X` і допоміжні runner-и без чіткої межі між каналами.

**Рішення/Процедура/Факт:**
1. **Ролі скілів зафіксовано жорстко**: `/n-fix` = привести структуру проєкту у відповідність до `.cursor/rules/` (через `npx @nitra/cursor check`); `/n-lint` = привести код у відповідність до `bun run lint`. `/n-fix` не викликає `/n-lint` і не запускає жодного лінту — ці виміри незалежні.
2. **Структура `rules/<rule>/` переорієнтована за технологією реалізації**: `fix/<concern>/` — JS, що живить `npx @nitra/cursor check`; `lint/` — JS, що живить `bun run lint-X` через `n-cursor lint-X`; `policy/<concern>/` — Rego, що живить `npx @nitra/cursor check` (залишається незміненим). Принцип: *технологія реалізації визначає директорію*, а не призначення.
3. **Фаза 1 dual-mode**: `discover-checkable-rules.mjs` і `run-rule.mjs` оновлено для підтримки обох форматів `js/<concern>/` (legacy) і `fix/<concern>/` (новий) одночасно. Поле `rootDir: 'js' | 'fix'` додано до shape концерну. Дублікат концерну в обох директоріях одного правила — fatal-помилка з підказкою про незавершену міграцію.
4. **`npm/scripts/lint-conftest.mjs` видалено**: скрипт дублював `discoverCheckableRules` → `runConftestBatch`, яку `npx @nitra/cursor check` вже виконує. Всі посилання на `bun run lint-conftest` у `.mdc`, `.rego`, `package.json` замінені на `npx @nitra/cursor check` або видалені.
5. **`npm/README.md` оновлено**: секція «Структура пакету» документує нову `rules/<id>/{fix,lint,policy}/` ієрархію замість застарілої `mdc/`.

**Обґрунтування:** `policy/` залишається сиблінгом `fix/` і `lint/`, а не вкладається у `fix/policy/`, бо Rego і JS мають різні runner-и (`runConftestBatch` vs dynamic `import`), різні namespace-конвенції та різну `target.json`-логіку — змішування технологій в одній директорії збиває з пантелику. `lint-conftest.mjs` видалено, бо його єдина функція (ітерувати policy-концерни через `conftest test`) повністю покривається `npx @nitra/cursor check`, яка вже викликає `runConftestBatch` для всіх `policy/<concern>/target.json`. Dual-mode дозволяє мігрувати правила по одному без big-bang і зберігати тести зеленими на кожному кроці.

**Розглянуті альтернативи:** (1) Покласти policy у `fix/policy/` — відхилено: зайвий рівень nesting і розчинення rego-ecosystem у JS-сусідах; (2) Покласти policy плоско в `fix/<concern>/` поряд з `check.mjs` — відхилено: ризик колізії імен концернів між JS і Rego одного правила; (3) Залишити `policy/` зовні нової ієрархії (варіант A) — відхилено на користь принципу «технологія визначає директорію»; (4) Big-bang міграція замість поетапної — відхилено через ризик зламати всі тести одразу.
