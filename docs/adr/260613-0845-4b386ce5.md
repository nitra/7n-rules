## Два рівні incremental

### Рівень 1 — змінені в цьому PR/push (просто, надійно)

Відмінність між `pull_request` і `push` — різне джерело базового SHA:

```yaml
- name: Collect changed code files
id: changed
run: |
# PR: diff vs base branch; Push: diff vs commit before push
if [ "${{ github.event_name }}" = "pull_request" ]; then
BASE="origin/${{ github.base_ref }}"
else
BASE="${{ github.event.before }}"
fi
# guard: first push (before = 000...0) → full scan
if echo "$BASE" | grep -qE '^0+$'; then BASE="HEAD~1"; fi
FILES=$(git diff --name-only "$BASE"...HEAD \
| grep -E '\.(js|mjs|ts|vue|py|rs)$' \
| tr '\n' ' ')
echo "files=$FILES" >> $GITHUB_OUTPUT

- name: Lint doc
run: |
[ -z "${{ steps.changed.outputs.files }}" ] && { echo "No code files changed"; exit 0; }
bun run lint-doc ${{ steps.changed.outputs.files }}
```

### Рівень 2 — змінені з моменту останнього успішного прогону

GitHub Actions зберігає SHA кожного run. Отримуємо через `gh` CLI:

```yaml
- name: Find base SHA (last successful run)
id: base
env:
GH_TOKEN: ${{ github.token }}
run: |
SHA=$(gh run list \
--workflow="${{ github.workflow }}" \
--branch="${{ github.ref_name }}" \
--status=success --limit=1 \
--json headSha -q '.[0].headSha' 2>/dev/null || true)
# fallback: event.before (якщо ще не було успішних прогонів)
FALLBACK="${{ github.event.before }}"
echo "sha=${SHA:-$FALLBACK}" >> $GITHUB_OUTPUT

- name: Collect changed code files
id: changed
run: |
FILES=$(git diff --name-only "${{ steps.base.outputs.sha }}"...HEAD \
| grep -E '\.(js|mjs|ts|vue|py|rs)$' | tr '\n' ' ')
echo "files=$FILES" >> $GITHUB_OUTPUT
```

Потребує `permissions: actions: read` у job (для `gh run list`).

## Нюанс: зміни тільки в doc-файлах

Якщо хтось редагує `src/docs/foo.md` (без зміни `foo.mjs`), фільтр за розширеннями пропустить це. Два варіанти:

**A.** Передати в `lint-doc` ВСІ змінені файли (код + docs) і нехай `lint-doc` сам розсортує — чисто, але вимагає щоб `lint-doc [paths…]` розумів doc-шляхи та резолвив зворотно в source.

**B.** Залишити поточний `paths`-тригер workflow (`**/docs/**` вже є) — workflow запускається, але потім `lint-doc` отримає тільки code-файли. Якщо відповідний source-файл **не** змінився, CRC у doc все одно матчитиме → `lint-doc` не скаржиться. Фактично це безпечно: doc-редагування без зміни source = legit update, CRC перепишеться після `fix-doc`.

**Рекомендація для цього PR**: Рівень 1 з варіантом Б — `github.event.before` для push, `origin/${{ github.base_ref }}` для PR, фільтр тільки code-extensions. Рівень 2 ("since last success") — додавати лише якщо прогін на `main` регулярно охоплює кілька merged PR'ів і повний скан стає повільним. Зараз `lint-doc` — чистий детермінований скан (0 LLM), тому він навіть у повному режимі — секунди; incremental — оптимізація, не необхідність.

Підтверджуєш Рівень 1? Тоді можу записати це рішення в ADR і запустити Workflow для генерації документа.
