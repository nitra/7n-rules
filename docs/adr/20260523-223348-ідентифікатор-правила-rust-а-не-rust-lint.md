## ADR Ідентифікатор правила `rust`, а не `rust-lint`

## Context and Problem Statement
У пакеті `@nitra/cursor` нове правило для Rust-лінтингу треба назвати. Існує два очевидних варіанти: `rust-lint` (за аналогом `js-lint`/`style-lint`) або `rust` (за аналогом `vue`/`tauri`/`capacitor`).

## Considered Options
* `rust-lint` — парність з `js-lint`, `style-lint`
* `rust` — парність з `vue`, `tauri`, `capacitor`

## Decision Outcome
Chosen option: "`rust`", because користувач явно вказав парність з `vue`, `tauri`, `capacitor` — правила, де ідентифікатор відповідає екосистемі, а не дії.

### Consequences
* Good, because `npm/rules/rust/`, `.cursor/rules/n-rust.mdc` — коротко, без дублювання суфікса `-lint`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/rules/rust/`, `npm/rules/rust/rust.mdc`, `.cursor/rules/n-rust.mdc`. Аналоги: `npm/rules/tauri/`, `npm/rules/vue/`, `npm/rules/capacitor/`.

## ADR Rego-heavy архітектура нового правила `rust`

## Context and Problem Statement
Для правила `rust` треба обрати, як розподілити логіку перевірки між JS і Rego: JS-heavy (як у `tauri`), rego-heavy (як у `js-lint`/`style-lint`), або гібрид.

## Considered Options
* Rego-heavy — три rego-пакети: `package_json`, `vscode_extensions`, `lint_rust_yml`; `js/tooling/check.mjs` лише FS-existence + `runConftestBatch`
* JS-heavy — лише `policy/vscode_extensions/`; `package.json` і workflow перевіряються JS-логікою
* Гібрид — `package_json` і `vscode_extensions` у rego, workflow у JS

## Decision Outcome
Chosen option: "Rego-heavy", because правило `rust` має ті самі три документи (`package.json`, `.vscode/extensions.json`, `.github/workflows/lint-rust.yml`), що й `js-lint`/`style-lint`, тому дрейф від встановленої конвенції створить технічний борг.

### Consequences
* Good, because структура однорідна з `js-lint`/`style-lint`; шаблони (`template/*.snippet`) генеруються через rego-тест.
* Bad, because більше boilerplate-файлів (три `.rego` + три `_test.rego` + три `target.json`).

## More Information
Структура: `npm/rules/rust/policy/package_json/`, `npm/rules/rust/policy/vscode_extensions/`, `npm/rules/rust/policy/lint_rust_yml/`. Аналог: `npm/rules/js-lint/policy/`, `npm/rules/style-lint/policy/`.

## ADR Канонічна команда `lint-rust`: `--fix` локально, `--check` у CI

## Context and Problem Statement
Канонічний скрипт `scripts.lint-rust` у `package.json` та крок у `.github/workflows/lint-rust.yml` мають різну семантику — локально потрібна авто-корекція, в CI — тільки перевірка.

## Considered Options
* `fmt --check + clippy -D warnings` скрізь (жорсткий, без fix)
* `fmt (with fix) + clippy --fix + clippy -D warnings` — локально з auto-fix, CI без fix

## Decision Outcome
Chosen option: "`fmt (with fix) + clippy --fix + clippy -D warnings` локально, `fmt --check` + `clippy -D warnings` у CI", because користувач явно обрав цей варіант; локальний скрипт авто-виправляє, CI тільки верифікує.

### Consequences
* Good, because локальний `lint-rust` діє як `bun run lint-js` (виправляє і потім перевіряє суворо); CI завжди завершується помилкою за будь-якого порушення.
* Bad, because `cargo clippy --fix --allow-staged --allow-dirty` вимагає пояснення в `rust.mdc` — без нього флаги виглядають підозріло.

## More Information
Канон `package.json`: `cargo fmt --all && cargo clippy --fix --allow-staged --allow-dirty --all-targets --all-features && cargo clippy --all-targets --all-features -- -D warnings`. Канон CI: `cargo fmt --all -- --check` + `cargo clippy --all-targets --all-features -- -D warnings`. Spec: `docs/superpowers/specs/2026-05-23-rust-rule-design.md`.

## ADR Власність VSCode-розширення `rust-lang.rust-analyzer` — правило `rust`, не `tauri`

## Context and Problem Statement
`rust-lang.rust-analyzer` вимагався у `tauri.policy.vscode_extensions`. Після появи нового правила `rust` виникає дублювання: Tauri-проєкт завжди має `src-tauri/Cargo.toml`, тому обидва правила вимагатимуть одне й те саме розширення.

## Considered Options
* Залишити `rust-lang.rust-analyzer` у `tauri.policy.vscode_extensions`
* Перенести `rust-lang.rust-analyzer` до `rust.policy.vscode_extensions`, `tauri` вимагає лише `tauri-apps.tauri-vscode`

## Decision Outcome
Chosen option: "Перенести до `rust`", because `rust-lang.rust-analyzer` є Rust-generic (не Tauri-specific); `tauri`-проєкт активує правило `rust` автоматично через `src-tauri/Cargo.toml`, тому `rust-analyzer` приходить транзитивно.

### Consequences
* Good, because кожне правило вимагає лише власні розширення; rego-перевірки не дублюються.
* Bad, because `tauri.mdc` потребує пояснювального речення, щоб не виникло питання «чому rust-analyzer відсутній у tauri».

## More Information
Файли до зміни: `npm/rules/tauri/policy/vscode_extensions/vscode_extensions.rego` → `required_extensions := {"tauri-apps.tauri-vscode"}`. `npm/rules/tauri/policy/vscode_extensions/vscode_extensions_test.rego` — видалити кейси про `rust-analyzer`. `npm/rules/rust/policy/vscode_extensions/vscode_extensions.rego` — `required_extensions := {"rust-lang.rust-analyzer", "tamasfe.even-better-toml"}`.

## Context and Problem Statement
`.github/workflows/lint-rust.yml` потребує встановлення Rust toolchain у CI. Є кілька варіантів: `rustup` вручну, `dtolnay/rust-toolchain` action, або вбудований `actions-rust-lang/setup-rust-toolchain`.

## Considered Options
* `dtolnay/rust-toolchain@stable` безпосередньо у `lint-rust.yml`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`dtolnay/rust-toolchain@stable` + `Swatinem/rust-cache@v2`", because `dtolnay/rust-toolchain` є індустрійним стандартом для Rust CI; `Swatinem/rust-cache@v2` доданий додатково, бо `dtolnay/rust-toolchain` не кешує `target/`/registry — без нього повторні запуски витрачають кілька хвилин.

## More Information
Канонічна послідовність steps у `.github/workflows/lint-rust.yml`: `actions/checkout@v6` → `dtolnay/rust-toolchain@stable` (components: `rustfmt, clippy`) → `Swatinem/rust-cache@v2` → `cargo fmt --all -- --check` → `cargo clippy --all-targets --all-features -- -D warnings`.
