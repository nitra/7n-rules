## ADR Спостережувані сліди як основний сигнал для покращення проєкту

## Context and Problem Statement
Проєкт накопичує LLM-сесії через кілька інструментів (Claude Code, Codex, Cursor) і потребує сигналів для виявлення «де правила/скіли недовизначені». Розглядалась можливість аналізу reasoning-тексту моделей, але провайдери або шифрують його (Codex/OpenAI — `encrypted_content`), або він неструктурований (Cursor).

## Considered Options
* Reasoning-текст моделі як основний сигнал
* Спостережувані сліди (tool-calls, ok/err, latency, retry, token cost) — портативні через всі інструменти

## Decision Outcome
Chosen option: "Спостережувані сліди", because reasoning недоступний або зашифрований у Codex/Cursor, тоді як сліди є у всіх трьох інструментах, об'єктивні (дія сталася або ні), дешеві для детектування без LLM, і прямо маппяться на конкретні правки `.mdc`/`SKILL.md`/allowlist.

### Consequences
* Good, because transcript фіксує очікувану користь: 9 виявлених сигнальних патернів (серії однотипних дій, retry-after-failure, висока частка розвідки, permission-промпти, аномальна вартість, тощо) прямо перетворюються на конкретні правки в `CLAUDE.md` або `.mdc`.
* Bad, because reasoning як вторинний канал (де доступний — `message.reasoning_content` на omlx/Qwen3-Thinking) все одно захоплюється в trace; але для хмарних pi-викликів thinking-текст відсутній, тож картина неповна.

## More Information
Таксономія 9 сигналів задокументована в transcript. Перевірено на реальному rollout-файлі Codex (`~/.codex/sessions/2026/06/05/rollout-*.jsonl`): 17 дій, 5 помилок, серія `gcloud logging read ×6` (агент намацував GCP-параметри). Cursor виключений зі scope (SQLite `state.vscdb`, відсутність hooks). Файли: `npm/lib/omlx-trace.mjs`, `npm/lib/llm.mjs`.

## ADR Єдина точка `callLlm` для wire-trace (Option C)

## Context and Problem Statement
Після додавання прямих `callOmlx`-викликів у кількох скілах (docgen, fix, coverage) виникла потреба захопити wire-trace з reasoning та usage. Перша версія спеки цілила в `callOmlx` як точку перехоплення (Option A — local-only), але `npm/lib/llm.mjs` вже існував як спільний шар над обома бекендами з мінімальним trace.

## Considered Options
* Option A — wrapper навколо `callOmlx` (local omlx-only, хмарні pi-виклики сліпі)
* Option B — окремі перехоплення в omlx-гілці та pi-гілці по всіх скілах
* Option C — збагатити єдиний `callLlm` над обома бекендами, мігрувати прямих callerів

## Decision Outcome
Chosen option: "Option C — єдиний `callLlm`", because один чокпойнт для обох бекендів, нуль рефактору на рівні скілів (вони вже через `callLlm`), і rich-поля з omlx (`reasoning_content`, `usage`) піднімаються всередині без розкидання логіки по callerах. Прямі callerи (`fix/llm-worker.mjs`, `coverage-classify/index.mjs`) мігровані на `callLlm` з `opts.caller` для атрибуції.

### Consequences
* Good, because transcript фіксує очікувану користь: `fix`, `coverage`, `docgen` тепер трасуються одноманітно; `caller`-поле дає атрибуцію без зміни скілів.
* Bad, because pi-гілка дає деградований запис (prompt/response/ms/ok, без `usage`/`reasoning`), бо `spawnSync('pi')` не повертає structured metadata. Transcript це явно фіксує.

## More Information
Файли: `npm/lib/llm.mjs` (заміна наявного shallow `trace()` блоку), `npm/lib/omlx.mjs` (новий `callOmlxRaw` з rich-return; `callOmlx` = тонка string-обгортка), `npm/lib/omlx-trace.mjs` (новий). Мігровані: `npm/skills/fix/js/llm-worker.mjs`, `npm/scripts/coverage-classify/index.mjs`. ADR 260610-2228 (в коді) — попередній архітектурний запис про `callLlm`.

## ADR Дворівнева raw→aggregate модель зберігання wire-trace

## Context and Problem Statement
Wire-trace накопичує повні `messages` (з вихідним кодом файлів), reasoning і usage на кожен виклик. Потрібно зберігати сировину локально і при цьому не роздувати git-репозиторій, але накопичені знання (агрегати після batch-аналізу) мають зберігатись назавжди і відстежуватись через git.

## Considered Options
* Gitignored лог з деструктивною ротацією (перезапис `.1`)
* Дворівнева модель: gitignored raw з недеструктивною ротацією + git-committed aggregate

## Decision Outcome
Chosen option: "Дворівнева модель raw→aggregate", because деструктивна ротація знищувала б дані до batch-агрегації (суперечить меті «зберігати назавжди»); а git-commit сирого потоку роздував би репозиторій вихідним кодом файлів у кожному коміті.

### Consequences
* Good, because transcript фіксує очікувану користь: `.n-cursor/llm-trace.jsonl` gitignored → не роздуває репо; недеструктивна ротація (`llm-trace.<seq>.jsonl`) → дані доживають до агрегації; `docs/omlx-insights/` в git → дистильовані знання відстежуються назавжди.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Raw: `.n-cursor/llm-trace.jsonl` (gitignored через `.gitignore`, ротація 50 MB, недеструктивні нумеровані архіви). Aggregate: `docs/omlx-insights/` — committed до git (рішення підтверджено явно в transcript). Логіка ротації в `npm/lib/omlx-trace.mjs`: `rotateIfNeeded`. Kill-switch: `N_CURSOR_OMLX_TRACE=0`.

## ADR Стратегія витягування reasoning із omlx-відповіді

## Context and Problem Statement
Живий тест `Qwen3-4B-Thinking-2507-4bit` виявив дві форми reasoning у відповідях omlx-сервера: структурне поле `message.reasoning_content` (коли `max_tokens` достатній) і витік у `content` без тегів (коли `finish_reason === 'length'`). Потрібна стратегія, яка правильно класифікує кожен варіант.

## Considered Options
* Тільки поле `message.reasoning_content`
* Ієрархічна: поле → `<think>`-теги → маркер `truncated`

## Decision Outcome
Chosen option: "Ієрархічна стратегія з маркером truncated", because живий тест показав: при `max_tokens:256` `reasoning_content` порожнє, думки виявились у `content` без `<think>`-тегів, і лише `finish_reason === 'length'` сигналізує про обрізку. Без цього маркера факт обрізаного thinking губиться.

### Consequences
* Good, because transcript фіксує очікувану користь: поле `reasoning_source` (`field` / `think_tag` / `truncated` / `none`) дозволяє відрізнити повний reasoning, витягнутий через теги, обрізаний і відсутній.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовано в `npm/lib/omlx.mjs`: функція `extractReasoning(message, finishReason)`. Перевірено: gemma-4 повертає `reasoning_source:none` (не thinking-модель); Qwen3-Thinking при достатньому `max_tokens` — `reasoning_source:field` з непорожнім `reasoning`. Smoke-тест з кореня репо підтвердив запис обох полів у `.n-cursor/llm-trace.jsonl`.
