## ADR Атомарний примітив блокування через `mkdirSync` на macOS

## Context and Problem Statement
Проєкт потребує механізму взаємного виключення для важких перевірок (lint, check тощо), що запускаються паралельно різними агентами на macOS. Системна команда `flock` на macOS відсутня — потрібна альтернатива, яка не вносить сторонніх залежностей.

## Considered Options
* Атомарний `mkdirSync` (APFS-гарантія унікальності)
* `shlock` (`/usr/bin/shlock`, BSD-утиліта, наявна на mac)
* npm-пакет `proper-lockfile`

## Decision Outcome
Chosen option: "Атомарний `mkdirSync`", because transcript фіксує: `shlock` дає лише взаємне виключення — дедуп усе одно писати самим, і він вносить залежність від зовнішнього бінарника та `spawn`; `proper-lockfile` — зайва залежність заради ~150 рядків, без підтримки дедупу, з непотрібною кросплатформністю. `mkdirSync` на APFS атомарний, дає повний контроль над JSON-станом дедупу.

### Consequences
* Good, because transcript фіксує очікувану користь: мінімум рухомих частин, дедуп природно лягає в той самий код без зовнішніх залежностей.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл реалізації: `npm/scripts/utils/with-lock.mjs`. Дир-лок `<key>.lock/` містить `owner.json` `{pid, host, startedAt, fingerprint}`. Живість PID перевіряється через `process.kill(pid, 0)`. Стан: `node_modules/.cache/n-cursor/<key>/`. Застарілий лок (мертвий PID або вік > 30 хв) автоматично видаляється. Спека: `docs/superpowers/specs/2026-05-22-lint-ga-concurrency-lock-design.md`.

## ADR Поведінка при зайнятому локу: очікування з дедуплікацією

## Context and Problem Statement
Коли агент Б намагається запустити важку команду, а лок вже тримає агент А, потрібно визначити, що Б має робити: завершитись із помилкою, пропустити виконання чи чекати.

## Considered Options
* Чекати в черзі (poll-and-wait) і запустити після звільнення локу
* Завершити з помилкою (fail fast)
* Пропустити виконання тихо (exit 0)
* Чекати і дедуплікувати (якщо A успішно завершив із тим самим станом — Б пропускає)

## Decision Outcome
Chosen option: "Чекати + дедуплікувати", because transcript фіксує явний вибір користувача.

### Consequences
* Good, because transcript фіксує очікувану користь: Б не втрачає свіжий результат А — або повертає його через дедуп, або запускається з актуальним станом після звільнення.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Таймаут очікування — 20 хв, після чого лог-попередження і виконання без локу (щоб не вічно висіти). Дедуп спрацьовує лише при `exitCode === 0` — невдалий прогон завжди перезапускається. `npm/scripts/utils/with-lock.mjs`, спека `docs/superpowers/specs/2026-05-22-lint-ga-concurrency-lock-design.md`.

## ADR Гранулярність локу: окремий лок на кожну команду

## Context and Problem Statement
Потрібно вирішити, чи всі важкі перевірки (`n-lint`, `n-check`, `n-start-check`) виключають одна одну, чи кожна команда має власний незалежний лок.

## Considered Options
* Один глобальний лок (лише одна важка команда одночасно на весь проєкт)
* Лок на команду (`lint-ga` і `check` можуть йти паралельно)

## Decision Outcome
Chosen option: "Лок на команду", because transcript фіксує явний вибір користувача — різні команди не конкурують між собою.

### Consequences
* Good, because transcript фіксує очікувану користь: ключ `<key>` задає незалежні простори виключення, паралельні команди різного типу не блокують одна одну.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`withLock('lint-ga', fn)`, `withLock('check', fn)` — незалежні. Стан: `node_modules/.cache/n-cursor/lint-ga/` та `node_modules/.cache/n-cursor/check/` окремо. Спека `docs/superpowers/specs/2026-05-22-lint-ga-concurrency-lock-design.md`.

## ADR Чинність кешу: fingerprint робочого дерева + TTL

## Context and Problem Statement
При дедуплікації треба вирішити, за якою умовою результат попереднього прогону вважати «ще чинним».

## Considered Options
* Хеш робочого дерева (git HEAD + diff + untracked)
* TTL-таймер без хешу
* Лише «git clean» (тільки якщо нема незакомічених змін)
* Хеш + TTL (обидва критерії разом)

## Decision Outcome
Chosen option: "Хеш + TTL", because transcript фіксує явний вибір користувача.

### Consequences
* Good, because transcript фіксує очікувану користь: сценарій «N субагентів лінтують однаковий стан» дає ідеальний збіг fingerprint → дедуп без повторного запуску; TTL захищає від використання застарілого кешу при збігу хешу з давнім станом.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Fingerprint = sha256(`git rev-parse HEAD` + `git diff HEAD` + вміст untracked-файлів). Поза git-репо → fingerprint `null` → дедуп вимкнено, лок залишається. TTL: 10 хв (константа в `with-lock.mjs`, пізніше може бути винесена в `.n-cursor.json`). Стан: `<key>.result.json` `{finishedAt, exitCode, fingerprint}`. Модуль: `npm/scripts/utils/worktree-fingerprint.mjs`.

## ADR Лок вбудований у функцію (не окрема команда-обгортка)

## Context and Problem Statement
Спочатку розглядалась окрема підкоманда `n-cursor guard <key> -- <команда>`, яку скіли мали б явно викликати. Користувач зазначив, що скіл може забути її викликати — і агент запустить `bun run lint` напряму, обійшовши лок.

## Considered Options
* Окрема команда-обгортка `n-cursor guard <key> -- <cmd>` (скіли змінюються)
* Лок вбудований у саму CLI-функцію перевірки (скіли і `package.json` не змінюються)
* Лок через start-hook (примусово для всіх запусків)

## Decision Outcome
Chosen option: "Лок вбудований у функцію", because transcript фіксує: користувач прямо відхилив окрему команду («навіщо мені нова команда якщо я хочу щоб лок був в кожній js перевірці»), hook відхилений як механізм інтеграції.

### Consequences
* Good, because transcript фіксує очікувану користь: скіли, `package.json`, агенти не змінюються — лок спрацьовує при будь-якому способі запуску `lint-ga`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізація: поточна логіка `runLintGaCli` в `npm/rules/ga/lint/lint.mjs` переноситься в приватну `runLintGaSteps()`, публічний експорт стає `export const runLintGaCli = () => withLock('lint-ga', runLintGaSteps)`. Виклик у `n-cursor.js` case `'lint-ga'` не змінюється.

## ADR Пілот механізму локу на `lint-ga`

## Context and Problem Statement
Механізм локу планується застосувати до кількох команд (`lint-ga`, `lint-rego`, `lint-text`, `lint-k8s`, `lint-docker`, `check`). Потрібно вирішити, чи вносити його одразу всюди, чи поетапно.

## Considered Options
* Вбудувати лок одразу в усі команди
* Почати з пілоту на одній команді (`lint-ga`)

## Decision Outcome
Chosen option: "Пілот на `lint-ga`", because transcript фіксує явний вибір користувача («давай почнемо з того що додамо тільки в наш скрипт lint-ga»).

### Consequences
* Good, because transcript фіксує очікувану користь: підхід перевіряється на реальній команді; той самий зразок (`withLock`) потім розкочується на решту команд.
* Neutral, because transcript не містить підтвердження наслідку щодо ризику неузгодженості між командами до повного розкотювання.

## More Information
Наступні команди для розкотювання (за спекою): `lint-rego`, `lint-text`, `lint-k8s`, `lint-docker`, `check`. Спека зафіксована в `docs/superpowers/specs/2026-05-22-lint-ga-concurrency-lock-design.md`, закомічено як `fd8d471`.
