## ADR: `template/`-директорія як єдине джерело канону merge-фрагментів у правилах `npm/rules/`

**Контекст:** У правилах `npm/rules/<id>/` канонічні значення для перевірки (рядки у `package.json`, поля `.gitleaks.toml`, YAML-фрагменти тощо) були розкидані у трьох місцях одночасно: у `.mdc` як fenced-блоки (для AI-контексту), у `fix/<concern>/check.mjs` як inline-regex/literals і у `policy/<concern>/*.rego` як рядкові константи. Зміна будь-якого канонічного значення вимагала синхронного оновлення у 2–3 місцях.

**Рішення/Процедура/Факт:**
Введено архітектурний патерн `template/` — каталог усередині кожного концерну (`policy/<concern>/template/` як canonical home; `fix/<concern>/template/` для FS-only концернів без Rego) із трьома семантично розрізненими файлами:

- `snippet.json` — required merge-фрагмент у синтаксисі цільового файлу; семантика: equals для leaf-значень, subset-of для масивів.
- `deny.json` — заборонені шляхи; наявність будь-якого ключа у реальному файлі = fail; значення-листи = опис помилки.
- `contains.json` — масиви підрядків, кожен з яких має бути присутнім у відповідному leaf цільового файлу.

Нові утиліти:
- `npm/scripts/utils/template-loader.mjs` — завантажує template для концерну за шляховою конвенцією.
- `npm/scripts/utils/template-checks.mjs` — pure-функції `checkSnippet`, `checkDeny`, `checkContains`.
- `runConftestBatch` розширено опціональним `templateData`, що передається як `--data <tempfile>` у conftest; tempfile чиститься після запуску.

Rego-полісі отримують дані через `data.<ruleId>.<concernId>.template.*` (завантажує runner). JS-`check.mjs`-и стають тонкими: FS-логіка залишається inline, leaf-перевірки — через `checkSnippet`/`checkContains`.

У `.mdc` inline fenced-блоки зі snippets замінюються на посилання `[template/](./template/)` — без дублювання канону.

Міграція поетапна всередині одного branch: Phase 0 (інфраструктура + unit tests), Phase 1 (pilot на `security`), Phase 2–4 (batch за типом target), Phase 5 (classify `k8s`/`abie` — non-eligible), Phase 6 (оновлення `.mdc`).

**Обґрунтування:** Єдине джерело істини знімає drift між `.mdc`, `check.mjs` і `*.rego`. Зміна канонічного значення (наприклад нова версія команди `lint-security`) — одна правка в `snippet.json`, яка автоматично підхоплюється і Rego, і JS-перевіркою, і AI-контекстом через посилання. Concern-level розташування (а не rule-level) обрано після аналізу реальних конфліктів: у `k8s` 4 концерни матчать один і той самий `*.yaml` glob з різними очікуваннями, у `npm-module` 2 концерни на `package.json` — rule-level template не може їх розрізнити.

**Розглянуті альтернативи:**
- *Rule-level `template/`* — розглядався (файли типу `template/package.json.snippet.json`); відхилено через виявлені collision-cases: `k8s` і `npm-module` мають кілька концернів на один basename.
- *Concern-level template лише для `policy/`* — variant concern-level `policy/<concern>/template/` без `fix/<concern>/template/`; відхилено на користь fallback-конвенції, що дозволяє FS-only концернам теж мати template без примусового Rego.
- *Expectations DSL (один YAML per rule)* — розглядався; відхилено через необхідність нового DSL, незрозумілого AI без явного навчання.

**Зачіпає:** `npm/scripts/utils/run-conftest-batch.mjs`, `npm/scripts/utils/run-rule.mjs`, усі `npm/rules/*/policy/<concern>/` і `npm/rules/*/fix/<concern>/` (Rego + JS), усі `npm/rules/*/<id>.mdc`; нові файли: `template-loader.mjs`, `template-checks.mjs`.
