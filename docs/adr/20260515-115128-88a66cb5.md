Ось правильна симетрична структура:

```
npm/rules/<id>/
├── <id>.mdc                     ← опис для агентів
├── auto.md                      ← (опційно) умови авто-додавання в .n-cursor.json
│
├── js/                          ← (опційно) імперативна частина
│   └── <name>/
│       ├── check.mjs            ← exports check(): Promise<0|1>
│       │                           exports applies?(): Promise<boolean>  ← гейт concern
│       └── *.test.mjs           ← пропускається CLI
│
└── policy/                      ← (опційно) декларативна частина
└── <name>/
├── <name>.rego
├── <name>_test.rego
└── target.json          ← маніфест файлів для conftest
```

Спільні хелпери → `npm/scripts/utils/` (вже існує, не дублюється всередині правил).

## Чому саме так

**Симетрія повна.** CLI робить одне і те ж для обох каталогів: `readdir` → директорії → конвенційний файл усередині (`check.mjs` / `target.json`). Немає різниці в логіці discovery.

**Ізоляція concern.** `js/firebase/check.mjs` і `js/k8s-base/check.mjs` — незалежні модулі. Кожен сам вирішує через `applies()`, чи він доречний. Shared логіка (наприклад `findK8sYamlFiles` з кешем) → `npm/scripts/utils/`.

**`applies()` — per-concern, а не rule-level.** Якщо треба заблокувати все правило одним гейтом (як `rego` → «немає `.rego` файлів»), додається один concern `js/tooling/check.mjs` що повертає `applies() = false`, і всі інші concerns (і policy-targets того ж правила) теж перевіряють ту саму умову через спільний хелпер. Rule-level синглтон — не потрібен.

## Discovery в CLI (симетрично)

```js
async function discoverCheckableRules() {
for (const ruleDir of await readdir(BUNDLED_RULES_DIR)) {
const jsConcerns = existsSync(join(ruleDir, 'js'))
? (await readdir(join(ruleDir, 'js'), { withFileTypes: true }))
.filter(d => d.isDirectory() && existsSync(join(ruleDir, 'js', d.name, 'check.mjs')))
.map(d => d.name).toSorted()
: []

const policyConcerns = existsSync(join(ruleDir, 'policy'))
? (await readdir(join(ruleDir, 'policy'), { withFileTypes: true }))
.filter(d => d.isDirectory() && existsSync(join(ruleDir, 'policy', d.name, 'target.json')))
.map(d => d.name).toSorted()
: []

if (jsConcerns.length || policyConcerns.length) {
out.push({ id: ruleDir.name, jsConcerns, policyConcerns })
}
}
}
```

Ключове: обидва блоки мають однаковий shape — `readdir → filter-by-directory → filter-by-convention-file → toSorted`.
