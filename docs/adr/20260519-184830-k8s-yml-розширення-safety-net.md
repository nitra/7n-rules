# Обробка `.yml`-розширення у правилах `k8s`: safety-net замість тихого пропуску

**Status:** Accepted
**Date:** 2026-05-19

## Context and Problem Statement
Правила `check k8s` і `lint-k8s` обробляли обидва розширення — `.yaml` і `.yml`. У проєкті канонічне розширення для k8s — лише `.yaml`; файли `.yml` заборонені окремим правилом. Виникло питання: чи повинен `check k8s` тихо ігнорувати `.yml`-файли під `k8s/`, чи активно повідомляти про потребу перейменування.

## Considered Options
* Фільтрувати лише `*.yaml` — тихо ігнорувати `.yml`
* Пропускати `.yml` через обхідник, але одразу завершувати `checkK8sYamlFile` з fail-повідомленням про перейменування

## Decision Outcome
Chosen option: "Пропускати `.yml` через обхідник із fail-повідомленням", because тихий skip дозволяв `.yml`-файлам під `k8s/` уникати будь-якої перевірки — safety-net з явним `fail(...)` є єдиним gate для випадково доданих файлів.

### Consequences
* Good, because `.yml`-файли не залишаються непоміченими — `check k8s` одразу вказує на потребу перейменування.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `YAML_EXTENSION_RE` і `K8S_YAML_EXT_RE` у `npm/rules/k8s/fix/manifests/check.mjs` залишено як `/\.ya?ml$/iu`; `target.json` для rego-policies (`manifest/`, `base_manifest/`, `gateway/`, `hpa_pdb/`) змінено на `"**/k8s/**/*.yaml"` (rego не валідує розширення — fail відбувається раніше, у JS).
- `base_kustomization.rego`: `is_hpa_or_pdb_filename` позбавлено `.yml`-варіантів.
- `npm/package.json` `1.13.53` → `1.13.54`; `npm/CHANGELOG.md` запис `[1.13.54]`.

## Update 2026-05-19

Поточне рішення (safety-net `fail` для `.yml`) замінило початково розгляпропуститий підхід повного видалення підтримки `.yml`: у першій ітерації `YAML_EXTENSION_RE` змінювалось на `/\.yaml$/iu`, а `.yml`-файли просто не потрапляли в обхідник (`target.json` → `["**/k8s/**/*.yaml"]`). Після аналізу обрано safety-net з явним fail-повідомленням, щоб уникнути тихого пропуску випадково доданих `.yml`-файлів.
