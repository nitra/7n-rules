## ADR Схема іменування change-файлів у монорепо

## Context and Problem Statement
Потрібно було визначити унікальну схему іменування `.changes/*.md`-файлів, щоб уникнути колізій у worktree-сценаріях, де два паралельні агенти можуть писати change-файли одночасно.

## Considered Options
* Хеш контенту файлу
* `<branch>-<n>` (детермінована схема)
* `<timestamp>-<short-rand>` (timestamp + короткий рандомний суфікс)

## Decision Outcome
Chosen option: "`<timestamp>-<short-rand>.md`", because timestamp забезпечує читабельний порядок, а рандомний суфікс усуває колізії за однакової мілісекунди у паралельних worktree.

### Consequences
* Good, because файли природньо сортуються хронологічно, а ймовірність колізій практично нульова.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схема реалізована в `npm/rules/release/lib/change-file.mjs` (функція `newChangeFileName`). Рішення закрило відкрите питання зі spec-документа `docs/superpowers/specs/2026-05-29-n-cursor-release-design.md`.

## ADR Git-теги per-package у монорепо (`<name>@<version>`)

## Context and Problem Statement
У монорепо з незалежним версіонуванням по workspace потрібно вирішити: ставити окремий git-тег на кожен опублікований пакет (`<name>@<version>`) чи один тег на весь CI-прогін (`release-<timestamp>` тощо).

## Considered Options
* Per-package тег: `<name>@<version>` (канон changesets/Beachball/Lerna independent mode)
* Один тег на реліз: `release-<timestamp>` / `v2026.05.29`

## Decision Outcome
Chosen option: "Per-package тег `<name>@<version>`", because він семантично відповідає незалежній per-workspace моделі версіонування та безкоштовно надає базу для fallback-синтезу запису з комітів (через `git describe --match '<name>@*'`).

### Consequences
* Good, because `git log <name>@<prev>..HEAD` дає точний commit-range для fallback і для майбутніх GitHub Releases / провенансу.
* Bad, because для scoped-пакетів (`@nitra/cursor`) тег стає `@nitra/cursor@1.33.0`, що створює вкладений refs/tags-шлях — стандартна поведінка changesets, але може дивувати.

## More Information
Тег ставиться в `npm/rules/release/release.mjs`. Fallback-синтез реалізований у `npm/rules/release/lib/fallback.mjs`, де `git tag --list '<name>@*'` визначає базу commit-range. За відсутності будь-якого тегу (bootstrap) fallback повертає `null` і не виконує bump — рішення закрито комітом `8b74ea1 fix(release): bootstrap no-op fallback + CI workflow deps/identity`.

## ADR Fallback-синтез CHANGELOG-запису лише за наявності попереднього тегу

## Context and Problem Statement
При першому запуску `n-cursor release` в репо ще немає тегів `<name>@*`. Якщо fallback запускається без baseline-тегу, він бере всю git-історію і безпідставно генерує запис та виконує bump, конфліктуючи з ручним переходом до нової схеми.

## Considered Options
* Завжди синтезувати fallback (брати `HEAD` від початку репо як базу)
* No-op fallback за відсутності тегу (bootstrap-gate)

## Decision Outcome
Chosen option: "No-op fallback за відсутності тегу", because bootstrap без тегу означає, що немає попереднього релізу для обчислення delta — синтез був би семантично беззмістовним і подвоїв би bump у перехідний період.

### Consequences
* Good, because transcript фіксує очікувану користь: перший реліз після переходу публікується без паразитного bump (перевірено тестом `12 passed, 0 failed` у `fallback.test.mjs`).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовано в `npm/rules/release/lib/fallback.mjs` (рядки ~37–39): `if (!lastTag) return null`. Закрито комітом `8b74ea1`. Рішення відкоригувало специфікацію `docs/superpowers/specs/2026-05-29-n-cursor-release-design.md` (рішення #3), де спочатку fallback не мав bootstrap-gate.

## ADR CI-workflow: `npm-publish.yml` — один job для release + publish

## Context and Problem Statement
Потрібно було визначити, як організувати CI для публікації npm-пакетів у монорепо: окремий job для релізу і окремий для публікації, чи один об'єднаний.

## Considered Options
* Два окремих jobs (release / publish)
* Один об'єднаний job `release-publish`

## Decision Outcome
Chosen option: "Один об'єднаний job `release-publish`", because це спрощує передачу стану (bumped version, tag) між кроками та уникає race condition між jobs.

### Consequences
* Good, because transcript фіксує очікувану користь: `lint-ga` підтвердив синтаксично коректний workflow після злиття фічі.
* Bad, because `cancel-in-progress: true` (enforced rego-поліcі `ga.workflow_common`) скасовує паралельний push mid-release; прийнятно, бо реліз дурабельний лише після push і half-release не виникає.

## More Information
Файл `.github/workflows/npm-publish.yml`. Composite-template для scope B: `npm/github-actions/release/action.yml`. Rego-поліcі `ga.workflow_common` примусово встановлює `cancel-in-progress: true`. Тригер `**/.changes/**` прибрано тимчасово (`check-ga` відхиляє glob без tracked-файлів); варто повернути після появи `.changes/` у репо.

## ADR Запуск `n-cursor release` через локальний бінарник, а не `npx`

## Context and Problem Statement
CI-job спочатку запускав `npx @nitra/cursor release`, що дало помилку `n-cursor: not found` — пакет не встановлено в runner, а self-publishing репо не може тягнути саме себе з реєстру (chicken-and-egg).

## Considered Options
* `npx @nitra/cursor release` (тягнути з npm-реєстру)
* `node npm/bin/n-cursor.js release` (локальний код після `bun install`)

## Decision Outcome
Chosen option: "`node npm/bin/n-cursor.js release` після кроку `setup-bun-deps`", because self-publishing репо мусить релізувати локальний код, а не попередню опубліковану версію; `bun install` лінкує workspace-bin і встановлює залежності (`smol-toml` тощо).

### Consequences
* Good, because transcript фіксує очікувану користь: помилку `n-cursor: not found` усунуто комітом `8b74ea1`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Кроки в `.github/workflows/npm-publish.yml`: `- uses: ./.github/actions/setup-bun-deps` → `node npm/bin/n-cursor.js release`. Git-identity (`user.name`, `user.email`) також додано, щоб `release` міг виконати `git commit`/`tag`. Зміни введено комітом `8b74ea1 fix(release): bootstrap no-op fallback + CI workflow deps/identity`.
