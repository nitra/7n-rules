## ADR Правило `test` завжди активне — видалення dog-food умови

## Context and Problem Statement
Консьюмер-монорепо `ai` використовує `n-cursor coverage`, що вимагає `vitest`, `@vitest/coverage-v8` і `@stryker-mutator/vitest-runner` у root `devDependencies`. `bun.mdc` (v1.9) описував дозвіл на ці пакети як виняток лише для dog food-репо `@nitra/cursor`, тоді як `package_json.rego` глобально дозволяв їх через `allowed_root_dev_dependency` без перевірки імені репо. Цей розсинхрон спричиняв помилку запуску `bun run coverage` в `ai` та суперечливі правила для консьюмерів.

## Considered Options
* Звузити `package_json.rego` під mdc-текст (перевіряти ім'я репо, залишити як dog-food виняток).
* Зробити правило `test` always-on (аналогічно `adr`, `security`, `text`) і прибрати умову з `bun.mdc`.

## Decision Outcome
Chosen option: "Зробити правило `test` always-on і прибрати dog-food умову", because перший варіант зламав би `ai` та всіх інших консьюмерів `n-cursor coverage`; структурна причина (published workspace `npm/` не має `devDependencies` за `npm-module.mdc`, тому Vitest/Stryker-tools мусять жити у root будь-якого монорепо) однаково стосується кожного споживача правила.

### Consequences
* Good, because `bun run coverage` у будь-якому консьюмер-монорепо перестає падати з помилкою про відсутній `vitest`.
* Good, because `bun.mdc` і `package_json.rego` тепер узгоджені: обидва дозволяють root-only test-peer без умов на ім'я репо.
* Bad, because `test` додається в `.n-cursor.json` автоматично навіть для репо без тестів — transcript не містить підтверджених негативних наслідків від цього.
