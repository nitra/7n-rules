**Status:** Accepted
**Date:** 2026-06-15

## Context and Problem Statement

Поточний `normalize-decisions.sh` використовував single-shot підхід: один великий prompt з усіма чернетками та clean-файлами очікував одну велику JSON-відповідь. На cloud-моделях цей підхід працював, але локальна 4B-модель `omlx/gemma-4-e4b-it` не могла стабільно обробити такий prompt: transcript фіксує `omlx curl exit 18` або зависання понад 5 хвилин.

Потрібно було зберегти ADR-нормалізацію без хмарної залежності й не перекладати глобальні інваріанти батчу на малу модель.

## Considered Options

- Single-shot наївний перенос: той самий prompt і той самий JSON-контракт через `omlx`.
- Конвеєрний підхід: JS оркеструє глобальний стан, а LLM відповідає лише на вузькі verifiable-питання для одного драфта або пари драфтів.

## Decision Outcome

Chosen option: "Конвеєрний підхід з інверсією керування", because single-shot arm `naive-local` впав на infra-рівні через `omlx curl exit 18` або hang, тоді як конвеєрний arm `new-local` завершив батч локально без cloud-викликів і зі структурно валідними MADR.

### Consequences

- Good, because transcript фіксує `cloudCalls: 0` і роботу через локальну модель без передачі чернеток зовнішньому LLM-сервісу.
- Good, because глобальні інваріанти — coverage «кожен файл рівно один раз», унікальність slug-ів, union-find кластеризація і MADR structural lint — гарантує JS, а не модель.
- Good, because hardened версія додала детермінований `isNoDecision` gate: драфти з «не обрано / не прийнято» у `Decision Outcome` ідуть на `delete` без виклику моделі.
- Bad, because wall-clock час hardened локального конвеєра близько 17 хвилин на 10 драфтів і не кращий за cloud-gold близько 14 хвилин.
- Neutral, because transcript фіксує 3 голоси з порогом 0.6 для draft-draft edge-judge як hardening genuine judgment call, але не містить окремого порівняння альтернативних порогів.

## More Information

- Файли реалізації у worktree `main-adr-exp`: `npm/scripts/lib/adr/normalize-pipeline.mjs`, `npm/scripts/lib/adr/normalize-cli.mjs`.
- Subcommand зареєстровано у `npm/bin/n-cursor.js` як `adr-normalize-local`.
- Local-backend гілка додана в `npm/.claude-template/hooks/normalize-decisions.sh`; активується, коли немає `claude`/`cursor-agent` або встановлено `ADR_NORMALIZE_BACKEND=local`.
- A/B результати: `experiments/adr-exp/RESULTS.md`, `experiments/adr-exp/out/*/summary.json`.
- Конвеєрні стадії: lexical retrieval top-K у JS, binary edge-judge моделлю, union-find кластеризація, kind-judge, constrained gen-MADR, validation gate.
- Модель: `omlx/gemma-4-e4b-it-OptiQ-4bit`, також згадана через `N_LOCAL_MIN_MODEL`.
- Transcript фіксує 45 зелених тестів і 13 нових unit-тестів детермінованих функцій у `npm/scripts/lib/adr/tests/normalize-pipeline.test.mjs`.
- Зміни у worktree `main-adr-exp` на момент transcript не були закомічені.

## Update 2026-06-14

Перед конвеєрною реалізацією зафіксовано вихідний мотив: поточна ADR-нормалізація була bash-hook-ом `.claude/hooks/normalize-decisions.sh` на 498 рядків, який запускався через Claude-hook і викликав хмарного LLM-провайдера.

Референсним патерном для нового оркестратора визначено `npm/scripts/lib/fix/orchestrator.mjs`, а модельну маршрутизацію — через `npm/lib/models.mjs`, де префікс `omlx/<model>` спрямовується у локальний HTTP через `npm/lib/omlx.mjs`, тоді як інші моделі йдуть у хмарний backend.

Neutral, because transcript фіксує наявність `N_LOCAL_MIN_MODEL=omlx/gemma-4-e4b-it-OptiQ-4bit` і супутній skill `.cursor/skills/n-adr-normalize/SKILL.md` для ручного worktree-only запуску одного батчу чернеток.

## Update 2026-06-15

A/B-експеримент у worktree `main-adr-exp` уточнив причину відмови від single-shot:

- `naive-local` з тим самим ~32k-токенним prompt-ом падав на infra-рівні через `omlx curl exit 18` або зависання.
- `cloud-gold` завершувався успішно, але вимагав хмарної моделі та близько 811 секунд.
- `new-local` виконав батч локально через серію вузьких LLM-мікрозадач і детермінований JS-контроль глобальних інваріантів.

Декомпозицію зафіксовано як інверсію керування: JS тримає union-find кластери, coverage-гейт «кожен файл рівно один раз», унікалізацію слагів і MADR structural lint; LLM виконує edge-judge, kind-judge і constrained gen-MADR для одного документа.

Bad, because transcript фіксує регресію ранньої версії: kind-judge не розпізнав no-decision draft і згенерував `Status: Accepted`; це вимагало детермінованого гейту «не обрано → delete».

## Update 2026-06-15

- Зафіксовано A/B-підтвердження локального pipeline backend: 7/7 MADR валідних, повне покриття, 0 cloud-викликів і 0 крахів на тих самих 10 фікстурах.
- Додано деталі реалізації: `npm/scripts/lib/adr/normalize-pipeline.mjs`, `npm/scripts/lib/adr/normalize-cli.mjs`, `npm/scripts/lib/adr/tests/normalize-pipeline.test.mjs`, backend-селектор `ADR_NORMALIZE_BACKEND=local|claude|cursor`, subcommand `adr-normalize-local`.
- Зафіксовано харднінги після першого прогону: deterministic `isNoDecision` gate та conservative edge-judge з 3 голосами і confidence ≥ 0.6.
- Відомий компроміс: wall-clock близько 17 хвилин на 37 серійних `omlx`-викликів; один edge-judge over-merge для docgen-пари.
