**Status:** Accepted
**Date:** 2026-06-15

## Context and Problem Statement

Поточний `normalize-decisions.sh` використовував single-shot підхід: один великий prompt з усіма чернетками та clean-файлами → одна JSON-відповідь з усіма операціями. Cloud-моделі виконували цей сценарій, але локальна 4B-модель `omlx/gemma-4-e4b-it` не могла стабільно опрацювати приблизно 32k-токенний prompt: виникав `omlx curl exit 18` або зависання понад 5 хвилин.

## Considered Options

- Single-shot наївний перенос: той самий prompt і JSON-контракт, але через `omlx`.
- Конвеєрний підхід: JS оркеструє глобальний стан, LLM відповідає лише на вузькі verifiable-питання per draft.

## Decision Outcome

Chosen option: "конвеєрний підхід з інверсією керування", because transcript фіксує, що `naive-local` впав на infra-рівні, а `new-local` завершив батч повністю локально без cloud calls, infra-failures і з валідними MADR.

### Consequences

- Good, because конвеєр виконується без хмарних викликів (`cloudCalls: 0`).
- Good, because приватні чернетки, локальні шляхи й внутрішня архітектура не передаються зовнішнім LLM-сервісам.
- Good, because детермінований JS гарантує coverage, slug-uniqueness і structural MADR validation, а не перекладає ці глобальні інваріанти на малу модель.
- Bad, because wall-clock час локального конвеєра приблизно 17 хвилин на 10 драфтів і не кращий за cloud-gold приблизно 14 хвилин.
- Neutral, because transcript фіксує серійну природу локального omlx-сервера як практичне обмеження.

## More Information

Файли реалізації з transcript:

- `npm/scripts/lib/adr/normalize-pipeline.mjs` — конвеєр.
- `npm/scripts/lib/adr/normalize-cli.mjs` — CLI wrapper `adr-normalize-local`.
- `npm/bin/n-cursor.js` — реєстрація subcommand.
- `npm/.claude-template/hooks/normalize-decisions.sh` — local-backend branch, активується коли немає `claude`/`cursor-agent` або `ADR_NORMALIZE_BACKEND=local`.
- `npm/scripts/lib/adr/tests/normalize-pipeline.test.mjs` — 13 нових unit-тестів детермінованих функцій.

Експериментальні факти:

- `experiments/adr-exp/RESULTS.md` і `experiments/adr-exp/out/*/summary.json` містять A/B результати.
- Hardened `new-local`: 6 rewrite, 3 merge, 1 delete; MADR valid 6/6; no-decision coverage OK; 0 cloud calls.
- Загалом пройшло 45 тестів.
- Два hardening-кроки: детермінований `isNoDecision` gate для no-decision драфтів і 3 голоси з порогом 0.6 для draft-draft edge-judge.
- Стадії конвеєра: lexical top-K retrieval, binary edge-judge, JS union-find clustering, constrained gen-MADR, deterministic validation gate.

## Update 2026-06-14

Початковий аналіз зафіксував, що локальну ADR-нормалізацію варто будувати за патерном `npm/scripts/lib/fix/orchestrator.mjs`, а не як прямий перенос bash-hook.

Додаткові факти з transcript:

- Поточний hook: `.claude/hooks/normalize-decisions.sh` — 498 рядків.
- Референсний патерн: `npm/scripts/lib/fix/orchestrator.mjs`.
- Модельна маршрутизація: `npm/lib/models.mjs` маршрутизує префікс `omlx/<model>` до `npm/lib/omlx.mjs` як локального HTTP-провайдера.
- Змінна середовища для локальної моделі: `N_LOCAL_MIN_MODEL=omlx/gemma-4-e4b-it-OptiQ-4bit`.
- Супутній skill для ручного запуску одного батчу: `.cursor/skills/n-adr-normalize/SKILL.md`.

## Update 2026-06-15

A/B-експеримент уточнив, що проблема single-shot підходу не лише у швидкості, а й у межах локальної 4B-моделі та omlx infra.

Додаткові факти з transcript:

- Поточний `.claude/hooks/normalize-decisions.sh` надсилав приблизно 32k-токенний prompt: 10 чернеток + 294 clean-файли.
- `naive-local` через `omlx/gemma-4-e4b-it-OptiQ-4bit` падав з `omlx curl exit 18` або зависав понад 5 хвилин.
- `cloud-gold` завершувався успішно, але потребував хмари й приблизно 811 секунд.
- `new-local` завершив 10 чернеток локально: `each-once=true`, `madrInvalid=0`, `cloudCalls=0`, `escalations=0`.
- Важливий design point: JS тримає глобальний стан — union-find кластери, coverage-гейт, унікалізацію slug і MADR structural lint; LLM відповідає на вузькі задачі `edge-judge`, `kind-judge`, `gen-MADR`.
- Зафіксована регресія: kind-judge не розпізнав no-decision draft, тому потрібен детермінований gate `не обрано → delete`.
- Wire-trace: `.n-cursor/llm-trace.jsonl` із caller-prefix `adr-pipe:kind`, `adr-pipe:gen`.
