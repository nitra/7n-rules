## ADR doc-files: міграція скіла в правило зі збереженням тонкого скіла

## Context and Problem Statement
В репо існував `npm/skills/doc-files/` — агентський скіл із важкою JS-логікою в `js/`. Водночас `scripts.mdc` вимагає checkable-механізм для CI-ланцюжка (policy, rego, `lint`/`fix`-канал) через правило `npm/rules/<id>/`. Потрібно було вирішити, як розмістити doc-files, щоб задовольнити обидва канали.

## Considered Options
* Зробити `doc-files` виключно скілом (без правила `doc`)
* Перейменувати правило на `doc-files` і лишити тонкий скіл поверх

## Decision Outcome
Chosen option: "Перейменувати правило на `doc-files` і лишити тонкий скіл поверх", because user явно вибрав цей варіант; він зберігає policy-канал (GA-workflow, rego-enforcement, `fix.mjs` через `runStandardRule`) у правилі, а агентський workflow — у тонкому `SKILL.md`.

### Consequences
* Good, because `git mv npm/skills/doc-files/js → npm/rules/doc-files/js` зберіг ті самі відносні імпорти `../../../` без жодних змін у модулях.
* Good, because transcript фіксує очікувану користь: правило отримало `meta.json` (`auto:"завжди"`, `lint:"quick"`), тому `n-cursor lint` агрегатор автоматично включає `doc-files` у quick-прогін зміненого набору.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Ключові файли: `npm/rules/doc-files/meta.json`, `npm/rules/doc-files/fix.mjs`, `npm/rules/doc-files/js/lint.mjs` (адаптер агрегатора, both-direction mapping source↔doc), `npm/rules/doc-files/lint/lint.mjs` (CLI `runLintDocFilesCli`), `npm/skills/doc-files/SKILL.md` (тонкий скіл). Заборонено фізично зливати каталоги детекторів в `lint` — порушує `scripts.mdc` «одне правило — один каталог».

## ADR lint-doc-files / fix-doc-files: іменування команд CLI за шляхом правила

## Context and Problem Statement
При міграції `js/` у `npm/rules/doc-files/` треба було надати нові підкоманди `n-cursor` для детектора і генератора. Попередня spec планувала скорочені `lint-doc`/`fix-doc`; після рішення про `doc-files` як id правила виникло питання іменування.

## Considered Options
* Скорочені `lint-doc` / `fix-doc` (як у початковій спеці)
* `lint-doc-files` / `fix-doc-files` (похідні від id правила `doc-files` за конвенцією `scripts.mdc`)

## Decision Outcome
Chosen option: "`lint-doc-files` / `fix-doc-files`", because user підтвердив перейменування правила на `doc-files`, а `scripts.mdc` виводить lock-ключ і ім'я команди зі шляху каталогу — `rules/doc-files` → `lint-doc-files`/`fix-doc-files` детермінований вивід.

### Consequences
* Good, because transcript фіксує очікувану користь: lock-ключ `lint-doc-files` виводиться автоматично, маркер у `sync-claude-config.mjs` (`DOC_FILES_HOOK_COMMAND_MARKER`) і шаблон `settings.template.json` відповідно оновлені; PostToolUse/Stop-хуки в `.claude/` перейменовано без дублювання логіки.
* Good, because старий `doc-files <sub>` лишився deprecated-аліасом з warning для поступового переходу.
* Bad, because команди довші за `lint-doc`/`fix-doc`; вплив на UX у transcript не зафіксований.

## More Information
`npm/bin/n-cursor.js`: `case 'lint-doc-files'`, `case 'fix-doc-files'`, `case 'doc-files'` (deprecated alias). `npm/.claude-template/settings.template.json`: хуки оновлені. `npm/scripts/sync-claude-config.mjs`: `DOC_FILES_HOOK_COMMAND_MARKER = '@nitra/cursor lint-doc-files'`. Changeset `npm/.changes/260614-0724.md` (bump: minor). Авто-реліз опублікував `@nitra/cursor@5.4.0`.

## ADR N_LOCAL_MIN_MODEL як тир-канон для локальної omlx-моделі

## Context and Problem Statement
`fix-doc-files` вимагає локальну omlx-модель. Пакетний дефолт `DEFAULT_OMLX_MODEL = 'mlx-community--gemma-4-e2b-it-4bit'` не збігався з іменами, які повертає сервер (`gemma-4-e2b-it-4bit`, без HF-org-префікса), що призводило до `Model not found`. Перше рішення — `N_CURSOR_DOCGEN_MODEL` у `~/.zshenv` — було docgen-специфічним. User поставив питання, чи є стандартний env.

## Considered Options
* `N_CURSOR_DOCGEN_MODEL` (docgen-specific env)
* `N_LOCAL_MIN_MODEL` (тир-канон з `npm/lib/models.mjs`)

## Decision Outcome
Chosen option: "`N_LOCAL_MIN_MODEL`", because пакет уже має `resolveModel('min')` у `npm/lib/models.mjs`, куди `N_CURSOR_DOCGEN_MODEL` каскадує: `env.N_CURSOR_DOCGEN_MODEL ?? resolveModel('min') ?? DEFAULT_OMLX_MODEL`. `N_LOCAL_MIN_MODEL` — канонічний «один раз у середовищі; кожен скіл посилається на потрібний тир» — налаштовує модель для **всіх** скілів через цей тир, без зміни пакета.

### Consequences
* Good, because `~/.zshenv` містить один рядок `export N_LOCAL_MIN_MODEL=omlx/gemma-4-e2b-it-4bit`; перевірено в clean-env (`env -i`) — `fix-doc-files` без жодного env-префікса дав `✓ OK score=100`.
* Good, because transcript фіксує очікувану користь: інші скіли на тирі `min` автоматично підхоплять ту саму модель.
* Bad, because `DEFAULT_OMLX_MODEL = 'mlx-community--gemma-4-e2b-it-4bit'` у пакеті застарілий (prefixed vs unprefixed); без env все одно дає `Model not found`. Self-heal у `npm/lib/omlx.mjs` (retry з de-prefixed id) залишився невирішеним у цій сесії.

## More Information
`npm/lib/models.mjs`: `resolveModel(tier)`, `N_LOCAL_MIN_MODEL = env.N_LOCAL_MIN_MODEL`. `~/.zshenv`: `export N_LOCAL_MIN_MODEL=omlx/gemma-4-e2b-it-4bit`. Модель `gemma-4-e4b-it-OptiQ-4bit` не вміщається (потребує 13.07GB > ceiling 11.84GB на 16GB RAM). Модель `gemma-4-e2b-it-4bit` — єдина що влазить на цій машині.
