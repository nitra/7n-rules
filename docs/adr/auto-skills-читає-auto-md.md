# Auto-skills читає auto.md замість hardcoded мапи

**Status:** Accepted
**Date:** 2026-05-15

## Контекст

`npm/scripts/auto-skills.mjs` мав hardcoded константи `SKILL_DEPENDENCIES` та `ALWAYS_ON_SKILLS` — дублювання тієї ж інформації, що вже була задокументована у файлах `npm/skills/<id>/auto.md` (формат: `завжди`, `[rule-id]`, або відсутній файл). Зміна будь-якої залежності потребувала правки і `auto.md`, і JS.

## Рішення/Процедура/Факт

`auto-skills.mjs` тепер сканує `npm/skills/<id>/auto.md` під час завантаження модуля і будує `AUTO_SKILL_RULE_DEPENDENCIES` та `ALWAYS_ON_SKILLS` із вмісту файлів. Три формати:
- `завжди` → always-on (скіл активується у кожному проєкті);
- `[rule, …]` → умова на правила (скіл активується лише якщо вказані правила є у `.n-cursor.json:rules`);
- відсутній або нерозпізнаний файл → opt-in (скіл не активується автоматично).

Commit `b7ad911`, версія `1.11.2`.

Примітка: файл `auto.md` у каталозі скіла є **документаційним** — його вміст дублює логіку для людини, яка переглядає каталог. Справжня логіка живе у `auto-skills.mjs`, який читає цей файл при завантаженні.

## Обґрунтування

Єдине джерело правди — `auto.md`. Після цієї зміни додавання нового скіла з умовами потребує тільки правки `auto.md` без змін у JS.

## Розглянуті альтернативи

Hardcoded підхід (`SKILL_DEPENDENCIES` у JS) — попередній стан, відхилено як дублювання двох файлів при кожній зміні.

## Зачіпає

`npm/scripts/auto-skills.mjs`, `npm/scripts/auto-skills.test.mjs`, `npm/skills/*/auto.md`

## Update 2026-05-14

Підтверджено синтаксис умовного авто-додавання: рядок вигляду `skill - [rules]` у `npm/bin/auto-skills.md` означає, що скіл включається лише якщо всі перелічені правила вже присутні в `.n-cursor.json` споживача. Скіли, не зареєстровані в `auto-skills.md`, потребують ручного включення або явної реєстрації з умовою.
