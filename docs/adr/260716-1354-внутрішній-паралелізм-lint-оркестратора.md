description: "Керований перехід detectAll() до безпечного паралельного виконання detector-ів після інвентаризації блокувальних меж і benchmark-ів."
---

**Status:** Proposed
**Date:** 2026-07-16

## Context and Problem Statement

`detectAll()` у `npm/scripts/lib/lint-surface/run-detectors.mjs` виконує план concern-ів послідовно. Це подовжує full-lint, але простий пул Promise не гарантує пришвидшення: багато detector-ів викликають `spawnSync`, який блокує Node event loop. Конверсія лише `run-conftest-batch.mjs` та `js/eslint/main.mjs` не охоплює ці межі, а `ensure-tool.mjs:runStep` у коді не існує.

Ціль — скоротити wall-clock time одного `n-rules lint` без зміни workflow-файлів споживачів, не створивши oversubscription на 1–2 vCPU CI-agent-ах, гонок в auto-install/cache, недетермінованого звіту чи process leaks після скасування.

## Considered Options

- Запустити всі concern-и через `Promise.all` або пул поверх наявних detector-ів.
- Конвертувати лише кілька спільних `spawnSync` helper-ів.
- Провести інвентаризацію blocking boundary, мігрувати придатні зовнішні процеси на async API та ввести bounded concern-pool.
- Використати `worker_threads` або child-process pool для CPU-bound JS detector-ів.
- Додати `resourceWeight` у `concern.json` і планувати за історичною тривалістю.

Перші два варіанти відхилено: вони не дають заявленої паралельності для detector-ів із прямим `spawnSync`. `worker_threads`, ваги та LPT відкладені до появи профілю реальних тривалостей.

## Decision Outcome

Chosen option: **спочатку вимірюваний і безпечний MVP, потім увімкнення паралелізму**.

1. Перед зміною scheduler-а створюємо inventory усіх detector-ів і shared helper-ів, що блокують event loop (`spawnSync`, sync I/O або інші process-wide side effects). Кожна позиція має бути або перенесена на async boundary, або явно позначена serial до окремої міграції. Не можна заявляти загальний паралелізм, доки невідомі blocker-и.
2. Async-conversion охоплює не абстрактні «кілька helper-ів», а конкретні execution boundaries та всіх їхніх caller-ів: `runConftestBatch()` стає async разом із production caller-ами; `runOxlintJson()` отримує async process wrapper; інші detector-и зі `spawnSync` входять до inventory. Wrapper приймає `AbortSignal`, timeout і повертає нормалізовані stdout/stderr/exit status.
3. Лише після цього `detectAll()` отримує bounded динамічний pool по concern-ах. Він не запускає нові задачі після infrastructure failure, зберігає progress по завершених concern-ах і стабільно сортує violations за `ruleId`, `concernId`, `file`, `line`, `reason`.
4. Auto-install не виконується конкурентно. Для lint-пайплайна потрібен async single-flight resolver за `toolId`; download пишеться в унікальну temp-локацію та публікується atomically. Альтернатива — serial preflight усіх потрібних tool-ів до старту pool-а. Спільні cache/archive paths без lock заборонені.
5. Контракт помилок розділяється: lint violations — звичайний результат і не зупиняють прогін; infrastructure error (неможливий import, запуск tool-а, timeout, невалідний результат detector-а) зупиняє планування, abort-ить підтримувані child-process-и, чекає settlement already-started задач і повертає exit code `2`. Часткові violations не маскують infrastructure failure.

`N_RULES_LINT_CONCURRENCY` лишається override для експериментів. Постійний default не фіксується до benchmark-ів; до їх позитивного завершення production default залишається `1`.

### Consequences

- Good, because pool не створить ілюзію паралельності поверх синхронних detector-ів, а scope кожної міграції буде перевірюваним.
- Good, because explicit cancellation, error classification і single-flight install прибирають orphan processes, flaky cold-cache CI та нечіткі partial results.
- Good, because стабільний comparator зберігає відтворюваний console output незалежно від порядку завершення задач.
- Bad, because перша реалізація торкнеться не лише orchestration layer: async `runConftestBatch()` вимагає змін у його caller-ах, а частину detector-ів доведеться тимчасово лишити serial.
- Bad, because на 1–2 vCPU agent-ах безпечний default може залишатися `1`; пришвидшення має бути доведене, а не припущене.

## More Information

Перед зміною default або merge реалізації потрібні такі acceptance gates:

1. Benchmark full-lint на GitHub runner та Azure agent, щонайменше для cold і warm tool cache, з `N_RULES_LINT_CONCURRENCY=1`, `2` і запропонованим default. Вимірюються wall-clock time, peak RSS і exit-code/output parity.
2. Заявлений default приймається лише якщо медіана wall-clock time не гірша за baseline і дає помітне скорочення на репрезентативному workload; точне значення та результати додаються в implementation PR.
3. Unit/integration tests покривають concurrency cap, stable violation order, progress `done/total`, abort без запуску нових concern-ів, очікування already-started задач, timeout, single-flight install і унікальні tmp paths.
4. Inventory підтверджує, що кожен concern у parallel lane не має blocking sync boundary або process-wide mutable state. Решта concern-ів виконуються serial lane до окремого рішення.
5. Реалізація не змінює workflow-файли споживачів і працює в plain Node та Bun.

Після проходження цих gates ADR можна перевести в `Accepted` разом із конкретним default concurrency та посиланням на benchmark evidence.
