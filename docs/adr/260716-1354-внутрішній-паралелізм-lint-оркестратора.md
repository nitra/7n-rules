# Внутрішній паралелізм lint-оркестратора @7n/rules

**Status:** Accepted
**Date:** 2026-07-16

## Context and Problem Statement

`detectAll()` у `npm/scripts/lib/lint-surface/run-detectors.mjs` проганяє план concern-ів (масив `{entry, files}`) послідовно через `for`-цикл. Мета — прискорити один прогін `n-rules lint` (`--full` і delta) для всіх CI-провайдерів (GitHub/Azure/майбутній GitLab) кратно (не просто "трохи швидше"), змінивши лише сам JS-оркестратор — без зміни жодного workflow-файлу споживача.

Обмеження: CI-агенти з обмеженою CPU (Azure Microsoft-hosted free tier — типово 1-2 vCPU на приватний org); живий progress у TTY (`⏱ N/M`) не має регресувати. Мотивація виникла після реального інциденту: на Azure-агенті `oxlint` (сам багатопотоковий, `threads_count: 10`) конфліктував за CPU при спробі підняти зовнішній паралелізм на рівні pipeline (N окремих job-ів), що коштувало б платних parallel jobs і множило б `bun install` — тому паралелізм зважили як задачу самого JS-оркестратора, а не інфраструктури CI.

## Considered Options

- **Механізм**: конверсія спільних `spawnSync`-хелперів на async `spawn`/`execFile` + `Promise.all`, без worker_threads/child-fork.
- **Гранулярність**: одиниця паралельного виконання — concern (крос-rule), правило — лише одиниця групування прогресу (не виконання).
- **Scheduling**: динамічний пул (семафор, "хто звільнився — бере наступного"), не фіксовані раунди.
- **Ресурсний ліміт**: консервативний дефолт (не наївний `cpus()-1`) + env-override для дебагу/opt-out.
- **Безпека**: аудит/рандомізація tmp-шляхів, ізоляція помилок per-задача, `AbortController` для fail-fast на інфраструктурних помилках.
- **Progress/детермінізм**: лічильник по факту завершення (без live-списку in-flight), сортування violations після збору.
- Відкладено: worker_threads для чисто CPU-bound JS-concern-ів; декларативне поле "вага" в `concern.json`; LPT-планування за історичною тривалістю.

## Decision Outcome

Chosen option: **MVP-пакет із п'яти елементів** — (1) async-spawn конверсія `run-conftest-batch.mjs`/`ensure-tool.mjs:runStep`/`eslint/main.mjs:runOxlintJson`; (2) динамічний пул на рівні concern-ів крос-rule, правило — лише групування прогресу; (3) консервативний дефолтний ліміт паралелізму + `N_RULES_LINT_CONCURRENCY` override; (4) обов'язковий безпековий бандл (tmp-шляхи, ізоляція помилок, AbortController); (5) progress-reporter із лічильником по факту завершення + сортування violations після збору.

Facilitator recommended цей самий пакет із гранулярністю "по concern" (не по rule) як дефолтний варіант кластера B; user спершу обрав гранулярність "по rules" (послідовно всередині правила, паралельно між правилами). Facilitator заперечив: профіль правил нерівномірний через mixin-архітектуру (правило `js` в azure-репо накопичує 9+ concern-ів з кількох плагінів, тоді як `security`/`bun` — 1-2), тож послідовність усередині одного важкого правила блокує кратний ефект, бо решта "доріжок" простоюють, поки найважче правило добігає. User погодився на гібрид: правило лишається логічною одиницею групування (прогрес, звітність, концептуальна мапа на плагін/домен), а фактичний пул паралелізму працює на рівні concern-ів незалежно від приналежності до правила.

Причина вибору MVP-пакета (не ширшого): профіль навантаження в цій кодовій базі — переважно зовнішні spawn-важкі тули (conftest, oxlint, eslint, cspell, kubeconform), не чистий CPU-bound JS, тож async-spawn конверсія вже забирає більшість досяжного ефекту без worker-пулів; консервативний дефолт+override напряму адресує інцидент з oxlint-oversubscription; жодних змін у щойно стабілізованому контракті `concern.json`/plugin-маніфесту (уникнуто найризикованішої під-ідеї — декларативного поля "вага").

### Consequences

- Good, because full-lint отримує кратне пришвидшення для домінантного паттерна навантаження (зовнішні spawn-виклики) з мінімально інвазивною зміною — торкається кількох спільних хелперів, а не ~20 різних `main.mjs` по всіх правилах і плагінах.
- Good, because гібридна гранулярність (concern-рівень виконання, rule-рівень групування прогресу) уникає load-imbalance пастки, яку виявили в дебатах кроку 5, і не потребує подальшого узгодження з mixin-архітектурою плагінів.
- Good, because безпековий бандл (tmp-шляхи, ізоляція помилок, AbortController) — явна передумова, зафіксована в рішенні, а не мовчазне припущення, тож майбутня реалізація не забуде про гонки стану.
- Bad, because worker_threads (для чисто CPU-bound JS-concern-ів, якщо такі виявляться вузьким місцем) свідомо відкладено — якщо профіль навантаження виявиться іншим на практиці (напр. великі k8s/hasura YAML-парсери домінують у часі, не spawn-виклики), MVP не дасть очікуваного ефекту без другої ітерації.
- Bad, because LPT-планування за історичною тривалістю відкладено — перший прогін після впровадження не матиме оптимального порядку виконання (можлива "довга хвіст"-задача в кінці), допоки не накопичиться кеш тривалостей.

## More Information

Усі ідеї сесії (сирий список):

1. Promise.all + concurrency-limiter по `cpus().length`.
2. Конвертація `spawnSync`→async `execFile`/`spawn` у спільних хелперах.
3. Worker_threads пул (кожен worker сам `import()` concern-модуль).
4. Дочірні Node-процеси (fork) замість worker_threads.
5. Гібрид #2+#3: async-spawn для I/O-важких, worker_threads — для CPU-важких JS.
6. Гранулярність "по concern".
7. Гранулярність "по ruleId" (обрано користувачем, потім скориговано до гібрида).
8. Гранулярність "по rulesDir/плагіну".
9. Гранулярність "по файлу" всередині одного per-file concern-а.
10. Двофазна модель: паралельна фаза + серіалізована "хвіст"-фаза для parallel-unsafe concern-ів.
11. Round-based батчі (фіксовані раунди по K concern-ів).
12. True async scheduler (динамічний семафор).
13. Longest-Processing-Time-First евристика за історичною тривалістю.
14. Кешування тривалості concern-ів між прогонами.
15. "Total budget" модель замість наївного count-of-processes (враховувати внутрішню багатопоточність зовнішніх тулів).
16. Розділення пулу на "легкі"/"важкі" категорії з окремими лімітами.
17. Явне поле "resourceWeight" у `concern.json` (декларативне).
18. Автоматична евристика ваги за наявністю зовнішнього spawn у коді (без розширення схеми).
19. Адаптивний concurrency-ліміт за `os.freemem()`/`os.cpus().length`.
20. Env-var override `N_RULES_LINT_CONCURRENCY`.
21. Fail-fast через `AbortController` на інфраструктурних помилках (`DetectorError`).
22. Аудит спільного mutable-стану (tmp-шляхи, кешування template-даних).
23. Рандомізація/унікалізація тимчасових шляхів.
24. Стабільне сортування violations після збору (не за порядком виконання).
25. Progress-reporter v2: live-дашборд in-flight concern-ів.
26. Progress-reporter MVP: лічильник по факту завершення, без live-списку.
27. Таймаут на concern (запобігає зависанню, напр. мережевий kubeconform).
28. Ізоляція крашу: виняток в одній паралельній задачі не вбиває решту.
29. Опційний `--profile` для вимірювання тривалості кожного concern-а окремо.
30. Cross-runtime сумісність: уникати Bun-специфічних API, лишатись сумісним з plain Node.
31. Групування за походженням правила (ядро vs плагін) як лінза безпеки для майбутнього fix-паралелізму.

Відкладені кластери: worker_threads (варіант #3/#5, кластер A) — друга ітерація, якщо профіль навантаження виявиться CPU-bound; декларативне поле "вага" в `concern.json` (#17, кластер D) — суперечить щойно стабілізованому контракту плагінів; LPT-планування (#13/#14, кластер C) — потребує кешу тривалостей із першого прогону; гранулярність "по файлу" (#9, кластер B) — нижчий пріоритет, бо більшість concern-ів у типових споживачах мають `scope=full`, не delta з великими changeset-ами; role-storming раунд генерації — не проводився (сесія зупинена після 3 раундів SCAMPER, 31 ідея, визнано достатнім обсягом).

Відкриті питання: конкретне числове значення дефолтного ліміту паралелізму (кластер D, #19) — не зафіксовано, визначиться емпірично під час реалізації/бенчмаркінгу; чи потрібен `--profile`-прапорець (#29) у першій ітерації, чи як окрема дрібна задача.
