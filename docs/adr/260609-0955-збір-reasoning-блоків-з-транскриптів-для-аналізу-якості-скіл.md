## ADR Збір reasoning-блоків з транскриптів для аналізу якості скілів

## Context and Problem Statement
Після кожного виклику скіла модель формує внутрішній «reasoning» (thinking-блоки). Команда хоче фіксувати ці міркування, щоб виявляти, де модель плуталась у правилах або де скіл недовизначений, і на цій основі покращувати `.mdc`-правила та `SKILL.md`-файли.

## Considered Options
* Використати вже наявні `.jsonl`-транскрипти Claude Code (`~/.claude/projects/…/*.jsonl`) разом із hooks-конвеєром (`SubagentStop`/`Stop`) для витягу thinking-блоків.
* Збирати reasoning через зовнішній Cursor або OpenAI Codex CLI (власні `.jsonl`-файли сесій у `~/.codex/sessions/`).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Використати `.jsonl`-транскрипти Claude Code + hooks-конвеєр", because reasoning-блоки (`"type":"thinking"`) вже зберігаються в 348 файлах поточного проєкту, а наявні Stop-hooks (`capture-decisions.sh`, `normalize-decisions.sh`) реалізують той самий pattern «зафіксувати сирий лог → дистилювати через LLM». Розширення цього конвеєра не вимагає нового механізму збору.

### Consequences
* Good, because transcript фіксує очікувану користь: thinking-блоки доступні вже зараз без додаткової інфраструктури; атрибуція до конкретного скіла здійснюється кореляцією `tool_use` типу `Skill` у тому ж `.jsonl`.
* Bad, because transcript попереджає: reasoning дуже багатослівний (сигнал/шум), транскрипти вже займають 262 МБ і зростатимуть; для Cursor сирий reasoning передається лише якщо провайдер моделі його віддає — не кожна модель і не кожна версія гарантує розгорнутий `thinking`, що робить цей підхід нестабільним поза Claude Code.
