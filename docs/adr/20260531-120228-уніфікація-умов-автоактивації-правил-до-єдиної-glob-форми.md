## ADR Уніфікація умов автоактивації правил до єдиної glob-форми

## Context and Problem Statement
Правила `@nitra/cursor` мали 3 різні форми перевірки файлового дерева: `anyFile` (список файлів), `rootFile` (тільки корінь), `dir` (наявність каталогу). Виникала потреба вибрати один декларативний предикат, що охоплює всі три варіанти для 13 правил Type A.

## Considered Options
* Три окремі форми: `anyFile`, `rootFile`, `dir`
* Єдина форма `{ "glob": "..." }` з OR-масивом і природнім кодуванням кореня (`package.json`) vs будь-якого рівня (`**/package.json`)

## Decision Outcome
Chosen option: "єдина форма `{ glob }`", because користувач підтвердив: "можна привести до glob виразу" — і обидва сторони погодилися, що glob природно виражає корінь (без `**/`) та рекурсію (`**/`), не потребуючи трьох окремих форм.

### Consequences
* Good, because transcript фіксує очікувану користь: одна форма замість трьох скорочує число варіантів схеми і спрощує реєстр предикатів.
* Bad, because семантика каталогів змінюється: `k8s/**` матчить лише якщо каталог непорожній, тоді як старий `existsSync(dir)` тригерив навіть на порожній каталог. Transcript фіксує це як прийнятний компроміс — порожній `k8s/` немає сенсу тригерити.

## More Information
- Таблиця маппінгу 13 правил до glob-патернів зафіксована в `docs/superpowers/specs/2026-05-31-rule-meta-json-design.md`.
- Порожній-каталог нюанс задокументований у spec явно як "свідома, прийнятна зміна семантики".

## ADR Чотири форми `auto` у `meta.json` правил

## Context and Problem Statement
Правила `@nitra/cursor` мали `auto.md` файли з довільним текстом для опису умов автоактивації. Потрібно було перейти на машиночитабельний `meta.json` з чітко типізованими умовами — дзеркалюючи вже наявну схему скілів, але враховуючи додаткові типи умов (файлові предикати, незводимі JS-предикати).

## Considered Options
* Три форми: `anyFile`/`rootFile`/`dir` + `"завжди"` + масив залежностей + `predicate`
* Чотири форми: `"завжди"` / `["rule"]` / `{ "glob": "..." }` / `{ "predicate": "...", "arg"?: ... }`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "чотири форми `auto`", because дизайн узгоджено в сесії після уніфікації файлових предикатів до glob-форми та підтвердження з боку користувача: "підтверджую".

### Consequences
* Good, because transcript фіксує очікувану користь: єдина схема для 33 правил, хардкод `AUTO_RULE_ORDER` та `AUTO_RULE_DEPENDENCIES` виводиться з даних, `auto.md` файли видалено.
* Bad, because незводимі предикати (6 штук) лишаються в JS-реєстрі `rule-predicates.mjs` — `meta.json` містить лише посилання на ім'я, не саму логіку. Transcript не містить підтверджених негативних наслідків цього рішення.

## More Information
- Реєстр предикатів: `repoUrlMarker`, `gqlTaggedTemplate`, `hasuraConfigMarker`, `depInAnyPackageJson`, `jsBunDbSignal`, `nestedPackageWithoutVite` — описані в `docs/superpowers/specs/2026-05-31-rule-meta-json-design.md`.
- Реалізовано в `npm/scripts/lib/rule-meta.mjs`, `npm/scripts/lib/rule-predicates.mjs`.
- Схема: `npm/schemas/rule-meta.json`; валідація: `npm/rules/npm-module/js/rule_meta.mjs`.

## ADR Увімкнення автодетекту правила `tauri` (раніше dead code)

## Context and Problem Statement
Правило `tauri` мало `auto.md` з умовою активації (`@tauri-apps/api`), але ніколи не було підключено до `detectAutoRules()` — фактично dead code. При міграції на data-driven `meta.json` виникав вибір: увімкнути або лишити opt-in.

## Considered Options
* Лишити `tauri` як opt-in (без `auto` у `meta.json`)
* Увімкнути автодетект через `{ "predicate": "depInAnyPackageJson", "arg": ["@tauri-apps/api"] }`
