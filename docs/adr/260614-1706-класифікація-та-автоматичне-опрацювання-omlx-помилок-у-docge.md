## ADR Класифікація та автоматичне опрацювання omlx-помилок у docgen-оркестраторі

## Context and Problem Statement
`docgen-files-batch.mjs` трактує всі omlx-збої однаково: `catch → ✗ → push → continue`. В логу зафіксовано три структурно різні класи: `spawnSync curl ETIMEDOUT` (transient), `memory ceiling exceeded` (systemic, каскадить на решту батчу), `Prompt too long: 9177917 tokens` (permanent, ретрай нічого не дасть). Через відсутність класифікатора systemic-збій після файлу №57 спалив ~200 наступних файлів без шансу на recovery.

## Considered Options
* Класифікатор `transient | systemic | permanent` + retry-з-backoff + circuit-breaker + окремий `skipped[]` — все в межах `docgen-files-batch.mjs` і `omlx.mjs`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Класифікатор + retry + circuit-breaker + skip", because transcript чітко демонструє, що поточний єдиний `catch` призводить до каскадного стрибка `✗` після пам'яттєвого збою, і healthcheck існує лише на старті, а не mid-run.

### Consequences
* Good, because transcript фіксує очікувану користь: circuit-breaker (K=3 підряд systemic) зупиняє марну витрату curl-раундів і надає actionable-повідомлення з кількістю оброблених файлів; невдалі файли без запису лишаються `stale` і підхоплюються при resume через CRC.
* Good, because transcript фіксує очікувану користь: ETIMEDOUT переводиться в transient-гілку (зараз там `break` замість `continue`), що прибирає неправомірне завершення одного таймауту.
* Good, because transcript фіксує очікувану користь: `permanent`-клас (`Prompt too long`) кладеться у `skipped[]` замість `errors[]`, усуваючи беззмістовні повтори для мініфікованих/vendored файлів.
* Bad, because transcript не містить підтверджених негативних наслідків.
