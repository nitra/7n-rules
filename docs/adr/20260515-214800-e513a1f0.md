## ADR Розмежування ролей `/n-fix` і `/n-lint` та дедублікація кроку лінтингу

**Контекст:** Скіл `/n-fix` у своєму кроці 6 дублював логіку `/n-lint`, перебираючи скрипти `lint-js`, `lint-text`, `lint-style` окремо. Це створювало ризик паралельних ESLint-процесів (заборонено в `CLAUDE.md`) і розмивало відповідальність: не було однозначної відповіді, чия задача — канонічний прогон `bun run lint`.

**Рішення/Процедура/Факт:** Закріплено чіткий розподіл ролей: `/n-fix` = приведення структури проєкту у відповідність до `.cursor/rules/` (перевіряється через `npx @nitra/cursor check`); `/n-lint` = чистота коду (єдиний запуск `bun run lint`). У `npm/skills/fix/SKILL.md` та `.cursor/skills/n-fix/SKILL.md` крок 6 переписано — перебір `lint-*` замінено делегуванням до `/n-lint` (`bun run lint`) з явним правилом "лінт-логіку не дублюй". У `npm/rules/style-lint/style-lint.mdc` cross-reference «повний набір `lint-*` (навичка `n-fix`)» замінено на «`bun run lint` (навичка `/n-lint`)`. Додано запис до `npm/CHANGELOG.md` секція Changed у 1.11.6.

**Обґрунтування:** Злиття команд відкинуто через різні сценарії використання (рідкий онбординг vs. частий pre-commit), надлишковий overhead та ризик порушення інваріанти "один ESLint-процес на сесію". Обрано мінімальну хірургію — дедублікацію: `/n-fix` делегує лінтинг до `/n-lint` замість власного перебору скриптів.

**Розглянуті альтернативи:** Повне злиття в одну команду — відхилено через різну частоту використання, різний час виконання і ризик паралельних ESLint; залишення status quo з дублюванням — відхилено як джерело плутанини.

**Зачіпає:** `npm/skills/fix/SKILL.md`, `.cursor/skills/n-fix/SKILL.md`, `npm/rules/style-lint/style-lint.mdc`, `npm/CHANGELOG.md`; публічний контракт обох скілів та правило `n-style-lint`.
