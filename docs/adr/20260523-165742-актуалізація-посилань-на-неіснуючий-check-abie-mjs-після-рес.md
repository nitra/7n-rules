## ADR Обов'язковий per-rule `fix.mjs` як точка входу (IoC модель)

## Context and Problem Statement
Правила `npm/rules/<id>/` не мали власної точки входу: CLI `n-cursor check` — єдиний спосіб їх запустити. Це блокувало пряме налагодження (`bun rules/abie/fix.mjs`), запуск через IDE Run button, per-rule CI-matrix jobs і майбутню автономізацію правил. CLI робив convention-based discovery і централізовано оркестрував applies-гейт → JS-concerns → policy → mdc-refs.

## Considered Options
* **Варіант A — Shim:** файл `<id>/fix.mjs` делегує `runStandardRule`; CLI лишається авторитативним.
* **Варіант B — Справжня IoC:** CLI перебирає каталоги і викликає `mod.run(ctx)` з кожного `fix.mjs`; правило саме вирішує оркестрацію, але за замовчуванням делегує `runStandardRule`.
* **Варіант C — Hybrid:** CLI спочатку шукає `<id>/fix.mjs`, інакше fallback на convention-discovery.

## Decision Outcome
Chosen option: "Варіант B (справжня IoC) з обмеженням дрейфу через `RuleContext`", because user обрав повну IoC-інверсію та mandatory entry-point для підтримки прямого запуску, IDE Run, CI per-rule matrix і майбутньої portability; convention-drift стримується забороною локальної логіки в `fix.mjs` — усі відхилення мають бути enumerated options у `RuleContext` у центральному `run-standard-rule.mjs`.

### Consequences
* Good, because transcript фіксує очікувану користь: `bun npm/rules/<id>/fix.mjs` запускає правило без CLI; CI може робити matrix по каталогах; `import.meta.main` блок забезпечує direct-run.
* Bad, because Variant B є mandatory — 30 правил мають отримати `fix.mjs` атомарно; перехідний fallback відсутній; існуючі `tests/*.test.mjs` з літеральними шляхами `fix/<concern>/` потребують оновлення після rename `fix/` → `js/`.

## More Information
Контракт `fix.mjs` (8 рядків, уніформний для 30 правил): `export async function run(ctx?) → Promise<number>`; CLI перебирає `rules/<id>/` через `listRuleIds` + `dynamic import('<id>/fix.mjs').run(ctx)`. Нові utils: `scripts/utils/run-standard-rule.mjs`, `scripts/utils/walk-cache.mjs`, `scripts/utils/list-rule-ids.mjs`. Spec: `docs/superpowers/specs/2026-05-23-per-rule-fix-mjs-entry-point-design.md`.

## ADR Rename `fix/` → `js/` у структурі правил

## Context and Problem Statement
Після рішення про корневий файл `rules/<id>/fix.mjs` виникла колізія імен: каталог `rules/<id>/fix/` і новий файл `rules/<id>/fix.mjs` мають однаковий basename, що призводить до плутанини при читанні структури каталогу та IDE-tree view.

## Considered Options
* Перейменувати `fix/` → `js/` (технологічна класифікація, аналогічно до `policy/` для Rego).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Rename `fix/` → `js/`", because усуває колізію імен між папкою і файлом; приводить у відповідність семантику: `js/` = JS-concerns (технологія), `policy/` = Rego-concerns (технологія), тоді як попереднє `fix/` означало функцію ("rule-fix"), що провокувало плутанину ще до появи `fix.mjs`.

### Consequences
* Good, because transcript фіксує очікувану користь: структура `rules/<id>/js/<concern>/check.mjs` і `rules/<id>/policy/<concern>/` стає симетричною і читається одноманітно.
* Bad, because transcript фіксує один підтверджений негативний наслідок: існуючі `tests/*.test.mjs` (наприклад `tests/check-empty-trees.test.mjs`) містять літеральні шляхи `'../rules/docker/fix/lint/check.mjs'` — вони потребують оновлення в рамках тієї самої атомарної PR.

## More Information
Rename виконується через `git mv rules/<id>/fix rules/<id>/js` для збереження git-history. Scope: 30 правил, де є каталог `fix/`. Команда-перевірка після міграції: `find npm/rules -maxdepth 2 -type d -name fix | wc -l` має повернути 0. Spec: `docs/superpowers/specs/2026-05-23-per-rule-fix-mjs-entry-point-design.md`, секція R8.

## ADR Module-level singleton для `walkCache`

## Context and Problem Statement
CLI `n-cursor check` потребує спільного FS-walk кешу між усіма правилами щоб уникнути повторних обходів одних і тих самих каталогів. Після переходу до IoC-моделі (CLI викликає `mod.run(ctx)`) виникло питання: як прокидати кеш — через параметр у `RuleContext` чи через module-level singleton.

## Considered Options
* **Module-singleton:** `utils/walk-cache.mjs` експортує lazy-init `Map`; всі споживачі імпортують без передачі через параметри.
* **Ctx-передача:** CLI створює `walkCache: Map` і прокидає як поле `RuleContext` через усі рівні (CLI → fix.mjs → runStandardRule → runRule).

## Decision Outcome
Chosen option: "Module-singleton", because CLI запускається як одиничний процес з одним деревом FS; singleton усуває потребу прокидати Map через 3 рівні; для тестів `resetWalkCache()` дозволяє clean-up у `beforeEach`; при паралельних воркерах (якщо з'являться) кожен воркер отримає власний module-instance — природна ізоляція без додаткового коду.

### Consequences
* Good, because transcript фіксує очікувану користь: нульовий boilerplate для прокидання; `getOrCreateWalkCache()` викликається в `runStandardRule` автоматично без участі `fix.mjs`.
* Bad, because module global state може вкусити в unit-тестах якщо `resetWalkCache()` не викликається між кейсами — transcript фіксує це як відомий ризик, вирішений явним `resetWalkCache()` у тестовому setup'і.

## More Information
API: `getOrCreateWalkCache(): Map<string, Promise<string[]>>`, `resetWalkCache(): void`. Файл: `scripts/utils/walk-cache.mjs`. Spec: `docs/superpowers/specs/2026-05-23-per-rule-fix-mjs-entry-point-design.md`, секція R6.
