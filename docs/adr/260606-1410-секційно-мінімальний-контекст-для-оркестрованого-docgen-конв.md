## ADR Секційно-мінімальний контекст для оркестрованого docgen-конвеєра

## Context and Problem Statement
Оркестрований docgen-конвеєр (gemma3:4b) у v1 інгестував повний вихідний код для кожної секції документа (`Огляд`, `Поведінка`, `Гарантії`, `Публічний API`), через що час генерації в 2-5 разів перевищував one-shot режим. Потрібно було зробити оркестрацію конкурентоспроможною за часом, зберігши якісну перевагу над one-shot.

## Considered Options
* Секційно-мінімальний контекст: код — лише у `behavior`-секції, решта секцій отримують лише факт-лист (Stage 0)
* Повний код у кожному секційному запиті (v1)

## Decision Outcome
Chosen option: "Секційно-мінімальний контекст", because бенчмарк підтвердив: overlay 310→77с (×4), k8s 141→55с (швидше за one-shot), якість g3 ORCH зросла до 92% проти 47% для g3 one-shot на тих самих файлах.

### Consequences
* Good, because transcript фіксує очікувану користь: час оркестрованого режиму порівняно з one-shot (52с vs 53с середнє), якість 92% vs 47% one-shot на `k8s-tree`-кейсі.
* Bad, because для gemma4:4b (Gemma 3n E4B) підхід фундаментально зламаний — модель повертає порожній рядок для секцій без коду в контексті; transcript підтверджує як для split system/user, так і для merged single-user.

## More Information
- Змінені файли: `npm/skills/docgen/js/docgen-gen.mjs`, `npm/skills/docgen/js/docgen-prompts.mjs`
- Функція `sectionMessages` у `docgen-prompts.mjs`: `behavior`-секція отримує `src` + `factsTxt`; `overview`, `api`, `guarantees` — лише `factsTxt`
- Бенч проведено на трьох файлах: `firebase_hosting.mjs`, `overlay-paths.mjs`, `k8s-tree.mjs` із `npm/rules/abie/`
- Commit: `17cfca32` (worktree `feat/docgen-orchestrator-pi`), злитий у `main` через `5e9d3a7b`

## ADR Stage-2 детермінований лінт і негативні маркери кешу

## Context and Problem Statement
Після переходу на секційно-мінімальний контекст залишилися два стійкі патерни деградації якості: 1) у секції `Поведінка` модель echo-їть сигнатури функцій (`isUaKustomizationPath(rel)`), 2) у `Гарантіях` модель вигадує кешування навіть для файлів без жодного кешу — pattern-matching «гарантії → кеш».

## Considered Options
* Детермінований `stripSignatures` (0 токенів) + негативний факт `'Кешування: НЕМАЄ'` у факт-листі
* Додатковий LLM-промпт для виправлення витоків
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Детермінований `stripSignatures` + негативний факт", because 0 токенів, модель-агностичний, перевірено на обох режимах (orchestrated і one-shot).

### Consequences
* Good, because transcript фіксує очікувану користь: витоки сигнатур усунені (overlay `Поведінка` — без `(rel)`), cache-галюцинація в k8s-tree зникла після додавання негативного маркера.
* Bad, because `stripSignatures` може видалити аргументи там, де вони є частиною смислового опису (не підтверджено в transcript як зафіксованою проблемою).

## More Information
- `stripSignatures` у `docgen-gen.mjs`: regex `([`\w$.]+)\([^()]*\)` → `$1`, 2 ітерації; застосовується після кожного секційного виклику і після one-shot
- Негативний маркер у `factsSummary` (`docgen-prompts.mjs`): `m.caches ? 'Кешування: так, у межах прогону' : 'Кешування: НЕМАЄ — не згадуй кеш у гарантіях'`
- `extractInternalSymbols` у `docgen-extract.mjs`: витягує імена з internal (`./`) імпортів і передає у `facts.internalSymbols` для заборони в `behavior`-промпті
- Commit: `45a7466c`, злитий у `main` через `5e9d3a7b`

## ADR Підхід B (LLM-суддя) непридатний як кількісний поріг якості

## Context and Problem Statement
Після тестового запуску docgen на `npm/rules/ga` (4 файли, середня якість ~63% за ручною оцінкою) постало питання: як автоматично виявляти документи нижче порогу якості для передачі на хмарний прохід?

## Considered Options
* Підхід B: gemma3:4b оцінює власний вивід за 4 критеріями (behavioral, no_leaks, structure, accuracy), числовий скор 0-100
* Підхід A: детермінований `scoreDoc` на основі фактів Stage 0 (без LLM)
* Підхід C (Hybrid): детермінований gate + LLM-текст як issue-reason

## Decision Outcome
Chosen option: "Hybrid (Підхід C)", because Підхід B виявив реальні проблеми у тексті коментарів, але скор системно завищений на +24 пп (діапазон +15..+35) — непридатний як числовий поріг. Детермінований скор як gate, LLM-текст як human-readable reason для хмарного агента.

### Consequences
* Good, because transcript підтверджує: LLM-суддя коректно ідентифікував усі три реальні проблеми (cache-галюцинація у fix.mjs, Rego-leak у workflows.mjs, внутрішнє ім'я `runLintGaCli` у lint/lint.mjs).
* Bad, because transcript фіксує підтверджений негативний наслідок: числові оцінки (+24 пп завищення в середньому) непридатні для автоматичного порогу без калібрування.

## More Information
- Бенч-скрипт: `~/docgen-bench3/judge_b.mjs`
- 4 файли `npm/rules/ga/`: `fix.mjs` (LLM=85%, manual=50%), `js/lint.mjs` (90%/75%), `js/workflows.mjs` (85%/58%), `lint/lint.mjs` (85%/67%)
- Запланована реалізація: `scoreDoc(md, facts)` у `docgen-gen.mjs` + Claude Haiku patch-промпт для секцій нижче порогу
