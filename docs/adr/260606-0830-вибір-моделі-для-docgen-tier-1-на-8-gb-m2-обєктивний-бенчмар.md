## ADR Вибір моделі для docgen Tier 1 на 8 GB M2: об'єктивний бенчмарк

## Context and Problem Statement
Для локального Tier 1 docgen-конвеєра на машині з 8 GB unified memory потрібно обрати модель Ollama, яка дає найкращий баланс якості документації та швидкості за умови повного вміщення в GPU без свопу.

## Considered Options
* `gemma3:4b` (3.3 GB) — 100% GPU
* `qwen2.5-coder:3b` (2.4 GB) — 100% GPU
* `qwen2.5:7b` / `qwen2.5-coder:7b` (5.1 GB) — 9%/91% CPU/GPU офлоад
* `llama3.1:8b` (5.7 GB) — 13%/87% CPU/GPU офлоад
* `llama3.2:3b`, `gemma2:2b`, `phi3.5`, `qwen3:4b` — відкинуті (мова/контроль виводу)

## Decision Outcome
Chosen option: "`gemma3:4b` (якість-first) + `gemma4:4b` (як alias `batiai/gemma4-e4b:q4`, якість-first з вищою якістю)", because об'єктивний бенчмарк з ручною оцінкою проти еталона показав: лише `gemma3:4b` та `qwen2.5-coder:3b` повністю вміщуються в GPU; `gemma3:4b` виграла за якістю (~86% vs 77%); пізніше виміряна `gemma4:4b` (5.3 GB, q4) дала ~92% при частковому офлоаді (56%/44%) і ~11 tok/s — обрана як нова якість-first альтернатива.

### Consequences
* Good, because transcript фіксує очікувану користь: `gemma3:4b` — 100% GPU, ~20 tok/s, ~14.5 год на 1042 файли; `gemma4:4b` — ~92% якості, читає system-prompt обмеження (не витікає stdlib/сигнатури на відміну від `gemma3:4b`).
* Bad, because `gemma4:4b` при 5.3 GB частково офлоадиться → ~11 tok/s, ~30 год на проєкт; вибір між якістю і швидкістю — ручний (`--model`).

## More Information
Бенчмарк-скрипти: `/tmp/docgen-bench2/run.py` (3 нові файли, еталони від Claude); виміряно `ollama ps` (RAM, processor split), wall_s, gen_tps, ручна оцінка за рубрикою 4×25%. `qwen3:4b` дискваліфікована: ігнорує `/no_think`, генерує «думання» російською. `ollama cp batiai/gemma4-e4b:q4 gemma4:4b` — alias без додаткового дискового місця; `~/.pi/agent/models.json` оновлено: `gemma4:4b` перша, `gemma3:4b` друга в провайдері `ollama`.

## ADR System-prompt — головний важіль якості локальної моделі, не транспорт

## Context and Problem Statement
Порівняння прямого Ollama-виклику (без system-prompt) з викликом через `pi` CLI показало різницю якості ~71% vs ~87%. Потрібно з'ясувати причину, щоб обрати правильну архітектуру Tier 1.

## Considered Options
* `pi` як транспорт (вбудований system-prompt coding-assistant)
* Прямий `ollama /api/chat` без system-prompt
* Прямий `ollama /api/chat` з явним system-prompt

## Decision Outcome
Chosen option: "Прямий `ollama /api/chat` з явним system-prompt", because контрольований експеримент (A: прямий без system ~71%, B: pi ~87%, C: прямий + system ~85%) довів: 15 п.п. приросту дає system-prompt, а не сам транспорт pi. Різниця B↔C (2 п.п.) — у межах шуму суб'єктивного оцінювання (~±3 п.п.).

### Consequences
* Good, because прямий HTTP без node-старту (~4 с/виклик на pi); повний контроль `num_ctx`/`num_predict` (усуває обрізання виводу, яке підвело варіант A на `overlay-paths`); простіша інтеграція в JS-оркестратор.
* Bad, because pi лишається валідним варіантом для разових запусків (`pi -p --no-tools --no-session --append-system-prompt "$STYLE"`); різниця між pi і прямим HTTP — лише в стелі якості system-prompt, яку закриває допрацювання промпта.

## More Information
Конфіг прямого виклику: `POST http://localhost:11434/api/chat`, `{model, messages:[{role:"system",...},{role:"user",...}], stream:false, options:{num_ctx:8192, num_predict:1100, temperature:0.2}}`. System-prompt: заборона `` ``` ``-огорожі, заборона stdlib/regex/типів/сигнатур, вимога фіксувати крайові деталі (що пропускається, fail-safe). Виміряний pi boot overhead: ~3.8–5 с/виклик (перший ~6.4 с). Скрипти: `~/docgen-bench3/run.py`, `~/docgen-bench3/duel.py`, `~/docgen-bench3/confirm.py`.

## ADR Per-file виклик замість RPC-сесії для docgen Tier 1

## Context and Problem Statement
`pi` підтримує два headless-режими: per-file процес (`pi -p`) і персистентна RPC-сесія (`pi --mode rpc`). Для docgen Tier 1 (1042 незалежних файлів) потрібно обрати патерн, що мінімізує час і не допускає крос-забруднення між файлами.

## Considered Options
* Персистентна pi RPC-сесія (`pi --mode rpc`) — один процес, батч файлів через stdin
* Per-file виклик `pi -p` — новий процес на кожен файл (~4 с node-boot)
* Прямий `ollama /api/chat` per-file — HTTP без pi-накладних витрат

## Decision Outcome
Chosen option: "Per-file виклик (або прямий HTTP per-file)", because RPC-сесія для docgen концептуально неправильна: файли незалежні, але в одній сесії контекст накопичується — файл N тягне за собою файли 1…N-1 та їхні доки → кожен наступний повільніший і зростає ризик крос-забруднення. Вимір підтвердив: RPC дав 267/144/296 с на 3 файли проти 31/124/70 с у per-file pi.

### Consequences
* Good, because per-file ізоляція гарантує відсутність крос-забруднення між файлами; прямий HTTP прибирає ~4 с node-boot (~+1.1 год на 1042 файлах).
* Bad, because per-file pi платить ~3.8–4 с node-boot на кожен файл; на 1042 файлах — зайва ~1.1 год відносно прямого HTTP. Прискорення RPC (boot ~1.1 с) не компенсує деградацію від накопиченого контексту.

## More Information
RPC boot overhead: ~1.1 с (boot-фаза), per-file pi ~3.8 с, прямий HTTP < 1 с. Уповільнення RPC (267/144/296 с): memory thrashing на 8 GB — pi-node + ollama + накопичений контекст конкурують за unified memory. `ollama ps` під RPC: `CONTEXT 4096, 100% GPU` — thrashing від розміру самого node-процесу pi, а не контекстного вікна. Персистентна сесія доцільна лише всередині одного файлу (секційні промпти → KV-cache спільного prefix).

## ADR JS-оркестратор як точка входу docgen (інверсія керування)

## Context and Problem Statement
One-shot підхід (модель отримує файл + один промпт і пише весь `.md`) дає нестабільну якість: витікають сигнатури, stdlib, внутрішні імена; галюцинації у «Гарантіях»; пропуск крайових деталей. Це стеля архітектури «один промпт» — не залежить від транспорту.

## Considered Options
* One-shot: один промпт → модель пише весь `.md` самостійно
* Секційні промпти через `pi` з `--append-system-prompt`
* JS-оркестратор: Stage 0 (детермінована екстракція, 0 токенів) → Stage 1 (секційні LLM-промпти з мінімальним контекстом) → Stage 3 (детермінована зборка)

## Decision Outcome
Chosen option: "JS-оркестратор як точка входу", because інверсія керування виносить факти й структуру з-під моделі: JS детерміновано знає імена експортів, stdlib-vs-internal, JSDoc, поведінкові маркери (`readOnly`, `catchesErrors`, `skips`, `caches`) — і передає їх моделі як контекст. Модель лишає тільки «перефразування відомих фактів у поведінкову прозу».

### Consequences
* Good, because Stage 0 `docgen-extract.mjs` витягує per-function JSDoc для всіх експортів — API-секція практично готова без LLM; правильна класифікація stdlib/internal; smoke-тест: без витоку stdlib і без довільних сигнатур.
* Good, because бенчмарк v1 підтвердив +6 п.п. якості vs one-shot (~86% vs ~80% на gemma3:4b), детермінована структура (без дублювання секцій, API авто-пропускається для однофункційних файлів).
* Bad, because складність вища за one-shot (3 модулі); Stage 0 прив'язаний до JS/MJS — для `.vue`/`.py` потрібні окремі екстрактори або fallback до one-shot; v1 у 2–5× повільніша (код інгеститься у кожну секцію повторно).

## More Information
Реалізація у `.worktrees/feat-docgen-orchestrator-pi` (гілка `feat/docgen-orchestrator-pi`): `npm/skills/docgen/js/docgen-extract.mjs` (Stage 0), `npm/skills/docgen/js/docgen-prompts.mjs` (Stage 1), `npm/skills/docgen/js/docgen-gen.mjs` (CLI-оркестратор, прапор `--oneshot` для AB). Конвенція: персистентна сесія між різними файлами шкідлива (накопичення); всередині одного файлу — доцільна (KV-cache стабільного `system + код` префіксу).

## ADR docgen-orchestrator v2: section-minimal context

## Context and Problem Statement
`docgen-prompts.mjs` v1 передає повний код файлу у system-контексті для всіх секційних промптів. Бенчмарк v1 показав: +6 п.п. якості vs one-shot (~86% vs ~80%), але у 2–5× повільніший (firebase: 102 с vs 19 с, overlay: 310 с vs 57 с). Потрібна v2-архітектура, що зберігає якість і наближає швидкість до one-shot.

## Considered Options
* v1: повний код у system-контексті для всіх 4 секцій
* v2: section-minimal context — код **тільки** у секцію `Поведінка`; `Огляд`, `Публічний API`, `Гарантії` отримують лише compact fact-list (exports, JSDoc, behavioral_markers)

## Decision Outcome
Chosen option: "v2: section-minimal context", because `Поведінка` потребує сирого коду (покрокова логіка, умови, крайові деталі); решта секцій повністю виводяться з compact fact-list (JSDoc, `exports[]`, маркери). Менший контекст скорочує prompt_eval_count і усуває необхідність платити за KV-інгест коду у кожній секції.

### Consequences
* Good, because `Огляд`/`API`/`Гарантії` отримують точно ті факти, що потрібні — без відволікаючого сирого коду; промпти менші → швидший prefill, менше ризику «деградувати» у код-переказ.
* Bad, because v2 не реалізована й не виміряна на момент фіксації рішення — цифри прогнозовані (очікується час < one-shot при якості ≥ v1).

## More Information
Бенчмарк v1 vs oneshot (gemma3:4b, ті самі 3 файли): oneshot ~48 с/файл, orchestrated v1 ~184 с/файл. Планові зміни: `docgen-prompts.mjs` — `sectionMessages(section, facts, src)` де `src` передається тільки для `section === 'behavior'`; `docgen-gen.mjs` — відповідне допасування. Реалізацію заплановано у `.worktrees/feat-docgen-orchestrator-pi`.
