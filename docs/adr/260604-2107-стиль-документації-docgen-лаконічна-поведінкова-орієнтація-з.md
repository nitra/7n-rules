## ADR Стиль документації docgen: лаконічна поведінкова орієнтація замість вичерпної реалізаційної

## Context and Problem Statement

Скіл `docgen` генерував md-документацію, що містила надлишкові реалізаційні деталі: таблиці залежностей (`node:fs`, `node:fs/promises`, `node:path`), сигнатури внутрішніх функцій (`isAbieRuleEnabled`), секції `## Залежності` і `Rebuild Test`-переліки. Такий стиль не відповідав бізнес-орієнтованій документації і не допомагав зрозуміти призначення файлу без читання коду. Еталонний файл `npm/rules/abie/lib/docs/enabled.md` демонстрував проблему.

## Considered Options

* Повністю поведінковий стиль: Огляд, Поведінка, Де використовується, Гарантії поведінки — без таблиць stdlib, без сигнатур, без `## Публічний API` для тривіальних предикатів
* Зберегти поточний «вичерпний» стиль із сигнатурами й переліком залежностей

## Decision Outcome

Chosen option: "Повністю поведінковий стиль", because користувач схвалив запропонований переписаний `enabled.md` і попросив адаптувати під нього скіл.

### Consequences

* Good, because transcript фіксує очікувану користь: перегенеровані 15 файлів `npm/rules/abie` вже в новому стилі й суб'агенти підтвердили, що прибрали `## Залежності`, сигнатури й `Rebuild Test`.
* Bad, because раніше згенерована документація поза `npm/rules/abie` (400+ файлів з `exists: true`) лишилася в старому реалізаційному стилі й потребуватиме повторного прогону з `--overwrite`.

## More Information

Змінені файли: `npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md`, `.pi/skills/n-docgen/SKILL.md`. Eталонний файл, перезаписаний вручну й потім перегенерований субагентом: `npm/rules/abie/lib/docs/enabled.md`. Ключові правки в промпті Tier 1: прибрано пункт «Публічні exports — можна назвати»; для тривіальних leaf-модулів `## Публічний API` пропускати; `## Помилки` → `## Гарантії поведінки`; заборонено сигнатурні таблиці типів і перелік stdlib. Слово «вичерпну» замінено на «лаконічну поведінкову» у `description` скіла в усіх трьох копіях.

## ADR Маршрутизація Tier 1 docgen на локальний Ollama замість хмарних субагентів

## Context and Problem Statement

Tier 1 скіла `docgen` (1042 файли-кандидати, ≈26 M хмарних токенів) є найбільшою статтею витрат (~$300–570 за один прогін). Кожен файл — незалежна й механічна задача «прочитай файл → запиши `.md`», що не потребує синтезу. Постало питання, чи можна замінити хмарні субагенти на локальну Ollama-модель.

## Considered Options

* Tier 1 → локальна Ollama (`gemma3:4b`), Tier 2/3 → Claude
* Tier 1 → локальна Ollama (`gemma4:e4b` — вже встановлена)
* Зберегти всі три рівні на Claude

## Decision Outcome

Chosen option: "Tier 1 → локальна Ollama (`gemma3:4b`), Tier 2/3 → Claude", because замір підтвердив: `gemma4:e4b` (9.6 GB) свопиться на 8 GB RAM (0.4 tok/s, непрацездатно), тоді як `gemma3:4b` (3.3 GB) вміщається і дає 14 tok/s; якість української визнана прийнятною для чорнового драфту Tier 1.

### Consequences

* Good, because transcript фіксує очікувану користь: усунення ~26 M хмарних токенів Tier 1 → $0; Tier 2/3 залишається на Claude (~2.6 M токенів, невисока вартість).
* Bad, because Tier 1 стає строго послідовним (concurrency = 1) — хмарний паралелізм «батчами по 5» зникає; повний прогін на 8 GB M2 займає ~22–38 год (≈1–1.5 доби). Якість 4B-моделі нижча за Claude: схильна огортати вивід у ` ```markdown `-fence, може називати внутрішні символи, не надійно тримає тонкі правила стилю — потребує quality-gate і пост-обробки.

## More Information

Вимірювальний стенд: Apple M2, 8 GB unified RAM, macOS. Команди: `ollama pull gemma3:4b`, `POST http://localhost:11434/api/generate`. Виміряно: cold load 24 с, warm інгест 24.1 tok/s, warm генерація 14 tok/s, реальний файл `enabled.mjs` — 76 с. Для порівняння `gemma4:e4b`: генерація 0.4 tok/s, реальний файл — таймаут >10 хв. Пропонована CLI-команда (ще не реалізована): `npx @nitra/cursor docgen gen --engine ollama --model gemma3:4b --concurrency 1 --max-bytes … --strip-fences`. Потребує виносу Tier-1 промпту в `npm/skills/docgen/js/docgen-prompt.mjs` як спільне джерело правди для обох двигунів.
