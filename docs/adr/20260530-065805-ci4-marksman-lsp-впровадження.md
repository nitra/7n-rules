# Впровадження Zed + marksman LSP в правило ci4

**Status:** Accepted
**Date:** 2026-05-30

## Context and Problem Statement

Правило `n-ci4.mdc` рекомендувало MkDocs Material як viewer для архітектурної документації. Єдина вимога: переглядати документацію файл-за-файлом у Zed без запуску dev-сервера. Необхідно оновити правило, забезпечити конфіг marksman через `.marksman.toml`, автоматизоване створення конфігу та enforcement VSCode-розширення `arr.marksman`.

## Considered Options

- MkDocs Material + `mkdocs serve`
- VitePress + `bun dev`
- Docusaurus, Starlight, Antora, Hugo, Sphinx — відхилені як такі, що вимагають запуску процесів
- Zed built-in MD preview + marksman LSP (без site-generator)

## Decision Outcome

Chosen option: "Zed built-in MD preview + marksman LSP без site-generator", because це єдиний варіант, що задовольняє вимогу «file-by-file без запуску сервера»: Zed рендерить CommonMark + GFM + Mermaid (`cmd+shift+m`), marksman дає `[[wiki-link]]`-навігацію і outline без збирача.

### Consequences

- Good, because нуль запущених процесів; marksman-LSP резолвить `[[slug]]` через `wiki.style = "file-stem"` — не ламається при зміні H1.
- Good, because `fix ci4` детермінистично виявляє відсутній `arr.marksman`; rego unit tests 5/5 PASS.
- Good, because `marksman_config.mjs` ідемпотентно створює `.marksman.toml` і не перетирає ручні правки.
- Good, because forward-compatible — site-generator підключається пізніше без переписування контенту.
- Bad, because framework-specific синтаксис (pymdownx `!!! note`, VitePress `:::`, MDX, Hugo shortcodes) не рендериться — введено portable-only subset: CommonMark + GFM + Mermaid + KaTeX + HTML5 `<details>`.
- Bad, because timestamp-draft-файли з `docs/adr/` потрапляють в автокомпліт marksman як шумні `[[20260530-...]]`; transcript обирає не змінювати це.
- Neutral, because `markdown.file_extensions` не розширено на `.mdc` — операційні правила не є частиною docs-workspace.

## Update 2026-05-30

**LLM prompt skeleton:** промпт-скелет LLM-проекцій у `ci4.mdc` оновлено під `<details>`: перевірки `??? engineer` → перевірки `<details>`-обрамлення; порожні рядки навколо `<details>` закріплені у валідаторній секції. Bump `npm` 1.30.1 → 1.31.0 (проміжний крок до marksman_config).

## Update 2026-05-30

**`wiki.style = "file-stem"` rationale:** ADR-slug є стабільним ідентифікатором у `sources="..."`, manifest і валідаторі — не ламається при зміні H1 під час normalize-проходу, тоді як `title-slug` ламав би посилання. `markdown.file_extensions` не розширено на `.mdc` — операційні правила не є частиною docs-workspace.

## Update 2026-05-30

Реалізовано JS-концерн `marksman_config` для автоматичного створення `.marksman.toml`.

**Вибір реалізації:** JS-концерн `marksman_config` за паттерном `test.stryker_config` (ідемпотентне копіювання baseline через `ensureBaselineFile`) замість Rego-policy (яка тільки валідує, але не створює файли) або документування з ручним кроком.

**Обґрунтування:** policy-рівень лише фіксує порушення; JS-концерн може копіювати baseline ідемпотентно — цей паттерн вже застосовано для `stryker.config.mjs`/`vitest.config.js` у правилі `test`.

**Результат:** перший `npx @nitra/cursor fix ci4` → `.marksman.toml` створюється; повторний прогін → idempotent pass (ручні правки не перетираються).
