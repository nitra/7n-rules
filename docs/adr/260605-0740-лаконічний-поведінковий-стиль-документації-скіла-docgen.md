## ADR Лаконічний поведінковий стиль документації скіла `docgen`

## Context and Problem Statement

Скіл `n-docgen` генерував `.md`-документацію, що містила зайві деталі реалізації: сигнатури функцій, таблиці типів, перелік stdlib-залежностей, секцію `## Залежності`, перелік тестів `Rebuild Test`. Еталонний файл `enabled.md`, створений старою версією промпту, демонстрував цей «bloat». Постала задача привести промпт до лаконічного поведінкового стилю, де документ описує «що і навіщо», а не «як».

## Considered Options

* Переписати Tier-1 промпт на поведінковий стиль (поведінка/гарантії без сигнатур, stdlib і внутрішніх імен)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Переписати Tier-1 промпт на поведінковий стиль", because існуючий промпт давав документацію, орієнтовану на реалізацію, а не на поведінку; після правки секція `## Публічний API` з'являється лише для нетривіальних зовнішніх поверхонь, секція `## Помилки` перейменована на `## Гарантії поведінки` (інваріанти: read-only, не кидає винятків, fail-safe), а `## Залежності` і `Rebuild Test` — вилучені.

### Consequences

* Good, because transcript фіксує очікувану користь: всі 15 перегенерованих файлів `npm/rules/abie` (3 батчі по 5 субагентів) отримали документи без сигнатур, без stdlib-переліків і без `Rebuild Test`; еталонний `enabled.md` переписано і підтверджено у стилі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Змінені файли (дві синхронні копії промпту): `npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md`. Стаб-делегат `.pi/skills/n-docgen/SKILL.md` оновлено лише у частині опису (слово «вичерпну» → «лаконічну поведінкову»); логіка в ньому не дублюється. Перегенеровані файли: `npm/rules/abie/docs/fix.md`, `npm/rules/abie/js/docs/*.md` (6 файлів), `npm/rules/abie/lib/docs/*.md` (8 файлів).

## ADR Вибір локальної моделі `gemma3:4b` для Tier 1 docgen через Ollama

## Context and Problem Statement

Tier 1 скіла `docgen` (~1042 файли на цьому проєкті) генерує ~26 M хмарних токенів (≈$300–570 за прогін). Постало питання: чи можна перенести Tier 1 на локальну Ollama на машині Apple M2 / 8 GB RAM, щоб усунути цей кошт, і яка модель оптимальна для цих обмежень.

## Considered Options

* `gemma4:e4b` (9.6 GB) — попередньо встановлена модель
* `gemma3:4b` (3.3 GB)
* `qwen2.5-coder:3b` (2.4 GB)
* `qwen3:4b` (3.6 GB, reasoning)
* `qwen2.5:7b` / `qwen2.5-coder:7b` / `llama3.1:8b` (~5–5.7 GB)
* `llama3.2:3b`, `gemma2:2b`, `phi3.5` (< 3 GB, але слабка українська)

## Decision Outcome

Chosen option: "`gemma3:4b`", because це єдина модель у класі «повністю влазить у 8 GB (100% GPU, 4.2 GB resident) + пристойна українська + достатня якість документації» — виміряний gen ~14–21 tok/s (бенчмарк підтвердив), час Tier 1 ≈ 14–22 год в один потік, середня якість vs еталон ~86%; `gemma4:e4b` дає ~91%, але займає 10 GB (CPU-офлоад, свопиться) і утричі повільніша; `qwen2.5-coder:3b` удвічі швидша, але ~77% якості з витоком сигнатур і мовними помилками.

### Consequences

* Good, because transcript фіксує очікувану користь: на 8 GB `gemma3:4b` дає 100% GPU без свопу; виміраний реальний gen ≥ 14 tok/s; якість документів прийнятна для чорнового Tier 1 із подальшим синтезом Claude на Tier 2/3.
* Bad, because Tier 1 стає строго послідовним (concurrency = 1) — паралелізм хмарних субагентів зникає; повний прогін 1042 файлів займає ~14–22 год; якість (86%) помітно нижча за Claude; великі файли можуть потребувати Claude-хвоста при провалі quality-gate.

## More Information

Бенчмарк проводився у два раунди: `npx @nitra/cursor docgen scan` повернув 1442 файли (400 `exists:true`, 1042 нові). Заміри: прямий `POST http://localhost:11434/api/chat` (`stream:false`); чистий старт перед кожною моделлю (`ollama stop`); RAM — з `ollama ps`. Команда очищення після бенчмарку: `ollama rm qwen2.5-coder:3b gemma4:e4b` (і решта тимчасово завантажених). `qwen3:4b` відкинута через неможливість надійно вимкнути reasoning (`/no_think` проігноровано, мова переключилась на російську). 7B-клас відкинутий — CPU-офлоад і ризик свопу без виграшу в якості.

## ADR Доступ до локальної `gemma3:4b` через `pi` vs прямий Ollama API зі system-prompt

## Context and Problem Statement

Визначившись із моделлю `gemma3:4b`, постало питання: як правильно до неї звертатися з docgen-скіла — через CLI `pi` (провайдер ollama через `~/.pi/agent/models.json`) чи напряму через `POST http://localhost:11434/api/chat`. Початкові прогони показали різницю в якості між цими двома способами (87% у pi vs 71% без system-prompt у прямому виклику), що вимагало з'ясування реальної причини.

## Considered Options

* Прямий виклик `ollama /api/chat` без system-prompt (варіант A)
* Виклик через `pi --provider ollama` (варіант B)
* Прямий виклик `ollama /api/chat` зі сильним system-prompt (варіант C / C2)

## Decision Outcome

Chosen option: "Прямий виклик із системним промптом (C2) з урахуванням ергономіки pi", because серія контрольованих експериментів довела: різницю якості між A і B давав **system-prompt pi, а не сам інструмент** — варіант C2 (прямий + сильний system) вийшов на ту ж смугу (85–89% vs 78–87% у pi залежно від прогону); варіативність між прогонами всередині одного варіанта (~±9 п.п.) більша за систематичну різницю між B і C2.

### Consequences

* Good, because transcript фіксує очікувану користь: прямий виклик дає повний контроль `num_ctx`/`num_predict` (виключає обрізання, що спостерігалось у варіанті A), відсутній ~4 с overhead pi-старту на файл (~+1.1 год на 1042 файлах), простіша оркестрація в CLI `docgen gen`.
* Bad, because pi — валідний вибір із двох причин: (1) pi обробляє «сантехніку» (читання/запис файлів своїми інструментами, ретраї, сесія) і (2) його оверхед нейтралізується персистентною сесією (`--no-session` не використовувати); відмова від pi означає, що оркестрацію (читання → виклик → запис → зрізання fence) треба реалізовувати вручну в CLI.

## More Information

Конфігурація pi для ollama: `~/.pi/agent/models.json` (ключ `providers.ollama`, `api: "openai-completions"`, `baseUrl: "http://localhost:11434/v1"`). `pi --list-models` підтвердив `gemma3:4b` з контекстом 128K. pi-стартовий оверхед виміряно окремо: ~3.8–6 с/виклик. Виміряний інференс tok/s однаковий (~20) в обох варіантах — різниця часу між B і C2 (напр. overlay: 124 с vs 74 с) пояснюється схильністю pi до довшого виводу (+20–69% символів), а не накладними витратами транспорту. Ключовий важіль якості — наявність system-prompt, а не вибір транспорту.
