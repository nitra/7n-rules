## ADR Принцип розподілу директорій rules/<id>/: технологія визначає папку

**Контекст:** При міграції `rules/<id>/js/` → `fix/` виникло питання: куди класти rego-політики (`policy/`) — залишати окремо, переносити під `fix/policy/`, чи розміщати плоско у `fix/<concern>/`. Потрібно зафіксувати єдине правило, щоб не плутати канали.

**Рішення/Процедура/Факт:** Технологія реалізації визначає директорію у `rules/<id>/`: JS-діагностика та автофікс (запускається через `npx @nitra/cursor check`) → `fix/<concern>/check.mjs`; JS-оркестратори зовнішніх лінт-інструментів (запускаються через `bun run lint-X`) → `lint/lint.mjs`; rego-діагностика (запускається через `npx @nitra/cursor check` → `runConftestBatch`) → `policy/<concern>/<concern>.rego`. `policy/` є третім сиблінгом поруч із `fix/` і `lint/`, а не вкладеним у них.

**Обґрунтування:** `policy/` і `fix/` живлять один і той самий CLI-канал (`n-cursor check`), але мають різні runner-и, різні namespace-конвенції (snake_case у rego vs kebab у JS) і різну роль `target.json`. Плоске об'єднання у `fix/<concern>/` змішало б ці світи та створило б ризик колізії імен (наприклад, `fix/cspell/check.mjs` поруч із `fix/cspell/<name>.rego`). Варіант `fix/policy/<concern>/` відкинуто через зайвий рівень nesting. Залишення `policy/` на одному рівні з `fix/` і `lint/` є найчіткішим: читач одразу бачить технологію за назвою директорії.

**Розглянуті альтернативи:** `fix/policy/<concern>/` (відкинуто — зайвий nesting), плоско `fix/<concern>/<name>.rego` поруч із `fix/<concern>/check.mjs` (відкинуто — колізії імен, розчинення rego-екосистеми у JS).

**Зачіпає:** `npm/rules/*/` (структура кожного правила), `npm/scripts/utils/discover-checkable-rules.mjs` (окремі масиви `jsConcerns` / `policyConcerns`), `npm/README.md` (нова секція «Структура одного правила»), `.cursor/rules/scripts.mdc`.

## ADR Видалення lint-conftest як дублюючого каналу запуску rego-політик

**Контекст:** `npm/scripts/lint-conftest.mjs` і `bun run lint-conftest` запускали rego-політики через фіксований список `TARGETS` у `package.json`-скрипті. Водночас `npx @nitra/cursor check` робить те саме через `discoverCheckableRules` → `runConftestBatch`, автоматично знаходячи `policy/<concern>/target.json`. Два канали розходились у покритті й синхронізації.

**Рішення/Процедура/Факт:** `npm/scripts/lint-conftest.mjs` видалено повністю. З кореневого `package.json` прибрано скрипт `lint-conftest` та його ланку у `lint`-chain (`bun run lint-rego && bun run lint-conftest && ...`). У 10+ `.rego` і `.mjs` файлах замінено `bun run lint-conftest` → `npx @nitra/cursor check`. `.cursor/rules/conftest.mdc` переписано: крок 5 (реєстрація нового пакета) тепер пояснює, що `discoverCheckableRules` підхоплює rego автоматично за наявністю `target.json` — окрема реєстрація не потрібна. Умовні rego-пакети (без `target.json`, наприклад `tauri/policy/vscode_extensions/`) позначено в коментарях як «не auto-discoverable».

**Обґрунтування:** `lint-conftest` вимагав ручної підтримки `TARGETS`-списку і при додаванні нового правила легко забувався. `n-cursor check` використовує discovery за конвенцією файлової системи, тому завжди актуальний. Видалення усуває ризик розходження між двома каналами і спрощує `lint`-chain.

**Розглянуті альтернативи:** Залишити `lint-conftest` як «fast-feedback IDE-gate» (відкинуто — `n-cursor check` виконує рівно ту саму роботу, другий канал лише збільшує час CI і плутає розробника).

**Зачіпає:** `package.json` (кореневий), `npm/scripts/lint-conftest.mjs` (видалено), `.cursor/rules/conftest.mdc`, `.cursor/rules/scripts.mdc`, `npm/rules/abie/abie.mdc`, ~17 файлів `.mjs`/`.rego` з оновленими коментарями.

## ADR Завершення реструктуризації rules/*/js/ → fix/: видалення dual-mode інфраструктури (Фаза 3)

**Контекст:** Фаза 1 додала dual-mode у `discover-checkable-rules.mjs` і `run-rule.mjs` (підтримка і `js/`, і `fix/` одночасно, поле `concern.rootDir`). Фаза 2 перенесла всі 26 правил із `js/` у `fix/`. Після завершення міграції dual-mode став мертвим кодом зі штучним полем `rootDir` в інтерфейсі.

**Рішення/Процедура/Факт:** `discover-checkable-rules.mjs` перезаписано: функція `listJsConcerns` тепер сканує лише `fix/`; видалено `mergeJsConcerns` (fatal на дублікат), видалено поле `rootDir` із типу `JsConcern`. `run-rule.mjs`: `resolveJsCheckPath` хардкодить `'fix'`; JSDoc оновлено з `js/applies/` → `fix/applies/`. У тестах видалено 6 dual-mode сценаріїв (legacy `js/`, mix `js/`+`fix/`, fatal-дублікат), додано 2 нових: legacy `js/` ігнорується, multi-concern сортування. `knip.json` оновлено: entry-патерни `rules/*/js/**/check.mjs` → `rules/*/fix/**/check.mjs`, додано `rules/*/lint/lint.mjs`, `rules/*/lint/run-*.mjs`. Версія `1.11.12`.

**Обґрунтування:** Dual-mode — тимчасова міра на час паралельного існування `js/` і `fix/`. Після повного переїзду він перетворився на джерело плутанини (поле `rootDir` в інтерфейсі без практичного значення) та ускладнював розуміння коду. Single-source `fix/` відповідає затвердженому принципу «технологія визначає директорію».

**Розглянуті альтернативи:** Залишити `rootDir` із дефолтом `'fix'` для backward compat (відкинуто — жодного `js/` більше немає, fallback — мертвий код).

**Зачіпає:** `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/discover-checkable-rules.test.mjs`, `npm/scripts/utils/run-rule.mjs`, `npm/scripts/utils/run-rule.test.mjs`, `npm/bin/n-cursor.js` (JSDoc-коментар), `knip.json`.
