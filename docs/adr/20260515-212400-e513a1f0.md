## ADR: `n-fix` і `n-lint` залишаються окремими скілами; крок 6 у `n-fix` дедуплікується через `bun run lint`

**Контекст:** Виникло питання, чи варто злити `/n-fix` і `/n-lint` в одну команду — зокрема через те, що крок 6 у `/n-fix` самостійно запускає `lint-js`/`lint-text`/`lint-style`, що частково перетинається з `/n-lint`.

**Рішення/Процедура/Факт:** Злиття відхилено. Натомість обрано **дедуплікацію**: крок 6 у `.cursor/skills/n-fix/SKILL.md` замінюється на єдиний виклик `bun run lint` (або делегування `/n-lint`) замість перебору окремих `lint-*` скриптів. Ролі залишаються чіткими: `/n-fix` — привести структуру проєкту у відповідність до `.cursor/rules/` (конфіги, залежності, workflows); `/n-lint` — перевірити код через канонічний `bun run lint` з кореневого `package.json`.

**Обґрунтування:** Скіли мають різну частоту використання (`/n-fix` — разово, `/n-lint` — перед кожним комітом) і різну вартість виконання. Злиття нав'язує повний `bun run lint`-ланцюжок (jscpd, cspell, ESLint тощо) на кожен структурний фікс і навпаки. Крім того, `CLAUDE.md` явно забороняє паралельні запуски ESLint; злитий скіл важче утримати в межах цієї інваріанти, якщо `/n-fix` тригерить власні `lint-*` і `bun run lint` одночасно. Нарешті, `bun run lint` — канонічний контракт CI; ховати його всередину `/n-fix` розмиває межу "перевірити як CI" vs "приведи до правил".

**Розглянуті альтернативи:** Повне злиття в одну команду — відхилено через overhead і ризик порушення обмеження на паралельний ESLint.

**Зачіпає:** `.cursor/skills/n-fix/SKILL.md` (крок 6), `.cursor/skills/n-lint/SKILL.md` (без змін), `CLAUDE.md` (інваріанта щодо паралельних ESLint-запусків).
