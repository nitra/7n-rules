## ADR Детермінована драбина ескалації моделей у fix-конформності

## Context and Problem Statement
Рушій конформності `lint --full` (успадкований від `/n-fix`) використовував грубий цикл `min→avg` із лічильником `failCount`: 2 провали підряд → перемикання на `MODEL_HEAVY`. Механізм не давав feedback попереднього рунга наступному, не розрізняв системні помилки від prompt-фейлів і не залишав spостережуваного сліду «чи допомогло».

## Considered Options
* Детермінована чотирирунгова драбина: `local-min → local-min(feedback) → cloud-min → cloud-avg` до першого зеленого re-check
* Збереження поточного `failCount`-циклу з дрібними покращеннями

## Decision Outcome
Chosen option: "Детермінована чотирирунгова драбина", because user підтвердив рекомендований дизайн: рунг 2 (retry-with-feedback) лише для prompt-фейлів; на systemic-помилці локального — одразу рунг 3; глобальний кеп `--max-avg` (default 3) на `cloud-avg`-виклики.

### Consequences
* Good, because transcript фіксує очікувану користь: кожен рунг отримує feedback (поля `changes` + `remainingViolation` з попереднього рунга у prompt), що дозволяє моделі перечитати свої попередні зміни.
* Bad, because вичерпаний `--max-avg` лог фіксує (не мовчить), але cloud-avg все одно не виконується для правил, що залишились.

## More Information
- `npm/scripts/lib/fix/orchestrator.mjs` — нові exported функції `buildLadder`, `escalateRule`, `parseOrchestratorArgs`, `runOrchestratorCli`
- `buildLadder({ localMin, cloudMin, cloudAvg })` — фільтрує незадані тири; `escalateRule(rule, ladder, worker, check, log, opts)` — цикл по рунгах
- Тести: `npm/scripts/lib/fix/tests/orchestrator.test.mjs` (12 тестів, інжектовані `worker`/`check`)

## ADR Feedback попереднього рунга вбудований у `diagnosis`-поле, а не окремий LLM-виклик

## Context and Problem Statement
При ескалації в хмару постало питання: як модель має «зрозуміти, чому попередній рунг не вдався» — через окремий LLM-виклик («аналіз»), який потім передається у fix-виклик, чи через включення feedback безпосередньо в prompt fix-виклику.

## Considered Options
* Окремий LLM-виклик «аналізуй, чому не спрацювало» → результат → fix-виклик (подвоює вартість)
* Включити feedback (попередні `changes` + залишковий `violationOutput`) безпосередньо у prompt і попросити модель повернути поле `diagnosis` у тій самій відповіді

## Decision Outcome
Chosen option: "Включити feedback у prompt і поле `diagnosis` у тій самій відповіді", because user прийняв рекомендацію: «дешевий варіант на локальних рунгах; окремий «аналіз»-виклик — тільки при ескалації в хмару». Фактично навіть для хмарних рунгів рішення згорнуто в один виклик заради вартості й fail-fast.

### Consequences
* Good, because один виклик замість двох зберігає вартість і узгоджується з принципом fail-fast.
* Bad, because слабка `local-min`-модель дасть слабкий `diagnosis` — але це прийнятний компроміс, явно зафіксований у ADR.

## More Information
- `npm/scripts/lib/fix/llm-worker.mjs` — `runLlmWorker` тепер приймає `opts.feedback` (рядок з попередніми змінами + violation); повертає `{ ok, changes, diagnosis, error }`
- `buildPrompt` розширено: параметр `feedback?: string`; якщо заданий — додається секція `## Попередня спроба` у prompt
- `MODEL_HEAVY` (стара ескалаційна константа) видалено з модуля як мертвий код

## ADR Append-only JSONL escalation-лог із per-рунговими записами

## Context and Problem Statement
Наявний wire-trace (`omlx-trace.mjs`) фіксує кожен LLM-виклик, але не знає, чи конкретний виклик допоміг правилу стати зеленим. Оркестратор знає результат re-check, але не пише його нікуди. Без цього кореляція «виклик → результат» була неможлива.

## Considered Options
* Окремий `escalation-log.mjs` — один JSONL-запис на рунг із полями `ruleId`, `tier`, `model`, `callOk`, `callError`, `recheckOk`, `remainingViolation`, `diagnosis`, `ms`; join із wire-trace за `caller`-міткою `fix:<ruleId>:<rung>`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Окремий escalation-log.mjs із per-рунговими JSONL-записами", because user прийняв рекомендований формат; розташування поряд із omlx-trace (той самий каталог `.n-cursor/`, та сама ротаційна конвенція).

### Consequences
* Good, because transcript фіксує очікувану користь: `recheckOk` явно показує «чи допомогло»; `remainingViolation` обнуляється до `''` при `recheckOk=true`; join із wire-trace тривіальний за `caller`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Тести: `npm/scripts/lib/fix/tests/escalation-log.test.mjs`
- `caller`-мітка у wire-trace: `fix:<ruleId>:<rung>` (передається через `opts.caller` у `callLlm`)

## ADR Аналітичний скрипт escalation-логу з викликом cloud-avg наприкінці `lint --full`

## Context and Problem Statement
Після впровадження escalation-логу постала задача автоматично аналізувати накопичені рунги, щоб знаходити закономірності та пропонувати зміни в скриптах або LLM-інструкціях `@nitra/cursor`. Аналіз не повинен зупиняти `lint` при помилці хмари.

## Considered Options
* Новий модуль `analyze-escalation.mjs`: читає записи від байтового зсуву (тільки нові за прогін), чанкує якщо даних > `DEFAULT_CHUNK_CHARS`, викликає `cloud-avg`-модель map-reduce, пише звіт у `.n-cursor/fix-escalation-analysis.md`; вбудований у кінець конформності-фази `lint --full` через `maybeAnalyzeEscalation`; також CLI-subcommand `n-cursor analyze-escalation` для ручного запуску
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "analyze-escalation.mjs із вбудуванням у lint --full та CLI-subcommand", because user поставив точну вимогу: читати логи, ділити на частини, звертатись до avg-cloud-моделі для аналізу «як доповнити скрипти або адаптувати LLM-інструкції», і викликати аналіз наприкінці `lint`.

### Consequences
* Good, because transcript фіксує очікувану користь: аналіз пропускається без помилки якщо лог порожній, `N_CLOUD_AVG_MODEL` не заданий, або kill-switch `N_CURSOR_FIX_ESCALATION_LOG=0`; не валить `lint` при мережевій помилці (ковтає у `null`).
* Bad, because cloud-avg-виклик збільшує час `lint --full` на час аналізу (timeout 180 000 мс на чанк).
