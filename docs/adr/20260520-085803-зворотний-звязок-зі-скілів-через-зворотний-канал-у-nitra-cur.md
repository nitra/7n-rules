## ADR Зворотний зв'язок зі скілів через зворотний канал у `@nitra/cursor`

## Context and Problem Statement
Після сесій зі скілами (`n-lint`, `n-fix`, `n-llm-patch`) у споживацьких репо накопичується сигнал: що в правилах `@nitra/cursor` незрозуміло, чого не вистачає в `check`, які кроки скіла зайві. Потрібен канал «назад у пакет» без прямого коміту з чужого репо.

## Considered Options
* **A.** `alwaysApply` правило `feedback.mdc` — агент після `n-*` скіла додає блок «Зворотний зв'язок» у відповідь у чаті
* **B.** Розширити `n-llm-patch` — скіл генерує промпт для окремої сесії в `@nitra/cursor` (explicit handoff)
* **C.** Stop-hook + `capture-decisions.sh` — після сесії LLM пише draft у `docs/adr/`

## Decision Outcome
Chosen option: "B + C", because скіл `n-llm-patch` забезпечує явний self-contained handoff, а stop-hook автоматично ловить архітектурні рішення з transcript без додаткових дій. Варіант A (`alwaysApply feedback.mdc`) залишається опцією лише для миттєвого UX у чаті, якщо очікування ADR небажане.

### Consequences
* Good, because після завершення відповіді агента очікується один draft ADR у `docs/adr/YYYYMMDD-HHMMSS-<session>.md`.
* Bad, because якщо сесія не містить блоку `## ADR` у відповіді LLM — `capture-decisions.sh` пропускає запис (зафіксовано в логу: `response missing '## ' header`).

## More Information
Задіяні файли та команди: `docs/adr/`, `.claude/hooks/capture-decisions.sh`, `.cursor/hooks.json`, `npm/skills/n-llm-patch/SKILL.md`. Stop-хуки спрацьовують після події **stop** агента з таймаутами 180s / 600s згідно `.cursor/hooks.json`. Normalize-хук має cooldown `ADR_NORMALIZE_MIN_INTERVAL_HOURS=6`; для ручного обходу — `/n-adr-normalize`.
