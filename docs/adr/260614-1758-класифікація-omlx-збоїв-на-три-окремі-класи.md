## ADR Класифікація omlx-збоїв на три окремі класи

## Context and Problem Statement
Масовий прогін docgen на 266 файлах давав каскад помилок, починаючи з файлу ~58: три принципово різні збої (`spawnSync curl ETIMEDOUT`, `memory ceiling`, `Prompt too long`) оброблялись однаково — `catch → ✗ → push → continue`, без зупинки і без розрізнення природи збою.

## Considered Options
* Єдиний handler для всіх збоїв (поточний стан)
* Класифікатор трьох класів: `transient | systemic | permanent` з різними стратегіями реакції

## Decision Outcome
Chosen option: "Класифікатор трьох класів", because це єдиний спосіб застосувати відповідну стратегію до кожного типу: ретрай для мережевих збоїв, circuit-breaker для проблем середовища, skip для детерміновано нерозв'язних файлів.

### Consequences
* Good, because transcript фіксує очікувану користь: каскад зупиняється на рівні circuit-breaker замість тихого виконання 266 фейлів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Функція `classifyOmlxError` розміщена в `npm/lib/llm.mjs` (переюзовує наявні константи `MEMORY_GUARD_MARKER`, `AUTH_ERROR_MARKER`). Клас `systemic` покриває `memory ceiling`; `transient` — `curl error`/exit-коди 18, 28, 52, 56, `ETIMEDOUT`; `permanent` — `Prompt too long`. Тести: `npm/lib/tests/llm.test.mjs` (`describe('classifyOmlxError')`).

## ADR Circuit breaker для systemic-каскаду: K=1, без cooldown, exit 2

## Context and Problem Statement
Коли omlx-сервер потрапляє під `memory ceiling` (у сесії — через конкурентну резидентну модель), кожен наступний файл гарантовано падає з тим самим збоєм. Оркестратор продовжував прогін попри системне обмеження середовища.

## Considered Options
* K≥3 підряд-systemic з cooldown і повторним health-check перед abort
* K=1 (перший же systemic → негайний abort, без cooldown)

## Decision Outcome
Chosen option: "K=1, без cooldown", because користувач явно зазначив пріоритет: «швидше помилятись і рухатись далі, ніж довго чекати».

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор не витрачає час на curl-раунди для файлів, які детерміновано впадуть.
* Bad, because одиничний transient, помилково класифікований як systemic, перерве весь прогін — але класифікатор розрізняє їх на рівні маркерів, тому ризик мінімальний.

## More Information
Константа `SYSTEMIC_ABORT_STREAK = 1` у `npm/rules/doc-files/js/docgen-files-batch.mjs`. При спрацюванні — повідомлення в stdout з вказівкою на звільнення пам'яті й поверненням exit-коду 2. Невдалі файли не пишуться, лишаються stale — наступний прогін підбере їх через CRC. Тести: `npm/rules/doc-files/js/tests/docgen-files-batch.test.mjs`.

## ADR Transient retry з backoff [2000, 8000] мс — хардкод і ETIMEDOUT у transient-класі

## Context and Problem Statement
`spawnSync curl ETIMEDOUT` потрапляв у гілку `r.error` і викликав `break` замість `continue`, тобто жодного ретраю для найбільш відновлюваного збою. Існуючий набір transient-кодів `{18, 52, 56}` не покривав таймаут і exit-код 28.

## Considered Options
* Transient-backoff через env-змінні (конфігурований)
* Хардкод backoff `[2000, 8000]` мс, `ETIMEDOUT` і код 28 у transient-множині; через `opts.backoffMs` — лише для тестів

## Decision Outcome
Chosen option: "Хардкод", because користувач явно: «хардкод».

### Consequences
* Good, because transcript фіксує очікувану користь: усуває непотрібний ретрай-провал на мережевому таймауті без конфігураційного overhead.
* Bad, because `opts.backoffMs` публічно в сигнатурі `callOmlxRaw` — але це trade-off для тест-ізоляції, а не API-поверхня для кінцевих користувачів.

## More Information
`BACKOFF_MS = [2_000, 8_000]` у `npm/lib/omlx.mjs`; `callOmlxRaw` приймає `{ backoffMs = BACKOFF_MS }` — тести передають `[0, 0]`. Transient-множина: `{18, 28, 52, 56}`. `sleepSync` через `Atomics.wait`. JSDoc оновлено.

## ADR Permanent помилки — skip з stdout-діагностикою, без size-guard

## Context and Problem Statement
`run/auth/src/lib/lib/euscp.js` — мініфікований vendored-файл на ~9.17M токенів — детерміновано падає з `Prompt too long`. Раніше він потрапляв у `errors[]` як «треба перегнати», хоча ретрай нічого не дасть.

## Considered Options
* Pre-send size-guard (підрахунок токенів/байтів до виклику, поріг > 0.5 × max_context_window)
* Тільки stdout-діагностика (`⚠ size:`) і окремий лічильник `skipped`, без size-guard

## Decision Outcome
Chosen option: "Stdout-діагностика без size-guard", because користувач: «не потрібен, але робимо вивід в stdout для дослідження».

### Consequences
* Good, because transcript фіксує очікувану користь: permanent-файли не ретраяться, не псують лічильник помилок, але залишаються видимими для ручного аналізу.
* Bad, because файл все одно ходить на сервер і витрачає curl-раунд — але зупиняється на відповіді API, не через локальний guard.

## More Information
Клас `permanent` у `classifyOmlxError` (`npm/lib/llm.mjs`) покриває маркер `Prompt too long`. Оркестратор (`npm/rules/doc-files/js/docgen-files-batch.mjs`) відображає `⊘ skip` і підсумковий рядок `skipped: N`.

## ADR Виключення файлів зі сканування через .gitignore (git check-ignore)

## Context and Problem Statement
`docgen-ignore.mjs` містив хардкод-глоби (`**/node_modules/**`, `**/dist/**`, тощо), а мініфіковані/vendored-файли типу `euscp.js` не були в .gitignore і потрапляли у прогін.

## Considered Options
* Розширений набір глобів у `DOCGEN_IGNORE_GLOBS` (`**/lib/lib/**`, `**/*.min.js`, евристика довгих рядків)
* Делегування виключень повністю в `.gitignore` через `git check-ignore --stdin`

## Decision Outcome
Chosen option: "`git check-ignore --stdin`", because користувач: «тільки те що в .gitignore» — єдине джерело правди про ігноровані файли вже існує в репозиторії.

### Consequences
* Good, because transcript фіксує очікувану користь: не потрібно дублювати правила виключень у двох місцях.
* Bad, because `euscp.js` наразі не в `.gitignore`, тож ця зміна сама по собі не виключить його — для цього потрібен окремий `.gitignore`-запис.

## More Information
`execFileSync('git', ['check-ignore', '--stdin'], { cwd: root, input: relPaths.join('\n'), stdio: ['pipe','pipe','ignore'], encoding: 'utf8' })` у `npm/rules/doc-files/js/docgen-scan.mjs`. `stderr: 'ignore'` — щоб не шуміти в не-git-проєктах. Тест: `docgen-scan.test.mjs` — тимчасовий `git init` у tmp-дереві з `.gitignore`.

## ADR Видалення хардкоду DEFAULT_OMLX_MODEL — fail-loud без fallback-моделі

## Context and Problem Statement
`omlx.mjs` містив хардкод `DEFAULT_OMLX_MODEL = 'mlx-community--gemma-4-e2b-it-4bit'` як останній fallback. Після видалення `e2b`-моделі з сервера цей хардкод вказував на неіснуючу модель — в будь-якому шелі без `N_LOCAL_MIN_MODEL` (cron, headless, CI) виклик мовчки звертався б до фантому.

## Considered Options
* Оновити хардкод на актуальну модель (`e4b-OptiQ`)
* Прибрати хардкод повністю, fail-loud якщо модель не визначена

## Decision Outcome
Chosen option: "Прибрати хардкод, fail-loud", because користувач: «прибрати хардкод, кожен локально сам налаштовує»; принцип: явна помилка краща за тихий wrong-model.

### Consequences
* Good, because transcript фіксує очікувану користь: некоректна конфігурація середовища виявляється одразу на preflight, а не після сотень згенерованих доків невідомою моделлю.
* Bad, because `callOmlxRaw` з порожнім `model` і без `fallbackModel` тепер кидає Error замість мовчки ходити на сервер — але `model_fallback: true` на сервері покриває runtime-кейс «модель не та».

## More Information
`export const DEFAULT_OMLX_MODEL` видалено з `npm/lib/omlx.mjs`; `docgen-gen.mjs` більше не імпортує його. `omlxHealthCheck` тепер приймає `{ model }` і перевіряє його наявність до curl. Тест у `llm.test.mjs` оновлено: `omlxHealthCheck({ model: 'omlx/m' })`.

## ADR model_fallback: true в omlx server config

## Context and Problem Statement
При `model_fallback: false` запит до будь-якої невідомої моделі повертав `not_found_error`. Після видалення хардкоду `DEFAULT_OMLX_MODEL` запит зі stale model-id (або без env-змінної) призводив би до миттєвого 404 і каскаду, замість того щоб дістати відповідь від єдиної наявної моделі.

## Considered Options
* Залишити `model_fallback: false` (fail-loud на сервері), покрити тільки кодовим рішенням
* Увімкнути `model_fallback: true` — сервер перенаправляє на доступну модель

## Decision Outcome
Chosen option: "`model_fallback: true`", because користувач явно попросив увімкнути.

### Consequences
* Good, because transcript фіксує підтверджений результат: запит до `foo-bar-baz` повернув реальну відповідь (`content: ok, error: None`) замість 404.
* Bad, because поле `model` у відповіді ехоїть запитаний id, а не фактичний — трейс/CRC-штамп фіксує запитану модель, не ту, що реально відповіла; це задокументований trade-off.

## More Information
`~/.omlx/settings.json` → `model.model_fallback: true`. Сервер перезапущено через `kill <PID> && omlx start` (GUI-app `oMLX` тримав порт; CLI `omlx restart` не керує GUI-процесом). Верифіковано curl-запитом до неіснуючої моделі `foo-bar-baz`.
