#### `taze` — **деструктивний, незахищений**
- Запускає `taze`, оновлює `package.json` у кожному знайденому воркспейсі

- `bun install` після апдейту

#### `lint` — **умовно небезпечний**
- Сам по собі лінт read-only, але кроки `eslint --fix`, `biome check --write` пишуть файли
- `bun run lint` шукає `package.json` вгору по дереву — знайде корінь, але `eslint --fix` оперує у межах знайденого `rootDir`, який може виявитися поточним підкаталогом

#### `docgen` — **деструктивний, незахищений**
- Пише `docs/` поряд із кожним кодовим файлом через обхід `cwd()`
- З піддиректорії обходить лише її — `docs/` опиняється не там

#### `start-check` — **не деструктивний**
- Read-only: просто запускає `bun run start` у кожному воркспейсі та фіксує stdout
- Пишуться лише stdout-звіти в пам'яті агента

#### `llm-patch` — **не деструктивний**
- Явно read-only (`без жодних змін у поточному репо`)

#### `publish-telegram` — **не деструктивний**
- Читає context, надсилає в Telegram — нічого не пише у файли

#### `worktree` — **низький ризик**
- Створює `.worktrees/<branch>/` з кореня (хук `n-cursor worktree start` сам визначає git-корінь), але неточне iкористання cwd може дати неправильну назву гілки

## ADR Заборона запуску default-sync поза коренем git-репо

## Context and Problem Statement
CLI `@nitra/cursor` (файл `npm/bin/n-cursor.js`) у default-гілці (`case undefined/''`) виконує `runSync()`, яка через `cwd()` скаффолдить `.cursor/rules/`, `.cursor/skills/`, `.claude/`, `CLAUDE.md`, `AGENTS.md`, `.n-cursor.json`, `.gitignore` і запускає `bun install`. При прямому виклику бінаря з піддиректорії git-репо (`bun npm/bin/n-cursor.js`) або з будь-якого іншого місця, де `cwd()` ≠ корінь репо, всі артефакти записуються не туди.

## Considered Options
* Guard через `git rev-parse --show-toplevel`: зупинити процес (`process.exit(1)`) якщо `cwd() ≠ git toplevel`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Guard через `git rev-parse --show-toplevel`", because це найточніший спосіб визначити корінь будь-якого проєкту-споживача без прив'язки до назви або структури конкретного монорепо; поза git-репо guard пропускає, щоб не ламати легітимні кейси ініціалізації нового проєкту.

### Consequences
* Good, because transcript фіксує очікувану користь: прямий виклик із `npm/bin/` завершується з `exit=1` і підказкою `cd <корінь>` — до жодної мутації файлів не доходить.
* Bad, because transcript не містить підтверджених негативних наслідків.
