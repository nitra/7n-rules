## ADR Flat концерн-лейаут для JS-правил у `npm/rules/`

## Context and Problem Statement

У монорепо `@nitra/cursor` кожне правило мало вкладену структуру `npm/rules/<rule>/js/<concern>/check.mjs`. Ця вкладеність ускладнювала discovery, поглиблювала шляхи імпортів і не відповідала фактичній конвенції, що вже застосовувалася в окремих правилах. Постало питання: як стандартизувати layout JS-концернів по всьому пакету.

## Considered Options

* Flat layout: `js/<concern>.mjs` як єдиний файл концерну
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Flat layout: `js/<concern>.mjs`", because це усуває зайвий підкаталог для кожного концерну, спрощує `listJsConcerns` (достатньо `isFile()` без додаткового `CHECK_FILENAME_RE`), і узгоджує структуру з уже наявною конвенцією `abie/utils/`.

### Consequences

* Good, because `listJsConcerns` у `npm/scripts/utils/discover-checkable-rules.mjs` більше не потребує фільтру `CHECK_FILENAME_RE` — файли-концерни і підкаталоги розрізняються через `entry.isFile()`.
* Good, because шляхи імпортів у тестах і concern-файлах стали коротшими на один рівень (`../../../scripts/` замість `../../../../scripts/`).
* Good, because transcript фіксує очікувану користь: всі 946 тестів після міграції показали ті самі 3 fail і 1 error, що були до неї (pre-existing), тобто нових регресів не з'явилося.
* Bad, because transcript не містить підтверджених негативних наслідків.

## ADR Розташування helpers: `utils/` peer до `js/` замість `_lib/` всередині `js/`

## Context and Problem Statement

Під час планування flat-концерн-лейауту виникло питання, куди переносити helper-модулі (файли, що раніше лежали поряд з `check.mjs` у `js/<concern>/`): у `js/_lib/<concern>/` всередині `js/` чи у `utils/` як окремий peer-каталог на рівні `js/`.

## Considered Options

* `js/_lib/<concern>/<helper>.mjs` — всередині `js/`, з `_` prefixed каталогом для виключення з discovery
* `<rule>/utils/<helper>.mjs` — peer до `js/`, плоско, без namespace-підкаталогів

## Decision Outcome

Chosen option: "`<rule>/utils/<helper>.mjs`", because правило `abie` вже мало `abie/utils/` за цим патерном, тому обрано наявну конвенцію замість введення нового `_lib/` механізму. Discovery (`listJsConcerns`) сканує лише `js/` і скіпає підкаталоги через `!entry.isFile()` — `utils/` взагалі не потрапляє у scope сканування.

### Consequences

* Good, because `listJsConcerns` не потребує перевірки `entry.name.startsWith('_')` — `utils/` просто знаходиться поза `js/`.
* Good, because для правила `abie` міграція є no-op (helpers вже у `utils/`), що підтверджено в transcript.
* Good, because transcript фіксує очікувану користь: 21 helper успішно переміщено у `<rule>/utils/` без конфліктів (collision guard у migration script).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Шлях helper з single-test: `from '../../utils/<helper>.mjs'`
- Шлях helper з multi-test: `from '../../../utils/<helper>.mjs'`
