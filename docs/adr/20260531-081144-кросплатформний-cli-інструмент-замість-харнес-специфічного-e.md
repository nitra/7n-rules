## ADR Набір підкоманд `add`/`remove`/`list`/`prune` (D3)

## Context and Problem Statement
Необхідно визначити мінімальний набір підкоманд нового CLI worktree-інструменту. Конвенція `n-worktrees.mdc` визначала три операції вручну; автоматизований інструмент мав покрити як мутуючі, так і прибиральні операції.

## Considered Options
* D1: `add` + `remove`
* D2: `add` + `remove` + `list`
* D3: `add` + `remove` + `list` + `prune`

## Decision Outcome
Chosen option: "D3", because `prune` вирішує проблему «осиротілих» записів між трьома станами (git-metadata, checkout на диску, `.md`-опис), що виникає з часом у реальному використанні.

### Consequences
* Good, because transcript фіксує очікувану користь: повний lifecycle управління worktree в одній команді.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`prune` реалізує семантику H-prune-b (агресивна): `git worktree prune` + видалення осиротілих `.md` без підтвердження. Worktree без `.md` — лише попередження у звіт, не видалення.

## ADR Санітизація слеша в дефіс для пласкої структури `.worktrees/`

## Context and Problem Statement
Git-гілки з слешем (`feat/skill-meta`) породжують проблему: `.worktrees/feat/skill-meta.md` потрапляє у підкаталог, і `cat .worktrees/*.md` (не рекурсивний glob) його не знаходить. Конвенція в `n-worktrees.mdc` використовує дефіс у прикладах (`feat-skill-meta`), що суперечить звичній `feat/skill-meta` формі гілок.

## Considered Options
* E-slash-a: санітизувати слеш у дефіс для шляху, зберігати git-гілку з слешем
* E-slash-b: зберігати slash-структуру в каталозі, шукати описи рекурсивно

## Decision Outcome
Chosen option: "E-slash-a", because `cat .worktrees/*.md` завжди працює (пласка структура), відповідає прикладам в `n-worktrees.mdc`, і git-гілка лишається з нормальним `feat/skill-meta`.

### Consequences
* Good, because transcript фіксує очікувану користь: glob `.worktrees/*.md` в `list`/`prune` завжди коректний без рекурсії.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовано у `sanitizeBranch()` в `npm/scripts/lib/worktree.mjs`. Команда друкує фактичний шлях після `add`, щоб агент не вгадував.

## ADR Єдина glob-форма для Type-A умов у `rules/*/meta.json`

## Context and Problem Statement
Поточна функція `detectAutoRules()` в `npm/scripts/auto-rules.mjs` використовує різнорідні предикати: `anyFile`, `rootFile`, `dir`, `glob` — кожен з окремою логікою в коді. Потрібно мігрувати 33 правила на data-driven `meta.json`; постало питання, яку форму декларації обрати для 13 правил, що детектуються за наявністю файлів/каталогів.

## Considered Options
* Окремі ключі `anyFile`, `rootFile`, `dir` у `meta.json`
* Єдина glob-форма: `{ "auto": { "glob": "<pattern>" } }` (рядок або масив)

## Decision Outcome
Chosen option: "Єдина glob-форма", because слеш у glob природно кодує семантику «корінь vs будь-де» (`package.json` — корінь, `**/Cargo.toml` — будь-де), прибирає окремі ключі і спрощує схему до одного декларативного виду для всіх 13 правил.

### Consequences
* Good, because transcript фіксує очікувану користь: єдиний код для матчингу, одна схема в `rule-meta.json`, відсутність спецкейсів.
* Bad, because `<dir>/**` не спрацьовує для порожнього каталогу (на відміну від поточного `existsSync(dir)`). Transcript визнає цю зміну семантики як прийнятну, оскільки `k8s/`, `.github/workflows/` та `npm/` на практиці завжди непорожні.

## More Information
Glob-мапінг 13 правил (Type A) зафіксовано в spec `docs/superpowers/specs/2026-05-31-rule-meta-json-design.md`. Реєстр предикатів для незводимих Type-D умов — `npm/scripts/lib/rule-predicates.mjs` (6 предикатів: `repoUrlMarker`, `gqlTaggedTemplate`, `hasuraConfigMarker`, `depInAnyPackageJson`, `jsBunDbSignal`, `nestedPackageWithoutVite`).
