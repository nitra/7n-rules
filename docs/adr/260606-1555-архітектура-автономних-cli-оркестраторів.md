## ADR Архітектура автономних CLI-оркестраторів

## Context and Problem Statement
CLI-команди `fix`, `lint`, `taze` в n-cursor є "чекерами": виводять порушення, але не виправляють їх. Виправлення лежить на агенті, який читає SKILL.md (8+ кроків), аналізує violation-output, пише код і повторно верифікує. Це робить скіли складними, а LLM витрачає токени на механічне оркеструванння замість реальної роботи.

## Considered Options
* CLI-команди як автономні оркестратори (повний цикл check → T0-auto → LLM → recheck)
* Залишити чекери + SKILL.md як оркестратор (поточна модель)
* MCP-сервер замість bash CLI

## Decision Outcome
Chosen option: "CLI-команди як автономні оркестратори", because агент повинен викликати одну команду і отримати exit 0/1, а не оркеструвати багатокроковий процес самостійно — це скорочує SKILL.md до одного рядка і усуває клас LLM-помилок через неправильне оркестрування.

### Consequences
* Good, because transcript фіксує очікувану користь: агент не парсить violation-output, не ризикує галюцинувати при виборі стратегії виправлення, SKILL.md спрощується до `npx @nitra/cursor fix`.
* Bad, because transcript не містить підтверджених негативних наслідків (складність реалізації — очікувана, не підтверджена проблема).

## More Information
- Зафіксовано в `docs/adr/260606-1553-autonomous-cli-orchestrators.md`
- Всі LLM-виклики через `pi` CLI (провайдер-агностичний, ключі налаштовуються per user)
- LLM-tier: script-extracts context → pi → script-applies (без tool-use у LLM)
- fix → lint: послідовно в тому самому worktree (fix будує структуру, lint залежить від неї)
- Паралельні під-ворктрі відхилено через конфлікти на спільних файлах (package.json, extensions.json)
- `fix --json` видаляється; команди завжди автономні
- Відповідні файли: `npm/skills/fix/js/t0.mjs`, `npm/skills/taze/js/diff.mjs`, `npm/bin/n-cursor.js`
