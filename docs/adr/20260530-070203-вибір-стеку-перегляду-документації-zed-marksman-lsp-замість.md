## ADR Вибір стеку перегляду документації: Zed + marksman LSP замість MkDocs Material

## Context and Problem Statement
Проєкт використовував `ci4.mdc`-правило зі стеком arc42 + Diátaxis + AUTOGEN + MkDocs Material як viewer. Під час обговорення альтернатив виникла єдина вимога: зручна file-by-file навігація документами **без запуску dev-сервера**, лише в редакторі Zed.

## Considered Options
* Zed built-in MD preview + marksman LSP (без site-generator-а)
* MkDocs Material з `mkdocs serve`
* VitePress з Vite HMR
* Docusaurus / Starlight (MDX)
* Antora (AsciiDoc)

## Decision Outcome
Chosen option: "Zed built-in MD preview + marksman LSP без site-generator-а", because єдиною вимогою є зручна навігація безпосередньо у файлах без запуску додаткових процесів; Zed нативно рендерить CommonMark + GFM + Mermaid, marksman LSP дає `[[wiki-link]]` навігацію та `cmd+click` по `[link](file.md)`.

### Consequences
* Good, because transcript фіксує очікувану користь: file-by-file перегляд без `mkdocs serve`/`vite dev`; `[[wiki-link]]` автокомпліт і `cmd+click` у Zed через marksman.
* Bad, because framework-specific синтаксис (MkDocs `!!! note`, VitePress `:::tip`, MDX компоненти) не рендериться у Zed preview — в `docs/` дозволено лише portable subset: CommonMark + GFM + Mermaid + KaTeX + `<details>`.

## More Information
- Правило оновлено: `npm/rules/ci4/ci4.mdc` v2.1 → v3.2; мирор `.cursor/rules/n-ci4.mdc` пересинкнено.
- Секцію `## MkDocs Material: collapsible engineer-блоки` (pymdownx, `??? engineer`, `mkdocs.yml`) замінено на `## Viewer/editor: Zed + marksman LSP` і `## Collapsible-блоки для змішаної аудиторії` (HTML5 `<details>`).
- Конвенція аудиторних блоків: `<summary><strong>Engineer:</strong> …</summary>` — перше слово з фіксованого словника (`Engineer:` / `Ops:` / `Security:` / `Manager:`).

## ADR Автоматичне створення `.marksman.toml` через `npx @nitra/cursor fix ci4`

## Context and Problem Statement
Після переходу на marksman LSP як основний viewer `docs/` потрібно забезпечити що кожне репо з правилом `ci4` отримує canonical `.marksman.toml` без ручного copy-paste.

## Considered Options
* JS-концерн `marksman_config` за паттерном `test.stryker_config` (автокопіювання baseline через `ensureBaselineFile`)
* Rego-policy (тільки валідує, не створює файли)
* Документувати у `ci4.mdc` з ручним кроком

## Decision Outcome
Chosen option: "JS-концерн `marksman_config` за паттерном `test.stryker_config`", because policy-рівень лише фіксує порушення, а JS-концерн може копіювати baseline ідемпотентно; цей паттерн вже застосовано для `stryker.config.mjs` / `vitest.config.js` у правилі `test`.

### Consequences
* Good, because transcript фіксує очікувану користь: перший `npx @nitra/cursor fix ci4` → `.marksman.toml` створюється; повторний прогон → idempotent pass (ручні правки не перетираються).
* Bad, because transcript не містить підтверджених негативних наслідків.
