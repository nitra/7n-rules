## ADR Стиль документації docgen — поведінковий замість реалізаційного

## Context and Problem Statement
Згенерована скілом `docgen` документація (наприклад, `npm/rules/abie/lib/docs/enabled.md`) містила реалізаційні деталі — секції із переліком `node:fs`/`node:fs/promises`/`node:path`, точні назви внутрішніх символів (`isAbieRuleEnabled`), таблиці типів і «Rebuild Test»-переліки — які не пояснюють бізнес-задачу файлу і старіють разом із кодом.

## Considered Options
* Зберегти поточний «вичерпний» стиль із секціями залежностей, сигнатур і тестових кейсів.
* Перейти на лаконічний поведінковий стиль: секції Огляд / Поведінка / Публічний API (лише за нетривіальної поверхні) / Де використовується / Гарантії поведінки — без stdlib, без внутрішніх імен хелперів, без таблиць типів.

## Decision Outcome
Chosen option: "Лаконічний поведінковий стиль", because користувач явно відхилив деталі рівня `node:fs` і точних назв як «тех деталі, які не впливають на бізнес задачу», та схвалив запропонований зразок, де залишено лише алгоритм рішення, fail-safe-гарантії та opt-in-семантику.

### Consequences
* Good, because документація описує поведінку та інваріанти, які не прив'язані до конкретних символів і не вимагають оновлення при внутрішньому рефакторингу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Промпт Tier 1 змінено у `npm/skills/docgen/SKILL.md` та `.cursor/skills/n-docgen/SKILL.md`. Слово «вичерпну» замінено на «лаконічну поведінкову» в описах і меті всіх трьох копій (включно зі стабом `.pi/skills/n-docgen/SKILL.md`). Еталонний перезаписаний файл: `npm/rules/abie/lib/docs/enabled.md`. Правила нового стилю: не перелічувати stdlib, не називати внутрішні хелпери, секцію `## Публічний API` включати лише за нетривіальної зовнішньої поверхні, `## Гарантії поведінки` замість `## Помилки`.

## ADR Використання локальної Ollama для Tier 1 docgen

## Context and Problem Statement
Tier 1 скіла `docgen` (генерація `.md` для ~1042 файлів кандидатів) коштує приблизно 26 M хмарних токенів (~$260–520). Оскільки задача механічна (один файл → одна лаконічна `.md`, незалежно), її можна перемістити на локальну Ollama, щоб звести хмарні витрати Tier 1 до нуля.

## Considered Options
* Залишити Tier 1 на хмарних Claude-субагентах (5 паралельних батчів).
* Перенести Tier 1 на локальну Ollama (`--engine ollama`, `concurrency=1`), Tier 2/3 лишити на Claude.
* Гібрид: Ollama для bulk Tier 1 + Claude для хвоста (файли, що перевищують `--max-bytes` або провалили quality-gate) + Claude для Tier 2/3.

## Decision Outcome
Chosen option: "Гібрид Ollama bulk + Claude tail + Claude Tier 2/3", because transcript фіксує таку рекомендацію як прагматичну: вона «обнуляє найбільшу статтю (~26 M хмарних токенів Tier 1)», лишаючи Claude лише ~2.6 M на Tier 2/3 та складний хвіст. Повноцінна реалізація в сесії не виконана — тільки зафіксована як стратегія.

### Consequences
* Good, because transcript фіксує очікувану користь: Tier 1 коштує $0 при прийнятній (80%) якості на локальній моделі; Tier 2/3 синтез лишається на Claude, де якість критична.
* Bad, because на 8 GB M2 прогін Tier 1 стає строго послідовним (concurrency=1, без паралелізму субагентів); весь Tier 1 займає ~22–38 годин на 1042 файли. Крім того, 4B-моделі не тримають тонкі правила стилю (наприклад, пропуск `## Публічний API` для leaf-модулів) і потребують quality-gate.

## More Information
Замірані значення (gemma3:4b, 8 GB M2, чистий запуск): RAM 4.2 GB, 100% GPU (без свопу), інгест 33.7 tok/s, генерація ~14 tok/s (в окремому замірі), ~76 с/файл. Модель `gemma4:e4b` (9.6 GB на 8 GB) — 0.4 tok/s через постійний своп — визнана непридатною. Кандидат для bulk: `gemma3:4b` (~3.3 GB q4). `qwen3:4b` — виміряний 22.4 tok/s з `/no_think`, але відповідь не була українською. Архітектурна передумова реалізації: винести Tier 1 промпт у спільний модуль `docgen-prompt.mjs` та додати CLI `docgen gen --engine ollama|claude`.
