# Розмежування ролей `/n-fix` і `/n-lint` та дедуплікація кроку лінтингу

**Status:** Accepted
**Date:** 2026-05-15

## Контекст

Скіл `/n-fix` у своєму кроці 6 перебирав скрипти `lint-js`, `lint-text`, `lint-style` окремо — дублюючи логіку, яку вже реалізує `/n-lint`. Це створювало ризик паралельних ESLint-процесів (заборонено в `CLAUDE.md`) і розмивало відповідальність: не було однозначної відповіді, чия задача — канонічний прогон `bun run lint`.

## Рішення/Процедура/Факт

Закріплено чіткий розподіл ролей: `/n-fix` = приведення структури проєкту у відповідність до `.cursor/rules/` (перевіряється через `npx @nitra/cursor check`); `/n-lint` = чистота коду (єдиний запуск `bun run lint`). Крок 6 у `npm/skills/fix/SKILL.md` і `.cursor/skills/n-fix/SKILL.md` переписано — перебір `lint-*` замінено делегуванням до `/n-lint` (`bun run lint`) з явним правилом «лінт-логіку не дублюй». Cross-reference «повний набір `lint-*` (навичка `n-fix`)» у `npm/rules/style-lint/style-lint.mdc` рядок 12 переписано на «`bun run lint` (навичка `/n-lint`)». Додано запис до `npm/CHANGELOG.md`, секція Changed у `1.11.6`.

## Обґрунтування

Злиття команд відкинуто через різні сценарії використання (рідкий структурний онбординг у `/n-fix` проти частого pre-commit у `/n-lint`), надлишковий overhead та ризик порушення інваріанти «один ESLint-процес на сесію». `bun run lint` — канонічний контракт CI; ховати його всередину `/n-fix` розмиває межу «перевірити як CI» проти «привести до правил». Дедуплікація усуває конкретне місце дублювання і зберігає чіткі межі відповідальності.

## Розглянуті альтернативи

Повне злиття в одну команду `/n-fix` — відхилено через різні частоти виклику, різний час виконання і ризик порушення інваріанти «один ESLint на сесію». Залишення status quo з дублюванням — відхилено як джерело плутанини та потенційних паралельних ESLint-процесів.

## Зачіпає
