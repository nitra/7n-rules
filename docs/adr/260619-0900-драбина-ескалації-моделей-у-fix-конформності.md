## ADR Драбина ескалації моделей у fix-конформності

## Context and Problem Statement
Рушій конформності (`orchestrator.mjs`) виправляв порушення лише з двома тирами (`min`→`avg`) після 2 сумарних фейлів у глобальному циклі до 3 ітерацій — без feedback між спробами, без кореляції «чи допомогла ця модель», і без окремого логу для подальшого аналізу ефективності ескалацій.

## Considered Options
* Детермінована драбина рунгів: `local-min` → `local-min` (повтор з feedback) → `cloud-min` → `cloud-avg`, із per-рунговим escalation-лог файлом
* Retry того самого тиру на systemic-помилках (memory-guard / auth / down)
* Окремий LLM-виклик «поясни, чому не спрацювало» на кожному рунгу

## Decision Outcome
Chosen option: "Детермінована драбина рунгів із fail-fast на systemic-помилках і дешевим feedback у prompt", because transcript фіксує: retry того самого `local-min` на systemic-помилках марний (узгоджено з `fail-fast-over-waiting`); окремий «аналіз-виклик» на кожному рунгу подвоює вартість при слабкому local-min; дешевий варіант — класти попередні зміни + залишковий violation у prompt наступного рунга, а окремий аналіз — лише при ескалації в хмару.

### Consequences
* Good, because transcript фіксує очікувану користь: кожен рунг записується в escalation-JSONL (`.n-cursor/fix-escalation.jsonl`) із полями `ruleId`, `tier`, `model`, `recheckOk`, `callError`, `remainingViolation`, `diagnosis`, `ms` — дає матеріал для аналізу, яка модель і на якому рунгу реально закриває порушення.
* Good, because transcript фіксує очікувану користь: `MODEL_HEAVY` (грубе поле) видалено з `llm-worker.mjs` — тепер драбина будується в `buildLadder` з `LOCAL_MIN`, `CLOUD_MIN`, `CLOUD_AVG`, перевизначуваних через env.
* Bad, because transcript не містить підтверджених негативних наслідків; потенційно: N правил × 4 рунги можуть спалити cloud-avg, але глобальний `maxAvg` (дефолт 2) обмежує кількість avg-викликів на весь прогін.
