## ADR CLI скілів у пакеті `@nitra/cursor` замість окремого пакета

## Context and Problem Statement
Команда хотіла запускати скіли (`lint`, `taze`, `publish-telegram` тощо) у зовнішніх репозиторіях без синхронізації `.cursor/skills` вручну. У transcript користувач надав готову архітектуру окремого пакета `@nitra/skills` з власним `bin/skills.js`.

## Considered Options
* Окремий npm-пакет `@nitra/skills` з власним bin і каталогом `skills/`
* Додати підкоманду `skill` до існуючого `@nitra/cursor` із читанням `npm/skills/<id>/SKILL.md`

## Decision Outcome
Chosen option: "Додати підкоманду `skill` до існуючого `@nitra/cursor`", because скіли вже фізично розміщені в `npm/skills/` цього пакета, а новий пакет вимагав би їх дублювання або залежності між пакетами.

### Consequences
* Good, because transcript фіксує очікувану користь: один `npx @nitra/cursor skill list` дає доступ до всіх скілів без додаткових пакетів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `npm/scripts/skills-cli.mjs` — логіка CLI
- `npm/bin/n-cursor.js` — підключення через `case 'skill'`
- Скіли читаються з `npm/skills/<id>/SKILL.md` встановленого пакета (або кешу `npx`)

## ADR Спрощення UX підкоманди `skill` до чотирьох форматів

## Context and Problem Statement
Перша реалізація мала підкоманди `prompt`, `claude`, `cursor`, `list`, окремий «claude-first» режим (`n-cursor claude <id>`), а також два допоміжних bin: `n-skills` і `n-claude`. Користувач запросив спростити до чотирьох конкретних форматів.

## Considered Options
* Повний набір: `skill prompt <id>`, `skill claude <id>`, `skill cursor <id>`, `skill list`, bin `n-skills`, bin `n-claude`, підкоманда `claude` у `n-cursor`
* Чотири форми: `skill list` / `skill <id>` / `skill cursor <id> ["task"]` / `skill claude <id> ["task"]`

## Decision Outcome
Chosen option: "Чотири форми", because користувач явно визначив саме ці формати як цільовий UX та попросив видалити зайве.

### Consequences
* Good, because transcript фіксує очікувану користь: видалено `skill prompt`, `n-skills`, `n-claude`, підкоманду `claude` у `n-cursor`; залишається один `n-cursor` bin.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Видалені файли: `npm/bin/n-skills.js`, `npm/bin/n-claude.js`
- `npm/package.json` `"bin"` повернуто до `{ "n-cursor": "./bin/n-cursor.js" }`
- Версія пакета: `1.13.72`
- `npm/CHANGELOG.md` оновлено під версію `1.13.72`
- Команди: `npx @nitra/cursor skill list`, `npx @nitra/cursor skill taze`, `npx @nitra/cursor skill cursor taze "task"`, `npx @nitra/cursor skill claude taze`

## ADR Склад промпта скілу: інструкція + контекст CWD

## Context and Problem Statement
Агент, якому передається промпт скілу, має розуміти поточний проєкт (структура, конфігурація), а не лише інструкцію скілу. Без контексту CWD агент не знає, з яким саме репо він працює.

## Considered Options
* Промпт містить лише `SKILL.md` і текст завдання
* Промпт містить `SKILL.md` + завдання + файли CWD (`package.json`, `tsconfig.json`, `.n-cursor.json`)

## Decision Outcome
Chosen option: "Промпт містить `SKILL.md` + завдання + файли CWD", because файли `package.json`, `tsconfig.json`, `.n-cursor.json` дають агентові мінімально достатній контекст для виконання скілу у конкретному проєкті.

### Consequences
* Good, because transcript фіксує очікувану користь: агент отримує директорію запуску (`Directory: <cwd>`) та ключові конфіги без ручного копіювання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Реалізовано у функції `buildSkillPrompt` у `npm/scripts/skills-cli.mjs`
- Секції промпта: `# Task`, `# Skill`, `# Current project` (з `package.json`, `tsconfig.json`, `.n-cursor.json` якщо існують)
- Файли читаються з `process.cwd()` на момент виклику `npx`
