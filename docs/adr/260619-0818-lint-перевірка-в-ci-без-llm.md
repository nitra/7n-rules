**Workflow:**

| Середовище | Команда | Що робить |
|---|---|---|
| **Dev (локально)** | `npx @nitra/cursor lint` | fix-режим: мутує файли, LLM opt-in де дозволено |
| **CI / deploy** | `npx @nitra/cursor lint --full --read-only` | detect-only: жодних мутацій, жодного LLM, fail якщо є порушення |

**Логіка:** розробник виправляє локально (з можливим LLM), пушить — CI перевіряє що результат вже відповідає всім правилам. Якщо CI фейлиться — значить розробник не запустив `lint` локально.

**Чому `--full`в CI:** без нього конформність-фаза (фаза ❹) не виконується — не перевіряються структурні правила (конфіги, залежності тощо). У dev достатньо дельти, у CI хочемо повну картину.

**Нюанс:** якщо правило має `llmFix:true` у `meta.json`, але код не виправлено до пушу → CI впаде з non-zero exit. Це навмисно: LLM-fix — це зручність для dev, а не обхід CI.

## ADR Lint-перевірка в CI без LLM

## Context and Problem Statement
Деякі правила `n-cursor lint` використовують LLM для автоматичного виправлення (`meta.json → llmFix:true`). У CI/deploy-середовищі немає доступу до LLM, але потрібна гарантія, що код відповідає всім правилам перед розгортанням.

## Considered Options
* `lint --full --read-only` в CI (detect-only, без LLM) + `lint` локально у розробника (fix-mode, LLM opt-in)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`lint --full --read-only` в CI, `lint` локально", because `--read-only` явно вимикає мутації та LLM (`orchestrate.mjs:6-7`: "detect-only, без мутацій, без LLM — CI-mode"), а `--full` активує конформність-фазу яка без прапора пропускається.

### Consequences
* Good, because CI не викликає LLM і гарантовано завершується у детермінований спосіб незалежно від середовища.
* Good, because transcript фіксує очікувану користь: розробник виправляє локально, CI лише верифікує результат — стандартна lint-locally/check-in-CI схема.
* Bad, because transcript не містить підтверджених негативних наслідків.
