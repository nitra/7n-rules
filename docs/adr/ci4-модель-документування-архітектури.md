# CI4-модель: документування архітектури `@nitra/cursor`

**Status:** Accepted
**Date:** 2026-05-10

## Контекст

Монорепозиторій `@nitra/cursor` (npm-пакет, CLI, GitHub Action, Claude-хуки, набір `.mdc`-правил) не мав формального опису архітектури: незрозуміло, які компоненти існують, хто зовнішні актори і як вони взаємодіють. Також планувалося двостороннє посилання між архітектурою та ADR-рішеннями.

## Рішення/Процедура/Факт

Прийнято рішення описати проєкт у форматі C4-моделі (конвенція проєкту: **CI4**) зі структурою файлів:

```
docs/ci4/
├── README.md         # індекс + інструкція кросс-лінкування з ADR
├── 01-context.md     # L1: зовнішні актори і межа системи
├── 02-containers.md  # L2: CLI (subcommands: check, stop-hook, rename-yaml-extensions), Rule Sync, AGENTS Builder, GH Action
├── 03-components.md  # L3: внутрішні модулі кожного контейнера з якорями
├── 04-code.md        # L4: drill-in для Check Runner і AGENTS Builder
└── decisions.md      # таблиця "елемент CI4 ↔ суміжні ADR"
```

Формат діаграм — Mermaid у Markdown (нативний рендер GitHub/Cursor/VS Code). Межа системи — тільки пакет `@nitra/cursor` (`npm/`); `demo/`, CI, npm registry, цільові репозиторії та AI-агенти (Cursor, Claude Code) є зовнішніми. Описуються всі 4 рівні C4 (L4 — для Check Runner і AGENTS Builder). Стабільні якорі у файлах CI4 (`## Check Runner {#check-runner}`) дозволяють ADR-файлам посилатися на конкретні елементи архітектури.

## Обґрунтування

Окремі файли замість одного `ci4-model.md` забезпечують стабільні URL (`docs/ci4/02-containers.md#check-runner`), які не ламаються при редагуванні сусідніх секцій. `decisions.md` як центральна cross-reference таблиця простіша у підтримці, ніж розкидані inline-посилання. Назва `ci4` — власна конвенція проєкту. Mermaid обрано бо проєкт text-first (markdownlint, cspell) і рендеринг відбувається без зовнішніх інструментів.

## Розглянуті альтернативи

- Один файл `docs/ci4-model.md` — відхилено: URL-якорі нестабільні при переписуванні розділів.
- Structurizr DSL — відхилено: додаткова залежність Docker/CLI, надмірно для одного пакету.
- PlantUML — відхилено: не рендериться нативно на GitHub без CI-екшну.
- Mermaid + Structurizr паралельно — відхилено як зайве.

## Зачіпає

`docs/ci4/` (новий каталог), `docs/adr/_inbox/*.md` (зворотні посилання на CI4), конвенція іменування архітектурних документів у монорепо `@nitra/cursor`.

## Update 2026-05-10

**Реалізація `docs/ci4/`**: Каталог створено з шістьма файлами: `README.md` (конвенція якорів і ADR-лінкування), `01-context.md` (L1 — 7 категорій зовнішніх акторів), `02-containers.md` (L2 — 6 контейнерів: `cnt-rule-sync`, `cnt-check-runner`, `cnt-stop-hook`, `cnt-capture-decisions`, `cnt-gh-action`, `cnt-pkg-artifact`), `03-components.md` (L3 — 4 Mermaid-діаграми і таблиці компонентів із колонкою `Tests`), `04-code.md` (L4 — 5 code-flow діаграм для всіх runtime-контейнерів), `decisions.md` (централізований cross-ref CI4 ↔ ADR).

**Конвенція стабільних якорів**: Контейнери мають префікс `cnt-*`, компоненти — `cmp-*`, L4 code-flows — `code-*`. Для стабільності посилань при реорганізації тексту використовуються HTML-теги `<a id="...">` замість заголовків.

**Суміжні зміни**: `.markdownlint-cli2.jsonc` оновлено (`MD033.allowed_elements: [a]` для дозволу `<a>`-якорів); `.cspell.json` поповнено CI4-термінами. `decisions.md` як центральна таблиця CI4 ↔ ADR доповнює inline-секції `## Related decisions` у кожному файлі рівня — централізована таблиця дає огляд, inline-секції — локальний контекст без переходу між файлами.

## Update 2026-05-10

### Канонічне розташування C4-діаграм у `docs/ci4/`

До `npm/mdc/ci4.mdc` додано секцію `## Розташування`, яка фіксує `docs/ci4/` як єдину канонічну теку для всіх рівнів C4-моделі. Очікувані файли: `01-context.md`, `02-containers.md`, `03-components.md`, `04-code.md`, `README.md`, `decisions.md`. Розміщення C4-схем в інших теках репозиторію заборонено.

Обґрунтування: розпорошення схем по підтеках сервісів порушує навігацію «від системи вглиб» і підвищує вартість агентського аналізу. Єдина тека зберігає ієрархію рівнів у одному місці відповідно до концепції C4 як джерела істини поряд з кодом. Формалізовано фактичний стан, який вже існував — рішення полягало лише у фіксації його в правилі.
