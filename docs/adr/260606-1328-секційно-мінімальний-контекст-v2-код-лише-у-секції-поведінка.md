## ADR Секційно-мінімальний контекст (v2): код лише у секції `Поведінка`

## Context and Problem Statement
У v1 оркестрованого docgen-конвеєра повний вихідний код передавався у системний контекст кожного секційного LLM-виклику (`Огляд`, `Поведінка`, `Публічний API`, `Гарантії`). Це призводило до того, що оркестрація була у 2–5× повільніша за one-shot (overlay-paths 310s vs 57s), бо кожна секція повторно інгестила весь код; KV-cache між stateless-запитами `/api/chat` не перевикористовувався надійно на 8 GB M2.

## Considered Options
* Секційно-мінімальний контекст: код — лише у `Поведінку`, решта секцій отримують тільки факт-лист (`factsSummary`)
* Спільний повний системний контекст (код + факти) для всіх секцій (v1, поточний на момент рішення)
* Multi-turn сесія: один `system`-промпт із кодом, подальші секції — окремі `user`-turns в рамках однієї розмови

## Decision Outcome
Chosen option: "Секційно-мінімальний контекст", because бенч підтвердив: overlay 310s→77s (×4), k8s 141s→55s (швидше за one-shot); якість лишилася ~86% (+6 п.п. над g3 one-shot). Повний код потрібен лише `Поведінці`; `Огляд` вимагає лише заголовок файлу, `API` — список експортів, `Гарантії` — маркери (всі вже у факт-листі).

### Consequences
* Good, because час оркестрації став конкурентним з one-shot і не залежить від надійності KV-cache між stateless-запитами.
* Bad, because `Поведінка` — єдина секція з кодом — почала «echo-їти» сигнатури функцій у прозі (регрес, що усувається Stage-2 лінтом).

## More Information
Реалізовано у `npm/skills/docgen/js/docgen-prompts.mjs` (функція `sectionMessages`): `behavior`-об'єкт містить `src` у системному промпті, всі інші секції — лише `factsTxt`. Бенч: `bench_orch.mjs` на 3 файлах (`firebase_hosting.mjs`, `overlay-paths.mjs`, `k8s-tree.mjs`). Коміт `45a7466c`.

## ADR Детермінований Stage-2 лінт: `stripSignatures` (0 токенів)

## Context and Problem Statement
Після переходу на секційно-мінімальний контекст секція `Поведінка` (єдина, що бачить код) почала витікати сигнатури функцій у прозу: `isUaKustomizationPath(rel): …`, `overlay-paths(…)`. Спроба виправити це через промпт-інструкцію «лише назва без дужок/аргументів» призвела до регресу — модель прибирала і описи разом з аргументами, залишаючи «голі назви».

## Considered Options
* Детермінований post-processing `stripSignatures(text)`: regex `([`\w$.]+)\([^()]*\)` → `$1`, два проходи (0 LLM-токенів)
* Промпт-інструкція «НЕ пиши аргументи функцій у дужках» (без детермінованого зрізу)
* Окремий LLM-виклик для фінального clean-up

## Decision Outcome
Chosen option: "Детермінований post-processing `stripSignatures`", because regex-зріз усуває сигнатури механічно без додаткових токенів і без ризику вбити описи, що підтверджено бенчем: overlay `Поведінка` стала `isUaKustomizationPath: Перевіряє…` (замість `(rel)`), внутрішні імена не течуть.

### Consequences
* Good, because transcript фіксує очікувану користь: витік сигнатур усунуто без додаткового часу.
* Bad, because агресивний regex може помилково зрізати технічні вирази з дужками у прозі (наприклад, math-notation). Transcript цього не фіксує, але це теоретична крайність.

## More Information
Реалізовано у `npm/skills/docgen/js/docgen-gen.mjs` (функція `stripSignatures`), застосовується як до `generateOrchestrated`, так і до `generateOneShot`. Коміт `45a7466c`.

## ADR Негативні маркери кешу у `factsSummary` для запобігання галюцинацій

## Context and Problem Statement
Модель `gemma3:4b` у секції `Гарантії` систематично вигадувала рядок про кеш навіть для файлів, де кешування відсутнє (`firebase_hosting.mjs`, `overlay-paths.mjs`). Промпт-інструкція «Згадуй кеш лише якщо у фактах прямо є `Кешування: так`» без явного негативного факту не стримувала модель.

## Considered Options
* Явний негативний маркер у факт-листі: `'Кешування: НЕМАЄ — не згадуй кеш у гарантіях'`
* Лише позитивні маркери (якщо кешу нема — просто не додавати рядок про кеш)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Явний негативний маркер у факт-листі", because з позитивними-лише маркерами модель pattern-match-ить «гарантії → кеш» і вигадує його; явний «НЕМАЄ» як частина вхідних даних (а не лише промпту) зупиняє цей патерн. Аналогічно додано `'Робота з мережею: немає'`.

### Consequences
* Good, because transcript фіксує очікувану користь: галюцинація кешу у `firebase_hosting/Гарантії` зникла після додавання негативного маркера.
* Bad, because Neutral, because transcript не містить підтвердження наслідку — теоретично модель може ігнорувати негативний маркер на складніших файлах (наприклад, `fix.mjs` показав 6/12 з підозрілим рядком про кеш у фінальному тесті на `npm/rules/ga`).

## More Information
Реалізовано у `npm/skills/docgen/js/docgen-prompts.mjs` (функція `factsSummary`): `m.caches ? 'Кешування: так, у межах прогону' : 'Кешування: НЕМАЄ — не згадуй кеш у гарантіях'`. Коміт `45a7466c`.

## ADR `gemma4:4b` — лише one-shot режим; `gemma3:4b` orchestrated — рекомендація для Tier 1 docgen

## Context and Problem Statement
Після валідації orchestrated-конвеєра на `gemma3:4b` була спроба перенести його на `gemma4:4b` (`batiai/gemma4-e4b:q4`, 5.3 GB, 56%/44% CPU/GPU split на 8 GB M2). Orchestrated-режим для `gemma4:4b` повертав порожні рядки (`""`) для секцій без коду в контексті (підтверджено двома варіантами: split `system/user` і merged single-user message).

## Considered Options
* `gemma3:4b` orchestrated (3.3 GB, 100% GPU, ~14-20 tok/s)
* `gemma4:4b` one-shot (5.3 GB, ~11 tok/s, 162s avg/файл)
* `gemma4:4b` orchestrated (відхилено — архітектурна проблема Gemma 3n E4B)

## Decision Outcome
Chosen option: "`gemma3:4b` orchestrated як рекомендація для Tier 1 docgen", because фінальний бенч (3 файли × 3 режими) показав: g3 ORCH 92% / 52s avg, g4 ONE 75% / 162s avg, g3 ONE 47% / 53s avg. `gemma3:4b` orchestrated виграє по обох осях — якість вища, час нижчий, апаратний відбиток менший (3.3 vs 5.3 GB). `gemma4:4b` придатна лише для one-shot через архітектурне обмеження (Gemma 3n E4B повертає `""` без коду в контексті).

### Consequences
* Good, because transcript фіксує очікувану користь: 92% якості, 3× швидше за g4, залишається в GPU без свопу.
* Bad, because `gemma3:4b` one-shot на складних файлах (k8s-tree) дає лише 17% — значить, без orchestrated-конвеєра g3 непридатна; при проблемах з конвеєром fallback на one-shot неприйнятний.
