## ADR Автономні CLI-оркестратори як єдина точка входу

## Context and Problem Statement
Команди `fix`, `lint`, `taze` були "чекерами": виводили список порушень і очікували що агент-LLM або людина вручну пройдуть SKILL.md (8 кроків, аналіз JSON, вибір стратегії виправлення, convergence-loop). Це вимагало від агента повної оркестрації і призводило до нестабільних результатів залежно від того, як агент інтерпретував кроки.

## Considered Options
* Агент читає SKILL.md і сам оркеструє (поточний стан до цієї сесії)
* CLI-команда сама є оркестратором: check → T0-auto → LLM-tier → check-gate → loop

## Decision Outcome
Chosen option: "CLI-команда сама є оркестратором", because агент не повинен знати як виправляти — лише коли викликати. SKILL.md зводиться до одного рядка `npx @nitra/cursor fix`; логіка convergence-loop, ескалації, check-gate живе в `npm/skills/fix/js/orchestrator.mjs`.

### Consequences
* Good, because агент отримує простий контракт: exit 0 = чисто, exit 1 = не вдалось автоматично.
* Bad, because LLM-tier всередині CLI потребує налаштованого `pi` з cloud-моделлю; без нього — тільки T0 і T0-auto тири.

## More Information
Реалізовано в `npm/skills/fix/js/orchestrator.mjs` (convergence-loop), `npm/skills/fix/js/llm-worker.mjs` (LLM-tier через `pi`), `npm/skills/fix/js/t0.mjs` (T0-auto). Команда: `npx @nitra/cursor fix [rules] [--max-iter N]`.

## ADR `meta.json` прапор `orchestrator: true` як декларація автономії

## Context and Problem Statement
Потрібен стандартний спосіб декларувати що скіл є автономним оркестратором (вся логіка всередині CLI) на відміну від скіла-інструкції (агент читає SKILL.md і виконує кроки вручну). Без цього прапора немає машиночитаємого сигналу для агентів і CI.

## Considered Options
* Окремий файл `orchestrator.json` поряд з `meta.json`
* Прапор `"orchestrator": true` у існуючому `meta.json`
* Конвенція по імені файлу (`orchestrator.mjs` присутній = автономний)

## Decision Outcome
Chosen option: "прапор `\"orchestrator\": true` у `meta.json`", because `meta.json` вже є точкою конфігурації скіла (`worktree`, `auto`); додавання прапора до того самого файлу мінімізує кількість місць де описано скіл.

### Consequences
* Good, because агент може перевірити `meta.json` перед запуском і знати: якщо `orchestrator: true` — не треба читати SKILL.md для логіки, достатньо однієї команди.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/skills/fix/meta.json`: `{ "auto": "завжди", "worktree": true, "orchestrator": true }`. Наступний крок: додати той самий прапор у `taze` і `lint` після стабілізації `fix`.

## ADR `fix --json` видалено з публічного API → `_fix-check` як внутрішня команда

## Context and Problem Statement
`n-cursor fix --json` існував як публічний structured-check інтерфейс для CI, T0-auto і агентів. Після переходу до автономного оркестратора цей режим перестав бути потрібним зовнішнім споживачам, але залишився потрібним внутрішньо — orchestrator і t0.mjs викликали його як subprocess.

## Considered Options
* Залишити `fix --json` як публічний флаг паралельно з orchestrator
* Видалити `--json` з публічного API; внутрішній check перейменувати на `_fix-check` (underscore = private)

## Decision Outcome
Chosen option: "видалити `--json` з публічного API, `_fix-check` як приватна команда", because публічний `--json` суперечить контракту "fix = автономний оркестратор"; underscore-конвенція сигналізує що команда не для зовнішнього використання і може змінитись без попередження.

### Consequences
* Good, because transcript фіксує очікувану користь: єдина точка входу `fix` без прихованих режимів.
* Bad, because `fix-t0` і будь-який зовнішній код що використовував `fix --json` потребує міграції на `_fix-check`.

## More Information
`n-cursor.js`: case `'fix'` тепер завжди викликає `runOrchestratorCli`. case `'_fix-check'` → `runFixCommand` з `{ json: true }` (той самий код, але непублічний). `t0.mjs` оновлено: `['_fix-check', ...ruleFilter]` замість `['fix', '--json', ...ruleFilter]`. `fix-run` залишився як deprecated alias на `fix`.

## ADR Послідовне виконання fix → lint vs. паралельні під-ворктрі

## Context and Problem Statement
Агент в своєму worktree може потребувати запустити і `fix`, і `lint`. Питання: чи виконувати їх послідовно в тому самому worktree, чи спавнити два дочірніх worktree на базі поточного і мержити результати назад.

## Considered Options
* Послідовне виконання: `fix && lint` в одному worktree
* Паралельні під-ворктрі: `fix-worktree` ‖ `lint-worktree` → merge назад в agent-worktree

## Decision Outcome
Chosen option: "послідовне виконання в одному worktree", because `fix` і `lint` не є незалежними: `fix` може створити або змінити ESLint-конфіги, `extensions.json`, `package.json` — файли які `lint` потім перевіряє. Паралельний запуск гарантовано породжує конфлікти на цих спільних файлах; merge-логіка додає складності без реального виграшу в часі.

### Consequences
* Good, because відсутні merge-конфлікти; `lint` бачить стан після `fix`; один worktree замість трьох.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Додаткової інформації в transcript не зафіксовано.

## ADR LLM-tier через `pi` (C1 pattern): script збирає контекст, LLM повертає заміну

## Context and Problem Statement
`llm-worker.mjs` спочатку використовував Anthropic SDK напряму з `ANTHROPIC_API_KEY` і tool-use (read_file/write_file/run_command через SDK). Це прив'язувало оркестратор до одного провайдера і не дозволяло користувачам налаштовувати власні ключі.

## Considered Options
* Anthropic SDK напряму (`ANTHROPIC_API_KEY` hardcoded провайдер)
* `pi` CLI як абстракція провайдера (C1 pattern: script-extracts context → pi returns replacement → script applies)
* `pi` з `--tools` (file-write через tool-use в pi)

## Decision Outcome
Chosen option: "`pi` CLI, C1 pattern без tool-use", because кожен користувач налаштовує провайдера і API ключі в `pi` самостійно (`~/.pi/agent/models.json`, env vars, `/login`); оркестратор не залежить від конкретного провайдера. C1 (script збирає контекст, LLM повертає заміну файлу цілком) простіший за tool-use через зовнішній процес і надійніший (pi підтримує обмежений набір tools).

### Consequences
* Good, because transcript фіксує очікувану користь: кожен користувач обирає модель (claude-haiku-4-5, gpt-5.4-mini, gemma3:4b) через `N_CURSOR_FIX_MODEL_HAIKU` / `N_CURSOR_FIX_MODEL_SONNET`.
* Bad, because 4B локальні моделі (gemma3:4b, gemma4:4b) не дотримуються JSON-формату відповіді і часто видають timeout; LLM-tier фактично вимагає cloud-модель в `pi`.

## More Information
`npm/skills/fix/js/llm-worker.mjs`: `spawnSync('pi', ['-p', prompt, '--model', MODEL_HAIKU, '--no-session', '--mode', 'text', '--no-tools'])`. Env vars: `N_CURSOR_FIX_MODEL_HAIKU` (default: `claude-haiku-4-5-20251001`), `N_CURSOR_FIX_MODEL_SONNET` (default: `claude-sonnet-4-6`). Ескалація: haiku × 2 fails → sonnet. Тест: gemma3:4b повертає пояснювальний текст замість JSON (підтверджено); ETIMEDOUT при gemma4:4b > 120s (підтверджено).
