## ADR Інтерактивний генератор change-файлів `ch` у `@7n/7`

## Context and Problem Statement

У проєкті `/Users/vitaliytv/www/vitaliytv/7n` (`@7n/7`) не було зручного способу стандартно генерувати `.changes/*.md`-файли (формат `n-cursor release`): CLI-команди `--bump`/`--section`/`--message` вимагали памʼятати флаги, а агенти/розробники часто пропускали цей крок або заповнювали неповно. Потрібна команда, сумісна з `n-cursor`-флоу релізу і доступна через `npx @7n/7 ch`.

## Considered Options

* Інтерактивний генератор як частина `@7n/7` CLI (гібрид: флаги → без промптів, відсутнє → питається)
* Взяти готовий `@changesets/cli` (`changeset add`)
* Окремий standalone-пакет поза `@7n/7`

## Decision Outcome

Chosen option: "Інтерактивний генератор як частина `@7n/7` CLI", because `@changesets/cli` генерує несумісний формат (`bump:` — маппінг пакет→бамп, немає `section:`; каталог `.changeset/` замість `.changes/`), і споживач (`n-cursor release`) не зміг би читати ці файли без суттєвого перетворення. Окремий standalone-пакет дублював би формат і споживача. Варіант 1 дає `npx @7n/7 ch` як єдину точку входу, що перевикористовує однаковий формат `.changes/` (той самий споживач).

### Consequences

* Good, because формат change-файлів залишається єдиним (`---\nbump:\nsection:\n---\nопис`), сумісним із `n-cursor release`; жодного дублювання коду або споживача.
* Good, because transcript фіксує очікувану користь: гібридний режим (флаги → без TTY; відсутнє → промпт) дозволяє використовувати команду і в CI (повні флаги), і інтерактивно.
* Bad, because `npx @7n/7 ch` (інтерактив) запрацює ззовні лише після публікації пакета; локально команда валідована через `node npm/bin/n-7.js ch …` і vitest-тестами.

## More Information

- `npm/ch.js` — модуль команди; чисті функції `serializeChange`, `parseChArgs`, `collectChange`, `runCh`; DI-ін'єкція `prompt/now/rand/writeFile/cwd`.
- `npm/tests/ch.test.mjs` — 13 vitest-тестів (формат, парсинг флагів, повний/частковий інтерактив, повтори, non-TTY guard, запис файлу, делегування через `run`).
- `npm/package.json` — додано bin `"7": "./bin/n-7.js"` поряд із `n-7`; резолвить `npx @7n/7 ch`.
- Формат change-файлу перевірено на сумісність із `@changesets/cli` у transcript: три блокери (каталог, схема frontmatter, відсутність `section`) підтверджують несумісність.
- Change-файл для самої зміни: `npm/.changes/1780426461517-b64370.md` (bump: minor, section: Added), створено через `node npm/bin/n-7.js ch --bump minor --section Added …` (dogfood).

## ADR Fail-fast preflight для `worktree: true` скілів (заміна м'якого банера)

## Context and Problem Statement

Скіли з `worktree: true` у `meta.json` (`n-fix`, `n-coverage-fix`, `n-fix-tests`, `n-taze`, `n-adr-normalize`) вшивали в `SKILL.md` лише прозовий банер-пораду (`> **Worktree:** виконуй цей скіл…`). Агент сам повідомив, що обробляв це як «рекомендацію»: чисте робоче дерево служило спокусою пропустити preflight-перевірку CWD. Це була системна сліпа пляма, а не одноразова помилка.

## Considered Options

* Жорсткий fail-fast preflight у `SKILL.md` через `worktree-notice.mjs` (централізовано)
* PreToolUse-хук у `.claude/settings.json`
* Правило в `CLAUDE.md`

## Decision Outcome

Chosen option: "Жорсткий fail-fast preflight у `SKILL.md` через `worktree-notice.mjs`", because PreToolUse-хук бачить session CWD (корінь проєкту), а не bash-CWD агента після `cd .worktrees/…` — тому хук завжди рапортував би «не у worktree» й блокував навіть коректний запуск. Текстове правило в CLAUDE.md — другий шар (також додано), але недостатнє саме по собі.

### Consequences

* Good, because transcript фіксує очікувану користь: один файл (`worktree-notice.mjs`) — усі `worktree: true` скіли отримують гейт на ре-синку; банер тепер містить `[!IMPORTANT]`, явний `STOP`/`ABORT` і runnable-перевірку `git rev-parse --show-toplevel | grep -q '/\.worktrees/'`.

* Bad, because transcript не містить підтверджених негативних наслідків.
