## ADR Конвенція розширень нових JS-файлів: `.mjs`/`.cjs` замість `.js`

## Context and Problem Statement
Голий `.js` файл інтерпретується Node.js/Bun як ESM або CJS залежно від поля `"type"` у `package.json`. Це створює неоднозначність: той самий файл може читатися по-різному в різних пакетах. Постала потреба у конвенції, що зробить тип модуля явним безпосередньо з розширення.

## Considered Options
* Статична інструкція в `.mdc` (без автоматичної перевірки нових vs існуючих файлів) — варіант (a)
* Git-aware перевірка: фейл на `.js` у `git status untracked/added` — варіант (b)
* Warn на всі `.js` без фейлу — варіант (c)

## Decision Outcome
Chosen option: "Варіант (a) — статична інструкція в `js-lint.mdc`", because stateless-скан не відрізняє новий файл від існуючого (немає git-history у чекерах), тому programmatic-перевірка ненадійна; натомість явна секція-конвенція в `js-lint.mdc` дає чітку інструкцію агенту без хибних позитивів.

### Consequences
* Good, because нові файли однозначно ідентифікують тип модуля без звертання до `package.json#type`, що узгоджується з наявною вимогою `"type": "module"` у `js-lint`.
* Bad, because transcript не містить підтверджених негативних наслідків. Існуючі `.js` не зачіпаються автоматично — потребують ручного поступового переходу.

## More Information
Конвенція додана в `npm/rules/js-lint/js-lint.mdc` (v1.29 → 1.30), секція «Розширення нових файлів — `.mjs`/`.cjs`, не `.js`». Стосується backend і frontend. Команда для перевірки: `npx @nitra/cursor fix changelog`.

## ADR Вибір `js-lint.mdc` як єдиного дому конвенції розширень

## Context and Problem Statement
Нова конвенція стосується backend і frontend, але правило `js-run.mdc` явно обмежене backend-only («Не застосовується до frontend-пакетів»). Треба було визначити, де розмістити правило, щоб воно покривало обидва типи проєктів.

## Considered Options
* Розмістити в `js-lint.mdc` (крос-платформне, globs `**/*.{js,mjs,cjs,jsx,ts,tsx}`)
* Розмістити в `js-run.mdc` (тільки backend)
* Створити окреме правило

## Decision Outcome
Chosen option: "`js-lint.mdc`", because він крос-файловий, його `globs` вже охоплюють усі JS-файли backend і frontend, а `description` — загальний «Перевірка JavaScript коду»; `js-run` явно backend-only і не підходить.

### Consequences
* Good, because transcript фіксує очікувану користь: одна точка правила для backend і frontend без дублювання.
* Bad, because `js-lint` традиційно описував перевірювані тулзи (oxlint/eslint/jscpd/knip), а нова секція — неперевірювана конвенція. Секцію явно позначено маркером відсутності автоперевірки, щоб не створювати хибного очікування.

## More Information
Файл: `npm/rules/js-lint/js-lint.mdc`. Версія бампнута 1.29 → 1.30 згідно з `n-changelog`-правилом. Change-файл: `npm/.changes/260611-1201.md` (minor / Added).

## ADR Оновлення прикладів нового вихідного коду на `.mjs`: «Кошик 2», конфіги тулінгу не чіпати

## Context and Problem Statement
Після введення конвенції `.mjs` існуючі приклади нового вихідного коду у правилах (`src/conn/pg.js`, `main.js`, `date-utils.js`, `src/conn/db.js` тощо) суперечили б новій конвенції. Водночас деякі `.js`-згадки у правилах — це імена конфігів, продиктовані тулінгом або прив'язані до власних чекерів.

## Considered Options
* (A) Тільки секція в `js-lint`, приклади не чіпати
* (B) Секція в `js-lint` + переписати приклади нового вихідного коду («Кошик 2»), конфіги тулінгу лишити
* (C) Ширший рефактор — оновити й конфіги тулінгу та фікстури чекерів

## Decision Outcome
Chosen option: "Варіант (B)", because масовий рефактор Кошика конфігів ризикував зламати узгодженість doc↔check (чекери хардкодили `vitest.config.js`, `eslint.config.js`); Кошик 2 — безпечний (виключно нові source-файли без прив'язки до checker-regex).

### Consequences
* Good, because приклади нового вихідного коду в `js-run.mdc`, `js-bun-db.mdc`, `vue.mdc` тепер показують `.mjs` і не суперечать конвенції.
* Bad, because фікстури чекера `check-fixture.test.mjs` залишились би на `.js` (стилістична розбіжність doc↔test). Окремо вирішено привести й їх до `.mjs` після підтвердження, що `SOURCE_FILE_RE = /\.([cm]?[jt]sx?)$/u` вже підтримує `.mjs`.

## More Information
Змінені файли (Кошик 2): `npm/rules/js-run/js-run.mdc` (v1.11→1.12), `npm/rules/js-bun-db/js-bun-db.mdc` (v1.14→1.15), `npm/rules/vue/vue.mdc` (v2.1→2.2). Свідомо НЕ чіпали: `vite.config.js`, `vitest.config.js`, `eslint.config.js` (на момент рішення).

## ADR Перехід `vitest.config.js` → `vitest.config.mjs` у чекерах і baseline

## Context and Problem Statement
Чекери `test/js/vitest-config-pool-forks.mjs` та `test/js/stryker_config.mjs` хардкодили ім'я `vitest.config.js`, а Stryker baseline-файли містили `vitest: { configFile: 'vitest.config.js' }`. Vitest як інструмент підтримує `.mjs`; це був технічний борг, що суперечив новій конвенції.

## Considered Options
* Лишити `vitest.config.js` як виняток із конвенції (чекери не чіпати)
* Виправити проєкт: оновити чекери, baseline та тести, щоб канон став `vitest.config.mjs`; зберегти backward-compat через підтримку обох розширень у runtime

## Decision Outcome
Chosen option: "Виправити проєкт", because інструмент підтримує `.mjs`, а збереження `.js`-хардкоду без потреби суперечило б конвенції, яку щойно ввели.

### Consequences
* Good, because `vitest.config.mjs` тепер є каноном у всьому проєкті; чекер приймає обидва розширення для backward-compat з існуючими `.js`-конфігами.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
