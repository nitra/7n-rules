**Status:** Accepted
**Date:** 2026-06-06

## Context and Problem Statement

Існували два незалежні шари: `n-cursor flow` як «Пасивний Турнікет» та архітектура автономного ОАГ у `docs/думка.MD`. Обидва використовували git worktrees, але мали несумісні протоколи стану: `.flow.json`, `docs/specs/`, `docs/plans/` проти `tasks/<node>/task.md` і `outputs_NNN.md`. Потрібно було обʼєднати їх так, щоб `graph` керував DAG і worktree lifecycle ззовні, а `flow` обслуговував один вузол зсередини.

## Considered Options

* Зберегти `flow` і `graph` незалежними системами.
* Повністю злити обидві системи в єдину.
* Переписати `flow` під контракт `думка.MD`: `flow` як протокол всередині вузла, `graph` як зовнішній оркестратор DAG.
* Два окремі кроки Stage 1: `flow spec` для design і `flow plan` для decomposition.
* Один спільний крок `flow plan`, який поглинає `flow spec` і виконує design та decomposition.
* Атрибут `mode` у front-matter `task.md` (`human` або `agent`).
* Прапорець у команді `graph run --actor` або `--mode`.
* Variant A для composite-шляху: `flow plan` сам викликає `graph spawn`.
* Variant B для composite-шляху: `flow plan` створює дочірні `task.md`, а агент явно викликає `graph spawn`.
* Тільки post-merge hook для запуску готових вузлів.
* Тільки постійний `n-cursor watch` демон.
* Обидва механізми: post-merge hook і `n-cursor watch`.

## Decision Outcome

Chosen option: "Переписати `flow` під контракт `думка.MD`: `flow` як протокол всередині вузла, `graph` як зовнішній оркестратор DAG", because `flow` і `graph` перекривались по відповідальності за worktrees і стан задачі, а поділ `graph` зовні / `flow` всередині одного вузла усуває дублювання та задає чітку межу відповідальності.

Також прийнято: Stage 1 виконується як один спільний `flow plan`; `mode: human|agent` зберігається у front-matter `task.md` з default `human`; для composite-шляху використовується Variant B — агент явно викликає `graph spawn`; автоматичний запуск готових вузлів забезпечують і post-merge hook, і `n-cursor watch` демон.

### Consequences

* Good, because один `task.md` / `outputs_NNN.md` стає джерелом істини для стану вузла замість `.flow.json`, `docs/specs/` і `docs/plans/`.
* Good, because Stage 1 дає явний gate: вузол або стає атомарним і отримує `plan_001.md`, або розкладається на дочірні `task.md`.
* Good, because `mode: human` за замовчуванням зберігає безпечний інтерактивний режим, а `mode: agent` дозволяє автономний запуск на рівні конкретного вузла.
* Good, because явний `graph spawn` дає агенту можливість перевірити дочірні задачі перед реєстрацією у графі.
* Good, because поєднання post-merge hook і `n-cursor watch` покриває і git-події, і файлові зміни чи stale worktrees.
* Bad, because видаляються чинні команди `flow init`, `flow spec`, `flow release` і `.flow.json`-стан, що є breaking change для існуючих flows.
* Bad, because два оркестраційні тригери потребують координації, щоб уникнути race condition при одночасному запуску.
* Neutral, because transcript не містить підтвердження, як саме буде виконана міграція існуючих `.flow.json`-станів.

## More Information

Файли й команди, які зникають або переписуються: `flow init`, `flow spec`, `flow release`, `.flow.json`, `docs/specs/<date>.md`, `docs/plans/<date>.md`, `lib/spec.mjs`, `lib/plan.mjs`, `lib/plan-panel.mjs`, `lib/flow-lock.mjs`, `lib/state-store.mjs`, `lib/snapshot.mjs`, `lib/artifact.mjs`, `commands.mjs`.

Нова файлова структура вузла: `tasks/<node>/task.md`, `tasks/<node>/plan_001.md`, `tasks/<node>/run_001.md`, `tasks/<node>/outputs_001.md`, sentinel `invalidated`. Для composite-шляху Stage 1 створює дочірні `tasks/<node>/<child>/task.md`, після чого агент викликає `n-cursor graph spawn <path>`.

Stage 2: агент виконує атомарну роботу, пише `outputs_NNN.md`, викликає `flow verify`, який перевіряє `## Done when` з `task.md`, і потім сигналізує `graph done`, `graph audit` або `graph failed`.

Оркестрація черги: `.git/hooks/post-merge` викликає `n-cursor graph run --auto`; `n-cursor watch` слідкує за `tasks/`, чергою та stale worktrees. Конфіг згаданий у transcript: `.n-cursor.json` з `max_worktrees` і `stale_worktree_min`. Документ архітектури: `docs/думка.MD`, розділ «Інтеграція з `n-cursor flow`».
