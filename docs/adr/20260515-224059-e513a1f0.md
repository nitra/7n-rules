## ADR Розмежування ролей `/n-fix` та `/n-lint` і структурна реорганізація `rules/<id>/js/`

**Контекст:** Два скіли (`/n-fix` і `/n-lint`) частково дублювали один одного: `/n-fix` на кроці 6 запускав окремі `lint-*` скрипти, тоді як `/n-lint` є канонічним оркестратором `bun run lint`. Паралельно, всі JS-артефакти правил (і fix-концерни, і lint-entry) лежали в одному каталозі `rules/<id>/js/`, що ускладнювало розуміння їхньої ролі.

**Рішення/Процедура/Факт:**
1. `/n-fix` отримав секцію «Скоуп» і позбувся кроку 6 (виклик `bun run lint`). Його єдина задача — приводити структуру проєкту у відповідність до `.cursor/rules/` через `npx @nitra/cursor check`. Лінт-порушення у коді — поза скоупом.
2. `/n-lint` лишається єдиним каноном для `bun run lint`.
3. Усі 26 правил реструктуровано: `rules/<id>/js/<concern>/` → `rules/<id>/fix/<concern>/` (JS-діагностика для `n-cursor check`), `rules/<id>/js/lint.mjs` / `run.mjs` → `rules/<id>/lint/lint.mjs` (CLI-entry для `bun run lint-X`). `policy/` лишилась незмінною.
4. Інфраструктура (`discover-checkable-rules.mjs`, `run-rule.mjs`) отримала dual-mode — підтримує обидва формати під час міграції, з fatal-помилкою при дублікаті концерну у `js/` і `fix/`.
5. Оновлено 5 static imports у `n-cursor.js`, 6 крос-модульних imports у `check.mjs`/`discover.test.mjs`, 3 hardcoded imports у `tests/`.

**Обґрунтування:** Чіткіше розмежування ролей (`fix/` = діагностика структури, `lint/` = лінт коду) усуває дублювання і ризик паралельних запусків ESLint, яке CLAUDE.md забороняє. `policy/` лишається окремо, бо `.rego`-концерни мають інший runner (`runConftestBatch` vs dynamic `import`) і namespace-конвенцію. Поетапна міграція (Фаза 1 — dual-mode, Фаза 2 — переміщення, Фаза 3 — прибирання legacy) дозволяла тримати тест-сюїт зеленим на кожному кроці.

**Розглянуті альтернативи:**
- Злиття `/n-fix` і `/n-lint` в одну команду — відхилено через різну частоту запуску і ризик запуску важкого `bun run lint` там, де потрібна лише структурна перевірка.
- Класти `policy/` під `fix/policy/` — відхилено на користь збереження `policy/` на рівні правила (rego-ecosystem ізольований, runner відрізняється від JS).
- Big-bang міграція в один PR — відхилено на користь поетапної (Фаза 1 → 2 → 3) для безпечного ревʼю.
