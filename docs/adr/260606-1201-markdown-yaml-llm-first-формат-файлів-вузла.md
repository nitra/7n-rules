## ADR Markdown + YAML-фронтматер як LLM-first формат файлів вузла

**Status:** Accepted
**Date:** 2026-06-06

## Context and Problem Statement

Файли вузлів у Рекурсивному складеному ОАГ читаються переважно агентами-LLM, а не парсерами. Потрібен формат де LLM може природно читати контекст і продовжувати без реконструкції структури з машинного формату.

## Considered Options

* JSON — структурований, але вимагає реконструкції контексту для LLM
* YAML — компактний, але незручний для багаторядкового тексту
* Markdown + YAML-фронтматер — машинозчитуваний фронтматер + вільний Markdown для LLM

## Decision Outcome

Chosen option: "Markdown + YAML-фронтматер", because LLM може читати `repair_history.md` як єдиний документ і продовжувати природно без парсингу, а фронтматер дає структуровані поля для оркестратора.

### Consequences

* Good, because `repair_history.md` як append-only журнал — інженер читає і продовжує природно, без реконструкції JSON-масиву.
* Good, because той самий формат використовується в `.mdc` файлах правил у цьому проєкті — узгодженість інструментарію.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Структура кожного файлу: YAML-фронтматер (машинозчитувані поля) + Markdown-тіло (вільний текст для LLM).

`meta.md` — місія та граф-метадані:
```markdown
---
id: вузол_7
стан: очікує
ребра:
  від: [вузол_5.вихід_a]
  до: [вузол_8.вхід_x]
---
## Місія
Перетвори технічне завдання на покроковий план.

## Оригінальна місія
(Пишеться один раз при spawn, ніколи не перезаписується — захист від drift намірів після патчів інженера)

## Критерій завершення
Структурований план з оціненими кроками.
```

`repair_history.md` — append-only журнал відновлення:
```markdown
# Журнал відновлення

## Спроба 1 — <timestamp>
**Міркування:** ...
**Патч:** ...
**Результат:** помилка — Y все ще виникає

## Спроба 2 — <timestamp>
**Міркування:** спроба 1 не вийшла, тепер...
```
