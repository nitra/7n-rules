## ADR Класифікація скілів за сумісністю з git worktree

## Context and Problem Statement
У проєкті 10 скілів (`n-fix`, `n-lint`, `n-taze`, `n-coverage-fix`, `n-fix-tests`, `mdc-check`, `n-adr-normalize`, `n-start-check`, `n-llm-patch`, `n-publish-telegram`), і постало питання: чи можна їх запускати у `git worktree` і який ефект це дає. Потрібно було визначити, які скіли виграють від ізоляції worktree, а які — не отримують жодних переваг або навіть зазнають шкоди.

## Considered Options
* Запускати всі скіли у worktree без класифікації
* Класифікувати скіли за ознаками «пише в репо» та «важкий/CPU-чутливий» і для кожної групи задати різні правила щодо worktree

## Decision Outcome
Chosen option: "Класифікувати скіли за ознаками «пише в репо» та «важкий/CPU-чутливий»", because це дозволяє чітко описати, де worktree дає ізоляцію результату (7 із 10 скілів), а де — зайвий overhead або конфлікт портів (`n-start-check`) чи взагалі нічого ізолювати (`n-llm-patch`, `n-publish-telegram`).

### Consequences
* Good, because transcript фіксує очікувану користь: для `n-taze` worktree автоматично задовольняє передумову «чисте дерево», а для `n-lint`/`n-coverage-fix`/`n-fix-tests` ізолює авторефакторинг та нові тести на окрему гілку.
* Bad, because правило заборони паралельного eslint/Stryker/`bun test` (з `CLAUDE.md` та `n-lint/SKILL.md`) діє на рівні машини, а не директорії — наявність кількох worktree НЕ обходить це обмеження і навпаки множить ризик перевантаження CPU.

## More Information
Класифікація зі стенограми:
- **Безпечно паралелити між worktree:** `n-fix`, `mdc-check` (легке навантаження; ⚠️ спільний version-bump у `npm/package.json` → потенційний merge-конфлікт).
- **Ізоляція так, але лише послідовно:** `n-lint`, `n-coverage-fix`, `n-fix-tests`, `n-taze`.
- **Ізоляція корисна, CPU-навантаження немає:** `n-adr-normalize`.
- **Worktree шкодить (конфлікт портів):** `n-start-check`.
- **Worktree безглуздий (read-only):** `n-llm-patch`, `n-publish-telegram`.

## ADR Крос-worktree gap у `withLock` і стратегія виправлення

## Context and Problem Statement
Механізм `withLock` (`npm/scripts/utils/with-lock.mjs`, ADR `20260522`, ADR `20260523`) серіалізує важкі CLI-команди та дедуплікує їх за git-fingerprint у межах одного checkout. Але кожен `git worktree` має власний `node_modules`, а отже власний `node_modules/.cache/n-cursor/`. Через це `withLock` не бачить локів в інших worktree — і паралельний запуск eslint/Stryker у двох worktree не серіалізується, що порушує запроваджений інваріант «один важкий прогін на машину».

## Considered Options
* Мітка в назві worktree (перевіряти `git worktree list | grep <tag>` перед запуском)
* Перенести стан локу в спільну для всіх worktree директорію (`git rev-parse --git-common-dir`)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Перенести стан локу в спільну для всіх worktree директорію (`git rev-parse --git-common-dir`)", because «мітка в назві» є ненадійною з кількох причин: stale-lock після краш/завершення worktree без cleanup, TOCTOU-гонка, вона покриває тільки скіли, запущені через worktree, і лишається LLM-інструкцією без автоматичного enforcement — саме ті аргументи, які ADR `20260522` вже навів проти схожих підходів.

### Consequences
* Good, because transcript фіксує очікувану користь: `withLock` вже приймає `opts.cacheDir` — достатньо ключувати цей параметр від `git common dir`, а не від `process.cwd()`, щоб `mkdirSync`-lock став крос-worktree мьютексом, зберігши PID-liveness та fingerprint-dedup.
* Bad, because transcript не містить підтверджених негативних наслідків; потенційна складність: у різних worktree різні git-fingerprint (`HEAD` + diff), тому dedup не зливатиме різні гілки помилково, але серіалізація додасть очікування для другого агента.

## More Information
Ключові файли та ADR:
- `npm/scripts/utils/with-lock.mjs` — поточна реалізація (`mkdirSync`-lock, PID-liveness, sha256-dedup, `opts.cacheDir`)
- `docs/adr/20260522-212907-guard-блокування-паралельних-команд.md`
- `docs/adr/20260523-225704-withlock-серіалізація-та-дедуплікація-cli-команд.md`
- Команда для отримання shared dir: `git rev-parse --git-common-dir`
- `node_modules/.cache/n-cursor/<key>/` — поточне розташування лок-файлів (per-worktree)
