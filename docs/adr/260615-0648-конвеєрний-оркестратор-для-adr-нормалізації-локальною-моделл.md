## ADR Конвеєрний оркестратор для ADR-нормалізації локальною моделлю

## Context and Problem Statement
Поточний `normalize-decisions.sh` використовує single-shot підхід: один великий промпт (усі 10–30 чернеток + 290+ clean-файлів) → одна велика JSON-відповідь з усіма операціями. Це добре працює на cloud-моделях (sonnet/opus), але локальна 4B-модель (`omlx/gemma-4-e4b-it`) не може опрацювати: 32k-токенний промпт викликає `omlx curl exit 18` (передача перервана) або зависання на 5+ хвилин.

## Considered Options
* Single-shot наївний перенос (той самий промпт, той самий JSON — але через `omlx`)
* Конвеєрний підхід: JS оркеструє, LLM відповідає лише на вузькі verifiable-питання (per-draft, малий JSON, детермінований retrieval)

## Decision Outcome
Chosen option: "Конвеєрний підхід з інверсією керування", because єдиний single-shot arm (`naive-local`) впав на інфраструктурі (`omlx curl exit 18`/hang), а конвеєрний arm (`new-local`) завершив ідентичний батч повністю локально (37 викликів, 0 хмарних, 0 infra-фейлів, 7/7 структурно валідних MADR).

### Consequences
* Good, because transcript фіксує очікувану користь: конвеєр виконується без хмарних викликів (`cloudCalls: 0`), не передає чернетки із повними шляхами `/Users/vitaliytv/…` та внутрішньою архітектурою зовнішнім сервісам (приватність), і досягає near-gold прозової якості (gemma на constrained-reformat зіставна із sonnet).
* Bad, because wall-clock час конвеєра (~17хв на 10 драфтах, 29 серійних викликів) не кращий за cloud-gold (~14хв), через серійну природу локального omlx-сервера.
