type: ADR
title: Ліквідація n-cursor flow та злиття з n-cursor graph
description: Namespace n-cursor flow ліквідовано, а planning, execution-сигнали й аудит уніфіковано під n-cursor graph.
---

**Status:** Accepted
**Date:** 2026-06-07

## Context and Problem Statement

`n-cursor flow` і `n-cursor graph` частково дублювали один одного. `flow` мав команди для spec, plan, verify, review, gate, run/resume/cancel/repair і спирався на `.flow.json`; `graph` описував DAG-оркестрацію через файловий стан вузлів. Після переходу на `tasks/<node>/task.md`, `plan_NNN.md`, `run_NNN.md`, `outputs_NNN.md` і `pending-audit_NNN.md` окремий namespace `flow` став надлишковим.

## Considered Options

- Залишити `flow` і `graph` як два окремі namespace з чіткою межею.
- Залишити `flow` як внутрішній протокол вузла, а `graph` як зовнішній DAG-оркестратор.
- Повністю ліквідувати `flow` і поглинути функціонал у `graph`.

## Decision Outcome

Chosen option: "Повністю ліквідувати `flow` і поглинути функціонал у `graph`", because після видалення `flow init`, `flow spec`, `flow verify`, `flow review`, `flow gate`, `flow run`, `flow resume`, `flow cancel`, `flow repair` залишався лише planning-крок, який природно стає `graph plan`.

### Consequences

- Good, because агент працює з одним namespace `n-cursor graph` для lifecycle вузла без перемикання між `flow` і `graph`.
- Good, because `.flow.json`, `docs/specs/` і `docs/plans/` зникають, а стан зберігається у файлах вузла.
- Good, because аудит винесено в асинхронну чергу через `pending-audit_NNN.md`, а не в синхронний `flow verify`/`flow review`.
- Bad, because transcript не містить підтверджених негативних наслідків.
- Neutral, because існуючі правила й документація з посиланнями на `flow` потребують міграції.

## More Information

Заміни команд:

| Команда | Рішення |
|---|---|
| `flow init` | замінено `graph run` |
| `flow spec` | поглинуто в `graph plan` |
| `flow plan` | перейменовано на `graph plan` |
| `flow verify` | видалено, замінено аудит-чергою |
| `flow review` | видалено, замінено аудит-чергою |
| `flow gate` | видалено як зайву обгортку |
| `flow run` | замінено `graph run` |
| `flow resume` | не потрібно, стан у файлах |
| `flow cancel` | замінено `graph kill` |
| `flow repair` | не потрібно, `.flow.json` зникло |

Stage 1: `graph plan` читає `task.md`, працює в `mode: human` або `mode: agent`, створює `plan_001.md` для atomic-шляху або дочірні `task.md` для composite-шляху. Для composite-шляху обрано explicit-варіант: агент окремо викликає `graph spawn`.

Stage 2: агент читає `plan_001.md`, виконує роботу, пише `outputs_NNN.md` і сигналізує `graph done`, `graph audit <path>` або `graph failed`.

Аудит: `graph audit` створює `pending-audit_NNN.md`, де `NNN` дорівнює номеру відповідного `outputs_NNN.md`. `n-cursor watch` або post-merge hook сканує чергу; auditor пише `run_NNN.md` з `actor: auditor` і `result: success|failed`.

Рішення зафіксовано у `docs/думка.MD` у секції про інтеграцію `n-cursor flow`.

## Update 2026-06-07

Для Stage 1 у `mode: human` обговорювався варіант тонкого helper-а: CLI робить preflight, читає `task.md`, форматує контекст, а IDE-агент планує і пише `plan_001.md`; `--finalize` лише валідує результат. Для `mode: agent` обговорювався subagent subprocess із timeout від `budget_sec`.

Transcript також фіксує, що self-contained інтерактивний CLI-діалог для planning відхилено як дублювання можливостей IDE-агента. Пізніше це рішення поглинуте фінальним переходом від `flow plan` до `graph plan`.

## Update 2026-06-07

### Уточнення фінального контракту `graph`

Пізніша сесія уточнила, що `n-cursor watch` є єдиним оркестратором: post-merge hook лише будить демон через файловий сигнал на кшталт `touch .n-cursor/wake`, а не запускає окремий `graph run --auto`. Це прибирає race condition між двома процесами, які могли б одночасно сканувати граф і запускати той самий вузол.

### `mode` та `interactive`

Для `task.md` зафіксовано два ортогональні атрибути:

- `mode: human|agent` — хто є директором задачі;
- `interactive: true|false` — чи дозволений інтерактивний діалог під час виконання.

`n-cursor watch` не запускає автоматично вузли з `mode: human, interactive: true`. Пріоритет значень: CLI-аргумент → `task.md` → `.n-cursor.json`.

### Аудиторський результат окремим файлом

Для обробки `pending-audit_NNN.md` обрано окремий immutable-файл `audit-result_NNN.md`, де `NNN` збігається з номером `pending-audit_NNN.md` і відповідного `outputs_NNN.md`. Це замінює timestamp-порівняння або `audit_ref` у `run_NNN.md`: факт обробки визначається наявністю `audit-result_NNN.md`.

Очікуваний frontmatter `audit-result_NNN.md`:

```yaml
---
created_at: 2026-06-07T11:00:00Z
actor: auditor
result: success # або failed
---
```

### Composite resolved через агрегацію дітей

Composite-вузол не пише власний `outputs_NNN.md`: його стан деривується з дочірніх вузлів знизу вверх. Якщо всі діти `resolved`, батько `resolved`; якщо є `running` або `pending-audit`, батько `running`; якщо є `failed` без активного running, батько `failed`; якщо є лише `waiting`, батько `waiting`.
