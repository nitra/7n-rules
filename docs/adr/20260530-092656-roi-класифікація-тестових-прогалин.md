# ROI-класифікація тестових прогалин замість 100%-покриттєвого таргету

**Status:** Accepted
**Date:** 2026-05-30

## Context and Problem Statement

При виявленні 147 JS-файлів з aggregate lines coverage 77.62% (functions 84.47%) постало питання: куди вкладати нові тести? Рівномірне прагнення до 100% покриває spawn-обгортки, 3-рядкові `fix.mjs`-стаби та CLI entrypoints — файли, де unit-тести не вловлюють реальних багів, бо поведінка або покрита інтеграційними тестами, або немає testable logic.

## Considered Options

* Максимальне покриття на всіх файлах, включно зі spawn-wrappers та CLI glue
* ROI-класифікація прогалин: A (0% concern-функції) → B (lib-helpers) → C (часткові прогалини) → D (свідомо не покривати)

## Decision Outcome

Chosen option: "ROI-класифікація A/B/C/D", because spawn-обгортки й CLI glue не додають mutation-signal при unit-моку (підтверджено прикладами `rules/docker/lint/lint.mjs` і `scripts/lib/run-lint-step.mjs`); 3-рядкові `fix.mjs`-стаби мають однаковий паттерн без логіки; concern-функції й pure lib-helpers мають тестабельну логіку з чіткими pass/fail-сценаріями.

### Consequences

* Good, because 10 нових тест-файлів (108+ кейсів, усі зелені) підняли lines 77.62% → 78.80%, fn 84.47% → 86.13%; категорія D виключена свідомо і задокументована.
* Bad, because `rules/docker/lint/lint.mjs` (32%) не зрушив — `runLintDockerSteps`/`runLintDocker` залежать від `process.cwd()` без DI-точки; реальна прогалина залишилася.
* Neutral, because підхід потребує ручної класифікації при кожному новому файлі; обговорено, але не реалізовано: `@test-policy` JSDoc-маркер, `coverage-policy.yaml`, LLM-класифікатор мутантів (`verdict: equivalent|defensive|glue|wrapper|worth-testing`).

## More Information

- **Категорія A** (0% → 82–100%): `rules/abie/js/tests/` — firebase_hosting, env_dns, hc_pairing, ua_node_selector, ua_http_route.
- **Категорія B** (11–12% → 100%): `rules/abie/lib/tests/hc-yaml.test.mjs`, `rules/abie/lib/tests/k8s-tree.test.mjs`, `rules/docker/lib/tests/docker-hadolint.test.mjs`.
- **Категорія C** (16–40% → 84–100%): `scripts/tests/coverage-fix.test.mjs`, `scripts/tests/upgrade-nitra-cursor-and-install.test.mjs`.
- **Категорія D** (не покривати): 31× `rules/*/fix.mjs`-стаби, `scripts/lib/run-rule-cli.mjs`, `scripts/lib/run-lint-step.mjs`, `rules/docker/lint/lint.mjs`.
- Покриття виміряно: `bunx vitest run --coverage --coverage.reporter=json-summary --coverage.reportsDirectory=/tmp/n-coverage`.
- Пов'язані файли: `npm/stryker.config.mjs`, `npm/vitest.config.js`, `npm/rules/test/coverage/coverage.mjs`.

## Update 2026-05-30

**Ручна валідація вцілілих мутантів (fallback при недоступному Stryker):**

Коли Stryker у режимі `--incremental` завис (invalid cache: `0 of 25053 mutant result(s) are reused`, конкуренція CPU зі застарілими bun-процесами), застосовано ручну мутацію source-файлу + `vitest run -t <pattern>`:

1. Внести мутацію вручну в source (наприклад `if (opts.fix) → true`).
2. Запустити: `bunx vitest run <test-file> -t "<test-pattern>"`.
3. Якщо тест падає — мутант killed; відновити оригінал.

Усі 4 вцілілих мутанти в `coverage.mjs` (L189, L211) підтверджено killed за <5 хв замість 50+ хв очікування Stryker. COVERAGE.md не оновлюється при ручній валідації — потрібен окремий `bun run coverage`.

- Команди: `bunx vitest run rules/test/coverage/tests/coverage.test.mjs -t "opts.fix gate"`.
- Стара Stryker-cache: `npm/reports/stryker/incremental.json` (mtime давніший за 27 год) — причина invalid cache.
