# doc-files: local-only конвеєр claims→render з AST-декомпозицією замість cloud-ескалації

**Status:** Accepted
**Date:** 2026-06-10

## Context and Problem Statement

Скіл `doc-files` генерує файлову документацію двома тирами: прості файли (`sym < 4`) — локальна модель, складні (`sym ≥ 4`, ~22% файлів) — pre-route у cloud. Нова вимога: документувати файли **будь-якої складності виключно локальною моделлю** (`omlx/mlx-community--gemma-4-e2b-it-4bit` на Mac M2 8GB), зберігши якість. Виміряні межі: той самий g3 дає 47% one-shot цілим файлом проти 92% при секційній оркестрації — деградація порогова за розміром вікна, а не плавна; det-скоринг сліпий до семантичної точності («score=100 оманливий»).

## Considered Options

* Декомпозиція: плоскі per-export зрізи (А1) / одразу bottom-up по call-graph (А2) / А1 → bench → А2 за порогом (А3)
* Незбіжність: degraded-маркер у frontmatter (Б1) / строгий фейл із exit 1 (Б2) / Б1 + ручний `--cloud` (Б3)
* Анти-галюцинація: claims→render всюди (В1) / лише де помилка каскадує (В2) / без claims (В3)
* Верифікація: det-verify + micro-refine (Г1) / Г1 + закритий LLM-верифікатор (Г2) / без micro-refine (Г3)
* Семплінг: render best-of-2 + адаптивний claims self-consistency (Д2) / лише render (Д1) / повний (Д3) / без (Д0)
* Few-shot: у всіх типах викликів (Е1) / лише claims-JSON (Е2) / без (Е0)
* Мови: vue+py одразу в юніт-шарі (Ж1) / інкрементом 2 (Ж2) / виключити (Ж3)
* Вимірювання: лише det-метрики (К1) / + ручний diff-рев'ю (К2) / + cloud-judge у bench (К3)

## Decision Outcome

Chosen option: **А3 + Б1 + В1 + Г1 + Д2 + Е1 + Ж1 + К1; cloud-код видаляється повністю**, because складність має поглинатись оркестрацією, а не моделлю: AST-юніти (`oxc-parser` — уже пряма залежність, домашній патерн `ast-scan-utils.mjs`) повертають кожен LLM-виклик під поріг вікна; двофазна генерація claims→render з детермінованим evidence-фільтром відсікає вигадки структурно (рендерер не бачить коду — лише схвалені твердження); det-verify (анкор-покриття, identifier-whitelist, дублі, рендер-лінт) з точковими micro-refine замінює слабкого відкритого LLM-критика; bottom-up мікро-резюме хелперів рекурсивно застосовує виміряний важіль «факти в промпті» (+15 п.п.) і гарантує збіжність на будь-якій складності. Маршрутизація транспорту — лише за префіксом `omlx/` (env-перемикач `N_CURSOR_DOCGEN_BACKEND` видаляється); дефолтна модель іде в omlx напряму (через pi CLI виміряно +46% часу). Незбіжність після всіх ярусів — `score`+`issues` у frontmatter при свіжому CRC: Stop-гейт не блокує задачі через слабкість моделі, борг видимий (`check --degraded`) і адресно перегенеровується (`gen --retry-degraded`).

Повний дизайн: `docs/superpowers/specs/2026-06-10-doc-files-local-only-pipeline.md`.

### Consequences

* Good, because зникає клас файлів, структурно недосяжний для локальної моделі: вікно кожного виклика обмежене незалежно від розміру файлу; мета «будь-яка складність локально» виконується конструкцією.
* Good, because claims-фаза дає новий детермінований семантичний сигнал (частка відкинутих тверджень, AST-суперечності) — закриває сліпу пляму det-скорингу без cloud-judge.
* Good, because кеш claims/резюме за hash юніта робить повторні прогони інкрементальними (30–60 с/файл) — критично для обов'язкового кроку кожної задачі — і стає структурованим входом для `doc-aggregate`.
* Good, because health-check omlx перед батчем відрізняє memory-guard зайнятої 8GB машини («відклади прогін») від справжніх помилок генерації.
* Bad, because складні файли коштують більше викликів і часу (12–18 викликів, 4–6 хв на файл рівня `safety.mjs`); перший повний прогін ~7–9 год проти ~4 год.
* Bad, because semantic-якість прямо не вимірюється (К1) — лише через det-проксі; помилки інтерпретації справжнього коду ловляться тільки адаптивним self-consistency за сигналом.
* Bad, because хибне мікро-резюме листка може каскадувати вгору по call-graph; мітигація — листки мають найменші вікна + evidence-фільтр, але ризик не нульовий.
