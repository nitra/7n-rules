# Правило `test` переведено у always-on — видалення dog-food умови

**Status:** Accepted
**Date:** 2026-05-28

## Context and Problem Statement

Консьюмер-монорепо `ai` використовує `n-cursor coverage`, що вимагає `vitest`, `@vitest/coverage-v8` і `@stryker-mutator/vitest-runner` у root `devDependencies`. `bun.mdc` (v1.9) описував дозвіл на ці пакети як виняток лише для dog food-репо `@nitra/cursor`, тоді як `package_json.rego` глобально дозволяв їх через `allowed_root_dev_dependency` без перевірки імені репо.

Цей розсинхрон давав суперечливі правила для консьюмерів і ламав запуск `bun run coverage` в `ai`. Глибша причина: правило `test` було opt-in (умова в `test/auto.md` — «якщо у проєкті є хоча б один файл `*.test.mjs`»), тому виняток у `bun.mdc` доводилося формулювати через приклад одного-єдиного always-on споживача — самого `@nitra/cursor`.

## Considered Options

* Звузити `package_json.rego` під mdc-текст (перевіряти ім'я репо, лишити dog-food виняток). Зламав би `ai` та всіх інших споживачів `n-cursor coverage`.
* Зробити правило `test` always-on (аналогічно `adr`, `security`, `text`) і прибрати дог-фуд умову з `bun.mdc`, спираючись на структурну причину (`npm-module.mdc` забороняє `devDependencies` у published workspace, тому Vitest/Stryker peer-tools мусять жити у root будь-якого монорепо).

## Decision Outcome

Chosen option: «Зробити `test` always-on і прибрати dog-food умову», because перший варіант зламав би всіх консьюмерів `n-cursor coverage`; структурна причина (root — єдине місце для peer-tools у монорепо з published workspace) однаково стосується кожного користувача правила, а не лише dog food-репо. Always-on `test` робить «корінь — єдине місце» правдою у конфізі за замовчуванням, без вимоги до споживача вручну вмикати правило.

### Consequences

* Good, because `bun run coverage` у будь-якому консьюмер-монорепо перестає падати з помилкою про відсутній `vitest`.
* Good, because `bun.mdc` і `package_json.rego` тепер узгоджені: обидва дозволяють root-only test peer-tools без умов на ім'я репо.
* Good, because споживачі без власних `*.test.mjs` усе одно отримують канонічний `vitest.config.js` + `stryker.config.mjs` через концерн `stryker_config` (правило `test`), і `n-cursor coverage` працює одразу після `npx @nitra/cursor`.
* Bad, because `test` тепер додається в `.n-cursor.json` навіть для репо без власних тестів — формально це extra-правило у whitelist, без негативних практичних наслідків у transcript.

## More Information

Змінені файли:

- `npm/rules/test/auto.md` — `завжди` (замість «якщо у проєкті є хоча б один файл `*.test.mjs`»).
- `npm/scripts/auto-rules.mjs` — `'test'` додано у `AUTO_RULE_ORDER` (алфавітно) і у безумовний блок `addRule('test')` поруч з `addRule('adr')`/`addRule('security')`/`addRule('text')`.
- `npm/scripts/tests/auto-rules.test.mjs` — `'test'` додано у `ALL_RULES` і у очікуваний результат «додає правила за ознаками».
- `npm/rules/bun/bun.mdc` (version `1.9 → 2.0`) і `.cursor/rules/n-bun.mdc` — знято dog-food виняток; причина переформульована на структурну (`npm-module.mdc` + always-on `test`).
- `npm/rules/bun/policy/package_json/package_json.rego` — оновлено docstring і коментар `allowed_root_dev_dependency`: посилання на `test/auto.md` (always-on) замість «dog food».
- `npm/package.json` — `1.28.3 → 1.28.4`.
- `npm/CHANGELOG.md` — запис `[1.28.4]`.

Тести після змін: rego 14/14 (`conftest verify -p npm/rules/bun/policy/package_json`), vitest 313/313 (`bun x vitest run npm/scripts`), bun-rules 10/10.
