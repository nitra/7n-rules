---

**Status:** Proposed
**Date:** 2026-07-19

## Context and Problem Statement

`resolvePluginList()` (`npm/scripts/lib/resolve-plugins.mjs:162-168`) сьогодні працює all-or-nothing: якщо `.n-rules.json` має явне поле `plugins: string[]` — воно повністю перекриває `detectPluginsFromRepo()`, незалежно від того, скільки категорій плагінів там перелічено. Порожній масив `[]` — це навмисний спосіб сказати «плагіни вимкнено».

Інцидент, що це виявив: репо `components` мало `.n-rules.json` з `"plugins": ["@7n/rules-ci-github"]` (записано до сьогоднішнього релізу). 2026-07-19 випуск `@7n/rules` 1.26.0-1.29.0 виніс усю JS/npm-родину правил (включно з `taze`-провайдером) у новий плагін `@7n/rules-lang-js`. `detectPluginsFromRepo()` коректно розпізнає кореневий `package.json` як сигнал для `@7n/rules-lang-js` (`KNOWN_LANG_PLUGINS.js`, рядок 39), але оскільки `plugins` у `components` вже був явно заданий (лише з `ci`-категорією), автодетект жодної мовної категорії не спрацював. Наслідок: `npx @7n/rules skill codex taze` мовчки не мав жодного провайдера (`⏭ Жодного taze-провайдера`), і `lint --full` падав на `js/eslint` через несинхронізований `node_modules` — обидва симптоми зрештою звелись до того, що новий плагін ніколи не потрапив у список `components`.

Проблема не одноразова: та сама пастка спрацює для будь-якого майбутнього нового плагіна/категорії (напр. гіпотетичний `@7n/rules-obs-*`) для будь-якого проєкту, що вже має непорожній `plugins`.

## Considered Options

- **Залишити як є** (all-or-nothing) — покладатись на те, що користувачі вручну дописують нові плагіни в `plugins` після мажорних релізів. Відхилено: нічим не сигналиться, що дію взагалі треба виконати; `docs/specs/2026-07-18-lang-plugins-extraction-spec.md` не описує міграційний шлях для проєктів з уже заповненим полем.
- **Автодетект завжди домішується, навіть при явному `plugins`** (декларація = allowlist, не override). Відхилено: ламає легітимний use case — свідома відмова від плагіна певної категорії (наприклад, `package.json` у корені є, але lang-js навмисно не потрібен).
- **Per-категорійне перекриття**: `plugins` і автодетект групуються за категорією (`ci`, `lang`, …, виведеною з префікса імені пакета `@7n/rules-<cat>-*`); категорія, відсутня в явному `plugins`, добирається автодетектом, категорія, присутня хоч одним пакетом — лишається зафіксованою користувачем як є.
- **Явний список категорій для вимкнення** (`pluginsAutoDetectExclude: ["lang"]` замість самого `plugins`). Відхилено як зайва складність: те саме досягається присутністю/відсутністю категорії в уже наявному полі `plugins`, без нового поля конфігу.

## Decision Outcome

Chosen option: **per-категорійне перекриття** автодетекту в `resolvePluginList()`.

1. Категорія плагіна виводиться з імені пакета за конвенцією `@7n/rules-<category>-<name>` (regex, не хардкод переліку — так само підхопить майбутні категорії без зміни цієї функції).
2. `resolvePluginList(projectRoot, config)`:
   - `declared = config?.plugins` (якщо не масив — старий шлях, повний автодетект).
   - Явний **порожній** масив (`[]`) і далі означає «усі плагіни вимкнено» — семантика не змінюється, per-категорійний backfill до нього не застосовується.
   - Непорожній `declared`: групуємо за категорією → множина `declaredCategories`. `detectPluginsFromRepo()` рахується повністю, але в результат домішуються лише ті знайдені пакети, чия категорія **відсутня** в `declaredCategories`.
   - Результат: `declared` (як є, у порядку користувача) + автодобрані пакети відсутніх категорій.
3. Автодобір нового плагіна цієї категорії логує попередження (`⚠️ автододано плагін <pkg> (категорія <cat>, не задекларована явно в .n-rules.json)`), щоб дописування в `package.json`/`.n-rules.json` (через `ensurePluginInstalled`, коли `allowInstall`) не було тихим сюрпризом.
4. `plugins` у `.n-rules.json` після синку так само переписується повним резолвленим списком (як і зараз для повного автодетект-шляху) — наступний запуск бачить категорію вже «явно» присутньою.

### Consequences

- Good, because релізи нових плагінів/категорій (як сьогоднішній `@7n/rules-lang-js`) більше не осиротжують проєкти з частково заповненим `plugins` — потрібної дії від користувача не вимагається.
- Good, because свідома відмова від категорії (хоч одним записом цієї категорії в `plugins`, включно з навмисно "неповним" списком) і далі поважається — це не full-override-always варіант.
- Good, because категорія виводиться з naming convention, а не хардкодиться — механізм не потребує змін під кожен новий плагін.
- Bad, because подвійна семантика `[]` (усе вимкнено) проти непорожнього списку (частковий override) — менш очевидна з першого погляду, ніж простий all-or-nothing; потребує явного докстрінгу/тесту на обидва edge case.
- Bad, because автодобір і далі виконує мережеву дію (`bun add -d`) без окремого підтвердження користувача — лишається той самий trade-off, що й для «поле відсутнє повністю» шляху сьогодні, просто застосований частіше.

## More Information

Торкається `npm/scripts/lib/resolve-plugins.mjs` (`resolvePluginList`, `detectPluginsFromRepo` — потрібна допоміжна `categoryOf(pkgName)`), і потребує unit-тестів на: непорожній `declared` без lang-категорії → lang домішується; `declared: []` → нічого не домішується; `declared` з обома категоріями → автодетект не викликається взагалі (не марнує файлові сигнали).

Виявлено під час діагностики `lint --full`, що мовчки завершувався в `components` (несинхронізований `node_modules`) і `skill codex taze`, що не мав jsPlugin-провайдера (`⏭ Жодного taze-провайдера`) через той самий корінь — уже заданий `plugins: ["@7n/rules-ci-github"]` заблокував автодетект `@7n/rules-lang-js` після його сьогоднішнього виносу з ядра.
