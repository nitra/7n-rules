## ADR Зміна стилю документації docgen — поведінковий замість вичерпного

## Context and Problem Statement
Скіл `docgen` генерував документацію, перевантажену деталями реалізації: розділи `## Залежності` з переліком `node:fs`/`node:fs/promises`/`node:path`, точними іменами символів (`isAbieRuleEnabled`), таблицями типів і сигнатурами функцій, а також рубриками `Rebuild Test`. Такий вміст не описує бізнес-задачу модуля й старіє разом із кодом.

## Considered Options
* Поведінковий стиль: секції Огляд / Поведінка / Гарантії поведінки / Публічний API (лише за нетривіальної поверхні)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Поведінковий стиль документації", because користувач схвалив запропонований зразок переписаного `enabled.md` — без stdlib-кухні, без прив'язки до імен символів, із покроковим алгоритмом рішення, opt-in-семантикою та інваріантами-гарантіями.

### Consequences
* Good, because transcript фіксує очікувану користь: «мені твій результат сподобався давай адаптуємо під це скіл»; субагенти Tier 1 у прогоні по `npm/rules/abie` підтвердили стиль на 15 файлах.
* Bad, because вже згенерована документація (400 `exists:true` файлів) відображає старий стиль і потребує перегенерації через `--overwrite`.

## More Information
Змінено промпт Tier 1 у: `npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md`. Стаб `.pi/skills/n-docgen/SKILL.md` не містить промпту — лише делегує до `.cursor`-копії. Слово «вичерпну» замінено на «лаконічну поведінкову» у `description`-полі всіх трьох копій. Еталонний файл: `npm/rules/abie/lib/docs/enabled.md`. Прогін `docgen scan --root npm/rules/abie` повернув 15 файлів; усі перегенеровано трьома батчами по 5 субагентів.

## ADR Вибір локальної LLM-моделі для Tier 1 docgen на 8 GB M2

## Context and Problem Statement
Tier 1 docgen генерує ~1042 нових `.md`-файлів через хмарні Claude-субагенти (~26 M токенів, ~$300–570). Розглядалась можливість перевести Tier 1 на локальну Ollama, щоб звести хмарні витрати до нуля. Обмеження: Apple M2 8 GB unified RAM.

## Considered Options
* `gemma4:e4b` (9.6 GB) — перевищує RAM, свопиться
* `gemma3:4b` (3.3 GB) — влазить у GPU-бюджет 8 GB
* `qwen2.5-coder:3b` (2.4 GB) — найшвидша, мова слабша
* `qwen3:4b`, `qwen2.5:7b`, `qwen2.5-coder:7b`, `llama3.1:8b`, `llama3.2:3b`, `gemma2:2b`, `phi3.5` — відсіяні: офлоад у CPU, слабка українська або відмова слідувати директивам

## Decision Outcome
Chosen option: "`gemma3:4b`", because бенчмарк двох раундів (9 моделей у марафоні + 3 моделі × 3 файли з еталоном) показав: `gemma3:4b` — єдина модель, що одночасно (1) повністю вміщується у GPU (4.2 GB, 100% GPU, processor split без CPU-офлоаду) і (2) дає найкращу якість документації (86–91% проти еталона); `qwen2.5-coder:3b` швидша (~36 tok/s vs ~14–21), але якість нижча (~77%) через мовні дефекти й витік сигнатур/stdlib.

### Consequences
* Good, because transcript фіксує очікувану користь: ~14–21 tok/s на теплій моделі (cold-start 24 с), Tier 1 реалістичний як «нічний прогін» (~14–22 год на 1042 файли), вартість $0.
* Bad, because на 8 GB паралелізм відсутній — Tier 1 стає строго послідовним (concurrency = 1); 7B-клас моделей фізично не вміщується без CPU-офлоаду.

## More Information
Вимірювальні прогони: `/tmp/docgen-bench/run.py` (марафон 9 моделей), `/tmp/docgen-bench2/run.py` (3 моделі × 3 файли з еталонними `.md`). Встановлені після бенчмарку: `gemma3:4b` (3.3 GB, залишена), решта видалені через `ollama rm`. Ключовий інваріант: модель має займати ≤~50% RAM, щоб не йти у своп — `gemma4:e4b` (9.6 GB) давала 0.4 tok/s через memory-thrashing.

## ADR Спосіб звернення до локальної Ollama — прямий API vs pi CLI

## Context and Problem Statement
Після вибору `gemma3:4b` постало питання, яким транспортом звертатися до неї у CLI `docgen gen`: прямий `ollama /api/chat` чи через `pi` CLI (налаштований на `http://localhost:11434/v1`). Спостережувалась різниця якості у попередніх замірах: виходи через pi (~87%) були чистішими, ніж прямий виклик (~71%).

## Considered Options
* Прямий `ollama /api/chat` без system-prompt
* Прямий `ollama /api/chat` з system-prompt
* Через `pi --provider ollama --model gemma3:4b` (OpenAI-сумісний ендпойнт `:11434/v1`)

## Decision Outcome
Chosen option: перевага pi підтверджена в межах поточних даних, але її джерело — system-prompt, а не сам інструмент, because перевірочний прогін (C: прямий + system-prompt) дав 85% проти 87% у pi; розрив 2 п.п. знаходиться в межах суб'єктивного шуму ручного оцінювання (±3 п.п.) і корелює з якістю конкретного system-prompt, а не з архітектурою транспорту. Рішення про фінальний транспорт лишилось відкритим — сесія завершилась до остаточного підтвердження.

### Consequences
* Good, because transcript фіксує очікувану користь: pi коректно підключається до `gemma3:4b` через `~/.pi/agent/models.json` (`api: "openai-completions"`, `baseUrl: "http://localhost:11434/v1"`); `pi --list-models gemma` підтверджує видимість моделі (context 128K, max-out 16.4K).
* Bad, because transcript не містить підтверджених негативних наслідків; відкрите питання — node-старт pi на кожен файл як оверхед у послідовному Tier 1.

## More Information
Конфіг провайдера: `~/.pi/agent/models.json`. Схема підключення: `api: "openai-completions"`, `compat.supportsDeveloperRole: false`, `compat.noSystemPrompt: false`. Джерело схеми: `/opt/homebrew/Cellar/pi-coding-agent/0.78.0/.../docs/models.md`. Прогони: `~/docgen-bench3/confirm.py` (C: прямий + system-prompt), попередній `bypyv00hm` (A vs B). Файли-еталони для оцінки: `firebase_hosting.mjs`, `overlay-paths.mjs`, `k8s-tree.mjs` з `npm/rules/abie/`.
