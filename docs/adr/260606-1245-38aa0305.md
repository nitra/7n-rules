### Чому я "зупинився" саме на js-lint і rego

Секвенційна обробка правил + одна ключова подія:

1. **`ga` — першим**, файлу немає → modель має **створити з нуля** → більше кроків (прочитати правило → придумати JSON → write → перечитати → validate) → ETIMEDOUT 220 с, нічого не записано → ескалація на **haiku** → haiku **створив файл** з `github.vscode-github-actions`.

2. **`js-lint` — наступним**, файл **вже існує** → задача стала **EDIT** (read → append extension IDs) замість CREATE → **1–2 tool calls** → gemma встигла за ~180–200 с → записала на диск → **ETIMEDOUT** вбив процес, але **файл уже було змінено** → check-gate зелено → `resolved: local#1`.

3. **`rego` — аналогічно** (append `tsandall.opa`), але local#1 не встиг або перезаписав неправильно → check-gate не зелено → local#2 → знову ETIMEDOUT, але цього разу зміна встигла → `resolved: local#2`.

4. **`text`, `vue`** — ті самі умови (EDIT на існуючому файлі), але:
- Або модель була повільнішою (thermal throttling після 30+ хв безперервної роботи ollama на M2).
- Або зробила conflicting edit (перезаписала файл без наявних extensions) → check-gate провалював всі N спроб.

**Ключовий механізм**: коли `spawnSync` з `timeout: 220_000` вбиває `pi` — **файл уже може бути записаний на диск**, але процес ще читав/валідував. Тому ETIMEDOUT і успіх — не суперечність: gemma писала, потім продовжувала верифікацію, і саме в тій фазі вбило.

### Що це означає для архітектури

```
CREATE FROM SCRATCH ≠ EDIT existing file
ga (create) → 5+ steps → >220s → local fails
js-lint (edit) → ~2 steps → <220s → local succeeds
```

Тобто **local-tier варто dispatch-ити не за правилом, а за типом задачі**:
- `EDIT known file, add 1–2 lines` → local (60 с вистачить)
- `CREATE file from template` → T0 якщо template відомий, або сразу haiku якщо ні
- `cascade / multi-file` → haiku/sonnet

Це і є **classifier перед dispatch** — перший крок до `md→orchestrator-codegen`.
