## ADR Лаконічний поведінковий стиль документації в скілі docgen

## Context and Problem Statement
Згенерована docgen документація (`npm/rules/abie/lib/docs/enabled.md` та інші) містила розділи `## Залежності` з переліком `node:fs`/`node:fs/promises`/`node:path`-викликів, таблиці типів та сигнатури внутрішніх функцій (наприклад, `isAbieRuleEnabled`), а також секцію «Rebuild Test» — деталі, що не впливають на розуміння бізнес-задачі файлу і старіють із кодом.

## Considered Options
* Зберегти «вичерпний» (comprehensive) стиль: усі залежності, сигнатури, внутрішні назви символів
* Перейти на лаконічний поведінковий стиль: бізнес-роль, процес, гарантії — без stdlib-кухні та прив'язки до внутрішніх імен

## Decision Outcome
Chosen option: "лаконічний поведінковий стиль", because саме такий варіант схвалив користувач після перегляду ручного прикладу переписаного `enabled.md` (секції Огляд / Поведінка / Де використовується / Гарантії поведінки, без `## Залежності` і без сигнатур).

### Consequences
* Good, because документація описує «що і навіщо» без деталей реалізації, що залишається актуальною після рефакторингу коду.
* Bad, because раніше згенеровані файли (усі, крім `abie/`-тек, перезаписаних у цій сесії) лишаються в старому стилі й потребують регенерації.

## More Information
Промпт Tier 1 змінено в: `npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md` (`.pi/`-копія — лише стаб, промпт не містить). Ключові правки: секція `## Публічний API` включається лише для нетривіальної зовнішньої поверхні; для модулів-предикатів пропускається; `## Помилки` замінено на `## Гарантії поведінки` (read-only, fail-safe, ігнор поганого вводу). Слово «вичерпну» в описі/меті скіла замінено на «лаконічну поведінкову». Регенеровано 15 файлів `npm/rules/abie/` трьома батчами по 5 субагентів.

## ADR Архітектура гібридного Ollama+Claude запуску для Tier 1 docgen

## Context and Problem Statement
Tier 1 docgen на 1042 файлах коштує ~26 M хмарних токенів (~\$300–570). Оскільки Tier 1 складається з 1042+ незалежних механічних задач «один файл → один md», розглядалась заміна Claude-субагентів на локальну Ollama для усунення хмарних витрат.

## Considered Options
* Повністю замінити Tier 1 на Ollama (єдиний двигун)
* Гібрид: Ollama для Tier 1 + Claude для Tier 2/3 та файлів, що не пройшли quality-gate
* Залишити повністю на Claude

## Decision Outcome
Chosen option: "гібридна архітектура Ollama Tier 1 + Claude Tier 2/3", because синтез (Tier 2/3) потребує якості та коштує лише ~2.6 M токенів, а Tier 1 — механічний і однаковий промпт придатний для локальної моделі. Реалізацію визначено через CLI-команду `docgen gen --engine ollama|claude` та виокремлення промпту в спільний модуль `docgen-prompt.mjs` (єдине джерело правди).

### Consequences
* Good, because transcript фіксує очікувану користь: Tier 1 (~26 M токенів, ~\$300–570) переходить у $0 за умови достатнього заліза.
* Bad, because замір на наявній машині (Apple M2, 8 GB RAM, gemma4:e4b 9.6 GB) показав **0.4 tok/s** у «теплому» стані через своп (модель перевищує RAM), що дає ~37 діб на 1042 файли — гібрид на цьому залізі нежиттєздатний. Рекомендований поріг: модель ≤ ~50% RAM; повний потенціал — від 32 GB.

## More Information
Команди заміру: `ollama list` (модель `gemma4:e4b`, 9.6 GB); вимірювання через `POST http://localhost:11434/api/generate`, результат: warm load 74 с, генерація 0.4 tok/s, інгест 5.4 tok/s. Архітектурні компоненти, визначені в transcript: CLI `docgen gen --engine ollama|claude [--root <dir>] [--concurrency 1] [--num-ctx 8192] [--max-bytes 24000] [--overwrite]`; quality-gate (вихід непорожній, починається з `# <stem>`, обов'язкові секції); shared модуль `npm/skills/docgen/js/docgen-prompt.mjs`. Реалізацію в код не внесено — сесія завершилась оцінкою feasibility.
