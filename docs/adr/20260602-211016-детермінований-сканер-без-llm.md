## Context and Problem Statement
Необхідно визначити список файлів і модулів для генерації документації. Питання: чи використовувати LLM або мережеві запити для обходу файлової системи, чи обмежитися синхронним детермінованим кодом.

## Considered Options
* Синхронний `readdirSync` без LLM/network (детермінований сканер)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Синхронний `readdirSync` без LLM/network", because сканер повинен бути передбачуваним, швидким і не залежати від зовнішніх сервісів; LLM підключається лише на рівні скіла під час генерації.

### Consequences
* Good, because сканер є повністю детермінованим і відтворюваним.
* Good, because відсутність мережевих викликів прискорює обхід великих репозиторіїв.
* Good, because чіткий поділ відповідальності: `docgen-scan.mjs` — лише список файлів, скіл — генерація.
* Neutral, because transcript не містить підтвердження наслідку щодо складності підтримки.

## More Information
Файл: `npm/skills/docgen/js/docgen-scan.mjs`. Використовується `Array#toSorted()` замість мутуючого `.sort()`.

## ADR `Array#toSorted()` замість `.sort()`

## Context and Problem Statement
Під час сортування файлів у сканері потрібно обрати між мутуючим `.sort()` та імутабельним `Array#toSorted()`.

## Considered Options
* `Array#toSorted()` — імутабельний метод
* `.sort()` — мутуючий метод

## Decision Outcome
Chosen option: "`Array#toSorted()`", because імутабельний підхід безпечніший і відповідає правилу `unicorn/no-array-sort`, яке є помилкою у цьому репозиторії.

### Consequences
* Good, because вихідний масив не мутується, що зменшує ризик побічних ефектів.
* Good, because відповідає активному `unicorn/no-array-sort` eslint-правилу репозиторію.
* Neutral, because transcript не містить підтвердження наслідку щодо підтримки середовищами.

## More Information
Файл: `npm/skills/docgen/js/docgen-scan.mjs`. Правило: `unicorn/no-array-sort` у eslint-конфігурації репозиторію.

## ADR Трирівнева архітектура генерації документації

## Context and Problem Statement
Необхідно визначити структуру згенерованої документації для монорепозиторію з багатьма файлами та пакетами. Питання: як організувати ієрархію doc-файлів щоб вони відображали реальну структуру коду.

## Considered Options
* Трирівнева ієрархія: Tier 1 (file-level) → Tier 2 (module-summary) → Tier 3 (domain docs)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Трирівнева ієрархія", because вона відображає природну структуру монорепозиторію: окремі файли → пакет → домен.

### Consequences
* Good, because документація масштабується разом зі структурою проєкту.
* Good, because Tier 2 і Tier 3 є агрегатами нижчих рівнів, що забезпечує узгодженість.
* Bad, because суворий порядок виконання Tier 1 → 2 → 3 унеможливлює повну паралелізацію.

## More Information
Tier 1: `<dir>/docs/<stem>.md`; Tier 2: `<module_root>/docs/ARCHITECTURE.md`; Tier 3: `docs/<domain>.md`. Tier 1 і Tier 2 виконуються батчами по 5 агентів паралельно; Tier 3 — один агент.

## ADR Рішення про перезапис документа приймає скіл, а не сканер

## Context and Problem Statement
Сканер знає, чи існує вже doc-файл для певного вихідного файлу. Питання: де має прийматися рішення про пропуск або перезапис існуючого документа.

## Considered Options
* Сканер лише вказує прапор `exists`; рішення про skip/overwrite приймає скіл
* Сканер самостійно фільтрує вже існуючі файли
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Сканер вказує `exists`, скіл вирішує", because це дотримується принципу єдиної відповідальності: сканер є read-only інструментом збору даних, а скіл містить бізнес-логіку генерації.

### Consequences
* Good, because скіл може реалізовувати різну поведінку: Tier 1 пропускає `exists:true` без `--overwrite`, Tier 2 завжди регенерує як агрегат.
* Good, because сканер залишається простим і незалежним від стратегії генерації.
* Neutral, because transcript не містить підтвердження наслідку щодо можливих майбутніх стратегій overwrite.

## More Information
Tier 1: skip якщо `exists:true` без `--overwrite`; Tier 2: завжди регенерується. Команди: `docgen scan`, `docgen modules` у `npm/bin/n-cursor.js`.
