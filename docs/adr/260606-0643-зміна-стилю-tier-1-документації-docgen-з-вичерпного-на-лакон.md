## ADR Зміна стилю Tier-1 документації docgen з вичерпного на лаконічний поведінковий

## Context and Problem Statement
Скіл docgen генерував документацію в «вичерпному» стилі: таблиці типів, сигнатури функцій, перелік stdlib-залежностей, секція `## Помилки`, перелік внутрішніх хелперів. Отримані документи дублювали реалізацію замість пояснення поведінки, а це суперечило задекларованій меті — зрозуміти файл **без** читання коду.

## Considered Options
* Лаконічний поведінковий стиль (секції Огляд / Поведінка / Де використовується / Гарантії поведінки; без сигнатур, stdlib, внутрішніх імен)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Лаконічний поведінковий стиль", because промпт у `SKILL.md` переписали у двох робочих копіях (`npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md`): прибрали сигнатури і таблиці типів, замінили `## Помилки` на `## Гарантії поведінки`, `## Публічний API` дозволено лише за нетривіальної зовнішньої поверхні, порожніх секцій не вигадувати.

### Consequences
* Good, because transcript фіксує очікувану користь: після переписування промпту прогін по `npm/rules/abie` (15 файлів) дав доки без дампу сигнатур, без `Rebuild Test`-переліку, без stdlib-переліку — агент підтвердив «стиль саме той».
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Змінені файли: `npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md`, `.pi/skills/n-docgen/SKILL.md` (стаб). Еталонний вивід: `npm/rules/abie/lib/docs/enabled.md`. Регенерація виконувалась через `npx @nitra/cursor docgen scan --root npm/rules/abie` + батчі субагентів по 5.

## ADR Вибір gemma3:4b як локальної моделі Tier 1 на 8 GB M2

## Context and Problem Statement
Для локального (офлайн, $0) прогону Tier 1 docgen (1042 файли, ~26 M хмарних токенів) необхідна Ollama-модель, що повністю вміщається в 8 GB unified memory Apple M2 і дає прийнятну якість документації українською.

## Considered Options
* gemma3:4b (~3.3 GB, 100% GPU)
* qwen2.5-coder:3b (~2.4 GB, 100% GPU)
* qwen2.5:7b (~5.1 GB, 9%/91% CPU/GPU офлоад)
* qwen2.5-coder:7b (~5.1 GB, офлоад)
* llama3.1:8b (~5.7 GB, офлоад)
* llama3.2:3b (~2.9 GB, 100% GPU)
* gemma2:2b (~3.0 GB, 100% GPU)
* phi3.5 (~4.0 GB, 100% GPU)
* qwen3:4b (~3.6 GB, 100% GPU) — відхилено: ігнорує `/no_think`, думає російською
* gemma4:e4b (~9.6 GB) — відхилено: не вміщається, 0.4 tok/s (своп), 69% CPU-офлоад

## Decision Outcome
Chosen option: "gemma3:4b", because benchmark (3 моделі × 3 файли, чистий старт) показав: лише gemma3:4b і qwen2.5-coder:3b мають 100% GPU (повністю в RAM); якість gemma3:4b — ~86% vs еталон, qwen2.5-coder:3b — ~77% з мовними дефектами («Перикоджує», «Рules», leaked stdlib/сигнатури). Розрив 9 п.п. зумовив вибір якості над швидкістю.

### Consequences
* Good, because transcript фіксує: gemma3:4b дає 4.2 GB resident, 100% GPU, ~14–21 tok/s (чистий прогін) — повністю в RAM без свопу; якість UA найкраща в fit-класі.
* Bad, because на 8 GB послідовний Tier 1 (concurrency=1) займає ~14–38 год на 1042 файли; qwen2.5-coder:3b швидша (~11 год), але посередня мова.

## More Information
Benchmark-скрипти: `/tmp/docgen-bench/run.py`, `/tmp/docgen-bench2/run.py`. Виміряні дані у `/tmp/docgen-bench2/results.jsonl`. Правило вибору: модель **має вміщатися** в ~50% RAM (лишити місце під KV-cache і ОС) — усі 7B+ на 8 GB дали CPU-офлоад і деградацію.

## ADR Гібридна архітектура docgen: Ollama для Tier 1, Claude для Tier 2/3

## Context and Problem Statement
Повний хмарний прогін docgen на 1442 файлах коштує ~28–38 M токенів (~$300–770). Tier 1 (файли) лінійний і механічний — пряме джерело витрат; Tier 2 (модулі) та Tier 3 (домени) вимагають синтезу і коштують лише ~2.6 M токенів.

## Considered Options
* Гібрид: Tier 1 → локальна Ollama (gemma3:4b), Tier 2/3 → Claude
* Повністю хмарний (Claude на всіх рівнях)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Гібрид: Tier 1 → локальна Ollama, Tier 2/3 → Claude", because Tier 1 — 1042 незалежних механічних задачі «один файл → один md»; якість тут терпима (чорновий драфт). Tier 2/3 — синтез, якість критична, одиниць мало (16 модулів + 1 синтезатор), коштує ~2.6 M — копійки. Перенесення Tier 1 на Ollama обнуляє найбільшу статтю.

### Consequences
* Good, because transcript фіксує очікувану користь: Tier 1 на Ollama → $0 за ~26 M токенів; Claude лишається лише на Tier 2/3 (~2.6 M).
* Bad, because Ollama Tier 1 послідовний (concurrency=1 на одній машині), займає ~14–38 год на 8 GB M2; якість 4B-класу ~77–86% vs Claude-еталон.

## More Information
Архітектурна вимога: винести Tier-1 промпт у `docgen-prompt.mjs` (спільний для Claude-субагентів і Ollama-CLI) щоб не розійтися. CLI-команда `npx @nitra/cursor docgen gen --engine ollama --model gemma3:4b --concurrency 1 --max-bytes <N>` — запропонована, але не реалізована в сесії. `DOCGEN_IGNORE_GLOBS` у `docgen-ignore.mjs` — найдешевший важіль зменшення обсягу Tier 1.

## ADR System-prompt як головний чинник якості локальної генерації docgen

## Context and Problem Statement
Під час порівняння «прямий Ollama API» vs «через pi CLI» виявилась несподівана різниця в якості документів (+14–16 п.п.) при однаковій моделі gemma3:4b. Необхідно встановити справжню причину різниці, щоб правильно вибрати транспорт.

## Considered Options
* System-prompt як причина різниці (pi вбудовує coding-assistant system-prompt; прямий виклик без system)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "System-prompt як причина різниці", because контрольний прогін C (прямий `/api/chat` + system-role) дав ~85% vs еталон — рівень pi (~87%); прямий без system давав ~71%. Тобто два п.п. між B(pi) і C — шум семплінгу, а не властивість транспорту.

### Consequences
* Good, because transcript фіксує: два повідомлення (`system` — правила стилю; `user` — файл) у прямому виклику дають якість pi-рівня без node-оверхеду і з повним контролем `num_ctx`/`num_predict`.
* Bad, because transcript фіксує: gemma3:4b лише частково виконує негативні обмеження («без stdlib/regex») — 4B-стеля модель не перетинає незалежно від транспорту; залишковий `fence`-wrap потребує post-process.

## More Information
Підтверджуючий експеримент: `/tmp/docgen-bench3/confirm.py` → `C_sys__*.md`. Виміри: A (без system) ~71%, B (pi) ~87%, C (прямий+system) ~85%. Транспорти A і C ідентичні за tok/s (~20). Різниця у відсотках між B і C (~2 п.п.) — в межах суб'єктивного шуму оцінювання ±3 п.п.

## ADR Вибір per-file виклику pi замість RPC-сесії для docgen

## Context and Problem Statement
Щоб прибрати ~4 с node-оверхеду pi на файл (≈+1.1 год на 1042 файли), розглянуто персистентну RPC-сесію (`pi --mode rpc`) з однократним boot і багатьма промптами через stdin.

## Considered Options
* Per-file виклик pi (`pi -p --no-session ...`) — незалежні виклики
* Персистентна RPC-сесія (`pi --mode rpc`) — один процес на батч

## Decision Outcome
Chosen option: "Per-file виклик pi", because RPC-прогін (boot ~1.1 с) показав 267/144/296 с на файл (проти ~30/74/70 с у per-file) — контекст накопичується між незалежними файлами, кожен наступний файл тягне за собою попередні виводи, що призводить до memory pressure на 8 GB і крос-забруднення. Для docgen файли незалежні й потребують чистого контексту.

### Consequences
* Good, because per-file виклик дає чистий контекст на кожен файл — без крос-забруднення між документами.
* Bad, because per-file платить ~4 с node-старту на файл (≈+1.1 год на 1042 файли); цей оверхед не усувається без RPC.

## More Information
Benchmark: `~/docgen-bench3/pi_recipe.py`. `ollama ps` під час RPC показав `CONTEXT 4096, 100% GPU` — накопичення йде, своп збільшує latency. Рецепт: `pi -p --provider ollama --model gemma3:4b --no-tools --no-session --append-system-prompt "$STYLE" ... | sed '/^```/d'`. Конфіг провайдера: `~/.pi/agent/models.json` з `api: "openai-completions"`, `baseUrl: "http://localhost:11434/v1"`.
