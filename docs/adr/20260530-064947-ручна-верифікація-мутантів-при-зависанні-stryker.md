## ADR Ручна верифікація мутантів при зависанні Stryker

## Context and Problem Statement
Під час сесії `n-coverage-fix` повний прогін `bun run coverage` (Stryker incremental) тривав понад 54 хвилини без результату — incremental cache стала недійсною (`0 of 25053 mutant result(s) are reused`) і CPU додатково споживали залишені з попередніх сесій процеси `bun -e …fixSurvivedMutants` (PID 90242, 91321). Потрібно було підтвердити, що 4 нові тести дійсно вбивають 4 вцілілих мутанти у `coverage.mjs` (рядки 189 і 211), не чекаючи на весь Stryker.

## Considered Options
* Дочекатися завершення Stryker — зупинити і перезапустити повний прогін
* Вручну застосувати кожен мутант до source, запустити цільовий `bunx vitest run -t "<test-name>"` і перевірити, що тест падає

## Decision Outcome
Chosen option: "Вручну застосувати кожен мутант і запустити targeted vitest", because Stryker завис (54+ хв), а вручну валідація дає відповідь за кілька секунд і не залежить від стану incremental cache.

### Consequences
* Good, because transcript фіксує очікувану користь: кожен із 4 мутантів (`if(true)`, `if(false)`, `{}`, `fix:true`) був застосований до `npm/rules/test/coverage/coverage.mjs`, викликав падіння відповідного тесту, і source було відновлено — усі 57 тестів зелені.
* Bad, because COVERAGE.md не оновився автоматично — mutation score 93.62% (132/141) залишається з попереднього прогону; наступний `bun run coverage` має оновити до ≥136/141.

## More Information
- Завислі процеси вбиті: `kill 90242 91321`
- Мутанти перевірялись у `npm/rules/test/coverage/coverage.mjs` L188–211
- Vitest команди: `bunx vitest run rules/test/coverage/tests/coverage.test.mjs -t "opts.fix gate"` та `-t "2-ї стрілки"`
- Повний Stryker прогін з `npm/stryker.config.mjs` (`perTest: true`, incremental report у `npm/reports/stryker/incremental.json`)

## ADR Стратегія тестів для gate `opts.fix` і дефолту `{ fix: false }` у `coverage.mjs`

## Context and Problem Statement
Stryker зафіксував 4 вцілілих мутанти у `npm/rules/test/coverage/coverage.mjs`: два на L189 (`if (opts.fix)` → `if (true)` / `if (false)`) і два на L211 (`runCoverageSteps({ fix: false })` → `runCoverageSteps({})` / `runCoverageSteps({ fix: true })`). Потрібно дописати тести, що вбивають ці мутанти, не порушуючи 49 існуючих тестів.

## Considered Options
* Мок-перехоплення `withLock` і `fixSurvivedMutants` + `mock.calls` inspection для L189; `.toString()` source inspection для L211
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Мок-перехоплення + `.toString()` inspection", because ці техніки ізольовані від файлової системи (не потребують реального `bun run coverage`) і безпосередньо розрізняють логічні/літеральні мутанти без end-to-end прогону.

### Consequences
* Good, because transcript фіксує очікувану користь: 3 нові тести (`opts.fix=false → fixSurvivedMutants НЕ викликається`, `opts.fix=true → викликається`, `2-ї стрілки toString містить fix:false`) стали 57/57 замість 49/49, і кожен мутант падає в цільовому тесті.
* Bad, because тест `2-ї стрілки` покладається на `fn.toString()` — якщо рушій JS зміниться або мінімайзер видалить токен `fix`, тест може хибно провалитись. Transcript не містить підтвердження, що цей ризик обговорювався.

- Regex-константи в тестах: `FIX_TOKEN_RE`, `FALSE_TOKEN_RE`, `FIX_TRUE_RE` — перевіряють `.toString()` другого аргументу `withLock.mockImplementation`
