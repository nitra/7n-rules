## ADR Розмежування скоупів `/n-fix` і `/n-lint`

**Контекст:** Скіли `/n-fix` та `/n-lint` частково дублювали один одного: крок 6 у `/n-fix` перебирав окремі `lint-*` скрипти, що перетинались з роботою `/n-lint`. Обговорювалось злиття в одну команду.

**Рішення/Процедура/Факт:** Вирішено не зливати, а чітко розмежувати: `/n-fix` відповідає виключно за структурну відповідність `.cursor/rules/` (перевірки через `npx @nitra/cursor check`), а `/n-lint` — за чистоту коду через `bun run lint`. З `npm/skills/fix/SKILL.md` та `.cursor/skills/n-fix/SKILL.md` прибрано крок 6 (виклик `bun run lint`), додано секцію **«Скоуп»** з явною забороною запускати `bun run lint` із цього скілу. Cross-reference у `npm/rules/style-lint/style-lint.mdc` оновлено з «повний набір `lint-*` (навичка `n-fix`)» → «`bun run lint` (навичка `/n-lint`)».

**Обґрунтування:** `/n-fix` — рідкісна операція (онбординг/міграція), `/n-lint` — частий прогін перед комітом; запуск важкого ланцюжка `bun run lint` (≈2 хв) при структурних правках — зайвий overhead. Розділення також зберігає `bun run lint` як єдиний канонічний контракт, ідентичний до CI, без прихованих викликів із середини іншого скілу.

**Розглянуті альтернативи:** Злиття в одну команду (відхилено через різну частоту і вартість запуску); делегування — `/n-fix` викликає `/n-lint` наприкінці (відхилено: порушує принцип однієї відповідальності та CLAUDE.md-заборону паралельних ESLint-запусків).

**Зачіпає:** `npm/skills/fix/SKILL.md`, `.cursor/skills/n-fix/SKILL.md`, `npm/rules/style-lint/style-lint.mdc`, `npm/CHANGELOG.md`.

## ADR Реструктуризація `rules/<id>/js/` → `rules/<id>/{fix,lint}/`

**Контекст:** Каталог `npm/rules/<rule>/js/` містив два принципово різні типи артефактів в одному місці: `<concern>/check.mjs` (діагностика для `n-fix`-каналу) і `lint.mjs` / `run.mjs` (CLI-entry для `bun run lint-X`). Це ускладнювало навігацію й суперечило новому розмежуванню скоупів.

**Рішення/Процедура/Факт:** Обрана триетапна стратегія. **Фаза 1 (виконана):** dual-mode інфраструктура без переміщення файлів — `discover-checkable-rules.mjs` сканує обидва корені (`js/` і `fix/`), кожен концерн штампується полем `rootDir: 'js' | 'fix'`; дублікат одного концерну в обох каталогах — fatal-помилка. `run-rule.mjs` використовує `rootDir` при побудові шляху з fallback на `'js'`. Тести розширено до 19 (було 12). Версія `1.11.8 → 1.11.9`. Фаза 2 (не виконана): правила по одному `git mv js/<concern>/ fix/<concern>/` + `git mv js/lint.mjs lint/lint.mjs`. Фаза 3: прибирання legacy fallback і оновлення 5 статичних імпортів у `n-cursor.js`. Rego-політики визначено класти у `fix/policy/<concern>/` (варіант B), а не плоско в `fix/<concern>/` — щоб зберегти ізоляцію між JS і rego runner-ами в межах одного концерну.

**Обґрунтування:** Назва каталогу виражає канал (`fix`/`lint`), а не технологію (`js`). Поетапна стратегія обрана для збереження зелених тестів на кожному кроці; dual-mode у Фазі 1 дозволяє перемащати правила поступово без Big Bang.

**Розглянуті альтернативи:** Big-bang в одному PR (відхилено — важко ревʼювити, один зламаний імпорт валить усе); розміщення policy плоско в `fix/<concern>/` поруч із `check.mjs` (варіант C, відхилено — колізія імен концернів між JS і rego, `runConftestBatch` і dynamic `import` потребують різних runner-ів).

**Зачіпає:** `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/run-rule.mjs`, обидва `*.test.mjs`, `npm/bin/n-cursor.js` (фаза 3), `npm/package.json`, `npm/CHANGELOG.md`.
