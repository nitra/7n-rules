## ADR Заміна окремого бінарника `kustomize` на вбудовану підкоманду `kubectl kustomize`

## Context and Problem Statement
`lint-k8s` (файл `npm/rules/k8s/lint/lint.mjs`) викликав окремий бінарник `kustomize build <dir>` для збірки маніфестів перед передачею в kubescape. Це вимагало наявності самостійного виконуваного файлу `kustomize` у `PATH` — окремої залежності, яка могла бути відсутня (наприклад, у CI чи у локальних середовищах без неї), що спричиняло падіння лінтера.

## Considered Options
* Продовжувати використовувати окремий бінарник `kustomize build <dir>`
* Перейти на вбудовану підкоманду `kubectl kustomize <dir>`

## Decision Outcome
Chosen option: "Перейти на вбудовану підкоманду `kubectl kustomize <dir>`", because `kubectl` (v1.36.1, Kustomize v5.8.1) уже є обов'язковим інструментом у проєкті, його вбудована підкоманда `kustomize` функціонально еквівалентна окремому бінарнику, і це усуває потребу встановлювати та підтримувати зайву залежність.

### Consequences
* Good, because transcript фіксує очікувану користь: `kubectl` доступний на runner-ах та локально (`/opt/homebrew/bin/kubectl`), а крок «Install kustomize» у CI-сніпеті `k8s.mdc` прибрано — пайплайн став коротшим.
* Good, because всі 222 тести `npm/rules/k8s/` залишилися зеленими після зміни (`222 pass, 0 fail`).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Змінені файли: `npm/rules/k8s/lint/lint.mjs` (функції `runKustomizeBuild` і `runKubescape`), `npm/rules/k8s/k8s.mdc` (рядки залежностей і CI-сніпет), `npm/CHANGELOG.md`, `npm/package.json`.
- Версія правила `k8s.mdc`: `1.37` → `1.38`; версія пакета `@nitra/cursor`: `1.13.49` → `1.13.50`.

## ADR Зберігання нагадування про CHANGELOG лише у feedback-пам'яті

## Context and Problem Statement
Під час сесії асистент не оновив `CHANGELOG.md` і не зробив version bump після змін у коді. Виникло питання, як унеможливити або знизити ймовірність повторення цього в майбутніх сесіях.

## Considered Options
* Stop-hook у `settings.json` + feedback memory (автоматичний запуск `npx @nitra/cursor check changelog` після кожної зупинки)
* Тільки feedback memory (нагадування лише через збережений контекст без hook-механізму)

## Decision Outcome
Chosen option: "Тільки feedback memory", because користувач явно обрав цей варіант, відмовившись від автоматичного hook-а.
