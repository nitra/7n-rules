## ADR Concern-based layout і `target.json` для правил `@nitra/cursor`

**Контекст:** CLI `@nitra/cursor check` запускав перевірки через один `rules/<id>/js/check.mjs` на правило, а targeting для rego-полісі — через жорстко-кодовану таблицю `TARGETS` у `lint-conftest.mjs`. Ні масштабованості, ні ізоляції між концернами одного правила.

**Рішення/Процедура/Факт:** Введено нову структуру каталогів: `rules/<id>/js/<concern>/check*.mjs` (симетрично до `policy/<concern>/`) + `target.json`-маніфест поряд із кожним rego-пакетом. CLI виконує discovery автоматично: для JS — шукає `check*.mjs` у підкаталогах `js/` (не `utils/`); для rego — шукає підкаталоги `policy/` із `target.json`. Кебаб-ідентифікатори правил (`style-lint`, `image-compress`) конвертуються у snake для namespace rego (`style_lint.package_json`). Applies-гейт реалізується через named export `applies()` у `js/applies/check.mjs`. Додано три нові утиліти: `resolve-target-files.mjs` (picomatch-резолвер `single`/`walkGlob` із shared walk-кешем), `discover-checkable-rules.mjs`, `run-rule.mjs`. `conftest.combine` — не включено до schema (YAGNI).

**Обґрунтування:** Симетрія `js/<concern>/` ↔ `policy/<concern>/` дає одну точку входу для кожної логічної перевірки незалежно від реалізації. `target.json` per-policy — мінімальна декларація поряд із rego-файлом, уникає розростання центрального `lint-conftest.mjs:TARGETS`. picomatch — нуль транзитивних залежностей, один regex-test на шлях. `applies()` в JS замість окремого `rule.json` (що спростило поверхню) — рішення користувача.

**Розглянуті альтернативи:** (1) Feed-everything через `conftest test .` — відкинуто: rego не знає, проти чого запускатись без targeting; (2) name-convention (ім'я каталогу → файл) — відкинуто: конфлікт між `single` і walk-таргетами для однойменних полісі у різних правилах; (3) `required: true` для `walkGlob` — заборонено у schema; applicability gate лишається в `applies()`.

**Зачіпає:** `npm/bin/n-cursor.js`, `npm/scripts/utils/{resolve-target-files,discover-checkable-rules,run-rule}.mjs`, `npm/schemas/target.json`, `npm/rules/*/policy/*/target.json` (19 нових файлів у 12 правилах), `npm/rules/rego/js/{check.mjs → видалено, applies/check.mjs → новий}`, `bun.lock`, `npm/package.json` v1.11.0.

## ADR JS-сторона: конвенція запуску check-файлів

**Контекст:** Discovery JS-перевірок потребував чіткого правила, які саме `.mjs` запускати, щоб випадково не захопити тести, хелпери або загальний код у підкаталогах.

**Рішення/Процедура/Факт:** Запускати тільки `check.mjs` і `check-*.mjs` у відповідному підкаталозі; `*.test.mjs` пропускати; спільний код класти в `utils/` поза `js/` (правило-рівень) або в `npm/scripts/utils/` (пакет-рівень). Порядок — алфавіт. Спільний стан між кількома `check-*.mjs` — module-level singleton-кеш у хелпері (платить тільки перший виклик, наступні отримують готове). Walk-кеш — `Map`, що передається зверху вниз через `runChecks`-scope, щоб уникнути глобального стану в тестах.

**Обґрунтування:** Однозначна конвенція без окремого конфігурацій-файлу. Алфавітний порядок передбачуваний; якщо між файлами потрібен explicit-порядок — це сигнал про залежність по стану, яку треба виразити через явний виклик у хелпері. Module-level кеш природний для одноразового прогону (новий процес — новий кеш).

**Розглянуті альтернативи:** Явна анотація в файлі (наприклад, `export const runnable = true`) — відкинуто, зайва громіздкість. Числові префікси для порядку (`01-firebase.mjs`) — відкинуто на користь змістовних імен без штучного порядку.

**Зачіпає:** `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/run-rule.mjs`, майбутній розпил `abie/js/check.mjs` на концерни.
