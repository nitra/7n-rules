## ADR Драбина ескалації моделей у fix-конформності

## Context and Problem Statement
Движок конформності (`orchestrator.mjs`) використовував грубу ескалацію — один глобальний `failCount`-цикл: після 2 фейлів підряд модель `min` замінялась на `avg`, без зворотного зв'язку у prompt і без структурованого логу. Потреба: спостережуваний, покроковий каскад із знанням «чому не вдалося» і аналітикою для покращення.

## Considered Options
* Детермінована драбина per-rule із feedback-петлею і per-рунговим escalation-логом
* Існуючий глобальний `failCount`-цикл `min→avg` (без змін)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Детермінована драбина per-rule із feedback-петлею і per-рунговим escalation-логом", because глобальний цикл не відрізняв systemic-помилки (memory-guard / auth) від prompt-фейлів і не давав ніякого контексту наступному виклику; детермінована послідовність рунгів дозволяє fail-fast та аналіз «чи допомогло».

Драбина (функція `escalateRule`, `orchestrator.mjs`): `local-min` → `local-min`+feedback → `cloud-min` (pi) → `cloud-avg` (pi, під кепом `--max-avg`, дефолт 3). Кожен рунг: виклик worker → re-check цього правила → запис у `.n-cursor/fix-escalation.jsonl`. Достроковий вихід: systemic-помилка локального → пропуск рунгів тієї ж моделі; no-key на хмарному → abort драбини; вичерпаний avg-кеп → log (не мовчки). Feedback-петля: у prompt передаються попередні зміни + залишковий violation; модель повертає поле `diagnosis` у JSON-відповіді (один виклик, не окремий «аналіз»-виклик на локальних тирах).

Формат запису escalation-логу: `{ts, ruleId, tier, model, callOk, callError, recheckOk, remainingViolation, diagnosis, ms}`. Kill-switch: `N_CURSOR_FIX_ESCALATION_LOG=0`. Перевизначення моделей: `N_CURSOR_FIX_MODEL` / через `N_LOCAL_MIN_MODEL` / `N_CLOUD_MIN_MODEL` / `N_CLOUD_AVG_MODEL`.

### Consequences
* Good, because transcript фіксує очікувану користь: кожен рунг незалежно логується → join із наявним wire-trace за `caller: fix:<rule>:<rung>` дає повну картину вартості ескалацій.
* Good, because systemic-помилки локального (класифіковані `classifyOmlxError`) не витрачають quota хмарних рунгів — fail-fast узгоджено з принципом `[[fail-fast-over-waiting]]`.
* Bad, because «аналіз чому» на локальних тирах — лише поле `diagnosis` у тому самому виклику, а не окремий LLM-крок; глибина аналізу обмежена спроможністю min-моделі.

## ADR Аналітика escalation-логу через cloud-avg модель

## Context and Problem Statement
Після впровадження escalation-логу (`escalation-log.mjs`) виникла можливість систематично аналізувати, які правила ескалюють у хмару або падають повторно, і на підставі цього покращувати детермінований T0-рівень або `.mdc`-інструкції. Бракувало автоматизованого кроку, який перетворює лог на actionable рекомендації.

## Considered Options
* Новий JS-модуль `analyze-escalation.mjs` із чанкінгом і викликом cloud-avg моделі, запускається авто-хуком наприкінці `lint --full` та через CLI-subcommand
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Новий JS-модуль `analyze-escalation.mjs` із чанкінгом і викликом cloud-avg моделі", because це єдиний розглянутий підхід; автоматичний хук прив'язаний до `lint --full` гарантує, що аналіз відбувається відразу після прогону, де накопичились дані.

Реалізація: `readEscalationRecords(path, sinceOffset)` читає або весь лог, або записи одного прогону (від байтового зсуву). `chunkRecords(records, maxChars=40000)` стискає записи та ділить за бюджетом символів. `analyzeEscalations` — map-reduce: cloud-avg виклик по кожному чанку → синтез за кількох чанків. Мета-prompt пріоритизує правила, що ескалювали в хмару або падали повторно, і просить три типи рекомендацій: (A) детермінований T0-патерн для `t0.mjs`; (B) уточнення `.mdc`-інструкцій для локальної min-моделі; (C) зміни у check-скриптах. Звіт append у `.n-cursor/fix-escalation-analysis.md`. CLI: `npx @nitra/cursor analyze-escalation`. Авто-хук у `runLint` (`npm/rules/lint/js/orchestrate.mjs`): зсув записується до конформність-фази, `maybeAnalyzeEscalation` викликається після. Gated: kill-switch `N_CURSOR_FIX_ANALYZE=0`, наявність `N_CLOUD_AVG_MODEL`, наявність ескалацій. Помилки моделі не валять lint.

### Consequences
* Good, because transcript фіксує очікувану користь: рекомендації пріоритизуються за вартістю (cloud-escalations та повторні фейли); T0-патерни повністю прибирають LLM з дешевих випадків.
* Bad, because авто-хук робить cloud-avg виклик наприкінці кожного `lint --full` із ескалаціями, що генерує вартість; передбачений kill-switch (`N_CURSOR_FIX_ANALYZE=0`) для вимкнення авто-режиму.
