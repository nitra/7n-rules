## ADR Архітектура перевірки — rego-heavy (три rego policy-пакети)

## Context and Problem Statement
Нове правило `rust` має перевіряти три документи: `package.json` (наявність `scripts.lint-rust`), `.vscode/extensions.json` (VSCode-розширення), `.github/workflows/lint-rust.yml` (CI-workflow). Потрібно обрати, де зосередити логіку перевірки: у rego (`runConftestBatch`) чи у JS (`check.mjs`).

## Considered Options
* Rego-heavy — три rego policy-пакети + `js/tooling/check.mjs` лише як FS-existence orchestrator (як `js-lint`/`style-lint`)
* JS-heavy — лише один rego policy-пакет `policy/vscode_extensions/`, решта — JS-логіка (як `tauri`)
* Гібрид — rego для `package_json` і `vscode_extensions`, CI workflow — JS

## Decision Outcome
Chosen option: "rego-heavy", because нове правило має ті самі три документи, що й `js-lint`/`style-lint`; дрейф від цієї конвенції накопичив би технічний борг при майбутніх змінах перевірок.

### Consequences
* Good, because `rust.policy.package_json`, `rust.policy.vscode_extensions`, `rust.policy.lint_rust_yml` — одноманітний патерн, легко розширювати (напр. `cargo deny`); drift-тести rego гарантують sync між template і перевіркою.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Канонічний аналог: `npm/rules/style-lint/policy/`, `npm/rules/js-lint/policy/`. Утиліта: `npm/scripts/utils/run-conftest-batch.mjs`. Спека: `docs/superpowers/specs/2026-05-23-rust-rule-design.md`.

## ADR Канонічна команда `lint-rust` — fmt з fix + clippy fix + clippy -D warnings

## Context and Problem Statement
Потрібно зафіксувати єдину канонічну послідовність команд для `scripts.lint-rust` у `package.json`: обрати жорсткість (fail-fast `--check` vs автовиправлення `--fix`) для локального виконання.

## Considered Options
* `cargo fmt --check && cargo clippy -D warnings` (жорсткий, без fix)
* `cargo fmt --all && cargo clippy --fix … && cargo clippy … -D warnings` (fmt з fix + clippy fix + перевірка)
* `cargo fmt --check && cargo clippy --fix … && cargo clippy … -D warnings` (fmt без fix + clippy fix)

## Decision Outcome
Chosen option: "`cargo fmt --all && cargo clippy --fix --allow-staged --allow-dirty --all-targets --all-features && cargo clippy --all-targets --all-features -- -D warnings`", because користувач обрав варіант «fmt (with fix) + clippy --fix + clippy -D warnings» для локального dev-workflow з автовиправленням.

### Consequences
* Good, because transcript фіксує очікувану користь: `--allow-staged --allow-dirty` необхідні для роботи на брудному дереві у локальному dev; триетапна послідовність гарантує і форматування, і виправлення, і фінальну перевірку.
* Bad, because CI-workflow використовує `--check` (без `--fix`) — відмінна від локальної команди семантика; rego-перевірка `rust.package_json` валідує підрядки, тож точна послідовність прапорів в `package.json` є джерелом правди.

## More Information
Canonical template: `npm/rules/rust/policy/package_json/template/package.json.contains.json`. CI-команди (без `--fix`): `cargo fmt --all -- --check` та `cargo clippy --all-targets --all-features -- -D warnings` у `lint-rust.yml`.

## ADR CI-workflow — `dtolnay/rust-toolchain@stable` + `Swatinem/rust-cache@v2`

## Context and Problem Statement
Для `.github/workflows/lint-rust.yml` потрібно визначити спосіб встановлення Rust toolchain у CI та наявність кешування `target/`/registry для скорочення часу виконання.

## Considered Options
* `dtolnay/rust-toolchain@stable` безпосередньо у `lint-rust.yml`
* `actions-rs/toolchain` (застарілий)
* Ручна установка через `curl rustup.rs`

## Decision Outcome
Chosen option: "`dtolnay/rust-toolchain@stable` + `Swatinem/rust-cache@v2`", because `dtolnay/rust-toolchain` — підтримуваний world-standard action; `Swatinem/rust-cache` додано як best practice, оскільки `dtolnay/rust-toolchain` самостійно не кешує `target/` і `registry`, що економить хвилини на повторних запусках.

### Consequences
* Good, because transcript фіксує очікувану користь: `Swatinem/rust-cache@v2` — індустрійний стандарт для Rust CI; `components: rustfmt, clippy` встановлюються одним кроком.
* Bad, because `Swatinem/rust-cache@v2` — третя стороння action (додаткова довіра до зовнішнього постачальника).

## More Information
Template: `npm/rules/rust/policy/lint_rust_yml/template/lint-rust.yml.snippet.yml`. Rego: `rust.lint_rust_yml` перевіряє `uses: Swatinem/rust-cache@v2` у послідовності steps.

## ADR Рефакторинг `tauri` — `rust-lang.rust-analyzer` переноситься у правило `rust`

## Context and Problem Statement
`npm/rules/tauri/policy/vscode_extensions/vscode_extensions.rego` вимагав наявності `rust-lang.rust-analyzer` у `.vscode/extensions.json`. Після введення правила `rust` ця вимога дублюється: Tauri-проєкт завжди містить `src-tauri/Cargo.toml`, тому `rust`-правило теж активується й перевіряє `rust-analyzer`.

## Considered Options
* Перенести `rust-lang.rust-analyzer` у `rust.policy.vscode_extensions`, видалити з `tauri.policy.vscode_extensions`
* Залишити дублювання (обидва правила вимагають однакового запису)

## Decision Outcome
Chosen option: "Перенести у `rust`, видалити з `tauri`", because Tauri-проєкт завжди активує обидва правила; дублювання створює конфлікт власності та ускладнює майбутні зміни.

### Consequences
* Good, because `tauri.policy.vscode_extensions` залишається вузьким: лише `tauri-apps.tauri-vscode`; `rust.policy.vscode_extensions` стає єдиним джерелом правди для `rust-lang.rust-analyzer` + `tamasfe.even-better-toml`.
* Bad, because для Tauri-проєктів без `Cargo.toml` у FS-walk (нетиповий сценарій) `rust-analyzer` більше не перевірятиметься — але transcript визначив маркер `rust` саме через `Cargo.toml`, тому такий проєкт `rust` не активує.

## More Information
Змінювані файли: `npm/rules/tauri/policy/vscode_extensions/vscode_extensions.rego`, `npm/rules/tauri/policy/vscode_extensions/vscode_extensions_test.rego`, `npm/rules/tauri/tauri.mdc`. Спека: `docs/superpowers/specs/2026-05-23-rust-rule-design.md`, секція «Refactor `tauri.mdc`».
