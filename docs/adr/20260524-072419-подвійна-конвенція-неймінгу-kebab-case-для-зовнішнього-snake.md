## ADR Подвійна конвенція неймінгу: kebab-case для зовнішнього, snake_case для концернів і Rego

## Context and Problem Statement
У монорепо `npm/rules/` файли та директорії іменуються двома різними стилями — kebab-case і snake_case. Відсутність явного пояснення цього розподілу спричиняє враження хаотичного «різнобою». Необхідно зафіксувати, що подвійний стиль є навмисним і мотивованим технічними обмеженнями Rego.

## Considered Options
* Єдиний kebab-case для всіх файлів і директорій
* Єдиний snake_case для всіх файлів і директорій
* Подвійна конвенція: kebab-case для «зовнішнього» рівня, snake_case для концернів і Rego

## Decision Outcome
Chosen option: "Подвійна конвенція: kebab-case для «зовнішнього» рівня, snake_case для концернів і Rego", because Rego `package`-імена та ідентифікатори правил (`deny`, `allow`) не можуть містити дефіс — `package npm-module.package_json` є синтаксичною помилкою. Щоб ім'я `js/<concern>.mjs` і `policy/<concern>/<concern>.rego` збігалися 1:1 (для discovery і крос-посилань), концерни іменуються в snake_case. Усе «зовнішнє» (id правила, npm-пакет, CLI-скрипти, helpers) слідує стандартній kebab-case конвенції npm/Node.js.

### Конкретний розподіл (з `scripts.mdc:34`)

| Що | Стиль | Приклади |
|---|---|---|
| Каталог правила | kebab-case | `js-bun-db/`, `nginx-default-tpl/` |
| `.mdc` файл правила | kebab-case | `js-bun-db.mdc` |
| Концерн (`js/<concern>.mjs`, `policy/<concern>/`) | snake_case | `internal_urls.mjs`, `package_setup/` |
| Rego-пакети та тести | snake_case | `hasura_httproute.rego`, `hpa_pdb_test.rego` |
| Helpers у `utils/` | kebab-case з префіксом | `docker-mirror.mjs`, `vue-forbidden-imports.mjs` |
| Інфра-скрипти / тести | kebab-case | `walk-cache.test.mjs`, `auto-rules.mjs` |

### Consequences
* Good, because transcript фіксує очікувану користь: ім'я `js/<concern>.mjs` і `policy/<concern>/<concern>.rego` збігаються 1:1, що забезпечує автоматичний discovery без додаткового маппінгу.
* Good, because kebab-case для id правила відповідає полю `id` у `.n-cursor.json:rules`, зберігаючи консистентність конфігурації.
* Bad, because transcript не містить підтверджених негативних наслідків.
