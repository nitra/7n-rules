## ADR Вибір локальної моделі Tier 1 для docgen на 8 GB M2

## Context and Problem Statement
Необхідно обрати локальну LLM-модель через Ollama для автономної генерації технічної документації (Tier 1 pipline) на MacBook з 8 GB unified memory. Модель мусить генерувати читабельну українську документацію за розумний час без своппінгу.

## Considered Options
* gemma3:4b (3.3 GB, 100% GPU)
* qwen2.5-coder:3b (2.4 GB, 100% GPU)
* qwen2.5:7b (5.1 GB, частковий CPU-офлоад)
* qwen2.5-coder:7b (5.1 GB, частковий CPU-офлоад)
* llama3.1:8b (5.7 GB, частковий CPU-офлоад)
* qwen3:4b (3.6 GB, 100% GPU)
* gemma2:2b, llama3.2:3b, phi3.5 (1.6–4.0 GB, 100% GPU)
* gemma4:4b / batiai/gemma4-e4b:q4 (5.3 GB, 56%/44% CPU/GPU)

## Decision Outcome
Chosen option: "gemma3:4b", because об'єктивний бенчмарк (2 незалежних прогони на 3 різнопланових файлах з еталонами, складеними вручну) показав найкращий баланс якість/практичність: ~85% відповідності еталону при one-shot, 100% GPU, ~20 tok/s, ~14.5 год на 1042 файли, без ризику своппінгу.

### Consequences
* Good, because transcript фіксує очікувану користь: повністю влазить у 8 GB GPU-пам'ять (RAM 4.2 GB, 100% GPU), стала якість ~85–88% проти вручну складеного еталону, ~20 tok/s без деградації.
* Bad, because gemma4:4b (q4-перепак) дає ~92% якості за ті самі гроші — але ціною 56% CPU-офлоаду, 6.2 GB RAM та ~11 tok/s (~30 год на проєкт). Вибір gemma3 означає поступку ~4 п.п. якості заради ×2 швидкості та гарантованого GPU-запуску.

## More Information
- Бенчмарк: 3 файли (`npm/rules/abie/js/firebase_hosting.mjs`, `npm/rules/abie/lib/overlay-paths.mjs`, `npm/rules/abie/lib/k8s-tree.mjs`), чистий старт перед кожною моделлю, `ollama stop` між прогонами.
- Метрика якості: ручна рубрика 4×25% (точність, повнота поведінки, українська, формат/лаконічність) vs еталони, складені вручну перед прогоном.
- qwen3:4b відкинуто: ігнорує `/no_think`, думає російською, не дає вихід за бюджет.
- Моделі ≥5 GB (7–8B клас) відкинуто: CPU-офлоад, ризик своппінгу, якість не виправдовує часові витрати на 8 GB.
- Всі бенч-моделі прибрано після прогону; `gemma4:4b` лишено як alias `ollama cp batiai/gemma4-e4b:q4 gemma4:4b`.

## ADR System-prompt як головний важіль якості, а не вибір транспорту

## Context and Problem Statement
Під час порівняння «прямий ollama API» vs «через pi CLI» прямий виклик без system-prompt давав ~71% якості (витік stdlib, сигнатури, обрізання), а pi — ~87%. Потрібно було з'ясувати, чи є pi архітектурно кращим транспортом, чи це артефакт його вбудованого system-prompt.

## Considered Options
* Прямий ollama `/api/chat` без system-prompt
* pi CLI (вбудований coding-assistant system-prompt)
* Прямий ollama `/api/chat` з явним system-prompt

## Decision Outcome
Chosen option: "Прямий ollama `/api/chat` з явним system-prompt", because контрольований експеримент (3 файли × 3 варіанти) підтвердив: прямий + system-prompt досягає 85% (~рівень pi 87%), тоді як прямий без system — лише 71%. Різниця pi↔прямий+system = 2 п.п. (в межах шуму семплінгу). Справжній важіль — **наявність system-prompt**, а не інструмент.

### Consequences
* Good, because transcript фіксує очікувану користь: прямий виклик дає повний контроль над `num_ctx`/`num_predict` (усуває обрізання, яке завалило `overlay-paths` у варіанті A), без node-старту (~4 с/виклик у pi), без накопичення контексту між файлами.
* Bad, because pi залишається зручним, коли потрібні агентні можливості (read/write інструменти, сесії). Для one-shot docgen вони зайві, але якщо pipeline розшириться — pi легше.

## More Information
- Файли тесту: `firebase_hosting.mjs`, `overlay-paths.mjs`, `k8s-tree.mjs`.
- pi startup overhead виміряно: ~3.8–6.4 с/виклик; pi в RPC-режимі (~1.1 с boot) не підходить для docgen через накопичення контексту між незалежними файлами.
- Конфіг pi: `~/.pi/agent/models.json` з провайдером `ollama` (baseUrl `http://localhost:11434/v1`, `api: "openai-completions"`).
- System-prompt у фінальному рецепті: заборона ` ``` `-обгортки, заборона сигнатур/типів, заборона stdlib і приватних імен, вимога ловити крайові деталі (що пропускається, що не перевіряється, fail-safe).

## ADR JS-оркестратор з секційними промптами і persistent session як архітектура Tier 1 docgen

## Context and Problem Statement
One-shot LLM-виклик для генерації документації дає ~75–85% якості через типові помилки 4B-моделей: витік stdlib-імен, сигнатур, внутрішніх назв, галюцинації в гарантіях, обрізання для великих файлів. Потрібен спосіб підняти якість без переходу на важчу/повільнішу модель.

## Considered Options
* One-shot (один LLM-промпт на весь документ)
* JS-оркестратор: Stage 0 (детермінована екстракція факт-листа) + Stage 1 (секційні промпти) + Stage 3 (детермінована зборка), stateless виклики
* JS-оркестратор з persistent messages array (KV-cache reuse в межах одного файлу)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "JS-оркестратор з persistent messages array", because (а) секційна декомпозиція підняла якість gemma3:4b з ~75% до ~88% (+13 п.п.) завдяки заземленню фактів і вузьким задачам; (б) persistent messages array (incremental history у межах одного файлу) скоротив часовий оверхед з ×3.8 до ×1.14 vs one-shot — практично той самий час за суттєво вищу якість.

### Consequences
* Good, because transcript фіксує очікувану користь: gemma3:4b orchestrated (~88%) досягає якості gemma4:4b one-shot (~92%) за вдвічі менший час (~14 год проти ~30 год); JS володіє структурою й фактами (API-імена, skip-маркери, stdlib-класифікація) — модель не може вигадати назву або всунути сигнатуру.
* Bad, because конвеєр складніший за one-shot: 3 модулі (`docgen-extract.mjs`, `docgen-prompts.mjs`, `docgen-gen.mjs`), додаткова точка відмови при JS-парсингу. Stage 0 прив'язаний до JS/MJS — для `.vue`/`.py` потрібен окремий екстрактор або fallback до one-shot.

## More Information
- Реалізація: `.worktrees/feat-docgen-orchestrator-pi/npm/skills/docgen/js/` — `docgen-extract.mjs`, `docgen-prompts.mjs`, `docgen-gen.mjs`.
- `docgen-gen.mjs` API: `generateDoc(file, { model, mode })`, `mode ∈ {'orchestrated','oneshot'}`.
- Persistent session: `messages` array розширюється після кожної секції (`role: 'assistant'` response + новий `role: 'user'` prompt) — одна сесія на файл, між файлами сесія закривається.
- KV-cache: спільний `system`-prefix (містить код файлу) незмінний між секціями → ollama кешує інгест коду; оплачується один раз, а не 4×.
- Fallback: якщо `facts.unsupported === true` (не-JS файл або parse-error) — деградація до one-shot автоматично.
- Бенч: `firebase_hosting` orch2 28 с vs one-shot 18 с (×1.56); `overlay-paths` 57 с vs 47 с (×1.21); `k8s-tree` 59 с vs 62 с (×0.95 — orchestrated швидше).
