## ADR Перегрупування міграції умовних правил по механізму (Група 1 vs Група 2)

## Context and Problem Statement
Потрібно мігрувати 6 умовних правил (`docker`, `python`, `k8s`, `image-compress`, `php`, `rust`) з `bun run lint-*` на `n-cursor lint <rule>`. Початковий план фазування A (готові) → B (адаптер) → C (нова логіка) виявився хибним: docker формально відносився до фази A, але його повна міграція вимагає рефактора `bun/js/layout.mjs:checkCursorRuleScripts`, який є спільним для docker/k8s/image.

## Considered Options
* Фазування A→B→C (за наявністю `js/lint.mjs`)
* Перегрупування за спільним механізмом: Група 1 (docker+k8s+image, спільний `bun layout` рефактор) і Група 2 (php+rust, незалежні CI)

## Decision Outcome
Chosen option: "Перегрупування за спільним механізмом", because часткова docker-міграція без рефактора `checkCursorRuleScripts` залишала б active-consumer'ів у напівзламаному стані (workflow кличе `bun run lint-docker`, а `package_json` вже видалено).

### Consequences
* Good, because Група 2 (python/php/rust) закомічена й запушена в `main` як самодостатній проміжний результат; рефактор bun layout не розпорошений по частинах.
* Bad, because docker/k8s/image лишились незмігрованими в цій сесії — Група 1 відкладена на окрему задачу.

## More Information
- Спільний механізм: `npm/rules/bun/js/layout.mjs:checkCursorRuleScripts` + `RULE_SCRIPTS` (масив `[{rules: ['docker'], script: 'lint-docker', doc: 'docker.mdc'}, ...]`)

## ADR Rust CI лишається cargo-direct після міграції

## Context and Problem Statement
При міграції правила `rust` на `n-cursor lint rust` виникло питання: чи мігрувати CI-воркфлоу (`lint_rust_yml`) так само, як python/php (замінити `bun run lint-rust` на `n-cursor lint rust --read-only`).

## Considered Options
* Мігрувати CI: `n-cursor lint rust --read-only` у lint-rust.yml
* Залишити CI cargo-direct: `cargo fmt --check` + `cargo clippy` без n-cursor

## Decision Outcome
Chosen option: "Залишити CI cargo-direct", because CI-воркфлоу rust вже використовує cargo безпосередньо (не через `bun run lint-rust`) — `lint_rust_yml` template явно запускає `cargo fmt --check` і `cargo clippy` окремими кроками через rustup toolchain. Це навмисне рішення (toolchain-pinning через rustup, недоступний через n-cursor).

### Consequences
* Good, because transcript підтверджує: `lint_rust_yml` conftest 6/6 pass без жодних змін; rust CI залишається стабільним.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- CI-воркфлоу: `npm/rules/rust/policy/lint_rust_yml/`
- Новий адаптер (лише для локального `n-cursor lint rust`): `npm/rules/rust/js/lint.mjs` — викликає `cargo fmt --all` (fix-режим) або `cargo fmt --check` (readOnly) + `cargo clippy`
- Видалено: `npm/rules/rust/policy/package_json/`

## ADR Канонічний інтерфейс оркестраторного адаптера `js/lint.mjs`

## Context and Problem Statement
Кожне правило потребує адаптерного файлу `js/lint.mjs`, щоб `n-cursor lint <rule>` міг викликати лінтер-фазу уніфіковано. Python вже мав такий адаптер; php і rust були створені в цій сесії з нуля.

## Considered Options
* Канон `export function lint(files, cwd, {readOnly})` — делегує в `lint/lint.mjs`
* Окрема сигнатура на кожне правило

## Decision Outcome
Chosen option: "Канон `export function lint(files, cwd, {readOnly})`", because python (еталон) вже використовує цей інтерфейс; php і rust реалізовані за тим самим патерном: `readOnly=true` → read-only режим (dry-run/check), `readOnly=false` → fix-режим.

### Consequences
* Good, because transcript фіксує очікувану користь: усі три адаптери (python/php/rust) пройшли conftest + vitest + integration без регресій.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Адаптери: `npm/rules/php/js/lint.mjs` (делегує в `lint/lint.mjs:run`), `npm/rules/rust/js/lint.mjs` (cargo-команди напряму)
- Python-еталон: `npm/rules/python/js/lint.mjs`
- Документація-супровід: `js/docs/lint.md` (crc-stamp) + оновлений `js/docs/index.md` для кожного правила
