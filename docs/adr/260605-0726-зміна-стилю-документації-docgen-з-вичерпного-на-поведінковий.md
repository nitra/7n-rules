## ADR Зміна стилю документації docgen з «вичерпного» на «поведінковий лаконічний»

## Context and Problem Statement
Скіл `docgen` генерував документацію, що змішувала бізнес-поведінку з імплементаційними деталями: перелік stdlib-залежностей (`node:fs`, `node:fs/promises`, `node:path`), таблиці типів, сигнатури внутрішніх функцій, `Rebuild Test`-переліки. Це призводило до надлишкових і швидко застарілих `.md`-файлів, які не описували задачу, що вирішує файл.

## Considered Options
* Залишити поточний «вичерпний» стиль (перелік залежностей, сигнатури, Rebuild Test)
* Перейти на «поведінковий лаконічний» стиль (Огляд / Поведінка / Публічний API лише для нетривіальних поверхонь / Де використовується / Гарантії поведінки)

## Decision Outcome
Chosen option: "Поведінковий лаконічний стиль", because ручно написаний еталон для `enabled.mjs` отримав схвалення користувача; він описує opt-in-семантику, покроковий алгоритм і fail-safe-інваріанти без прив'язки до `node:fs` чи імені `isAbieRuleEnabled`.

### Consequences
* Good, because документи описують поведінку та гарантії, що не старіють разом зі змінами реалізації.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Змінено три файли: `npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md` (обидва — промпт Tier 1 і слово «вичерпну» → «лаконічну поведінкову»); `.pi/skills/n-docgen/SKILL.md` (лише опис — стаб без промпту). Tier-1-промпт у секції «Секції (включай лише доречні)» тепер явно забороняє `## Залежності`, таблиці типів, і вимагає `## Гарантії поведінки` замість `## Помилки`.

## ADR Вибір локальної моделі для Tier 1 docgen на 8 GB M2: `gemma3:4b`

## Context and Problem Statement
Tier 1 docgen (1042+ незалежних файлів → хмара) коштує ~26 M токенів. Постало питання: чи можна замінити хмарних Claude-субагентів локальною моделлю через Ollama, щоб знизити витрати до нуля.

## Considered Options
* `gemma4:e4b` (9.6 GB) — перевищує 8 GB RAM → своп → 0.4 tok/s (виміряно)
* `gemma3:4b` (3.3 GB) — повністю в GPU, 14–21 tok/s (виміряно), якість ~86% vs еталон
* `qwen2.5-coder:3b` (2.4 GB) — 28–36 tok/s, якість ~77%, мовні помилки, витік сигнатур/stdlib
* `qwen2.5:7b`, `qwen2.5-coder:7b`, `llama3.1:8b` — частковий CPU-офлоад (87–91% GPU), повільніші, ризик свопу при великому контексті
* `llama3.2:3b`, `gemma2:2b`, `phi3.5` — слабка або покручена українська
* `qwen3:4b` — reasoning-модель, ігнорує `/no_think`, дає вихід рос. мовою

## Decision Outcome
Chosen option: "`gemma3:4b`", because єдина модель у списку, що водночас повністю влазить у 8 GB RAM (100% GPU, 4.2 GB resident) і дає найкращу українську серед моделей, що влазять (~86% vs еталон).

### Consequences
* Good, because transcript фіксує очікувану користь: Tier 1 прогін ~14–22 год замість ~37 діб (`gemma4:e4b`) або ~9–16 год із гіршою якістю (`qwen2.5-coder:3b`); вартість ~$0.
* Bad, because на 8 GB паралелізм неможливий (один інстанс насичує GPU/RAM), тобто `concurrency = 1` і час — суворо послідовний; якість 86% потребує ручного або Claude-рев'ю.

## More Information
Бенчмарк: замір `ollama ps SIZE` + `processor split` на чистому старті кожної моделі; 3 тестові файли (`firebase_hosting.mjs`, `overlay-paths.mjs`, `k8s-tree.mjs`); оцінка 4×25% (точність, повнота, українська, формат). `gemma4:e4b` (9.6 GB) показала 0.4 tok/s через memory-thrashing. Результати у `/tmp/docgen-bench2/` (видалено після сесії).

## ADR System-prompt як основний важіль якості при зверненні до Ollama

## Context and Problem Statement
Порівнюючи два способи доступу до `gemma3:4b` — прямий `ollama /api/generate` без system та через `pi` CLI — виявилась стабільна різниця якості (+14–16 п.п. на користь pi). Постало питання, чи варто вбудовувати pi як обов'язкову ланку для Tier 1.

## Considered Options
* A — прямий `POST /api/generate` без system-prompt
* B — через `pi --provider ollama --model gemma3:4b` (pi додає вбудований coding-assistant system-prompt)
* C — прямий `POST /api/chat` з явним `system`-повідомленням (та сама поведінкова інструкція)

## Decision Outcome
Chosen option: "Прямий `/api/chat` з явним system-prompt (варіант C)", because після додавання system-role прямий ollama досяг рівня pi (~85 vs 87% vs еталон), без node-оверхеду pi і з повним контролем `num_ctx`/`num_predict`.

### Consequences
* Good, because transcript фіксує: варіант C усунув обрізання виводу (проблему A на `overlay-paths`) і витік реалізації (regex, `fs`/`path`, сигнатури) — завдяки явному system-prompt, а не завдяки pi.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Підтверджено контрольованим експериментом: 3 файли × 3 способи, тепла модель, однаковий user-промпт. pi роутиться через `~/.pi/agent/models.json` з `baseUrl: http://localhost:11434/v1`, `api: openai-completions`. Для Tier 1 CLI-команди достатньо `/api/chat` з двома повідомленнями (system + user); pi залишається доцільним лише за потреби в агентних інструментах (read/write, сесії). Результати підтвердження у `~/docgen-bench3/` (залишено після сесії).
