## ADR Видалення надлишкових CLI-точок входу n-cursor

## Context and Problem Statement
Команда `@nitra/cursor` накопичила дві публічні CLI-підкоманди (`lint-ci` і `doc-files <sub>`), які стали мертвим кодом: перша — чистий аліас `lint --read-only --full` без жодного живого callera у CI чи `package.json`; друга — deprecated-аліас `scan|check|gen|stamp`, де hook і skill вже перейшли на `lint-doc-files`/`fix-doc-files`. Мета — мінімальна поверхня CLI.

## Considered Options
* Лишити аліаси до запланованого major-релізу
* Знести обидва аліаси зараз (breaking → major bump)

## Decision Outcome
Chosen option: "Знести обидва аліаси зараз", because аліаси мають нуль живих callerів у реальних workflow, `.github`, root-`package.json`, а тримати deprecated-шар суперечить цілі мінімальної поверхні.

### Consequences
* Good, because transcript фіксує очікувану користь: CLI зменшено до двох семантично різних команд (`lint-doc-files` — hook/detect, `fix-doc-files` — generation), без dead-code.
* Bad, because зміна breaking — потребує major-버전 (`npm/.changes/260615-0638.md`: `bump: major, section: Removed`).

## More Information
Видалено: `case 'lint-ci'` і `case 'doc-files'` з `npm/bin/n-cursor.js`; рядки у шапці-довідці й у `default`-помилці; згадки у ROOT_GUARDED-коментарі. ## ADR Opportunistic LLM-fix tier у lint-правилах (doc-files як референс)

## Context and Problem Statement
Lint-крок `doc-files` (`npm/rules/doc-files/js/lint.mjs`) виявляв застарілі доки, але фізично не міг їх полагодити — генерація потребує локальної LLM (omlx), яку заборонено у дешевому/детермінованому lint-циклі. Виникла асиметрія: всі інші правила в fix-by-default реально правлять (oxlint `--fix`), а doc-files — лише детект + делегування.

## Considered Options
* Лишити lint-крок detect-only, фікс виключно через `fix-doc-files`
* Opportunistic-gen у lint-кроці: omlx up → генерує; omlx down → skip+exit 1

## Decision Outcome
Chosen option: "Opportunistic-gen у lint-кроці", because omlx-down → skip+exit 1 зберігає гейт (ніякого false-green), а guard «health-check лише коли є stale» убезпечує від зайвих omlx-пробоїв на чистому дереві.

### Consequences
* Good, because transcript фіксує очікувану користь: в fix-by-default lint-крок тепер сам генерує та re-детектить, а `fix-doc-files` лишається лише для явних/scoped прогонів.
* Bad, because lint-крок став side-effecting (тягне omlx-машинерію); тести детектора втратили герметичність — вимагають `{readOnly:true}` + mock генерації.

## More Information
Контракт: `lint(files, cwd, {readOnly})` — `readOnly:true` → detect-only (CI); `readOnly:false` + omlx up → `runGenerationBatch` → re-detect; omlx down → друкує причину + exit 1. Спільне ядро: `preflightProblem` і `runGenerationBatch` витягнуто в exports `npm/rules/doc-files/js/docgen-files-batch.mjs`. Прапор `llmFix: true` у `npm/rules/doc-files/meta.json`; схема `rule-meta.json` розширена. Тести переписано: `npm/rules/doc-files/js/tests/lint.test.mjs`. Спека: `docs/specs/2026-06-15-opportunistic-llm-fix-tier.md`. Changeset: `npm/.changes/260615-0907.md` (`bump: minor, section: Changed`).

## ADR cspell-fix: класифікація-до-словника замість whole-file apply

## Context and Problem Statement
Наявний `npm/rules/text/lint/cspell-fix.mjs` використовував `llmLintFix` — підхід «whole-file rewrite» (модель повертає весь файл як JSON). Емпіричний експеримент на репо (1 406 знахідок cspell / 292 файли) показав: ~90% «Unknown word» — валідні українські/технічні слова (кандидати у словник), а не одруки. Whole-file rewrite на файлах >3 k токенів давав parse-error (відповідь обрізана) або HTTP timeout 120 с — тобто код не працює на реальних файлах репо.

## Considered Options
* (a) Залишити `llmLintFix` whole-file apply, додати preflight і circuit-breaker
* (b) Classify→dict: модель класифікує слова (valid|typo), валідні авто-дописуються у `.cspell.json`, typo — список на рев'ю
* (c) Лишити cspell detect-only (без LLM)

## Decision Outcome
Chosen option: "(b) Classify→dict", because: (1) whole-file output unbounded → timeout, виключає (a); (2) 90% знахідок — словникові кандидати, а не одруки, тому classify→dict влучає в реальну природу проблеми; (3) пропозиції (не мутації) безпечні — шкідлива класифікація відхиляється в рев'ю diff.

### Consequences
* Good, because transcript фіксує очікувану користь: bounded output (classify повертає компактний JSON незалежно від розміру файлу), нуль timeout; валідні слова авто-ростуть у `.cspell.json` — реальний ремедіейшн для цього репо.
* Bad, because одруки не фіксуються авто (лише список на рев'ю); `llmLintFix` для cspell повністю замінено — якщо хтось покладався на apply-поведінку, треба явна міграція.

## More Information
Ключовий інваріант (з D3-дискусії): стратегія LLM-fix мусить давати **bounded output**; «apply через перепис усього входу» — заборонений анти-патерн. Нові публічні exports: `unknownWords(output)` (замість `groupFindingsByFile`), `appendWordsToDict(words, cfgPath)`, `runCspellText(files, cwd, {readOnly})`. Словник: `words[]` у `.cspell.json` (корінь репо), sorted+dedup. Scope: quick-only (змінені файли). Файли: `npm/rules/text/lint/cspell-fix.mjs`, `npm/rules/text/lint/tests/cspell-fix.test.mjs`. ESLint: 0 errors, 0 warnings після виправлення `Array#toSorted()`, ReDoS-regex, nested template.
