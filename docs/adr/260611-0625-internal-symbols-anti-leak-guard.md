# ADR: `internalSymbols` як whitelist-based anti-hallucination у LLM-pipeline

**Status:** Accepted
**Date:** 2026-06-11

## Context and Problem Statement

При генерації документації файлів (`doc-files`) LLM мав тенденцію описувати у публічному документі внутрішні деталі — функції та константи, що імпортуються з інших модулів і є деталями реалізації, а не публічним контрактом. Традиційні способи боротьби (стиль-інструкція в промпті) дають загальне правило, але не конкретний список заборонених слів для цього файлу.

## Considered Options

* Статично витягувати список імпортованих символів із внутрішніх залежностей і штрафувати скорер за їхній витік у документ.
* Обмежитись загальним правилом у system-промпті («не описуй внутрішні деталі»).
* Не перевіряти — залишити на розсуд моделі.

## Decision Outcome

Chosen option: "Статичний список `internalSymbols` + штраф скорера", because список утворюється детерміновано: `extractFacts` класифікує імпорти на stdlib / npm / внутрішні і збирає символи з внутрішніх; ці символи є деталями реалізації модуля B, якщо B їх імпортує з A, — вони не повинні з'являтися в публічній доці B. Скорер штрафує за кожне знайдене `internal-name:<symbol>`. Це персоналізований anti-hallucination фільтр для кожного файлу.

### Consequences

* Good, because штраф точковий: «витік `callOmlx` у доку `llm.mjs`» є конкретним дефектом, а не загальним «документ надто технічний».
* Good, because список формується без LLM і без токенів — чистий статичний аналіз.
* Good, because патерн масштабується на будь-який LLM-pipeline де важлива інкапсуляція: замовник → виконавець, інфраструктура → бізнес-логіка.
* Bad, because список захоплює лише прямі імпорти; транзитивні internal-символи (A → B → C, у доці C з'являється щось з A) не відстежуються.
