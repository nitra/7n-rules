## ADR Вибір моделі та транспорту для локального Tier 1 docgen

## Context and Problem Statement
Потрібно вибрати оптимальну модель і спосіб виклику для локального Tier 1 (1042 файли) docgen на машині Apple M2 8 GB RAM, де єдиними доступними моделями є `gemma3:4b` (3.3 GB) і `gemma4:4b` (5.3 GB), а транспортні варіанти — прямий HTTP до `ollama /api/chat` і `pi` CLI.

## Considered Options
* `gemma3:4b` + прямий `/api/chat` + system-prompt
* `gemma3:4b` через `pi` per-file + `--append-system-prompt`
* `gemma4:4b` + прямий `/api/chat` + system-prompt
* `gemma4:4b` через `pi` per-file + `--append-system-prompt`

## Decision Outcome
Chosen option: "`gemma3:4b` + прямий `/api/chat` + system-prompt", because модель повністю вміщається у GPU (4.2 GB, 100% GPU), дає ~20 tok/s і стабільні ~53 с/файл, якість 85% vs еталону — передбачувано і без memory thrash.

### Consequences
* Good, because transcript фіксує очікувану користь: ~15 год на 1042 файли ($0), 100% GPU без своп, стабільна якість.
* Bad, because якість 85% — нижча за `gemma4:4b` (89%); деякі виходи потребують post-process (зрізання fence, виправлення мовних дефектів).

## More Information
- Бенчмарк-файли: `~/docgen-bench3/`, `~/docgen-bench3/out/`
- `gemma4:4b` (5.3 GB) через pi зависає на довгих контекстах через memory pressure (8 GB = 6.2 GB model + pi node process → своп)
- Ключова причина зависання pi + gemma4: `--append-system-prompt` з бектіками (`\`\`\``) або memory thrash при 5.3 GB + node
