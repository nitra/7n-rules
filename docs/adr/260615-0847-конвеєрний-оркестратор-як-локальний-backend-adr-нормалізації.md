## ADR Конвеєрний оркестратор як локальний backend ADR-нормалізації

## Context and Problem Statement
Поточний механізм нормалізації (`normalize-decisions.sh`) надсилає всі чернетки батчу (~10 драфтів + ~294 імені clean-файлів) як єдиний single-shot промпт. На малій локальній моделі `omlx/gemma-4-e4b-it-OptiQ-4bit` це призводило до `omlx curl exit 18` (закриття з'єднання через переповнення) або зависання >5 хвилин, тобто повної неможливості запуску локально. Хмарна генерація через `claude sonnet` (cloud-gold) коштувала ~14 хвилин та розкривала внутрішні шляхи й архітектуру у чернетках назовні.

## Considered Options
* **Single-shot local** — той самий 32k-токенний промпт перенаправлений на локальну модель (naive-local arm)
* **Pipeline-оркестратор** — декомпозиція: JS-retrieval → per-edge бінарне судження (LLM) → union-find кластери (JS) → gen-MADR на anchor (LLM) → детермінований validation-гейт (JS)

## Decision Outcome
Chosen option: "Pipeline-оркестратор", because naive-local виявився нежиттєздатним (infra-крах на першій спробі, hang >5 хв на другій), тоді як новий конвеєр пройшов A/B-тест з near-gold якістю (7/7 MADR валідних, ідеальне покриття, 0 хмарних викликів, 0 крахів) на тих самих 10 фікстурах.

### Consequences
* Good, because transcript фіксує очікувану користь: повна приватність (чернетки з `/Users/vitaliytv/…` та внутрішньою архітектурою не залишають машину), $0 токенів, 0 cloud-залежності; прозова якість reformat ≈ sonnet (gemma сильна у constrained-reformat, бо capture вже пише MADR-структуру).
* Bad, because transcript фіксує: wall-clock ~17 хв на 37 серійних omlx-викликів (gemma на одному сервері не паралелиться); один edge-суддя over-merge-нув docgen-пару (defensible, але відхилення від cloud-gold).
