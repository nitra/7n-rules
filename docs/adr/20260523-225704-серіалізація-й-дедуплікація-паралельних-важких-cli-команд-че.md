## ADR Серіалізація й дедуплікація паралельних важких CLI-команд через `withLock`

## Context and Problem Statement

Проєкт запускає кілька агентів паралельно; кожен може самостійно тригерити важкі CLI-команди (`lint-ga`, `lint-rego`, `lint-text`, `lint-k8s`, `lint-docker`, `fix <rule>`). Паралельний запуск декількох `eslint` / `hadolint` / `kubeconform` перевантажує диск і CPU. Крім того, якщо N агентів лінтять той самий незмінений стан git-дерева, повторний прогін витрачає ресурси без будь-якого нового результату.

## Considered Options

* Нова команда-обгортка `n-cursor guard <key> -- <команда>` — агент явно викликає обгортку.
* Вбудований лок безпосередньо у реалізаційний код кожної команди — агент продовжує запускати `bun run lint` / `npx @nitra/cursor lint-ga` як завжди.
* Лок у `runFixCommand` на рівні диспетчера `n-cursor.js`.
* Лок у `runStandardRule` — спільна точка входу для всіх `fix-<ruleId>`.
* Спільна точка входу `runStandardLint(lintDir, stepsFn)` — дзеркально до `runStandardRule`.

## Decision Outcome

Chosen option: "Вбудований лок через `runStandardLint` і `runStandardRule`", because користувач визначив, що агенти не повинні змінювати команду запуску; лок має бути інтринсивним — а після реалізації прямого `withLock` у кожному `lint.mjs` окремо, архітектурна підказка усунула дублювання: замість 5 прямих викликів `withLock(...)` у 5 файлах — дві спільні точки входу (`runStandardLint` для лінтів, `runStandardRule` для `fix`), де ключ лок-файлу виводиться зі шляху `import.meta.dirname`.

Механізм примітиву: `mkdirSync`-based lock (атомарний на APFS), власник у `owner.json` `{pid, host, startedAt, fingerprint}`, виявлення застарілого локу через `process.kill(pid, 0)` + max-age 30 хв. Стан — `node_modules/.cache/n-cursor/<key>/` (gitignored). Дедуп — тільки для `exitCode === 0` + збіг sha256-fingerprint + TTL 10 хв. Fingerprint = sha256 від `git rev-parse HEAD` + `git diff HEAD` + untracked-файли через `git ls-files --others -z` (NUL-розділення, щоб не ламати не-ASCII імена).

### Consequences

* Good, because transcript фіксує очікувану користь: паралельні агенти серіалізуються (`🔒 lint-rego: лок взято`), повторний прогін на тому самому git-дереві пропускається (`♻️ lint-rego: те саме дерево, 2с тому — пропускаю`).
* Good, because майбутні cross-cutting зміни (телеметрія, dry-run, env-toggle вимкнення) вносяться в одне місце (`run-standard-lint.mjs` або `run-standard-rule.mjs`), а не в 30+ `fix.mjs` / 5 `lint.mjs`.
* Bad, because `runStandardRule` викликається і з `rules/<id>/fix.mjs` (bun-процес), і з `run(ctx)`-композиції — при spawning кожного правила як окремого процесу lock-ключ `fix-<ruleId>` захоплюється в дочірньому процесі, що є очікуваною поведінкою, але це необхідно пам'ятати при написанні майбутніх тестів.
