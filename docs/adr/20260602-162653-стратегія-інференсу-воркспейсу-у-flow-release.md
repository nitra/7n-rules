## ADR Стратегія інференсу воркспейсу у `flow release`

## Context and Problem Statement
При виклику `flow release` без явного `--ws` change-файл лягав у корінь монорепо (`.changes/`), а не у відповідний subworkspace. Потрібно автоматично визначати змінений воркспейс із git diff відносно `base_commit` зі стану flow.

## Considered Options
* Авто-інференс одного зміненого subworkspace (ін'єктовані `listWorkspaces` + `collectChangedFilesSince`), fail-hard при кількох збігах
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Авто-інференс одного зміненого subworkspace", because це закриває беклог #9: якщо рівно один subworkspace має змінені файли — додавати `--ws <dir>` у `change` автоматично; нуль або кілька → fall-through (без `--ws`, поведінка `change` по дефолту або помилка відповідно).

### Consequences
* Good, because transcript фіксує очікувану користь: change-файл тепер опиняється у правильному `<ws>/.changes/`, а не в корені монорепо.
* Bad, because при кількох змінених subworkspace `release` завершується з `exit 1` і вимагає явного `--ws` — але transcript фіксує це як свідому поведінку, не баг.

## More Information
- Реалізовано у `npm/scripts/dispatcher/lib/commands.mjs` — функції `matchChangedWorkspaces` (export) і `resolveChangeWsArgs`.
- `--ws=value` (inline-форма) виявляє присутність через `.some(a => a === '--ws' || a.startsWith('--ws='))` — виправлено після review-finding про `--ws=`.
- Вкладені воркспейси (`apps/` + `apps/web/`) розвʼязуються через `toSorted` за depth-desc — кожен файл відноситься до найглибшого збігу.
- Fail-soft при недосяжному `base` або помилці `listWorkspaces` — `change` запускається без `--ws`.
- 27 тестів у `commands.test.mjs`; коміт `282332c` на `origin/flow-release-infer-ws`, злито в `main` (`ca3bf57`).

## ADR Промпт рецензента верифікує cross-file твердження через Read

## Context and Problem Statement
Рецензент `flow review` отримував findings типу «з diff не видно» — нефальсифіковні твердження про cross-file стан, які генерували шум. При цьому runner вже надавав рецензентові інструмент `Read` і `cwd` воркспейсу, але промпт цього не використовував.

## Considered Options
* Оновити промпт `reviewerPrompt` — дозволити й зобов'язати дочитувати referenced-файли через `Read` для верифікації cross-file тверджень
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Оновити промпт reviewerPrompt", because рецензент вже має `Read` у allowed-tools, але промпт не давав йому дозволу/інструкції ним користуватись — зміна лише в тексті промпта, без зміни конфігурації runner-а.

### Consequences
* Good, because transcript фіксує очікувану користь: рецензент перевіряє cross-file твердження перед репортом, а не видає нефальсифіковний шум.

## More Information

- Тести у `npm/scripts/dispatcher/lib/tests/review.test.mjs` — 14 тестів (зокрема `READ_ACCESS_RE`, `VERIFY_RE`); oxlint чистий.

## ADR Запуск Stryker через локальний core-bin замість npx

## Context and Problem Statement
`flow verify` (coverage-gate) падав із помилкою `Cannot find TestRunner plugin "vitest"`: `runStryker` запускав core через `npx @stryker-mutator/core`, що тягнуло його у npx-кеш без сусіднього `vitest-runner` → plugin-discovery Stryker не знаходив плагін.

## Considered Options
* Запускати локально встановлений `@stryker-mutator/core` напряму (через node-shebang bin), резолвлячи шлях через `package.json` пакета
* Продовжувати через `npx` (поточний стан — не працює)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Запускати локально встановлений core напряму", because `vitest-runner` встановлений поруч у тому ж `node_modules/@stryker-mutator/`, тоді як `npx` ізолює core у власний кеш без плагінів. Bin-файл має `#!/usr/bin/env node` shebang → завжди виконується node незалежно від bun/node батька.

### Consequences
* Good, because transcript фіксує валідацію e2e (двічі): `coverage --changed` завантажує «vitest test runner with "perTest" coverage analysis», initial test run succeeded, exit 0.
* Bad, because фікс прив'язаний до локально встановленого core — якщо `@stryker-mutator/core` не встановлено як devDependency, резолвер повернеться до `npx` (fail-soft з логом попередження).

## More Information
- Змінено `npm/rules/js-lint/coverage/coverage.mjs` — функція `resolveLocalStrykerBin`: резолв через `createRequire(import.meta.url).resolve('@stryker-mutator/core/package.json')` → `dirname` → `bin.stryker`; `spawnSync(strykerBin, ...)` замість `spawnSync('npx', ['@stryker-mutator/core', ...])`.
- Перша спроба (`require.resolve('@stryker-mutator/core/bin/stryker.js')`) зазнала `ERR_PACKAGE_PATH_NOT_EXPORTED` — `exports` пакета не відкриває `./bin/stryker.js`; виправлено через `package.json` → `bin`.
- `process.execPath` (запропонований рецензентом) відхилено: під bun вказує на bun-bin, а не node — shebang надійніший.
- 34 тести у `npm/rules/js-lint/coverage/tests/coverage.test.mjs`; oxlint чистий.
- Коміт `373ce42` на `origin/flow-coverage-stryker-local`, злито в `main` (`9286833`).
