title: K8s-середовище та UI для task-графа
description: Task-граф виконується у Kubernetes, а UI читає стан через API/REST/SSE замість прямого монтування `tasks/`.
---

**Status:** Accepted
**Date:** 2026-06-07

## Context and Problem Statement

Після введення файлової структури `tasks/` потрібно визначити середовище виконання задач і спосіб перегляду стану графа. Користувач окреслив вимогу, що задачі будуть жити в Kubernetes, а для перегляду стану потрібен окремий UI.

## Considered Options

- Dev pod, що монтує той самий PVC, що й worker-поди.
- `kubectl port-forward` + SSH у поді як тимчасовий доступ.
- Tailscale / WireGuard у поді як постійна мережа.
- UI читає стан через `graph scan --json` через REST або SSE.
- Пряме монтування `tasks/` для UI.

## Decision Outcome

Chosen option: "Dev pod з тим самим PVC для розробницького доступу та UI через REST/SSE", because worker-поди і dev pod можуть бачити спільний `tasks-pvc`, а UI не потребує локального монтування `tasks/` і може отримувати стан через API-сервер у Kubernetes.

### Consequences

- Good, because `tasks-pvc` одночасно доступний worker-подам і dev pod, тож Zed remote може бачити живий файловий стан задач.
- Good, because UI не залежить від Zed remote або прямого доступу до файлової системи; достатньо HTTP/SSE до API.
- Bad, because transcript не містить підтвердження негативних наслідків.
- Neutral, because остаточна назва UI-проєкту в transcript не підтверджена.

## More Information

Запропонована команда доступу для dev pod: `kubectl port-forward pod/n-graph-dev 2222:22`.

Dev pod потребує SSH-сервера та встановленого `zed --headless`.

Діаграма з transcript: `tasks-pvc ←── n-cursor-graph (worker pods)` і `tasks-pvc ←── dev-pod (SSH + zed)`.

UI читає стан через `graph scan --json` за REST або SSE. Запропоновані назви UI-проєкту: `n-graph`, `graphwatch`, `taskflow`; фінальне підтвердження назви в transcript не зафіксоване.

## Update 2026-06-07

Додати принцип доступу до Kubernetes dev-середовища без видачі `kubectl` розробникам:

- `kubectl port-forward` як прямий шлях відхилено, бо він вимагає Kubernetes credentials у розробника.
- Доступ має проходити через SSH gateway / bastion з backend-контролем авторизації.
- Розглянуті реалізації: Teleport або власний gateway.
- Dev pod монтує `tasks-pvc`; backend визначає, хто і з якими правами може отримати SSH/Zed remote доступ.
- Вибір між Teleport і власною реалізацією в transcript не зафіксований.
