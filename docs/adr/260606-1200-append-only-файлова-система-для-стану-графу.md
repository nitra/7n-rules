## ADR Append-only файлова система для стану графу задач

**Status:** Accepted
**Date:** 2026-06-06

## Context and Problem Statement

Для зберігання стану вузлів у Рекурсивному складеному ОАГ розглядалась мутабельна JSON-схема де файл `meta.json` перезаписувався при кожній зміні стану. Це створювало дві проблеми: (1) відсутність атомарності — агент міг записати `outputs.json` але впасти до виклику CLI-хука, залишаючи граф у невалідному стані; (2) конфлікти при злитті git-ворктрі — два агенти могли змінити один файл.

## Considered Options

* Мутабельні файли — стан зберігається в одному файлі, перезаписується
* Append-only підхід — файли лише створюються або видаляються, ніколи не модифікуються

## Decision Outcome

Chosen option: "Append-only підхід", because `create file` є атомарною операцією на POSIX-системах, а naming convention ворктрі `<ідВузла>-виконання` стає природним мютексом через `git worktree add`, який атомарно провалюється якщо директорія вже існує.

### Consequences

* Good, because стан вузла визначається наявністю файлів-сентинелів: поява `outputs.md` = сигнал "вирішено" без окремого CLI-хука для зміни стану.
* Good, because конфлікти при злитті ворктрі неможливі за визначенням: кожен агент лише створює нові файли з унікальними іменами.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Файлова структура стану вузла:
```
tasks/<ідВузла>/
  meta.md          ← створюється при spawn, незмінний
  inputs.md        ← створюється при spawn, незмінний
  виконується      ← порожній sentinel; наявність = агент активний
  outputs.md       ← наявність = вирішено
  error.md         ← наявність = помилка
  знедійснений     ← наявність = invalidated
  patches/
    001-<timestamp>.md   ← новий файл на кожен патч інженера
```

Переходи стану:
- `spawn` → створити `meta.md + inputs.md`
- `start` → створити `виконується`
- `done` → видалити `виконується` + створити `outputs.md` (через `write tmp → fsync → rename`)
- `failed` → видалити `виконується` + створити `error.md`

Агент перевіряє наявність `знедійснений` перед записом `outputs.md` і зупиняється якщо файл знайдений.

TOCTOU race condition вирішується через `git worktree add .worktrees/<ідВузла>-виконання/` — ця команда атомарно провалюється якщо ворктрі вже існує; другий планувальник отримує помилку і пропускає вузол.

## Update 2026-06-06

Уточнено LLM-first формат файлів графу:

- Основний формат — Markdown з YAML-фронтматером, а не JSON.
- YAML-фронтматер містить машинозчитувані поля; тіло Markdown лишається природним для LLM-агента.
- Поле `created_at` є першим у фронтматері всіх файлів.
- Секції, які парсить оркестратор, мають англійські заголовки (`## Task`, `## Summary`, `## Done when`); довільні секції можуть бути будь-якою мовою.
- Стан вузла визначається наявністю файлів: `outputs.md` = resolved, `error.md` = failed, активний worktree = running, `invalidated` = invalidated.
- `inputs.md` не створюється: inputs є секцією `## Inputs` у `task.md`.
- Wrapper для repair-сеансу компілює попередні `repair/*-fact.md` у контекст інженера без mutable `repair/summary.md`.

## Update 2026-06-06

Додано фінальну схему `outputs.md`:

```markdown
---
created_at: 2026-06-06T10:05:00Z
---
## Summary
Коротке резюме результату — для observability і людини.

## Results

### report
ref: tasks/research/subgraph/analyze/artifacts/report.md

### findings
Inline-текст, якщо окремого артефакту немає.
```

Правила:

- `## Summary` обовʼязкова для observability-скрипту.
- `## Results` обовʼязкова для downstream agents.
- Якщо дані вже є у файлі — використовувати `ref:`; inline дозволено лише коли окремого артефакту немає.
- `docs/думка.MD` має лишатися контрактом схем файлів і правил, без дублювання архітектурних ADR-описів.

## Update 2026-06-06

Уточнено розмежування error і repair файлів:

- `error.md` — snapshot збою, пишеться один раз при першому failed state і не є журналом.
- `error.md` має секції `## System` для wrapper-даних (`exit_code`, `stderr`) і `## Analysis` для аналізу агента.
- `repair_history.md` — append-only журнал спроб відновлення, який пише EngineerAgent.
- Для `ops/spawn-*` обрано timestamp у назві файлу, бо динамічний spawn може трапитися кілька разів.
- Перед виконанням spawn оркестратор читає `ops/spawn-fact-*.md`; якщо відповідний fact уже існує, spawn не повторюється.
- `patches/` також використовує timestamp: `patches/<ts>-plan.md` / `patches/<ts>-fact.md`.

## Update 2026-06-06

Додано фінальний контракт LLM-first файлового формату вузлів:

- Всі імена файлів і директорій — англійською.
- Атрибути фронтматеру — англійською, `snake_case`.
- `id` вузла не дублюється у фронтматері, а береться з назви директорії.
- `parent:` є відносним шляхом від `tasks/`.
- `deps:` містить тільки sibling ids у межах того самого `subgraph/`.
- `ref:` є основним способом передачі даних між вузлами; копіювання даних не використовується, якщо існує файл-джерело.
- Патерн `plan → action → fact` поширюється на `ops/spawn-*`, `ops/kill-*` і `patches/*`.
- `task.md` є планом агента, а `outputs.md` — фактом виконання атомарного вузла.

## Update 2026-06-06

Уточнено остаточні правила файлового state store для графу:

- Стан вузла визначається файлами, а не полем `state`: `task.md` = pending, активний worktree = running, `outputs.md` = resolved, `error.md` = failed, `invalidated` = invalidated.
- Append-only межа починається з моменту створення worktree; до цього файли вузла можна редагувати й видаляти.
- `task.md` обʼєднує місію та inputs через секції `## Task`, `## Done when`, `## Inputs`.
- Топологія розподілена через `deps:` у `task.md` кожного child-вузла; центральний `graph.md` не створюється.
- `plan → action → fact` є recovery-протоколом для `spawn`, `kill` і `patch`.
- Для ретроактивного patch використовується kill-before-patch: спершу kill залежних worktree у топологічному порядку від листів, потім patch target-вузла, потім restart cascade.
- Timestamp у назвах `ops/*` і `patches/*` потрібен для динамічного spawn і повторних patch-операцій.

## Update 2026-06-06

- `task.md` обʼєднує завдання й inputs: окремий `inputs.md` прибрано, секція `## Inputs` живе всередині `task.md`.
- `created_at` додається першим полем frontmatter у `task.md`.
- Посилання в `outputs.md` мають бути відносними (`artifacts/...`), без дублювання абсолютного префікса `tasks/research/subgraph/...`.
- Топологія DAG зберігається розподілено: кожен дочірній вузол декларує попередників у `deps:` власного `task.md`, а не через центральний `graph.md`.
- Межа immutability — створення git worktree: до worktree файли можна редагувати, після старту агента дозволені лише нові файли; зміна після старту відбувається через kill → edit → restart.
- WAL-патерн однаковий для операцій: `*-plan.md` створюється до дії, відповідний `*-fact.md` — після дії; незавершені плани без fact є сигналом recovery.

## Update 2026-06-06

- Уточнено контракт файлової схеми графу: усі імена файлів і директорій мають бути англійською, бо їх обробляють bash/Node.js скрипти, glob-патерни та path-matching.
- `id` вузла не дублюється у frontmatter: ідентифікатор читається з назви директорії вузла, щоб уникнути розсинхронізації між шляхом і metadata.
- `inputs.md` обʼєднано в `task.md` як секцію `## Inputs`, бо агент має читати один файл з місією, критеріями завершення і вхідними даними.
- `created_at` зафіксовано як перше поле frontmatter у файлах `task.md`, `outputs_NNN.md` і `run_NNN.md`.
- `ops/` і `patches/` вилучено з поточної схеми: transcript фіксує, що spawn відбувається у worktree, а сценарій crash-recovery поза worktree поки ігнорується.
- Наступні вузли запускаються через git `post-merge` hook, який тригерить `graph run --auto` після merge worktree.

## Update 2026-06-06

- Стан вузла виводиться зі сканування файлів: тільки `task.md` → `waiting`; є `running.lock` → `running`; є `outputs_<epoch>.md` → `resolved`; є `invalidated` → `invalidated`.
- Межа immutable/append-only інваріанта — створення worktree: до `git worktree add` файли вузла можна редагувати; після старту потрібно `graph kill → edit → graph run`.
- Спроби виконання уніфіковано у `run_<epoch>.md` з `actor`, а результати — в `outputs_<epoch>.md`.
- Для ідентифікації запусків обрано epoch timestamp у назвах `run_<epoch>.md` і `outputs_<epoch>.md`, бо він не потребує лічильника і природно сортується хронологічно.
- Для виходу вузла обрано вільний `## Summary` замість структурованих портів, бо transcript описує систему як LLM-first і не фіксує потреби в жорсткій схемі output-портів.

## Update 2026-06-06

- `run_NNN.md` зафіксовано як єдиний запис будь-якої спроби вузла незалежно від виконавця: `actor: agent | engineer | human | auditor`.
- `outputs_NNN.md` і `run_NNN.md` залишаються окремими immutable файлами; `outputs_NNN.md` створюється для успішного результату, а `run_NNN.md` описує спробу.
- Нумерація `NNN` zero-padded до 3 цифр; wrapper рахує наявні `run_*.md` перед стартом як `count + 1`.
- Великі артефакти вузла зберігаються у фіксованій директорії `tasks/<node>/artifacts/`, а `ref:` в `outputs_NNN.md` має вказувати всередину неї.
- Каскадна інвалідація запускається тільки людиною через `graph invalidate <path>`; cascade вниз виконується автоматично.
- Bootstrap кореневого вузла використовує той самий формат `task.md`, але без поля `parent`.
- Паралелізм обмежується `max_worktrees` у `.n-cursor.json`; `graph run --auto` тримає чергу.
- Observability: `graph scan` для pull-перевірки, а при `result: failed` wrapper виводить terminal bell (`\a`).

## Update 2026-06-06

- `inputs.md` обʼєднано в `task.md` як секцію `## Inputs`, бо `task.md` і inputs мають одного автора, один lifecycle і однаково immutable після старту worktree.
- `## Inputs` містить підсекції `### name` з `ref: <path>` або inline-текстом; `ref:` обовʼязковий, якщо дані вже існують у файлі.
- `run_NNN.md` уніфікує `outputs.md`, `error.md` і `repair_history.md`: один immutable файл на спробу з frontmatter `created_at`, `actor`, `result`, опційним `worktree` і секціями `## Reasoning`, `## Script`, `## Ref`.
- Numbered immutable файли (`run_001.md`, `outputs_001.md`, …) замінюють append-only файли; wrapper рахує `count(run_*.md) + 1`, zero-padded до 3 цифр.
- `budget_sec` перенесено у frontmatter `task.md`, бо часовий бюджет є атрибутом вузла і застосовується до агента та інженера однаково.
- `ops/` і `patches/` видалено: spawn/kill crash recovery у main-гілці свідомо не покривається на цьому етапі, а деталі інженерських змін описуються в `## Reasoning` файлу `run_NNN.md`.
- Git `post-merge` hook тригерить `graph run --auto` після merge worktree, щоб запускати розблокованих наступників без ручного втручання.
