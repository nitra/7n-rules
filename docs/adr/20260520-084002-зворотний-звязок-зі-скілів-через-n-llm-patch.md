## ADR Зворотний зв'язок зі скілів через `n-llm-patch`

## Context and Problem Statement
Скіли `npm/skills` (`n-lint`, `n-fix`, `n-adr-normalize` тощо) запускаються у сторонніх репозиторіях, але збирають цінну інформацію (тертя, відсутні перевірки, неоднозначна документація), яка може покращити правила `npm/rules/` та самі скіли в пакеті `@nitra/cursor`. Потрібен канал доставки цих сигналів назад у пакет без прямого запису до нього із чужого репо.

## Considered Options
* Зберігати резюме безпосередньо у репо `@nitra/cursor` (dogfooding-варіант)
* Доставляти резюме через `n-llm-patch` — скіл генерує self-contained промпт, який потім застосовується окремим сеансом проти `@nitra/cursor`

## Decision Outcome
Chosen option: "Доставляти резюме через `n-llm-patch`", because скіли виконуються у зовнішніх проєктах і прямий запис у `@nitra/cursor` з них неможливий; `n-llm-patch` вже існує і розрахований саме на передачу контексту між репозиторіями.

### Consequences
* Good, because transcript фіксує очікувану користь: канал `n-llm-patch` не потребує нової інфраструктури і вписується в існуючий flow.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Скіл `npm/skills/llm-patch/SKILL.md` — існуючий механізм, що буде реюзований
- Чернетки зберігаються у `docs/cursor-feedback/<timestamp>-<skill>.md` у споживацькому репо
- Новий скіл `/n-feedback` (`npm/skills/feedback/`) збирає чернетки й готує промпт для `n-llm-patch`

## ADR Два канали збору сигналу під час виконання скілу

## Context and Problem Statement
Потрібно вирішити, як збирати корисний сигнал (тертя, прогалини в правилах) під час роботи `n-*` скілів: покладатися на само-звіт агента, що виконує скіл, чи на окремий Stop-хук, що майнить транскрипт.

## Considered Options
* Лише self-report агентом наприкінці скілу
* Лише Stop-хук (аналог `capture-decisions.sh`) для майнінгу транскрипту
* Обидва методи одночасно

## Decision Outcome
Chosen option: "Обидва методи одночасно", because так вирішив користувач: агент, що щойно виконав скіл, дає найточніший сигнал (знає який крок `SKILL.md` був неоднозначним), а Stop-хук є підстраховкою для випадків, коли агент не написав чернетку.

### Consequences
* Good, because transcript фіксує очікувану користь: self-report точніший (агент знає контекст), Stop-хук підстраховує.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Існуючий механізм: `.claude/hooks/capture-decisions.sh` — зразок для Stop-хука
- `alwaysApply: true` правило `n-skill-feedback.mdc` у `npm/rules/skill-feedback/` — навчає агента писати чернетку після скілу
- Жорстка схема чернетки: `npm/schemas/cursor-feedback.json` з полями `target`, `id`, `kind`, `evidence`, `suggestion`

## ADR Жорстка схема чернетки зворотного зв'язку

## Context and Problem Statement
Чернетки зворотного зв'язку по скілах мають бути придатними до автоматичної батч-нормалізації й дедуплікації через LLM. Вільний текст ускладнює групування за цільовим файлом пакета та підрахунок частоти проблем.

## Considered Options
* Вільний текстовий формат (есе)
* Жорстка JSON-схема з фіксованими полями

## Decision Outcome
Chosen option: "Жорстка JSON-схема з фіксованими полями", because вільний текст дає слабший і дорожчий сигнал для LLM-нормалізації; структура дозволяє групувати за `target`/`id` і рахувати частоту (`kind`) без додаткового парсингу.

### Consequences
* Good, because transcript фіксує очікувану користь: консолідоване резюме групується за цільовим файлом (`npm/rules/<id>/<id>.mdc`, `npm/skills/<id>/SKILL.md`, `check-*.mjs`) зі severity і частотою.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Файл схеми: `npm/schemas/cursor-feedback.json`
- Поля: `target` (rule | skill | check), `id`, `kind` (missing-check | ambiguous-doc | false-positive | no-autofix | recurring-pattern), `evidence`, `suggestion`
- Нормалізація реюзить машинерію `normalize-decisions.sh`
