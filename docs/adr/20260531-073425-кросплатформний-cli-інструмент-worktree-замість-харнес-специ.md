## ADR кросплатформний CLI-інструмент worktree замість харнес-специфічного EnterWorktree

## Context and Problem Statement
Харнес Claude Code надає вбудований інструмент `EnterWorktree`, але він захардкоджує розташування worktree у `.claude/worktrees/` і за замовчуванням відгалужується від `origin/main`, а не локального HEAD. Це несумісно з поточними локальними комітами та захищеною директорією `.claude/worktrees/` в `cursor/CLAUDE.md`. Крім того, `EnterWorktree` не доступний у Cursor — потрібен інструмент, що працює однаково в Claude Code, Cursor і терміналі.

## Considered Options
* `EnterWorktree` (харнес-специфічний інструмент Anthropic) з налаштуванням `baseRef: head`
* Ручний `git worktree add` у sibling-каталог (наприклад, `../cursor-skill-meta`)
* Нова CLI-підкоманда `n-cursor worktree` у пакеті `@nitra/cursor`

## Decision Outcome
Chosen option: "Нова CLI-підкоманда `n-cursor worktree` у пакеті `@nitra/cursor`", because це єдина форма, що дає ідентичну поведінку в Claude Code, Cursor і терміналі без залежності від інструментів конкретного харнесу; CLI гарантує одні правила розташування й формату `.md`-опису незалежно від середовища виклику.

### Consequences
* Good, because CLI реалізує конвенцію `.worktrees/` програмно (санітизація гілки, генерація `.md`-опису, `prune`), знімаючи залежність від дисципліни агента щодо ручного виконання рецепту.
* Bad, because transcript не містить підтверджених негативних наслідків.

## ADR правило `worktree` — нормалізація сироти в канонічне джерело пакету

## ADR інвентарний файл `.worktrees/<b>.md` — де зберігати і хто генерує

## Context and Problem Statement
Користувач поставив вимогу: кожен git-worktree має містити агентно- і людинозрозумілий опис призначення; цей опис не потрібен у git-історії, лише для інвентаризації поточних worktree. Питання — де зберігати файл-опис і хто відповідає за його генерацію (агент вручну, CLI-команда, чи обидва).

## Considered Options
* Файл-опис усередині checkout worktree (`.worktrees/<b>/.n-worktree.md`)
* Файл-опис поруч із checkout, але не всередині (`.worktrees/<b>.md`)
* Генерація описового файлу як частина CLI `worktree add` (атомарна операція)
* Генерація описового файлу агентом вручну за markdown-конвенцією

## Decision Outcome
Chosen option: "Файл-опис `.worktrees/<b>.md` поруч із checkout + генерація CLI-командою `worktree add`", because розміщення поруч (а не всередині) дозволяє `cat .worktrees/*.md` знайти всі описи без рекурсивного glob; CLI-генерація робить операцію атомарною і не залежить від дисципліни агента; `.worktrees/` gitignored — файл ніколи не потрапляє в git-історію.

### Consequences
* Good, because `cat .worktrees/*.md` і `git worktree list` — дві команди для повної інвентаризації; опис обовʼязковий (`add` без нього падає), тому не виникають безіменні worktree.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Шаблон `.md`: заголовок `# <branch>`, поля **Задача**, **Дата** (з `git`), **База (коміт)** (`git rev-parse --short HEAD`), гілка, рядок «Прибрати: `npx @nitra/cursor worktree remove <branch>`»
- `worktree add <branch> "<опис>"` — опис обовʼязковий аргумент (G1)
- Конвенція зафіксована в `n-worktrees.mdc` (зараз сирота, нормалізується в `npm/rules/worktree/`)
