# k8s: зниження дефолту `resources.requests.cpu` до `0.05` для dev

**Status:** Accepted
**Date:** 2026-05-09

## Контекст

У dev-кластері pod-и займали `0.5` CPU request за замовчуванням, що заважало щільному bin-packing навіть за реального споживання в мілісекунди. `HorizontalPodAutoscaler` в dev-середовищі тримає `maxReplicas: 1`, тому зменшення request не впливає на автомасштабування.

## Рішення/Процедура/Факт

У `npm/mdc/k8s.mdc` дефолтне значення `resources.requests.cpu` для Deployment знижено з `'0.5'` до `'0.05'`. Додатково внесено:

- примітку про HPA-шум (`ScalingLimited` у статусі) при такому низькому порозі — щоб не вводило в оману;
- підсекцію «CPU request у прод-overlay» з прикладом Kustomize patch для підняття до реального значення.

`check-k8s.mjs` не змінювався — він перевіряє лише наявність непорожнього значення, а не конкретну цифру.

## Обґрунтування

При `maxReplicas=1` HPA фізично не масштабує pod-и, тому низький request безпечний: `ScalingLimited` у dev — косметичний статус, а не реальна проблема. Прод-значення — відповідальність команди через overlay; правило рекомендує значення, але не enforce-ить конкретну цифру.

## Розглянуті альтернативи

Env-залежна перевірка в `check-k8s.mjs` (аналогічно до HPA/PDB min/max) — відхилено: зміна стосується лише `.mdc`-документа. Проміжне значення `0.1`/`100m` — відхилено на користь явного `0.05` для однозначності.

## Зачіпає

`npm/mdc/k8s.mdc` (рядки 99–128); усі проєкти, що читають це правило і мають Deployment з дефолтним CPU request.
