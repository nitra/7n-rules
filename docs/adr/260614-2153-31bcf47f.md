## Структура результату

```js
{
total: number,    // скільки правил реально запущено
failed: number,   // скільки завершились з кодом ≠ 0
rules: Array<{
ruleId: string, // id правила (= ім'я каталогу npm/rules/<id>)
ok: boolean,    // exit-status === 0
output: string  // злита stdout+stderr запущеного fix.mjs
}>
}
```

---

## Навіщо кожне поле — по споживачах

**`total` + `failed` (чисельник/знаменник)**

→ `orchestrator.mjs`: прогрес-рядок `"2/5 порушень (adr, changelog)"` і швидкий early-exit без ітерації масиву.
→ `post-tool-use-fix.mjs`: `if (failed === 0) return 0` — hot-path без аллоку нового масиву.

Два числа замість `rules.filter(r => !r.ok).length` — це контракт зручності для fast-check.

---

**`rules[i].output` — головне, що передається вперед**

Це не просто лог. В orchestrator T1-кроці:
```js
// orchestrator.mjs:52
await runLlmWorker(rule.ruleId, rule.output, cwd, { model })
```
`output` — діагностика з `fix.mjs` (що саме зламано, де, як виглядає порушення), яку LLM отримує як context для авто-виправлення. Без нього LLM не знав би, що виправляти.

В `orchestrate.mjs` (read-only lint) та `post-tool-use-fix.mjs` — той самий `output` виводиться на stderr як user-facing звіт: людина або CI бачать реальне повідомлення від правила, не абстрактний `rule failed`.

---

**`rules[i].ok` vs `rules[i].ruleId`**

- `ok` — дозволяє фільтрувати `failed = initial.rules.filter(r => !r.ok)` і diff-ити між ітераціями (T0 fix → re-check → які правила зникли з failed-списку).
- `ruleId` — ключ identity при diff: `t0Fixed = failed.filter(r => !failedAfterT0.some(f => f.ruleId === r.ruleId))`.

---

## Чому саме такий дизайн

`runFixCheck` — це **checkpoint**: викликається тричі на ітерацію в orchestrator (initial → after T0 → after LLM). Кожного разу результат порівнюється зі попереднім. Тому масив `rules` із стабільними `ruleId` як ключами — єдиний спосіб зробити ці diff-и без зовнішнього стану. `output` при цьому захоплюється свіжим на кожному checkpoint — LLM завжди бачить актуальний стан після попереднього фіксу.
