**Status:** Accepted

**Date:** 2026-06-09

## Context and Problem Statement

Виникла ідея фіксувати reasoning після виклику скілів і на основі нього покращувати правила та `SKILL.md`. Питання поширилось на Claude Code, Codex і Cursor: чи можна застосувати один підхід до всіх трьох інструментів. Transcript показав, що raw reasoning доступний нерівномірно: у Claude Code він може бути в JSONL, у Codex reasoning зашифрований, а в Cursor доступ залежить від недокументованого стану без lifecycle hooks.

## Considered Options

- Витягувати raw `thinking`-блоки з JSONL-транскриптів Claude Code.
- Витягувати зашифровані `encrypted_content` з Codex rollout-файлів.
- Скрапити SQLite `state.vscdb` у Cursor.
- Використовувати спостережувані сліди: tool-calls, помилки, повтори, exit-коди та фінальні diff.

## Decision Outcome

Chosen option: "Використовувати спостережувані сліди як основний сигнал", because raw reasoning доступний лише в Claude Code, у Codex він зашифрований OpenAI, а в Cursor доступний лише крихкий недокументований шлях; натомість послідовності tool-calls, exit-коди, помилки й повтори присутні в усіх трьох інструментах і показують, де правила або скіли недовизначені.

### Consequences

- Good, because підхід portable між Claude Code, Codex і Cursor.
- Good, because transcript фіксує очікувану користь: сліди описують фактичну поведінку агента — де він кружляв, ретраїв або отримував `exit_code=1`.
- Bad, because для Cursor потрібно скрапити недокументований SQLite-формат `globalStorage/` або `workspaceStorage/<hash>/state.vscdb`, що може ламатись при оновленнях Cursor.
- Bad, because відмова від raw reasoning втрачає сигнал про внутрішні вагання моделі, які не завжди проявляються у поведінці.
- Neutral, because для Claude Code raw `thinking` усе ще можна збирати як додатковий інструментальний сигнал через наявний hook-конвеєр, але не як основний portable контракт.

## More Information

Transcript-факти:

- Claude Code transcripts: `~/.claude/projects/-Users-vitalii-www-nitra-cursor/*.jsonl`; поле `isSidechain` розрізняє головну модель і субагентів.
- Codex sessions: `~/.codex/sessions/2026/06/05/rollout-*.jsonl`; reasoning має структуру `{"type":"reasoning", "summary":[], "encrypted_content":"..."}`.
- Codex hooks: `~/.codex/config.toml`, поле `notify`, подія `turn-ended`; transcript не підтвердив доступ до raw reasoning через цей hook.
- Claude hooks: `.claude/hooks/capture-decisions.sh` і `.claude/hooks/normalize-decisions.sh` уже реалізують pattern «зафіксувати лог → LLM-дистиляція».
- Для Claude Code можна розширити Stop-hook pipeline: витягувати thinking-блоки, корелювати їх із `tool_use` типу `Skill`, накопичувати append-log і дистилювати пакетно.
- Рекомендований фокус LLM-дистиляції raw reasoning: моменти, де модель вагалась, помилялась або не знайшла правило.
- У `.claude/settings.local.json` на момент session було виявлено невирішений git merge conflict, через що файл був невалідним JSON.

## Update 2026-06-09

Додаткові факти щодо збору reasoning-блоків Claude Code:

- У наявних `.jsonl` transcript-файлах Claude Code вже є блоки `{"type":"thinking"}`; у session згадано 348 файлів поточного проєкту.
- Атрибуцію до конкретного скіла можна робити через кореляцію `tool_use` типу `Skill` у тому самому `.jsonl` з наступними thinking-блоками.
- Наявні hooks `.claude/hooks/capture-decisions.sh` і `.claude/hooks/normalize-decisions.sh` реалізують pattern «зафіксувати сирий лог → дистилювати через LLM».
- Запропонований pipeline: `SubagentStop` hook витягує thinking-блоки, привʼязані до виклику скіла → append-log → LLM-прохід → висновки у `docs/skill-insights/`.
- Transcript також зафіксував проблему: `.claude/settings.local.json` містив невирішений git merge conflict і через це не парсився як JSON.

Це уточнення не змінює основний висновок: raw reasoning корисний для Claude Code, але не є portable сигналом для всіх інструментів.
