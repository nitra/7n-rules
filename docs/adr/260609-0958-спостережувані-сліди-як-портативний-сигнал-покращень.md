**Status:** Accepted
**Date:** 2026-06-09

## Context and Problem Statement

Виникла ідея фіксувати reasoning після виклику скілів і на його основі покращувати `.mdc`-правила та `SKILL.md`-файли. Потрібно було зрозуміти, чи один підхід можна застосувати до Claude Code, Codex і Cursor.

## Considered Options

- Витягувати raw `thinking` blocks з JSONL transcript-ів Claude Code.
- Витягувати зашифровані `encrypted_content` з Codex rollout-файлів.
- Scrape-ити SQLite `state.vscdb` у Cursor.
- Використовувати спостережувані сліди: tool-calls, помилки, повтори, exit codes і фінальні diff-и.

## Decision Outcome

Chosen option: "Використовувати спостережувані сліди", because raw reasoning доступний лише в Claude Code, у Codex він зашифрований як `encrypted_content`, а в Cursor доступний лише як summary без lifecycle hooks; натомість tool-calls, exit codes, помилки та повтори присутні в усіх трьох інструментах і показують, де правила або скіли недовизначені.

### Consequences

- Good, because підхід портативний між Claude Code, Codex і Cursor.
- Good, because transcript фіксує очікувану користь: сліди описують фактичну поведінку агента — де він кружляв, retry-ив або отримував `exit_code=1`.
- Bad, because для Cursor потрібно scrape-ити недокументований SQLite-формат `globalStorage/` або `workspaceStorage/<hash>/state.vscdb`, що є крихким.
- Bad, because відмова від raw reasoning втрачає сигнал про внутрішні вагання моделі, які не завжди проявляються у поведінці.
- Neutral, because raw reasoning Claude Code все одно можна збирати як додатковий непортативний сигнал через Stop-hook pipeline.

## More Information

- Claude Code transcript-и: `~/.claude/projects/-Users-vitalii-www-nitra-cursor/*.jsonl`; поле `isSidechain` розрізняє головну модель і субагентів.
- Codex sessions: `~/.codex/sessions/2026/06/05/rollout-*.jsonl`; reasoning має форму `{"type":"reasoning", "summary":[], "encrypted_content":"..."}`.
- Codex hooks: `~/.codex/config.toml`, поле `notify`, подія `turn-ended`.
- Наявні hooks у `.claude/hooks/`: `capture-decisions.sh` і `normalize-decisions.sh`.
- Для Claude Code raw reasoning можна збирати через розширення існуючого Stop-hook pipeline, append-log і пакетний LLM-прохід.
- Команда для перевірки наявності thinking у transcript-файлі: `grep -l '"type":"thinking"' ~/.claude/projects/-Users-vitalii-www-nitra-cursor/*.jsonl`.
- Цільова директорія для distillation insights, запропонована в transcript: `docs/skill-insights/`.
- У `.claude/settings.local.json` на момент сесії був невирішений git merge conflict, тому файл не був валідним JSON.

## Update 2026-06-09

Драфт додає деталі для Claude Code reasoning-збору:

- raw `thinking` blocks уже зберігаються у `.jsonl` transcript-файлах Claude Code;
- Stop-hooks `capture-decisions.sh` і `normalize-decisions.sh` реалізують pattern «зафіксувати сирий лог → дистилювати через LLM»;
- attribution до конкретного скіла можлива через кореляцію `tool_use` типу `Skill` у тому ж `.jsonl`;
- запропонована директорія для висновків: `docs/skill-insights/`;
- окрема проблема сесії: `.claude/settings.local.json` містив невирішений git merge conflict і не парсився як JSON.

Transcript попереджає, що reasoning багатослівний, transcript-и вже займали 262 МБ, а для Cursor raw reasoning залежить від provider-а моделі.
