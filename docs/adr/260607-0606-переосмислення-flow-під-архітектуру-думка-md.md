## ADR Переосмислення `flow` під архітектуру `думка.MD`

## Context and Problem Statement
Існуючий `n-cursor flow` — монолітний протокол із 5-фазним циклом (`init/spec/plan/verify/release`) і `.flow.json` як джерелом стану. При інтеграції з автономним ОАГ із `думка.MD` (файлова-система-як-стан, `tasks/<node>/task.md`) виникає суперечність: flow має внутрішній стан, граф — зовнішній. Потрібно переосмислити `flow` як протокол всередині вузла графу.

## Considered Options
* Залишити flow незмінним, запускати його всередині `graph run` як black-box
* Повністю переосмислити flow у два окремих етапи: planning + execution

## Decision Outcome
Chosen option: "Повністю переосмислити flow у два окремих етапи", because `graph` вже відповідає за worktree-lifecycle ззовні, а `flow` повинен обслуговувати лише те, що відбувається всередині одного вузла.

### Consequences
* Good, because transcript фіксує очікувану користь: `.flow.json`, `docs/specs/`, `docs/plans/` зникають, стан вузла повністю деривується з файлів у `tasks/<node>/`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Рішення зафіксовано в `docs/думка.MD` розділ "Інтеграція з `n-cursor flow`". Нова команда: `npm/scripts/dispatcher/index.mjs`. Файли, що зникають: `.flow.json`, `docs/specs/`, `docs/plans/`.

## ADR Явний виклик `graph spawn` після `flow plan` (Variant B)

## Context and Problem Statement
Після того як `flow plan` у складеному шляху створив дочірні `task.md`, потрібно сигналізувати оркестратору про нові вузли. Питання: хто саме викликає `graph spawn` — `flow plan` автоматично, чи агент вручну.

## Considered Options
* Variant A — bundled: `flow plan` сам викликає `graph spawn` в кінці
* Variant B — explicit: `flow plan` тільки створює дочірні `task.md`, агент викликає `graph spawn` явно

## Decision Outcome
Chosen option: "Variant B — explicit", because transcript: "Варіант B — потрібен скрипт який буде проходити весь граф і визначати де потрібно запустити і згідно черги буде вирішувати де запустити."

### Consequences
* Good, because transcript фіксує очікувану користь: `graph spawn` потрапляє в чергу оркестратора, а не запускається негайно — це дозволяє `n-cursor watch` керувати порядком і обмеженнями паралелізму.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Оркестратор черги реалізується через post-merge hook (`graph run --auto`) і демон `n-cursor watch`. Обидва скануть граф і приймають рішення про наступний вузол для запуску.

## ADR Схема нумерації `pending-audit_NNN.md` — NNN дзеркалює `outputs_NNN.md`

## Context and Problem Statement
Після завершення роботи агент запитує аудит. Аудит ставиться у чергу (файловий sentinel). Оркестратор має знати яку саме версію `outputs_NNN.md` перевіряти, якщо між запитом і виконанням аудиту агент міг написати нові версії.

## Considered Options
* Порожній sentinel-файл `.pending-audit` (overwrite при новому запиті)
* Numbered файли `pending-audit_NNN.md`, де NNN = NNN відповідного `outputs_NNN.md`
* Lifecycle: `.pending-audit` → `.running-audit` → видалення

## Decision Outcome
Chosen option: "Numbered файли `pending-audit_NNN.md`, де NNN = NNN відповідного `outputs_NNN.md`", because "якщо агент завершив на `outputs_003.md` і саме цю версію він вважає єдиною доцільною аудиту, то створюється `pending-audit_003.md` і так далі — щоб нумерація не губилася на всіх етапах."

### Consequences
* Good, because transcript фіксує очікувану користь: ім'я файлу саме по собі є посиланням на версію (`ref:` у front-matter вже не потрібен), аудитор завжди знає яку версію перевіряти.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схема нумерації по вузлу: `outputs_001.md` → `pending-audit_001.md` → `run_002.md (actor: auditor)`. `run_NNN.md` — незалежний лічильник (всі актори). `outputs_NNN.md` і `pending-audit_NNN.md` — спільний ключ.

## ADR Видалення Фасаду B (`flow run/resume/cancel/repair`)

## Context and Problem Statement
Існуючий Фасад B (`flow run`, `flow resume`, `flow cancel`, `flow repair`) реалізував повний автономний 5-фазний цикл через `.flow.json`. З переходом до `graph`-оркестрації і файлової-системи-як-стану ці команди дублюють функціональність `graph run`, `graph kill` та implicit state recovery.

## Considered Options
* Залишити Фасад B для зворотної сумісності
* Видалити Фасад B повністю, передати відповідальність `graph`-шару

## Decision Outcome
Chosen option: "Видалити Фасад B повністю", because `flow run` замінюється `graph run` → агент → `flow plan` → `flow verify`; `flow resume` не потрібен (стан у файлах, повторний `graph run` читає наявні `run_NNN.md`); `flow cancel` = `graph kill`; `flow repair` не потрібен (`.flow.json` зникає).

### Consequences
* Good, because transcript фіксує очікувану користь: менше команд, єдина точка входу для lifecycle вузла через `graph`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Видаляються: `flow run`, `flow resume`, `flow cancel`, `flow repair`, `flow review`, `flow gate`. Залишаються: `flow plan` (Stage 1), `flow verify` (Stage 2). Додаються сигнали: `graph done`, `graph audit`, `graph failed`, `graph spawn`.
