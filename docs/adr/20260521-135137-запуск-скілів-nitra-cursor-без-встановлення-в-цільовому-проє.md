## ADR Запуск скілів `@nitra/cursor` без встановлення в цільовому проєкті

## Context and Problem Statement
У зовнішніх проєктах потрібно виконувати скіли пакета `@nitra/cursor` через `npx`, не додаючи пакет як `devDependency` і не синхронізуючи `.cursor/rules/`. Скіли зберігаються як `SKILL.md`-файли у `npm/skills/<id>/SKILL.md` установленого пакета.

## Considered Options
* Окремий пакет `@nitra/skills` зі своїм bin (за аналогією з прикладом у завданні)
* Підкоманда `skill` у наявному bin `n-cursor` того ж пакета `@nitra/cursor`

## Decision Outcome
Chosen option: "Підкоманда `skill` у наявному bin `n-cursor`", because скіли вже лежать у `npm/skills/` того ж пакета, тож окремий пакет — зайва одиниця публікації; `npx @nitra/cursor skill …` достатньо.

### Consequences
* Good, because transcript фіксує очікувану користь: `npx @nitra/cursor skill list/taze/cursor/claude` працює без `devDependencies` у цільовому проєкті.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізація: `npm/scripts/skills-cli.mjs` (логіка), `npm/bin/n-cursor.js` (підкоманда `skill`). Версія пакета: **1.13.72**.

## ADR Спрощення UX CLI скілів до чотирьох канонічних форматів

## Context and Problem Statement
Після першої реалізації з'явилося кілька дублюючих форм запуску: `skill prompt <id>`, `skill claude <id>`, `skill cursor <id>`, окремі bins `n-skills` та `n-claude`, підкоманда `claude` верхнього рівня у `n-cursor`. Користувач хотів мінімальний, однозначний набір команд.

## Considered Options
* Повний набір: `skill list`, `skill prompt <id>`, `skill claude <id>`, `skill cursor <id>`, `claude <id>` (top-level), окремі bins `n-skills` та `n-claude`
* Чотири форми: `skill list`, `skill <id>`, `skill cursor <id>`, `skill claude <id>`

## Decision Outcome
Chosen option: "Чотири форми без `prompt`-підкоманди та без окремих bins", because користувач явно сформулював бажаний набір команд; `skill <id>` (без `prompt`) виконує ту ж роль і коротше; окремі bins `n-skills`/`n-claude` видалено як зайві.

### Consequences
* Good, because transcript фіксує очікувану користь: єдина точка входу `skill` у `n-cursor`, жодних конфліктних bin-імен (`n-claude` конфліктував би із системним `claude` CLI).
* Bad, because зворотна сумісність форм `skill prompt …` і `skill claude …` (як top-level) втрачена — але пакет ще не публікувався з цими підкомандами.

## More Information
Видалені файли: `npm/bin/n-skills.js`, `npm/bin/n-claude.js`. Видалені підкоманди: `prompt`, top-level `claude`. Збережені підкоманди: `skill list`, `skill <id> [task]`, `skill cursor <id> [task]`, `skill claude <id> [task]`. `npm/package.json` → `"bin": { "n-cursor": "./bin/n-cursor.js" }`.

## ADR Склад промпту для скілу

## Context and Problem Statement
CLI повинен передавати агенту (Claude/Cursor) достатній контекст: інструкцію скілу й стан цільового проєкту. Потрібно визначити, що саме включати в згенерований промпт.

## Considered Options
* Лише вміст `SKILL.md` і текст завдання
* `SKILL.md` + завдання + контекстні файли CWD (`package.json`, `tsconfig.json`, `.n-cursor.json`)

## Decision Outcome
Chosen option: "`SKILL.md` + завдання + контекстні файли CWD", because агент повинен орієнтуватися на поточний проєкт, а не тільки на абстрактну інструкцію скілу; `package.json` і `tsconfig.json` — мінімальний стандартний контекст JS/TS-проєкту.

### Consequences
* Good, because transcript фіксує очікувану користь: агент отримує інструкцію скілу з пакета і одразу бачить структуру цільового проєкту.
* Bad, because Neutral, because transcript не містить підтвердження наслідку щодо можливого перевантаження промпту для великих `package.json`.

## More Information
Секції промпту: `# Task` (текст або заглушка `Execute the skill instructions for this project.`), `# Skill` (вміст `skills/<id>/SKILL.md`), `# Current project` (CWD, `package.json`, `tsconfig.json`, `.n-cursor.json` — якщо існують). Логіка: `npm/scripts/skills-cli.mjs`, функція `buildSkillPrompt`. Id скілу нормалізується: `n-taze` → `taze`.
