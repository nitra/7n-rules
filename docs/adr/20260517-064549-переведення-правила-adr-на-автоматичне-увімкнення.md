## Knowledge Самопосилання `devDependencies` у `npm/package.json` — побічний ефект stop-хука

**Контекст:** При запуску `npx @nitra/cursor check` (або sync) з робочого каталогу `npm/` замість кореня репозиторію скрипт `ensure-nitra-cursor-in-root-dev-dependencies.mjs` записує `@nitra/cursor` у `devDependencies` поточного `package.json`. Це призводить до появи self-reference `"@nitra/cursor"` у самому `npm/package.json`, що ламає `check npm-module`.

**Рішення/Процедура/Факт:** Видаляти `devDependencies` вручну після кожного випадкового запуску sync/check з підкаталогу. Правильна точка запуску — корінь репо. Stop-хук Claude Code запускається з кореня, тому в CI/CD проблема не відтворюється.

**Обґрунтування:** Логіка `ensure-nitra-cursor-in-root-dev-dependencies` навмисно не перевіряє, чи поточне репо є самим пакетом `@nitra/cursor` — це edge case, актуальний лише для розробників самого пакета.

**Розглянуті альтернативи:** Додати guard `if (pkg.name === '@nitra/cursor') return` до скрипту — не було реалізовано в цій сесії, зафіксовано як known gotcha.

**Зачіпає:** `npm/scripts/ensure-nitra-cursor-in-root-dev-dependencies.mjs`, `npm/package.json`

## ADR Нове правило `security` — gitleaks-лінт через `lint-security`

**Контекст:** Жодне з існуючих правил `@nitra/cursor` не перевіряло наявність секрет-сканера в проєкті. Витоки ключів API / токенів у git-історії — типова ризик-зона, яку варто автоматизувати.

**Рішення/Процедура/Факт:**
- `npm/rules/security/security.mdc` — `alwaysApply: false`, `globs: "**/.gitleaks.toml,**/package.json,.github/workflows/**/*.yml"`, версія `1.1`
- `npm/rules/security/auto.md` = `завжди` — правило автодетектується без умов
- `npm/rules/security/fix/gitleaks/check.mjs` — FS-перевірка: `.gitleaks.toml` існує + `useDefault = true` у `[extend]`; 5 JS-тестів (check.test.mjs)
- `npm/rules/security/policy/package_json/package_json.rego` — 5 deny-правил: `scripts.lint-security` існує і містить `gitleaks detect|git`; агрегований `lint` містить `bun run lint-security`; `gitleaks` не в `dependencies`/`devDependencies`; 9 rego-тестів (package_json_test.rego)
- `npm/scripts/auto-rules.mjs` — `addRule('security')` між `addRule('adr')` і `addRule('text')`
- Dogfood у `cursor` репо: `package.json#scripts.lint-security = "gitleaks detect --no-banner"`, `lint` оновлено, `.gitleaks.toml` з `useDefault = true`
- Bump `1.11.17` → `1.12.0` → `1.12.1` (після уточнення `alwaysApply`)

**Обґрунтування:** `gitleaks detect --no-banner` — стандартний CLI-спосіб перевірки git-репо на секрети; не вимагає конфігурації для базового захисту. `.gitleaks.toml` з `useDefault = true` — мінімальний канон, що дозволяє додавати власний allowlist без вимкнення дефолтних правил. `alwaysApply: false` + вузькі globs — замість `alwaysApply: true`, бо правило повністю покрите `check`, і AI-контекст потрібен лише при редагуванні релевантних конфігів.

**Розглянуті альтернативи:** `gitleaks detect` через `npx` (без глобального install) — відхилено, бо `gitleaks` не є npm-пакетом; `alwaysApply: true` — початково реалізовано, потім змінено на `false` + globs за аналогією з `changelog`.

**Зачіпає:** `npm/rules/security/`, `npm/scripts/auto-rules.mjs`, `npm/scripts/auto-rules.test.mjs`, `.gitleaks.toml`, `package.json#scripts`, `npm/package.json`, `npm/CHANGELOG.md`

## ADR Масовий перехід 9 правил з `alwaysApply: true` на `alwaysApply: false` + `globs:`

**Контекст:** Більшість правил `@nitra/cursor` мали `alwaysApply: true`, що завантажує весь їх текст у AI-контекст при кожному зверненні до репо — навіть коли редагується несповоренний файл. При цьому кожне правило має повне покриття `check` (JS + Rego), тобто AI-контекст потрібен лише при редагуванні відповідних конфігів.

**Рішення/Процедура/Факт:** Перемкнуто 9 правил:

| Правило | Globs |
|---|---|
| `bun` | `**/package.json,**/bunfig.toml,**/bun.lock` |
| `capacitor` | `**/capacitor.config.json,**/android/**,**/ios/**` |
| `js-bun-db` | `**/package.json,**/src/conn/**` |
| `js-bun-redis` | `**/package.json,**/src/conn/**` |
| `js-mssql` | `**/package.json,**/src/conn/mssql-*` |
| `npm-module` | `npm/**,**/package.json,**/hk.pkl,.github/workflows/npm-publish.yml` |
| `tauri` | `**/src-tauri/**,**/tauri.conf.json` |
| `js-lint` | `**/{.oxlintrc.json,eslint.config.js,.jscpd.json,knip.json,package.json,*.{js,ts,mjs,cjs,jsx,tsx}}` |
| `js-run` | `**/package.json,**/src/**/*.{js,ts,mjs,cjs}` |

Залишено `alwaysApply: true`: `text` (cross-cutting — cspell/стиль у будь-якому файлі), `adr` (процес, не прив'язаний до файлів), `ci4` (0 check.mjs + 0 rego — без AI-контексту мертве), `abie` (динамічний `applies`-гейт + розкидані типи файлів).

Bump `1.12.1` → `1.13.0`.

**Обґрунтування:** Патерн `changelog` і `security`: коли є `npx @nitra/cursor check`, помилка буде виявлена програматично незалежно від AI-контексту. Вузький glob гарантує, що AI отримує правило **саме тоді**, коли редагує відповідний файл. Це знижує споживання контексту без втрати покриття.

**Розглянуті альтернативи:** Залишити `alwaysApply: true` — відхилено (марно палить контекст); переключити `text` — відхилено (cspell-поради потрібні при будь-якому текстовому редагуванні); переключити `ci4` — відхилено (правило без `check` — єдиний носій логіки в AI-контексті).

**Зачіпає:** `npm/rules/{bun,capacitor,js-bun-db,js-bun-redis,js-mssql,npm-module,tauri,js-lint,js-run}/*.mdc`, `npm/package.json`, `npm/CHANGELOG.md`
