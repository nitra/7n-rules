**Status:** Accepted
**Date:** 2026-06-06

## Context and Problem Statement

Існували два незалежні шари: `n-cursor flow` як пасивний турнікет для ізоляції, планування, верифікації та релізу, і архітектура автономного DAG-оркестратора з `docs/думка.MD`. Обидва використовували git worktrees, але мали несумісні протоколи стану: `.flow.json`, `docs/specs/`, `docs/plans/` у старому `flow` проти `tasks/<node>/task.md`, `plan_001.md`, `outputs_NNN.md` у графі.

Потрібно було обʼєднати ці системи так, щоб `graph` керував DAG і lifecycle worktrees ззовні, а `flow` залишився протоколом виконання одного вузла зсередини.

## Considered Options

- Зберегти `flow` і `graph` незалежними системами.
- Повністю злити обидві системи в одну без розділення відповідальностей.
- Переписати `flow` під контракт `думка.MD`: `flow` як протокол всередині вузла, `graph` як зовнішній оркестратор DAG.
- Для Stage 1 залишити два кроки: `flow spec` для design/brainstorm і `flow plan` для декомпозиції.
- Для Stage 1 обʼєднати design і decompose в один `flow plan`.
- Зберігати режим інтерактивності як аргумент команди.
- Зберігати режим інтерактивності як `mode: human|agent` у frontmatter `task.md`.
- Для composite-шляху викликати `graph spawn` автоматично всередині `flow plan`.
- Для composite-шляху створювати дочірні `task.md`, після чого агент явно викликає `graph spawn`.
- Для автоматичного запуску ready-вузлів використовувати тільки post-merge hook.
- Для автоматичного запуску ready-вузлів використовувати тільки `n-cursor watch` daemon.
- Використовувати і post-merge hook, і `n-cursor watch` daemon.

## Decision Outcome

Chosen option: "Переписати `flow` під контракт `думка.MD`: `flow` як протокол всередині вузла, `graph` як зовнішній оркестратор DAG", because `flow` і `graph` перекривалися за відповідальністю, а поділ на зовнішній DAG-оркестратор і внутрішній execution protocol усуває дублювання worktree-логіки та дає одне джерело істини для стану вузла.

Stage 1 виконується через один `flow plan`, який поглинає попередній `flow spec`: користувач явно обрав спільний крок. `flow plan` читає `task.md`, враховує `mode: human|agent` з default `human`, і або пише `plan_001.md` для атомарного вузла, або створює дочірні `task.md` для складеного вузла.

Для composite-шляху обрано явний Variant B: `flow plan` тільки створює дочірні задачі, а агент після перевірки викликає `n-cursor graph spawn <path>`.

Для черги ready-вузлів обрано обидва механізми: post-merge hook викликає `n-cursor graph run --auto` після merge, а `n-cursor watch` daemon моніторить `tasks/`, файлові зміни та stale worktrees.

### Consequences

- Good, because стан вузла переходить до єдиної файлової моделі `task.md`, `plan_001.md`, `outputs_NNN.md` замість `.flow.json`, `docs/specs/` і `docs/plans/`.
- Good, because `graph` відповідає за DAG, чергу, worktree lifecycle і `max_worktrees`, а `flow` — лише за протокол виконання одного вузла.
- Good, because `flow plan` як єдиний Stage 1 зменшує кількість команд і стикувальних станів між design і decompose.
- Good, because `mode: human` як default у `task.md` є безпечним режимом за замовчуванням.
- Good, because явний `graph spawn` дає агенту змогу перевірити дочірні `task.md` перед реєстрацією нових вузлів.
- Good, because комбінація post-merge hook і `n-cursor watch` покриває як git-події, так і прямі файлові зміни або stale worktrees.
- Bad, because видаляються чинні команди `flow init`, `flow spec`, `flow release` і стан `.flow.json`, що є breaking change для наявних користувачів `flow`.
- Bad, because `flow plan` стає складнішим: один крок відповідає і за design, і за декомпозицію.
- Bad, because transcript не містить підтвердження механізму, який гарантує, що агент не забуде викликати `graph spawn` після створення дочірніх задач.
- Bad, because два тригери оркестрації — hook і daemon — потребують координації, щоб уникнути race condition при одночасному запуску.

## More Information

Файлова модель вузла:

- `tasks/<node>/task.md` — вхідний контракт вузла.
- `tasks/<node>/plan_001.md` — immutable numbered plan для атомарного шляху.
- `tasks/<node>/outputs_001.md` — результат виконання.
- `tasks/<node>/<child>/task.md` — дочірні вузли composite-шляху.
- `invalidated` — sentinel для невалідного вузла.

Frontmatter `task.md` містить `mode: human|agent`; default — `human`.

Старі елементи, що зникають або переписуються:

- Команди: `flow init`, `flow spec`, `flow release`.
- Стан: `.flow.json`.
- Директорії: `docs/specs/`, `docs/plans/`.
- Dispatcher modules: `lib/spec.mjs`, `lib/plan.mjs`, `lib/plan-panel.mjs`, `lib/flow-lock.mjs`, `lib/state-store.mjs`, `lib/snapshot.mjs`, `lib/artifact.mjs`.
- `commands.mjs` має прибрати `init`, `spec`, `release`.
- Правило `n-flow.mdc` потребує повного переписування.

Stage 2 для атомарного вузла: агент виконує роботу, пише `outputs_NNN.md`, викликає `flow verify`, після чого сигналізує `graph done`, `graph audit` або `graph failed`.

Оркестрація ready-вузлів:

- `.git/hooks/post-merge` → `n-cursor graph run --auto`.
- `graph run --auto` виконує one-shot scan DAG, знаходить ready-вузли і запускає worktrees з урахуванням `max_worktrees` з `.n-cursor.json`.
- `n-cursor watch` працює як daemon, слідкує за `tasks/`, активністю worktrees і stale worktrees.
- Конфіг згаданий у transcript: `.n-cursor.json` з полями `max_worktrees` і `stale_worktree_min`.

Архітектурний документ: `docs/думка.MD`, розділ «Інтеграція з `n-cursor flow`».
