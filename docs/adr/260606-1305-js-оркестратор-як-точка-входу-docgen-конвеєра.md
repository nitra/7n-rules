## ADR JS-оркестратор як точка входу docgen-конвеєра

## Context and Problem Statement
Попередній docgen-скіл покладався на LLM-агента (Claude / pi) для читання коду, генерації та складання документації. При переході на локальні моделі (gemma3:4b, gemma4:4b) one-shot підхід давав нестабільну якість: галюцинації фактів, витік сигнатур і внутрішніх імен, відсутні або перемішані секції.

## Considered Options
* LLM-агент (pi / Claude) як драйвер: читає файл і пише `.md` в одному або кількох промптах
* Детермінований JS-скрипт як оркестратор, локальна модель — лише "сервіс перефразування" (обраний варіант)

## Decision Outcome
Chosen option: "Детермінований JS-скрипт як оркестратор", because точкою входу стає `docgen-gen.mjs`, який детерміновано екстрагує факти (`docgen-extract.mjs`), формує точкові промпти (`docgen-prompts.mjs`), викликає ollama та збирає результат; LLM отримує лише вузьку задачу на перефразування конкретних фактів. Цей патерн зафіксовано в `scripts.mdc`: «максимум логіки — в CLI, 0 токенів; агент отримує лише зріз».

### Consequences
* Good, because transcript фіксує очікувану користь: структура і факти (назви експортів, маркери поведінки, перелік пропускань) тримаються JS-кодом, а не моделлю — вона не може вигадати назву або втратити секцію.
* Bad, because складність зросла: замість одного промпта — `docgen-extract.mjs` + `docgen-prompts.mjs` + `docgen-gen.mjs` + `docgen-assemble` (Stage 3); більше точок відмови. Transcript фіксує конкретні баги під час реалізації (жадібний JSDoc-regex, неправильний відносний імпорт у bench-скрипті).

## More Information
Файли: `npm/skills/docgen/js/docgen-gen.mjs`, `npm/skills/docgen/js/docgen-extract.mjs`, `npm/skills/docgen/js/docgen-prompts.mjs`. Ollama API викликається напряму через `POST /api/chat` (без pi) — це рішення прийнято явно у дискусії: «pi тут зайвий».

## ADR Секційно-мінімальний контекст — код лише у секції `Поведінка`

## Context and Problem Statement
v1 оркестрованого конвеєра надсилав повний вихідний код у кожну секцію (Огляд / Поведінка / API / Гарантії). На 8 GB Apple M2 кожен з 4 секційних викликів повторно інгестував код → KV-cache між stateless `/api/chat` не перевикористовується → оркестрований режим у 2–5× повільніший за one-shot.

## Considered Options
* Спільний system-контекст (код + факти) для всіх секцій — v1 (повільно через повторний інгест)
* Секційно-мінімальний контекст: код — лише в `Поведінку`; Огляд / API / Гарантії отримують тільки факт-лист (обраний варіант)
* Персистентна multi-turn сесія (code у system раз, секції — turn-by-turn)

## Decision Outcome
Chosen option: "Секційно-мінімальний контекст", because `Огляд` потребує лише header-коментар, `Публічний API` — список експортів з JSDoc (вже в факт-листі), `Гарантії` — лише маркери (крихітні). Скорочення інгесту зробило час конкурентним: overlay-paths: 310с → 77с (×4), k8s-tree: 141с → 55с.

### Consequences
* Good, because transcript фіксує: після v2 час orch ≈ oneshot (firebase 55с/46с, overlay 56с/58с, k8s 55с/68с), при цьому якість g3 ORCH зберігається на рівні ~92% проти ~47% g3 ONE на тих самих файлах.
* Bad, because gemma4:4b (Gemma 3n E4B) повертає порожній рядок (`""`) для секцій без коду в контексті — перевірено обома форматами (split system/user і merged single user). Секційно-мінімальний підхід для gemma4:4b непрацездатний; вона обмежена one-shot режимом.

## More Information
Функція `sectionMessages` у `docgen-prompts.mjs` реалізує розподіл: `behavior` — з `src`; `overview` / `api` / `guarantees` — лише `factsSummary`. Stage-2 `stripSignatures` застосовується після кожного секційного виклику детерміновано (0 токенів).

## ADR Негативні маркери в факт-листі для придушення галюцинацій

## Context and Problem Statement
Навіть після секційно-мінімального контексту v2 модель gemma3:4b в секції `Гарантії` вигадувала кеш для файлів без кешування (`firebase_hosting`, `overlay-paths`). Модель патерн-матчить «гарантії → кеш» незалежно від фактів у промпті.

## Considered Options
* Лише позитивні маркери (`caches: true` якщо є) — недостатньо, модель ігнорує відсутність факту
* Явні негативні маркери (`'Кешування: НЕМАЄ — не згадуй кеш у гарантіях'`) — обраний варіант
* Додатковий LLM-критик (Stage 2 LLM): перевіряє вигадані твердження

## Decision Outcome
Chosen option: "Явні негативні маркери", because детермінована (0 токенів) зміна в `factsSummary` всередині `docgen-prompts.mjs` додає рядки `'Кешування: НЕМАЄ — не згадуй кеш у гарантіях'` та `'Робота з мережею: немає'` якщо відповідні маркери відсутні. Це дешевший і надійніший підхід, ніж LLM-критик.

### Consequences
* Good, because transcript фіксує: після додавання негативних маркерів cache-галюцинація в `Гарантіях` зникла у фінальному бенчі; k8s-tree (де `caches: true` — реальний факт) коректно описує кеш.
* Bad, because негативні маркери охоплюють лише відомі патерни галюцинацій (кеш, мережа); інші вигадані інваріанти (наприклад, «нормалізація зворотних слешів» у g4) залишаються без детермінованого фільтра. Transcript не містить підтвердження усунення цих периферійних галюцинацій.

## More Information
Реалізовано в `factsSummary()` у `npm/skills/docgen/js/docgen-prompts.mjs`. Паралельно реалізовано детермінований Stage-2 `stripSignatures` у `docgen-gen.mjs` (видаляє `name(args)→name` з прозового тексту). Закомічено як `45a7466c` з change-файлом `.changes/1780740228846-271939.md`.
