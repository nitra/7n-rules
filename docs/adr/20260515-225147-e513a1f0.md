## ADR Розмежування ролей `/n-fix` і `/n-lint` та реструктуризація `rules/<id>/js/`

**Контекст:** Скіл `/n-fix` мав крок 6, що запускав увесь ланцюжок `lint-js / lint-text / lint-style`, змішуючи структурну діагностику (`npx @nitra/cursor check`) із лінтом коду (`bun run lint`). Це призводило до надлишкового запуску важкого ESLint при кожній структурній правці і до неясності щодо того, яке з двох умінь відповідає за яку якість.

**Рішення/Процедура/Факт:**
- `/n-fix` — відповідає **лише за структуру** проєкту: щоб `.cursor/rules/` + `npx @nitra/cursor check` не давали `❌`. Не запускає `bun run lint`, не виправляє ESLint/stylelint/cspell/knip-порушення.
- `/n-lint` — відповідає за **чистоту коду**: `bun run lint` і всі його кроки. Те, що `/n-lint` діагностує, виправляється там само.
- Крок 6 у `npm/skills/fix/SKILL.md` і `.cursor/skills/n-fix/SKILL.md` — видалено повністю; додано секцію **«Скоуп»** з явною забороною запускати `bun run lint` із цього скілу.
- Структура директорій правил змінена з `rules/<id>/js/` на три паралельних сиблінги: `fix/<concern>/` (JS-діагностика для `n-cursor check`), `lint/` (CLI-entry для `bun run lint-X`), `policy/<concern>/` (Rego-діагностика для `n-cursor check`, залишилась без змін). Усі 26 правил мігровано через три фази: (1) dual-mode інфраструктура в `discover-checkable-rules.mjs` і `run-rule.mjs`; (2) `git mv js/<concern>/ fix/<concern>/` + `git mv js/lint.mjs lint/lint.mjs` для кожного правила; (3) прибирання legacy fallback `rootDir ?? 'js'`.
- **Принцип:** технологія реалізації визначає директорію — JS у `fix/`, JS-орекстратор лінту у `lint/`, Rego у `policy/`.

**Обґрунтування:** Один скіл — одна відповідальність. `/n-fix` не повинен запускати важкий (≈2 хв) `bun run lint` при кожній структурній правці. Rego і JS-concerns технологічно різні (різні runner-и, namespace-конвенції, роль `target.json`), але обидва є fix-каналом — тому `policy/` лишається паралельним сиблінгом, не вкладеним у `fix/`. Нова структура директорій робить призначення кожного файлу очевидним без читання вмісту.

**Розглянуті альтернативи:**
- Вкладити `policy/` під `fix/policy/` (variant B) — відхилено: зайвий рівень nesting, rego-блок розчиняється серед JS-сусідів; крім того, rego і JS мають різні CLI-runner-и.
- Вкласти rego-концерни плоско у `fix/<concern>/` поруч із `check.mjs` — відхилено: ризик колізії імен, змішування двох різних рантайм-технологій в одному каталозі.
- Залишити `js/` без змін із делегуванням `/n-lint` із кроку 6 — відхилено: дублювання відповідальності між скілами.

**Зачіпає:** `npm/skills/fix/SKILL.md`, `.cursor/skills/n-fix/SKILL.md`, `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/run-rule.mjs`, `npm/bin/n-cursor.js` (CLI-imports lint entry-points), `npm/rules/<all 26 rules>/` (структура директорій), `npm/README.md` (секція «Структура пакету»), `.cursor/rules/scripts.mdc`.

## ADR Видалення `lint-conftest.mjs`

**Контекст:** `npm/scripts/lint-conftest.mjs` і `bun run lint-conftest` у кореневому `package.json` дублювали функціональність, яку `npx @nitra/cursor check` вже надає через `discoverCheckableRules` + `runConftestBatch`. Обидва інструменти ітерували policy-концерни з `target.json` і запускали `conftest test`; різниця полягала лише у точці виклику (скрипт vs CLI).

**Рішення/Процедура/Факт:** `npm/scripts/lint-conftest.mjs` видалено. З кореневого `package.json` прибрано скрипт `lint-conftest` і його ланку з `lint`-chain. Усі посилання на `bun run lint-conftest` у `check.mjs` і `*.rego` файлах замінено на `npx @nitra/cursor check`. Документацію в `conftest.mdc` (крок 5 реєстрації нового rego-пакету) оновлено: `discoverCheckableRules` автоматично підхоплює policy-концерни за наявністю `target.json`, окремої реєстрації в TARGETS більше не потрібно. Conditional rego-пакети (без `target.json`, наприклад `tauri/vscode_extensions`, `graphql/vscode_extensions`) описані як «не auto-discoverable — викликаються JS-check через `runConftestBatch` безпосередньо».

**Обґрунтування:** Два шляхи для однієї операції (rego-перевірка policy-концернів) — джерело розбіжностей: `lint-conftest` міг бути не синхронізований зі списком TARGETS при додаванні нового правила. `n-cursor check` є єдиним авторитетним джерелом, бо сам виконує discovery через файлову систему.

**Розглянуті альтернативи:** Залишити `lint-conftest` як "fast-feedback" у watch-режимі — відхилено: `n-cursor check` вже достатньо швидкий, а подвоєння інструментів вимагає подвійної підтримки.

**Зачіпає:** `npm/scripts/lint-conftest.mjs` (видалено), `package.json#scripts.lint`, `.cursor/rules/conftest.mdc`, `.cursor/rules/scripts.mdc`, `npm/rules/abie/abie.mdc`, ~17 `check.mjs` і `*.rego` файлів у `npm/rules/`.
