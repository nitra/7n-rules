## ADR Конвенція розширень нових JS-файлів: `.mjs`/`.cjs` замість `.js`

## Context and Problem Statement
Нові JS-файли з розширенням `.js` залежать від `package.json#type` для визначення інтерпретації (ESM чи CJS), тому одна і та ж назва читається по-різному у різних пакетах монорепо. Потрібна конвенція, що усуває цю неоднозначність для всіх нових файлів — backend і frontend.

## Considered Options
* Явне розширення `.mjs`/`.cjs` для нових файлів, без автоматичної перевірки (інструкція в `.mdc`)
* Git-aware перевірка: фейлити на `.js`-файлах, що позначені як untracked/added у `git status`
* Warn-only перевірка на всі `.js` без фейлу
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Явне розширення `.mjs`/`.cjs` для нових файлів, без автоматичної перевірки (інструкція в `.mdc`)", because stateless-скан не відрізняє новий файл від існуючого, а git-aware підхід визнано складним і крихким; достатньо декларативної інструкції для агента.

### Consequences
* Good, because тип модуля стає однозначним незалежно від `package.json#type` у будь-якому пакеті монорепо.
* Bad, because автоматичного `fix`-у немає — конвенція покладається виключно на агента/розробника при створенні файлу; transcript фіксує це як усвідомлений компроміс.

## More Information
- Конвенція додана як нова секція в `npm/rules/js-lint/js-lint.mdc` (v1.29 → 1.30), прив'язана до наявної вимоги `"type": "module"`.
- Правило поширюється на backend і frontend: `src/`, тести `*.test.*`, `scripts/`, `src/conn/`.
- Існуючі `.js`-файли лишаються як є — масовий рефактор не передбачено.
- Change-файл: `npm/.changes/260611-1201.md` (minor / Added).

## ADR Стратегія оновлення `.js`-прикладів у документації правил: «Кошик 2»

## Context and Problem Statement
Після введення конвенції `.mjs`/`.cjs` приклади вихідного коду в `.mdc`-правилах показували `.js`, що суперечило б новій нормі. Водночас частина `.js`-прикладів — конфіги тулінгу, хардкодовані в чекерах — не могла бути перейменована без порушення узгодженості doc↔check.

## Considered Options
* Не чіпати жодних прикладів (тільки нова секція в `js-lint`)
* Переписати лише приклади «нового вихідного коду» — Кошик 2 (`src/conn/`, `src/utils/`, `store/`, `main.*`) — без торкання конфігів тулінгу
* Переписати всі `.js` у прикладах, включно з фіксацією чекерів/фікстур

## Decision Outcome
Chosen option: "Переписати лише приклади «нового вихідного коду» — Кошик 2", because конфіги тулінгу (`vite.config.js`, `vitest.config.js`, `eslint.config.js`) хардкодовані в чекерах (зокрема `test/js/vitest-config-pool-forks.mjs:20`, `test/js/stryker_config.mjs:357`) і перейменування зламало б узгодженість doc↔check при практично нульовій користі.

### Consequences
* Good, because приклади вихідного коду (`src/conn/pg.mjs`, `src/utils/col.mjs`, `store/customerPage.mjs`, `main.mjs` тощо) тепер ілюструють конвенцію; тести шаблонів пройшли (22 passed).
* Bad, because фікстури чекера `js-run` (`check-fixture.test.mjs`) досі містять `pg-write.js` — doc і фікстури частково розходяться; transcript фіксує це як відкритий нюанс, що може потребувати окремого кроку.

## More Information
- Змінено приклади в: `npm/rules/js-run/js-run.mdc` (v1.11 → 1.12), `npm/rules/js-bun-db/js-bun-db.mdc` (v1.14 → 1.15), `npm/rules/vue/vue.mdc` (v2.1 → 2.2).
