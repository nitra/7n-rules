## ADR auto.md → meta.json: уніфікований файл конфігурації скілів і правил

## Context and Problem Statement
Кожен скіл і кожне правило в `npm/skills/<id>/` та `npm/rules/<id>/` мали файл `auto.md` з умовою автоактивації у вигляді текстового рядка (`завжди`, `[bun]`, тощо). Розширення механізму (зокрема, нове поле `worktree` для скілів) у текстовому форматі ускладнює валідацію і парсинг, а наявність двох різних споживачів (`auto-skills.mjs`, `auto-rules.mjs`) без спільної схеми ускладнює подальшу еволюцію.

## Considered Options
* Зберегти `auto.md`, розширити синтаксис новими рядками
* Міграція на структурований `meta.json` (обраний варіант)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Міграція на структурований `meta.json`", because JSON легко валідувати JSON-схемою (репо вже практикує схеми в `npm/schemas/`), легко парсити програмно, і дозволяє безпечно розширити набір полів (зокрема, `worktree`) без змін у синтаксичному парсері.

### Consequences
* Good, because transcript фіксує очікувану користь: єдиний формат для скілів і правил; нові поля додаються без ламання парсера; валідація схемою замінює ad-hoc парсинг рядків.
* Good, because `scripts.mdc` і `auto-skills.mjs`/`auto-rules.mjs` отримують одне джерело правди замість двох різних форматів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
* Skills `meta.json`: `{ "auto": "завжди" | ["rule-id"], "worktree": true | false }` — обидва поля (`worktree` обовʼязкове для скілів).
* Rules `meta.json`: `{ "auto": "завжди" | ["rule-id"] }` — тільки `auto`; поле `worktree` відсутнє (правила не мають концепції worktree).
* Дві окремі JSON-схеми у `npm/schemas/`.
* `syncSkills` у `n-cursor.js`: при `worktree:true` вшиває ідемпотентний markdown-блок між маркерами `<!-- n-cursor:worktree:start -->` / `<!-- n-cursor:worktree:end -->` в копію `SKILL.md`; `meta.json` у `.cursor/skills/` не копіюється.
* Поле `worktree:true` також означає заборону паралельного запуску скіла (один інстанс за раз, поверх наявного `withLock`).
* Початкові значення `worktree`: `true` — `fix`, `taze`, `coverage-fix`, `fix-tests`, `adr-normalize`; `false` — `lint`, `start-check`, `llm-patch`, `publish-telegram`.
* Принцип вибору: `worktree:true` — для генеративних скілів; `worktree:false` — для реактивних (lint) і read-only.
* Міграція rules `auto.md` уніфікується одночасно зі скілами (29+ файлів); споживач — `auto-rules.mjs`.

## Update 2026-05-31

Деталі дизайну `meta.json` з transcript:

Структура: `{ "auto": "завжди" | "[rule,...]", "worktree": true | false }`. Якщо поле `worktree` відсутнє — вважається `false`.

Таблиця значень для всіх 9 скілів:

| Скіл | `auto` | `worktree` |
|---|---|---|
| `fix` | `"завжди"` | `true` |
| `lint` | `"завжди"` | `true` |
| `taze` | `"[bun]"` | `true` |
| `coverage-fix` | `"[test]"` | `true` |
| `fix-tests` | `"[test]"` | `true` |
| `adr-normalize` | `"[adr]"` | `true` |
| `start-check` | `"завжди"` | `false` |
| `llm-patch` | `"завжди"` | `false` |
| `publish-telegram` | `"завжди"` | `false` |

Механіка `syncSkills` (в `bin/n-cursor.js`): якщо `worktree: true`, вставляє `worktree: true` у YAML-frontmatter скопійованого `.cursor/skills/n-<id>/SKILL.md`. Файл `meta.json` не копіюється в проєкт (аналогічно `auto.md`). `auto-skills.mjs` перемикається з hand-crafted парсера `auto.md` на `JSON.parse(readFileSync(meta.json))` → поле `auto`; логіка `discoverSkillAutoActivation` не змінюється. Усі 9 `auto.md` під `npm/skills/` видаляються.

## Update 2026-05-31

### Семантика поля `worktree`: булеве C1 над enum C2

При виборі між булевим `worktree` (C1) та enum `required`/`allowed`/`forbidden` (C2) обрано C1: паралельність вже захищена `withLock` через git-common-dir; три стани C2 надлишкові — «заборонено» це `false`, а не окрема категорія. `worktree: true` автоматично означає «один інстанс за раз». Початкові значення: `true` — генеративні скіли (`fix`, `taze`, `coverage-fix`, `fix-tests`, `adr-normalize`); `false` — реактивні (`lint` — залежить від незакомічених змін checkout) і read-only (`llm-patch`, `publish-telegram`, `start-check`).

### Rules `auto.md` — міграція відкладена до окремого spec (G1)

`npm/rules/<id>/auto.md` у 29 правилах містить лише людиночитаний опис умови; фактична логіка автодетекту захардкоджена в `auto-rules.mjs` (`AUTO_RULE_ORDER`, `AUTO_RULE_DEPENDENCIES`, `collectAutoRuleFacts`, `autoRuleChecks`). Уніфікація F1 (проза 1:1 в `meta.json`) відхилена на користь повного data-driven рефакторингу G1 (умови в `meta.json`, реєстр предикатів для незводимих перевірок). G1 виноситься в окремий spec після завершення skills-міграції, оскільки приблизно подвоює обсяг змін. Тимчасово в репо співіснують два механізми — `rules/*/auto.md` і `skills/*/meta.json` — що прийнятно і явно зафіксовано в transcript.

## Update 2026-05-31

### Семантика поля `worktree` та D2-доставка

`worktree: true` — генеративні скіли (зміни з детермінованого джерела: `fix`, `taze`, `coverage-fix`, `fix-tests`, `adr-normalize`); `worktree: false` — реактивні (лінтять незакомічені зміни поточного checkout, як `lint`) і read-only (`llm-patch`, `publish-telegram`, `start-check`). `worktree: true` автоматично означає заборону паралельного запуску — серіалізується через наявний `withLock`.

При `worktree: true` у копію `.cursor/skills/n-<id>/SKILL.md` вшивається ідемпотентний блок між маркерами `<!-- n-cursor:worktree:start -->` / `<!-- n-cursor:worktree:end -->`. При `worktree: false` блок видаляється при ре-синку. Реалізація: `npm/scripts/lib/worktree-notice.mjs` (функція `injectWorktreeNotice`), виклик з `syncSkills` у `npm/bin/n-cursor.js`.

Spec B (data-driven автодетект 28 `rules` через `meta.json` з реєстром предикатів G1) відкладено в окремий цикл spec→plan→impl після завершення Spec A. Частина правил (≈8) матиме `{"auto":{"predicate":"gqlTaggedTemplate"}}` з реалізацією в реєстрі; решта (≈20) — декларативні (`anyFile`, `pathExists`, `requiresRules`).

## Update 2026-05-31

### Збереження літерала `"завжди"` замість `"always"`

Константа `ALWAYS_LITERAL = 'завжди'` захардкоджена в `npm/scripts/auto-skills.mjs`. Паралельна сесія помилково вписала `"always"` у spec і проміжний план; канонічний spec (коміт `79e38d1`) виправлено. `meta.json` зберігає `"завжди"` для 1:1 сумісності з парсером без рефакторингу runtime-коду — зміна літерала без синхронного оновлення `ALWAYS_LITERAL` зламала б авто-активацію без видимих тестових помилок.

### Видалення `auto.md` без deprecation-fallback

Check-концерн `npm/rules/npm-module/js/skill_meta.mjs` явно перевіряє відсутність `auto.md` як умову валідності скіла. Два паралельних формати для одного поля створюють подвійне джерело правди; `auto.md` = помилка валідатора, `meta.json` = норма.
