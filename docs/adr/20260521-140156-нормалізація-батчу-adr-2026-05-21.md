# Нормалізація батчу ADR 2026-05-21

**Status:** Accepted
**Date:** 2026-05-21

## Context and Problem Statement

У репозиторії накопичилось 66 ADR-чернеток (файли з `session:` у frontmatter). Скрипт `normalize-decisions.sh` має вбудований throttle — `ADR_NORMALIZE_THRESHOLD` (мінімальна кількість чернеток для запуску) і `ADR_NORMALIZE_MIN_INTERVAL_HOURS` (мінімальний інтервал між прогонами) — що блокує ручний масовий запуск поза межами автоматичного хука.

## Considered Options

- Обробити всі чернетки за один великий прогін
- Обробляти батчами по 10 із проміжним `git commit` після кожного батчу та dry-run перевіркою перед першим реальним прогоном
- Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Обробляти батчами по 10 із проміжним `git commit` після кожного батчу", because скрипт визначає batch size=10 (найстаріші чернетки), коміт після кожного батчу ізолює зміни і кожен результат зберігається в git незалежно від збоїв у наступних батчах; обхід throttle: `ADR_NORMALIZE_THRESHOLD=0 ADR_NORMALIZE_MIN_INTERVAL_HOURS=0`.

### Consequences

- Good, because кожен батч атомарно закомічується; часткові результати не втрачаються при збої LLM — батч 3 повернув хибний JSON, але попередні коміти (`7c62d53`, `c45b8c9`) збереглися.
- Bad, because throttle вимикається на весь час сесії — transcript не містить підтвердження щодо вартості або лімітів API.
- Bad, because у батчі 3 LLM повернув некоректний JSON — знадобився ручний retry; автоматичного retry в скрипті немає.
- Neutral, because за 7 батчів кількість чернеток скоротилась із 66 до 17; batch size фіксований — 10 чернеток за один прогін.

## Update 2026-05-21

### Стратегія: послідовні батчі по 10 із комітом після кожного

При обробці 66 чернеток розглядалися варіанти:
- Один масовий прогон усіх чернеток — відхилено (складно відкотити окремий збій).
- Один коміт після завершення всіх батчів — відхилено (ті самі причини).
- Послідовні батчі по 10 із окремим комітом після кожного — обрано: користувач явно підтвердив «Закомітити й продовжити» після першого батчу.

### Деталі виконання: збої та відновлення

Процес обробки 66 чернеток у 10 батчах (по 10, останній — 1 чернетка):
