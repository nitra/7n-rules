## ADR Примусова заборона ручного бампу `version` через check-функцію

## Context and Problem Statement
У монорепо `@nitra/cursor` паралельні feature-гілки конфліктували на полі `version` у `package.json`, бо субагенти самостійно бампили `version` (спонукані правилом `n-changelog.mdc` v3.0, яке містило legacy-дозвіл ручного bump, та перевіркою `checkDirtyNpmRequiresVersionBump` у `npm-module`, яка **падала** за відсутності bump). Потрібно гарантувати, що `version` змінює **лише** CI-крок `release.mjs`.

## Considered Options
* Лише документація — прибрати legacy-рядок з правила, нічого більше не змінювати.
* Документація + check активно відхиляє будь-який дрейф `version` поза CI — зміна `consistency.mjs` так, що `Vbase ≠ Vcurrent` або `Vpublished ≠ Vcurrent` → `fail` на **будь-якій** гілці, навіть із change-файлом (обраний варіант).
* Окремий write-time блок / хук як незалежний guard.

## Decision Outcome
Chosen option: "Документація + check активно відхиляє будь-який дрейф `version`", because наявний pre-commit hook `npm-changelog` вже запускає `check changelog`, тому новий `fail` у `consistency.mjs` автоматично блокує коміт без окремого механізму; legacy-лазівка у правилі видаляється одночасно, щоб агенти більше не читали «можна вручну».

### Consequences
* Good, because `consistency.mjs` перевіряє drift **до** change-file check, тому ручний bump не проходить навіть за наявності change-файлу (transcript фіксує явну вимогу: «навіть із change-файлом»).
* Good, because правило `n-changelog.mdc` оновлено до v3.1 — прибрано legacy-рядок, STOP-блок став категоричним; синхронізовано дзеркало `.cursor/rules/n-changelog.mdc`.
* Bad, because у `npm-module` є перевірка `checkDirtyNpmRequiresVersionBump`, яка активно форсує ручний bump (падає без нього) — transcript зафіксував це як окрему незавершену задачу (Task 6), що залишилась заблокованою через конкурентну активність у тому ж робочому дереві.
