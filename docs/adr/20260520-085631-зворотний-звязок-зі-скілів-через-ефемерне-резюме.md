## ADR Зворотний звʼязок зі скілів через ефемерне резюме

## Context and Problem Statement
Скіли `@nitra/cursor` (наприклад `n-lint`, `n-fix`) виконуються в споживацьких репозиторіях і накопичують цінний сигнал про «тертя» з правилами та SKILL.md пакета — неоднозначні кроки, відсутні autofix, хибні спрацювання. Цей сигнал зникає після сесії, бо жодного каналу назад у пакет не існувало.

## Considered Options
* Persistent-інбокс: чернетки `docs/cursor-feedback/<timestamp>.md` + Stop-хук + батч-скіл `/n-feedback` → llm-patch у репо `@nitra/cursor`
* Ефемерне резюме наприкінці скілу: агент аналізує тертя у чаті, нічого не пише у файли

## Decision Outcome
Chosen option: "Ефемерне резюме наприкінці скілу", because користувач явно обрав ефемерний варіант — він простіший в реалізації (одне `alwaysApply`-правило, нуль інфраструктури) і достатній для ручного перенесення спостережень у пакет через `/n-llm-patch`.

### Consequences
* Good, because transcript фіксує очікувану користь: нульова інфраструктура — жодного хука, інбоксу чи фази 2; агент, що щойно виконав скіл, має найточніший контекст для аналізу тертя.
* Bad, because відсутня персистентність: без накопичення неможливо бачити частоту повторення тих самих проблем між сесіями; якщо не зреагувати одразу — резюме втрачене.

## More Information
Реалізація: `npm/rules/feedback/feedback.mdc` (`alwaysApply: true`, version 1.0), синкнутий як `.cursor/rules/n-feedback.mdc`; `"feedback"` додано до `.n-cursor.json`→`rules`; `AGENTS.md` і `CLAUDE.md` оновлено; версія пакета підвищена до `1.13.59`.
Схема звіту тертя (в тексті правила): `target` (rule|skill|check), `id`, `kind` (missing-check|ambiguous-doc|false-positive|no-autofix|recurring-pattern), `evidence`, `suggestion`.
Інструкція міститься в одному файлі-правилі — не в кожному `SKILL.md` окремо — щоб не порушувати обмеження jscpd на дублювання.
Для cross-repo маршруту сигналу у `@nitra/cursor` (якщо пізніше додати персистентність) обрано формат `n-llm-patch`.
