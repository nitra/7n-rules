# Заборона ручного бампу `version` — лише CI

**Status:** Accepted
**Date:** 2026-05-30

## Context and Problem Statement

У монорепо `@nitra/cursor` паралельні feature-гілки конфліктували на полі `version` у `package.json`. Субагенти (під час виконання `n-fix`, `n-coverage-fix` тощо), читаючи правило `n-changelog.mdc` v3.0, самостійно бампали `version`. Перевірка `consistency.mjs` давала pass при ручному bump, тому коміт проходив і паралельні гілки конфліктували. Перевірка `checkDirtyNpmRequiresVersionBump` у `npm-module` додатково форсувала ручний bump — падала за його відсутності. Потрібно гарантувати, що `version` змінює лише CI-крок `release.mjs` через агрегацію `.changes/*.md`.

## Considered Options

- Лише документація: прибрати legacy-рядок з правила, нічого не змінювати у `consistency.mjs`. Ручний bump лишається тихо прохідним.
- Документація + check активно відхиляє будь-який дрейф `version` поза CI: зміна `consistency.mjs` так, що `Vbase ≠ Vcurrent` → `fail` на будь-якій гілці, навіть із change-файлом.
- Окремий write-time guard / хук як незалежний механізм блокування.

## Decision Outcome

Chosen option: "Документація + check активно відхиляє будь-який дрейф `version`", because наявний pre-commit hook `npm-changelog` вже запускає `check changelog`, тому новий `fail` у `consistency.mjs` автоматично блокує коміт без окремого механізму. Legacy-лазівка у правилі видаляється одночасно. Заборона однорідна для всіх гілок, включно з `main`. CI-реліз-коміт не гейтиться changelog-workflow, тому release-коміт не блокується. Підхід 3 (окремий guard) відхилено як надлишковий.

### Consequences

- Good, because `consistency.mjs` перевіряє drift до change-file check — ручний bump не проходить навіть за наявності change-файлу.
- Good, because правило `n-changelog.mdc` оновлено до v3.1 — прибрано legacy-рядок, STOP-блок став категоричним; синхронізовано `.cursor/rules/n-changelog.mdc`.
- Bad, because у `npm-module` є перевірка `checkDirtyNpmRequiresVersionBump` (в `package_structure.mjs`), яка активно форсує ручний bump — залишилась незавершеною (Task 6) через конкурентну активність у тому ж робочому дереві.

## Update 2026-05-30

Префлайт-контекст: hook-файл `hk.pkl` — локальний pre-commit hook, що запускає лише `check changelog`, без жодного бампу версії. Саме він автоматично блокує коміт із незаконним bump через новий `fail` у `consistency.mjs`, без окремого механізму (Підхід 3 відхилено: pre-commit вже дає той самий ефект).

## Update 2026-05-30

Деталі реалізації: у `consistency.mjs` видалено функцію `verifyChangelogEntry` та її імпорт; функція `checkPublishedWorkspacePendingGitChanges` змінена так, що `Vbase ≠ Vcurrent` → `fail` незалежно від наявності change-файлу. Команда для розробника в повідомленні про помилку: `n-cursor change …` (поклади change-файл і відкоти version до базового).
