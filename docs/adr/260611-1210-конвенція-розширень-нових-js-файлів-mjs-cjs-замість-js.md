## ADR Конвенція розширень нових JS-файлів: `.mjs`/`.cjs` замість `.js`

## Context and Problem Statement
У монорепо Node.js/Bun-пакетів runtime (і тулінг) визначав тип модуля нового `.js`-файлу за `package.json#type`. Це призводило до неоднозначності: один і той самий патерн назви міг інтерпретуватись по-різному в різних workspace-пакетах залежно від наявності та значення поля `type`.

## Considered Options
* Нові файли — явне розширення `.mjs` (ESM) або `.cjs` (CJS), `.js` — заборонено
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Нові файли — явне розширення `.mjs`/`.cjs`", because явне розширення робить тип модуля однозначним незалежно від `package.json#type` будь-якого пакету; існуючі `.js` лишаються без змін (масовий рефактор не вимагається).

### Consequences
* Good, because runtime не потребує читати `package.json` для визначення типу модуля, що усуває неоднозначність між workspace-пакетами.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Конвенція зафіксована в `npm/rules/js-lint/js-lint.mdc` (v1.29 → 1.30) у секції «Розширення нових файлів — `.mjs`/`.cjs`, не `.js`». Стосується backend і frontend. Автоматичної stateless-перевірки немає — скан не відрізняє нові файли від існуючих. Change-файл: `npm/.changes/260611-1201.md` (minor / Added).

## ADR Дім правила `.mjs`-конвенції — `js-lint`, не `js-run`

## Context and Problem Statement
Потрібно було обрати правило, в якому розмістити нову конвенцію розширень файлів. Кандидати — існуючі `js-run` та `js-lint`, або нове правило.

## Considered Options
* Розмістити в `js-lint.mdc`
* Розмістити в `js-run.mdc`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Розмістити в `js-lint.mdc`", because `js-run` явно обмежений backend-проєктами, тоді як конвенція охоплює backend і frontend; `js-lint` уже має globs `**/*.{js,mjs,cjs,jsx,ts,tsx}` і загальний опис «Перевірка JavaScript коду», що відповідає крос-проєктній конвенції.

### Consequences
* Good, because transcript фіксує очікувану користь: єдине місце для конвенції, що охоплює обидва типи проєктів без дублювання.
* Bad, because `js-lint` зосереджений на інструментах (oxlint/ESLint/jscpd/knip), тоді як нова секція — неперевірювана інструкція; потенційна концептуальна неоднорідність правила.

## More Information
Зміна: `npm/rules/js-lint/js-lint.mdc` (v1.29 → 1.30). Секція явно позначена як конвенція без автоперевірки, щоб не створювати хибне очікування, що `fix js-lint` ловитиме `.js`.

## ADR Розподіл прикладів для оновлення (Кошик 2 vs конфіги тулінгу)

## Context and Problem Statement
Після прийняття `.mjs`-конвенції виникло питання: які з існуючих `.js`-прикладів у `.mdc`-файлах переписати на `.mjs`. Масовий рефактор ризикував порушити узгодженість doc↔check, бо частина назв диктується тулінгом або хардкодиться у чекерах.

## Considered Options
* Переписати лише приклади нового вихідного коду (Кошик 2): `src/conn/`, `src/utils/`, `main.js`, store-файли
* Переписати всі `.js` у прикладах, включно з `vite.config.js`, `vitest.config.js`, `eslint.config.js`
* Не переписувати нічого (тільки нова секція в `js-lint`)

## Decision Outcome
Chosen option: "Кошик 2 — лише приклади нового вихідного коду", because конфіги тулінгу (`vitest.config.js`, `eslint.config.js`) хардкодяться у власних чекерах (`test/js/vitest-config-pool-forks.mjs:20`, `test/js/stryker_config.mjs:357`); перейменування в доках при незмінних чекерах створило б розсинхрон doc↔check.

### Consequences
* Good, because transcript фіксує очікувану користь: doc↔check залишається узгодженим; усі 22 тести шаблонів/посилань пройшли після змін.
* Bad, because конфіги тулінгу (`vite.config.js`, `vitest.config.js`) лишаються `.js` у прикладах — потенційна неузгодженість із загальною конвенцією, але вона виправдана обмеженням тулінгу.

## More Information
Кошик 2 — оновлені файли: `npm/rules/js-run/js-run.mdc` (v1.11 → 1.12), `npm/rules/js-bun-db/js-bun-db.mdc` (v1.14 → 1.15), `npm/rules/vue/vue.mdc` (v2.1 → 2.2). Конфіги тулінгу, свідомо НЕ змінені: `vite.config.js` (але `abie/lib/overlay-paths.mjs` вже підтримує `vite.config.mjs`), `vitest.config.js`, `eslint.config.js`.

## ADR Оновлення інтеграційних фікстур `check-fixture.test.mjs` на `.mjs`

## Context and Problem Statement
Інтеграційні тести `js-run/js/tests/runtime/tests/check-fixture.test.mjs` створювали conn-файли з розширенням `.js` (`pg-write.js`, `mssql-write.js`), тоді як оновлені `.mdc`-приклади та конвенція вимагають `.mjs` для нового коду.

## Considered Options
* Оновити інтеграційні фікстури на `.mjs`, юніт-матрицю розширень лишити як є
* Лишити всі тести без змін
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Оновити інтеграційні фікстури, зберегти юніт-матрицю", because `conn-file-rules.mjs` використовує регекс `SOURCE_FILE_RE = /\.([cm]?[jt]sx?)$/u`, що приймає `.mjs`; юніт-тест `conn-file-rules.test.mjs` навмисно покриває всю матрицю розширень (`.js/.ts/.mjs/.cjs/.mts`) як доказ backward-compat — конвертація цих кейсів втратила б покриття.

* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Змінений файл: `npm/rules/js-run/js/tests/runtime/tests/check-fixture.test.mjs`. Перейменовані фікстури: `pg-write.js` → `pg-write.mjs`, `mssql-write.js` → `mssql-write.mjs`, `lib/connections/pg-write.js` → `lib/connections/pg-write.mjs`.
