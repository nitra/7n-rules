## ADR Розмежування ролей скілів n-fix і n-lint

**Контекст:** Скіл `/n-fix` містив крок 6, де послідовно запускав `lint-js`, `lint-text`, `lint-style` — частково дублюючи `/n-lint`. Це порушувало правило "один ESLint-процес на сесію" з `CLAUDE.md` і розмивало межу між перевіркою структури та перевіркою коду.

**Рішення/Процедура/Факт:** З `npm/skills/fix/SKILL.md` і `.cursor/skills/n-fix/SKILL.md` повністю видалено крок "Лінт коду". Додано секцію **«Скоуп»**, яка явно забороняє запускати `bun run lint` або будь-які `lint-*` скрипти з `/n-fix`. Перелік `lint-*` у `npm/rules/style-lint/style-lint.mdc` виправлено з посиланням на `/n-lint`. Усі правки зафіксовано в CHANGELOG 1.11.7. Ролі зафіксовані як: `/n-fix` = структура проєкту відповідає `.cursor/rules/`; `/n-lint` = код чистий згідно `bun run lint`.

**Обґрунтування:** Один скіл — одна відповідальність. `/n-fix` є рідкісною операцією (онбординг/міграція), `/n-lint` — регулярною (перед комітом). Запускати важкий `bun run lint` (~2 хв) при кожній структурній правці — зайвий overhead. Що `/n-lint` діагностує — там само й виправляється, без дублювання логіки в двох `SKILL.md`. Заборона паралельних ESLint-процесів у `CLAUDE.md` — додатковий аргумент проти виклику `lint-*` із `/n-fix`.

**Розглянуті альтернативи:** Злиття двох скілів в один (відхилено: різний час виконання, різна частота, ризик паралельних ESLint-запусків); делегування (крок 6 → `/n-lint`, реалізовано в 1.11.6, але потім визнано зайвим — 1.11.7).

**Зачіпає:** `npm/skills/fix/SKILL.md`, `.cursor/skills/n-fix/SKILL.md`, `npm/rules/style-lint/style-lint.mdc`, `npm/CHANGELOG.md`

## ADR Фаза 1: dual-mode підтримка `rules/<id>/{fix,lint}/` у discover/run-rule

**Контекст:** Вирішено реструктурувати `npm/rules/<rule>/js/` на дві семантичні директорії: `fix/` (концерни для `npx @nitra/cursor check` і `/n-fix`) та `lint/` (CLI-entry для `bun run lint-X`). Це велика міграція 26 правил, яку не можна зробити одним коміком без проміжного зламу `n-cursor check`.

**Рішення/Процедура/Факт:** Фаза 1 вводить dual-mode: `discover-checkable-rules.mjs` тепер сканує концерни у `fix/<concern>/` (новий формат) **і** `js/<concern>/` (legacy) в межах одного правила. `run-rule.mjs` відповідно обчислює шлях до `check.mjs` через нову функцію `resolveJsConcernPath`, яка перевіряє обидва варіанти. Фізичного переміщення файлів у фазі 1 немає — тільки інфраструктура. Тести обох файлів розширено сценаріями для нового формату (`fix/<concern>/check.mjs`), усі 684 тести проходять. Версія піднята до 1.11.9.

**Обґрунтування:** Поетапний підхід (3 фази) вибраний, щоб після кожної фази `bun test` і `npx @nitra/cursor check` залишались зеленими. Big-bang міграція всіх 26 правил одним PR ускладнила б ревʼю і будь-який зламаний import валив би все. Dual-mode дозволяє мігрувати правила по одному (фаза 2) без блокувань.

**Розглянуті альтернативи:** Big-bang міграція всіх 26 правил відразу (відхилено: важке ревʼю, ризик масового зламу імпортів).

**Зачіпає:** `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/discover-checkable-rules.test.mjs`, `npm/scripts/utils/run-rule.mjs`, `npm/scripts/utils/run-rule.test.mjs`, `npm/CHANGELOG.md`, `npm/package.json`
