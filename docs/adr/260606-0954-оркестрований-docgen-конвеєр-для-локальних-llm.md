## ADR Оркестрований docgen-конвеєр для локальних LLM

## Context and Problem Statement

Потрібно автоматично генерувати `.md`-документацію для JS-файлів проєкту, використовуючи локальні моделі через Ollama. One-shot підхід давав нестабільну якість: витік внутрішніх імен, сигнатур, stdlib-деталей, а також обрізання виводу на великих файлах. Постало питання, чи підніме якість декомпозиція на секційні LLM-виклики з детермінованим заземленням.

## Considered Options

* One-shot виклик моделі з system-prompt (прямий `/api/chat`)
* Секційно-оркестрований конвеєр: JS екстрагує факти → окремі LLM-виклики на кожну секцію → детермінована зборка
* Виклик через `pi` CLI (з вбудованим system-prompt)

## Decision Outcome

Chosen option: "Секційно-оркестрований конвеєр з детермінованою екстракцією та зборкою", because транскрипт фіксує прямий запит користувача і підтверджений результат: JS як точка входу готує точкові завдання моделі і збирає результат, а модель виступає лише сервісом перефразування.

Конвеєр складається з:
- `docgen-extract.mjs` — Stage 0: детермінована екстракція факт-листа (0 токенів): список `export`-ів, JSDoc, stdlib vs internal imports, поведінкові маркери (`readOnly`, `catchesErrors`, `skips`, `caches`), внутрішні символи для бану
- `docgen-prompts.mjs` — Stage 1: секційно-мінімальний контекст (v2): код іде **лише** у секцію `Поведінка`, решта секцій (`Огляд`, `Публічний API`, `Гарантії`) отримують тільки факт-лист
- `docgen-gen.mjs` — оркестратор: Stage 1 → LLM → Stage 2 (детермінований зріз сигнатур `(\w+, …)` та бан внутрішніх імен) → Stage 3 (фіксована зборка заголовків)
- Ollama викликається напряму через `/api/chat` (без `pi` CLI у гарячому циклі)

Усі файли розташовані у worktree `.worktrees/feat-docgen-orchestrator-pi/npm/skills/docgen/js/`.

### Consequences

* Good, because transcript фіксує очікувану користь: gemma3:4b підняла якість з ~80% (one-shot) до ~86% (оркестровано) при time-competitive часі (orch ≈ oneshot після v2 секційно-мінімального контексту).
* Good, because детермінований Stage-2 лінт (зріз сигнатур, бан внутрішніх імен) усуває цілий клас помилок без LLM-токенів і є модель-агностичним.
* Bad, because оркестрований режим **несумісний з `gemma4:4b`** (Gemma 3n E4B повертає порожній рядок для секцій на факт-листі без коду); для `gemma4:4b` рекомендований one-shot + Stage-2 лінт.
* Bad, because залишковий дефект `gemma3:4b`: галюцинація «кешу» у секції `Гарантії` для файлів без кешування — через відсутність негативних маркерів (`Кешування: немає`) у факт-листі.

## More Information

- Виміряний порівняльний результат: `gemma3:4b` one-shot ~80%, orchestrated ~86%, `gemma4:4b` one-shot ~92%
- `gemma4:4b` — канонічний alias (`ollama cp batiai/gemma4-e4b:q4 gemma4:4b`), ID `d682bf87e3a3`, 5.3 GB, 56%/44% CPU/GPU офлоад, ~11 tok/s на 8 GB
- `gemma3:4b` — 3.3 GB, 100% GPU, ~20 tok/s
- pi-провайдер `ollama` сконфігуровано у `~/.pi/agent/models.json`: `gemma4:4b` — перша модель, `gemma3:4b` — друга
- Знайдена причина переваги `pi` над «голим» one-shot — лише наявність вбудованого system-prompt, не архітектура інструмента (підтверджено прогоном C: прямий + system ≈ pi за якістю)
- Нефіксований дефект: негативні маркери відсутні у факт-листі → передавати `caches: false` поряд з `readOnly`, `catchesErrors`
