## ADR Кросплатформний CLI-інструмент для керування worktree замість харнес-специфічного EnterWorktree

## Context and Problem Statement
Команда використовувала харнес-специфічний інструмент `EnterWorktree`, який створює worktree від `origin/main` за замовчуванням, ігноруючи локальні коміти. Це призводило до втрати незмердженої роботи. Крім того, `EnterWorktree` доступний лише в конкретному харнесі (не в Cursor, не в терміналі), що унеможливлює уніфіковане використання конвенції `.worktrees/`.

## Considered Options
* Форма A — лише skill (SKILL.md з ручними інструкціями `git worktree add`)
* Форма B — лише CLI-команда (виконавець без skill)
* Форма C — CLI-команда (виконавець) + тонкий skill-вказівник

## Decision Outcome
Chosen option: "Форма C — CLI-команда + тонкий skill", because CLI-команда `npx @nitra/cursor worktree` працює ідентично в Claude Code, Cursor і терміналі (харнес-незалежно), а skill SKILL.md лише спрямовує агента до CLI без дублювання логіки. Логіка генерації інвентарного `.md`-опису перенесена в tool, що гарантує єдине джерело істини.

### Consequences
* Good, because transcript фіксує очікувану користь: усуває пастку `EnterWorktree` (втрата локальних комітів) і забезпечує уніфіковане виконання конвенції `.worktrees/` у всіх середовищах.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/scripts/worktree-cli.mjs`, `npm/scripts/lib/worktree.mjs`, `npm/skills/worktree/SKILL.md`, `npm/skills/worktree/meta.json`. Підключено у `npm/bin/n-cursor.js` як `case 'worktree'`. Реліз `@nitra/cursor@1.38.0` (`b4d50d6`).

## ADR Підкоманди worktree CLI: add + remove + list + prune (D3)

## Context and Problem Statement
Для CLI-команди `n-cursor worktree` потрібно визначити мінімально достатній набір підкоманд, що покриває CRUD-операції над worktree та інвентарними `.md`-файлами.

## Considered Options
* D1 — лише `add` + `remove`
* D2 — `add` + `remove` + `list`
* D3 — `add` + `remove` + `list` + `prune`

## Decision Outcome
Chosen option: "D3", because `prune` потрібен для очищення «осиротілих» станів (worktree видалено, але `.md` лишився, або навпаки) — це невідʼємна частина підтримки інвентаризації в консистентному стані. `list` виправданий власним форматом, що обʼєднує вивід `git worktree list` із вмістом описів.

### Consequences
* Good, because transcript фіксує очікувану користь: повний lifecycle управління worktree в одній команді.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовано в `npm/scripts/worktree-cli.mjs`. Тести: `npm/scripts/tests/worktree-cli.test.mjs` (6 тестів).

## ADR Санітизація слеша у дефіс для пласкої структури `.worktrees/`

## Context and Problem Statement
Git-гілки зазвичай містять слеш (наприклад, `feat/skill-meta`). Якщо каталог і файл опису дублюють цю структуру (`.worktrees/feat/skill-meta.md`), то `cat .worktrees/*.md` (нерекурсивний glob, зафіксований у конвенції `n-worktrees.mdc`) не знаходить вкладені файли.

## Considered Options
* E-slash-a — санітизувати слеш у дефіс для імені каталогу/файлу; git-гілка лишається з нормальним slash
* E-slash-b — зберігати slash-структуру (`.worktrees/feat/skill-meta/`); `list`/`prune` шукають описи рекурсивно

## Decision Outcome
Chosen option: "E-slash-a — санітизація слеша в дефіс", because зберігає пласку структуру `.worktrees/`, сумісну з `cat .worktrees/*.md` у конвенції, і відповідає прикладу `feat-skill-meta` у `n-worktrees.mdc`. Git-гілка (`feat/skill-meta`) лишається незміненою; лише шлях на диску санітизується.

### Consequences
* Good, because transcript фіксує очікувану користь: `cat .worktrees/*.md` завжди знаходить усі описи без рекурсивного пошуку.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовано у `sanitizeBranch()` в `npm/scripts/lib/worktree.mjs`. Тести: `npm/scripts/lib/tests/worktree.test.mjs`.

## ADR База гілки від поточного HEAD при `worktree add` (F1)

## Context and Problem Statement
`EnterWorktree` використовував `origin/main` як базу за замовчуванням, що тихо ігнорувало локальні коміти. Потрібно визначити, від якого ref нова гілка worktree відгалужується за замовчуванням.

## Considered Options
* F1 — завжди від поточного HEAD (без флагів)
* F2 — HEAD за замовчуванням + опційний `--from <ref>`
* F3 — дефолт від `origin/<default>` (поведінка EnterWorktree)

## Decision Outcome
Chosen option: "F1 — завжди від HEAD, без флагів", because найпоширеніший кейс — «відгалузити поточну роботу в ізоляцію» — саме HEAD. F3 відтворює пастку EnterWorktree. F2 ускладнює інтерфейс без YAGNI-обгрунтування на цьому етапі.

### Consequences
* Good, because transcript фіксує очікувану користь: унеможливлює тиху втрату локальних комітів.
* Bad, because transcript не містить підтверджених негативних наслідків. `--from` можна додати пізніше.

## More Information
`git worktree add .worktrees/<sanitized> -b <branch>` без ref у `npm/scripts/worktree-cli.mjs`.

## ADR Семантика `remove` (безпечний + `--force`) і `prune` (агресивний)

## Context and Problem Statement
Потрібно визначити семантику двох деструктивних підкоманд: чи `remove` відмовляється на брудному дереві, чи `prune` питає підтвердження перед видаленням.

## Considered Options
* `remove` H1 — без `--force` (ніколи не видаляє брудний)
* `remove` H2 — безпечний дефолт + опційний `--force` для брудного
* `prune` H-prune-a — лише звітує, видаляє з `--yes`
* `prune` H-prune-b — одразу чистить осиротілі `.md` і виконує `git worktree prune`

## Decision Outcome
Chosen option: "`remove` = H2 (безпечний + `--force`) і `prune` = H-prune-b (агресивний)", because `remove` повинен захищати від випадкової втрати незакомічених змін (H2), а `prune` виправляє структурні невідповідності, які вже стались — тут dry-run тільки відкладає проблему (H-prune-b).

### Consequences
* Good, because transcript фіксує очікувану користь: `remove` безпечний за замовчуванням; `prune` не вимагає двох викликів для очищення.
* Bad, because `prune` незворотно видаляє `.md` без підтвердження — потенційна втрата опису при хибному спрацюванні.

## More Information
Реалізовано в `npm/scripts/worktree-cli.mjs`. `git worktree remove` сам відмовляється на брудному дереві без `--force`.

## ADR Перейменування поля `auto` → `suggest` у `meta.json` правил

## Context and Problem Statement
При проектуванні `meta.json` для правил поле спочатку називалося `auto` за аналогією зі скілами. Однак `n-cursor fix` перевіряє **всі** правила незалежно від наявності поля. Поле відповідає лише за те, чи пропонується правило до додавання у `.n-cursor.json` при sync — а не за те, чи запускається перевірка.

## Considered Options
* `auto` — назва за аналогією зі скілами
* `suggest` — назва, що відображає фактичну семантику (пропозиція при sync)

## Decision Outcome
Chosen option: "`suggest`", because точно описує призначення: «запропонувати додати в `.n-cursor.json`» при sync. Значення `"завжди"` означає «пропонувати завжди, незалежно від умов файлової системи», а не «завжди запускати перевірку».

### Consequences
* Good, because transcript фіксує очікувану користь: усуває концептуальну плутанину між детекцією (коли пропонується) і виконанням (коли запускається).
* Bad, because transcript не містить підтверджених негативних наслідків; потребує перейменування у всіх `meta.json` скілів та відповідному коді.

## More Information
Рішення прийнято наприкінці сесії. Поточний стан: скіли вже мають поле `auto` (`npm/skills/*/meta.json`); правила ще не мігровані. Наступний крок — Spec B із полем `suggest`.
