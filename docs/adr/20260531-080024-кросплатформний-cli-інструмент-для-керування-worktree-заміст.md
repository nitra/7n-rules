## ADR Кросплатформний CLI-інструмент для керування worktree замість харнес-специфічного `EnterWorktree`

## Context and Problem Statement
Харнес Claude Code надає вбудований інструмент `EnterWorktree`, але він доступний лише в Claude Code і відсутній у Cursor та терміналі. Команда потребувала єдиного способу створення git-worktree з інвентарним файлом-описом, який би працював ідентично в обох середовищах.

## Considered Options
* **(A)** Лише CLI-підкоманда `n-cursor worktree`
* **(B)** Лише skill (markdown-інструкція для агента)
* **(C)** CLI-команда (виконавець) + тонкий skill, що вказує агенту на неї

## Decision Outcome
Chosen option: "**(C) CLI + тонкий skill**", because CLI за визначенням платформо-незалежний і дає ідентичну поведінку в Claude Code, Cursor і терміналі, тоді як skill-тільки підхід залежить від дисципліни агента і не гарантує однаковості.

### Consequences
* Good, because `npx @nitra/cursor worktree <sub>` виконується однаково у всіх середовищах без залежності від харнесу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовано в `npm/scripts/worktree-cli.mjs` (оркестратор), `npm/scripts/lib/worktree.mjs` (чиста логіка), `case 'worktree'` у `npm/bin/n-cursor.js`. Skill: `npm/skills/worktree/SKILL.md` + `meta.json` з `{ "worktree": false }`.

## ADR Семантика `remove --force` і агресивний `prune`

## Context and Problem Statement
Операції `remove` і `prune` незворотні; потрібно визначити, який дефолт безпечний для `remove` і чи `prune` повинен запитувати підтвердження.

## Considered Options
* **remove H1** — видаляє checkout + `.md`, без `--force`
* **remove H2** — безпечний дефолт (git відмовиться на брудному/незакомічному дереві) + опційний `--force`
* **prune H-prune-a** — за замовчуванням лише звітує; видаляє з `--yes`
* **prune H-prune-b** — агресивно: `git worktree prune` + негайно видаляє осиротілі `.md`

## Decision Outcome
Chosen option: "**remove = H2, prune = H-prune-b**", because для `remove` незворотність потребує явного `--force` при брудному дереві (git сам є запобіжником); `prune` прибирає лише осиротілі технічні артефакти (worktree вже немає), тому агресивний дефолт виправданий — очищення безпечне.

### Consequences
* Good, because transcript фіксує очікувану користь: `remove` без `--force` не знищує незакомічену роботу; `prune` не плодить «сміттєвих» `.md` після видалення worktree.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовано в `npm/scripts/worktree-cli.mjs`: `remove` делегує в `git worktree remove [--force]`; `prune` викликає `git worktree prune` і видаляє `.md`-файли без відповідного worktree через `findOrphanDescFiles`. Гілку git `remove` завжди лишає.

## ADR Нормалізація правила `worktree` — канонічне джерело у пакеті, pure-doc

## Context and Problem Statement
Правило `n-worktrees.mdc` існувало лише локально в `.cursor/rules/`, без канонічного джерела в `npm/rules/`. Це порушувало інваріант репо: `discoverBundledRuleNames` не знає про правило → sync може видалити його при наступному запуску. Також виникло питання, чи додавати programmatic check структури `.worktrees/`.

## Considered Options
* **(J1)** Pure-doc: лише `npm/rules/worktree/worktree.mdc` (конвенція + посилання на CLI), без `check.mjs`
* **(J2)** Doc + programmatic check: валідує, що worktree лежать у `.worktrees/`, кожен має `.md`, немає осиротілих

## Decision Outcome
Chosen option: "**J1 pure-doc**", because CLI сам гарантує правильне розташування і `.md`-опис при кожному `add`; окремий `check` дублював би гарантії інструмента. YAGNI.

### Consequences
* Good, because transcript фіксує очікувану користь: sync тепер розповсюджує правило через звичайний механізм `discoverBundledRuleNames`; назва нормалізована на `worktree` (однина, як підкоманда CLI і каталог skill).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/rules/worktree/worktree.mdc` (нове канонічне джерело), видалено `.cursor/rules/n-worktrees.mdc` (сирота), sync створив `.cursor/rules/n-worktree.mdc`. `CLAUDE.md` і `AGENTS.md` оновлені (`n-worktrees.mdc` → `n-worktree.mdc`). `worktree` додано до `rules` у `.n-cursor.json`.
