## ADR

## Context and Problem Statement

Розробляється CLI-утиліта `docgen-gen.mjs` для автоматичної генерації документації. Раніше вона виключно викликала локальний `ollama` безпосередньо через HTTP. Виникло питання: як зробити утиліту придатною для запуску в середовищах де локальний `ollama` відсутній (K8s, CI), при цьому не погіршуючи якість і швидкість там де `ollama` є.

## Considered Options

* `checkOllama()` — перевірка доступності ollama на старті й вибір між ollama HTTP та pi orchestrated
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "`checkOllama()` + pi fallback", because benchmark показав: ollama HTTP швидше в 2× (68s vs 135s), але pi orchestrated дає рівнозначну або кращу якість (97.0% vs 96.1%) і не має проблеми `cache-hallucination`. При відсутності ollama (K8s без GPU, CI) — pi обробляє через cloud API без змін у коді.

### Consequences

* Good, because transcript фіксує очікувану користь: нульові зміни коду для K8s деплою — достатньо виставити `OLLAMA_HOST`.
* Good, because transcript фіксує очікувану користь: складні файли (sym≥2) у pi дають +10% якості.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Реалізовано в `npm/skills/docgen/js/docgen-gen.mjs`:
- `_ollamaAvailable` кешована змінна (рядки ~100-108)
- `checkOllama()` — GET `/api/tags` з `AbortSignal.timeout(3000)`, повертає `bool` (рядки ~99-108)
- `OLLAMA_HOST` env var (дефолт `http://localhost:11434`)
- Routing: `const useOllama = model && await checkOllama()` (~рядок 267)
- Fallback: `piOrchestrated` якщо ollama недоступний

K8s deployment: `OLLAMA_HOST=http://kubeai-svc.kubeai.svc:11434` (kubeai OllamaEngine) без змін у коді.
