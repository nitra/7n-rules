## ADR MADR-каркас у JS-збірку — LLM повертає лише зміст секцій

**Status:** Accepted
**Date:** 2026-06-16

## Context and Problem Statement
У `GEN_SYS` промпті `npm/scripts/lib/adr/normalize-pipeline.mjs` повний MADR-каркас (заголовки секцій, `**Status:** Accepted`, `**Date:**`, fallback-фрази «Інші варіанти не обговорювалися.» та «Додаткової інформації не зафіксовано.») диктувався моделі — і потім перевірявся `validateMadr` regex-ами. Класична петля: промпт диктує формат → код валідує формат → ретрай при розбіжності. Аналогічно `## Update <date>` у `MERGE_SYS` зашивався в промпт і валідувався `RE_UPDATE_HEAD`.

## Considered Options
* «Чисті поля»: LLM повертає JSON без жодного markdown (prose без backticks, рядки без розмітки)
* «Гібрид»: JS будує увесь каркас (`assembleMadr`), LLM повертає зміст секцій як markdown-prose у JSON-полях (`context`, `options[]`, `outcome`, `consequences[]`, `more_info`)

## Decision Outcome
Chosen option: "Гібрид", because A/B-прогін на 6 реальних чернетках показав, що «чисті поля» є лосіі — модель вилучає inline-code backticks і згортає багаторядкові секції у плаский рядок; гібрид дав 5/5 IDENTICAL дифів між старим і новим виходом.

### Consequences
* Good, because JS (`assembleMadr`) гарантує каркас без ретраїв за формат — `validateMadr` для `genMadr` стає зайвим.
* Good, because `## Update <date>` у `MERGE_SYS` тепер будує JS (`genMerge`), усуваючи ще одну форматну валідацію.
* Good, because transcript фіксує очікувану користь: нові функції `assembleMadr`, `normalizeSections`, `madrDate` покриті 8 детермінованими юніт-тестами (21/21 зелені).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
