## ADR Спостережувані сліди як портативний сигнал покращень замість raw reasoning

## Context and Problem Statement
Виникла ідея: після виклику скілів із моделями, що мають reasoning, фіксувати лог міркувань і на основі нього робити висновки щодо покращення проєкту. Питання поширилося на Codex і Cursor — чи можна один підхід застосувати до всіх трьох інструментів.

## Considered Options
* Витягувати raw `thinking`-блоки з JSONL-транскриптів (тільки Claude Code)
* Витягувати зашифровані `encrypted_content` з Codex rollout-файлів (неможливо читати)
* Скрапити SQLite `state.vscdb` у Cursor (недокументований формат, без hooks)
* Використовувати спостережувані сліди (tool-calls, помилки, повтори, фінальні дифи) — портативно через всі три інструменти

## Decision Outcome
Chosen option: "Спостережувані сліди як основний сигнал для крос-інструментного аналізу", because raw reasoning доступний лише в Claude Code (Anthropic віддає його сирим у JSONL), тоді як у Codex він зашифрований OpenAI (`encrypted_content`), а в Cursor доступний лише як summary без lifecycle-hooks; спостережувані сліди — послідовності tool-calls, exit-коди, помилки, повтори — присутні в усіх трьох і дають той самий діагностичний сигнал («де правила/скіли недовизначені»).

### Consequences
* Good, because підхід портативний: Claude Code `thinking`-блоки, Codex `rollout-*.jsonl` (tool-calls + messages), Cursor `state.vscdb` — можна звести до єдиного формату для LLM-проходу.
* Good, because transcript фіксує очікувану користь: сліди описують **фактичну поведінку** (де агент кружляв, ретраїв, отримував exit_code=1), а не самоопис моделі, що часто є більш інформативним сигналом.
* Bad, because для Cursor потрібно скрапити недокументований SQLite-формат (`globalStorage/`, `workspaceStorage/<hash>/state.vscdb`) — це крихке і може ламатись при оновленнях Cursor.
* Bad, because відмова від raw reasoning означає втрату сигналу про внутрішні вагання моделі, які не завжди проявляються у поведінці (наприклад, модель обрала правильний шлях, але через неправильне reasoning).

## ADR Зберігання raw reasoning Claude Code через існуючий Stop-hook конвеєр

## Context and Problem Statement
Потрібно вирішити, як саме збирати `thinking`-блоки з сесій Claude Code та атрибутувати їх до конкретних скілів для подальшого LLM-проходу з висновками щодо покращення проєкту.

## Considered Options
* Розширити існуючі Stop-hooks (`capture-decisions.sh`, `normalize-decisions.sh`) для витягування thinking-блоків
* Написати окремий незалежний збирач reasoning
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Розширення існуючого Stop-hook конвеєра", because в проєкті вже є `capture-decisions.sh` (Stop-hook) і `normalize-decisions.sh` (Stop-hook), які реалізують той самий патерн; додавання збору reasoning є розширенням наявного механізму, а не новим.

### Consequences
* Good, because transcript фіксує очікувану користь: атрибуція thinking-блоків до скіла можлива через кореляцію `tool_use` типу `Skill` у JSONL з наступними `thinking`-блоками до завершення скіла.
* Good, because підхід описаний як накопичення в append-лог (аналогічно `capture-decisions.log`) з окремим LLM-проходом — не аналізувати щоразу, а дистилювати пакетно.
* Bad, because reasoning дуже багатослівний; без жорсткого LLM-промпта на дистиляцію сигнал буде зашумлений. Цінність лише у вузькому фокусі: «моменти, де модель вагалась/помилялась/не знайшла правило».
* Bad, because наявність і формат `thinking`-блоків залежить від моделі та версії — не всі моделі пишуть розгорнутий reasoning, і це може змінюватись між релізами Claude.
