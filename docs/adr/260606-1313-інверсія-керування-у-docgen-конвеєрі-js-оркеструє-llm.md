## ADR Інверсія керування у docgen-конвеєрі: JS оркеструє LLM

## Context and Problem Statement
При генерації документації локальними LLM (gemma3:4b, gemma4:4b через Ollama) агент-підхід (Claude/pi як драйвер) давав нестабільний формат: відсутні заголовки секцій, витік внутрішніх імен, галюцинації в гарантіях. Потрібен підхід, який дозволить отримати стабільну структуру документу без залежності від здатності 4B-моделі слідувати складним інструкціям.

## Considered Options
* **Агент-підхід**: pi або Claude агент читає файл і генерує весь `.md` в одному виклику
* **JS-оркестратор**: детермінований JS-скрипт готує точкові завдання, викликає LLM посекційно, детерміновано збирає результат

## Decision Outcome
Chosen option: "JS-оркестратор", because користувач обрав «точкою входу JS, який підготує вже точкові завдання до локальної моделі а потім збере все докупи» — модель стає сервісом перефразування, JS володіє структурою та фактами.

### Consequences
* Good, because transcript фіксує очікувану користь: структурна дисципліна гарантується кодом (`assemble()` зі фіксованими `##`-заголовками), незалежно від поведінки моделі; g3 ORCH отримав 12/12 на firebase (еталонний результат).
* Bad, because transcript фіксує: складніший код (3 окремі модулі `docgen-extract`, `docgen-prompts`, `docgen-gen`) та більше точок відмови, ніж один one-shot виклик.

## More Information
Файли: `npm/skills/docgen/js/docgen-gen.mjs`, `npm/skills/docgen/js/docgen-prompts.mjs`, `npm/skills/docgen/js/docgen-extract.mjs`. Команда генерації: `generateDoc(filePath, { model, mode })`. Принцип закріплений у коментарях модулів як «JS володіє фактами/структурою, модель — лише прозою». Commit `17cfca32`.

## ADR Секційно-мінімальний контекст (v2): код тільки у секцію Поведінка

## Context and Problem Statement
У v1 конвеєра повний вихідний код подавався у КОЖНУ секційну інструкцію (Огляд/Поведінка/API/Гарантії). Це спричиняло повторний інгест коду 4 рази на файл: v1 в 3–5× повільніший за one-shot через відсутність надійного KV-cache між stateless `/api/chat` запитами на 8 GB M2.

## Considered Options
* **Повний код у кожній секції (v1)**: спільний system-prefix → ollama теоретично кешує, але на практиці — 310 с на overlay-paths
* **Секційно-мінімальний контекст (v2)**: код лише у секцію `behavior`; `overview`/`api`/`guarantees` отримують тільки компактний факт-лист

## Decision Outcome
Chosen option: "Секційно-мінімальний контекст (v2)", because вимірювання показало ×4 прискорення (overlay 310→77 с) при збереженні якості (~86%); факт-лист достатньо заземлює секції без коду.

### Consequences
* Good, because transcript фіксує очікувану користь: overlay-paths 310→77 с (×4), k8s-tree 141→55 с; сумарно g3 ORCH v2 = 157 с ≈ g3 one-shot = 158 с.
* Bad, because transcript фіксує: gemma4:4b (Gemma 3n E4B архітектура) повертає `""` для секцій без коду в контексті — секційно-мінімальний підхід з нею несумісний; gemma4 використовується лише в one-shot режимі.

## More Information
Файл: `npm/skills/docgen/js/docgen-prompts.mjs`, функції `sectionMessages()` та `oneShotMessages()`. Перевірка несумісності з gemma4: `node -e "..."` → `split system/user → "" / merged single user → ""`. Commit `45a7466c`.

## ADR Stage-0 детермінована екстракція фактів (0 токенів LLM)

## Context and Problem Statement
Локальні 4B-моделі галюцинували факти документації: вигадували назви функцій, вказували stdlib-залежності, описували неіснуюче кешування, витікали внутрішні символи (`walkDir`, `readAndParseYamlDocs`). Найбільший клас помилок — не «погана проза», а «невірні або відсутні факти».

## Considered Options
* **LLM-екстракція**: модель читає код і сама визначає факти в промпті
* **Детермінована JS-екстракція**: парсер на регулярних виразах витягує факти до першого LLM-виклику (0 токенів)

## Decision Outcome
Chosen option: "Детермінована JS-екстракція", because JS-парсер дає точні, не-галюциновані дані (список `export`-ів, stdlib/internal класифікація, JSDoc-заголовок, поведінкові маркери) та коштує 0 токенів.

### Consequences
* Good, because transcript фіксує очікувану користь: факт-лист для `overlay-paths` витягнув всі 6 експортів з описами з JSDoc — «API-секція практично написана» без LLM-виклику; маркери (`readOnly`, `caches`, `skips`) заземлюють гарантії.
* Bad, because transcript фіксує: Stage-0 реалізований лише для JS/MJS (regex/AST-парсер); `.vue`/`.py` деградують до one-shot без факт-листа.

## More Information
Файл: `npm/skills/docgen/js/docgen-extract.mjs`, функція `extractFacts(src, relPath)`. Структура факт-листа: `{ header, exports[], imports:{stdlib,npm,internal}, internalSymbols[], markers:{readOnly,caches,skips[],catchesErrors,network} }`. Commit `17cfca32` + оновлення `internalSymbols` в `45a7466c`.

## ADR Stage-2 детермінований зріз сигнатур (0 токенів LLM)

## Context and Problem Statement
Навіть при секційно-мінімальному контексті секція `Поведінка` (яка отримує повний код) дозволяла 4B-моделям витікати сигнатури функцій у проз у (`isUaKustomizationPath(rel)…`, `readAndParseYamlDocs`). Промпт-рівнева заборона («НЕ пиши аргументи у дужках») не тримала достатньо надійно.

## Considered Options
* **Промпт-заборона**: інструкція «без аргументів/дужок» у системному промпті
* **Детермінований пост-лінт**: `stripSignatures()` регулярним виразом після кожного LLM-виклику

## Decision Outcome
Chosen option: "Детермінований пост-лінт (Stage-2)", because пост-лінт є модель-агностичним і не вимагає додаткового LLM-виклику; застосовується і в orchestrated, і в one-shot режимах.

### Consequences
* Good, because transcript фіксує очікувану користь: overlay `Поведінка` після Stage-2 — `isUaKustomizationPath: Перевіряє…` (без `(rel)`); сигнатурні витоки зникли механічно.
* Bad, because `stripSignatures` `/([`\w$.]+)\([^()]*\)/g` не розрізняє «опис виклику в прозі» від «назва функції у тексті» — може зрізати корисні дужки у проявах, що не є сигнатурами. Transcript не містить підтвердженого кейсу такого побічного ефекту.

## More Information
Файл: `npm/skills/docgen/js/docgen-gen.mjs`, функція `stripSignatures(text)`. Застосовується у `generateOrchestrated()` та `generateOneShot()`. Commit `45a7466c`.

## ADR Негативні маркери для запобігання cache-галюцинаціям у Гарантіях

## Context and Problem Statement
Секція «Гарантії» генерувалась із самого факт-листа (без коду). 4B-моделі стабільно вигадували «кешування» навіть для файлів без кешу (`firebase_hosting`, `overlay-paths`) — pattern-matching «гарантії → кеш» незалежно від контексту.

## Considered Options
* **Промпт «згадуй кеш ЛИШЕ якщо є у фактах»**: умовна заборона у системному промпті
* **Явний негативний маркер у факт-листі**: `'Кешування: НЕМАЄ — не згадуй кеш у гарантіях'` поряд із `'Кешування: так, у межах прогону'`

## Decision Outcome
Chosen option: "Явний негативний маркер + умовна заборона у промпті", because подвійне заземлення (факт + інструкція) надійніше однієї умовної заборони для 4B-моделей.

### Consequences
* Good, because transcript фіксує очікувану користь: g3_ORCH__firebase_hosting Гарантії в фінальному бенчі містять «Не використовує кешування» — галюцинація відсутня.
* Bad, because transcript не містить підтвердженого негативного наслідку.

## More Information
Файл: `npm/skills/docgen/js/docgen-prompts.mjs`, функція `factsSummary(facts)`. Рядки: `lines.push(m.caches ? 'Кешування: так, у межах прогону' : 'Кешування: НЕМАЄ — не згадуй кеш у гарантіях')`. Commit `45a7466c`.
