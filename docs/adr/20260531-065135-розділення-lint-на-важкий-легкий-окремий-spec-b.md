### Consequences
* Good, because Spec A лишається сфокусованим (9 скілів, meta.json, worktree); transcript фіксує очікувану користь: «кожна задача має чистий власний цикл spec→plan→impl».

## ADR Міграція `skills/*/auto.md` → `meta.json` + поле `worktree`

## Context and Problem Statement
Скіли в `npm/skills/<id>/` мали `auto.md` — плоский текстовий файл з умовою автоактивації (`завжди` / `[rule,...]`). З появою потреби в полі `worktree` однофайловий підхід вичерпав себе: розширення вимагало б або другого файлу, або нестандартного парсингу.

## Considered Options
* Залишити `auto.md`, додати окремий `worktree.md`.
* Замінити `auto.md` на структурований `meta.json` з полями `auto` і `worktree`.

## Decision Outcome
Chosen option: "Замінити `auto.md` на `meta.json`", because структурований JSON дає одне джерело правди для обох полів, підтримує JSON-схему-валідацію і не залишає `auto.md` як deprecation-fallback.

### Consequences
* Good, because transcript фіксує очікувану користь: єдиний `meta.json` замість двох файлів; `check`-перевірка підтверджує валідний формат; `auto.md` більше не існує.
* Bad, because `auto-skills.mjs` потребує міграції парсера: вхід змінюється з рядка markdown на JSON-значення; тести `auto-skills.test.mjs` потребують нових фікстур.

## More Information
Формат: `{ "auto": "завжди", "worktree": true }`. Поле `auto`: `"завжди"` | `["adr"]` | відсутнє (opt-in). Поле `worktree`: обовʼязкове булеве. Розкладка `worktree:true` — `fix`, `taze`, `coverage-fix`, `fix-tests`, `adr-normalize`; `worktree:false` — `lint`, `llm-patch`, `publish-telegram`, `start-check`. JSON-схема: `npm/schemas/skill-meta.json`. D2-sync: ідемпотентний блок між маркерами `<!-- n-cursor:worktree:start -->` / `<!-- n-cursor:worktree:end -->` у `.cursor/skills/n-<id>/SKILL.md`. Принцип: `worktree:true` — генеративні скіли (зміни з детермінованого джерела); `worktree:false` — реактивні (незакомічені зміни поточного checkout) і read-only.

## ADR Дві окремі JSON-схеми для `rules` і `skills` (E2)

## Context and Problem Statement
Обидва типи артефактів (`npm/rules/<id>/` і `npm/skills/<id>/`) матимуть `meta.json`, але поле `worktree` має сенс лише для скілів — правила є програмними перевірками, а не скілами, що виконуються в worktree. Треба вирішити, як уніфікувати формат без семантичних суперечностей.

## Considered Options
* (E1) Спільна схема з опційними полями за контекстом розташування.
* (E2) Та сама назва файлу `meta.json`, але дві тонкі незалежні схеми: `rules` знає лише `auto`, `skills` знає `auto` + `worktree`.
* (E3) Уніфікувати лише `auto` зараз, `worktree` додати скілам — формулювати як «спільне поле + скілова добавка».

## Decision Outcome
Chosen option: "(E2) Та сама назва, дві окремі схеми", because «менше умовної логіки, ціною двох файлів-схем» (цитата з transcript); кожна схема описує рівно свій контекст без умовних гілок.

### Consequences
* Good, because `rules`-схема ніколи не бачить `worktree` — помилка «worktree у правилі» вловлюється валідатором.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схеми живуть у `npm/schemas/`. `check`-перевірка розрізняє їх за розташуванням: `skills/*/meta.json` проти `rules/*/meta.json`. Повна міграція rules на `meta.json` — Spec B (окремий spec).

## ADR Інвентарний файл `.worktree.md` не в git-історії

## Context and Problem Statement
При роботі з кількома worktree одночасно агент і людина потребують зрозуміти, для чого саме той чи інший worktree створено. Коміт цього опису у git-історію засмічує її технічним артефактом, що не є частиною продукту.

## Considered Options
* Зберігати опис у `git commit message` при створенні worktree-гілки.
* Зберігати як `.worktree.md` у корені worktree, виключений із git через `.gitignore` або `.git/info/exclude`.

## Decision Outcome
Chosen option: "`.worktree.md` виключений з git", because «не потрібен в git-історії, тільки для інвентаризації поточних ворктрі» (цитата з transcript); файл залишається доступним для `git worktree list` + читання кожного `.worktree.md`, але не забруднює `git log`.

### Consequences
* Good, because transcript фіксує очікувану користь: файл є агентно- і людинозрозумілим; містить `Branch`, `Base` SHA, `Purpose`, список `Files`, `Notes` (зокрема хто/коли запустив сесію).
* Bad, because файл зникає разом з worktree при `git worktree remove`; централізованого реєстру немає — інвентаризація потребує обходу всіх `.worktrees/<name>/.worktree.md`.

## More Information
Шаблон файлу зафіксовано в `.cursor/rules/n-worktrees.mdc`. Виключення: `.gitignore` або `.git/info/exclude` у конкретному worktree. Task 9 плану `docs/superpowers/plans/2026-05-31-skill-meta-json-worktree.md` описує, як агент заповнює файл після `git worktree add`.
