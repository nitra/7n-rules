## ADR Поділ відповідальності між шарами валідації GitHub Actions workflows

## Context and Problem Statement
У монорепо `@nitra/cursor` потрібно перевіряти workflow файли GitHub Actions на кількох рівнях: синтаксис, безпека та відповідність проєктному канону (наприклад, обов'язковий `actions/checkout@v6`). Виникло питання, який саме інструмент відповідає за перевірку конкретної версії (`@v4` замість `@v6`).

## Considered Options
* Єдиний інструмент (actionlint) для всіх перевірок
* Layered підхід: actionlint + zizmor + conftest (Rego)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Layered підхід: actionlint + zizmor + conftest (Rego)", because кожен інструмент покриває окремий клас перевірок: actionlint — синтаксис та валідність `uses:`, zizmor — security-аудит і ref-pinning, conftest (Rego) + `check ga` — відповідність канонічним версіям Nitra (зокрема `checkout@v6`).

### Consequences
* Good, because transcript фіксує очікувану користь: `actionlint` ловить структурні помилки GHA, `zizmor` покриває `unpinned-uses` і security-ризики, а Rego-полісі (`npm/rules/ga/policy/`) фіксують канонічний стандарт проєкту незалежно від загальних лінтерів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Запуск: `bun run lint-ga` → `n-cursor lint-ga` → `bunx github-actionlint` + `uvx zizmor --offline --collect=workflows .`
- Канонічні шаблони: `npm/rules/ga/policy/*/template/*.yml`
- Rego-пакети: `lint_ga`, `workflow_common`, `lint_js_yml` та ін. у `npm/rules/ga/policy/`
- JS cross-file перевірки: `npm/rules/ga/lint/lint.mjs`, `npm/scripts/utils/gha-workflow.mjs`
- Конфіг security-аудиту: `.github/zizmor.yml` (параметр `ref-pin` для `unpinned-uses`)

## ADR Відсутність локальної перевірки `runs.using` у third-party actions та визначення відповідального шару

## Context and Problem Statement
`Infisical/secrets-action@v1.0.8` містить `runs.using: "node20"` у своєму `action.yaml`. GitHub анонсував deprecation Node.js 20 runtime для actions з червня 2026. Виникло питання, хто з наявних інструментів має ловити цей клас проблем.

## Considered Options
* Розширення `check ga` (`@nitra/cursor`) — резолюція `action.yml` по `uses:` та `deny` для `node12/node16/node20`
* Перевірка runner-ом на GitHub під час job-виконання (з `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true`)
* actionlint — після офіційного виводу `node20` з GHA і оновлення корпусу «outdated»
* Dependabot — оновлення тегу, але без гарантії `node24` (навіть `v1.0.12` Infisical лишається `node20`)
* Окремий скрипт/полісі поза `@nitra/cursor`

## Decision Outcome
Chosen option: "Відкрите питання — жоден з наявних шарів це не закриває", because transcript фіксує, що `lint-ga` (actionlint + zizmor + `check ga`) не перевіряє `runs.using` у `action.yml` третіх сторін; GitHub runner показує annotation лише під час реального запуску job-у, не локально.

### Consequences
* Good, because transcript фіксує очікувану користь від розширення `check ga`: резолюція `action.yml` через GitHub API/кеш дасть можливість `deny`, якщо `runs.using` ∈ `{node12, node16, node20}`, ще до push.
* Bad, because розширення `check ga` потребує FS/API-резолюції ref для кожного `uses:` — це виходить за межі поточної Rego per-document моделі та вимагає нового cross-file кроку.

## More Information
- `Infisical/secrets-action@v1.0.8` → `runs.using: "node20"` (підтверджено через `action.yaml`)
- `Infisical/secrets-action@v1.0.12` — також `node20`; `main`-гілка вже містить `node24` (без стабільного релізу на момент сесії)
- Варіант для CI до червня 2026: `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true` у тестовому job
- Поточна Rego-модель: per-document, не читає `action.yml` зовнішніх дій
- Команда для перевірки канону: `npx @nitra/cursor check ga`
