# Рекурсивний складений ОАГ із динамічним розкладом вузлів

**Status:** Accepted
**Date:** 2026-06-06

## Context and Problem Statement

Потрібна структура для системи де оркестратор розбиває задачі на підзадачі під час виконання. Вузли верхнього рівня не знають наперед чи буде підзадача атомарною операцією чи розкладеться у цілий підграф. Директорна ієрархія проєктів може бути довільною глибиною, і на кожному рівні можуть існувати як листові задачі так і вкладені workflow.

## Considered Options

* Рекурсивний складений ОАГ — вузол або атомарний або містить власний підграф; рішення динамічне при запуску
* Статичний ОАГ — структура задається наперед; не підходить через unknown-depth декомпозицію
* 3D-граф із просторовими вимірами — термін описує відображення, а не структуру зв'язків

## Decision Outcome

Chosen option: "Рекурсивний складений ОАГ", because батьківський вузол завжди бачить однаковий інтерфейс незалежно від внутрішньої складності дочірнього, а агент вирішує тип при запуску на основі вхідних даних.

### Consequences

* Good, because атомарний вузол можна "розкрити" у підграф без змін у батьківському графі — повна замінюваність.
* Good, because executor однаковий на всіх рівнях рекурсії: `execute(node)` → якщо атомарний запустити, інакше виконати підграф.
* Good, because динамічний spawn дозволяє агенту додавати нові дочірні вузли під час виконання без змін у батьківській структурі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Ребра несуть дані (outputs → inputs), не лише залежності. Кілька exit-вузлів у підграфі — їх виходи зливаються в outputs батька. Топологія зберігається розподілено: кожен дочірній `task.md` містить поле `deps:` зі списком попередників-siblings. Центрального файлу графу немає — оркестратор відновлює топологію скануванням `task.md`.

Близькі реалізації: Dask, Prefect dynamic tasks, LangGraph.

## Update 2026-06-06

### Термінологічне уточнення: Composite-абстракція вузла

Обговорення у `думка.txt` (репозиторій `/Users/vitaliytv/www/nitra/cursor`) розглядало альтернативи: `Recursive Compound Graph`, `Hierarchical Task Network (HTN)`, `Nested Directed Acyclic Graph`, `Multi-level Graph`, `Graph of Graphs`, `3D Graph`.

`HTN` відкинуто — статична декомпозиція (відома до виконання). `3D Graph` відкинуто — метафора просторового відображення, не топологія зв'язків.

Ключовий принцип: для батьківського рівня вузол є або **вирішеним**, або **не вирішеним** — незалежно від того, чи він атомарний, чи розгортається у підграф. Це класична Composite-абстракція (GoF): батько не залежить від внутрішньої структури вузла.

### Node-local repair memory — stateless EngineerAgent

Пам'ять про попередні спроби відновлення зберігається у `repair_history.json` на рівні вузла, де інженер застосував патч (не обов'язково на рівні вузла що впав). Між викликами інженер stateless — при кожному виклику читає `repair_history` з файлової системи, щоб не повторювати вже випробуваних стратегій.

Структура запису:
```json
[
  {
    "attempt": 1,
    "engineer_reasoning": "<діагноз>",
    "patch_applied": {},
    "result": "failed",
    "failure_reason": "<причина>"
  }
]
```

Алгоритм завершення: якщо стратегії вичерпано → `node.state = "unresolvable"` → escalate вгору по ієрархії. Конкретний поріг `max_attempts` і те, хто його перевіряє (сам інженер чи зовнішній guard), в transcript не визначені.

## Update 2026-06-06

### Деталізація архітектурних рішень (сесія d0294f80)

**Файлова схема вузла:**
```
<nodeId>/
  meta.json          — тип, state, edges
  inputs.json
  outputs.json       — з'являється при resolved
  error.json         — з'являється при failed
  repair_history.json
  patches/
  graph/<childId>/   — якщо Compound
```
Мутаційний лог: `patches/patch_NNN.json` накопичує зміни (`DAG_v1 → [EngineerPatch] → DAG_v2 → ...`). Граф відновлюється скануванням файлів без центрального оркестратора.

**Mutable with invalidation.** При патчі вузла `outputs.json` отримує `{ stale: true, reason: "patch_by_engineer" }`; каскад інвалідації поширюється до листів. При зміні батька через помилку дочірнього: `child.repair_history` логує `triggered_parent_patch`, `parent.repair_history` — деталі патчу.

**EngineerAgent як мета-рівень поза графом.** Повний доступ до будь-якого рівня ієрархії. `GraphPatch` включає: replace node, insert nodes, rewire edges, modify inputs.

**repair_history.json на вузлі.** Інженер stateless; знання — у файлах вузла. Структура запису: `{ attempt, engineer_reasoning, patch_applied, result, failure_reason }`. При ескалації повний масив attempts передається батькові.

**Time-budget convergence guard.** `repair_context.json`: `{ deadline, started_at, time_budget_sec, attempts[] }`. `deadline = parent.deadline - reduction_factor`. При timeout: `node.state = "unresolvable"`, ескалація вгору. Кожен батьківський рівень при ескалації отримує **свіжий** бюджет (не залишок).

**Ієрархічна ескалація; root → SeniorEngineer.** Максимальний час до людини: `depth × budget`. Root timeout → `senior_report.json`: `{ failed_node, escalation_chain: [{level, attempts, time_spent}], current_graph_snapshot, suggested_next_steps }`.

**Spec-driven execution через LLM-executor.** `meta.json` містить лише опис/промпт; LLM генерує виконання динамічно. Executor може самостійно розкрити атомарний вузол у Compound. Executor і engineer — два окремих агенти з різними ролями.

## Update 2026-06-06 (формат файлів вузла та паралельне виконання)

**Markdown + YAML-фронтматер як формат файлів вузла (LLM-first).** Фронтматер містить машинозчитувані поля (стан, ребра, мітки часу); тіло — вільний Markdown для LLM. Файли: `meta.md`, `inputs.md`, `outputs.md`, `error.md`, `repair_history.md`. `repair_history.md` — append-only журнал: інженер читає як єдиний документ і дописує наступну спробу без JSON-парсингу. Поле `застарілі: true` у фронтматері `outputs.md` при каскаді інвалідації.

**Паралельне виконання через git-worktrees з агентом-медіатором.** Щасливий шлях: незалежні вузли пишуть у різні директорії `tasks/<nodeId>/` — конфліктів немає за визначенням. Складний шлях (інженер у ворктрі A патчить спільний `meta.md` батька поки агент у ворктрі B ще виконується): вирішується агентом-медіатором за тим самим патерном що й відновлення вузла. Кожен агент-вузол працює у власному worktree (`.worktrees/<назва>/`); після завершення — merge до основної гілки. Конфлікт злиття на `meta.md` батька → spawn агента-медіатора.

## Update 2026-06-06 (динамічний розклад та протокол агента)

**Динамічний розклад вузла при запуску.** Поле `тип` відсутнє у `meta.md` — агент читає `inputs.md` і самостійно вирішує, чи задача атомарна чи потребує підграфу. Граф не потребує переоголошення при зміні складності; батьківський граф не зачіпається.

**Протокол агента: system_prompt оркестратора + CLI хуки.** Файли вузла містять лише місію та дані; протокол живе виключно у `system_prompt` оркестратора. Push-модель — агент після запису файлів сам викликає CLI:
- `n-cursor graph done <шлях>` — вузол resolved (outputs.md записано)
- `n-cursor graph spawn <шлях>` — вузол розкрився в Compound (дочірні meta.md+inputs.md записано)
- `n-cursor graph failed <шлях>` — вузол failed (error.md записано)

Порядок: спочатку запис файлів, потім CLI хук. `meta.md` орієнтовано на місію без поля `тип`; незмінна секція `## Оригінальна місія` захищає від drift намірів після патчів інженера.

## Update 2026-06-06 (git-worktree як атомарний мютекс)

**Git-worktree atomicity вирішує TOCTOU race condition.** `git worktree add` з іменем `<nodeId>-виконання` атомарно провалюється якщо ворктрі вже існує — природний мютекс без додаткових механізмів блокування. Усуває TOCTOU між читанням `стан: очікує` і запуском другого агента на той самий вузол.

**Зафіксоване обмеження:** 100+ паралельних вузлів = 100+ ворктрі на диску. Transcript не містить підтвердженого рішення для цього обмеження.

## Update 2026-06-06

Transcript додав наскрізне уточнення до рішення про рекурсивний складений ОАГ:

- Батьківський вузол бачить дочірній лише через інтерфейс стану та outputs, незалежно від того, атомарний він чи складений.
- Ребра мають data-flow семантику: outputs попередників стають inputs наступників.
- Топологія підграфу розподілена по `task.md` кожного дочірнього вузла через `deps:`.
- `inputs.md`/`outputs.md` можуть містити посилання `ref:` на файли, секції або рядки, щоб не дублювати великі дані.

Окремого нового ADR не потрібно: це деталізація вже прийнятого рішення про динамічний Recursive Compound DAG.

## Update 2026-06-06

Transcript додав мета-рівень відновлення для task graph:

- `EngineerAgent` є мета-рівневим агентом, а не звичайним вузлом графу.
- Він може змінювати граф на будь-якому рівні ієрархії, включно з батьківськими вузлами.
- Контекст ремонту зберігається у `repair_history.md` і `repair_context.md`.
- Перед patch вузла потрібно інвалідувати або kill-ити залежні worktree у топологічному порядку від листів до цілі.
- Ескалація йде вгору по ієрархії; root timeout приводить до повідомлення SeniorEngineer.

Ці факти доповнюють рішення про Recursive Compound DAG як self-modifying task graph.
