## ADR Вибір CLI-форми для worktree-інструмента

## Context and Problem Statement
Anthropic-специфічний інструмент `EnterWorktree` кладе worktree в `.claude/worktrees/` (захищену директорію) і не сумісний з Cursor. Конвенція `n-worktrees.mdc` описує правила вручну, але не гарантує однакову поведінку в різних LLM-середовищах.

## Considered Options
* CLI-підкоманда `n-cursor worktree` (виконавець) + тонкий skill
* Лише skill (markdown-інструкція для агента)
* Лише CLI без skill

## Decision Outcome
Chosen option: "CLI-підкоманда + тонкий skill (варіант C)", because CLI-команда `npx @nitra/cursor worktree …` однакова в Claude Code, Cursor і терміналі без залежності від харнесу, а тонкий skill лише направляє агента до CLI замість ручного `git worktree add`.

### Consequences
* Good, because transcript фіксує очікувану користь: однакова поведінка у всіх середовищах, атомарне створення worktree + `.md`-опису, агент не може відхилитись від конвенції.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Spec: `docs/superpowers/specs/2026-05-31-n-cursor-worktree-cli-design.md` (коміт `bf1842f`). Реалізація: `npm/scripts/lib/worktree.mjs` + case `'worktree'` у `npm/bin/n-cursor.js` + `.cursor/skills/n-worktree/SKILL.md`.

## ADR Набір субкоманд `n-cursor worktree`

## Context and Problem Statement
При проєктуванні нового CLI-інструменту `n-cursor worktree` потрібно визначити мінімальний набір операцій над git-worktree.

## Considered Options
* `add` + `remove` (YAGNI-мінімум, без `list` і `prune`)
* `add` + `remove` + `list`
* `add` + `remove` + `list` + `prune`

## Decision Outcome
Chosen option: "`add` + `remove` + `list` + `prune`", because користувач явно обрав варіант D3; `list` дає об'єднаний вивід git worktree + `.md`-описів в одному форматі, `prune` прибирає осиротілі `.md`-файли після видалення worktree.

### Consequences
* Good, because transcript фіксує очікувану користь: повна інвентаризація та прибирання без ручних операцій.
* Bad, because transcript не містить підтверджених негативних наслідків.

## ADR Санітизація імені гілки для шляху worktree

## Context and Problem Statement
Гілки git часто містять слеш (`feat/skill-meta`), тоді як `.worktrees/*.md` glob не рекурсивний — файл опису `feat/skill-meta.md` не знайде `cat .worktrees/*.md`.

## Considered Options
* Слеш → дефіс для пласкої структури (`.worktrees/feat-skill-meta/` + `.worktrees/feat-skill-meta.md`)
* Зберігати слеш-структуру (`.worktrees/feat/skill-meta/`) і шукати описи рекурсивно

## Decision Outcome
Chosen option: "Слеш → дефіс (варіант E-slash-a)", because пласка структура `.worktrees/` відповідає прикладу в `n-worktrees.mdc`, зберігає `cat .worktrees/*.md` робочим і спрощує реалізацію `list` і `prune`.

### Consequences
* Good, because transcript фіксує очікувану користь: `cat .worktrees/*.md` завжди знаходить усі описи без рекурсивного пошуку.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Функція `sanitizeBranch` у `npm/scripts/lib/worktree.mjs`. Unit-тест: `sanitizeBranch('feat/skill-meta')` → `'feat-skill-meta'`.

## ADR База гілки при `n-cursor worktree add`

## Context and Problem Statement
`EnterWorktree` (Anthropic) за замовчуванням гілкується від `origin/<default>`, що призводить до втрати локальних непоxаних комітів. Потрібно визначити, від якого ref гілкується новий worktree.

## Considered Options
* Завжди від поточного HEAD (F1, без флагів)
* Дефолт HEAD + опційний `--from <ref>` (F2)
* Дефолт від `origin/<default>` (F3, як `EnterWorktree`)

## Decision Outcome
Chosen option: "Завжди від поточного HEAD (F1)", because YAGNI: флаг `--from` — надмірність на старті; HEAD вирішує конкретну проблему `EnterWorktree` (локальні коміти губились), а F3 відтворює той самий баг.

### Consequences
* Good, because transcript фіксує очікувану користь: worktree гарантовано успадковує локальні коміти поточної гілки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`git worktree add .worktrees/<sanitized> -b <branch>` (без явного ref → бере HEAD). Spec: `docs/superpowers/specs/2026-05-31-n-cursor-worktree-cli-design.md`.

## ADR Обовʼязковий опис при `n-cursor worktree add`

## Context and Problem Statement
Інвентарний файл `.worktrees/<branch>.md` має сенс лише якщо поле «Задача» заповнено; без нього інвентаризація не дає відповіді «навіщо цей worktree створено».

## Considered Options
* Опис — обовʼязковий позиційний аргумент (G1); без нього команда падає з exit 1
* Опис опційний; якщо не передано — плейсхолдер у `.md` (G2)
* Структуровані прапорці `--task`, `--skill`, `--plan` (G3)

## Decision Outcome
Chosen option: "Обовʼязковий позиційний аргумент (G1)", because без опису інвентаризація безсенсова; один позиційний аргумент простіший за прапорці; обовʼязковість гарантує, що `cat .worktrees/*.md` завжди дає осмислену картину.

### Consequences
* Good, because transcript фіксує очікувану користь: `list` завжди показує змістовний опис кожного worktree.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Решта полів (Дата, База, Шлях) заповнюються автоматично з `new Date()` і `git log`. Spec: `docs/superpowers/specs/2026-05-31-n-cursor-worktree-cli-design.md`.
