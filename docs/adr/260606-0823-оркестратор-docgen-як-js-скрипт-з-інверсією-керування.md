## ADR Оркестратор docgen як JS-скрипт з інверсією керування

## Context and Problem Statement
One-shot виклики локальних LLM для генерації документації систематично «течуть» реалізацією: stdlib (`fs`, `path`), internal імена (`yamlCache`, `readAndParseYamlDocs`), сигнатури функцій, regex-деталі, вигадані гарантії. Проблема виникає тому, що модель одночасно «читає» код і «вирішує», що писати — без зовнішнього факт-заземлення.

## Considered Options
* JS-оркестратор керує pipeline (Stage 0 детермінована екстракція → точкові LLM-промпти → детермінована зборка)
* One-shot виклик LLM з повним кодом файлу і загальними поведінковими правилами у system-prompt
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "JS-оркестратор з інверсією керування", because: JS-скрипт (docgen-gen.mjs) є точкою входу; він детерміновано витягує факти (Stage 0, 0 токенів LLM), готує вузькі секційні промпти та збирає результат. Локальна модель — лише «сервіс перефразування» для відомих фактів; вона не може вигадати назву, всунути сигнатуру чи загубити секцію, бо ці рішення ухвалює скрипт.

### Consequences
* Good, because transcript фіксує очікувану користь: оркестрація підняла `gemma3:4b` з ~80% до ~86% відповідності еталону (+6 п.п.) на трьох тестових файлах (`firebase_hosting.mjs`, `overlay-paths.mjs`, `k8s-tree.mjs`).
* Bad, because v1 (спільний повний код у кожній секції) виявилась у 3–5× повільнішою за one-shot через повторний інгест коду; виправлено у v2.

## More Information
Файли: `npm/skills/docgen/js/docgen-extract.mjs`, `npm/skills/docgen/js/docgen-prompts.mjs`, `npm/skills/docgen/js/docgen-gen.mjs` у worktree `.worktrees/feat-docgen-orchestrator-pi`.

## ADR Секційно-мінімальний контекст (v2): код лише у секцію Поведінка

## Context and Problem Statement
v1 оркестратора надсилала повний вихідний код файлу у кожен із чотирьох секційних LLM-запитів (`Огляд`, `Поведінка`, `Публічний API`, `Гарантії`). На 8 GB MacBook (частковий CPU-офлоад, `gemma3:4b` ~20 tok/s) повторний інгест коду призвів до часу 102–310 с/файл — у 3–5 разів повільніше за one-shot.

## Considered Options
* Надсилати повний код лише у `Поведінку`; решта секцій отримують тільки компактний факт-лист (JSDoc-header, список експортів, маркери `readOnly`/`throws`/`skips`/`caches`)
* Спільна persistent multi-turn сесія: код інгестується в `system` один раз, секційні інструкції — окремими `user`-turns (тестувалось як варіант у transcript)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Секційно-мінімальний контекст (v2)", because persistent multi-turn сесія між незалежними файлами призводить до накопичення контексту (cross-file забруднення) і не підходить для batch docgen; натомість мінімальний контекст під кожну секцію робить інгест дешевим безвідносно до KV-cache поведінки ollama.

### Consequences
* Good, because transcript фіксує очікувану користь: `overlay-paths` прискорився з 310 с до 77 с (×4), `k8s-tree` з 141 с до 55 с (швидше за one-shot); `Публічний API` більше не витікає stdlib і приватні імена кешу.
* Bad, because `firebase_hosting` у v2-прогоні показав 118 с — аномалія першого виклику (4.6 tok/s, контенція при холодному старті); transcript не підтверджує стабільного покращення для першого файлу в серії.

## More Information
Функція `sectionMessages(section, facts, src)` у `docgen-prompts.mjs`: код (`src`) передається лише коли `section === 'behavior'`; для `overview`, `api`, `guarantees` — лише `facts` (header + exports + markers). Бенч-скрипт: `~/docgen-bench3/bench_orch.mjs`.

## ADR Підключення ollama як провайдера pi через models.json

## Context and Problem Statement
`pi` CLI (v0.78.0, `@earendil-works/pi-coding-agent`) за замовчуванням підтримує лише хмарні провайдери (`openai-codex`); прапорець `--provider ollama` повертав `Error: Unknown provider`. Для порівняльного експерименту (pi vs прямий ollama) потрібна була можливість звертатись до `gemma3:4b` через pi.

## Considered Options
* Додати кастомний провайдер `ollama` у `~/.pi/agent/models.json` з `api: "openai-completions"` і `baseUrl: "http://localhost:11434/v1"` (ollama має OpenAI-сумісний ендпойнт)
* Використовувати pi лише з хмарними моделями; locol ollama — тільки прямим HTTP
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Додати ollama у ~/.pi/agent/models.json", because ollama підтримує OpenAI-сумісний ендпойнт на `:11434/v1`, і документація pi (`docs/models.md`) прямо описує цей формат для локальних моделей.

### Consequences
* Good, because transcript фіксує очікувану користь: `pi --list-models gemma4` повернув `ollama  gemma4:4b  128K  16.4K`; smoke-test `pi -p --provider ollama --model gemma4:4b "одне слово: ок"` відповів `Так.`
* Bad, because pi-старт (~4 с на виклик при per-file запуску) додає фіксований оверхед; для batch docgen прямий `/api/chat` без node-boot є швидшою альтернативою.

## More Information
Конфіг: `~/.pi/agent/models.json`. Поля провайдера: `api: "openai-completions"`, `apiKey: "ollama"`, `compat.supportsDeveloperRole: false`, `compat.supportsReasoningEffort: false`. Глобальний default pi (`~/.pi/agent/settings.json`) свідомо **не змінювався** (лишився `openai-codex / gpt-5.5`), щоб не перемикати весь pi-кодинг на локальну модель.

## ADR Вибір моделей для Tier 1 docgen на 8 GB M2

## Context and Problem Statement
Після бенчмарку семи моделей (перший прогін) і трьох моделей (другий прогін, на нових файлах з самостійно написаним еталоном) потрібно було визначити, які локальні моделі лишати для Tier 1 docgen на машині з 8 GB RAM (M2).

## Considered Options
* `gemma3:4b` (3.3 GB, 100% GPU, ~20 tok/s) — швидкість-first
* `gemma4:4b` / `batiai/gemma4-e4b:q4` (5.3 GB q4, 56%/44% CPU/GPU, ~11 tok/s) — якість-first
* Моделі 7–8B (`qwen2.5:7b`, `qwen2.5-coder:7b`, `llama3.1:8b`) — частковий CPU-офлоад, ~0.5–17 tok/s
* Малі моделі (`qwen2.5-coder:3b`, `llama3.2:3b`, `gemma2:2b`, `phi3.5`) — повністю в GPU, але нижча якість / ненадійний контроль виводу
* `qwen3:4b` — ігнорував директиву `/no_think`, відповідав **російською**

## Decision Outcome
Chosen option: "gemma3:4b + gemma4:4b (два полюси спектра)", because 7–8B-моделі не вміщаються в GPU-бюджет 8 GB → своп/офлоад робить їх непрактичними; малі моделі (окрім `qwen2.5-coder:3b`) показали мовні дефекти або непередбачуваний формат виводу; `gemma3:4b` дає ~85% якості за ~14–15 год на 1042 файли; `gemma4:4b` (q4-перепак) дає ~92% за ~30 год — реалістична альтернатива без свопу.

### Consequences
* Good, because transcript фіксує очікувану користь: разом обидві моделі покривають обидва кінці спектра якість/швидкість без виходу за межі 8 GB (при ізольованому запуску).
* Bad, because `gemma4:4b` частково офлоадиться (56% CPU) і за наявності інших навантажень може деградувати так само, як `batiai/gemma4-e4b:q4` під час марафонного бенчмарку (0.4 tok/s через memory thrashing).

## More Information
Команди очищення: `ollama rm qwen3:4b qwen2.5:7b qwen2.5-coder:7b llama3.1:8b llama3.2:3b gemma2:2b phi3.5 qwen2.5-coder:3b gemma4:e4b batiai/gemma4-e4b:q4`. Alias: `ollama cp batiai/gemma4-e4b:q4 gemma4:4b`. `OLLAMA_MAX_LOADED_MODELS=1`, `OLLAMA_NUM_PARALLEL=1` — зафіксовано в env.

## ADR Самостійно написаний еталон як база для відсоткового порівняння якості LLM-документації

## Context and Problem Statement
Перший бенчмарк порівнював виходи LLM суб'єктивно — «зіркова» рейтингова шкала без формалізованої бази. Для другого бенчмарку (нові тестові файли, три моделі) потрібен відтворюваний критерій якості, щоб числово зіставляти виходи різних моделей і конфігурацій.

## Considered Options
* Написати еталонну документацію самостійно (Claude) для кожного тестового файлу і порівнювати вихід моделей з нею у відсотках за чотирма рубриками: точність, повнота поведінки, якість украї́нської, формат/лаконічність
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Самостійно написаний еталон (Claude)", because це дає фіксовану, відтворювану базу порівняння незалежно від конкретної моделі; рубрики 4×25% покривають ті ж виміри, за якими модель може помилитись (факти, поведінкова повнота, мова, формат).

### Consequences
* Good, because transcript фіксує очікувану користь: метрика дозволила кількісно зіставити всі варіанти (gemma3:4b ~85%, gemma4:4b ~92%, orchestrated v1 ~86%, orchestrated v2 очікується ≥86%) і виявити регрес між прогонами одного й того ж pi (78% vs 87%).
* Bad, because ручне оцінювання вносить суб'єктивний шум ±3 п.п.; transcript явно фіксує: «розрив B↔C — 2 п.п., в межах мого власного шуму»; порівняння між різними сесіями оцінювача ненадійне.

## More Information
Еталони записувались у `/tmp/docgen-bench2/etalon/` (другий бенчмарк) і відновлювались у `~/docgen-bench3/etalon/` (третій бенчмарк) після втрати `/tmp` між сесіями. Тестові файли: `npm/rules/abie/js/firebase_hosting.mjs` (50 рядків), `npm/rules/abie/lib/overlay-paths.mjs` (97 рядків), `npm/rules/abie/lib/k8s-tree.mjs` (110 рядків).
