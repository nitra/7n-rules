type: ADR
title: OKF v0.1 сумісність автоматично-генерованих ADR
description: ADR-нормалізатор і clean ADR-файли отримують OKF frontmatter з `type: ADR` поверх MADR body.
---

**Status:** Accepted
**Date:** 2026-06-17

## Context and Problem Statement

Автоматично нормалізовані ADR не мали YAML frontmatter у clean-стані, тоді як OKF v0.1 conformance вимагає parseable YAML frontmatter з непорожнім полем `type` у кожному `.md` файлі. Через це `docs/adr/*.md` не відповідали OKF v0.1, хоча тіло ADR залишалося MADR-сумісним.

## Considered Options

- Додати OKF-сумісний YAML frontmatter у rewrite-операції normalize-pipeline.
- Мінімальний ADR frontmatter із `type: ADR` і `title`, з можливими custom fields.
- Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Додати OKF-сумісний YAML frontmatter у rewrite-операції", because це дає відповідність OKF v0.1 conformance без зміни структури MADR body.

### Consequences

- Good, because кожен `.md` файл у `docs/adr/` отримує parseable YAML frontmatter з непорожнім `type: ADR`.
- Good, because MADR-секції тіла залишаються без структурних змін.
- Bad, because transcript фіксує це як відхід від попереднього MADR v4.0.0 minimal принципу без frontmatter у clean-файлах.
- Neutral, because OKF дозволяє довільні додаткові ключі, але transcript не фіксує окремої політики для всіх custom fields.

## More Information

- Змінені файли: `npm/scripts/lib/adr/normalize-pipeline.mjs`, `npm/.claude-template/hooks/normalize-decisions.sh`, `.claude/hooks/normalize-decisions.sh`, `npm/rules/adr/adr.mdc`, `.cursor/rules/n-adr.mdc`, `npm/scripts/lib/adr/tests/normalize-pipeline.test.mjs`.
- `GEN_SYS` оновлено так, щоб вимагати OKF frontmatter.
- `validateMadr()` почав перевіряти `type: ADR` замість відхилення будь-якого frontmatter.
- У shell fallback додано гарантію OKF frontmatter, якщо LLM його не сформував.
- Transcript фіксує 14 passing tests і міграцію 300 clean ADR-ів.
- Clean-стан в правилах описано як `OKF v0.1 + MADR body`.
- Формат, згаданий у transcript: `type: ADR`, `title`, `tags: [adr]`, `timestamp`; мінімальним conformance-полем є `type`.

## Update 2026-06-17

- Transcript уточнює мінімальну вимогу OKF v0.1 conformance: кожен `.md` файл має parseable YAML frontmatter і непорожнє поле `type`.
- Для ADR мінімальна зміна формулювалася як frontmatter з `type: ADR` і `title`, де `session` може лишатися custom field, бо OKF дозволяє довільні додаткові ключі.
- `description` визнано опціональним і таким, що може братися з першої фрази `Context and Problem Statement`.

## Update 2026-06-18

- Clean ADR-файли після нормалізації отримують OKF v0.1 frontmatter з `type: ADR` і `title:`; `description`, `tags` і `timestamp` прибрані як надлишкові для каталогу ADR.
- `# ADR: <заголовок>` прибрано з тіла, щоб не дублювати `title:` у frontmatter; тіло починається зі status/date рядків MADR.
- `validateMadr()` оновлено так, щоб приймати `type: ADR` frontmatter замість відхиляти будь-який frontmatter.
- Міграційний скрипт `/tmp/adr-okf-migrate.sh` обробив 300 clean ADR-файлів; після cleanup лишків `tags`/`timestamp` не залишилось.
- Змінені місця: `npm/scripts/lib/adr/normalize-pipeline.mjs`, `npm/.claude-template/hooks/normalize-decisions.sh`, `.claude/hooks/normalize-decisions.sh`, `npm/rules/adr/adr.mdc`, `.cursor/rules/n-adr.mdc`, `npm/scripts/lib/adr/tests/normalize-pipeline.test.mjs`.
