# Серіалізація та дедуплікація паралельних важких CLI-команд через `withLock`

**Status:** Accepted
**Date:** 2026-05-23

## Context and Problem Statement

Проєкт запускає кілька агентів паралельно; кожен може самостійно тригерити важкі CLI-команди (`lint-ga`, `lint-rego`, `lint-text`, `lint-k8s`, `lint-docker`, `fix <rule>`). Паралельний запуск декількох `eslint` / `hadolint` / `kubeconform` перевантажує диск і CPU. Якщо N агентів лінтять той самий незмінений стан git-дерева, повторний прогін витрачає ресурси без нового результату.

## Considered Options

- `n-cursor guard <key> -- <команда>` — окрема команда-обгортка, агент явно викликає її
- Вбудований лок у реалізаційний код кожної команди через `runStandardLint` / `runStandardRule`
- Атомарний лок через `mkdirSync` як примітив (bun-native, без зовнішніх залежностей)
- `/usr/bin/shlock` (BSD-утиліта macOS) — взаємне виключення без дедупу
- npm-пакет `proper-lockfile` — стороння залежність без потрібної функціональності
- Fail-fast: агент Б отримує помилку, якщо лок зайнятий
- Skip-and-continue: агент Б пропускає команду без очікування

## Decision Outcome

Chosen option: "Вбудований лок через `runStandardLint` і `runStandardRule` з `mkdirSync`-примітивом та дедупом за git-fingerprint", because `fs.mkdirSync()` атомарний на APFS — рівно один процес виграє (`EEXIST` для решти); fail-fast унеможливлює підтвердження коректності для агента Б, skip пропускає валідацію; вбудований лок прозорий — агент далі запускає `bun run lint` як завжди і його неможливо обійти прямим викликом; дедуп дає оптимізацію без втрати гарантій; дві спільні точки входу усувають дублювання (`withLock` у кожному з 5 `lint.mjs`).

Механізм: `mkdirSync`-lock, власник у `owner.json` `{pid, host, startedAt, fingerprint}`, виявлення застарілого локу через `process.kill(pid, 0)` + max-age 30 хв. Стан — `node_modules/.cache/n-cursor/<key>/`. Дедуп — тільки для `exitCode === 0` + збіг sha256-fingerprint + TTL 10 хв. Fingerprint = sha256 від `git rev-parse HEAD` + `git diff HEAD` + untracked-файли через `git ls-files --others -z` (NUL-розділення для non-ASCII імен). Ключ лок-файлу виводиться зі шляху `import.meta.dirname`.

### Consequences

- Good, because паралельні агенти серіалізуються (`🔒 lint-rego: лок взято`), повторний прогін на тому самому git-дереві пропускається (`♻️ lint-rego: те саме дерево, 2с тому — пропускаю`)
- Good, because майбутні cross-cutting зміни (телеметрія, dry-run, env-toggle) вносяться в одне місце (`run-standard-lint.mjs` або `run-standard-rule.mjs`)
- Bad, because `runStandardRule` викликається і з `rules/<id>/fix.mjs` (bun-процес), і з `run(ctx)`-композиції — lock-ключ `fix-<ruleId>` захоплюється в дочірньому процесі; необхідно враховувати при написанні тестів
- Neutral, because таймаут очікування 20 хв → лог-попередження + виконання без локу (захист від зависання)

## Update 2026-05-23

### Деталі вбудування lock у конкретні команди

- `lint-ga`: `export const runLintGaCli = () => withLock('lint-ga', runLintGaSteps)` — `npm/rules/ga/lint/lint.mjs:168`
- `lint-rego`, `lint-text`, `lint-k8s`, `lint-docker`: аналогічний патерн `*Steps` + `*Cli` у відповідних `lint.mjs`
- `fix` (per-rule): лок вбудовано в `npm/scripts/utils/run-standard-rule.mjs`; `runFixCommand` у `n-cursor.js` делегує до spawn-процесів `rules/<id>/fix.mjs`, кожен з яких проходить через `runStandardRule` → `withLock('fix-<id>')`
- Перші тести: 12 тестів `with-lock.test.mjs` + `worktree-fingerprint.test.mjs` зелені після першої реалізації; `npx @nitra/cursor fix changelog` виводить `🔒 fix-changelog: лок взято`
- Комміт реалізації: `c4b9ea3`; розгортання на всі команди: `17fd868` (v1.13.85)
