## ADR Тільки правило для worktree-конвенції, без окремого skill

## Context and Problem Statement
Під час проєктування worktree-tool розглядалися три форми: лише CLI, CLI + skill, CLI + rule. Після того як форму C (CLI + skill + rule) було обрано, виникло питання: чи не дублюють skill і rule той самий зміст з різним «технічним обгортанням»?

## Considered Options
* CLI + rule `npm/rules/worktree/worktree.mdc` + skill `npm/skills/worktree/SKILL.md`
* CLI + тільки rule `npm/rules/worktree/worktree.mdc` (без skill)
* CLI + тільки skill (без rule)

## Decision Outcome
Chosen option: "CLI + тільки rule, без skill", because якщо CLI виконує всю роботу однією командою, і rule, і skill говорили б одне й те саме — «виклич цю команду». Skill потрібен лише там, де є складний покроковий сценарій із прийняттям рішень (як `n-fix`); тут такого сценарію немає.

### Consequences
* Good, because transcript фіксує очікувану користь: rule завантажується пасивно у `.cursor/rules/` й завжди в контексті; немає розсинхрону між двома артефактами.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Порівняння в transcript: `n-fix` — складний сценарій (lint → читати помилки → виправляти), skill виправданий; `n-cursor worktree add` — одна команда, skill зайвий. Файл: `npm/rules/worktree/worktree.mdc` (ще не створений, заплановано у плані реалізації). Попередній «сирота» — `.cursor/rules/n-worktrees.mdc` без джерела в `npm/rules/` — буде нормалізований: перенесений у `npm/rules/worktree/worktree.mdc` і перейменований у `n-worktree.mdc` (однина, відповідає назві команди).

## ADR Санітизація слешів гілки у дефіс для пласкої структури `.worktrees/`

## Context and Problem Statement
Конвенція `n-worktrees.mdc` зберігає інвентарний `.md`-опис поруч із checkout: `.worktrees/<name>.md`. Git-гілки зазвичай містять слеш (`feat/skill-meta`). Якщо ім'я файлу повторює гілку дослівно, опис лягає в підкаталог (`.worktrees/feat/skill-meta.md`), а glob `cat .worktrees/*.md` його не знаходить.

## Considered Options
* Санітизувати слеш у дефіс: `feat/skill-meta` → `.worktrees/feat-skill-meta/` + `.worktrees/feat-skill-meta.md` (пласка структура)
* Зберігати слеш як є (`.worktrees/feat/skill-meta/`), `list`/`prune` шукають рекурсивно

## Decision Outcome
Chosen option: "Санітизувати слеш у дефіс (пласка структура)", because glob `cat .worktrees/*.md` інвентаризації завжди працює без рекурсії; приклад у `n-worktrees.mdc` (`feat-skill-meta`) вже використовує цю форму. Git-гілка лишається з нативним слешем (`feat/skill-meta`), змінюється лише ім'я каталогу і файлу на диску.

### Consequences
* Good, because transcript фіксує очікувану користь: інвентаризація `cat .worktrees/*.md` завжди дає повну картину без додаткових glob-ускладнень.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Логіку санітизації планується винести в `npm/scripts/lib/worktree.mjs` (чиста функція, покривається юнітами). CLI друкує фактичний шлях після `add`, щоб агент не вгадував. Spec: `docs/superpowers/specs/2026-05-31-worktree-cli-design.md`.

## ADR Семантика `prune` — агресивне прибирання, `remove` — безпечний дефолт з `--force`

## Context and Problem Statement
Дві підкоманди `worktree` виконують незворотні дії (видалення checkout, `.md`-опису). Треба визначити, коли вони діють одразу, а коли вимагають підтвердження, щоб не втратити роботу випадково.

## Considered Options
* `remove` — безпечний дефолт (`git worktree remove` відмовляється на брудному дереві), `--force` форсує; `prune` — dry-run за замовчуванням, видаляє тільки з `--yes`
* `remove` — безпечний дефолт + `--force`; `prune` — одразу прибирає осиротілі `.md` і `git worktree prune` без підтвердження

## Decision Outcome
Chosen option: "`remove` = H2 (безпечний + `--force`), `prune` = H-prune-b (агресивний)", because `prune` за визначенням вже є «прибирання сміття», тому додаткове підтвердження зайве; `remove` натомість може мати незмерджену гілку — тут безпечний дефолт важливіший.

### Consequences
* Good, because transcript фіксує очікувану користь: незворотність `remove` захищена дефолтом; `prune` не вимагає зайвих підтверджень для рутинного прибирання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Що саме видаляє `prune` (H-prune-b): `git worktree prune` (видаляє git-метадані стертих checkout) + осиротілі `.worktrees/<name>.md` без відповідного worktree; worktree без `.md` — лише попередження у звіт, не видалення. Логіка виявлення осиротілих — `findWorktreesWithoutDesc()` у `npm/scripts/lib/worktree.mjs`. Spec: `docs/superpowers/specs/2026-05-31-worktree-cli-design.md`.

## ADR Обовʼязковий текстовий опис при `worktree add`, авто-поля з git

## Context and Problem Statement
Ціль інвентарного файлу `.worktrees/<name>.md` — дати відповідь на питання «навіщо створено цей worktree», щоб `cat .worktrees/*.md` давав осмислену картину поточних ізольованих задач. Постало питання, які поля заповнює tool автоматично, а які вимагають від агента/користувача.

## Considered Options
* Опис — обовʼязковий аргумент `add`; інші поля (дата, коміт-база, гілка, команда прибирання) — автоматично з git
* Опис опційний, при відсутності — плейсхолдер `<!-- опиши задачу -->`
* Структуровані прапорці `--task`, `--skill`, `--plan` окремо

## Decision Outcome
Chosen option: "Опис обовʼязковий, решта авто з git (G1)", because без опису інвентаризація безглузда — заради неї весь механізм і будувався. Плейсхолдер (варіант 2) дозволяє пропустити головне; прапорці (варіант 3) — зайва церемонія для CLI, яку викликає агент.

### Consequences
* Good, because transcript фіксує очікувану користь: `cat .worktrees/*.md` завжди дає осмислений вивід; tool гарантує заповнення полів дата/коміт/гілка/команда-прибирання без ручного ввімкнення.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Авто-поля: дата — з `Date` (CLI-процес, заборони `new Date()` у `.cursor/rules/*.mdc` не знайдено); коміт-база — `git rev-parse --short HEAD`; рядок «Прибрати» — `npx @nitra/cursor worktree remove <branch>`. Spec: `docs/superpowers/specs/2026-05-31-worktree-cli-design.md`. Шаблон `.md` узгоджено з конвенцією `n-worktrees.mdc`.
