## ADR Розподіл відповідальності: pi як шар хмарних/агентних викликів, прямий HTTP до omlx для локальних one-shot

## Context and Problem Statement
У проєкті всі виклики до LLM (локальних і хмарних) проходили через CLI `pi`. Потрібно зменшити overhead і залежність від `pi` для локальних моделей, при цьому зберегти pi як повноцінний шар там, де потрібна SDK-агентика або хмарні провайдери.

## Considered Options
* Перевести всі локальні виклики на прямий HTTP до `http://localhost:8000` (omlx), залишити pi лише для хмари
* Залишити єдиний шар `pi` для всіх викликів (статус-кво)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Перевести all локальні one-shot виклики на прямий omlx HTTP, залишити pi для хмарних і агентних (SDK tool-loop) викликів", because transcript підтверджує: one-shot точки (`coverage-classify`, `fix/llm-worker`, docgen Tier 1) не потребують tool-loop — їх легко замінити прямим HTTP; `coverage-fix` уже де-факто хмарний (`LOCAL_MAX=''` → `CLOUD_MAX`), а JS-owned tool-free loop покриває локальну агентику без pi (зафіксовано в ADR `260610-1349`).

### Consequences
* Good, because transcript фіксує очікувану користь: усунення CLI-overhead ~97с/tool-крок для one-shot точок; детерміновані сайд-ефекти в JS-оркестраторі; pi залишається тонким шаром лише там, де потрібен справжній SDK tool-loop.
* Bad, because omlx (MLX-сервер) text-only: не приймає `tools` і не повертає `tool_calls` — JS-owned loop не є прямим замінником pi-агента для складних завдань; потрібна ескалація на cloud-tier при провалі.

## More Information
Файли: `npm/lib/omlx.mjs` (новий спільний транспорт), `npm/scripts/coverage-classify/index.mjs`, `npm/skills/fix/js/llm-worker.mjs`, `npm/skills/docgen/js/docgen-gen.mjs`. ADR агентної пастки: `docs/adr/260610-1349-агентна-пастка-js-owned-loop-через-omlx-замість-pi-tool-loop.md`. Change-файл: `npm/.changes/260610-1402.md`.

## ADR Конвенція префікса `omlx/` для маршрутизації моделі без окремого env-прапора

## Context and Problem Statement
Після рішення розділити pi/omlx потрібен механізм, за яким `callModel` (раніше `callPi`) визначає транспорт — прямий HTTP до omlx або CLI `pi` — без додаткового env-прапора на рівні кожного скіла.

## Considered Options
* Префікс у назві моделі: `omlx/<model-id>` → прямий HTTP; усе інше → pi CLI
* Окремий env-прапор per-skill (як `N_CURSOR_DOCGEN_BACKEND=omlx`)
* Єдиний глобальний перемикач `N_CURSOR_LLM_BACKEND`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Префікс `omlx/` у назві моделі як єдиний дискримінатор маршруту", because transcript фіксує рішення оформити routing у дефолтній реалізації `callModel` — сигнатура незмінна (`callPi` → `callModel`), ін'єкція в тестах збережена; per-skill env-флаги (зокрема `N_CURSOR_DOCGEN_BACKEND`) прибрані на користь єдиної конвенції.

### Consequences
* Good, because transcript фіксує очікувану користь: жодного per-skill env-боргу; routing живе в одному місці — `npm/lib/omlx.mjs` (`isOmlxModel`, `omlxModelId`, `callOmlx`); existuючий fallback (omlx-помилка → cloud-тир) спрацьовує через наявний `catch` без окремої логіки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізація: `npm/lib/omlx.mjs` — `isOmlxModel(model)` перевіряє `model.startsWith('omlx/')`, `omlxModelId(model)` повертає частину після префікса. `npm/lib/models.mjs` оновлено — коментар про конвенцію `omlx/`-префікса для `N_LOCAL_MIN_MODEL` тощо. Тести: `npm/lib/tests/omlx.test.mjs` (19 тестів, 2 файли проходять). Rename ін'єкції: `callPi` → `callModel` / `opts.callModel` у `coverage-classify/index.mjs` і його тестах.
