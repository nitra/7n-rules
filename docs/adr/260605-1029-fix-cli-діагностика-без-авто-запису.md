## ADR `fix`-CLI діагностика без авто-запису

## Context and Problem Statement
Workflow `.github/workflows/npm-publish.yml` не містив кроку `Release (bump + CHANGELOG + tag)`, що блокувало реліз: `JS-DevTools/npm-publish` публікував ту саму версію без bump/CHANGELOG/тегу. Питання полягало у тому, чому скіл `n-fix` не виправив це автоматично, хоча канонічний template у `npm/rules/npm-module/policy/npm_publish_yml/template/npm-publish.yml.snippet.yml` описує крок правильно.

## Considered Options
* CLI `npx @nitra/cursor fix` — лише друкує `❌`/`✅`, файли не змінює
* Агент (крок 3 скіла `n-fix`) — читає `❌` і вручну вставляє канонічний контент

## Decision Outcome
Chosen option: "Агент застосовує виправлення, CLI лише діагностує", because `runRule` (`npm/scripts/lib/run-rule.mjs`) за архітектурою є orchestrator-only: він запускає `checkSnippet`/`checkContains`/`checkDeny` і виводить статус, але ніколи не пише файли. Запис — це крок 3 скіла `n-fix`, де агент читає `❌`-рядки й застосовує зміни.

### Consequences
* Good, because `checkSnippet` subset-перевірка по `steps[].name` коректно виявила відсутній елемент (`❌ npm_publish_yml: jobs."release-publish".steps має містити елемент з name: "Release (bump + CHANGELOG + tag)"`).
* Bad, because якщо агент пропустив `❌`-рядок або скіл не запускався end-to-end, крок так і не повертається — порушення залишається без виправлення до наступного повного запуску `n-fix`.

## More Information
- Канонічний template: `npm/rules/npm-module/policy/npm_publish_yml/template/npm-publish.yml.snippet.yml`
- Файл правила: `npm/scripts/lib/run-rule.mjs` (рядки 1–40 — логіка applies-gate + print-only)
- Виправлення застосоване в worktree `.worktrees/main-fix` — крок `Release (bump + CHANGELOG + tag)` (`node npm/bin/n-cursor.js release`) вставлено між `Configure git identity` і `Publish package`
- Після виправлення: `✅ npm_publish_yml: .github/workflows/npm-publish.yml відповідає канону (template subset)`
- Форматування: `oxfmt .github/workflows/npm-publish.yml` (101 ms, 1 file)
