## ADR Нове правило `rust` — rego-heavy архітектура

## Context and Problem Statement
Проєкти на Rust у монорепо не мали стандартизованих перевірок форматера (`rustfmt`) та лінтера (`cargo clippy`), CI workflow і VSCode-розширень. Постала необхідність додати правило, аналогічне `js-lint`/`style-lint`.

## Considered Options
* **A: rego-heavy** — три окремі rego-policy-пакети (`package_json`, `vscode_extensions`, `lint_rust_yml`), JS-check лише для gating та orchestration (як `js-lint`/`style-lint`)
* **B: JS-heavy** — лише один rego-пакет `vscode_extensions`, решту перевіряти в `check.mjs` (як `tauri`)
* **C: Гібрид** — rego для `package_json` і `vscode_extensions`, CI-workflow — JS-логікою

## Decision Outcome
Chosen option: "rego-heavy (A)", because це забезпечує паритетність з `js-lint`/`style-lint` і уникає технічного боргу при майбутньому розширенні перевірок (наприклад, `cargo deny`, MSRV). Дрейф від конвенції при JS-heavy підході потребував би рефакторингу при додаванні нових документів.

### Consequences
* Good, because транскрипт фіксує очікувану користь: консистентність з існуючими правилами, можливість додавати нові документи через rego без зміни JS-коду.
* Bad, because транскрипт не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/rules/rust/policy/package_json/`, `npm/rules/rust/policy/vscode_extensions/`, `npm/rules/rust/policy/lint_rust_yml/`. Зразок — `npm/rules/js-lint/` і `npm/rules/style-lint/`. Spec: `docs/superpowers/specs/2026-05-23-rust-rule-design.md`.

## ADR Канонічна команда `scripts.lint-rust`

## Context and Problem Statement
Необхідно визначити єдину канонічну команду `lint-rust` у `scripts` в `package.json`, яка покриватиме і авто-виправлення, і hard-fail на попередження.

## Considered Options
* `cargo fmt --check && cargo clippy -D warnings` (жорсткий, без fix)
* `cargo fmt --all && cargo clippy --fix --allow-staged --all-targets --all-features && cargo clippy --all-targets --all-features -- -D warnings` (fmt з фіксом + clippy --fix + hard-check)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "fmt (with fix) + clippy --fix + clippy -D warnings", because це обрав користувач для локального запуску: авто-виправлення дозволяє разом провести `fmt` і `clippy --fix`, а фінальний `clippy -D warnings` гарантує, що після fix залишились нульові попередження.

### Consequences
* Good, because транскрипт фіксує очікувану користь: розробник виконує одну команду й отримує і форматування, і виправлення, і звіт про помилки.

## More Information
Канонічна команда: `cargo fmt --all && cargo clippy --fix --allow-dirty --allow-staged --all-targets --all-features && cargo clippy --all-targets --all-features -- -D warnings`. CI використовує `--check`-варіант (без `--fix`), бо PR-оточення read-only. Rego в `policy/package_json/` перевіряє `contains(scripts["lint-rust"], "cargo")`.

## ADR Перенесення `rust-lang.rust-analyzer` з правила `tauri` у правило `rust`

## Considered Options
* Залишити `rust-lang.rust-analyzer` в обох правилах (дублювання)
* Перенести `rust-lang.rust-analyzer` у `rust`-rule, `tauri`-rule перевіряє лише `tauri-apps.tauri-vscode`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Перенести `rust-lang.rust-analyzer` у `rust`-rule", because Tauri-проєкт — це завжди Rust-проєкт, тому `rust`-rule завжди активне поряд із `tauri`; `tauri`-rule покриває лише Tauri-специфічне розширення.

### Consequences
* Good, because транскрипт фіксує очікувану користь: без дублювання, чіткий separation of concerns — `rust` відповідає за Rust-інструментарій, `tauri` — за Tauri-специфіку.
* Bad, because транскрипт не містить підтверджених негативних наслідків.

## More Information
Зміни у `tauri`: `npm/rules/tauri/tauri.mdc` (прибрати `rust-lang.rust-analyzer` з прикладу), `npm/rules/tauri/policy/vscode_extensions/vscode_extensions.rego` (залишити лише `tauri-apps.tauri-vscode`), `npm/rules/tauri/policy/vscode_extensions/vscode_extensions_test.rego` (оновити тести: 1 violation замість 2, видалити `test_fail_missing_rust_analyzer`).

## ADR Без канонічних конфігів `rustfmt.toml` / `clippy.toml`

## Context and Problem Statement
При проєктуванні правила `rust` постало питання: вимагати канонічний `rustfmt.toml` і `clippy.toml` у корені (аналогічно `oxlint-canonical.json` у `js-lint`), чи залишити defaults.

## Considered Options
* Вимагати `rustfmt.toml` і `clippy.toml` з канонічним вмістом (як `oxlint-canonical.json`)
* Не вимагати конфігів — `rustfmt` і `clippy` працюють з defaults

## Decision Outcome
Chosen option: "Не вимагати канонічних конфігів", because defaults достатні; надмірне закріплення конфігів ускладнить впровадження без реальної потреби.

### Consequences
* Good, because транскрипт фіксує очікувану користь: менше boilerplate, простіше впровадження правила.
* Bad, because транскрипт не містить підтверджених негативних наслідків.

## More Information
До scope виключено: `rustfmt.toml`, `clippy.toml`, `rust-analyzer` налаштування у `settings.json`, `cargo-deny`, MSRV pin. Можна додати пізніше окремим правилом або розширенням поточного.
