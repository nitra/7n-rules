## ADR Видалення скіла `/n-fix` (делегат без власної логіки)

## Context and Problem Statement
Скіл `/n-fix` був делегатом — він не містив власної логіки, лише перенаправляв до `/n-lint`. Наявність двох скілів з однаковим ефектом створювала зайвий реєстраційний ріпл (дзеркала, CLAUDE.md, auto-skills фікстури) і плутанину для користувача.

## Considered Options
* Залишити `/n-fix` як публічний аліас до `/n-lint`
* Повністю видалити `/n-fix` — функціонал у `/n-lint`

## Decision Outcome
Chosen option: "Повністю видалити `/n-fix`", because скіл уже був делегатом без власної поведінки; видалення прибирає зайвий реєстраційний ріпл і однозначно вказує на `/n-lint` як єдиний вхід.

## ADR Інлайн `_fix-check`/`fix-t0` як прямі функції (без internal subcommands)

## Context and Problem Statement
Конформність-фаза lint і T0-автофікс (orchestrator.mjs, t0.mjs) викликали самих себе через `spawnSync('bun', [N_CURSOR_BIN, '_fix-check', ...])` і `spawnSync('bun', [N_CURSOR_BIN, 'fix-t0', ...])` — subprocess-round-trip усередині того ж процесу. `bin/n-cursor.js` містив локальні функції `runFixCommand`/`resolveFixRuleIds`/`runRuleFixProcesses`/`listProjectRulesMdcFiles`, які більше ніде не перевикористовувались.

## Considered Options
* Залишити `_fix-check`/`fix-t0` як internal subcommands (поточний стан)
* Винести `runFixCheck` і `listProjectRulesMdcFiles` у lib і викликати їх напряму

## Decision Outcome
Chosen option: "Винести у lib і викликати напряму", because subprocess-round-trip додає overhead, ускладнює стек і унеможливлює type-safe передачу параметрів між фазами; пряма функція в lib є природним рефакторинговим кроком.

### Consequences
* Good, because transcript фіксує очікувану користь: `_fix-check`/`fix-t0` вилучені з `bin/n-cursor.js` (dispatch-switch, `runFixCommand`/`resolveFixRuleIds`/`runRuleFixProcesses`); `run-fix-check.mjs` і `list-project-rules-mdc.mjs` у `scripts/lib/`; `orchestrator.mjs`, `t0.mjs`, `rules/lint/js/orchestrate.mjs`, `scripts/post-tool-use-fix.mjs` рефакторовані на прямі виклики.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

## ADR Патерн per-tool omlx-фіксера зі спільним ядром `llm-fix-apply`

## Context and Problem Statement
Тули без нативного `--fix` (cspell, knip, actionlint тощо) раніше залишали знахідки без авто-корекції у fix-режимі lint. `llm-worker.mjs` вже містив логіку `parseResponse`/`readFilesForFix`/`applyChanges`, але тільки для конформності-фіксу; дублювання цієї логіки для кожного нового тулу було б проблемою.

## Considered Options
* Реалізовувати per-tool фіксер для кожного тулу окремо (без спільного ядра)
* Вибрати `llm-fix-apply.mjs` як спільне ядро (parse/read/apply) + generic `llmLintFix` + per-tool wiring

## Decision Outcome
Chosen option: "Спільне ядро `llm-fix-apply` + generic `llmLintFix`", because усуває дублювання parse/read/apply між конформністю і per-tool фіксерами; новий тул вмикається одним викликом `llmLintFix({ toolOutput, toolName, projectRoot })` без повторення omlx-інтеграції.

### Consequences
* Good, because transcript фіксує валідацію на живому omlx: `quik→quick, jumpps→jumps, teh→the` (cspell, `npm/rules/text/lint/cspell-fix.mjs`); `llm-worker.mjs` рефакторовано на спільне ядро без поломки 16 тестів.
* Bad, because transcript фіксує явний ризик: knip/jscpd/trufflehog залишені поза авто-фіксом навмисно (dynamic-usage хибнопозитиви, деструктивність dedup, рішення людини для секретів); репо було чисте від knip-знахідок, що унеможливило валідацію knip-фіксера in-repo.
