## ADR Concern-based JS + per-policy `target.json` — повна міграція всіх правил `@nitra/cursor`

**Контекст:** Монолітний `rules/<id>/js/check.mjs` на правило ускладнював підтримку: один файл суміщав applies-гейт, FS-перевірки і прямі виклики `runConftestBatch`. Для `abie` це виросло до 1153 рядків. Потрібна була структура, де JS відповідає тільки за FS/cross-file логіку, а запуск rego — за декларативний `target.json`.

**Рішення/Процедура/Факт:**
Реалізовано повну concern-based структуру для всіх ~25 правил пакету `@nitra/cursor`:

1. **Інфраструктура** (`npm/scripts/utils/`): `resolve-target-files.mjs` (picomatch-резолвер `single`/`walkGlob` зі shared walk-кешем, захист від path-traversal), `discover-checkable-rules.mjs` (сканує `js/<concern>/check*.mjs` + `policy/<name>/target.json`, legacy-fallback на плаский `js/check.mjs`), `run-rule.mjs` (applies-гейт → JS-концерни → policy-концерни).
2. **Контракт**: `npm/schemas/target.json` (JSON Schema для `target.json`-маніфестів, без `conftest.combine` і без `walkGlob.required`); `picomatch@^4.0.4` у `dependencies`.
3. **CLI**: `npm/bin/n-cursor.js` — `discoverCheckScripts` повертає `CheckableRule[]`, `runChecks` делегує в `runRule` зі спільним walk-кешем.
4. **Пілот + batch-міграція** (25 правил): `rego` (пілот з `js/applies/check.mjs`), `bun`, `text`, `style-lint`, `php`, `docker`, `npm-module`, `js-lint`, `image-compress`, `capacitor`, `hasura`, `adr` — додано `target.json`; walk-glob: `js-mssql`, `js-bun-db`, `js-bun-redis`, `js-run` (2 концерни), `vue`, `image-avif`, `k8s.*` (8 концернів).
5. **Розпил `abie`**: монолітний `js/check.mjs` → 6 концернів (`applies`, `firebase_hosting`, `hc_pairing`, `env_dns`, `ua_node_selector`, `ua_http_route`) + 7 утиліт (`k8s-tree.mjs` з кешованим `findK8sYamlFiles`/`collectDeploymentDirs`, `overlay-paths.mjs`, `kustomization-patches.mjs`, `http-route.mjs`, `hc-yaml.mjs`, `env-dns.mjs`, `enabled.mjs`).
6. **Виправлені баги**: (a) `namespace = ruleId.concernName` для kebab-id правил давав `style-lint.package_json` — rego забороняє `-` у пакетному ідентифікаторі; виправлено на snake. (b) Picomatch у вигляді масиву `['pos', '!neg']` трактує `!neg` як окремий позитивний матчер; виправлено розділенням на positives/negatives з окремим `isExcluded`-фільтром. (c) `**/` у backtick всередині JSDoc-коментаря ламав bun-парсер (`hc_pairing/check.mjs`).
7. **Discovery-пріоритет**: subdir-концерни (`js/<concern>/`) мають пріоритет над legacy `js/check.mjs`, щоб у перехідному стані правило не запускалось двічі.
8. **Закриті дизайн-питання**: legacy-fallback — без явного іменування концерну (`jsLegacyCheck: 'check.mjs'` окремим полем у `CheckableRule`); `walkGlob.required` залишається забороненим у schema; `conftest.combine` прибрано з schema до першого юзкейса (YAGNI).
9. Bump `1.10.0 → 1.11.0`, запис у `CHANGELOG.md`. ADR — `docs/adr/20260515-123025-concern-target-design.md`.

**Обґрунтування:** Відокремлення JS (cross-file/FS) від rego (per-document структура) дає: CLI контролює запуск rego без дублювання у JS; shared walk-кеш виключає повторний обхід дерева між концернами одного правила; legacy-fallback дозволяє мігрувати правила інкрементально без breaking change. YAGNI щодо `conftest.combine` і `walkGlob.required` — не вводимо механізм без реального юзкейса.

**Розглянуті альтернативи:** (1) Залишити legacy-fallback назавжди — відхилено, запланований cleanup після повної міграції. (2) `conftest.combine: false/true` у schema як булевий прапор «об'єднати документи» — відкладено до першого правила, яке це потребує. (3) Одночасний розпил всіх JS-концернів у одному PR — відхилено на користь інкрементальної міграції з legacy-fallback для ненятронутих правил.
