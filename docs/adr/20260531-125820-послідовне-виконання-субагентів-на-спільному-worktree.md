## ADR Послідовне виконання субагентів на спільному worktree

## Context and Problem Statement
Сесія спробувала запустити всіх субагентів паралельно (Tasks 1–9) в одному блоці Agent-викликів. Усі задачі працювали з одним worktree (`feat-rule-meta-json`), що спричинило каскадне скасування: git-конфлікти/race-condition між агентами. Реально завершився лише Task 1 (`616f832`), решта — скасовані з `tool_use_error: Cancelled`.

## Considered Options
* Паралельний запуск усіх субагентів одним блоком
* Строго послідовне виконання: один агент → перевірка результату → наступний

## Decision Outcome
Chosen option: "Строго послідовне виконання", because субагенти, що ділять один worktree, не можуть працювати паралельно — вони конфліктують по git-операціях та файловій системі. CLAUDE.md прямо забороняє паралельні lint-сесії з тієї ж причини; аналогічний принцип застосовується до будь-яких субагентів на спільному worktree.

### Consequences
* Good, because transcript фіксує очікувану користь: після переходу на послідовний режим усі 9 задач завершились успішно (1978 тестів passed, фінальний review APPROVED 6/6).
* Bad, because загальний час виконання зростає — задачі, що могли б бути незалежними, чекають одна одну.

## More Information
Правило зафіксовано в `npm/rules/worktree/worktree.mdc` (розділ «Заборони»). Commit `7b07bcc fix(worktree-skill)` містить оновлення правила. Паралельний запуск агентів допустимий лише якщо кожен агент має **свій** ізольований worktree.

## ADR Squash-merge як рекомендований спосіб завершення worktree-гілки

## Context and Problem Statement
Після завершення Spec B користувач запитав: «Змерджити локально в `main` — це буде одним комітом?» Виявилося, що fast-forward дає 9 окремих комітів, а `git merge --squash` — один. Користувач хотів squash, але фонова сесія вже виконала FF-merge. Виникла потреба зафіксувати цей патерн у правилі worktree, щоб агент пропонував squash за замовчуванням при завершенні будь-якої гілки.

## Considered Options
* Fast-forward merge (зберігає всі N комітів гілки)
* Squash merge (зводить усі зміни в один коміт)
* Merge-коміт (`--no-ff`, N комітів + 1 merge-коміт)

## Decision Outcome
Chosen option: "Squash merge як рекомендований варіант при завершенні worktree-гілки", because worktree-гілка реалізує одну логічну фічу, і проміжні TDD-кроки не потрібні в `main`. Рекомендація зафіксована в правилі `npm/rules/worktree/worktree.mdc` — агент зобов'язаний пропонувати squash першим; конкретний вибір лишається за користувачем.

### Consequences
* Good, because transcript фіксує очікувану користь: `main` отримує один описовий коміт на фічу замість N технічних кроків TDD.
* Bad, because втрачається покроковий TDD-history гілки. У цій конкретній сесії користувач обрав варіант B (лишити 9 комітів як є), бо FF-merge вже стався через фонову сесію — squash не виконувався.

## More Information
Зміна внесена в `npm/rules/worktree/worktree.mdc` (commit `b2b8e11 feat(worktree-rule)`), change-файл `npm/.changes/1780218783124-a30f10.md`. Нове правило набуде сили в споживачах після релізу наступної версії пакета (sync бере правила з опублікованого `node_modules`, не з локального джерела).

## ADR Data-driven lint-split: поле `meta.json.lint` + CLI-оркестратор

## Context and Problem Statement
Кореневий `bun run lint` — монолітний хардкод-ланцюг із 6 під-лінтів + `oxfmt`. Він важкий для швидкої перевірки поточних змін (агент або розробник): запускає всі інструменти по всіх файлах. Потрібно розділити на `lint` (швидкий, по змінених) і `lint-ci` (повний, по всіх).

## Considered Options
* Додати нові npm-скрипти в кореневий `package.json` (хардкод нового ланцюга)
* CLI-команда пакета як виконавець (F1 + E1 + D3)
* Генерація скриптів через sync (F2)
* Мінімальна зміна: обгорнути наявний ланцюг у diff-фільтр без meta-driven (F3)

## Decision Outcome
Chosen option: "CLI-оркестратор у пакеті + поле `meta.json.lint` + розщеплення `js-lint`", because це єдиний варіант, що реалізує ідею «meta.json керує набором»: оркестратор зчитує атрибут і будує потрібний набір інструментів. Архітектурно узгоджується з `auto-rules.mjs` (data-driven після Spec B) і наявним CLI-патерном (`lint-ga` = CLI-виконавець + тонкий скрипт-делегат).

**Деталі рішень:**

| Вісь | Вибір |
|---|---|
| Конфіг | Поле `meta.json.lint`: `"quick"` \| `"ci"` \| відсутнє; семантика quick ⊆ ci |
| Гранулярність | Атрибут на правилі; `js-lint` розщеплено на quick (oxlint+eslint) і ci (jscpd+knip) |
| База quick | working-tree vs HEAD + untracked нові файли |
| Fix-поведінка | `--fix` в обох режимах; падає на незакритих порушеннях |
| Назви | `lint` (швидкий), `lint-ci` (повний) |
| Scope | Лише пакет `@nitra/cursor`; кореневий `package.json` репо мігрує через sync |

### Consequences
* Good, because transcript фіксує очікувану користь: набір інструментів швидкого/повного lint контролюється декларативно через `meta.json`, без хардкоду в скриптах.
* Bad, because transcript не містить підтверджених негативних наслідків. Відкрите питання: CLI-команди `lint-ga`/`lint-rego`/`lint-text` приймають список файлів для quick-режиму — треба звірити при імплементації; якщо ні — лишаються `ci`.

## More Information
Spec зафіксований у `docs/superpowers/specs/2026-05-31-lint-split-quick-ci-design.md`. Нові файли пакета: `npm/scripts/lint-cli.mjs`, `npm/scripts/lib/changed-files.mjs`. Схема `rule-meta.json` розширюється полем `lint` (enum `quick`/`ci`, optional). Правило `n-js-lint` оновлює канонічні скрипти `lint`/`lint-ci`/`lint-js`/`lint-js-ci`. Строга CI-no-fix семантика (H2) і зміна GitHub workflow — поза scope цього spec.
