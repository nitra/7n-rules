## ADR CLI-екстрактори замість LLM-парсингу великих артефактів у скілах

## Context and Problem Statement
Скіли (`n-coverage-fix`, `n-taze`, `n-start-check`, `n-fix`) містили кроки, де LLM-агент вручну парсив великі або гетерогенні артефакти (`COVERAGE.md` ~2.76 МБ ≈ 700K токенів, `package.json.taze-bak`, glob-воркспейси, текстовий вивід `fix`), тоді як ця робота детермінована і могла виконуватись скриптом за 0 LLM-токенів. Після впровадження `n-cursor coverage-fix index|slice` патерн узагальнили на решту скілів.

## Considered Options
* Read-only CLI-екстрактор (`n-cursor <skill> <verb>`): скрипт несе весь детермінований парсинг, агент отримує компактний `index` або `slice`
* Залишити LLM-парсинг у SKILL.md (попередній підхід)

## Decision Outcome
Chosen option: "Read-only CLI-екстрактор", because детермінований парсинг є нульовою цінністю для LLM і водночас найбільшим токен-навантаженням (COVERAGE.md ≈ 700K токенів): скрипт виконує його за 0 LLM-токенів, а агент отримує лише когнітивно необхідну порцію.

### Consequences
* Good, because `coverage-fix index` (7.4 КБ / ~2K токенів) замінює читання COVERAGE.md (~700K токенів) — ≈350× менше для оркестратора; transcript фіксує очікувану користь.
* Good, because `taze diff`, `start-check scan|run`, `fix --json` усувають ручне порівняння бекапів, розгортання glob і парсинг текстового виводу — детермінована робота йде в CLI.
* Good, because патерн зафіксований у `scripts.mdc` v1.13 як обов'язковий чекліст перед фіналізацією будь-якого скіла.
* Bad, because нові підкоманди CLI (`taze`, `start-check`, `coverage-fix`, `fix --json`) активуються для споживачів лише після публікації `@nitra/cursor`; до релізу між SKILL.md і бінарем є version-skew (3.24.0 local vs 3.20.0 installed).

## More Information
Нові файли: `npm/scripts/coverage-fix-extract.mjs`, `npm/skills/taze/js/diff.mjs`, `npm/skills/start-check/js/check.mjs`; змінено: `npm/bin/n-cursor.js` (case `coverage-fix`, `taze`, `start-check`; прапорець `--json` у `runFixCommand`); правило `scripts.mdc` v1.13 (секція «🔴 ВИСОКИЙ ПРІОРИТЕТ — детермінований парсинг у скіла»). Контракт: `index` повертає `[{file, mutants}]`, `slice --file` — промпт під один файл, `taze diff` — `{major, minorPatch, totalChanged, comparedWorkspaces}`, `start-check scan` — `[{workspace, name, hasStart, startCmd, type}]`, `start-check run <ws>` — `{type, exitCode, timedOut, status, ready, firstError, logTail, sideEffects}`. Скіли `worktree` та `adr-normalize` визнані еталонами (уже чисті). Скіл `lint` (`n-lint`) визнаний непридатним для цього патерну: `&&`-ланцюг ~10 тулів без єдиного JSON-формату, human-in-the-loop гейти, ризик подвійного прогону.
