## ADR Egress-правило `169.254.0.0/16` для DNS через GKE NodeLocal DNSCache у NetworkPolicy

## Context and Problem Statement
У GKE-кластерах з увімкненим NodeLocal DNSCache kubelet налаштовує `/etc/resolv.conf` подів не на ClusterIP kube-dns, а на link-local DNS-агент ноди (адреса в діапазоні `169.254.0.0/16`, RFC 3927). NetworkPolicy з `policyTypes: [Egress]` без правила на цей діапазон блокувала DNS-запити до того, як вони могли дістатися kube-dns, що призводило до збоїв у резолвінгу всіх cluster-сервісів.

## Considered Options
* Додати egress-правило `ipBlock: 169.254.0.0/16`, порти 53/UDP+TCP
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Додати egress-правило `ipBlock: 169.254.0.0/16`, порти 53/UDP+TCP", because kubelet на GKE спрямовує DNS-трафік пода до link-local агента (NodeLocal DNSCache), і правило на `kube-system/kube-dns` його не покриває — трафік йде на `169.254.x.x:53`, а не на ClusterIP.

### Consequences
* Good, because transcript фіксує очікувану користь: DNS-резолвінг в подах з NetworkPolicy на GKE більше не блокується NodeLocal DNSCache.
* Bad, because transcript не містить підтверджених негативних наслідків. У non-GKE кластерах правило не впливає (на `169.254.x.x` там ніхто не слухає).

## More Information
Правило додано до `npm/rules/k8s/policy/network_policy/template/networkpolicy.snippet.yaml` (між блоком kube-dns і блоком 0.0.0.0/0). Діапазон `169.254.0.0/16` — link-local (RFC 3927), не маршрутизується в інтернет. Змінено через Edit в поточному transcript.

## ADR `networkpolicy.snippet.yaml` як єдине джерело правди для структури NetworkPolicy

## Context and Problem Statement
Канон `spec` NetworkPolicy (egress-правила, ingress, policyTypes) дублювався у п'яти місцях одночасно: рядкова константа `NETWORK_POLICY_EGRESS_YAML` і масив `NETWORK_POLICY_IN_CLUSTER_DEFAULT_PORTS` у `manifests.mjs`, функція `networkPolicyManifestViolations` (JS-валідатор), фікстура `valid_np` у `network_policy_test.rego`, і сам `networkpolicy.snippet.yaml` як «еталон для очей». Зміна будь-якого правила (наприклад, додавання link-local DNS) потребувала ручної синхронізації в усіх п'яти місцях.

## Considered Options
* `networkpolicy.snippet.yaml` → JS читає snippet, rego синхронізується вручну (документована угода)
* `networkpolicy.snippet.yaml` → JS читає snippet, rego перевіряє через `conftest --data snippet` + deep-equal
* Повне видалення rego; єдина перевірка через JS
* Rego як єдиний канон (snippet як допоміжний файл)

## Decision Outcome
Chosen option: "`networkpolicy.snippet.yaml` → JS читає snippet, rego синхронізується вручну", because цей варіант мав найменше рухомих частин, reалізація Tasks 2–3 вже була виконана в попередній сесії і тести зеленіли (182 pass), а conftest deep-equal підхід — значно складніший без відповідного приросту цінності.

### Consequences
* Good, because transcript фіксує очікувану користь: `buildNetworkPolicyYaml` більше не містить власного вбудованого шаблону; зміна canonical egress = редагування одного YAML.
* Good, because функція `networkPolicyManifestViolations` видалена — структурна перевірка делегована rego, JS зменшився до тонкого I/O-шару.
* Bad, because `valid_np` у `network_policy_test.rego` оновлюється вручну (конвенція «в тому ж коміті»), а не автоматично зі snippet.
