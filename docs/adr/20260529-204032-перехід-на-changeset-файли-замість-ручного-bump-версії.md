## ADR Перехід на changeset-файли замість ручного bump версії

## Context and Problem Statement

У bun-монорепо `@nitra/cursor` кожен PR зобов'язаний вручну підняти `version` у `package.json` і дописати секцію в `CHANGELOG.md`. Якщо два розробники виконують цей крок одночасно від однієї бази `main` — гарантований git-конфлікт у цих двох файлах, оскільки обидва редагують ті самі рядки.

## Considered Options

* Власний мінімальний флоу «всередині n-cursor» — новий subcommand `n-cursor release`, файли `.changes/<workspace>/`
* `@changesets/cli` з коробки
* Conventional Commits + `release-please`

## Decision Outcome

Chosen option: "Власний мінімальний флоу «всередині n-cursor»", because він не тягне нових npm-залежностей, узгоджується з поточним стилем CLI (`n-cursor fix`, `n-cursor check`), і при переході до цілі B (споживачі пакету) — команда вже «всередині інструменту». `@changesets/cli` вимагає конфігурації і дублює наявний publish-крок; `release-please` вимагає зміни workflow команди (коміти у форматі Conventional Commits) і найбільшого setup.

### Consequences

* Good, because transcript фіксує очікувану користь: конфлікт у `package.json` і `CHANGELOG.md` зникає by design — два розробники пишуть різні файли у `.changes/<workspace>/`.
* Bad, because transcript фіксує: `parseChangelog.mjs` потребує розширення для режиму запису (зараз лише читає); перехідний період вимагає одночасної міграції всієї команди.

## More Information

Специфікація: `docs/superpowers/specs/2026-05-29-n-cursor-release-design.md`. План реалізації: `docs/superpowers/plans/2026-05-29-n-cursor-release-plan.md`. Нові файли: `npm/scripts/release.mjs`, `.changes/npm/`, `.changes/rego/`. Зміни: `npm/rules/changelog/js/consistency.mjs`, `.cursor/rules/n-changelog.mdc`, `.github/workflows/npm-publish.yml`.

## ADR Формат файлів `.changes/<workspace>/`

## Context and Problem Statement

Для нового changeset-флоу потрібно визначити, як розробник ідентифікує, до якого workspace належить його changeset-файл, і як у файлі кодується рівень semver-bump.

## Considered Options

* Директорія = workspace: `.changes/<workspace-slug>/<slug>.md` (A)
* YAML front-matter з полем `workspace:` і `bump:` у плоскому `.changes/<slug>.md` (C)
* Ім'я файла кодує workspace: `.changes/<workspace>__<slug>.md` (B)
* Перший рядок файла — тег `patch`/`minor`/`major`; ідентифікація через директорію (A+A)
* Conventional Commits у git-лозі — без `.changes/`-файлів взагалі (B для bump)

## Decision Outcome

Chosen option: "Директорія = workspace + plaintext тег першим рядком", because це мінімально проблемний варіант: `ls .changes/npm/*.md` дає все потрібне без парсингу метаданих, YAML-парсера або ризику помилки у полі; тег першим рядком — простий grep без залежностей.

### Consequences

* Good, because transcript фіксує очікувану користь: нульова складність парсингу, нуль нових залежностей, читається людиною без інструментів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Формат файла: рядок 1 — `patch` | `minor` | `major`; рядки 2+ — один або кілька рядків опису, кожен непорожній рядок стає окремим пунктом у CHANGELOG. Правило злиття рівнів: `major` > `minor` > `patch` — береться максимум при кількох файлах. Специфікація: `docs/superpowers/specs/2026-05-29-n-cursor-release-design.md`, секція 1.

## ADR CI-патерн для автоматичного релізу

## Context and Problem Statement

Після вибору changeset-флоу потрібно визначити, де і як `n-cursor release` запускається в CI: як додати bump/commit у наявний `npm-publish.yml` (тригер `push main`, шлях `npm/**`) без зациклення workflows і без введення додаткових токенів.

## Considered Options

* Патерн A — додаткові steps у наявному `npm-publish.yml` (release + commit-back + publish в одному job)
* Патерн B — окремий `release.yml` + тригер на push тега `v*` для publish (потребує PAT або Deploy Key)
* Патерн C — `release-please`-стиль з Release PR (найбільше роботи, окремий бот)

## Decision Outcome

Chosen option: "Патерн A — додаткові steps у наявному `npm-publish.yml`", because мінімальна зміна відносно поточного workflow: тільки `contents: read` → `contents: write` і два нових кроки перед publish. `GITHUB_TOKEN` за замовчуванням не тригерить інші workflows — loop-guard забезпечений. Патерн B вимагає PAT/Deploy Key; Патерн C — найбільше інфраструктурних змін.

### Consequences

* Good, because transcript фіксує очікувану користь: існуючий `concurrency`-блок серіалізує паралельні merge-и; `[skip ci]` у commit-повідомленні — додаткова страховка; коли `.changes/<ws>/` порожній — `git diff --cached --quiet` = true і коміт не відбувається, publish лишається без змін.
* Bad, because bump і publish мають бути в одному job (інакше commit-back не тригерить publish при Патерні A з `GITHUB_TOKEN`).

## More Information

Конкретні зміни у `.github/workflows/npm-publish.yml`: `permissions.contents: write`; два нових steps — `node npm/bin/n-cursor.js release` і `git add npm/package.json npm/CHANGELOG.md .changes/ && git commit -m "chore: release [skip ci]" && git push`. Специфікація: `docs/superpowers/specs/2026-05-29-n-cursor-release-design.md`, секція 3.
