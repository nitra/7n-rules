## ADR Двосторонній зв'язок spec→plan через поле `Plan:` у специфікаціях

## Context and Problem Statement
Специфікації у `docs/superpowers/specs/` мають поле `**Status:**`, але не мають зворотного лінку на відповідний план у `docs/superpowers/plans/`. Це створює однобічний зв'язок: план посилається на spec через `**Spec:**`, але spec не відображає факт переходу до стадії планування.

## Considered Options
* Додати порожнє поле `**Plan:**` у шаблон spec (`brainstorming`) і патчити його у `writing-plans`
* Автоматизувати оновлення статусу через хук або CI
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Додати поле `Plan:` у шаблон spec і патчити через `writing-plans`", because це суто конвенція без додаткової автоматики: скіл `brainstorming` включає порожнє `**Plan:**` у шаблон для узгодженості, а скіл `writing-plans` під час створення плану патчить файл spec, вписуючи лінк назад — оскільки обидва скіли вже відповідають за ці файли.

### Consequences
* Good, because transcript фіксує очікувану користь: стан спеки можна відстежувати за наявністю лінку — порожнє `Plan:` означає "ще не перейшла до планування", заповнене — є готовий план.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Скіли для патчу: `superpowers/skills/brainstorming/SKILL.md` (шаблон spec), `superpowers/skills/writing-plans/SKILL.md` (патч spec-файлу)
- Поточний стан `writing-plans` вже генерує `**Spec:**` лінк у плані та `**Status:** In progress`
- Поточний стан `brainstorming` зберігає spec у `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- Поле `**Plan:**` залишається порожнім до моменту запуску `writing-plans`
