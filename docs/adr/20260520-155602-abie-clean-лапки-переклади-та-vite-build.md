# Конвенція лапок та `bun vite build` у скілі `abie-clean`

**Status:** Accepted
**Date:** 2026-05-20

## Context and Problem Statement

Скіл `abie-clean` генерує англійські рядки перекладу, що можуть містити апострофи (скорочення `'s`, `'t`, `'re`). Якщо такий рядок обгортати одинарними лапками, апостроф або зламує парсинг, або потребує екранування, яке легко зробити неправильно. Синтаксичні помилки у JS-файлах ламали всі три CI-пайплайни (`android`, `ios`, `site`) без попереднього локального виявлення.

## Considered Options

- Залишати одинарні лапки й екранувати апостроф
- Перемикатися на подвійні лапки, коли переклад містить апостроф
- Додати `bun vite build` як обов'язковий крок верифікації перед комітом

## Decision Outcome

Chosen option: "Подвійні лапки при апострофах + `bun vite build` перед комітом", because подвійні лапки повністю усувають клас помилок з екрануванням; `bun vite build` ловить синтаксичні помилки в перекладах локально до потрапляння у CI.

### Consequences

- Good, because синтаксичні помилки у translation-об'єктах виявляються локально через `bun vite build`, не ламаючи CI.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Рішення зафіксоване у `npm/skills/abie-clean/SKILL.md` — новий розділ §5.1 «Апострофи та спецсимволи в рядках перекладу» і доповнений §6.
- Алгоритм: після генерації перекладів перевіряти наявність апострофа, подвійних лапок і зворотної косої — обирати безпечний роздільник.
- `bun vite build` застосовується лише в проєктах із Vite.
- Команда верифікації CI: `npx @nitra/cursor check abie`.
- Версія пакета: `1.13.64` → `1.13.65`; запис у `npm/CHANGELOG.md`.
