## ADR Автономний оркестратор n-fix: CLI як виконавець замість агента

## Context and Problem Statement
Поточна реалізація `n-cursor fix` є лише перевірником: вона виводить порушення, але не виправляє їх. Логіка виправлення живе у SKILL.md, яку читає агент-LLM і виконує кроками. Це вимагає від агента аналізу JSON, прийняття рішень і ескалації — всього, що можна автоматизувати у CLI.

## Considered Options
* Залишити `fix` як чекер, агент сам виправляє (поточний стан)
* Зробити `fix`, `lint`, `taze` автономними оркестраторами (convergence-loop + LLM-tier всередині CLI)
* Додати MCP-сервер як інтерфейс для агентів

## Decision Outcome
Chosen option: "Автономні CLI-оркестратори (`fix`, `lint`, `taze`) з тонкими скілами", because агент не повинен знати як виправляти — він викликає `npx @nitra/cursor fix` і перевіряє exit code; логіка convergence-loop, T0-auto, LLM-tier і ескалація переносяться всередину CLI.

### Consequences
* Good, because SKILL.md зводиться до однієї команди (`npx @nitra/cursor fix`), агент не парсить JSON і не приймає рішень.
* Good, because transcript фіксує очікувану користь: T0-auto вже доведено (3/3 правила за 4s, 0 LLM-токенів); оркестратор розширює цей патерн.
* Bad, because `fix --json` видаляється — CI/скрипти, які на нього покладаються, потребують адаптації (у transcript явно сказано "fix --json не потрібен").

## More Information
- `npm/skills/fix/js/t0.mjs` — T0-auto патерн (vscode-ext-add, rm-forbidden-file)
- `npm/bin/n-cursor.js` — case `fix-t0` вже реалізований
- LLM-tier: всі виклики через `pi` (C1-патерн: orchestrator збирає контекст → pi повертає replacement → orchestrator застосовує; без tool-use на стороні LLM)
- Ескалація: haiku → sonnet через `pi --model`
- Worktree: fix → lint послідовно в одному ворктрі (не паралельні під-ворктрі; fix і lint не незалежні — fix створює конфіги, які lint перевіряє)
- MCP-сервер: не в поточній реалізації; розглянути пізніше якщо потрібні типізовані параметри або streaming прогрес

## ADR T0-auto: детермінований рівень виправлень (0 LLM-токенів)

## Context and Problem Statement
Під час tool-free експерименту з gemma3:4b виявлено, що violation-output `fix --json` вже містить точні значення для виправлення (наприклад, `recommendations має містити "tsandall.opa"`). Спроба використати local LLM для цього класу задач виявилась ненадійною: warm inference 78-90s на M2 8GB, галюцинації ("stoutler.dotenv" замість "stylelint.vscode-stylelint").

## Considered Options
* Використати local LLM (gemma3:4b/gemma4:4b) для config-фіксів
* Regex-парсинг violation-output → програмний фікс (T0-auto)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "T0-auto: regex-парсинг violation-output → програмний фікс", because violation-messages структуровані і містять конкретні значення; детермінований патерн надійніший (100%) і миттєвий (0 LLM-токенів) порівняно з local 4B моделлю.

### Consequences
* Good, because transcript фіксує очікувану користь: 3/3 правил закрито за 4.2s (wall time rules loop), 100% коректність.
* Good, because нова категорія в ієрархії: T0 (rm/create, hardcoded) → T0-auto (parse violation → fix) → T1 (LLM); кожен тір простіший і дешевший за наступний.
* Bad, because T0-auto покриває лише паттерни з чіткою структурою у violation-output; нові типи порушень потребують нових регекспів у `npm/skills/fix/js/t0.mjs`.

## More Information

- `npm/skills/fix/js/tests/t0.test.mjs` — 11 тестів, всі pass

## ADR Worktree-стратегія для fix + lint: послідовне виконання в одному ворктрі

## Context and Problem Statement
При виконанні `fix` і `lint` разом постало питання ізоляції: чи виконувати послідовно в одному ворктрі агента, чи паралельно в окремих під-ворктрях (створених від поточного ворктрі), а потім мержити.

## Considered Options
* Послідовно в одному ворктрі: fix → lint → done
* Паралельні під-ворктрі: fix-worktree ∥ lint-worktree → merge назад

## Decision Outcome
Chosen option: "Послідовно в одному ворктрі", because fix і lint не є незалежними — fix створює конфіги (extensions.json, package.json), які lint потім перевіряє; паралельне виконання гарантує конфлікти при мержі спільних файлів.

### Consequences
* Good, because немає конфліктів при мержі; lint бачить стан після fix.
* Bad, because повільніше ніж паралельне виконання; transcript не містить підтверджених негативних наслідків окрім часу виконання.

## More Information
Додаткової інформації в transcript не зафіксовано.

## ADR Всі LLM-виклики оркестратора через `pi` (C1-патерн)

## Context and Problem Statement
Оркестратор потребує LLM для складних виправлень (T1-тір). Потрібно вирішити як передавати API-ключі та яку архітектуру промпту використовувати (з tool-use чи без).

## Considered Options
* Anthropic SDK напряму (`ANTHROPIC_API_KEY` в env)
* `pi` CLI як proxy (C1): orchestrator збирає контекст → `pi` повертає replacement → orchestrator застосовує
* `pi` CLI з `--tools` (C2): LLM сам читає/пише файли через tool-use

## Decision Outcome
Chosen option: "`pi` CLI (C1-патерн, без tool-use на стороні LLM)", because `pi` дозволяє кожному користувачу самостійно налаштовувати ключі доступу та провайдери; C1 простіший за C2 — LLM stateless, отримує self-contained prompt, повертає replacement, orchestrator застосовує програмно.

### Consequences
* Good, because transcript фіксує очікувану користь: провайдер-нейтральність, user-controlled ключі, ескалація `--model claude-haiku-4-5` → `--model claude-sonnet-4-6`.
* Bad, because Neutral, because transcript не містить підтвердження наслідку щодо обмежень pi в production-use.

## More Information
- `pi` вже використовується у `benchmarks/tool-free/run.mjs` (`spawnSync('pi', ['-p', prompt, '--provider', 'ollama', '--model', 'gemma3:4b', '--no-session', '--no-tools', '--mode', 'text'])`)
- Ескалація: haiku (default) → sonnet (якщо haiku fails 2x на тому ж rule)
- `~/.pi/agent/models.json` — конфіг провайдерів (openai-completions, localhost:11434)
