# Префікс `n-` як межа між керованими та користувацькими файлами в `.cursor/`

**Status:** Accepted
**Date:** 2026-05-21

## Context and Problem Statement

CLI `@nitra/cursor` синхронізує правила та скіли в `.cursor/rules/` і `.cursor/skills/` проєкту з пакетом, але проєктна директорія також може містити власні правила та скіли без відношення до пакета. Потрібен механізм, щоб CLI знав, які файли він може перезаписувати або видаляти, а які залишати без змін.

## Considered Options

- Префікс `n-` для файлів, керованих CLI; без префікса — власність проєкту
- Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Префікс `n-` для файлів, керованих CLI", because `npx @nitra/cursor` перезаписує лише `n-*.mdc` і `n-*/SKILL.md` (за списком у `.n-cursor.json`), видаляє лише `n-*`-orphan'и; файли без префікса (`conftest.mdc`, `dev-dep.mdc`, `mdc-check/`) CLI не чіпає.

### Consequences

- Good, because ручні правила (`*.mdc` без `n-`) і власні скіли (директорії без `n-`) зберігаються між запусками `npx @nitra/cursor` і автоматично потрапляють в індекси `CLAUDE.md` / `AGENTS.md` через `listProjectRulesMdcFiles`, яка сканує всі `*.mdc` без фільтрації за префіксом.
- Bad, because будь-які ручні правки всередині `n-*.mdc` або `n-*/SKILL.md` перезаписуються при наступному синку — канонічне джерело для них лише в `npm/rules/` і `npm/skills/` пакета.
- Neutral, because `CLAUDE.md` і `AGENTS.md` повністю перегенеруються кожним запуском `npx @nitra/cursor`; ручні правки в цих файлах не зберігаються. Статичний шаблон — `npm/AGENTS.template.md` у пакеті.

## More Information

- `npm/bin/n-cursor.js:489–496` — `listProjectRulesMdcFiles` сканує весь `RULES_DIR` без фільтра за префіксом: `readdir(rulesDir)` + `.filter(n => n.endsWith('.mdc'))`. Аналогічно `listProjectSkillDirNames()` повертає всі підкаталоги `.cursor/skills/`.
- Конфігурація керованих id правил і скілів — `.n-cursor.json` у споживчому проєкті, секції `rules` і `skills`.
- Orphan-cleanup видаляє лише `n-*`-файли та `n-*`-каталоги, що не входять до поточного списку `.n-cursor.json`.

## Update 2026-05-21

### Мотивація повної перегенерації замість патч-оновлення

Вибір повної перегенерації `CLAUDE.md` / `AGENTS.md` при кожному синку обумовлений тим, що `CLAUDE.md` містить попередження про автогенерацію, а статичний контент для всіх проєктів зберігається в `npm/AGENTS.template.md` пакета `@nitra/cursor`. Це гарантує, що індекс завжди відображає поточний стан `.cursor/rules/` та `.cursor/skills/` без ручного обслуговування. Будь-які ручні правки безпосередньо в цих файлах — не зберігаються: наступний `npx @nitra/cursor` їх перезапише.

## Update 2026-05-21

### Пропозиція: disk-only модель для `check`, hooks і slash-команд (Варіант A — м'який синк)

Поточна архітектура має дві паралельні моделі: індекс агентів формується з диска, але `check`, ADR hooks, orphan-cleanup і slash-команди залежать від масивів `rules`/`skills` у `.n-cursor.json`. Рекомендований мінімальний крок до уніфікації (Варіант A; реалізація поза scope сесії):

1. `check` без аргументів сканує `.cursor/rules/*.mdc` замість парсингу `AGENTS.md`; id: `n-bun.mdc` → `bun`, `my-rule.mdc` → `my-rule`.
2. ADR hooks перевіряють `existsSync('.cursor/rules/n-adr.mdc')` замість `config.rules.includes('adr')`.
3. `.claude/commands/*.md` генеруються для кожного каталогу в `.cursor/skills/`, не лише за масивом `skills` у `.n-cursor.json`.
4. Синк не перезаписує наявні `n-*` без `--force`; orphan-prune — окрема команда `npx @nitra/cursor prune`.

Варіант B (жорсткий disk-only: синк взагалі не чіпає `.cursor/`; пакет лише надає `npm/rules/` для ручного копіювання) відхилений як надто радикальний крок.

Каверза: якщо старий `n-adr.mdc` лежить на диску після вимкнення в конфігу, ADR hook лишається активним без автоматичного orphan-delete.
