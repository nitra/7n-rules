## ADR Рекурсивний складений ОАГ з динамічним розкладом вузлів

## Context and Problem Statement
Потрібна структура для представлення задач, де директорія містить проєкти, кожен проєкт має граф задач, а кожен вузол цього графу може сам містити підграф — рекурсивно та без обмежень глибини. Постало питання як правильно назвати і змоделювати цю структуру.

## Considered Options
* Recursive Compound DAG (рекурсивний складений орієнтований ациклічний граф)
* "3D-граф" (відхилено: описує просторове відображення, а не структуру зв'язків)
* Multi-level Graph

## Decision Outcome
Chosen option: "Recursive Compound DAG", because вузол або є атомарним (виконується безпосередньо), або є складеним (містить підграф); для батьківського рівня інтерфейс однаковий — вузол або `вирішено` або ні, незалежно від внутрішньої реалізації.

### Consequences
* Good, because transcript фіксує очікувану користь: будь-який атомарний вузол можна "розкрити" у підграф без змін у батьківському графі (замінюваність через інтерфейс).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Executor однаковий на всіх рівнях: `execute(node)` → якщо атомарний — запускає fn, інакше рекурсивно виконує підграф і чекає exit-вузлів. Ребра — data-flow: `from: NodeId+portId → to: NodeId+portId`. Кілька exit-вузлів підграфу merge-уються в `outputs` батька. Близькі реалізації: Dask, Prefect dynamic tasks, LangGraph.

## ADR Файлове сховище стану в режимі append-only

## Context and Problem Statement
Граф задач виконується розподілено агентами. Постало питання де і як зберігати стан: централізовано в оркестраторі чи кожен вузол сам; мутабельні файли чи append-only; як гарантувати атомарність запису.

## Considered Options
* Append-only файли: стан кодується наявністю файлів, нові файли лише створюються
* Мутабельні JSON-файли зі зміною `state` поля

## Decision Outcome
Chosen option: "Append-only файли", because `create file` є атомарною операцією на рівні ФС; наявність `вихідні.md` сама є сигналом `resolved` без окремого хука; два паралельних ворктрі пишуть у різні директорії і `git merge` завжди чистий.

### Consequences
* Good, because transcript фіксує очікувану користь: відсутність race conditions на запис (файл створюється один раз), природній аудит-trail, безкоштовна персистентність.
* Bad, because для визначення поточного стану потрібен scan файлів — немає індексу; при великих графах це дорого.

## More Information
Файлова структура вузла: `місія.md`, `вхідні.md`, `вихідні.md`, `помилка.md`, `скасовано`, `знедійснений`, `операції/spawn-план.md`, `операції/spawn-факт.md`, `патчі/001-план.md`, `патчі/001-факт.md`, `repair_history.md`. Формат: Markdown з YAML-фронтматером (LLM-friendly). `вхідні.md`/`вихідні.md` містять **посилання** на файли або рядки — не копії даних.

## ADR Топологія графу розподілена по `місія.md` дочірніх вузлів

## Context and Problem Statement
При динамічному spawn підграфу постає питання де зберігати топологію (ребра між вузлами): в одному центральному `граф.md` батька чи розподілено по файлах дочірніх вузлів.

## Considered Options
* Розподілена топологія: кожен дочірній вузол у своїй `місія.md` оголошує `залежності:` (своїх попередників)
* Центральний `граф.md` у батьківській директорії з повним списком вузлів і ребер

## Decision Outcome
Chosen option: "Розподілена топологія в `місія.md` дочірніх", because це забезпечує універсальність — кожен вузол самодостатній; динамічний spawn (додавання нового вузла в процесі) не потребує оновлення центрального файлу; оркестратор відновлює повний граф скануванням `місія.md` усіх дочірніх.

### Consequences
* Good, because transcript фіксує очікувану користь: динамічний spawn зводиться до створення нового `місія.md` з `залежності:` — жодного центрального файлу оновлювати не потрібно.
* Bad, because відновлення топології потребує сканування всіх дочірніх директорій; при великій кількості вузлів — потенційно дорого (поєднується зі слабкою стороною append-only стору).

## More Information
Поле `залежності:` у фронтматері `місія.md` містить список `NodeId` попередників. Агент-автор підграфу (батьківський агент) пише всі `місія.md` дочірніх при spawn. Динамічний spawn можливий і в процесі виконання — нові вузли просто додаються з посиланнями на вже існуючі.

## ADR Патерн "план → дія → факт" для всіх операцій системи

## Context and Problem Statement
Агент може впасти між кроком "kill залежних ворктрі" і "патч вузла". Без механізму відновлення система опиняється в невизначеному стані: залежні вбиті, але патч не застосований.

## Considered Options
* Write-ahead log через файли: спочатку `*-план.md` (намір), потім дія, потім `*-факт.md` (результат)
* Транзакційний менеджер (зовнішній координатор)

## Decision Outcome
Chosen option: "Write-ahead log через файли", because це природньо вписується в append-only файлову модель; recovery зводиться до scan `*-план.md` без відповідного `*-факт.md` → незавершена операція → продовжити.

### Consequences
* Good, because transcript фіксує очікувану користь: детермінований recovery після збою без зовнішнього координатора; операції аудитуються через пари план/факт файлів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Патерн застосовується до всіх операцій: spawn (`spawn-план.md`/`spawn-факт.md`), kill залежних (`kill-план.md`/`kill-факт.md`), патч інженера (`патч-план.md`/`патч-факт.md`). Протокол ретроактивного патчу вузла: (1) записати `патч-план.md`, (2) kill залежних ворктрі у топологічному порядку від листів, (3) застосувати патч, (4) записати `патч-факт.md`, (5) рестарт каскаду.

## ADR Ворктрі як межа атомарності для паралельного виконання

## Context and Problem Statement
Кілька агентів можуть виконуватись паралельно над незалежними вузлами. Постає проблема race conditions при запису стану і конфліктів при злитті.

## Considered Options
* Кожен вузол/патч виконується у власному git worktree; merge = єдина точка рішення
* Централізований lock-менеджер

## Decision Outcome
Chosen option: "Git worktree як межа атомарності", because поки ворктрі не злито — результати ізольовані; наступники не стартують до merge; merge є єдиною точкою де оркестратор (людина) вирішує чи приймати результат.

### Consequences
* Good, because transcript фіксує очікувану користь: race condition між "агент пише вихідні.md" і "інженер скасовує" не є проблемою — merge gate не дозволить наступникам стартувати зі stale даними.
* Bad, because масштаб — 100+ паралельних ворктрі на MacBook; вирішено жорсткими мінімальними лімітами на старті.

## More Information
Конфлікт при злитті (два ворктрі модифікували спільний батьківський файл) вирішує агент-медіатор — той самий патерн відновлення що і при помилці вузла. Оркестратор на даний момент — людина вручну.

## ADR Часовий бюджет замість ліміту спроб для самовідновлення

## Context and Problem Statement
Потрібен механізм запобігання нескінченному циклу відновлення: агент-інженер може нескінченно патчити вузол що раз за разом падає.

## Considered Options
* Time budget: інженеру виділяється фіксований час (наприклад 10 хв) з необмеженою кількістю спроб
* Max attempts: жорсткий ліміт на кількість спроб

## Decision Outcome
Chosen option: "Time budget", because реалістичніше обмеження ніж лічильник; інженер може адаптувати стратегію залежно від залишку часу (багато часу → складна спроба, мало → швидкий fix); глибина ескалації × бюджет = передбачуваний максимальний час до сповіщення старшого інженера.

### Consequences
* Good, because transcript фіксує очікувану користь: кожен рівень ієрархії отримує **свіжий** бюджет при ескалації — максимальний час до людини = `глибина × бюджет`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`repair_context.json` містить `deadline`, `started_at`, `time_budget_sec`. `repair_history.md` — append-only журнал спроб, зберігається на вузлі де відбулись зміни (не на інженері — інженер stateless). При `timeout` → `стан = unresolvable` → escalate на батьківський рівень. Root timeout → сповіщення `SeniorEngineer` з повним `senior_report.json`. Інженер є мета-рівнем поза графом, має необмежений доступ до будь-якого рівня ієрархії, включно з коренем.
