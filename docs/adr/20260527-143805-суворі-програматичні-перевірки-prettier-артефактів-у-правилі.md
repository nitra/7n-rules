## ADR Суворі програматичні перевірки Prettier-артефактів у правилі `text`

## Context and Problem Statement
Правило `text.mdc` вже забороняло Prettier у полях `dependencies`/`devDependencies`/top-level `package.json`, але не перевіряло програматично: (а) наявність конфіг-файлів (`.prettierignore`, `.prettierrc*`, `prettier.config.*`) у корені проєкту; (б) скрипти `package.json#scripts.*`, що містять виклик `prettier`. Ці прогалини дозволяли Prettier-налаштуванням «просочуватися» без видимого порушення.

## Considered Options
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Окремий JS concern `forbidden-prettier.mjs` + Rego deny для `scripts.*`", because transcript містить явні acceptance checks: repo з `.prettierignore` повинен давати `❌ .prettierignore заборонено`, repo з `"fix": "bunx prettier --write ."` — `❌ package.json: scripts.fix містить prettier`, а чистий repo — exit 0.

### Consequences
* Good, because transcript фіксує очікувану користь: `n-cursor fix text` тепер падає на `.prettierignore` + `bunx prettier` у fixture-прогоні (exit 1), а чистий tmpdir → exit 0.
* Good, because логіка розділена на два рівні: FS-перевірка у `js/forbidden-prettier.mjs` (auto-discovered runner'ом), token-based Rego-перевірка у `policy/package_json/package_json.rego`; дублюючий inline-цикл прибрано з `formatting.mjs`.

## ADR Усунення `process.chdir` race у паралельних vitest-тестах

## Context and Problem Statement
`withTmpCwd` у `npm/scripts/utils/test-helpers.mjs` мутує глобальний `process.cwd()` навколо тестових блоків. У паралельному vitest (default `pool: 'threads'`) різні тестові файли поділяють один процес і один `process.cwd()`. Тести `rules/changelog/js/tests/consistency/tests/check.test.mjs` виконують `git init`/`git commit -m init`/`git checkout -b feat/<x>` — коли race перехоплював `cwd` назад до кореня репо, ці `git`-команди потрапляли у реальний репозиторій: з'являлись rogue branches (`feat/x`, `feat/docs`, `feat/sync`) і rogue commits (`Author: test <test@test>`), які руйнували `npm/CHANGELOG.md` та `npm/package.json` (2350+ deletions).

## Considered Options
* Варіант А: окремий `vitest.stryker.config.js` з `include: ['rules/test/coverage/tests/**']` — Stryker не завантажує git-тести, симптом зникає, корінь лишається
* Варіант Б: повний refactor `withTmpCwd` → `withTmpDir` (без `process.chdir`), явна передача `dir`/`cwd:` у всі виклики; доповнення `test.mdc` і policy guard у `npm/rules/test/`

## Decision Outcome
Chosen option: "Варіант Б — повний refactor + policy guard", because користувач явно зазначив: «робимо Б і доповнюємо правило test, і політиками контролюємо щоб подобне повторилось в жодному проекті».

### Consequences
* Good, because transcript фіксує очікувану користь: race-bug усувається в кореневій причині; будь-який pool (threads/forks/inline Stryker) стає безпечним для паралельного запуску.
* Good, because policy guard у `npm/rules/test/` (новий JS concern `no-process-chdir.mjs` + Rego-перевірка `pool: 'forks'` у `vitest.config.js`) унеможливлює повторення в будь-якому підключеному проєкті.
* Bad, because transcript фіксує: зміна торкається 43+ тестових файлів і потребує додавання `cwd` параметра до ~10–20 production `check()`/`fix()` функцій; це breaking change у публічному API `test-helpers.mjs` → SemVer мінор (1.28.0).
