## ADR Concern-based layout і `target.json` для правил `@nitra/cursor`

**Контекст:** `npm/rules/<id>/js/check.mjs` ріс у монолітний файл (наприклад `abie/check.mjs` — 1153 рядки), а прив'язка rego-полісі до цільових файлів жила або всередині JS, або в centralized `lint-conftest.mjs:TARGETS`-таблиці. Обидва підходи ускладнювали додавання нових правил і порушували cohesion (один concern — один каталог, але код розкиданий у трьох місцях).

**Рішення/Процедура/Факт:** Прийнята нова concern-based структура правила:
```
npm/rules/<id>/
├── <id>.mdc
├── auto.md
├── js/
│   └── <concern>/
│       ├── check.mjs / check-<name>.mjs   ← CLI запускає
│       └── *.test.mjs                     ← пропускається
├── utils/                                 ← shared для <id>, не запускається
└── policy/
└── <concern>/
├── <concern>.rego
├── <concern>_test.rego
└── target.json                    ← маніфест таргета (files.single / files.walkGlob)
```
Симетрія `js/<name>/` ↔ `policy/<name>/`: однойменний concern є або в JS, або в policy, або в обох (hybrid). Пакет-рівень `npm/scripts/utils/` лишається для cross-rule shared утиліт. Rule-level gate виноситься в `js/applies/check.mjs` з named export `applies()`. Glob-матчинг — picomatch 4.x. Walk-кеш — `Map` на scope одного `runChecks`-прогону (не global singleton). Визначені нові утиліти CLI: `discover-checkable-rules.mjs`, `resolve-target-files.mjs`, `run-rule.mjs` — всі покриті тестами (9+9+5 pass). CLI підтримує legacy-fallback `js/check.mjs` під час інкрементальної міграції. Таргет-маніфест: `{ files: { single|walkGlob, required? }, missingMessage?, conftest? }` — схема в `npm/schemas/target.json`.

**Обґрунтування:** Нова структура усуває три джерела інформації для одного правила (JS, TARGETS-таблиця, rego-каталог) і замінює їх одним каталогом `rules/<id>/`. pure-rego правила не потребують JS-обгортки поза applies-гейтом. Picomatch обраний за відсутність транзитивних залежностей і вищу швидкість проти `minimatch`. `applies()` інлайниться у JS замість окремого `rule.json`, бо всі use-cases conditional-гейтингу вже можна виразити JS-кодом (`projectHasRegoFiles`, `isAbieRuleEnabled`), а `rule.json` з примітивами `requireRule`/`requireFilesByGlob` створив би зайвий DSL.

**Розглянуті альтернативи:** (1) `rule.json` з примітивами `applies` на рівні правила — відхилено на користь JS-inlining; (2) centralized `lint-conftest.mjs:TARGETS` (поточна схема) — залишається на час міграції, потім видаляється; (3) `conftest test .` (feed-everything) без таргет-маніфесту — відхилено, бо не підтримує перевірки конкретних файлів без `kind`-гейту й спричиняє N повних обходів дерева; (4) name-convention (ім'я каталогу → шлях файлу) — відхилено через конфлікт `single` vs `walkGlob` для однойменних полісі різних правил.

**Зачіпає:** `npm/rules/*/js/`, `npm/rules/*/policy/`, `npm/bin/n-cursor.js` (функції `discoverCheckScripts`, `runChecks`), `npm/scripts/lint-conftest.mjs` (TARGETS — буде видалено після міграції), нові файли `npm/scripts/utils/{discover-checkable-rules,resolve-target-files,run-rule}.mjs`, `npm/schemas/target.json`, залежність `picomatch` у `npm/package.json`.
