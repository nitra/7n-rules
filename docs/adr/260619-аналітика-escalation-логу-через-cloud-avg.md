# Аналітика escalation-логу через cloud-avg модель

**Status:** Accepted
**Date:** 2026-06-19

## Context and Problem Statement

Драбина ескалації fix-конформності пише per-рунговий escalation-лог (`escalation-log.mjs`), де видно, які правила потребували LLM, скільки рунгів і чи допомогло. Сирий лог сам по собі не закриває петлю покращення: треба регулярно дивитися, де LLM витрачається даремно, і перетворювати це на конкретні правки пакета `@nitra/cursor` — щоб більше роботи робили детерміновані скрипти або щоб локальна модель влучала з першого рунга.

## Considered Options

* Ручний перегляд логу людиною.
* Скрипт-аналітик, що читає лог, чанкує і просить хмарну avg-модель сформулювати рекомендації; виклик наприкінці `lint --full`.
* Аналіз кожного рунга в реальному часі (inline у драбині).

## Decision Outcome

Chosen option: "Скрипт-аналітик (`analyze-escalation.mjs`)", because він автоматизує петлю покращення без втручання в гарячий шлях драбини. `readEscalationRecords` читає лог від байтового зсуву (scope «цей прогін» — зсув фіксується до конформність-фази; межа рядка зберігає мультибайт), `chunkRecords` ділить стиснені записи за бюджетом символів, `analyzeEscalations` робить map-reduce: виклик avg-моделі (`CLOUD_AVG`) по чанках + фінальний синтез за кількох чанків. Мета-prompt просить пріоритезовані рекомендації: (A) новий детермінований T0-патерн для `t0.mjs`, (B) уточнення `.mdc`-інструкцій правила, (C) зміна скрипта. Звіт дописується в `.n-cursor/fix-escalation-analysis.md`. Точки входу: CLI `n-cursor analyze-escalation` (весь лог) і хук наприкінці `lint --full` (non-read-only) — gated kill-switch `N_CURSOR_FIX_ANALYZE`, наявністю `CLOUD_AVG` і записів цього прогону. Помилки моделі ковтаються — аналіз не впливає на exit-код lint.

### Consequences

* Good, because рекомендації формуються автоматично з реальних даних логу, замикаючи петлю «де LLM марнується → що змінити в пакеті».
* Good, because inline-аналіз драбини відкинуто — він би сповільнював гарячий шлях і дублював виклики; пост-аналіз дешевший і бачить повну картину прогону.
* Bad, because кожен `lint --full` із LLM-ескалаціями робить додатковий cloud-avg-виклик (вартість) — пом'якшено гейтами (kill-switch, наявність cloud-avg, наявність записів).
* Bad, because якість рекомендацій залежить від моделі; звіт — це підказки для людини/агента, не авто-патч.

## More Information

Реалізація: `npm/scripts/lib/fix/analyze-escalation.mjs`; хук — `npm/rules/lint/js/orchestrate.mjs` (`runLint`, гілка `--full`); CLI — `npm/bin/n-cursor.js` (`analyze-escalation`). Залежить від `escalation-log.mjs` (джерело даних) і `lib/models.mjs` (`CLOUD_AVG`). Звіт: `.n-cursor/fix-escalation-analysis.md`.
