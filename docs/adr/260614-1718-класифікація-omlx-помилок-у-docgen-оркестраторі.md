## ADR Класифікація omlx-помилок у docgen-оркестраторі

## Context and Problem Statement
Оркестратор `docgen-files-batch.mjs` обробляв усі збої виклику omlx однаково — `catch → ✗ → push → continue` — без розрізнення причини. Це призводило до каскадного списання ~200 файлів підряд після одного systemic-збою (memory-guard), хоча їхній зміст взагалі не намагалися генерувати.

## Considered Options
* Залишити поточну поведінку (єдиний `catch`, без класифікації)
* Триступенева класифікація: `transient | systemic | permanent` з різними стратегіями реакції

## Decision Outcome
Chosen option: "Триступенева класифікація: `transient | systemic | permanent`", because кожен клас вимагає іншої дії, і це видно з логу: `ETIMEDOUT` потрібен ретрай із backoff, memory-guard — circuit breaker із cooldown, `Prompt too long` — детерміновано-незворотний skip.

### Consequences
* Good, because функція-класифікатор `classifyOmlxError` вже частково живе в `omlxHealthCheck` (`lib/llm.mjs:137-150`) — розширення природне.
* Good, because transient-гілка отримає ретрай із паузою замість миттєвого `break` (`omlx.mjs:144-147`), a `ETIMEDOUT` потрапить у `transient` замість `break`.
* Good, because systemic (streak ≥ K) зупиняє батч із повідомленням і не витрачає curl-раунди на решту файлів.
* Good, because permanent (`Prompt too long`, vendored-файли) потрапить у окремий `skipped[]` і не вважатиметься помилкою для повторного прогону.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/rules/doc-files/js/docgen-files-batch.mjs` (цикл рядки ~188-191, preflight рядки 66-80), `npm/lib/omlx.mjs` (ретрай-логіка рядки 144-151).
Конкретні помилки в логу: `spawnSync curl ETIMEDOUT` (transient), `Cannot load … memory ceiling` (systemic), `Prompt too long: 9177917 tokens` (permanent, файл `run/auth/src/lib/lib/euscp.js`).

## ADR Unload конкурентних моделей у preflight замість скидання кешу

## Context and Problem Statement
Починаючи з файлу [58/266], усі виклики omlx повертали `memory ceiling exceeded` через те, що в пам'яті одночасно перебували дві gemma-моделі: резидентна `gemma-4-e4b-it-OptiQ-4bit` (~7.86 GB) і потрібна батчу `gemma-4-e2b-it-4bit` (~3.76 GB) — разом більше стелі 12.7 GB. Одразу постало питання: чи вирішить це скидання KV-/prompt-кешу між файлами через `/admin/api/hot-cache/clear`.

## Considered Options
* Скидати KV-/prompt-кеш між файлами (`POST /admin/api/hot-cache/clear`, `POST /admin/api/ssd-cache/clear`)
* Вивантажувати конкурентну модель через `POST /v1/models/{id}/unload` у preflight / recovery

## Decision Outcome
Chosen option: "Вивантажувати конкурентну модель через `/v1/models/{id}/unload`", because per-request KV-cache omlx звільняє автоматично після кожного completion; prefix/hot-cache навпаки прискорює батч (спільний system-prompt); персистентним споживачем є ваги моделі, а не кеш запитів.

### Consequences
* Good, because `GET /v1/models/status` + `POST /v1/models/{id}/unload` доступні у тій самій `resolveOmlxApiKey`-авторизації і вже бачені в `/openapi.json` сервера.
* Good, because preflight отримує детермінований стан: одна потрібна модель у RAM, без конкурентів.
* Good, because hot-cache (prefix-кеш) зберігається → повторний прогін з тим самим system-promptом docgen не втрачає прогріву.
* Good, because transcript фіксує очікувану користь: user вже видалив зайву модель вручну і каскад припинився — `unload` автоматизує той самий крок.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Ендпоінти: `POST /v1/models/{id}/unload`, `POST /v1/models/{id}/load`, `GET /v1/models/status` (всі наявні в `/openapi.json` локального omlx-сервера `http://127.0.0.1:8000`).
Preflight існує в `docgen-files-batch.mjs:66-80` (`omlxHealthCheck`), відповідний транспорт — `npm/lib/omlx.mjs`.
Розбіжність моделей: `N_LOCAL_MIN_MODEL=omlx/gemma-4-e4b-it-OptiQ-4bit` (`~/.zshenv`), тоді як `DEFAULT_OMLX_MODEL` у `npm/lib/omlx.mjs` вказував на `gemma-4-e2b-it-4bit` — саме вона спричинила конкуренцію.
