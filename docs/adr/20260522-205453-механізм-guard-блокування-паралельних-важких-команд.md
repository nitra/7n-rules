## ADR Механізм guard-блокування паралельних важких команд

## Context and Problem Statement

Кілька AI-агентів в одному проекті можуть паралельно запускати скіли (`n-lint`, `n-fix`, `n-check`, `n-start-check`), що спричиняє конкуренцію за ресурси CPU/диску та неефективні дублюючі запуски. macOS не має `flock`, тому потрібен альтернативний механізм взаємного виключення.

## Considered Options

* **A. Bun-native атомарний лок через `mkdir`** (рекомендовано)
* **B. `shlock` (BSD, `/usr/bin/shlock`)**
* **C. npm-пакет `proper-lockfile`**

## Decision Outcome

Chosen option: "A. Bun-native атомарний лок через `mkdir`", because `fs.mkdirSync()` є атомарним на APFS і не потребує зовнішніх залежностей; дає повний контроль над JSON-дедупліканцією в ~150 рядках bun-коду без `spawn` стороннього бінарника.

### Consequences

* Good, because transcript фіксує очікувану користь: жодних зовнішніх залежностей, дедуп природно лягає в той самий код, що й лок.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Стан зберігається в gitignored `.n-cursor/guard/`: `<key>.lock/` (dir-lock з `owner.json`) + `<key>.result.json`
- Нова підкоманда: `n-cursor guard <key> -- <команда...>` (напр. `n-cursor guard lint -- bun run lint`)
- Реалізація: `npm/scripts/guard.mjs` + `guard.test.mjs`, підключена в `npm/scripts/cli-entry.mjs`
- Файли скілів оновлюються: `npm/skills/{lint,fix,start-check}/SKILL.md`

## ADR Гранулярність та семантика черги guard

## Context and Problem Statement

Потрібно визначити, чи блокувати всі важкі команди єдиним локом, або per-команда, і що робити агенту, який прийшов другим — відмовити, чекати, або перевикористати результат.

## Considered Options

* Один глобальний лок (будь-яка команда блокує всі інші)
* Лок на команду (однаковий `key` = взаємне виключення; `lint` і `check` можуть йти паралельно)
* Завершення агента Б з помилкою (fail-fast)
* Чекати в черзі (wait-and-retry)
* Чекати + дедуплікувати (wait, потім skip якщо результат ще чинний)

## Decision Outcome

Chosen option: "Лок на команду + чекати + дедуплікувати", because користувач явно вибрав: `lint` і `check` ізольовані одне від одного, а агент Б не відкидається, а чекає і потім перевикористовує результат якщо він ще чинний.

### Consequences

* Good, because transcript фіксує очікувану користь: незалежні команди можуть виконуватись паралельно; агент Б не губить результат без причини.
* Bad, because полінг (~1–2 с) до 15 хв тримає агент Б у стані очікування, що може бути непомітним.

## More Information

- Дедуп активується лише на успішний результат (`exitCode === 0`); невдалий прогон завжди запускається заново
- Таймаут очікування — ~15 хв; виявлення застарілого локу — через `process.kill(pid, 0)` + вік > 30 хв

## ADR Fingerprint робочого дерева для дедуплікації

## Context and Problem Statement

Дедуплікація потребує способу визначити, чи змінився стан репозиторію з моменту останнього успішного прогону команди, щоб вирішити — пропустити повторний запуск чи виконати заново.

## Considered Options

* Хеш робочого дерева (`git rev-parse HEAD` + `git diff HEAD` + untracked-файли)
* Хеш + TTL (хеш збігається І час з останнього прогону < порогу)
* Лише TTL (фіксований таймаут без перевірки стану файлів)

## Decision Outcome

Chosen option: "Хеш + TTL", because користувач явно вибрав цей варіант: TTL = 10 хв, fingerprint = sha від `git rev-parse HEAD` + `git diff HEAD` + список і вміст untracked-файлів.

### Consequences

* Good, because transcript фіксує очікувану користь: якщо файли змінились, дедуп не спрацьовує навіть у межах TTL — агент отримує свіжий результат.
* Bad, because поза git-репозиторієм дедуп вимикається, лишається лише взаємне виключення (прийнятний fallback згідно з дизайном).

## More Information

- TTL зберігається у `<key>.result.json` → поле `finishedAt`
- Fingerprint вираховується після захоплення локу, до запуску команди
- Поза git → дедуп вимкнено, лок продовжує працювати
