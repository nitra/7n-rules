---

**Status:** Accepted
**Date:** 2026-06-15

## Context and Problem Statement

У `npx @nitra/cursor lint --full` conformance-фаза визначала правила через discovery файлів `.cursor/rules/*.mdc` на диску, а не через `.n-cursor.json`. Через це правило, увімкнене в конфізі, але без свіжого `sync` і відповідного `.mdc`, тихо не потрапляло в перевірку.

Проблема полягала в тому, що джерело правди для селекції було неявним і залежало від стану файлової системи.

## Considered Options

- Config-first: `resolveCheckRuleIds` бере `available ∩ isRuleEnabled(cfg)`, а `.cursor/rules/*.mdc` використовується лише як fallback за відсутнього конфіга.
- Дволанцюговий гейт: discovery по `.mdc` на першому шарі селекції та `isRuleEnabled` всередині per-rule `runRuleCli`.
- Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Config-first", because `.n-cursor.json` уже є авторитетним конфігом через `readNCursorConfigLite` і `isRuleEnabled`, а дволанцюговий гейт створював silent-failure клас помилок, коли enabled-правило без `.mdc` зникало з conformance-перевірки.

### Consequences

- Good, because `resolveCheckRuleIds` повертає `available ∩ enabled(cfg)` незалежно від наявності `.mdc` на диску.
- Good, because прибрано дублювання: `isRuleEnabled`-гейт у `runRuleCli` видалено, а автоматичні шляхи `lint`, orchestrator, T0 і hook отримують уже відфільтрований список.
- Bad, because за відсутнього конфіга fallback лишається на `.mdc`-discovery, тому поведінка режимів із конфігом і без конфіга відрізняється.
- Neutral, because transcript не містить підтвердження інших наслідків.
