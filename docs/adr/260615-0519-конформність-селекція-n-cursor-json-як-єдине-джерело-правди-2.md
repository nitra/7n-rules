**Status:** Accepted
**Date:** 2026-06-15

## Context and Problem Statement

У `npx @nitra/cursor lint --full` conformance-фаза визначала список правил через discovery файлів `.cursor/rules/*.mdc` на диску. Через це правило, увімкнене в `.n-cursor.json`, але без актуального `.mdc` після sync, тихо пропускалося. Селекція залежала від стану файлової системи, а не від конфіга.

## Considered Options

- Config-first: `resolveCheckRuleIds` бере `available ∩ isRuleEnabled(cfg)`, а `.cursor/rules/*.mdc` лишається fallback за відсутнього конфіга.
- Дволанцюговий гейт: discovery по `.mdc` на першому шарі та `isRuleEnabled` у per-rule `runRuleCli`.
- Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Config-first", because `.n-cursor.json` уже є авторитетом через `readNCursorConfigLite`/`isRuleEnabled`, а попередній дволанцюговий гейт створював silent-failure: enabled-правило без `.mdc` зникало з conformance-перевірки.

### Consequences

- Good, because enabled-правило без `.mdc` більше не пропускається тихо: `resolveCheckRuleIds` повертає `available ∩ enabled(cfg)` незалежно від диска.
- Good, because прибрано дублювання `isRuleEnabled` у `runRuleCli`: автоматичні шляхи отримують уже відфільтрований список.
- Bad, because за відсутнього конфіга fallback лишається на `.mdc` discovery, тому поведінка config-mode і debug/open-by-default mode розходиться.
- Neutral, because transcript фіксує експорт `resolveCheckRuleIds` для тестування.
