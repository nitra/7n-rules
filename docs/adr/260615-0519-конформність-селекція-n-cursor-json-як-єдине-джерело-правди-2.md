**Status:** Accepted
**Date:** 2026-06-15

## Context and Problem Statement

У `npx @nitra/cursor lint --full` конформність-фаза визначала список правил через discovery `.cursor/rules/*.mdc` на диску. Це створювало silent-failure: правило, увімкнене в `.n-cursor.json`, але без свіжого `sync` і відповідного `.mdc`, тихо не потрапляло в конформність-перевірку.

## Considered Options

- Config-first: `resolveCheckRuleIds` бере `available ∩ isRuleEnabled(cfg)`, а `.cursor/rules/*.mdc` лишається fallback за відсутнього конфіга.
- Дволанцюговий гейт: discovery по `.mdc` для селекції й додатковий `isRuleEnabled` всередині per-rule `runRuleCli`.

## Decision Outcome

Chosen option: "Config-first", because `.n-cursor.json` уже є авторитетом через `readNCursorConfigLite`/`isRuleEnabled`, а discovery по `.mdc` створював клас помилок `enabled-but-no-.mdc → silently skipped`.

### Consequences

- Good, because enabled-правило без `.mdc` більше не зникає з конформності: `resolveCheckRuleIds` повертає `available ∩ enabled(cfg)` незалежно від стану `.cursor/rules`.
- Good, because прибрано дублювання: `isRuleEnabled`-гейт у `runRuleCli` видалено, а всі автоматичні шляхи отримують уже відфільтрований список.
- Bad, because за відсутнього конфіга fallback лишається на `.mdc` discovery, тож два режими мають різну основу селекції.
- Neutral, because transcript не містить підтвердження інших наслідків.
