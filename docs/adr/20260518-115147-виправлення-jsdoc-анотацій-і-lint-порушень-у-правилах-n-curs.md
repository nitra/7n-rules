## ADR Виправлення JSDoc-анотацій і lint-порушень у правилах n-cursor

## Context and Problem Statement
Після додавання нових правил (зокрема `network_policy`) і рефакторингу утиліт (`template.mjs`, `inline-template-links.mjs`, `package-manifest.mjs`) повний запуск `bun run lint` завершувався з ненульовим кодом через численні ESLint-порушення: `jsdoc/require-param`, `jsdoc/require-param-type`, `jsdoc/require-param-description`, `jsdoc/require-returns`, `no-empty-function`, `e18e/prefer-static-regex`, `sonarjs/slow-regex`, `unicorn/consistent-function-scoping`, а також cspell-помилки, jscpd-клони і v8r-попередження про відсутні схеми.

## Considered Options
* Виправити порушення вручну — файл за файлом
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Виправити порушення вручну — файл за файлом", because lint-пайплайн у проєкті є блокуючим (`exitCode: 1` у `.jscpd.json`, `bun run lint` завершується з кодом 1), і кожне порушення потребувало конкретного точкового виправлення.

### Consequences
* Good, because `bun run lint` завершується з EXIT=0 після усіх правок.
* Good, because transcript фіксує очікувану користь: regex-літерали винесено на рівень модуля (`YAML_LS_MODELINE_RE`, `LEADING_DOTSLASH_RE`, `MD_LINK_RE`), що відповідає правилам `e18e/prefer-static-regex` і `sonarjs/slow-regex`; JSDoc-анотації стали повними; `() => {}` замінено на `() => undefined` для обходу `no-empty-function`.
* Bad, because transcript не містить підтверджених негативних наслідків.
