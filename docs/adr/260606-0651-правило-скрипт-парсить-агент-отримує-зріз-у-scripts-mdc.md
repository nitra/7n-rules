## ADR: Правило «Скрипт парсить — агент отримує зріз» у `scripts.mdc`

## Context and Problem Statement
Декілька скілів у `npm/skills/` (зокрема `coverage-fix`, `taze`, `start-check`, `fix`) інструктували LLM-агента виконувати детермінований парсинг великих артефактів (`COVERAGE.md` ~2.76 МБ, `package.json.taze-bak`, сирий термінальний вивід `fix`). Це витрачало сотні тисяч токенів без будь-якої когнітивної роботи LLM і унеможливлювало роботу скілів при перевищенні контекстного вікна.

## Considered Options
* Залишити поточний підхід (агент читає повні артефакти)
* Детермінований парсинг — у read-only CLI-екстрактор; агент отримує компактний `index` + `slice`/`run` під один елемент

## Decision Outcome
Chosen option: "read-only CLI-екстрактор з `index`/`slice`/`run`", because детермінований парсинг/класифікація/лічба — 0 LLM-токенів у скрипті; агент отримує лише порцію під справді когнітивну роботу (CHANGELOG, рефакторинг, діагностика причини FAIL). Правило закріплено як секція «🔴 ВИСОКИЙ ПРІОРИТЕТ» у `.cursor/rules/scripts.mdc` v1.13.

### Consequences
* Good, because transcript фіксує очікувану користь: `coverage-fix index` = 7.4 КБ (~2K токенів) замість 2.76 МБ (~700K токенів) — ≈350× менше для оркестратора; `taze diff` та `start-check scan/run` усувають LLM-ручне розгортання glob-воркспейсів і semver-порівняння.
* Bad, because нові команди (`coverage-fix`, `taze`, `start-check`, `fix --json`) активуються лише після публікації нової версії `@nitra/cursor`: до релізу є version-skew між локальним `npm/` (3.24.0) і встановленою копією (3.20.0).
