* Інші варіанти в transcript не обговорювалися.

## ADR flow release: інференс зміненого воркспейсу для `--ws`

## Context and Problem Statement
`flow release` завжди передавав аргументи `change` без `--ws`, тому change-файл потрапляв у дефолтну директорію `.` замість реального subworkspace. У монорепо це призводить до неправильної атрибуції змін.

## Considered Options
* Інференс `--ws` зі змін від `base_commit` через `collectChangedFilesSince` + `getMonorepoProjectRootDirs`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Інференс `--ws` зі змін від `base_commit`", because це дозволяє автоматично визначити один subworkspace й передати `--ws <ws>` у `n-cursor change`; якщо змінено нуль або більше одного subworkspace — fail (чіткий сигнал для ручного втручання) або skip-інференс відповідно; ін'єктовні залежності (`listWorkspaces`, `changedFilesSince`) дозволяють тестувати без реального git.

### Consequences
* Good, because `flow review` не зможе отримати два конфліктні `--ws`; кейс «кілька subworkspaces» → exit 1 із зрозумілим повідомленням.
* Bad, because inline-форма `--ws=value` (де `=` замість пробілу) не ловилась першою перевіркою `rest.includes('--ws')` — виявлено під час `flow review` та виправлено в тій самій сесії.

## More Information
Змінені файли: `npm/scripts/dispatcher/lib/commands.mjs` (нова `matchChangedWorkspaces` + `resolveChangeWsArgs`), `npm/scripts/dispatcher/lib/tests/commands.test.mjs`. Тести: 27 passed (commands), 211 passed (full dispatcher). Гілка `flow-release-infer-ws` (`282332c`) змерджена у `main` разом із `flow-cwd-state-resolution`. ## ADR flow review: верифікація cross-file findings через `Read`

## Context and Problem Statement
Рецензентський subagent у `flow review` отримував у промпті лише diff і інструкцію «знайди баги». Без явної вказівки на read-доступ він не звертався до referenced-файлів/spec для перевірки cross-file тверджень, видаючи нефальсифіковні findings «з diff не видно».

## Considered Options
* Додати до промпту явну вказівку: перед репортом перевіряти cross-file твердження читанням файлів через `Read`; обмежити scope — репортувати лише баги, *введені* цим diff (не pre-existing у сусідніх файлах)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Явна вказівка у промпті + обмеження scope", because subagent вже мав `allowedTools: ['Read', 'Edit', 'Bash']` і `cwd` = worktree; єдине, чого бракувало — інструкція скористатись ними перед репортом; звуження scope знижує шум від pre-existing помилок.

### Consequences
* Good, because transcript фіксує очікувану користь: менше хибно-позитивних findings «з diff не видно»; рецензент верифікує твердження перед публікацією.
* Bad, because промпт декларує наявність `Read`, але не залежить від реальної конфігурації runner-а — якщо `allowedTools` змінять, промпт буде неточним. Виявлено під час `flow review` самої цієї зміни.

## More Information
Змінений файл: `npm/scripts/dispatcher/lib/review.mjs` (`reviewerPrompt`). Тести: 14 passed, oxlint rc=0. Гілка `flow-review-read-access` (`b0308d4`) змерджена у `main`.

## ADR coverage-gate: запуск Stryker через локально встановлений core, а не `npx`

## Context and Problem Statement
`flow verify` постійно падав на coverage-gate через `StrykerError: Cannot find TestRunner plugin "vitest"`. `runStryker` у `npm/rules/js-lint/coverage/coverage.mjs` запускав core через `npx @stryker-mutator/core`, що завантажувало core у власний кеш npx без сусіднього `@stryker-mutator/vitest-runner`.

## Considered Options
* Резолвити локально встановлений `@stryker-mutator/core` через `package.json` → поле `bin`, запускати `bin/stryker.js` напряму (через його `#!/usr/bin/env node` shebang)
* Перша спроба: `createRequire(import.meta.url).resolve('@stryker-mutator/core/bin/stryker.js')` — відхилено, бо `exports` пакета не відкриває цей subpath (`ERR_PACKAGE_PATH_NOT_EXPORTED`)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Резолв через `package.json` + пряме виконання `bin/stryker.js`", because `exports` пакета не відкриває `./bin/stryker.js`, тому резолвити треба через `package.json` → `bin`; пряме виконання bin-скрипта (з `#!/usr/bin/env node`) гарантує node-рантайм незалежно від батьківського процесу (bun чи node).

### Consequences
* Good, because transcript фіксує очікувану користь: `flow verify` coverage-gate пройшов e2e (vitest test runner завантажився, 34 тести, exit 0).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Змінений файл: `npm/rules/js-lint/coverage/coverage.mjs` (нова `resolveLocalStrykerBin`, `runStryker` спавнить bin напряму без `process.execPath`). Тести: 34 passed, oxlint rc=0. Гілка `flow-coverage-stryker-local` (`373ce42`) змерджена у `main`. Залежність: `@stryker-mutator/core` і `@stryker-mutator/vitest-runner` мають бути сусідами у `node_modules/@stryker-mutator/`.
