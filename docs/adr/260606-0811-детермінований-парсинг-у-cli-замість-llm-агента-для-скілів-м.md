## ADR Детермінований парсинг у CLI замість LLM-агента для скілів монорепо

## Context and Problem Statement

У проєкті `npm/skills/` накопичились скіли (`n-fix`, `n-taze`, `n-start-check`, `n-coverage-fix`), у яких SKILL.md зобов'язував LLM-агента вручну парсити великі артефакти, порівнювати бекапи, розгортати glob-шаблони, лічити рядки та класифікувати semver-діапазони. Це призводило до нерепродукованих результатів, зайвих токенів і помилок класифікації (наприклад, `workspace:1.0.0` неправильно класифікувалось як semver).

## Considered Options

* Залишити важкий парсинг у LLM-агенті (тогочасний підхід)
* Винести детермінований парсинг/класифікацію/лічбу в CLI (`n-cursor <skill> <verb>`), щоб агент отримував компактний зріз

## Decision Outcome

Chosen option: "Винести детермінований парсинг у CLI", because такий підхід дає 0 LLM-токенів на механічну частину, відтворюваний результат і відповідає канону `scripts.mdc`, де читання/сканування/класифікація є програмними (не когнітивними) операціями.

### Consequences

* Good, because transcript фіксує очікувану користь: `n-cursor taze diff` (semver-diff із правильним anchor-regex), `n-cursor start-check scan|run` (розгортання glob, класифікація server/cli, крос-платформний таймаут через `spawnSync`), `n-cursor fix --json` (per-rule результати без забруднення stdout), `n-cursor coverage-fix index|slice` — усі реалізовані, oxlint чисто, 60 тестів зелені.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- `scripts.mdc` оновлено до версії `v1.13`, додано секцію «🔴 ВИСОКИЙ ПРІОРИТЕТ — детермінований парсинг у скіла: у CLI, не в LLM» з Red flags і контрактом `n-cursor <skill> <verb>`.
- Нові модулі: `npm/skills/taze/js/diff.mjs`, `npm/skills/start-check/js/check.mjs`; оновлено `npm/bin/n-cursor.js` (case `taze`, `start-check`; прапорець `--json` для `fix`).
- Change-файли: `npm/.changes/260605-0716.md`, `npm/.changes/260605-0731.md`, `npm/.changes/260606-0636.md`.

## ADR Виявлення баґу: відсутність `permissionMode` у headless SDK-воркері для `coverage --fix`

## Context and Problem Statement

Під час feasibility-перевірок для експерименту `n-fix A/B/C` було виявлено, що `npm/scripts/coverage-fix.mjs` запускає `query()` з `@anthropic-ai/claude-agent-sdk` без `permissionMode`. Без цього параметра headless-агент отримує дозволи за замовчуванням і не виконує `Edit`/`Bash`-виклики — файли не записуються.

## Considered Options

* Залишити без `permissionMode` (тогочасний стан)
* Додати `permissionMode: 'bypassPermissions'` для headless-прогонів `coverage --fix`

## Decision Outcome

Chosen option: "Додати `permissionMode: 'bypassPermissions'`", because два контрольні проби в transcript підтвердили: без режиму — файл не створюється (`(missing)`), з `bypassPermissions` — `WORKER_OK`. Для CI/headless `coverage --fix` потрібен autofile.

### Consequences

* Good, because transcript фіксує очікувану користь: `coverage --fix` тепер реально редагує файли headless; тест оновлено (`11/11`), change-файл `npm/.changes/260606-0721.md`.
* Bad, because `bypassPermissions` вимикає всі prompt-gates — підходить лише для CI/автономного прогону, де оператор свідомо запускає команду. Для інтерактивного режиму був би ризик небажаних змін.

## More Information

- Змінений файл: `npm/scripts/coverage-fix.mjs` (додано `permissionMode: 'bypassPermissions'` в `options` до `query()`).
- Тест: `npm/scripts/tests/coverage-fix.test.mjs`, опис test-кейсу оновлено.
- Probe-скрипти запускались безпосередньо з `npm/` (де доступний `@anthropic-ai/claude-agent-sdk`).

## ADR Hybrid-orchestrator (T0 механіка + holistic LLM) замість per-rule fan-out або монолітного агента для `n-fix`

## Context and Problem Statement

Потрібно було емпірично порівняти три підходи запуску `n-fix` при 6 реальних порушеннях у worktree: **A** (один LLM-агент зі SKILL.md), **B** (per-rule fan-out: окремий воркер на кожне правило), **C** (hybrid: детермінований autofix + holistic LLM на залишок). Три прогони проводились в ізольованому git-worktree `.worktrees/main-fixexp` від HEAD з ідентичним baseline.

## Considered Options

* A — один LLM-агент зі SKILL.md (28 turns)
* B — наївний per-rule fan-out (8 воркерів)
* C — hybrid: детермінований autofix (T0) + один holistic LLM-воркер на залишок

## Decision Outcome

Chosen option: "C — hybrid orchestrator", because C набрав найкращі показники за output-токенами (3.1K vs 5.0K у A та 9.6K у B), рівну з A швидкість (105 с проти 210 с у B), чистий мінімальний diff (без scope-creep, який мав A), та 0 токенів на механічні фікси (видалення lockfiles). Ключовий висновок transcript: **для задач зі спільними цільовими файлами per-rule fan-out — антипатерн** (5 воркерів B правили один `.vscode/extensions.json` 5 разів); holistic LLM тримає файл в одній операції.

### Consequences

* Good, because transcript фіксує очікувану користь: T0-механіка (видалення заборонених файлів) без LLM; 1 holistic LLM-воркер для decision-задач; відсутність scope-creep; `failed 0` так само, як A і B.
* Bad, because `n=1` (LLM недетермінований), тому числа індикативні. Transcript зафіксував потенційний наступний крок: якби `fix` ще й генерував конфіги з template/rego (T0), то навіть один LLM-воркер зник би для більшості сценаріїв.

## More Information

- Інструменти: `_exp/lib.mjs`, `_exp/run-a.mjs`, `_exp/run-b.mjs`, `_exp/run-c.mjs` у worktree (прибрані після прогону).
- Baseline: 6 реальних порушень (`bun, ga, js-lint, rego, text, vue`) — наслідок merge `origin/main`; `.vscode/extensions.json` відсутній, закоммічені lockfiles.
- Harness: `fix --json` → механічний autofix (JS) → `query()` з `bypassPermissions`/`acceptEdits`, результат `fix --json` post-run.
- Наступна запланована стадія: T1 (локальні моделі: `pi`+`gemma3:4b`) + router з порогом ≤10% втрати якості + md→codegen для специфікацій оркестраторів.
