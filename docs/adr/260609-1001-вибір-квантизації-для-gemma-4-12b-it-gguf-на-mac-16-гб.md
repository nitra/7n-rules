## ADR Вибір квантизації для gemma-4-12b-it-GGUF на Mac 16 ГБ

## Context and Problem Statement
Треба запустити модель `unsloth/gemma-4-12b-it-GGUF` (11.95B params) на Mac з 16 ГБ unified memory без свопу. Потрібно вибрати квантизацію, яка вміститься у вільні ~9–10 ГБ (після системи та застосунків) разом із KV-кешем і обчислювальними буферами.

## Considered Options
* Q4_K_M (7.12 ГБ) — стандартна llama.cpp k-quant схема
* UD-Q4_K_XL (7.37 ГБ) — Unsloth Dynamic 2.0 з нерівномірним розподілом бітів
* Q5_K_M (8.41 ГБ) — більш точна, але впритул до ліміту
* Q6_K / Q8_0 — не розглядались серйозно через перевищення бюджету

## Decision Outcome
Chosen option: "UD-Q4_K_XL", because за майже однаковий розмір (лише +0.25 ГБ відносно Q4_K_M) Unsloth Dynamic дає кращу якість завдяки нерівномірному per-layer розподілу бітів, що підтверджено завантаженням через `ollama pull hf.co/unsloth/gemma-4-12b-it-GGUF:UD-Q4_K_XL`.

### Consequences
* Good, because 7.37 ГБ залишає ~2–3 ГБ під KV-кеш без свопу на 16 ГБ.
* Bad, because потрібен свіжий llama.cpp/Ollama (0.24.0+) для підтримки dynamic IQ-типів; на старих версіях рантайму можлива несумісність.

## More Information
- Модель завантажена: `hf.co/unsloth/gemma-4-12b-it-GGUF:UD-Q4_K_XL` (7.5 ГБ, ID `1b84f575dd0b`)
- Ollama version: `0.24.0`
- Рекомендований контекст: `/set parameter num_ctx 8192`
- Опційно: `OLLAMA_KV_CACHE_TYPE=q8_0` для зменшення KV-кеша
- Команда запуску: `ollama run hf.co/unsloth/gemma-4-12b-it-GGUF:UD-Q4_K_XL`
