## ADR Concern-based layout та `target.json` для правил CLI

**Контекст:** Поточна структура `npm/rules/<id>/` змішує всю логіку в одному `js/check.mjs` (до 1153 рядків у `abie`) і таргетинг Rego-полісі зберігається централізовано у `lint-conftest.mjs:TARGETS`-таблиці (рядки 96–373), а не поряд із самими полісі. CLI `discoverCheckScripts()` не бачить правила без `js/check.mjs`, тому pure-Rego правила неможливі.

**Рішення/Процедура/Факт:**
Нова структура каталогу:
```
npm/rules/<id>/
├── <id>.mdc
├── auto.md
├── js/
│   └── <concern>/
│       ├── check.mjs / check-<variant>.mjs  ← запускає CLI
│       └── *.test.mjs                        ← пропускається
├── utils/                                    ← shared у межах правила
└── policy/
└── <concern>/
├── <concern>.rego
├── <concern>_test.rego
└── target.json                       ← маніфест таргета
```
`target.json` має дві форми: `{ "files": { "single": "pkg.json" } }` або `{ "files": { "walkGlob": "**/*.yaml" } }`. CLI читає `target.json`, резолвить список файлів через `picomatch` + `walkDir` з кешем (`Map` у scope `runChecks`), і викликає `runConftestBatch` без участі JS. JS concern-и відповідають тільки за обчислювані/динамічні таргети та cross-file логіку. Namespace автоматичний: `<id>.<concern>`. Гейтинг (`applies`) лишається в `js/<concern>/check.mjs` як named export `applies()`. Discovery підтримує legacy `js/check.mjs` паралельно для інкрементальної міграції.

Нові файли, вже створені:
- `npm/schemas/target.json` — JSON Schema для `target.json`
- `npm/scripts/utils/resolve-target-files.mjs` — resolver (single/walkGlob + walk-cache + path-traversal guard)
- `npm/scripts/utils/resolve-target-files.test.mjs` — 9 тестів (всі зелені)
- `npm/scripts/utils/discover-checkable-rules.mjs` — новий discovery з підтримкою обох структур
- `npm/scripts/utils/discover-checkable-rules.test.mjs` — тести discovery
- `docs/superpowers/specs/2026-05-15-npm-rules-concern-and-target-design.md` — plan документ

**Обґрунтування:** Один `check.mjs` на 1000+ рядків важко читати і тестувати; TARGETS-таблиця в `lint-conftest.mjs` дублює targeting, що живе поза самим правилом. Нова структура дає: locality (все правило — у своєму каталозі), symmetry (`js/<name>/` ↔ `policy/<name>/`), discovery без конфігурації, підтримку pure-Rego правил без JS-обгортки. `picomatch` обраний як glob-бібліотека — 1 пакет, нуль транзитивних залежностей, ~30KB.

**Розглянуті альтернативи:**
- `rule.json` per-rule з централізованим `targets`-масивом — відхилено: locality гірша, targeting не поряд із `.rego`
- name-convention без маніфесту (ім'я каталогу → файл) — відхилено: конфлікт між `single`/`walk` для одноіменних полісі в різних правилах
- `conftest test .` feed-everything — відхилено: не підтримує перевірки на конкретні безкіндові файли (`.cspell.json`, `.vscode/settings.json`), performance: 30+ повних обходів репо
- `rule.json` з `applies`-полем — відхилено: `applies` вирішено інлайнити в JS через named export `applies()`, щоб не плодити конфіг-файл для кожного правила

**Зачіпає:** `npm/bin/n-cursor.js` (функції `discoverCheckScripts`, `runChecks`), `npm/scripts/lint-conftest.mjs` (таблиця `TARGETS` стане похідною після міграції), `npm/rules/*/js/check.mjs` (усі, інкрементально), `npm/rules/*/policy/` (додаються `target.json`), `npm/package.json` (додана залежність `picomatch@4.0.4`), нові утиліти в `npm/scripts/utils/`
