## ADR Стиль документації docgen: поведінково-орієнтований замість реалізаційного

## Context and Problem Statement
Скіл `n-docgen` генерував `.md`-документацію з надлишковими реалізаційними деталями: розділами `## Залежності` з переліком `node:fs`/`node:path`, точними іменами внутрішніх функцій (`isAbieRuleEnabled`), таблицями типів і сигнатурами. Такий контент швидко старіє разом із кодом і не пояснює бізнес-задачу, яку файл вирішує.

## Considered Options
* Поведінково-орієнтований стиль: секції `## Огляд`, `## Поведінка`, `## Публічний API` (лише за нетривіальної поверхні), `## Де використовується`, `## Гарантії поведінки`; без `## Залежності` і без сигнатурних блоків
* Зберегти наявний реалізаційний стиль («вичерпна» документація з усіма іменами, таблицями, `Rebuild Test`-переліком)

## Decision Outcome
Chosen option: "Поведінково-орієнтований стиль", because користувач явно вказав, що розділи з `node:fs`, `node:fs/promises`, `node:path` і точними іменами символів не впливають на бізнес-задачу і їх не потрібно бачити; мета — зрозуміти файл без читання реалізації (Behavior Test).

### Consequences
* Good, because документація описує інваріанти і гарантії (read-only, fail-safe, opt-in-семантика, нормалізація регістру), які залишаються стабільними навіть при рефакторингу реалізації.
* Bad, because для модулів із нетривіальним публічним API промпт залишає вибір секцій на розсуд субагента — без чіткого критерію «нетривіальності», що може призводити до непослідовного результату між файлами.

## More Information
Змінені файли:
- `npm/skills/docgen/SKILL.md` — джерело (Tier-1 промпт, description)
- `.cursor/skills/n-docgen/SKILL.md` — встановлена копія
- `.pi/skills/n-docgen/SKILL.md` — стаб (description)
