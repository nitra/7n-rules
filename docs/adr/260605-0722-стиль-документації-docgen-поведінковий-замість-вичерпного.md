## ADR Стиль документації docgen — поведінковий замість вичерпного

## Context and Problem Statement
Скіл `n-docgen` генерував документацію, що містила розділи «Залежності» з переліком `node:fs`/`node:path`/`node:fs/promises`, таблиці сигнатур, точні внутрішні імена хелперів і «Rebuild Test»-переліки. Такий стиль не відповідав бізнес-задачі файлу, старів разом із кодом і не допомагав розуміти поведінку без читання реалізації.

## Considered Options
* Поведінковий стиль — секції Огляд / Поведінка / Де використовується / Гарантії поведінки; Публічний API лише для нетривіальних exports; без stdlib, внутрішніх імен і сигнатур
* Вичерпний стиль — повний перелік залежностей, сигнатур, параметрів, Rebuild Test (попередній стан)

## Decision Outcome
Chosen option: "Поведінковий стиль", because користувач відхилив документацію з `node:fs`-деталями та `isAbieRuleEnabled`-сигнатурою як технічний шум, що не впливає на розуміння задачі файлу; еталонний варіант (узгоджений у дискусії) — opt-in-семантика, покроковий алгоритм рішення, fail-safe, read-only — без жодного stdlib-рядка.

### Consequences
* Good, because transcript фіксує очікувану користь: перегенеровані доки (15 файлів `npm/rules/abie`) стали лаконічнішими й читабельними без читання коду.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Промпт Tier 1 змінено в `npm/skills/docgen/SKILL.md` і `.cursor/skills/n-docgen/SKILL.md` (опис «вичерпну» → «лаконічну поведінкову»; правила секцій уточнені: `## Публічний API` — лише за нетривіальної поверхні; `## Гарантії поведінки` замість `## Помилки`; заборона stdlib і внутрішніх імен).
- Еталонні доки: `npm/skills/docgen/bench/etalon/` (3 файли: `firebase_hosting.md`, `overlay-paths.md`, `k8s-tree.md`).
- Перегенеровано `npm/rules/abie/lib/docs/enabled.md` як ручний, а потім автоматичний еталон.

## ADR Локальний Ollama для docgen Tier 1 — gemma3:4b з system-prompt

## Context and Problem Statement
Повний прогін скіла `n-docgen` Tier 1 по цьому проєкту (1442 файли) коштує ~26–36 M хмарних токенів Claude (~$300–770). Розглядалася альтернатива — локальна Ollama на M2 8 GB — для зниження вартості до $0.

## Considered Options
* `gemma3:4b` (3.3 GB, 100% GPU на 8 GB) — модель, що повністю влазить у пам'ять
* `gemma4:e4b` (9.6 GB) — не влазить, постійний своп, 0.4 tok/s
* `qwen2.5-coder:3b` (2.4 GB) — найшвидша (36 tok/s), але якість UA ~77% (мовні помилки, витік stdlib)
* 7–8B моделі (`qwen2.5:7b`, `llama3.1:8b`) — офлоад на CPU, повільніші, без виграшу в якості
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`gemma3:4b` з прямим `POST /api/chat` + `system`-role", because бенчмарк на 3 моделях × 3 файлах показав: `gemma3:4b` дає ~86% якості проти еталона (100% GPU, без свопу); system-prompt — єдина причина різниці між прямим API (74%) і pi-транспортом (86%); pi як посередник для batch-генерації не потрібен.

### Consequences
* Good, because transcript фіксує: `gemma3:4b` встановлена, pi підключений через `~/.pi/agent/models.json`, підтверджений виклик `Так.`, оновлена оцінка ~13.6 год для 1042 файлів (concurrency=1, ~47 с/файл).
* Bad, because на 8 GB M2 Tier 1 послідовний (concurrency=1) — паралелізм батчів зникає; повний прогін займає ~13–14 год (без паузи).

## More Information
- Підтверджені команди: `ollama pull gemma3:4b`, `POST http://localhost:11434/api/chat` з `system`+`user` roles, без явного `num_ctx`.
- Конфіг pi: `~/.pi/agent/models.json` з `"api": "openai"`, `"baseUrl": "http://localhost:11434/v1"`.
- Еталони бенчмарку: `npm/skills/docgen/bench/etalon/` (3 файли).
- Видалені після бенчмарку: `qwen2.5-coder:3b`, `gemma4:e4b` та 5 інших моделей.
- Наступний крок (не реалізований): CLI `docgen gen --engine ollama --model gemma3:4b --concurrency 1`.
