## ADR Заміна окремого бінарника `kustomize` на вбудовану підкоманду `kubectl kustomize`

## Context and Problem Statement

У `npm/rules/k8s/lint/lint.mjs` kubescape-сканування k8s-маніфестів спиралося на окремий виконуваний файл `kustomize` у `PATH`. Користувач вказав, що перевірку треба виконувати через `kubectl kustomize`, оскільки kubectl містить цю підкоманду вбудовано (Kustomize вшито в kubectl).

## Considered Options

* Зберегти окремий бінарник `kustomize` (і далі вимагати його в `PATH` та CI)
* Використовувати вбудовану підкоманду `kubectl kustomize <dir>`

## Decision Outcome

Chosen option: "Використовувати вбудовану підкоманду `kubectl kustomize <dir>`", because kubectl 1.36+ вже містить Kustomize v5.8.1, тому окремий бінарник є зайвою залежністю — підтверджено виводом `kubectl version --client=true` під час сесії.

### Consequences

* Good, because transcript фіксує очікувану користь: прибрано крок «Install kustomize» з CI-сніпета в `k8s.mdc`, тому GitHub Actions–ворклоу стає коротшим і не потребує окремого `curl`-встановлення.
* Good, because версія Kustomize завжди синхронізована з версією kubectl, що усуває потенційний drift між `kustomize build` і `kubectl apply`.
* Bad, because transcript не містить підтверджених негативних наслідків.
