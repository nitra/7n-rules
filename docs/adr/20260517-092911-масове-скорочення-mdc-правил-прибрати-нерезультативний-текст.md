## ADR Масове скорочення `.mdc`-правил: прибрати нерезультативний текст
**Контекст:** Правила в `npm/rules/<id>/*.mdc` накопичили пояснювальний текст (опис інструментів, інструкції встановлення, VS Code блоки, "## Перевірка" хвости, дублі, міграційні нотатки), який збільшував контекстне вікно AI без будь-якого впливу на кінцевий результат застосування правила.
**Рішення/Процедура/Факт:** Почищено 23 з 26 правил у `npm/rules/` — прибрано ~314 рядків (~7,6%): всі "## Перевірка" секції (18 файлів), `.vscode/extensions.json`/`settings.json` блоки (8 файлів), `brew`/`apt`-інструкції (5 файлів), описи зовнішніх інструментів (`hadolint`, `gitleaks`, `kubeconform`), активаційну мету, дублі (наприклад `bun.mdc` — тричі той самий список), міграційний контекст для старих версій. Пропущено `vue.mdc`, `php.mdc`, `text.mdc` — потребують перевірки з check-скриптами або вирішення внутрішніх суперечностей.
**Обґрунтування:** `.mdc` читає тільки AI-агент: він отримує файл у промпт за `globs:`-матчем і на підставі нього генерує/редагує цільові файли (`package.json`, `Dockerfile` тощо). CLI n-cursor (`npx @nitra/cursor check <id>`) дізнається про концерни та скрипти виключно з файлової структури (`fix/<concern>/check*.mjs`, `policy/<concern>/target.json`) — `.mdc` для цього не читається. Тому пояснювальний текст "для людини-оператора" є шумом у AI-промпті.
**Розглянуті альтернативи:** Лишити як є; прибирати лише найбільші файли вибірково; зберегти "## Перевірка" рядок як компактний one-liner — обрали баланс: one-liner не повертали, бо аудиторія .mdc — AI, а не людина.
**Зачіпає:** `npm/rules/rego/rego.mdc`, `npm/rules/adr/adr.mdc`, `npm/rules/style-lint/style-lint.mdc`, `npm/rules/image-compress/image-compress.mdc`, `npm/rules/image-avif/image-avif.mdc`, `npm/rules/nginx-default-tpl/nginx-default-tpl.mdc`, `npm/rules/graphql/graphql.mdc`, `npm/rules/docker/docker.mdc`, `npm/rules/bun/bun.mdc`, `npm/rules/k8s/k8s.mdc`, `npm/rules/js-lint/js-lint.mdc`, `npm/rules/ga/ga.mdc`, `npm/rules/hasura/hasura.mdc`, `npm/rules/capacitor/capacitor.mdc`, `npm/rules/changelog/changelog.mdc`, `npm/rules/npm-module/npm-module.mdc`, `npm/rules/ci4/ci4.mdc`, `npm/rules/js-run/js-run.mdc`, `npm/rules/js-bun-db/js-bun-db.mdc`, `npm/rules/js-bun-redis/js-bun-redis.mdc`, `npm/rules/tauri/tauri.mdc`, `npm/rules/js-mssql/js-mssql.mdc`, `npm/rules/abie/abie.mdc`

## ADR `template/` — єдине джерело канонічних merge-фрагментів у правилах npm/rules
**Контекст:** Канонічні значення (скрипти `package.json`, leaf-поля `.gitleaks.toml`, рядки workflow YAML) дублювалися між `.mdc`-сніпетами, inline-літералами в `check.mjs` та inline-рядками в `*.rego`. Зміна одного канонічного значення вимагала оновлень у трьох місцях.
**Рішення/Процедура/Факт:** Для кожного концерну правила, де перевірка зводиться до merge-assertions, додається `template/` субдиректорія у `policy/<concern>/template/` (canonical) або `fix/<concern>/template/` (FS-only концерни без rego). Три опціональних файли: `snippet.json` (required leaf-eq; для масивів — subset-of), `deny.json` (заборонені шляхи), `contains.json` (substring-перевірки). Всі файли в JSON незалежно від формату цільового файлу (TOML/YAML/JSONC парсяться у JS-об'єкт перед порівнянням). Нова утиліта `template-loader.mjs` завантажує concern template; `template-checks.mjs` — три pure-функції `checkSnippet`/`checkDeny`/`checkContains`; `runConftestBatch` отримує опціональний `templateData`, що передається conftest через `--data`. У `.mdc` inline-сніпети замінюються на markdown-посилання. Spec: `docs/superpowers/specs/2026-05-17-template-dir-design.md`.
**Обґрунтування:** Один канонічний JSON-фрагмент замість трьох копій забезпечує, що зміна значення автоматично впливає на JS-check і Rego-полісі одночасно. Native-синтаксис (JSON залишається JSON, не DSL) знижує поріг входу для AI, який редагує template. Concern-level layout обраний замість rule-level через реальні collision-cases: `k8s` має 4 концерни з `*.yaml` glob і 2 з `kustomization.yaml`; `npm-module` — 2 концерни на `package.json` — у rule-level ці концерни конкурували б за один template-файл.
**Розглянуті альтернативи:** Rule-level `template/` з flatten-іменуванням файлів (відхилено через collision у k8s/npm-module); DSL-файл `check.json` з ключами `required`/`forbidden`/`contains` (попередній spec, відхилено на користь native-syntax суфіксів); Scaffold Сценарій A — повні статичні файли у `template/` (наприклад повний `.gitleaks.toml`) — відхилено, бо для merge-assertions не потрібний повний файл; концерни `k8s.manifests`, `abie.*` (cross-file / kustomize-resolution) не підлягають template (лишаються inline).
**Зачіпає:** `npm/scripts/utils/template-loader.mjs` (новий), `npm/scripts/utils/template-checks.mjs` (новий), `npm/scripts/utils/run-conftest-batch.mjs` (розширення `templateData`), `npm/scripts/utils/run-rule.mjs` (передача template у check), всі `policy/<concern>/*.rego` та `fix/<concern>/check.mjs` для 25 правил у `npm/rules/` за 6-фазовим планом міграції.
