## ADR Конвенція назв npm-скриптів: `lint` (швидкий) та `lint-ci` (повний)

## Context and Problem Statement
Монолітний `bun run lint` (~1–2 хв) блокує агентів між ітераціями. Потрібно розрізати lint на два набори з різними назвами npm-скриптів для локального і CI-використання.

## Considered Options
* `lint-quick` / `lint-all` — початкова назва, що використовувалась у першій версії spec
* `lint` / `lint-ci` — запропоновано користувачем як фінальна конвенція

## Decision Outcome
Chosen option: "`lint` / `lint-ci`", because користувач продиктував цю конвенцію: `lint` — швидкий (по змінених), `lint-ci` — повний (по всіх).

### Consequences
* Good, because `lint` залишає звичний short-name для локального запуску, а суфікс `-ci` явно сигналізує про CI-контекст.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Spec: `docs/superpowers/specs/2026-05-31-lint-quick-ci-split-design.md`. CLI: `n-cursor lint` (quick) / `n-cursor lint --ci` (full). Commit `788f4ba`.

## ADR `meta.json.lint` — атрибут `"quick"|"ci"` з семантикою `quick⊆ci`

## Context and Problem Statement
Для data-driven lint-оркестрації кожне правило має декларувати, чи воно входить до швидкого набору, повного, або обох. Потрібен мінімальний атрибут у `meta.json`.

## Considered Options
* E1: поле `lint: "quick" | "ci"` (enum, `quick⊆ci` — швидкі кроки входять до обох наборів)
* E2: об'єкт `lint: { phase, scope }` — окремо фаза і scope
* E3: булеві прапорці `lintQuick`/`lintCi`

## Decision Outcome
Chosen option: "E1 — поле `lint: "quick"|"ci"`", because просте enum-значення покриває всі випадки; scope виводиться з фази (`quick`→по змінених, `ci`→по всіх); E2 і E3 — YAGNI.

### Consequences
* Good, because transcript фіксує очікувану користь: мінімальна схема, легко валідувати (`rule-meta.json` schema + Ajv), узгоджено з вже існуючим `auto`-атрибутом.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схема: `npm/schemas/rule-meta.json`, поле `lint: { type: string, enum: ["quick", "ci"] }`. Парсер: `npm/scripts/lib/rule-meta.mjs`. Plan: `docs/superpowers/plans/2026-05-31-lint-quick-all-meta-json.md`, Задача 1–2.

## ADR ga / rego / text — класифікація як `"ci"` (крос-файлові правила)

## Context and Problem Statement

### Consequences
* Good, because transcript фіксує очікувану користь: швидкий `lint` не ганяє важкі крос-файлові перевірки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Spec таблиця класифікації: `docs/superpowers/specs/2026-05-31-lint-quick-ci-split-design.md`. Commit `788f4ba` (оновлення spec: "lint-ga → ci, GA rules are cross-file").

## ADR js-lint розщеплення (D3): quick = oxlint+eslint, ci = jscpd+knip

## Context and Problem Statement
`lint-js` є композитним скриптом: `oxlint --fix && eslint --fix . && jscpd . && knip`. Два перших інструменти приймають список файлів (придатні для quick), два останніх — крос-файлові аналізатори (тільки ci). Атрибут на рівні правила не може виразити обидві фази одночасно.

## Considered Options
* D1: грубий атрибут на правилі — весь `js-lint` у `quick` або весь у `ci`
* D2: атрибут гранулярніший — на рівні lint-кроку/інструменту
* D3: атрибут на правилі + спеціальне розщеплення тільки для `js-lint`

## Decision Outcome
Chosen option: "D3 — одне свідоме розщеплення `js-lint`", because 90% правил природно лягають у простий атрибут на `meta.json`; `js-lint` — єдиний реальний композит. D2 — YAGNI; D1 втрачає сенс quick.

### Consequences
* Good, because transcript фіксує очікувану користь: простота для решти правил, точне розділення для єдиного виключення.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Нові кроки: `lint-js` (quick: oxlint+eslint), `lint-js-ci` (ci: jscpd+knip). Plan: Задача 6. Нові файли: `npm/rules/js-lint/meta.json`, `npm/rules/js-lint-ci/meta.json`.
