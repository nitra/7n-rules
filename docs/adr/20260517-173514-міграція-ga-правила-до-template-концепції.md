## ADR Міграція ga-правила до template/-концепції

## Context and Problem Statement
Правила перевірки `ga` (GitHub Actions) мали inline-літерали (YAML-канони workflow, назви extensions, значення scripts) безпосередньо у Rego-файлах. Після того як `security`-правило вже мігрувало до патерну `template/`-каталогу (де канон зберігається у JSON/YAML і передається у Rego через `--data`), виникла необхідність зробити те саме для `ga`.

## Considered Options
* Залишити inline-літерали у Rego (поточний стан до 1.13.9)
* Перенести канонічні значення у `template/`-підкаталог і передавати їх через `data.template.*` (вже обкатаний патерн `security`)

## Decision Outcome
Chosen option: "Перенести канонічні значення у `template/`-підкаталог", because це вже обкатаний патерн проекту (`security` правило), і він дозволяє тримати канон в одному місці (JSON/YAML-файл) без дублювання між Rego і `.mdc`.

### Consequences
* Good, because `findMissingMdcRefs("ga")` повертає `[]` — усі template-файли автоматично верифікуються на наявність посилань у `ga.mdc`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фази міграції:
- **Phase 2** (v1.13.9): `npm/rules/ga/policy/package_json/template/package.json.contains.json`, `npm/rules/ga/policy/vscode_extensions/template/extensions.json.snippet.json`, `npm/rules/ga/policy/vscode_settings/template/settings.json.snippet.json`, `npm/rules/ga/policy/zizmor_yml/template/zizmor.yml.snippet.yml`
- **Phase 3.5** (v1.13.12): backfill для 4 full-canon workflow концернів (`clean_ga_workflows`, `clean_merged_branch`, `lint_ga`, `git_ai`), пропущених у Phase 2

## ADR Ігнорування regal-правил directory-package-mismatch та unresolved-reference

## Context and Problem Statement
Після виправлення `LINT_TARGETS` `regal lint` почав репортувати 111 violations `directory-package-mismatch` і 28 violations `unresolved-reference`. Обидва — наслідок архітектурних рішень проекту, а не помилок.

## Considered Options
* Вимкнути ці правила глобально у `.regal/config.yaml`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Вимкнути ці правила у `.regal/config.yaml`", because `directory-package-mismatch` — інтенціональна конвенція (`<rule>.<concern>` package у `<rule>/policy/<concern>/`), а `unresolved-reference` — наслідок runtime-injection через `--data` (`data.template.*`), яку regal не може зарезолвити статично.

### Consequences
* Good, because `regal lint npm/rules` завершується з "No violations found" (111 файлів), що дозволяє CI-перевірці бути дієвою без false-positives.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл конфігурації: `.regal/config.yaml`. Додаткові налаштування в цьому ж коміті: `style.line-length.max-line-length: 220` (узгоджено з поведінкою `opa fmt`, який тримає inline-об'єкти single-line). Комміт v1.13.10 (`81d8ea3`).

## ADR Патерн full-canon для workflow Rego-концернів: field-by-field з template/

## Context and Problem Statement
При міграції 4 ga workflow-концернів (`clean_ga_workflows`, `clean_merged_branch`, `lint_ga`, `git_ai`) потрібно було вибрати спосіб зберігання канонічного YAML: generic recursive subset-walker або per-concern field-by-field перевірки з template.

## Considered Options
* Generic walker (повний subset-of деревом) — новий рекурсивний snippet-walker у Rego
* Per-concern field-by-field (як є) — існуючі field-by-field денії, тільки literals виносяться у template

## Decision Outcome
Chosen option: "Per-concern field-by-field", because зберігає явність перевірок і не потребує нового generic механізму.

### Consequences
* Good, because transcript фіксує очікувану користь: підхід узгоджено з існуючими концернами (`security`, `ga.zizmor_yml`), opa test 354/354 після міграції.
* Bad, because `workflow_common` (cross-workflow forbidden-patterns) виключено зі скоупу як окремий partial-випадок.

## More Information
Виявлений практичний нюанс: Go-yaml (conftest) парсить `no` як `bool false`, тоді як npm `yaml` (v1.2) — як рядок. Вирішено через коментар у `template/clean-merged-branch.yml.snippet.yml` і перевірку типу у Rego: `is_boolean(input.jobs["clean-merged-branch"].steps[_].with.dry_run)`. Комміт v1.13.12 (`55a6751`).
