## ADR Реструктуризація `rules/<rule>/js/` → `{fix,lint,policy}/`

**Контекст:** У `npm/rules/<rule>/js/` накопичились два принципово різних типи файлів — `<concern>/check.mjs` (діагностика для `npx @nitra/cursor check`, fix-канал) і `lint.mjs`/`run.mjs` (CLI-entry для `bun run lint-X`, lint-канал). Структура не відображала цього поділу.

**Рішення/Процедура/Факт:** Проведено триетапну реструктуризацію (версії 1.11.9–1.11.12):
- **Фаза 1** — `discover-checkable-rules.mjs` і `run-rule.mjs` оновлено для підтримки dual-mode (`js/` та `fix/` одночасно); поле `rootDir` у `JsConcern`; fatal-помилка на дублікат концерну в обох каталогах.
- **Фаза 2** — усі 26 правил: `js/<concern>/` → `fix/<concern>/`, `js/lint.mjs` → `lint/lint.mjs`. CLI-entry `n-cursor.js:80-84` оновлено. `policy/` лишилась на місці як третій сиблінг.
- **Фаза 3** — legacy dual-mode прибрано з `discover-checkable-rules.mjs` і `run-rule.mjs`; залишено `fix/`-only.

Принцип: **технологія реалізації визначає директорію**:

| Директорія | Вміст | Канал |
|---|---|---|
| `fix/<concern>/` | `check.mjs`, `autofix.mjs` (опц.), тести | `npx @nitra/cursor check` |
| `lint/` | `lint.mjs`, runner-помічники | `bun run lint-X` |
| `policy/<concern>/` | `<concern>.rego`, `target.json` | `npx @nitra/cursor check` (rego) |

**Обґрунтування:** `policy/` залишено окремим сиблінгом (не в `fix/policy/`), бо rego і JS мають різні runner-и, namespace-конвенції та роль `target.json` — змішування ускладнює навігацію. `lint-conftest.mjs` видалено як дублікат: `discoverCheckableRules` автоматично підхоплює rego-concerns за `target.json`, тому окремий скрипт-ітератор не потрібен.

**Розглянуті альтернативи:**
- `fix/policy/<concern>/` (policy під fix-парасолею) — відхилено через різні runner-и і збільшення nesting.
- `fix/<concern>/` плоско для rego поруч із JS (варіант C) — відхилено через ризик колізії імен у правилах з однойменними JS- і rego-концернами (`text.cspell` vs JS `cspell/`).
- Big-bang в одному PR — відхилено на користь поетапного підходу.

**Зачіпає:** `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/run-rule.mjs`, `npm/bin/n-cursor.js:80-84`, `knip.json`, `npm/README.md`, `.cursor/rules/scripts.mdc`, `.cursor/rules/conftest.mdc`, усі 26 правил у `npm/rules/`; версія `@nitra/cursor` 1.11.9–1.11.13.

## ADR Видалення `lint-conftest` як окремого lint-кроку

**Контекст:** `npm/scripts/lint-conftest.mjs` запускав rego-policy через `conftest` по всіх активних правилах і реєструвався як крок у `bun run lint`. Фактично дублював `npx @nitra/cursor check`, який теж запускає policy-concerns через `runConftestBatch`.

**Рішення/Процедура/Факт:** `npm/scripts/lint-conftest.mjs` видалено. З `package.json#scripts.lint` прибрано ланку `lint-conftest`. Всі cross-reference у `.mdc`, `check.mjs` і `.rego` замінено з `bun run lint-conftest` → `npx @nitra/cursor check`. Rego-policy з умовним запуском (без `target.json`, наприклад `tauri/policy/vscode_extensions/`) переформульовано як «не auto-discoverable».

**Обґрунтування:** Після впровадження `discoverCheckableRules` з авто-підхопленням rego за `target.json`, `lint-conftest.mjs` став зайвим — він дублював ту саму ітерацію, але через ручну реєстрацію у `TARGETS`-масиві. Один канал простіший у підтримці й усуває ризик розсинхронізації TARGETS зі структурою `policy/`.

**Розглянуті альтернативи:** Не обговорювались — рішення прийнято після встановлення того, що обидва канали роблять одне й те саме.

**Зачіпає:** `npm/scripts/lint-conftest.mjs` (видалено), `package.json#scripts.lint`, `.cursor/rules/conftest.mdc:90`, `.cursor/rules/scripts.mdc:30`, `npm/rules/abie/abie.mdc`, 10 `check.mjs`-файлів, 7 `.rego`-файлів.
