`readOnly` у lint-правилах — це guard проти автофіксу. Але `lint()` doc-files **ніколи** не фіксить — вона detect-only за конструкцією. Тобто `readOnly` тут немає чого блокувати.

Логіка така:

```
lint(files, cwd, { readOnly })   ← readOnly гасить автофікс у правилах що пишуть
↓
doc-files/js/lint.mjs
→ scanForDocFiles + filter(stale)
→ reportStale()   ← лише process.stderr.write + return 1
← ніколи не пише жодного файла
```

Якщо додати `readOnly` — він просто нічого б не змінив. Відсутність параметра — правдива сигнатура.

## Чому генерацію не можна перенести в lint-крок

Якщо ти думаєш «якби lint фіксив доки сам, `fix-doc-files` був би зайвий» — ось чому це не працює:

| | lint-крок | fix-doc-files |
|---|---|---|
| Ціна | ~0 (CRC hash) | omlx LLM call на файл |
| Запуск | кожен `git commit` hook, CI | ручно / opt-in |
| Fail-fast | ✅ exit 1 | не детект |
| `--read-only` | незмінний (не пише) | n/a |

Запускати локальну LLM на кожен змінений файл у кожному lint-запуску — занадто дорого і занадто повільно для pre-commit/CI.

**Поділ правильний:** `lint` = «чи свіже?» (дешево, завжди), `fix-doc-files` = «зроби свіжим» (дорого, opt-in). Єдине що лишилось прибрати — deprecated `doc-files <sub>`. Знести?
