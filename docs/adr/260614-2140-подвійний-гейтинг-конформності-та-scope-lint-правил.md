title: Подвійний гейтинг конформності та scope lint-правил
description: Конформність lint має читати авторитетний конфіг напряму, а scope правил має відповідати реальній поведінці сканерів.
---

**Status:** Accepted
**Date:** 2026-06-14

## Context and Problem Statement

Конформність-фаза `npx @nitra/cursor lint --full` визначала правила через `.cursor/rules/*.mdc` на диску, а не через прямий список `rules` у `.n-cursor.json`. Це створювало подвійний гейтинг: `sync` генерує `.mdc`, discovery читає `.mdc`, а `fix.mjs` правила все одно перевіряє конфіг через `runRuleCli` / `isRuleEnabled`. Окремо `security` було класифіковано як `lint: "per-file"`, хоча реалізація запускала `trufflehog filesystem .` по всьому репозиторію.

## Considered Options

- Discovery конформності через `.cursor/rules/*.mdc` і поточний `security` як `lint: "per-file"`.
- Discovery конформності напряму через `.n-cursor.json` + `isRuleEnabled`, а `.mdc` лишити як AI/Cursor-контекст або fallback.
- Рекласифікувати `security` у `lint: "full"` або реалізувати справжній per-file scan для trufflehog.

## Decision Outcome

Chosen option: "Discovery конформності через `.n-cursor.json` + виправлення scope metadata", because transcript показав, що `.n-cursor.json` є авторитетним джерелом, `.mdc` може дрейфувати без `sync`, а metadata має відповідати реальній поведінці сканера.

### Consequences

- Good, because `lint --full` не залежатиме від застарілого стану `.cursor/rules/` і читатиме авторитетний конфіг.
- Good, because `security` як full-scope правило не маскуватиме full-repo scan під delta-lint.
- Bad, because transcript фіксує можливий edge-case: за відсутнього `.n-cursor.json` open-by-default режим може запустити більше правил, якщо не лишити `.mdc` fallback.
- Bad, because якщо `security` стане full-only, звичайний delta-`lint` не перевірятиме секрети у змінених файлах.

## More Information

- Поточна discovery логіка: `npm/scripts/lib/discover-check-rules-from-cursor.mjs`.
- Пропонована точка зміни: `npm/scripts/lib/fix/run-fix-check.mjs`, `resolveCheckRuleIds`.
- Конфіг-утиліти: `npm/scripts/lib/read-n-cursor-config-lite.mjs`, `readNCursorConfigLite`, `isRuleEnabled`.
- Кожен `fix.mjs` уже проходить через `npm/scripts/lib/run-rule-cli.mjs`.
- `security` metadata: `npm/rules/security/meta.json`; реалізація: `npm/rules/security/js/lint.mjs`, де `_files` ігнорується і викликається `trufflehog filesystem .`.
- `npm/rules/lint/js/orchestrate.mjs` читає `meta.json` для scope-вибору.

## Update 2026-06-14

- `n-cursor lint` має дві незалежні осі: per-file lint-скан читає `meta.json` пакета, а конформність-фаза працює через discovery правил.
- Per-file правила з `lint: "per-file"` включають `doc-files`, `js-lint`, `security`, `style-lint`, `text`; full-only правила включають `ga`, `js-lint-ci`, `rego`.
- Конформність-фаза активується лише при `--full` або `lint-ci`.
- У per-file фазі зафіксовано fail-fast: перший ненульовий exit-код зупиняє прогін і наступні правила не запускаються.
- Transcript не містить підтвердження, чи fail-fast заважає побачити решту порушень за один прогін.

## Update 2026-06-14

- Схема має явно показувати `sync` як посередника між `.n-cursor.json` і `.cursor/rules/*.mdc`.
- `lint` не читає `.n-cursor.json` напряму для discovery конформності; він бачить стан `.cursor/rules/*.mdc` на диску.
- Якщо `sync` не запускався, `.mdc` можуть не відповідати конфігу, тому документація має описувати це як sync-інваріант, а не прямий причинний звʼязок.
