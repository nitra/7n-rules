## ADR concern-based JS + per-policy target.json у @nitra/cursor

**Контекст:** Кожне правило `@nitra/cursor` мало один монолітний `rules/<id>/js/check.mjs` (abie — 1153 рядки) та дублювало targeting для rego-полісі у hardcoded `TARGETS`-таблиці `lint-conftest.mjs` (~280 рядків) і в JS-викликах `runConftestBatch`. Розширення правила потребувало змін у трьох місцях одночасно.

**Рішення/Процедура/Факт:**
- Введено декларативні `target.json` поруч із кожним `<concern>.rego`: `{ "files": { "single" | "walkGlob" } }`. CLI резолвить файли через picomatch і запускає `conftest` — JS не дублює виклики.
- Кожне правило розпиляно на `js/<concern>/check.mjs` (25 правил, 5 batch-ів). Applies-гейт через `js/applies/check.mjs` з опційним `applies()` export — `false` скіпає правило цілком.
- `abie`: монолітний `check.mjs` (1153 рядки) → 6 концернів + 8 utils із кешованим shared state (`utils/k8s-tree.mjs` — module-level map, ключ `(root, ignorePaths)`).
- `lint-conftest.mjs`: hardcoded `TARGETS`-таблиця замінена на `discoverCheckableRules` + читання `target.json`.
- Legacy-fallback (flat `js/check.mjs`) прибрано з `discoverCheckableRules` після повного concern-split.
- Виправлено 3 баги: kebab→snake namespace у `run-rule.mjs`, picomatch array-negation у `resolve-target-files.mjs`, subdir-preference у `discoverCheckableRules`.
- 9 коммітів: `c0f7e92` (feat), `19b9ab4`–`27ea6cd` (batch 1-5), `93b9d62` (legacy removal).

**Обґрунтування:** Один каталог = одне правило, чітка межа між декларативним (rego) і імперативним (JS). Нові полісі додаються без змін у `lint-conftest.mjs`. Pure-rego правила можливі без JS-обгортки. Shared walk-cache усуває повторний `walkDir` при виклику кількох концернів одного правила.

**Розглянуті альтернативи:** Не обговорювалися явно; попередня архітектура (монолітний check.mjs + hardcoded TARGETS) була відправною точкою.

**Зачіпає:** `npm/scripts/lint-conftest.mjs`, `npm/scripts/utils/discover-checkable-rules.mjs`, `npm/scripts/utils/run-rule.mjs`, `npm/scripts/utils/resolve-target-files.mjs`, `npm/rules/*/js/<concern>/check.mjs` (25 правил), `npm/rules/*/policy/*/target.json` (~50 файлів), `npm/rules/abie/utils/k8s-tree.mjs`, `npm/tests/integration-repo-checks.test.mjs`

## ADR auto-skills читає auto.md замість hardcoded мапи

**Контекст:** `npm/scripts/auto-skills.mjs` мав hardcoded константи `SKILL_DEPENDENCIES` та `ALWAYS_ON_SKILLS` — дублювання тієї ж інформації, що вже була задокументована у файлах `npm/skills/<id>/auto.md` (формат: `завжди`, `[rule-id]`, або відсутній файл). Зміна будь-якої залежності потребувала правки і `auto.md`, і JS.

**Рішення/Процедура/Факт:** `auto-skills.mjs` тепер сканує `npm/skills/<id>/auto.md` під час завантаження модуля і будує `AUTO_SKILL_RULE_DEPENDENCIES` та `ALWAYS_ON_SKILLS` із вмісту файлів. Три формати: `завжди` → always-on, `[rule, …]` → умова на правила, відсутній / нерозпізнаний → opt-in. Commit `b7ad911`, версія `1.11.2`.

**Обґрунтування:** Єдине джерело правди — `auto.md`. Після цієї зміни додавання нового скіла з умовами потребує тільки правки `auto.md`, без змін у JS.

**Розглянуті альтернативи:** Не обговорювалися; hardcoded-підхід був pre-existing станом.

**Зачіпає:** `npm/scripts/auto-skills.mjs`, `npm/scripts/auto-skills.test.mjs`, `npm/skills/*/auto.md`

## Knowledge path-depth після concern-split: +1 рівень `../`

**Контекст:** При механічному переміщенні `rules/<id>/js/check.mjs` → `rules/<id>/js/<concern>/check.mjs` усі відносні імпорти (`'../../../scripts/...'`, `'../../docker/js/...'`) та `join(here, '..', '..', '..', ...)` всередині файлів стають хибними — глибина зростає на один рівень (`<concern>/` — нова директорія).

**Рішення/Процедура/Факт:** Після кожного `git mv` треба замінити всі `../../../` на `../../../../` у переміщених файлах. Не тільки `import from`, а й `fileURLToPath`-based path-constants (`BUNDLED_HOOKS_DIR`, `OXLINT_CANONICAL_JSON_PATH`, `KNIP_CANONICAL_JSON_PATH`). Баги проявлялися як `ENOENT` у тестах, а не як синтаксичні помилки — їх легко пропустити без запуску тестів після кожного batch.

**Обґрунтування:** Виявлено при виконанні batch 1 (adr: `BUNDLED_HOOKS_DIR`), batch 3 (js-lint: `OXLINT_CANONICAL_JSON_PATH`), batch 4 (nginx-default-tpl: крос-правило `docker/js/lint/check.mjs`).

**Розглянуті альтернативи:** Абсолютні path-constants через `import.meta.resolve` або спільний `paths.mjs` — не обговорювалися; виправлено точковими `sed` патчами.

**Зачіпає:** Будь-який `check.mjs` або `check.test.mjs`, що містить `fileURLToPath`-based paths або відносні cross-rule imports — після concern-split автоматично потребує перегляду глибини.
