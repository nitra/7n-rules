# Інтерактивне рішення перед додаванням виключень у конфіги лінтерів

**Status:** Accepted
**Date:** 2026-05-20

## Context and Problem Statement

Скіл `/n-lint` мовчазно розширював виключення у конфігах (`.jscpd.json`, `.cspell.json`, `knip.json`, eslint/oxlint ignores) замість рефакторингу коду. Такі тихі зміни накопичуються непомітно й знецінюють lint-перевірки, оскільки реальні порушення перестають фіксуватися.

## Considered Options

- Мовчазне додавання виключень у конфіги (поведінка до змін)
- Інтерактивний `AskQuestion` перед будь-якою правкою конфігів з виключеннями

## Decision Outcome

Chosen option: "Інтерактивний `AskQuestion` перед будь-якою правкою конфігів з виключеннями", because користувач явно вказав, що LLM не має самостійно розширювати `ignore`/`words`/`minLines`; замість цього — пропонувати варіанти з поясненнями та варіант `refactor` як рекомендований.

### Consequences

- Good, because виключення більше не додаються мовчазно — кожне потребує явного підтвердження через `AskQuestion`.
- Good, because варіант `refactor` позначений як рекомендований, що спрямовує LLM до виправлення першопричини замість обходу.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Змінені файли: `npm/skills/lint/SKILL.md`, `.cursor/skills/n-lint/SKILL.md`
- Синхронізація: `npx @nitra/cursor` копіює `npm/skills/` → `.cursor/skills/`
- Новий блок у SKILL.md: «Винятки в конфігурації — інтерактивне рішення» — таблиця конфігів, обов'язковий `AskQuestion`, варіанти `refactor` / `ignore-once` / `skip` / `explain`
- Після `ignore-once` — фіксація причини відмови від рефакторингу у коментарі
