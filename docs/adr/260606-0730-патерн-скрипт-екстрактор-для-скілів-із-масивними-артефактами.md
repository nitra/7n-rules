## ADR Патерн «скрипт-екстрактор» для скілів із масивними артефактами

## Context and Problem Statement
Скіли `n-coverage-fix`, `n-taze`, `n-start-check` та `n-fix` змушували LLM-агента читати та парсити великі сирі артефакти (COVERAGE.md ~2.76 МБ, `package.json.taze-bak`, монорепо-воркспейси, вивід `fix`), витрачаючи сотні тисяч токенів на детерміновану механічну роботу, яку міг виконати скрипт.

## Considered Options
* Read-only CLI-екстрактор `n-cursor <skill> <verb>` (скрипт несе парсинг, агент отримує компактний JSON-зріз)
* Широкий `lint-extract` для скіла `n-lint` (аналізувався й відхилений)

## Decision Outcome
Chosen option: "Read-only CLI-екстрактор `n-cursor <skill> <verb>`", because детермінований парсинг у CLI дає 0 LLM-токенів на механічну роботу, агент отримує лише компактний `index`/`slice` під один елемент і лишається з когнітивною роботою (рефакторинг, діагностика, пояснення).

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор `coverage-fix` читає 7.4 КБ (`index`) замість 2.76 МБ (~350× менше токенів); аналогічно для `taze diff`, `start-check scan|run`, `fix --json`.
* Bad, because широкий `lint-extract` визнаний шкідливим (андалізувався і відхилений): `&&`-ланцюг показує лише один інструмент за раз, половина без JSON, подвійний прогін = drift; `n-lint` лишився без змін.

## More Information
- Вимога зафіксована в `.cursor/rules/scripts.mdc` v1.13, секція «🔴 ВИСОКИЙ ПРІОРИТЕТ — детермінований парсинг у скіла: у CLI, не в LLM».
- Реалізовано: `npm/skills/taze/js/diff.mjs` (`taze diff`), `npm/skills/start-check/js/check.mjs` (`start-check scan|run`), `runFixCommand(…, {json:true})` у `npm/bin/n-cursor.js`, `npm/scripts/coverage-fix-extract.mjs` (`coverage-fix index|slice`).
- Change-файли: `npm/.changes/260604-2133.md`, `npm/.changes/260605-0716.md`, `npm/.changes/260605-0731.md`, `npm/.changes/260606-0636.md`.

## ADR Баг-фікс `permissionMode` у headless SDK-воркері `coverage --fix`

## Context and Problem Statement
`npm/scripts/coverage-fix.mjs` спавнив агента через `@anthropic-ai/claude-agent-sdk` `query()` без параметра `permissionMode`, тому headless-воркер не отримував дозволу на запис файлів і фактично нічого не редагував (підтверджено пробами: файл `(missing)` без режиму, `WORKER_OK` з режимом).

## Considered Options
* Додати `permissionMode: 'bypassPermissions'` до виклику `query()`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Додати `permissionMode: 'bypassPermissions'` до виклику `query()`", because це той самий режим, що дозволив `Write`/`Edit`/`Bash` у feasibility-пробах; `acceptEdits` теж спрацював, але `bypassPermissions` відповідає автономному характеру команди `coverage --fix`.

### Consequences
* Good, because transcript фіксує очікувану користь: headless-воркер тепер реально редагує файли (підтверджено функціональним пробом `mode=bypassPermissions msgs=10 file="WORKER_OK"`).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Файл: `npm/scripts/coverage-fix.mjs`, виклик `query({…, options:{…, permissionMode:'bypassPermissions'}})`.
- Тест оновлено: `npm/scripts/tests/coverage-fix.test.mjs` — assertion `toEqual` тепер перевіряє `permissionMode`.
- Change-файл: `npm/.changes/260606-0721.md` (Fixed, patch).
