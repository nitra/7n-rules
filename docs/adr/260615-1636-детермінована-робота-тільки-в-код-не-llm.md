## ADR Детермінована робота — тільки в код, не LLM

## Context and Problem Statement
У npm-підсистемах (`doc-files`, `adr`, `lint`, `coverage-classify`) LLM-виклики виконували й творчу, й детерміновану (механічну, обчислювану кодом) роботу в одному запиті. Це підвищувало вартість і знижувало передбачуваність: детермінована частина нестабільна при температурі > 0 і не потребує нейромережі.

## Considered Options
* Залишити статус-кво: LLM виконує обидва типи роботи в одному prompt
* Вилучити детерміновану роботу в JS-код, LLM залишається лише для творчих завдань

## Decision Outcome
Chosen option: "Вилучити детерміновану роботу в JS-код", because користувач явно поставив задачу «прибрати детерміновану роботу від LLM зовсім» — і аналіз підтвердив, що в `npm/rules/lint/js/orchestrate.mjs`, `npm/rules/adr/js/hooks.mjs`, `npm/scripts/coverage-classify/index.mjs`, `npm/scripts/lib/adr/normalize-pipeline.mjs` та docgen-пайплайні (`npm/rules/doc-files/js/`) є частини, котрі LLM виконує, але вони обчислювані детерміністично.

### Consequences
* Good, because transcript фіксує очікувану користь: детермінована логіка стає стабільною, тестованою й дешевшою.
* Bad, because transcript не містить підтверджених негативних наслідків.
