```

--data policy/<concern>/template/snippet.json \
--data-namespace <concern>.template
```


2. Витягуємо inline-значення, що дублюються у `.mdc`:
- required fields/values → `template/snippet.json`
- forbidden paths → `template/deny.json`
- contains-перевірки → `template/contains.json`
3. Якщо є нові файли — оновлюємо rego/check.mjs щоб читали через `checkTemplate()` / `data.<concern>.template.snippet`.
4. Оновлюємо `.mdc`: замінюємо inline code block на посилання.

**Правила, де inline-значень мало або не буде template** (лишаємо як є):
- Правила з чисто структурними або версійними перевірками (regex, semver) — template не рятує від regex-ів.
- `tauri`, `js-bun-redis` — по 1-2 рядки inline, overhead-дискусійний, але однаковість важливіша.

# Writing Plans

Use this skill to write a detailed implementation plan from a spec or set of requirements.

Plans are lightweight — they're just a numbered list of tasks with enough context to execute without re-reading everything.

IMPORTANT: Before writing the plan, understand the CURRENT codebase state. The spec describes what to build; the codebase may already have parts of it. Your plan must reflect reality, not just the spec.

## Process

1. Read the spec (or understand the requirements from conversation)
2. Explore the current codebase to understand what already exists
3. Write the plan

## Plan Format

A plan is a numbered list of implementation tasks. Each task is a markdown section with:

```markdown
## {n}. {Title}

**Files:** `path/to/file.ts`, `path/to/other.ts`
**Tests:** `path/to/test.ts`
**Notes:** {brief context, gotchas, decisions already made}

Steps:
- {concise implementation steps}
```

## Key Principles

- **Explore before writing** — check what already exists, don't assume the spec is ground truth
- **Reference reality** — specific file paths, function names, line numbers from the actual codebase
- **Size tasks for single sessions** — each task should be completable in one focused work session
- **Front-load risks** — order tasks so unknowns surface early, before downstream work depends on them
- **One plan per file** — save to `docs/superpowers/plans/YYYY-MM-DD-<topic>-plan.md`
- **Non-negotiable: tests** — every task that writes code must have a test file specified

## Plan quality checks

Review the plan before saving. A good plan has:
- [ ] Tasks ordered so dependencies come first
- [ ] No "figure out how to" vague steps — every step is a concrete action
- [ ] Every implementation task has a test file
- [ ] File paths verified to exist (or noted as new files)
- [ ] Critical unknowns surfaced in early tasks, not buried

**Контекст:** Правила в `npm/rules/*/*.mdc` накопичили значний обсяг пояснювальних блоків (VS Code-налаштування, інструкції встановлення CLI, описи зовнішніх інструментів, секції «Перевірка», активаційна мета), які не впливають на поведінку AI при застосуванні правила — але витрачали токени контексту.

## ADR Архітектура `template/` — єдине джерело канонічних значень у правилах

**Контекст:** Очікувані значення (назви скриптів, поля файлів, заборонені ключі) дублювались у трьох місцях одночасно: у `.mdc` як inline code block, у `fix/<concern>/check.mjs` як hardcoded рядки, у `policy/<concern>/*.rego` як Rego-літерали. Зміна канонічного значення вимагала оновлення трьох файлів; ризик розсинхронізації.

**Рішення/Процедура/Факт:** Додати каталог `template/` всередині кожного `fix/<concern>/` та `policy/<concern>/` з трьома можливими JSON-файлами: `snippet.json` (required leaf-values, deep partial match, масиви — order-insensitive subset), `deny.json` (заборонені шляхи, leaf = опис порушення), `contains.json` (масиви рядків-підрядків). Новий хелпер `npm/scripts/load-template.mjs` (`checkTemplate(templateDir, parsed)`) використовується у `check.mjs`. Runner `buildConftestArgs` (n-cursor.js, рядок 247) передає template-файли як `--data <file> --data-namespace ${checkId}.${concernId}.template.<baseName>` у conftest. `.mdc` замінює inline snippet на посилання `[snippet.json](./fix/<concern>/template/snippet.json)`. Scope: 23 правила (усі з `fix/` або `policy/`).

**Обґрунтування:** Єдиний файл `template/snippet.json` стає source of truth, який одночасно читає JS-скрипт і отримує conftest через `--data`; AI бачить синтаксис цільового файлу без нового DSL; формат JSON дозволяє conftest нативно приймати `--data-file`; суфіксна конвенція (`.deny`, `.contains`) зберігає файли в синтаксисі target-файлу, але з різною семантикою перевірки.

**Розглянуті альтернативи:** (1) Expectations DSL (один YAML із JSONPath і операторами) — відхилено: новий DSL, AI треба вчити, rego отримує чужий формат; (2) Snippet-only без deny/contains — відхилено: частина правил потребує substring-перевірок, тому snapshot was incomplete; (3) template/ на рівні правила (не концерну) — відхилено: один концерн вже знає свій target через `target.json`; кілька концернів на один target-файл не конфліктують.

**Зачіпає:** `npm/bin/n-cursor.js` (функція `buildConftestArgs`); новий `npm/scripts/load-template.mjs` + `load-template.test.mjs`; `npm/rules/*/fix/<concern>/template/`; `npm/rules/*/policy/<concern>/template/`; всі 23 правила з `fix/`/`policy/`; `docs/superpowers/specs/2026-05-17-rule-templates-design.md`; `docs/superpowers/plans/2026-05-17-rule-templates-plan.md`.
