Ось як працює скіл `docgen`:

---

## Загальна ідея

Скіл генерує **поведінкову** українську markdown-документацію для кожного кодового файлу (`*.js`, `*.mjs`, `*.ts`, `*.vue`, `*.py`) і кладе її **поряд із кодом** — `<dir>/docs/<stem>.md`. Один субагент = один файл (паралельно, батчами по 5).

---

## 3 рівні документації (строго послідовно)

```
Tier 1  →  Tier 2  →  Tier 3
(файли)    (модулі)   (домени)
```

**Tier 1 — файловий.** Субагент на кожен файл пише `<dir>/docs/<stem>.md`. Секції: `## Огляд`, `## Поведінка`, `## Публічний API` (лише якщо нетривіальна поверхня), `## Де використовується`, `## Гарантії поведінки`. Без stdlib-кухні, без сигнатур листкових функцій.

**Tier 2 — module-summary.** Після Tier 1 — `npx @nitra/cursor docgen modules` дає список логічних модулів. Субагент читає файлові доки членів і пише `<module_root>/docs/ARCHITECTURE.md`.

**Tier 3 — доменні доки.** Один субагент-синтезатор читає всі `ARCHITECTURE.md`, виділяє бізнес-домени (можуть перетинати межі модулів) і пише `docs/<домен>.md` у кореневу `docs/`.

---

## Workflow по кроках

| Крок | Команда / дія |
|---|---|
| 1 | `npx @nitra/cursor docgen scan [--root <dir>]` → JSON-список файлів |
| 2 | Фільтр: `exists:true` пропускаємо (якщо не `--overwrite`) |
| 3 | Диспатч Tier 1 батчами по 5 субагентів одночасно |
| 4 | `npx @nitra/cursor docgen modules` → Tier 2 (так само батчами по 5) |
| 5 | Один субагент-синтезатор → Tier 3 |
| 6 | Підсумок: N описано / S пропущено / E помилок |

---

## Обмеження і правила

- **Scanner ігнорує:** `node_modules`, `dist`, `.git`, `__pycache__`, `coverage`, `.cursor`, `.claude`, `docs/`, `*.test.*`, `*.spec.*`, `*.d.ts`.
- **Не комітить** автоматично — користувач вирішує.
- **Помилка одного файлу не зупиняє батч** — решта продовжується.
- Скіл має **три копії:** `npm/skills/docgen/SKILL.md` (джерело), `.cursor/skills/n-docgen/SKILL.md` (активна), `.pi/skills/n-docgen/SKILL.md` (стаб-делегат).
