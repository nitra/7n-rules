# Guard-механізм блокування паралельних важких команд

**Status:** Accepted
**Date:** 2026-05-22

## Context and Problem Statement

Кілька AI-агентів в одному проекті можуть паралельно запускати важкі перевірки (`lint-ga`, `lint-rego`, `lint-text`, `lint-k8s`, `lint-docker`, `check`), що спричиняє конкуренцію за ресурси CPU/диску та дублюючі запуски. macOS не має `flock` — потрібна альтернатива без зовнішніх залежностей, що підтримує взаємне виключення і дедуплікацію результатів.

## Considered Options

- **Bun-native атомарний `mkdirSync`** (APFS-гарантія унікальності) — повний контроль над JSON-станом дедупу, ~150 рядків без зовнішніх залежностей.
- **`shlock` (`/usr/bin/shlock`, BSD)** — дає лише взаємне виключення; дедуп потрібно писати самостійно + залежність від зовнішнього бінарника і `spawn`.
- **npm-пакет `proper-lockfile`** — зайва залежність заради ~150 рядків, без підтримки дедупу, з непотрібною кросплатформністю.
- **Один глобальний лок** — будь-яка важка команда блокує всі інші незалежно від типу.
- **Лок per-команда** — `lint-ga` і `check` мають незалежні простори виключення.
- **Окрема команда-обгортка `n-cursor guard <key> -- <cmd>`** — скіл може забути її викликати й запустити `bun run lint` напряму, обійшовши лок.
- **Лок вбудований у CLI-функцію** — спрацьовує при будь-якому способі запуску.
- **Fail-fast при зайнятому локу** — агент Б завершується з помилкою.
- **Чекати + дедуплікувати** — агент Б чекає, потім повертає результат агента А якщо fingerprint збігається.

## Decision Outcome

Chosen option: "Bun-native `mkdirSync` + лок per-команда + чекати і дедуплікувати + лок вбудований у CLI-функцію; пілот на `lint-ga`", because:
- `mkdirSync` на APFS атомарний без зовнішніх залежностей; `shlock` і `proper-lockfile` вносять залежності без підтримки дедупу.
- Лок per-команда відповідає явному вибору користувача — різні команди не конкурують між собою.
- Вбудований лок: користувач явно відхилив окрему команду-обгортку («навіщо мені нова команда якщо я хочу щоб лок був в кожній js перевірці»); hook також відхилений.
- Дедуп при `exitCode === 0` + fingerprint + TTL: агент Б не губить результат без причини.
- Пілот на `lint-ga` перевіряє підхід на реальній команді перед розкочуванням.

**Семантика очікування:** polling ~1–2 с, таймаут 20 хв; після закінчення — лог-попередження і виконання без локу (щоб не вічно висіти). Дедуп спрацьовує лише при `exitCode === 0`; невдалий прогон завжди перезапускається.

**Fingerprint:** sha256(`git rev-parse HEAD` + `git diff HEAD` + вміст untracked-файлів). Поза git-репо — fingerprint `null`, дедуп вимкнено, лок залишається.

**Cache TTL:** 10 хв (поле `finishedAt` у `<key>.result.json`).

### Consequences

- Good, because мінімум рухомих частин; дедуп природно лягає в той самий код що й лок без зовнішніх залежностей.
- Good, because різні команди (`lint-ga` і `check`) не блокують одна одну; паралельні перевірки різного типу залишаються можливими.
- Good, because скіли, `package.json`, агенти не змінюються — лок спрацьовує при будь-якому способі запуску `lint-ga`.
- Bad, because polling до 20 хв тримає агент Б у стані очікування, що може бути непомітним.
- Neutral, because поза git-репо дедуп вимкнено; лок продовжує працювати (прийнятний fallback).

## More Information

- Реалізація: `npm/scripts/utils/with-lock.mjs`; fingerprint: `npm/scripts/utils/worktree-fingerprint.mjs`.
- Стан: `node_modules/.cache/n-cursor/<key>/`; дир-лок `<key>.lock/` містить `owner.json` `{pid, host, startedAt, fingerprint}`.
- Живість PID: `process.kill(pid, 0)`; застарілий лок (мертвий PID або вік > 30 хв) видаляється автоматично.
- Стан результату: `<key>.result.json` `{finishedAt, exitCode, fingerprint}`.
- Інтеграція в `lint-ga`: `runLintGaSteps()` приватна; публічний `export const runLintGaCli = () => withLock('lint-ga', runLintGaSteps)` у `npm/rules/ga/lint/lint.mjs`; виклик у `n-cursor.js` case `'lint-ga'` не змінюється.
- Наступні команди для розкотювання: `lint-rego`, `lint-text`, `lint-k8s`, `lint-docker`, `check`.
- Спека: `docs/superpowers/specs/2026-05-22-lint-ga-concurrency-lock-design.md` (закомічено як `fd8d471`).

## Update 2026-05-23

Реалізовано примітив блокування. `mkdirSync`-based lock (атомарний на APFS) замість `flock`/`shlock`/`proper-lockfile` — повний контроль над дедупом без зовнішніх залежностей (~150 рядків у `npm/scripts/utils/with-lock.mjs`). Поведінка при зайнятому локу: poll-and-wait (таймаут 20 хв, потім попередження і виконання без локу) + дедуп при `exitCode === 0` і збігу відбитку. Гранулярність: per-command key (`lint-ga`, `lint-rego` тощо — паралельні між собою). Відбиток: SHA-256 від `git rev-parse HEAD` + `git diff HEAD` + untracked-файли + TTL 10 хв (`npm/scripts/utils/worktree-fingerprint.mjs`); поза git-репо дедуп вимкнено. Стан: `node_modules/.cache/n-cursor/<key>/lock/owner.json` (`{pid, host, startedAt, fingerprint}`) + `result.json` (вже gitignored). Інтеграція: вбудована в кожну команду (`runLintGaCli = () => withLock('lint-ga', runLintGaSteps)`), не окрема CLI-обгортка. Чиста функція `shouldDedup` — окремо для юніт-тестів. Тести: `npm/scripts/utils/tests/with-lock.test.mjs`, `worktree-fingerprint.test.mjs`.
