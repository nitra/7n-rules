## ADR Детермінований парсинг у CLI, агент отримує компактний зріз

## Context and Problem Statement
Скіли `n-coverage-fix`, `n-taze`, `n-start-check` та `n-fix` змушували LLM-агента вручну читати великі артефакти (`COVERAGE.md` ≈ 2.76 МБ ≈ ~700K токенів) і виконувати детерміновану роботу: розгортати glob-воркспейси, diff-ити `package.json.taze-bak`, розпізнавати exit-коди, парсити `❌`-маркери з тексту. Це навантажувало контекстне вікно, збільшувало вартість і зменшувало надійність (парсинг тексту нестабільний порівняно з JS-кодом).

## Considered Options
* Лишити детермінований парсинг на боці LLM-агента (статус-кво).
* Перенести детермінований парсинг у read-only CLI-екстрактор (`n-cursor <skill> <verb>`), агенту віддавати лише компактний результат.

## Decision Outcome
Chosen option: "Перенести детермінований парсинг у CLI-екстрактор", because цей підхід дозволив зменшити токени оркестратора в ≈350× для `coverage-fix` (7.4 КБ index замість 2.76 МБ), знижує ризик помилкового парсингу та є реалізованою вимогою: `n-cursor coverage-fix index|slice`, `n-cursor taze diff`, `n-cursor start-check scan|run`, `n-cursor fix --json`. Патерн зафіксований у `.cursor/rules/scripts.mdc` v1.13 як вимога для всіх нових і рефакторованих скілів.

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор читає `index` 7.4 КБ (~2K токенів) замість `COVERAGE.md` 2.76 МБ (~700K токенів); smoke-тест `scan` правильно класифікував три воркспейси (root→cli, demo→server, npm→cli); `fix --json` повернув чистий JSON зі скоупованим per-rule виводом.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізовані файли: `npm/scripts/coverage-fix-extract.mjs` (`parseSurvivedBlock`, `buildIndex`, `readSurvived`, `runCoverageFixCli`); `npm/skills/taze/js/diff.mjs` (`parseVersion`, `isBreaking`, `diffPackageJson`, `collectTazeDiff`, `runTazeCli`); `npm/skills/start-check/js/check.mjs` (`scanStartWorkspaces`, `classifyStartType`, `parseStartLog`, `runWorkspaceStart`, `runStartCheckCli`); `npm/bin/n-cursor.js` — case `'coverage-fix'`, `'taze'`, `'start-check'`, прапорець `--json` у `runFixCommand`. Правило: `.cursor/rules/scripts.mdc` v1.13, секція «🔴 ВИСОКИЙ ПРІОРИТЕТ — детермінований парсинг у скіла: у CLI, не в LLM». Change-файли: `npm/.changes/260604-2133.md`, `npm/.changes/260605-0716.md`, `npm/.changes/260605-0731.md`, `npm/.changes/260606-0636.md`.

## ADR Відмова від широкого `lint-extract` для скіла `n-lint`

## Context and Problem Statement
Після аудиту 9 скілів скіл `n-lint` був ідентифікований як кандидат №1 за ROI для рефактора (LLM парсить сирий лог ~10 інструментів). Постало питання, чи реалізовувати широкий `n-cursor lint-extract`, що парсить увесь `&&`-ланцюг `lint`.

## Considered Options
* Реалізувати широкий `lint-extract`, що захоплює й уніфікує вивід усього ланцюга (oxlint, eslint, jscpd, knip, cspell, shellcheck, markdownlint, v8r, stylelint, trufflehog).
* Відмовитись від широкого `lint-extract`; залишити для on-demand scoped варіанту (лише oxlint+eslint).

## Decision Outcome
Chosen option: "Відмовитись від широкого `lint-extract`", because реалізація суперечить архітектурі repo: кореневий `lint` — `&&`-ланцюг, що зупиняється на першому фейлі, тому агент і так бачить лише один впалий інструмент за раз. Половина тулів не має JSON-виводу (trufflehog, jscpd, knip, cspell, stylelint), а JSON-режим oxlint/eslint вимагав би повторного прогону важких тулів — подвійний lint і дублювання канонічного `withLock`-ланцюга (drift-ризик, порушення `scripts.mdc`).

### Consequences
* Good, because transcript фіксує очікувану користь: уникнуто подвійного прогону lint і maintenance-surface крихких text-парсерів для ~5 інструментів без JSON-виводу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Рішення задокументовано в пам'яті проєкту: `project_skill_extractor_refactor_backlog.md` — «lint-broad шкідливий»; широкий sweep зупинено після `fix`; залишок (`llm-patch`, `docgen filter`, `publish-telegram`) — on-demand.

## ADR `n-cursor fix --json` пропускає `ensureHkInstall` для чистого stdout

## Context and Problem Statement
При додаванні прапорця `--json` до `n-cursor fix` функція `ensureHkInstall` друкувала `Installed hk hook: …` безпосередньо в stdout, забруднюючи JSON-вивід і унеможливлюючи машинний парсинг результату.

## Considered Options
* Перенаправити вивід `ensureHkInstall` у stderr.
* Пропускати `ensureHkInstall` у json-режимі (`if (!json) ensureHkInstall(hkBin)`).

## Decision Outcome
Chosen option: "Пропускати `ensureHkInstall` у json-режимі", because `--json` — діагностичний (read-only) режим: він не мутує проєкт і не потребує git-хука для цього прогону; пропуск безпечніший і простіший за зміну потоку виводу бічної функції.

### Consequences
* Good, because transcript фіксує очікувану користь: smoke-тест `fix bun --json` повернув чистий парсабельний JSON у stdout без артефактів hk-інсталяції.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Змінений рядок у `npm/bin/n-cursor.js` у функції `runFixCommand`: `if (!json) ensureHkInstall(hkBin)`. Smoke: `bun ./npm/bin/n-cursor.js fix bun --json >/tmp/fj.out 2>/tmp/fj.err` → `node -e 'JSON.parse(...)'` успішно.
