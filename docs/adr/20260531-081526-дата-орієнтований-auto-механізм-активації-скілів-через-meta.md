## ADR Дата-орієнтований `auto`-механізм активації скілів через `meta.json`

## Context and Problem Statement
Потрібен спосіб визначати, які скіли автоматично потрапляють у `.n-cursor.json` під час `n-cursor sync`. Захардкодена мапи в коді CLI важко підтримувати, тому механізм виніс конфігурацію автоактивації безпосередньо в кожен скіл.

## Considered Options
* Поле `auto` у `npm/skills/<id>/meta.json` як єдине джерело правди
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "поле `auto` у `meta.json` кожного скіла", because кожен скіл сам декларує умову своєї автоактивації; `npm/scripts/auto-skills.mjs` читає ці файли і виконує логіку без hardcoded-мапи в CLI.

Підтримуються два режими поля `auto`:
- `"завжди"` — скіл активується завжди (наприклад, `fix/meta.json`: `{ "auto": "завжди", "worktree": true }`).
- Масив рядків — скіл активується лише якщо `.n-cursor.json` містить хоча б один із перелічених rule-id (наприклад, `adr-normalize/meta.json`: `{ "auto": ["adr"] }`, `taze/meta.json`: `{ "auto": ["bun"] }`).

### Consequences
* Good, because додавання нового скіла вимагає лише `meta.json` без зміни коду CLI — `detectAutoSkills` / `mergeConfigWithAutoDetected` у `npm/bin/n-cursor.js` читають метадані динамічно.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- `npm/scripts/lib/skill-meta.mjs` — спільний парсер полів `meta.json` (`auto`, `worktree`).
- `npm/bin/n-cursor.js` — функції `detectAutoRules`, `mergeConfigWithAutoDetected`.
