## ADR Розширення `withLock` для крос-worktree серіалізації важких команд

## Context and Problem Statement
Проєкт використовує кілька git-worktree одночасно. `withLock` зберігав стан локу в `node_modules/.cache/n-cursor/<key>/` — директорії, яка є **окремою для кожного worktree**. Через це паралельні `lint`/`fix`-команди, запущені з різних worktree, не бачили локів одне одного й запускались одночасно, перевантажуючи CPU/диск.

## Considered Options
* Перемістити `cacheDir` за замовчуванням під `git-common-dir` (спільний для всіх worktree).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Перемістити `cacheDir` під `git-common-dir`", because це єдине місце, яке однакове для головного checkout і всіх linked-worktree; наявний `withLock` підтримує `opts.cacheDir`, тож зміна мінімальна й не порушує API.

### Consequences
* Good, because `mkdirSync`-mutex тепер серіалізує важкі команди (`lint-*`, `fix-*`) між будь-якими worktree того самого репо, не лише в межах одного checkout.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Нові файли: `npm/scripts/utils/lock-cache-dir.mjs` (функція `resolveLockCacheDir(key)` — повертає `<git-common-dir>/n-cursor/<key>`, fallback на `node_modules/.cache` поза git-репо); `npm/scripts/utils/tests/lock-cache-dir.test.mjs` (5 тестів: відносний/абсолютний `--git-common-dir`, крос-worktree однаковість, обидва fallback-и). Змінено: `npm/scripts/utils/with-lock.mjs` рядок 66 — `const cacheDir = opts.cacheDir ?? resolveLockCacheDir(key)`. ## ADR `auto.md` → `meta.json` з полем `worktree` для скілів і правил

## Context and Problem Statement
Скіли (`npm/skills/<id>/`) і правила (`npm/rules/<id>/`) мають `auto.md` з умовою автоактивації, але цей формат — вільний рядок, без структури. Для нового поля `worktree` (ізоляція скіла в git-worktree) `auto.md` не підходить. Паралельно виявлена асиметрія: skills `auto.md` машинно-парситься, rules `auto.md` — лише документація. Потрібна уніфікація.

## Considered Options
* B1 `meta.json` — структурований JSON з JSON-схемою.
* B2 `meta.yaml` / B3 `.md` з YAML frontmatter.
* Інші варіанти в transcript не обговорювалися (крім B1–B3).

## Decision Outcome
Chosen option: "`meta.json` (B1, дві схеми E2)", because JSON легко валідується наявними `npm/schemas/`; дві окремі схеми (`skill-meta` / `rule-meta`) уникають умовної логіки і чітко закріплюють, що `worktree` — тільки скіловий концепт.

### Consequences
* Good, because transcript фіксує очікувану користь: кілька `worktree:true`-скілів можна запускати паралельно без взаємних перешкод — кожен в ізольованому checkout; `withLock` (git-common-dir) серіалізує тільки однаковий ключ, різні скіли не блокують одне одного.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Spec: `docs/superpowers/specs/2026-05-31-meta-json-skill-worktree-design.md`. Поле `worktree: boolean` (обовʼязкове для skills): `true` = генеративний скіл (виконується у git-worktree, заборонений паралельний запуск); `false` = реактивний або read-only. D2-sync: при `worktree:true` у копію `.cursor/skills/n-<id>/SKILL.md` вставляється ідемпотентний блок між маркерами `<!-- n-cursor:worktree:start/end -->`. Для rules `meta.json.auto` — рядок-проза (1:1 міграція); `auto-rules.mjs` не змінюється (Spec 2). `meta.json` у `.cursor/skills/` не копіюється.
