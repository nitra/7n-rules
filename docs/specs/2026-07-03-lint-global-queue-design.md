# Глобальна черга запусків `n-cursor lint`

- Дата: 2026-07-03
- Статус: реалізовано
- Контекст: кілька агентів (сесій) запускають `npx @nitra/cursor lint` одночасно і конфліктують

## Проблема

«Один інстанс за раз» існував лише як текст у CLAUDE.md / SKILL.md — технічного примусу не було.
Одночасні запуски в одному робочому дереві конфліктують реально:

- обидва пишуть у ті самі файли проєкту (T0-автофікси `eslint --fix`/`oxfmt`/`cspell --fix`,
  потім LLM-worker) — на спільному дельта-наборі vs origin перетин гарантований;
- snapshot/rollback живе в пам'яті кожного процесу: якщо запуск A відкочує файл, який щойно
  пофіксив запуск B, — B цього не бачить і репортить успіх;
- detect-only читає дерево, яке паралельний fix мутує → нестабільні результати;
- навіть диз'юнктні прогони конкурують за CPU/диск/локальну LLM.

## Рішення (зафіксовані в brainstorm)

| Вісь | Рішення |
| --- | --- |
| Скоуп лока | **Один глобальний на машину** (per-user tmpdir), без прив'язки до дерева/репо |
| Поведінка при зайнятому локу | **Черга**: poll до дедлайну (20 хв), далі **fail-closed** (Error, exit 1) |
| `--no-fix` / `--read-only` | **Теж у чергу** — одна черга для всіх варіантів lint |
| Лок мертвого процесу | Перехоплюється **одразу** (PID-перевірка), без очікування |
| `hook --post-tool-use` | Лок **не** бере: read-only, per-file, має відповідати миттєво |

## Реалізація

Перевикористано наявний `withLock` (`npm/scripts/utils/with-lock.mjs`: mkdir-лок, PID-живість,
poll-черга, TTL-дедуплікація за fingerprint) через тонку обгортку
`npm/scripts/lib/lint-surface/lint-lock.mjs` (`withGlobalLintLock`), підключену в диспатч
`case 'lint'` у `npm/bin/n-cursor.js` — одна точка входу накриває delta, `--full`, scoped і
`--no-fix`. Відмінності від per-rule використання (`run-standard-lint.mjs`):

- `cacheDir` у `os.tmpdir()/n-cursor/lint-global` замість `<git-common-dir>` → скоуп machine-wide,
  а не per-repo;
- fingerprint дедуплікації домішує варіант виклику (`--full`/`--no-fix`/rules/cwd) до знімка
  дерева — інакше scoped-успіх (`lint js`, exit 0) хибно пропускав би повний `lint` на тому ж
  дереві; при `--cwd` не на процесний cwd або поза git-репо fingerprint = null (дедуплікація
  вимкнена, черга працює);
- `staleThreshold` піднято до 6 год: дефолтні 30 хв «перехоплювали» б живий лок довгого
  `--full`-прогону з LLM-ladder і знову паралелили запуски; краші покриває PID-перевірка;
- `onWaitTimeout: 'fail'` замість дефолтного `run-unlocked` — мовчазний паралельний запуск
  це саме те, що черга має унеможливити.

UX черги (stderr): `⏳ lint-global: чекаю, лок тримає pid …` → `🔒 lint-global: лок взято`;
дедуплікація: `♻️ lint-global: те саме дерево, N с тому — пропускаю`.

## Верифікація

- Юніт-тести: `npm/scripts/lib/lint-surface/tests/lint-lock.test.mjs` (fingerprint-осі,
  fail-closed таймаут, перехоплення лока мертвого PID) + наявні `with-lock.test.mjs`.
- Живий прогін: лок утримано сторонім живим процесом → `lint` чекав у черзі (5 poll-повідомлень),
  після звільнення взяв лок і завершився з exit 0.
