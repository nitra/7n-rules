# Винесення мовних специфік (js/rust/python) у плагіни `@7n/rules-lang-*`

**Дата:** 2026-07-18
**Статус:** погоджено — готово до реалізації
**Зв'язані документи:** `npm/scripts/lib/resolve-plugins.mjs` (наявна плагінна інфраструктура ci-github/ci-azure), `npm/skills/taze/js/orchestrate.mjs` (оркестратор із трьома екосистемними гілками — перший споживач порту)

## 1. Проблема / Мета

Мовна специфіка (Rust, Python, частково JS) розповзлась по ядру `@7n/rules`: taze-гілки (`cargo-diff.mjs`, `uv-diff.mjs`, гілки оркестратора), lint-правила (`rules/rust/`, `rules/python/`), .mdc-рули для агентів, мовні секції SKILL.md, cspell-терміни. Кожна нова мова робить ядро важчим для ВСІХ проєктів — включно з чисто-JS, яким Rust/Python-логіка не потрібна ніколи.

Водночас у репо вже є відпрацьована плагінна модель для CI (`@7n/rules-ci-github`/`@7n/rules-ci-azure`): автодетект за файловими сигналами, автоінсталяція через `bun add -d`, мердж `rules/`-тек, capability-гейт концернів, маніфест `n-rules` у package.json з `handlers` як v1-API без споживачів. Досвід експлуатації CI-плагінів: «налаштував раз — далі живеш як у раю» — zero-touch після онбордингу.

Мета: перенести цю ж модель на мови — плагіни `@7n/rules-lang-rust`, `@7n/rules-lang-python` (згодом і `lang-js`), з трьома рівноцінними драйверами: схуднення ядра, незалежний реліз-цикл, розширюваність третіми (нова мова = зовнішній пакет без PR у ядро). Головний інваріант — зберегти zero-touch: жодних нових ручних кроків у щоденному циклі.

## 2. Ухвалені рішення

| # | Питання | Рішення |
|---|---|---|
| А | Гранулярність пакування | **Плагін = мова** (`@7n/rules-lang-rust`), всередині — конвенційні підтеки-поверхні (`rules/`, `taze/`, `doc-files/`, `mdc/`), які ядро сканує конвенційно. Відхилено «плагін = мова×поверхня» (вибух пакетів; користувач думає мовами, не поверхнями) і чисто-декларативний маніфест (семантика гілок надто різна — uv-цикл із best-effort відновленням у дані не влазить без escape-hatch, який стане нормою) |
| Б | Контракт розширення | **Кодовий порт, але ОДИН на старті**: інтерфейс `EcosystemProvider` (`detect/backup/bump/diff/promptFor/cleanup`) лише для taze. Наступні порти (doc-files, lint і далі) додаються, коли доїжджає їхня міграція — YAGNI, не проєктувати всі п'ять наперед. Реєстрація через наявний `n-rules.contributes.handlers` (taze — перший реальний споживач v1-API) + capability `lang:rust` для гейта концернів |
| В | Активація | **Успадкувати механізм CI-плагінів без змін**: файловий автодетект (`Cargo.toml` → lang-rust, `pyproject.toml` → lang-python) у `detectPluginsFromRepo`, явний `.n-rules.json.plugins` перекриває, автоінсталяція `ensurePluginInstalled` + `allowInstall:false` на hot-path, кеш на процес |
| Г | Шлях міграції | **Внутрішні плагіни спершу**: пакети живуть у цьому ж монорепо, інтерфейс обкатується на first-party мовах, публічний контракт для сторонніх — лише після стабілізації. Пілот — **uv-гілка taze** (наймолодша, найчистіша, найменше зрощена з ядром) |
| Д | Плагін-до-плагіна композиція | **Свідомо відкладено** (rust-плагін контриб'ютить clippy-джобу в ci-github тощо) — граф міжплагінних залежностей загрожує zero-touch-простоті; повернутись після стабілізації базового контракту |
| Е | Чисто-JS проєкти | Нульовий слід: мовний плагін не встановлено → у звітах taze/doc-files нема навіть згадки про мову (тиша, не «пропущено») |

## 3. Деталі реалізації

### Фаза 1 — порт taze + пілот lang-python (uv)

1. **Інтерфейс `EcosystemProvider`** у ядрі (`npm/scripts/lib/plugin-api/` або сусідній модуль до `resolve-plugins.mjs`):
   - `detect(cwd)` → знайдені маніфести (сьогоднішні `findCargoManifests`/`findPyprojectManifest`);
   - `backup(cwd, manifests, deps)` / `cleanup(cwd, manifests, deps)`;
   - `bump(cwd, manifests, spawnFn, log, deps)` (сьогоднішні `cargo upgrade`-виклики / `bumpUvDependencies`);
   - `diff(cwd, manifests)` → `{major, minorPatch, totalChanged}` (сьогоднішні `collectCargoDiff`/`collectUvDiff`);
   - `promptFor(entry)` → промпт одного major-запису (сьогоднішні `buildCargoDependencyPrompt`/`buildUvDependencyPrompt`);
   - `available(spawnFn)` → чи встановлений тулчейн (`hasCargoEdit`/`hasUv`) + `unavailableReason` для звіту;
   - метадані звіту: `title` («Rust-крейти» / «Python-пакети (uv)»), `manifestNoun`.
   - JSDoc-типи експортуються як `@7n/rules/plugin-api` — контракт для авторів.
2. **Оркестратор** (`orchestrate.mjs`): замінити хардкод-гілки на цикл по провайдерах — ядро отримує список через `getHandlers(root, config, 'taze')` + вбудовані first-party провайдери; npm/bun-гілка поки лишається вбудованою (виїде в lang-js фазою 5).
3. **Пакет `npm-plugins/lang-python/`** у монорепо: `package.json` з блоком `n-rules` (`capabilities: ["lang:python"]`, `contributes.handlers.taze: "./taze/provider.mjs"`), переїжджають `uv-diff.mjs` + uv-функції оркестратора + тести. Реліз-конвеєр — той самий, що вже релізить `@7n/llm-lib` поруч із `@7n/rules`.
4. **Автодетект**: розширити `KNOWN_CI_PLUGINS`-патерн до `KNOWN_LANG_PLUGINS` (`pyproject.toml` → `@7n/rules-lang-python`); `resolvePluginList` мерджить обидва набори.
5. **Верифікація фази**: реальний прогін `skill codex taze` на python-проєкті (`/Users/vitalii/www/efes/strawberry`) з установленим плагіном і — на чисто-JS репо — перевірка нульового сліду (рішення Е).

### Фаза 2 — lang-rust тим самим портом

`cargo-diff.mjs`, cargo-функції оркестратора, тести → `npm-plugins/lang-rust/`. Це перевірка, що порт справді загальний, а не «підігнаний під uv». Розбіжності інтерфейсу (мультиманіфестність Cargo vs один pyproject) мають вміститись без зміни сигнатур — інакше правимо порт ЗАРАЗ, поки він внутрішній.

### Фаза 3 — lint-правила й .mdc

- Теки `rules/rust/`, `rules/python/` переїжджають у плагіни як є — `resolveRulesDirs` уже вміє зливати правила плагінів, нова механіка не потрібна.
- .mdc-рули (`n-rust.mdc`, `n-python.mdc`) — у плагін; sync інлайнить їх з плагінних `rules/`-тек тим самим механізмом, що й ядрові.
- cspell: плагін шипить свій словник (`cspell-dict.txt` у конвенційній підтеці), ядро агрегує в `import`-ланцюг — зникає ручне поповнення `.cspell.json` мовними термінами.
- Концерни цих правил гейтяться `requires.capability: lang:rust` — на чисто-JS репо вони навіть не завантажуються.

### Фаза 4 — doc-files і SKILL-фрагменти (окремий порт, коли дійде черга)

- doc-files: мовні екстрактори (мапінг розширень, парсинг rustdoc/docstring для майбутнього comment-hybrid) — другий порт `contributes.handlers['doc-files']`; конвеєр/CRC/LLM — ядро.
- SKILL.md: мовні секції (Rust-гілка, Python-гілка) генеруються композицією фрагментів із плагінів — знання їде разом з кодом, не сиротіє в ядрі.
- Плагін шипить власні doc-files-доки для свого коду (той самий CRC-конвеєр).

### Фаза 5 — lang-js: ядро стає чистим двигуном

Після стабілізації порту на rust/python — винести і npm/bun-гілку в `@7n/rules-lang-js`. Свідомо остання, бо: (а) виграшу в схудненні нема — плагін активний у 100% користувачів (кожне репо з `npx @7n/rules` має package.json); (б) bootstrap-циркулярність — ядро розробляється й лінтується собою, залежність `@7n/rules` ← `@7n/rules-lang-js` ← plugin-api ядра потребує окремого розв'язання в CI/локальній розробці; (в) це найстарша й найзрощеніша гілка (воркспейси, `bun.lock`, `n-rules taze diff`, changelog-механіка) — порт, обкатаний на двох «чужих» екосистемах, доведе загальність, а не вийде «JS-shaped». Мотив фази — симетрія контракту (lang-js як еталонна реалізація для сторонніх) і незалежний реліз-цикл JS-правил. Ціль кінцевого стану: ядро = двигун без жодної мовної специфіки; lang-js ставиться автодетектом за package.json, як решта мов.

### Наскрізні вимоги (всі фази)

- Ядро ніколи не імпортує плагін статично — лише динамічний `import()` за маніфестом; відсутність плагіна ≠ помилка.
- `pluginApi`-версія: ядро декларує ціле число, плагін — `requiresPluginApi`; несумісність → зрозумілий skip зі згадкою в звіті, не креш.
- Повідомлення порушень від плагінних правил містять ім'я плагіна-джерела (діагностика хибних спрацювань).
- CI ядра ганяє smoke-тести всіх first-party плагінів на кожен PR (монорепо — дешево).
- Зміна сигнатури порту = major bump `pluginApi`; стереже rule_meta-чек.

### Success signals

- Пілот (фаза 1): `skill codex taze` на strawberry-репо дає той самий звіт, що й до винесення; на чисто-JS репо — нуль згадок Python і нуль нових залежностей.
- Фаза 2: lang-rust сів у порт без зміни сигнатур.
- Фаза 5: у ядрі не лишилось жодного мовно-специфічного модуля; всі мовні прогони йдуть через провайдерів.
- Кінцевий: додати підтримку нової мови = створити пакет за шаблоном, без жодного PR у ядро.

## Відкриті питання

- Механізм розв'язання bootstrap-циркулярності lang-js (симлінк у монорепо / workspace-протокол / вендорений fallback) — вирішується на вході у фазу 5, з досвідом фаз 1-4.
- Публічний контракт для сторонніх (скаффолд `plugin init`, тестбед, курований реєстр) — свідомо ПІСЛЯ стабілізації внутрішнього порту; окрема сесія.
- Плагін-до-плагіна композиція (рішення Д) — повернутись після фази 3.
- Доля `github-actions/`-шаблонів з мовними кроками — залежить від рішення по композиції.
