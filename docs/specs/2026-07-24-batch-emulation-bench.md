# Бенч-калібрування дефолтів `llm_lib::batch` (Тип 2b — емуляція, задача T6)

**Дата:** 2026-07-24
**Пов'язано:** `docs/specs/2026-07-23-llm-cascade-single-source-spec.md` (рішення Р, задача T6),
`llm-lib/crates/llm-lib/src/batch.rs`, `llm-lib/crates/llm-lib/examples/batch_bench.rs`

## Методика

Приклад `cargo run --release -p llm-lib --example batch_bench` ганяє `llm_lib::batch::submit`
проти реального локального omlx (`http://127.0.0.1:8000/v1/`, модель — `N_LOCAL_MIN_MODEL`,
у прогоні — `omlx/gemma-4-e4b-it-OptiQ-4bit`) для 48 коротких промптів (по одному слову у
відповіді) — усі 9 комбінацій `chunk_size ∈ {5, 15, 35}` × `concurrency ∈ {1, 2, 4}`.
Кожна комбінація — окремий повний прогін 48 items з нуля (без розігріву кешу між
комбінаціями, порядок комбінацій — як у таблиці нижче).

Локальний сервер на момент бенчу мав `scheduler.max_concurrent_requests: 1` у
`~/.omlx/settings.json` — це очікувано обмежує ефект клієнтської конкурентності
(генерація на сервері серіалізована незалежно від того, скільки запитів клієнт тримає
"в польоті").

## Результати

| chunk_size | concurrency | час, с | ok | err |
|---|---|---|---|---|
| 5 | 1 | 29.80 | 48 | 0 |
| 5 | 2 | 17.93 | 48 | 0 |
| 5 | 4 | 18.97 | 48 | 0 |
| 15 | 1 | 18.19 | 48 | 0 |
| 15 | 2 | 17.90 | 48 | 0 |
| 15 | 4 | 18.34 | 48 | 0 |
| 35 | 1 | 18.16 | 48 | 0 |
| **35** | **2** | **102.15** | 48 | 0 |
| 35 | 4 | 19.20 | 48 | 0 |

Усі 9 прогонів завершились без жодної помилки (0 err) — сама емуляція batch (чанкування,
конкурентний прогін через 2a, `submit → progress → results`) працює коректно.

## Інтерпретація

- Крім однієї аномалії (`chunk_size=35, concurrency=2` — 102 с проти сусідніх ~18 с),
  час майже не залежить від `chunk_size`/`concurrency`: `scheduler.max_concurrent_requests: 1`
  на локальному сервері серіалізує фактичну генерацію незалежно від клієнтських налаштувань —
  клієнтська конкурентність тут лише формує чергу, не пришвидшує сам inference.
- `chunk_size=5, concurrency=1` — помітно повільніший (29.8 с) — очікувано: єдина
  комбінація, де і чанк, і конкурентність мінімальні, тобто найменше перекриття
  мережевого/чергового оверхеду з генерацією.
- Аномалія `35×2` (102 с) не відтворюється системно в жодній сусідній клітинці (ні `35×1`,
  ні `35×4`, ні `15×2`) — найімовірніше шумовий викид (фонове навантаження машини, теплова
  дроселізація, разова затримка ollx-кешу), а не системна деградація конкретно цієї
  комбінації. Зафіксовано чесно замість замовчування; не використано для вибору дефолтів.
- Головний практичний висновок бенчу: **для одиночного локального omlx-сервера з
  `max_concurrent_requests: 1` вибір `chunk_size`/`concurrency` майже не впливає на
  сумарний час** — вузьке місце не в клієнтській емуляції batch, а в сервері. Значення
  цих лімітів стають вагомими для конфігурацій із реальною серверною паралельністю
  (кластер omlx із `max_concurrent_requests > 1`, чи хмарний провайдер типу 2a) —
  саме на це й розраховані ліміти в `BatchConfig`, не лише на цей одиночний сервер.

## Обрані дефолти

`BatchConfig::default()` (`llm-lib/crates/llm-lib/src/batch.rs`):

```rust
chunk_size: 35,   // старт зі спеки (рішення Р)
concurrency: 2,   // помірна — бенч не показав виграшу від вищої конкурентності
                   // на локальному однопоточному сервері, і не показав системної
                   // шкоди від concurrency=2 (окрім разового шумового викиду)
```

Дефолти лишені такими, як у спеці ("старт: чанк ≤35"), — бенч не дав підстав ні
зменшити `chunk_size` (немає деградації при 35 — на відміну від емпіричного ліміту
`mlmail`, ~35 items, що був виведений конкретно для його клієнтського чанкінгу без
локального контролю конкурентності), ні підняти `concurrency` вище 2 (немає виграшу
на цьому сервері; вища конкурентність на реальному сервері з паралельним
обслуговуванням запитів чи на хмарному провайдері — окреме калібрування поза
обсягом цього бенчу).

## Відтворення

```bash
N_LOCAL_MIN_MODEL=omlx/<модель-з-`omlx list`> \
OMLX_API_KEY=<ключ із ~/.omlx/settings.json, auth.api_key> \
cargo run --release -p llm-lib --example batch_bench
```
