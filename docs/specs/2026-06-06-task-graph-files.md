# Task Graph — специфікація форматів файлів

Дата: 2026-06-06
Статус: Draft
Пов'язані ADR: `рекурсивний-складений-ОАГ-динамічний-розклад.md`, `файловий-стан-append-only-план-факт.md`, `ворктрі-межа-атомарності.md`

## Мета

Визначити точні формати файлів для системи динамічного самомодифікованого графу задач. Цей контракт є основою для observability-скрипту та wrapper-скрипту агентів.

---

## Загальні правила

**Формат:** Markdown з YAML-фронтматером.

**Мова:**
- Атрибути фронтматеру → англійські, snake_case
- Заголовки секцій що парсить скрипт → англійські
- Секції з довільними даними → будь-яка мова

**Append-only інваріант:** файли тільки створюються, ніколи не змінюються після створення ворктрі.

**Стан вузла через наявність файлів:**

| Які файли існують | Стан |
|---|---|
| тільки `task.md` (+ опц. `вхідні.md`) | `waiting` |
| є активний ворктрі | `running` |
| є `вихідні.md` | `resolved` |
| є `помилка.md` | `failed` |
| є `invalidated` | `invalidated` |
| є `вихідні.md` + `invalidated` | `resolved` (`вихідні.md` має пріоритет) |

**Синтаксис посилань:**
```
ref: tasks/collect-data/вихідні.md              # весь файл
ref: tasks/collect-data/вихідні.md#results      # секція за заголовком
```
Шляхи відносно кореня проекту. Можна посилатись на будь-які файли проекту, не лише в `tasks/`.

## Файлова структура вузла

`id` вузла = назва директорії (не дублюється у фронтматері). Дочірні вузли живуть безпосередньо в директорії батька — якщо директорія містить `task.md`, це вузол.

```
tasks/
  <node-id>/
    task.md              ← завдання (незмінний після створення ворктрі)
    вхідні.md            ← посилання на дані (незмінний, опціональний)
    вихідні.md           ← факт успіху (v1)
    вихідні-2.md         ← нова версія після патчу (v2, опціональний)
    помилка.md           ← факт збою (незмінний)
    invalidated          ← sentinel: порожній файл
    операції/
      spawn-план.md
      spawn-факт.md
      spawn-2-план.md    ← динамічне розширення
      spawn-2-факт.md
      kill-план.md
      kill-факт.md
    патчі/
      001-план.md
      001-факт.md
      002-план.md
      002-факт.md
    repair/
      001.md
      002.md
    <child-node-id>/     ← дочірній вузол (якщо складений)
      task.md
      ...
```

**Версіонування виходів:** `вихідні.md` (v1), `вихідні-2.md` (v2) — кожен незмінний. Resolver завжди бере файл з максимальним `version:`. `ref:` завжди вказує на `вихідні.md` без версії.

## `task.md`

```markdown
---
parent: research/collect-data   # відносно tasks/; відсутній у кореневого вузла
deps:                            # відсутній якщо немає попередників
  - collect-data
  - fetch-sources
---
## Task
Що саме має виконати цей вузол.

## Done when
Чіткий критерій: що означає "вирішено".

## Контекст
(опціональна секція — довільна мова)
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `parent` | ні | відносно `tasks/`; вказує на директорію вузла-батька |
| `deps` | ні | id siblings у тій самій директорії |
| `## Task` | так | англійська; читається observability-скриптом |
| `## Done when` | так | англійська |

Незмінний після створення ворктрі. Для зміни — видалити директорію і re-spawn.

Кореневий вузол без `вхідні.md` → статус `waiting-for-input` → людина створює `вхідні.md` вручну.

---

## `вхідні.md`

```markdown
---
created_at: 2026-06-06T10:00:00Z
---
## context
ref: tasks/collect-data/вихідні.md#results
ref: docs/specs/requirements.md

## raw-data
ref: tasks/fetch-sources/вихідні.md
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `created_at` | так | ISO 8601 |
| секції | ні | довільні назви; кілька `ref:` в одній секції — дозволено |

Опціональний файл — відсутній якщо немає `deps` і батько нічого не передає.

---

## `вихідні.md`

```markdown
---
created_at: 2026-06-06T10:05:00Z
version: 1
---
## results
ref: tasks/collect-data/вихідні.md#raw

## summary
(синтез агента — опціонально)
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `created_at` | так | ISO 8601 |
| `version` | так | починається з 1; нова версія = новий файл `вихідні-2.md` |
| секції | ні | довільні — посилання або дані на розсуд агента |

Для складеного вузла: `ref:` на виходи exit-вузлів (основний варіант), опціонально доповнений синтезом агента.

## `помилка.md`

```markdown
---
occurred_at: 2026-06-06T10:05:00Z
type: execution-error | timeout | unresolvable
author: agent | wrapper | engineer
---
## What happened
Що конкретно пішло не так.

## Last action
Що агент робив у момент збою.

## Trace
(технічний трейс — опціонально)
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `occurred_at` | так | ISO 8601 |
| `type` | так | `execution-error` \| `timeout` \| `unresolvable` |
| `author` | так | `agent` \| `wrapper` \| `engineer` |

Незмінний — один на першу невдачу. Наступні невдачі при repair-ітераціях → у `repair/`.

---

## `repair/001.md`

```markdown
---
attempt: 1
started_at: 2026-06-06T10:01:00Z
deadline: 2026-06-06T10:10:00Z
patch: патчі/001-факт.md       # відсутній якщо патчу не було
result: failed | succeeded | escalated
---
## Reasoning
Чому обрана ця стратегія, що спостерігав.

## What was tried
Опис підходу — що зроблено.

## Outcome
Що сталося після retry.
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `attempt` | так | порядковий номер |
| `started_at` | так | ISO 8601 |
| `deadline` | так | time budget дедлайн; спільний для кількох спроб одного рівня |
| `patch` | ні | ref на `патчі/NNN-факт.md` якщо патч застосовано |
| `result` | так | `escalated` = передає батьку |

Інженер створює файл і пише `result` в кінці своєї роботи. `deadline` ≤ `now()` → `result: escalated`.

## `операції/spawn-план.md` / `spawn-факт.md`

```markdown
---
created_at: 2026-06-06T10:00:00Z
nodes:
  - id: collect-data
    deps: []
  - id: analyze
    deps: [collect-data]
  - id: synthesize
    deps: [analyze]
---
## Why decomposed
Чому агент вирішив розбити на підграф.
```

```markdown
---
created_at: 2026-06-06T10:00:05Z
nodes_created:
  - collect-data
  - analyze
  - synthesize
---
```

Recovery: `nodes` у плані ≠ `nodes_created` у факті → spawn не завершився → продовжити.

Динамічне розширення → `spawn-2-план.md` / `spawn-2-факт.md`, нумерація зростає.

---

## `операції/kill-план.md` / `kill-факт.md`

Файли живуть у директорії **вузла-цілі патчу** (не вузлів що вбиваються).

```markdown
---
created_at: 2026-06-06T10:00:00Z
kill_order:
  - synthesize
  - analyze
reason: patch target collect-data
---
```

```markdown
---
created_at: 2026-06-06T10:00:30Z
nodes_killed:
  - synthesize
  - analyze
---
```

`kill_order` ≠ `nodes_killed` → kill не завершився → добити решту.

Kill = terminate process → delete worktree (послідовно).

---

## `патчі/001-план.md` / `001-факт.md`

```markdown
---
created_at: 2026-06-06T10:00:00Z
target_node: collect-data
---
## Reason
Чому потрібен патч.

## Changes
Що буде змінено.
```

```markdown
---
created_at: 2026-06-06T10:02:00Z
result: success | failure
---
## Applied changes
Що фактично змінено.
```

## Протокол патчу залежного вузла

```
1. записати патчі/NNN-план.md у collect-data   ← спочатку намір
2. знайти всі запущені ворктрі залежних
3. kill від листів до цілі (топологічний порядок)
4. записати операції/kill-факт.md
5. застосувати патч
6. записати патчі/NNN-факт.md
7. рестарт каскаду
```

Наявність `патчі/NNN-план.md` без `патчі/NNN-факт.md` = незавершений патч → продовжити з кроку 3.

## Observability

Скрипт відновлює стан графу скануванням:

```
для кожного tasks/**/task.md:
  стан       ← які файли існують поруч
  залежності ← поле deps: з task.md
  незавершені операції ← *-план.md без *-факт.md
  repair/    ← остання спроба і дедлайн

вивести:
  - дерево вузлів зі станами
  - незавершені операції
  - активні ворктрі
  - часовий бюджет інженерів що активні
```
