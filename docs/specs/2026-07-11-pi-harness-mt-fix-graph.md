# pi-harness у @nitra/llm-lib + fix-граф MT: автономне виправлення lint-хвоста

**Дата:** 2026-07-11
**Статус:** погоджено — готово до dev-design (деталізація Фази A — окрема задача)
**Зв'язані документи:** `docs/specs/2026-07-05-llm-lib-extraction-spec.md`; pi-migration-spec (памʼять сесій, Ф1 змерджено `b167ebcf`); архітектура MT 0.3.0-draft — `/Users/vitalii/www/nitra/mt/npm/docs/architecture/` (`graph.md`, `overview.md`), roadmap M0–M5

## 1. Проблема / Мета

`npx @nitra/cursor lint` у fix-режимі вже має драбину local-4B → cloud (gpt-5.4-mini → gpt-5.5) на pi-рушії (Ф1 pi-міграції), але **складні правила (doc-files/js-run/npm-module/test) не долітають**: невиправлений хвіст доводиться вручну переносити в Claude Code. Мета — нуль ручного перенесення: агентичний хвіст закривається самостійно, а де потрібна людина — це формалізована задача з повним контекстом спроб, а не копіпаста.

Рішення спирається на два шари:

- **@nitra/llm-lib** — не «просто виклики моделі», а pi-harness: якісне виконання **одного run** (evidence-гейт, надійні edits, web-доступ), спільне для n-cursor і 7n-test.
- **MT** (`@7n/mt`, платформа задач для людей і ШІ) — оркестрація **між** задачами: порушення стають вузлами графа з машинним контрактом (`## Done when` + `## Check`), retry ladder → EngineerAgent → людина, audit, бюджети, worktree-ізоляція, fenced publish.

Обмеження: локальні моделі omlx/gemma — тир-канон для тривіального; `lint --read-only` (CI) — нуль pi/MT-викликів (тверда межа з pi-migration-spec); MT зараз на M0 (headless dogfood), `mt attach` (інтерактивні сесії) — M1.

Контекст добору можливостей: проаналізовано топ-40 розширень екосистеми pi (awesome-pi.site) — найсильніші важелі, відсутні в поточному рушії: компресія контексту (pi-hypa, pi-lean-ctx), hash-anchored edits (pi-hashline-edit-pro, pi-readseek), goal/verify-loop (pi-codex-goal, gentle-pi evidence), web-access (pi-web-access), advisor-ескалація (rpiv-advisor), субагенти з typed API (@gotgenes/pi-subagents).

## 3. Деталі реалізації

### 3.1 Фаза A — run-harness у @nitra/llm-lib

Надбудова над наявним pi-шаром (pi-one-shot / pi-agent-fix уже в екосистемі; wiring: `ModelRegistry.create(AuthStorage.create())` → `createAgentSession`, конфіг pi перевикористовується):

- `harness.run(task)` — єдиний вхід з профілями (one-shot / agentic), який споживають і inline-драбина n-cursor, і майбутній MT-runner. Профіль = декларативний конфіг: tools, model-tier, budget, verify-команда.
- **Evidence-гейт:** tool `report_done` приймається лише з прикладеним виводом verify-команди (check-{id}); провал verify інʼєктиться фідбеком у ту саму сесію (внутрішній loop, ліміт ітерацій), а не одразу новий rung. Зовнішній re-check лишається джерелом правди (locked у pi-migration-spec).
- **Hash-anchored edit tools:** заміна built-in read/edit на строгі line-anchored операції без мовчазної автокорекції — проти collateral слабких моделей (прецеденти: літеральний `\n\n`, переписаний `llm.mjs` у 7n-test). Джерело патерну — pi-hashline-edit-pro (MIT, peer `@earendil-works/pi-coding-agent ≥0.74` — сумісно з pin 0.80.2); рішення vendored-код vs залежність — на dev-design після security-ревʼю.
- **Web-access tool** (лише cloud-профілі): SDK-ядро патерну pi-web-access, провайдери за наявними ключами; поверхня довіри обмежується whitelist-провайдерами.
- Телеметрія run (turns, tool-calls, cost, oldText→newText, тригер-сигнатура) — формат сумісний і з `~/.n-cursor/telemetry/` (поточний стор), і з майбутнім `fact_NNN.md` (§3.3).

### 3.2 Фаза B — адаптер lint→MT у @nitra/cursor

- Хвіст fix-прогону (порушення, які інлайн-щаблі не закрили) → `mt init` composite-вузла fix-batch; Stage 1 групує дітей за правило × кластер файлів; складні правила (doc-files) — вузол на файл, «мікро-вузли» з розжованим контекстом у `## Inputs`.
- Контракт дитини: `## Done when` = «check-{id} чистий на target-файлах»; `## Check` = shell-рядок `npx @nitra/cursor lint --rule <id> --files …` — машинний гейт MT з коробки.
- Тири → `a.md`: `model_tier` MIM/AVG/MAX = наявний мапінг local/cloud-min/cloud-avg; `retry_ladder` per-node (1 базова → 2 diagnose-first → 3 `model_tier: +1`). `run-summary.md` (LLM-аналіз патернів невдач) заміняє ідею «rung-адаптивного промпту».
- Ескалація: `failed_streak ≥ agent_retry_max` → EngineerAgent (повна історія ранів) — це формальна заміна «вставити в Claude Code»; далі `unresolvable.md` + `h.md` (кейс Е).
- Вузли зʼявляються в дереві лише штатними fenced-комітами MT (ніколи не лишаються untracked під час активного lint-прогону) — помʼякшення ризику брудного дерева при рішенні Ж.
- `lint --read-only` не торкається ні pi, ні MT (межа ціла); гібрид В означає: MT-шлях вмикається лише у fix-режимі і лише для хвоста.

### 3.3 Фаза C — телеметрія в графі та маховик

- `fact_NNN.md` fix-вузла несе дистиляційні дані (oldText→newText, сигнатура, occurrences) — дистиляційний читач Ф3-маховика (n-llm-patch → fix-*.mjs у @nitra/cursor, human-review обовʼязковий) ходить по графу; `cost_usd` з `run_NNN.md` дає звіт «скільки коштує правило» для пріоритезації дистиляції.
- Collateral-контроль повного циклу: `audit: required` на fix-вузлах з широким diff — аудитор = окрема модель (верифікатор-субагент безкоштовно зі штатного MT-audit).
- Ретро-цикл M5 поверх fix-графа — другий маховик (пропозиції «правило X фіксилось через Y; Z дешевше»), поза скоупом цієї спеки.

### Послідовність

Фаза A (llm-lib) не залежить від MT і дає ефект одразу (інлайн-драбина стає сильнішою) → Фаза B (адаптер, потребує M0-стабільності `@7n/mt`) → Фаза C (телеметрія в графі; до того — поточний `~/.n-cursor/telemetry/`). Dev-design Фази A — наступний окремий крок.

## Відкриті питання

- Пін `@7n/mt`: контракт M0 «зміцнюється» — стратегія сумісності адаптера (schema_version fail-closed уже в MT).
- Конвергенція журналів: pi SessionManager → `session.jsonl` у run ref MT (один журнал замість двох) — ідея на M1+.
- Rate-limit авто-створення вузлів per прогін (захист від graph-спаму при сотнях порушень) — поріг і поведінка при переповненні.
- Міграція телеметрії: `~/.n-cursor/telemetry/` ↔ графова (Фаза C) — міст чи паралельне життя до повного переходу.
- Web-access: перелік довірених провайдерів і зберігання ключів (pi auth.json vs env).
- Vendored vs dependency для hash-edit ядра (після security-ревʼю pi-hashline-edit-pro).
