# Прогрес-бар для `npx @nitra/cursor lint` (гібрид: концерни + порушення)

**Дата:** 2026-07-03
**Статус:** чернетка (після brainstorm-дискусії)
**Стосується:** `npm/scripts/lib/lint-surface/run-fix.mjs`, `run-detectors.mjs`, `npm/rules/doc-files/docgen-files-batch/main.mjs`

## Проблема

Довгий прогін `lint` (особливо `--full` чи `lint doc-files`) виглядає як «завис»: append-only
рядки (`✅ T0: …`, `⚡ haiku: …`) з'являються нерегулярно, загальної картини «скільки знайдено /
скільки вже виправлено / скільки лишилось» немає ні під час прогону, ні в моменті. `doc-files`
має власний примітивний `[done/total]` лічильник, не уніфікований з рештою.

## Ключове обмеження: total порушень не монотонний

Класичний бар «X/Y %» по **порушеннях** неможливий чесно:

- standalone-концерни (§8 spec `2026-07-02-text-check-per-file-split-design.md`) пропускають
  початковий detect — їхні порушення стають відомі лише після post-T0 re-detect;
- фікс одного тула може **відкрити** нові порушення іншого (маскування) — re-detect
  всередині ladder-циклу може повернути *більше*, ніж було.

Тому одиниця прогресу — **гібрид** (рішення дискусії): монотонний бар по **концернах**
(total відомий одразу з плану детекції), а лічильники **порушень** — як живий тикер поряд.

## Рішення

### Формат (TTY)

Один рядок унизу виводу, перемальовується:

```text
[██████████░░░░░░░░░░] 5/12 концернів · порушення: 47 знайдено / 32 виправлено · js/eslint (haiku)
```

- бар і `5/12` — закриті/усього концернів з плану (standalone-концерни входять у total);
- `знайдено / виправлено` — кумулятивні лічильники порушень (семантика нижче);
- суфікс — поточний концерн і активний ступінь (`T0`, `local`, `haiku`, `sonnet`).

### Рендер: зовнішня бібліотека `cli-progress` (рішення дискусії)

- `MultiBar`/`SingleBar` з кастомним `format`; існуючі рядки (`✅ T0: …`, `⚡ …`) друкуються
  через `multibar.log()` — інакше вони рвуть перемальовування бара.
- **Не-TTY (CI, hooks, пайпи) — авто-фолбек** (рішення дискусії): `process.stdout.isTTY !== true`
  → бар не створюється взагалі, лишаються поточні append-рядки БЕЗ змін + один додатковий
  append-рядок зі зведенням лічильників на кожне закриття концерну
  (`⏱ 5/12 · viol 47/32`). Жодного ANSI у CI-логах.
- `--verbose` не змінюється: verbose-рядки йдуть через той самий `log`/`multibar.log()`.
- Нова runtime-залежність `cli-progress` у `npm/package.json#dependencies` — свідоме рішення
  дискусії (рендер не пишемо самі).

### Внутрішня проводка: тонкий `ProgressReporter`

Зовнішня lib закриває лише рендер; лічильники крізь пайплайн веде новий модуль
`npm/scripts/lib/lint-surface/progress.mjs`:

```js
/**
 * createProgressReporter({ total, log, isTTY }) → {
 *   concernStart(ruleId, concernId, tier?),   // оновити суфікс
 *   detectSnapshot(concernKey, count),        // після кожного detect/re-detect
 *   concernClosed(concernKey),                // done++
 *   summary(),                                // { concernsDone, concernsTotal, found, fixed }
 *   stop()                                    // фінальний render + звільнити TTY-рядок
 * }
 * ```
 */
```

**Семантика лічильників порушень** (пер-концерн, агрегується в тикер):

- `found_c` — кумулятивний максимум відомих порушень концерну:
  `found_c = max(found_c, remaining_c + fixed_c)`. Якщо re-detect показав *більше*, ніж
  очікувалось (маскування, standalone) — `found` росте; лічильник ніколи не бреше «вниз».
- `remaining_c` — останній detect-знімок.
- `fixed_c = found_c − remaining_c` (не менше 0). Закритий концерн → `remaining_c = 0`.
- Тикер: `знайдено = Σ found_c`, `виправлено = Σ fixed_c`.

Дані вже протікають крізь код: кожен `detectOne`/re-detect у ladder-циклі повертає
`violations.length` — треба лише смикнути `detectSnapshot()` у 3-4 точках `run-fix.mjs`
(initial detect, post-T0 re-detect, post-tier re-detect, standalone post-apply detect).
Реporter **інжектиться** через `opts` (як `log`) — юніт-тести семантики лічильників не
залежать від `cli-progress` і TTY.

### `detect`-only (`--no-fix`) режим

`run-detectors.mjs` отримує спрощений варіант: бар по концернах плану
(`5/12 · знайдено 47`), без «виправлено». Той самий reporter, викликів менше.

### doc-files — у цій же спеці (рішення дискусії)

`docgen-files-batch/main.mjs` мігрує свій `[done/total]` на спільний reporter:

- одиниця = **файл** (концерн один, тому бар по файлах, не по концернах):
  `total` = кількість stale-файлів після скану, `done` = згенеровано (ok + degraded);
- суфікс — поточний `sourcePath` і розмір;
- фінальний рядок `✓ OK … ⚠ degraded … ✗ Err …` лишається (це підсумок, не прогрес);
- не-TTY: поточні `[done/total]`-рядки лишаються як фолбек — вони вже append-only.

Reporter для цього отримує опційний override одиниці (`unitLabel: 'файлів'`), без
окремого коду рендера.

## Не в обсязі

- JSON-стрім подій для зовнішніх оркестраторів (окрема майбутня спека; reporter-семантика
  сумісна — `summary()` вже структурований).
- Прогрес усередині одного концерну (скільки файлів обробив eslint) — зовнішні тули
  не звітують гранулярно.
- ETA — тривалість tier-кроків надто нестабільна (локальна LLM 90+ с/крок), оцінка брехатиме.

## Тести

- Юніт: семантика `found/fixed` (маскування: re-detect більший за очікуваний → `found` росте;
  standalone: перший знімок після apply; закриття концерну → `remaining=0`).
- Юніт: не-TTY фолбек емить append-рядки, не ANSI.
- Інтеграційний smoke: `runFixPipeline` з інжектнутим reporter-стабом отримує виклики
  у правильному порядку (start → snapshot → closed).

## План упровадження

1. `progress.mjs` (reporter + семантика лічильників) + юніт-тести.
2. Вплетення в `run-fix.mjs` (4 точки) і `run-detectors.mjs` (1 точка); `cli-progress`
   у deps; TTY/не-TTY розгалуження.
3. Міграція `docgen-files-batch` на reporter.
4. Change-файл (`bump: minor`, `section: Added`).
