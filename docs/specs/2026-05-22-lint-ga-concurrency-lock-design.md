# Лок паралельних запусків — пілот на `lint-ga` + розгортання на всі важкі команди

**Дата:** 2026-05-22 (оновлено 2026-05-23 під актуальні конвенції; розгортання — 2026-05-23)
**Статус:** реалізовано й розгорнуто на всі важкі команди — `lint-ga` (c4b9ea3), `lint-rego` / `lint-text` / `lint-k8s` / `lint-docker` + per-rule `fix-<id>` через `runStandardRule` (1.13.85). Канон зафіксовано у [`.cursor/rules/scripts.mdc`](../../../.cursor/rules/scripts.mdc) v1.9.

## Проблема

У проєкті паралельно працює кілька агентів (субагенти, окремі сесії Claude Code /
Cursor). Кожен самостійно запускає важкі перевірки — `bun run lint`,
`npx @nitra/cursor fix` тощо. У результаті одночасно крутиться N копій одного й того
самого ланцюжка лінту: кожна — окремий процес `eslint` + `oxlint` + `jscpd` + інші
інструменти. Диск і CPU на macOS перевантажуються, прогони сповільнюються або лагають.

Текстові інструкції в `.cursor/rules/` і скілі `n-lint` («не запускай лінт паралельно»)
не допомагають: оркестратор однаково розпаралелює роботу, і кожен дочірній агент сам
запускає перевірку. Потрібен **програмний механізм**, а не прохання.

Існуючий хук `normalize-decisions.sh` уже намагався блокувати через `flock`, але на
macOS `flock` відсутній — хук це визнає і просто пропускає блокування. Тобто реального
крос-процесного локу на macOS зараз нема.

## Обмеження

- Цільова платформа — **тільки macOS**.
- Рантайм — **bun** (Node-сумісний API доступний).
- `flock` недоступний. `/usr/bin/shlock` є, але дедупу не вміє.

## Рішення (затверджені параметри)

| Параметр                     | Рішення                                                                       |
| ---------------------------- | ----------------------------------------------------------------------------- |
| Поведінка при зайнятому локу | **Чекати** звільнення, потім **дедуплікувати**                                |
| Гранулярність                | **Лок на команду** (різні перевірки не блокують одна одну)                    |
| Чинність кешу для дедупу     | **Хеш робочого дерева + TTL** (обидві умови)                                  |
| Механізм                     | Лок **вбудований у саму перевірку** — без окремої команди-обгортки й без hook |
| Обсяг пілота                 | **Тільки `n-cursor lint-ga`**; решту розкочуємо за тим самим зразком          |

Окрема команда-обгортка відхилена: її легко обійти прямим викликом, і скіл/агент мусив
би пам'ятати про неї. Лок має бути інтринсивним — `n-cursor lint-ga` сам бере лок,
незалежно від того, викликаний він прямо, через `bun run lint-ga` чи в ланцюжку
`bun run lint`.

Примітив локу — **атомарний `mkdirSync`** (на APFS створення директорії атомарне: рівно
один процес виграє, решта отримують `EEXIST`). Без зовнішніх залежностей; дедуп-стан
природно лягає в той самий код. Альтернативи `shlock` і npm-пакет `proper-lockfile`
відхилені — додають залежність і не вміють дедупу.

## Архітектура

Три нові одиниці + одна точкова інтеграція.

### 1. `npm/scripts/utils/worktree-fingerprint.mjs`

Чистий хелпер: рахує детермінований відбиток стану робочого дерева.

- **Що робить:** повертає `sha256` від конкатенації:
  1. `git rev-parse HEAD`;
  2. `git diff HEAD` (усі tracked-зміни — staged + unstaged);
  3. для кожного untracked-файла з `git ls-files --others --exclude-standard` —
     його шлях + результат `git hash-object <file>`.
- **Як використовувати:** `computeFingerprint(cwd) → string | null`.
- **Залежить від:** `git` у `PATH`, `node:child_process`, `node:crypto`.
- **Межовий випадок:** не git-репозиторій (або `git` недоступний) → повертає `null`.
  Тоді дедуп вимкнено, але лок (взаємне виключення) працює далі.

Відбиток береться по **всьому дереву**: цільовий сценарій — «N субагентів лінтять
однаковий стан репо» — дає ідентичний відбиток у всіх, тобто чистий дедуп. Звуження
скоупу відбитка (напр. лише `.github/**`) можливе пізніше, окремо per-command.

### 2. `npm/scripts/utils/with-lock.mjs`

Перевикористовуваний модуль локу + дедупу. Серце механізму.

**Публічний API:**

```js
withLock(key, runFn, opts?) → Promise<number>
```

- `key` — ідентичність локу (для пілота `'lint-ga'`). Однаковий `key` = взаємне
  виключення; різні `key` незалежні.
- `runFn` — `() => Promise<number>`, реальна робота перевірки; повертає код виходу.
- Повертається код виходу: `0` на дедуп-пропуск або реальний код від `runFn`.
  Інформація «пропущено» транслюється логом `♻️ <key>: …`, не структурно — каліру
  достатньо exit-коду.
- `opts` (необов'язково, мають дефолти-константи): `ttl`, `pollInterval`,
  `waitTimeout`, `staleThreshold`, плюс точки розширення `cacheDir` (override
  розташування стану) і `getFingerprint` (DI-хук для тестів).

**Стан** (на корінь репозиторію, `process.cwd()`):
`node_modules/.cache/n-cursor/<key>/`

- `lock/` — директорія-лок (атомарний `mkdirSync`).
  - `owner.json` — `{ pid, host, startedAt, fingerprint }`.
- `result.json` — `{ finishedAt, exitCode, fingerprint }`.

`node_modules/.cache/` уже поза git і не псує робоче дерево. Знесення при `bun i`
безпечне: під час інсталяції залежностей лінт не виконується, активних локів нема.

**Потік виконання:**

1. Порахувати `fingerprint` поточного дерева (`computeFingerprint`).
2. **Захоплення локу** (цикл):
   - `mkdirSync(lockDir)` успішно → лок наш; пишемо `owner.json`.
   - `EEXIST` → читаємо `owner.json`:
     - застарілий (`host` той самий І `process.kill(pid, 0)` кидає `ESRCH` —
       процес мертвий) **або** `now - startedAt > staleAgeMs` (дефолт 30 хв) →
       видаляємо `lockDir`, повторюємо цикл;
     - інакше — лог `⏳ <key>: чекаю лок (тримає pid <pid>)…`, спимо `pollMs`
       (дефолт 1500 мс), повторюємо.
   - Перевищено `waitTimeoutMs` (дефолт 20 хв) → лог-попередження, **виконуємось без
     локу** (краще прогнати перевірку, ніж залишити агента висіти вічно).
3. **Дедуп** (уже з локом): читаємо `result.json`. Якщо
   `exitCode === 0` **І** `result.fingerprint === fingerprint` **І**
   `now - finishedAt < ttlMs` (дефолт 10 хв):
   - лог `♻️ <key>: те саме дерево, Nс тому — пропускаю`;
   - звільняємо лок, повертаємо `0`.
   - Дедуплікується **лише успіх**. Невдалий прогон (`exitCode !== 0`) завжди
     переганяється — наступному агенту потрібен свіжий вивід помилок.
4. **Виконання:** `exitCode = await runFn()`; пишемо `result.json`
   (`{ finishedAt: now, exitCode, fingerprint }`).
5. **Звільнення:** видаляємо `lockDir` у `finally` — лок звільняється навіть при
   винятку всередині `runFn`.
6. **Сигнали:** обробники `SIGINT` / `SIGTERM` звільняють лок і виходять. Жорсткий
   `SIGKILL` / краш процесу `finally` не виконає — такий випадок ловиться перевіркою
   застарілості з кроку 2 (мертвий PID).

**Чиста функція рішення дедупу** експортується окремо для юніт-тестів:

```js
shouldDedup(result, fingerprint, ttl) → boolean
```

(Поточний час береться всередині через `Date.now()`, тож сигнатура без параметра `now`.)

### 3. Інтеграція в `lint-ga`

Файл `npm/rules/ga/lint/lint.mjs`:

- Поточну логіку експортованої `runLintGaCli` перейменувати в приватну
  `runLintGaSteps()` (preflight → actionlint → zizmor → check-ga — без змін).
- Новий експорт:

  ```js
  export const runLintGaCli = () => withLock('lint-ga', runLintGaSteps)
  ```

Виклик у `npm/bin/n-cursor.js` (case `'lint-ga'`) **не змінюється** —
`process.exitCode = await runLintGaCli()` працює як раніше. Скіли, кореневий
`package.json`, поведінка агента — без змін. Лок повністю інтринсивний.

Внутрішні JS cross-file перевірки `runLintGaSteps` делегують у
`rules/ga/js/workflows/check.mjs` (раніше було `rules/ga/fix/workflows/check.mjs` —
дир перейменована на `js/` згідно з ADR 20260523-215236).

## Потік даних

```
n-cursor lint-ga
   └─ runLintGaCli()
        └─ withLock('lint-ga', runLintGaSteps)
             ├─ computeFingerprint(cwd) ─────────────► fingerprint
             ├─ acquire lock (mkdir, retry, stale-check)
             ├─ read result.json ─── shouldDedup? ──► так → release + return 0
             │                                       ні  ↓
             ├─ runLintGaSteps() ───────────────────► exitCode
             ├─ write result.json {finishedAt, exitCode, fingerprint}
             └─ release lock (finally)
```

## Обробка помилок і межові випадки

| Ситуація                                    | Поведінка                                              |
| ------------------------------------------- | ------------------------------------------------------ |
| Не git-репозиторій / нема `git`             | `fingerprint = null`, дедуп вимкнено, лок працює       |
| Власник локу — мертвий PID (той самий host) | Лок визнається застарілим, перехоплюється              |
| Лок старший за `staleAgeMs` (30 хв)         | Визнається застарілим (захист від зависань)            |
| `runFn` кидає виняток                       | `finally` звільняє лок; виняток прокидається далі      |
| `SIGINT` / `SIGTERM`                        | Обробник звільняє лок, процес виходить                 |
| `SIGKILL` / краш                            | `lockDir` лишається; наступний прибере по мертвому PID |
| Очікування довше `waitTimeoutMs` (20 хв)    | Попередження + виконання без локу (анти-deadlock)      |
| Пошкоджений `owner.json` / `result.json`    | Трактується як відсутній/застарілий, перезаписується   |
| Гонка двох `mkdir` одночасно                | Атомарність `mkdir`: рівно один виграє, решта чекають  |

## План тестування (`bun test`, у `npm/scripts/utils/tests/`)

Тести живуть у піддиректорії `tests/` поряд із кодом (`npm/scripts/utils/tests/`), а
не co-located поруч із джерелом — це нова конвенція проєкту (ADR 20260523-154806).

`tests/worktree-fingerprint.test.mjs`:

- детермінізм: двічі поспіль на тому самому дереві → однаковий відбиток;
- зміна tracked-файла → відбиток інакший;
- поява untracked-файла → відбиток інакший;
- поза git-репозиторієм → `null`.

`tests/with-lock.test.mjs`:

- `shouldDedup` (чиста функція): успіх + збіг відбитка + у межах TTL → `true`;
  невдача → `false`; розбіжність відбитка → `false`; протермінований TTL → `false`;
- виявлення застарілого локу: мертвий PID → перехоплення; вік > `staleAgeMs` →
  перехоплення;
- два паралельні `withLock` на штучну затримку → серіалізуються (сумарний час ≈ 2×,
  не 1×);
- другий `withLock` із тим самим відбитком після успішного першого → `skipped: true`,
  `runFn` не викликається;
- `runFn` кидає виняток → лок звільнено, директорія-лок прибрана.

## Розгортання після пілота (1.13.85)

- **`lint-rego` / `lint-text` / `lint-k8s` / `lint-docker`** — обгорнуто за тим самим
  патерном: приватна `runLint<Foo>Steps()` + публічна
  `runLint<Foo>Cli = () => withLock('lint-<rule>', runLint<Foo>Steps)`. Диспетчер у
  `bin/n-cursor.js` await-ить усі `lint-*` (стали async як наслідок).
- **`lint-rego` додатково:** експортує `runLintRegoSteps(cwd)` без локу — тести
  передають тимчасовий `cwd`, але fingerprint завжди обчислюється від `process.cwd()`,
  тож наївна обгортка дедупувала б другий тестовий виклик проти першого.
- **`fix`-лок переїхав у `runStandardRule`** замість зовнішньої обгортки навколо
  `runFixCommand`. Кожен `rules/<id>/fix.mjs` отримує лок «безкоштовно» через
  делегацію: `withLock('fix-<ruleId>', …)` всередині `runStandardRule`. Per-rule
  гранулярність — різні правила паралельно, однакові серіалізуються.
- **Канон у [`scripts.mdc`](../../../.cursor/rules/scripts.mdc) v1.9** — секція
  «Серіалізація важких CLI-команд: `withLock`» з таблицею ключів, патерном
  інтеграції для нових `lint-*` і red flags.

## Поза обсягом

- Виведення `ttl` / таймаутів у конфіг `.n-cursor.json` — поки константи в модулі
  (override через `opts` лишається доступним як точка розширення).
- Звуження скоупу відбитка per-command (напр. лише релевантні файли).
- Дедуп невдалих прогонів (зберігання й відтворення виводу помилок).
- Примусове застосування через PreToolUse-hook.
- Лок навколо самого `bun run lint-js` (oxlint / eslint / jscpd / knip — сторонні
  бінарники, не керовані пакетом `@nitra/cursor`).
