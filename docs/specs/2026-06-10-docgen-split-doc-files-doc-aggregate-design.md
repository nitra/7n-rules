# Розділення `docgen` на `doc-files` і `doc-aggregate` — дизайн-спека

Дата: 2026-06-10
Власник: @vitaliytv
Статус: Approved (2026-06-10)

## Мета

Розділити монолітний worktree-only скіл **`docgen`** (3 рівні документації в одному прогоні) на два незалежні скіли з різними життєвими циклами:

- **`doc-files`** — документація по кожному кодовому файлу окремо (нинішній Tier 1). Стає **обовʼязковим кроком кожної задачі** (як lint): після правки джерела його дока має бути актуальною. Біжить у поточному робочому дереві, інкрементально, керується **CRC**-звіркою через **PostToolUse hook**.
- **`doc-aggregate`** — агрегуюча документація (нинішні Tier 2 module-summary + Tier 3 доменні доки). Викликається **за запитом**, лишається **worktree-only** і важким.

Старий скіл/namespace `docgen` видаляється повністю, без fallback.

## Передісторія

Чинний `docgen` (`npm/skills/docgen`, скіл `n-docgen`) робить три строго послідовні рівні в одному worktree-прогоні:

1. **Tier 1** — `<dir>/docs/<stem>.md`, субагент на файл, батчі по 5. Scanner (`docgen scan`) лістить кодові файли, фільтр за полем `exists` (пропускає вже описані).
2. **Tier 2** — `<module>/docs/ARCHITECTURE.md`, агрегат, завжди регенерується.
3. **Tier 3** — `docs/<домен>.md`, один субагент-синтезатор.

Проблеми чинної моделі:

- **Дрейф файлових док.** Tier 1 пропускає файл, якщо дока вже існує (`exists:true`), навіть коли джерело **змінилося** — дока тихо застаріває. Немає механізму інкрементальної регенерації під конкретну зміну.
- **Все або нічого.** Щоб оновити одну файлову доку, треба запускати весь worktree-only прогон з усіма трьома рівнями.
- **Файлова дока — не частина процесу задачі.** Lint обовʼязковий, а документація — окремий ритуал, який легко забути; через це доки systematically відстають від коду.

Рішення: винести Tier 1 в окремий легкий скіл, привʼязаний до життєвого циклу задачі (як lint), а важкий агрегат лишити окремою командою за запитом.

## Scope

**In:**

- Новий CLI namespace `doc-files` (`scan`, `check`) і `doc-aggregate` (`modules`).
- Новий скіл `doc-files` (Tier 1, не worktree, обовʼязковий) і `doc-aggregate` (Tier 2+3, worktree-only, за запитом).
- **CRC-маркер у YAML-frontmatter** кожної файлової доки + детермінована звірка застарілості.
- **PostToolUse hook**, що флагує застарілу/відсутню файлову доку після правки джерела.
- Видалення `npm/skills/docgen`, CLI-кейса `docgen`, скіла `n-docgen`.
- Документ-конвенція в `CLAUDE.md` про обовʼязковість `doc-files` (дзеркало lint).

**Out:**

- Зміна стилю/правил самого тексту документації (українська, чистий Markdown, behavior-focus) — переноситься як є.
- git-diff-базований механізм детекції змінених файлів (відхилено на користь CRC, див. нижче).
- Автокоміт згенерованої доки (як і раніше — вирішує користувач).
- Зміна ignore-глобів (`docgen-ignore.mjs` — переноситься як є).

## Рішення: CRC-frontmatter як джерело правди про застарілість

**Відхилено:** git-diff-детекція змінених файлів. Привʼязка до base-ref крихка (rebase, незакомічене, невизначена «база задачі»), погано лягає на per-edit hook і не знає, чи доку вже оновили під цю зміну.

**Прийнято:** кожна файлова дока несе у **frontmatter** CRC байтів джерела на момент генерації:

```markdown
---
docgen:
  source: src/lib/foo.js
  crc: a3f1c9e0
---

## Огляд
…
```

- `crc` — **CRC32** вмісту файлу-джерела, hex (8 символів). Дешево рахувати, слова `crc` дотримано; колізії для цього юзкейсу нехтовні.
- `source` — відносний шлях до джерела (для самодостатності доки й діагностики).
- **Stale** = доки немає **АБО** `crc32(поточне джерело) ≠ crc у frontmatter`.
- Frontmatter — єдиний дозволений виняток із правила «чистий Markdown без HTML»: це машинні метадані, не контент.

Переваги над git-diff: O(1) перевірка одного файлу (hook бачить лише змінений шлях зі stdin), нуль залежності від git-стану, точно знає чи доку вже оновлено (CRC збігся → свіжа), стійкість між гілками/rebase.

Компроміс: whitespace-only зміна джерела фліпає CRC і спричиняє зайву регенерацію. Прийнятно — консервативно в бік свіжості.

## Рішення: оркестрацію генерації веде JS, не модель

Генерація файлових док **уже** має повноцінний JS-шлях (не Claude-субагенти):

- `docgen-gen.mjs` — `generateDoc()` кличе LLM **напряму** (`callOmlx`/`resolveModel`), багатостадійний pipeline (one-shot → секції → critic → refine, quality-threshold 70). Роутинг за складністю: `sym<4` → локальна модель (gemma/omlx, MLX), `sym≥4` → Claude cloud.
- `docgen-batch.mjs` — повний JS-оркестратор: scan → класифікація по symbol-count → цикл по файлах із `generateDoc` → накопичення stats.

**Рішення (питання 4):** скіл `doc-files` лишається **тонким** — він лише запускає JS-команду генерації, а **весь work-queue, батчинг, роутинг, повтори й прогрес веде JS**. Модель сесії **не** диспатчить субагентів і **не** тримає в контексті сотні файлів — тому не «заморюється» навіть на першому масовому прогоні всього репо. Це принципова відмінність від старого `docgen`, де батчами по 5 керувала сама модель.

**CRC штампує JS (питання 1):** frontmatter `source`+`crc` дописує/оновлює JS у тому ж `writeFile`-кроці, що пише тіло доки (детерміновано, без ризику, що модель зіпсує формат). Окремий `doc-files stamp` — для повторного штампування наявних док при першій міграції.

## Гейтинг: PostToolUse флагує + Stop-hook блокує

PostToolUse hook — детермінований CLI **без LLM**: він не генерує доку, лише детектує дрейф. Гейт двошаровий (питання 3 — сильніший гейт):

- **PostToolUse (мʼякий сигнал).** На `Edit|Write|MultiEdit` бере шлях зі stdin; якщо це кодове джерело (не ignored) і `crc(джерело) ≠ crc(дока)` (або доки немає) — повертає Claude фідбек: «дока для `<file>` застаріла → `/doc-files`».
- **Stop-hook (твердий гейт).** На `Stop` запускає `doc-files check` по змінених у задачі джерелах; якщо є stale/missing — **блокує завершення** задачі (non-zero / decision-block), доки доку не регенеровано. Це робить `doc-files` справді обовʼязковим, як lint.

```
Claude редагує foo.js
   │
   ▼
PostToolUse ── crc(foo.js) ≠ crc(docs/foo.md)? ──► мʼякий фідбек "застаріла"
   …
Claude завершує задачу
   │
   ▼
Stop-hook ── doc-files check ──► є stale? ──► БЛОК + "запусти /doc-files"
                                    │
                                    ▼ (модель запускає скіл)
                         doc-files gen ── JS-оркестрація ── LLM пише docs/*.md + CRC
```

## Deliverable

### 1. CLI: `doc-files`

`npx @nitra/cursor doc-files <scan|check|gen|stamp> [--root <dir>]`

- **`scan`** — лістинг кодових файлів зі станом застарілості. На відміну від старого `docgen scan`, поле більше не `exists`, а **`stale`** (missing OR crc-mismatch):

  ```json
  [{ "sourcePath": "src/lib/foo.js", "docPath": "src/lib/docs/foo.md", "stale": true, "reason": "missing" }]
  ```

  `reason`: `"missing"` | `"crc-mismatch"` | `null` (свіжа).

- **`check`** — детермінований режим для hook: приймає один або кілька шляхів-джерел (зі stdin або аргументів), друкує застарілі. Exit-код придатний як гейт. Точка входу **обох** hook'ів (PostToolUse — м'який, Stop — блокуючий).

- **`gen`** — **JS-оркестрована генерація** (нащадок `docgen-batch.mjs`): scan → фільтр `stale` (або всі з `--overwrite`) → класифікація по symbol-count → цикл по файлах через `generateDoc` (роутинг local/cloud) → запис доки **з вшитим свіжим CRC у frontmatter** → stats. Прапори `--limit N` / `--from N` для дозапуску великого прогону. Це команда, яку кличе скіл — уся важка робота тут, не в контексті моделі.

- **`stamp`** — детерміновано (пере)штампувати frontmatter `source`+`crc` у наявній доці без виклику LLM. Для першої міграції наявних док.

Усі команди читають/пишуть CRC через спільний (продубльований у пакеті, див. #7) модуль `crc`-обчислення й frontmatter-парсингу.

### 2. CLI: `doc-aggregate`

`npx @nitra/cursor doc-aggregate modules [--root <dir>]` — нинішній `docgen modules` як є (лістинг логічних модулів для Tier 2). Tier 3-синтез лишається інструкцією в скілі (детермінованого CLI не потребує).

### 3. Скіл `doc-files`

- **Не** worktree-only (`meta.json` без `worktree:true`). Біжить у поточному робочому дереві задачі.
- **Тонкий скіл:** workflow зводиться до запуску `doc-files gen` (за потреби з `--limit/--from` для дуже великих прогонів) + читання підсумкових stats. Уся оркестрація (черга, батчинг, роутинг local/cloud, CRC-штамп) — у JS, модель її не веде.
- Промпт генерації (нинішній Tier 1-стиль) живе в `docgen-prompts.mjs`-аналозі пакета й застосовується JS-пайплайном `generateDoc`, не диспатчем субагентів.
- Команда `/doc-files`.

### 4. Скіл `doc-aggregate`

- Worktree-only (`meta.json` → `worktree:true`), з тим самим preflight/bootstrap-блоком, що нині в `docgen`.
- Workflow: Tier 2 (`doc-aggregate modules` → батчі по 5) → Tier 3 (один синтезатор). Промпти Tier 2/Tier 3 переносяться як є.
- **Передумова:** агрегат припускає, що файлові доки свіжі (їх підтримує `doc-files`). Опційно на старті може запустити `doc-files check` по всьому дереву й попередити, якщо є stale-файли.
- Команда `/doc-aggregate`.

### 5. Hooks (двошаровий гейт)

- **PostToolUse** (`Edit|Write|MultiEdit`) — викликає `doc-files check` по щойно зміненому шляху; **не блокує, не генерує**, лише повертає м'який фідбек про stale-доку. Узгодити з наявним `post-tool-use-fix` (один прогін на правку, без конфлікту).
- **Stop-hook** — викликає `doc-files check` по змінених у задачі джерелах і **блокує завершення** (decision-block / non-zero) за наявності stale/missing, з вказівкою запустити `/doc-files`. Це твердий гейт, що робить doc-files обовʼязковим, як lint.
  - **Набір файлів** визначаємо **найшвидшим** способом — `git diff --name-only HEAD` (working-tree проти HEAD), фільтр по кодових джерелах (не ignored). Допускаємо, що зловить не завжди (напр. уже закомічені в межах задачі зміни випадуть) — це свідомий компроміс на користь простоти й швидкості; CRC лишається джерелом правди про застарілість для тих файлів, що в наборі.
  - **Поріг великого прогону:** якщо stale-джерел **> 50** — гейт **не блокує** (лише виводить попередження зі зведенням). Захист від нескінченного блоку на першій ініціалізації / масовій зміні; поріг — env-конфігурований (дефолт 50).
- Обидва прописуються в `.claude/settings.json` (і дзеркалиться в інлайн-снапшот sync, не руками).

### 6. Конвенція в `CLAUDE.md`

Додати секцію (дзеркало lint): doc-files — обовʼязковий крок задачі; після зміни кодового файлу його дока має бути регенерована (`/doc-files`); PostToolUse hook сигналить про дрейф.

### 7. Видалення

- `npm/skills/docgen/` (увесь каталог: SKILL.md, meta.json, `js/*`, bench, тести).
- CLI-кейс `case 'docgen'` у `npm/bin/n-cursor.js` + згадки в usage/JSDoc.
- Згенерований скіл `.cursor/skills/n-docgen/` (через sync, не руками).
- Модулі `docgen-*.mjs` (prompts, ignore, scan, gen, batch, extract, …) — **дублюються** в `npm/skills/doc-files/js` і `npm/skills/doc-aggregate/js` (питання 2). Спільного `_shared`-каталогу **не** робимо: скіли далі еволюціонують окремо, спільна абстракція їх звʼязала б. Свідома копія на старті дешевша за передчасну спільність.

## Вирішені рішення (раніше відкриті)

1. **CRC рахує JS** у `writeFile`-кроці генерації + окрема `doc-files stamp` для міграції. Не модель/субагент.
2. **Дублювання** модулів між `doc-files` і `doc-aggregate` — без спільного каталогу, бо скіли еволюціонують окремо.
3. **Сильніший гейт:** Stop-hook **блокує** завершення задачі за наявності stale-док (плюс м'який PostToolUse-сигнал).
4. **Першу ініціалізацію опрацьовуємо повністю** (усі `missing`), навіть якщо довго — але **оркеструє JS** (`doc-files gen`, за потреби порціями `--from/--limit`), щоб модель не «заморилася». Не лишаємо на ручний диспатч моделі.

## Параметри гейта (вирішено)

- **Детекція змінених джерел на Stop:** `git diff --name-only HEAD` (working-tree проти HEAD) + фільтр кодових джерел. Найшвидший шлях; допускаємо неповне покриття (закомічене в межах задачі випадає). CRC — джерело правди про застарілість у межах набору.
- **Поріг великого прогону:** stale-джерел > **50** → гейт **не блокує**, лише попереджає. Env-конфігурований дефолт 50.

## Міграційний план

1. Скопіювати `docgen-*.mjs` у `npm/skills/doc-files/js` і `npm/skills/doc-aggregate/js`; додати модуль CRC/frontmatter; розширити scan режимом `stale`; додати `check`/`gen`/`stamp` (`gen` = нащадок `docgen-batch.mjs` з CRC-штампом).
2. Додати CLI-кейси `doc-files` і `doc-aggregate`; прибрати `docgen`.
3. Створити `npm/skills/doc-files` (тонкий, не worktree) і `npm/skills/doc-aggregate` (worktree-only) — SKILL.md + meta.json, розщепивши нинішній SKILL.md.
4. Прописати PostToolUse + Stop hooks у `.claude/settings.json` (через inline-снапшот sync) + секцію-конвенцію в `CLAUDE.md`.
5. Sync → перегенерувати `.cursor/skills/*`; видалити `n-docgen`.
6. Масовий перший прогін `doc-files gen` по репо (JS-оркестрований, порціями за потреби), щоб усі наявні доки отримали frontmatter+CRC; за наявних док — `doc-files stamp`.
