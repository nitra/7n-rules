# Dev-design Фази B: адаптер lint-хвіст → MT-граф

**Дата:** 2026-07-11
**Статус:** частково історичний — B2 (executor-міст `mt-run-node`) видалено 2026-07-14
**Батьківські документи:** `docs/specs/2026-07-11-pi-harness-mt-fix-graph.md` (рішення В/Д/Е/Ж), `docs/specs/2026-07-11-llm-lib-run-harness-dev-design.md` (Фаза A — завершена)

> **Міграція 2026-07-14.** Після рішення nitra/mt «ACP — єдиний транспорт AI-викликів»
> (mt ADR `260713-2110`, `260713-2040`) fix-вузли виконує **вбудований** шлях MT —
> підписочні CLI (`claude` | `codex` | `cursor` | `pi`); тир-канон зберігається через
> user-level ENV: `MT_AGENT_CLI` (дефолтний CLI), `MT_CLOUD_AGENT_CLIS` (каскад підписок),
> `MT_AGENT_CLI_MODEL_MAP` (мапа «CLI → тир → модель»); локальні omlx-моделі — `pi`
> (pi.dev CLI обгортає omlx-сервер тим самим ACP). Тому B2-міст (`mt-run-node`,
> `ensureNodeExecutor`, `node_executor` у `.mt.json`) видалено з n-cursor, а в MT
> видаляється сама точка розширення `node_executor`. B1 (матеріалізація хвоста,
> `mt-tail.mjs`) чинна без змін. Розділи нижче про B2/executor-міст — історичний контекст.

## 1. Відправна точка (перевірено наживо)

Фаза A закрита: `@7n/llm-lib` має `createHarness` + evidence-gate/anchored/web профілі. Тепер — адаптер, що бере **хвіст** fix-прогону (порушення, які інлайн-драбина не закрила) і матеріалізує їх вузлами MT-графа.

**MT M0 фактичний стан (локальна валідація 2026-07-11, `@7n/mt@0.14.0`):**
- CLI працює, але потребує нативний addon (`cargo build --release -p mt-napi` → `libmt_napi.dylib`; env `MT_NATIVE_ADDON` або підпакет `@7n/mt-darwin-arm64`). **Блокер для CI/consumer-машин без Rust-тулчейну** — див. §5.
- Команди M0 доступні: `setup`, `init`, `plan`, `spawn`, `run`, `audit`, `done`, `failed`, `status`, `scan`, `watch`, `invalidate`, `kill`.
- `mt setup` створює `.mt.json` + `mt/` + `.worktrees/` + git post-commit hook.
- `mt init <name> [--mode agent|human] [--model-tier MIM|AVG|MAX] [--budget-sec N] [--hint] [--dep <id>]` — скафолдить `mt/<name>/task.md` (шаблон) + прапор `a.md`/`h.md`.

**Знайдені розбіжності контракту (тертя M0 — треба узгодити з MT перед реалізацією):**
1. Шаблон `task.md` вживає `## Mission`, а не `## Task` (як у `architecture/graph.md`).
2. Шаблон **не містить `## Check`** — хоча graph.md робить `## Check` машинним гейтом done/audit. Це критично для адаптера: re-check команда — серце fix-вузла.
3. `.mt.json` consumer-репо має `model_map` на **Claude-моделі** (`claude-sonnet-4-6` тощо) і `claude_model`/`audit_model`, а не на pi-тири llm-lib (omlx/openai-codex). Тобто MT-runner виконує вузли **власним** agent-шляхом (Claude), не через наш pi-harness — це означає, що «llm-lib виконує один run» (рішення В) наразі НЕ реалізується автоматично; MT має свій executor.

**Наслідок для дизайну:** рішення В спеки («агентичний хвіст = вузли MT, llm-lib виконує один run») розпадається на дві інтеграції:
- **B1 — матеріалізація** (adapter пише вузли): не залежить від того, ХТО їх виконує.
- **B2 — executor-міст** (MT-вузол виконується нашим pi-harness замість вбудованого Claude-шляху MT): потребує точки розширення в MT-runner, якої зараз НЕМАЄ.

**РІШЕННЯ ВЛАСНИКА (2026-07-11): проміжний стан «Claude-агент MT виконує хвіст» ВІДХИЛЕНО.** Тир-канон (локальні omlx → cloud pi-тири llm-lib) — обовʼязковий і для хвоста; виконувати вузли чужим Claude-шляхом MT неприйнятно. Наслідок: **B2 — ТВЕРДА ПЕРЕДУМОВА, не відкладене**. Фаза B не вмикається у продакшені, поки MT не дасть executor-міст. B1 можна написати й протестувати (scripted-MT) як фундамент, але enable — лише разом із B2.

**Дві тверді передумови в MT (обидві — робота на боці @7n/mt, блокують продакшн-Фазу B):**
1. **Executor-міст** — точка розширення MT-runner «виконай вузол зовнішньою командою» замість вбудованого Claude-agent-шляху (щоб вузол ішов через `createHarness().run(fixProfile)` llm-lib з нашими тирами й телеметрією). MT-задача.
2. **Платформні підпакети addon** (`@7n/mt-darwin-arm64` тощо, prebuilt) — щоб consumer-машина без Rust-тулчейну запускала `mt`. Без цього адаптер непридатний деінде, крім машин з локальним build. MT-задача.

## 2. Проєктні рішення

| # | Питання | Рішення |
|---|---|---|
| B-а | init через CLI чи пряме авторство файлів | **Пряме авторство** повного `task.md` (schema-сумісного) + `a.md`, БЕЗ `mt init`: init дає лише шаблон-заготовку без `## Check`, а нам потрібен повний контракт з re-check командою. Пишемо файли за схемою graph.md, `mt scan` валідує. `mt setup` — одноразово, якщо `.mt.json` відсутній. |
| B-б | Що є `## Check` вузла | Рядок-команда точного re-detect: `npx @nitra/cursor lint --rule <id> --concern <c> --files <…>` (детермінований гейт MT з коробки; exit 0 = чисто). |
| B-в | Тир → a.md | `model_tier`: MIM/AVG/MAX. Але каветка §1.3: MT мапить їх на Claude, не на наші тири — поки B2 не готовий, вузол виконає Claude-агент MT. Документуємо як свідомий проміжний стан. |
| B-г | Гранулярність вузлів | Composite fix-batch: батько `lint-fixes/<timestamp>`, діти = правило × кластер файлів; складні правила (doc-files) — вузол на файл. Один прогін → один composite. |
| B-д | Коли адаптер спрацьовує | Лише у fix-режимі, лише для **хвоста** (порушення після інлайн-драбини). `lint --read-only` не торкається MT (межа ціла). Env-гейт `N_LINT_MT_TAIL=1` — opt-in, поки не зрілий. |
| B-е | Ідемпотентність | Сигнатура вузла = `rule+concern+sha(files)`; повторний прогін не дублює наявний невирішений вузол (перевірка `mt scan` перед init). |

## 3. Деталі реалізації

### B1 — матеріалізація (реалізовна зараз)

Новий модуль у `@nitra/cursor` (напр. `npm/scripts/lib/lint-surface/mt-tail.mjs`), викликається central pipeline-ом наприкінці fix-прогону, коли лишились невиправлені порушення і `N_LINT_MT_TAIL=1`:

1. **Preflight:** `.mt.json` є? ні → `mt setup` (або пропуск із логом, якщо MT недоступний — fail-open, хвіст лишається як текст). Addon-перевірка: MT недоступний → лог «MT-tail пропущено» + звичайний exit (НЕ валимо lint).
2. **Групування:** порушення хвоста → мапа `rule → concern → files[]`; кластеризація у вузли (B-г).
3. **Авторство composite-плану:** `mt/lint-fixes/<ts>/task.md` (Mission = «закрити lint-хвіст прогону», decision: composite) + `plan_001.md` з `## Children` (кожна дитина: id, mode: agent, model_tier, `## Check` = re-detect команда, deps: []).
4. **Spawn:** `mt spawn --approve` матеріалізує дітей одним fenced commit (валідує naming/deps).
5. **Вивід:** посилання на створений граф + `mt status`; consumer-прогін НІЧОГО більше не робить (виконання — асинхронне, orchestrator-ом MT).

Тести: scripted MT (fake spawnSync, як у наявних lint-surface тестах) — перевіряємо згенеровані `task.md`/`plan`/`## Check` проти схеми, ідемпотентність, fail-open при відсутньому MT. Live-smoke (реальний addon) — nightly, не в CI.

### B2 — executor-міст (ТВЕРДА ПЕРЕДУМОВА, робота в MT)

Щоб MT-вузол виконувався нашим pi-harness (рішення В), MT-runner має дати точку розширення «виконай цей вузол командою X» замість вбудованого Claude-agent-шляху. Варіанти (для MT-власника): (а) `a.md: executor: cmd` + команда, яку MT спавнить; (б) MT викликає `npx @nitra/cursor mt-run-node <path>`, що всередині кличе `createHarness().run(fixProfile)` з тирами llm-lib. Обидва — зміни в MT. **Без B2 Фаза B не вмикається** (проміжний Claude-executor відхилено власником). n-cursor-бік B2 — тонкий CLI-ентрі `mt-run-node`, що читає контракт вузла й гейтить через harness; реалізується, коли MT дасть спосіб його викликати.

### B3 — human-loop (рішення Е, M0-мінімум)

Після вичерпання agent-retry MT сам пише `unresolvable.md` + (за наявності `h.md`) сигналить людині. Адаптеру тут робити нічого — це штатний MT-механізм; ми лише переконуємось, що складні правила мапляться у вузли з розумним `budget_sec` і `retry_ladder`, щоб ескалація спрацьовувала природно.

## 4. Послідовність (переглянута під рішення власника)

Продакшн-Фаза B гейтиться двома MT-передумовами. Порядок:

1. **MT: платформні підпакети addon** (передумова 2) — прибирає блокер запуску `mt` деінде.
2. **MT: executor-міст** (передумова 1) — точка розширення runner-а.
3. **n-cursor B1** — матеріалізація вузлів (можна ПИСАТИ й тестувати scripted-MT паралельно з 1-2, але enable лише після обох).
4. **n-cursor B2-ентрі** — `mt-run-node`, що виконує вузол через `createHarness().run(fixProfile)` (потребує executor-міст із кроку 2).
5. Узгодження контракту MT (тертя §1: `## Check`/`## Mission`/`## Task`) — паралельно, окрема MT-задача.
6. **Фаза C** — телеметрія у `fact_NNN`.

B1 БЕЗ B2 у продакшн НЕ вмикається (проміжний Claude-executor відхилено). Тому B1 пишемо як фундамент за env-гейтом `N_LINT_MT_TAIL`, а прод-enable — коли готові кроки 1-4.

## 5. Ризики / блокери

- **Нативний addon MT** — головний блокер широкого вжитку: consumer-машина без Rust-тулчейну не запустить `mt`. Потрібен опублікований платформний підпакет (`@7n/mt-darwin-arm64` тощо) або prebuilt. До того — B1 лише на машинах з addon, fail-open деінде.
- **Розбіжності контракту §1** — реалізовувати B1 проти `architecture/graph.md` чи проти фактичного шаблону 0.14.0? Рішення: проти graph.md (нормативний), а розбіжності шаблону — тертя на виправлення в MT (окрема задача).
- **Тир-канон:** поки B2 немає, «lint лікує через локальні omlx-тири» не виконується для хвоста — його бере Claude MT. Це прийнятний проміжний стан (хвіст усе одно закривається автономно, мета «нуль ручного Claude Code» досягається), але це НЕ той самий шлях, що інлайн-драбина.

## Відкриті питання

- Пін `@7n/mt` у devDeps `@nitra/cursor` для scripted-тестів (contract-фікстури) — версія й стратегія оновлення.
- Чи адаптер сам робить `mt setup` у чужому репо, чи вимагає, щоб репо було MT-onboarded заздалегідь (рішення Ж каже «завжди в mt/», але setup у чужому репо = слід у дереві — узгодити з fail-open).
- Формат `plan_NNN.md` `## Children` — точна YAML-схема, яку приймає `mt spawn --approve` 0.14.0 (валідувати живим spawn до реалізації).
