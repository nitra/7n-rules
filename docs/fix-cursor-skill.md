# Cursor Skill: як одна команда приводить проєкт до стандартів за хвилину

## Проблема

У вас є набір правил для проєкту: Bun замість npm, oxfmt замість Prettier, правильні GitHub Actions, cspell з українським словником, ESLint з корпоративним конфігом. Правила описані в `.cursor/rules/*.mdc` — AI-агент їх бачить і дотримується при написанні коду.

Але що робити, коли потрібно **привести весь проєкт до стандартів**? Наприклад:

- Підключили нові правила до існуючого репозиторію
- Оновили конфігурацію і треба перевірити/виправити всі файли
- Новий член команди клонував проєкт і хоче переконатися що все ок

Раніше це виглядало так:

```
Розробник: "Перевір чи проєкт відповідає правилам і виправ"
Агент: *читає 10 правил* *робить 40+ tool calls* *пропускає половину*
Розробник: "А yarn.lock видалив? А GitHub Actions перевірив?"
Агент: *ще 20 tool calls*
```

Повільно, дорого (тисячі токенів), ненадійно.

## Рішення: Cursor Skill `n-fix`

Cursor Skill — це markdown-файл з інструкціями для AI-агента, який автоматично підхоплюється при релевантних запитах. На відміну від правил (`.cursor/rules/`), skill описує не конвенції, а **конкретний workflow** — послідовність дій.

### Як працює

```
Розробник: "n-fix" (або "виправ проєкт", "apply rules")
                │
                ▼
        Cursor бачить Skill
        n-fix/SKILL.md
                │
                ▼
    ┌───────────────────────┐
    │ 1. npx @nitra/cursor  │ ── 1 tool call, 100ms
    │    check              │    замість 40+ tool calls
    │                       │
    │    ✅ Немає yarn.lock │
    │    ❌ Немає .oxfmtrc  │
    │    ❌ ESLint config   │
    │    ✅ bun.lock є      │
    └───────────┬───────────┘
                │
                ▼
    ┌───────────────────────┐
    │ 2. Аналіз ❌          │ ── Агент знає ЩО зламано
    │    .oxfmtrc.json      │    + має правила ЯК чинити
    │    eslint.config.js   │    (з .cursor/rules/*.mdc)
    └───────────┬───────────┘
                │
                ▼
    ┌───────────────────────┐
    │ 3. Виправлення        │ ── Агент створює/оновлює файли
    │    Створив .oxfmtrc   │    використовуючи LLM-інтелект
    │    Створив eslint.cfg │    для складних випадків
    │    Оновив package.json│
    └───────────┬───────────┘
                │
                ▼
    ┌───────────────────────┐
    │ 4. bun i              │ ── Встановлення залежностей
    └───────────┬───────────┘
                │
                ▼
    ┌───────────────────────┐
    │ 5. oxfmt .            │ ── Форматування
    └───────────┬───────────┘
                │
                ▼
    ┌───────────────────────┐
    │ 6. bun run lint-*     │ ── Всі лінтери з package.json
    └───────────┬───────────┘
                │
                ▼
    ┌───────────────────────┐
    │ 7. npx @nitra/cursor  │ ── Верифікація
    │    check              │
    │                       │
    │    ✅ Немає yarn.lock │
    │    ✅ .oxfmtrc.json   │
    │    ✅ ESLint config   │
    │    ✅ bun.lock є      │
    └───────────────────────┘
```

### Ключова ідея: програмна діагностика + LLM-інтелект

Skill поєднує два підходи:

| Етап         | Метод                 | Чому                                               |
| ------------ | --------------------- | -------------------------------------------------- |
| Діагностика  | `check-*.mjs` скрипти | Детерміновано, швидко, 1 tool call                 |
| Виправлення  | LLM-агент             | Гнучко, розуміє контекст, обробляє складні випадки |
| Форматування | `oxfmt`               | Детерміновано, однаковий результат                 |
| Лінтування   | `lint-*` скрипти      | Автоматичне виправлення з `--fix`                  |
| Верифікація  | `check-*.mjs` скрипти | Підтвердження результату                           |

Програмні скрипти кажуть агенту **ЩО** зламано. MDC правила кажуть **ЯК** чинити. Агент з'єднує це і діє.

## Анатомія Skill-файлу

```
.cursor/skills/n-fix/
└── SKILL.md
```

```yaml
---
name: n-fix
description: >-
  Fix project to comply with all n cursor rules. Use when the user asks
  to fix the project, apply rules, make project compliant, or mentions
  n-fix.
---
```

**Frontmatter:**

- `name` — ідентифікатор, до 64 символів
- `description` — коли Cursor має активувати цей skill. Пишеться від третьої особи, з trigger-термінами

**Body:** покроковий workflow з командами, які агент виконує послідовно.

### Повний workflow

```markdown
1. Діагностика → npx @nitra/cursor check
2. Аналіз → знайти ❌, визначити правила
3. Виправлення → створити/оновити файли згідно MDC
4. Встановлення → bun i
5. Форматування → oxfmt .
6. Лінтери → bun run lint-\* (всі з package.json)
7. Верифікація → npx @nitra/cursor check
8. Повтор → якщо ❌ залишились → крок 3
```

## Порівняння з альтернативами

### Варіант 1: Програмні fix-скрипти

```javascript
// fix-bun.mjs — скрипт що сам видаляє файли, оновлює JSON
/**
 *
 */
export async function fix() {
  if (existsSync('yarn.lock')) unlinkSync('yarn.lock')
  // ...
}
```

**Мінуси:**

- Потрібно писати і підтримувати окремий скрипт для кожного правила
- Дублювання логіки (правило + скрипт описують те саме)
- Скрипт не може обробити складні випадки (рефакторинг коду, контекстні зміни)
- Ламкість — зміна правила вимагає оновлення скрипта

### Варіант 2: LLM без діагностики

```
Агент: *читає 10 правил* *перевіряє файли вручну* *виправляє*
```

**Мінуси:**

- 40+ tool calls тільки на діагностику
- Тисячі токенів на reasoning
- Може пропустити порушення
- Повільно (хвилини замість секунд)

### Варіант 3: Skill + check-скрипти (наш підхід)

```
1 tool call (check) → агент бачить список проблем → виправляє → 1 tool call (verify)
```

**Переваги:**

- Діагностика за 100ms замість хвилин
- LLM фокусується на виправленнях, а не на пошуку проблем
- Складні випадки обробляються інтелектом LLM
- Немає дублювання — правила в MDC, перевірки в скриптах, виправлення в LLM
- Feedback loop — повторна перевірка гарантує результат

## Автоматична дистрибуція

Skill розповсюджується разом з пакетом `@nitra/cursor`:

```
npm/@nitra/cursor/
├── mdc/              ← правила (MDC файли)
├── scripts/          ← check-скрипти
├── skills/           ← Cursor Skills
│   └── n-fix/
│       └── SKILL.md
└── AGENTS.template.md
```

При запуску `npx @nitra/cursor` CLI автоматично:

1. Завантажує MDC правила до `.cursor/rules/`
2. Копіює skills до `.cursor/skills/`
3. Генерує `AGENTS.md` з посиланнями на правила і skills

Для інших AI-агентів (Claude Code, Codex) посилання на skill додається в `AGENTS.md`:

```markdown
## Skills

- `.cursor/skills/n-fix/SKILL.md` — автоматичне виправлення проєкту
```

Агент читає `AGENTS.md` → бачить skill → відкриває `SKILL.md` → виконує workflow.

## Як додати свій Skill

### 1. Створи директорію

```bash
mkdir -p .cursor/skills/your-skill-name
```

### 2. Напиши SKILL.md

```markdown
---
name: your-skill-name
description: >-
  What it does. Use when the user asks to...
---

# Your Skill

## Workflow

1. **Крок** — опис
2. **Крок** — опис
```

### 3. Принципи ефективного Skill

- **Конкретні команди** — агент повинен знати що саме запускати
- **Feedback loop** — перевірка після виправлень обов'язкова
- **Програмна діагностика** — де можливо, замінюй ручну перевірку скриптами
- **Не дублюй правила** — skill каже ЩО робити, MDC каже ЯК
- **До 500 рядків** — SKILL.md має бути компактним

## Підсумок

| Компонент     | Роль                                   | Де живе           |
| ------------- | -------------------------------------- | ----------------- |
| MDC правила   | Конвенції для LLM: як писати код       | `.cursor/rules/`  |
| Check-скрипти | Програмна діагностика: що зламано      | `npm/scripts/`    |
| Skill         | Workflow для агента: послідовність дій | `.cursor/skills/` |
| AGENTS.md     | Entry point для всіх агентів           | корінь проєкту    |

Skill `n-fix` — це точка, де програмна верифікація зустрічається з інтелектом LLM. Скрипти швидко знаходять проблеми, агент розумно їх вирішує, а потім скрипти підтверджують результат. Один workflow, одна команда, повна відповідність стандартам.
