# Cursor Skill `n-coverage-fix`: автоматичне підвищення mutation score

## Проблема

Покриття рядків (`lines coverage`) бреше. Тест може «зачепити» рядок, але нічого не перевірити — і метрика покаже 100%, хоча баг проскочить. Чесну відповідь дає **мутаційне тестування**: Stryker вносить дрібні зміни в код (`true → false`, `+ → -`, `=== → !==`) і дивиться, чи **впаде** хоч один тест. Якщо мутант **вижив** (`survived`) — значить, ця поведінка не перевіряється жодним тестом.

Ручний цикл «вбивання» вцілілих мутантів виглядає так:

```
Розробник: "підніми mutation score"
Агент: *запускає coverage* *читає 30 survived-мутантів*
       *відкриває кожен файл* *пише тести навмання* *забуває половину*
Розробник: "а ці 12 ще живі"
Агент: *ще один захід наосліп*
```

Повільно, без зупинки, без гарантії, що score взагалі зрушив.

## Рішення: один автономний скіл

`n-coverage-fix` замикає це в **детермінований цикл до конвергенції**: згенерувати звіт → прочитати вцілілих → дописати тести → перевірити → переміряти → повторити, поки score не перестане рости (або 3 ітерації).

> **Історична нота.** Раніше було два майже однакові скіли — `n-coverage-fix` (запускав coverage сам) і `n-fix-tests` (стартував по готовому `COVERAGE.md`). `n-fix-tests` був строгою підмножиною (~95% дубль) і його **видалено**; усе корисне з нього (детекція команд із `package.json#scripts`, старт по готовому звіту) поглинуто канонічним `n-coverage-fix`. Деталі — `docs/adr/260604-2008-злиття-n-fix-tests-у-n-coverage-fix-із-видаленням-дубляту.md`.

### Загальна схема

## Як працює покроково

### Крок 0a — preflight (worktree-only)

Скіл має `meta.json → worktree: true`, тож виконується **виключно** в окремому git-worktree. Перший крок — перевірка `git rev-parse --show-toplevel`: якщо ти **не** під `.worktrees/` — **STOP**, скіл сам створює worktree від поточної гілки (`<branch>-coverage-f`) literal-командами (без shell expansion) і ставить залежності (`bun install` + обгортка `n_cursor_npx` з retry на CDN-пропагацію щойно опублікованої версії).

**Навіщо worktree:** Stryker пише `mutation.json` і `incremental.json` в одну директорію — два паралельні прогони **зіпсують обидва файли**. Ізоляція + заборона паралелі гарантують один інстанс за раз.

### Крок 0b — детекція команд (з `package.json#scripts`)

Скіл читає кореневий `package.json` і фіксує дві команди (перша, що існує):

| Команда              | Джерело                                    | Fallback            |
| -------------------- | ------------------------------------------ | ------------------- |
| **coverage-команда** | `scripts["coverage"]` → `bun run coverage` | `n-cursor coverage` |
| **test-команда**     | `scripts["test"]` → `bun run test`         | `bun test`          |

Завдяки цьому скіл адаптується до проєкту, а не хардкодить інструментарій.

### Крок 1 — згенерувати або повторно використати `COVERAGE.md`

**Early-skip:** якщо `COVERAGE.md` уже існує, свіжий (новіший за останню зміну source/тестів) і має секцію `## Вцілілі мутанти` — одразу до Кроку 2. Інакше — запустити coverage-команду, яка згенерує звіт.

Саме це робить скіл універсальним для **обох** сценаріїв: «згенеруй і виправ» та «виправ по вже готовому звіту».

### Крок 2 — прочитати вцілілих

`COVERAGE.md` → секція `## Вцілілі мутанти` → огороджений ` ```json ` блок → JSON-масив. Якщо секції нема або масив порожній — **стоп** (`✓ Жодних вцілілих мутантів — mutation score повний`). Інакше запам'ятати `prevCount = масив.length` як базу для порівняння.

JSON-блок у звіт кладе `renderMarkdown` із `npm/rules/test/coverage/coverage.mjs` — це контракт між coverage-правилом і скілом.

### Крок 3 — диспатч Agent по файлах

Мутанти групуються за полем `file`. Для кожної групи:

1. **Визначити test-файл** — завжди `<dir>/tests/<basename>.test.mjs`. Якщо знайдено co-located тест (`.test.js`/`.test.mjs`) — переноситься у `tests/` з оновленням відносних imports.
2. **Сформувати промпт** — source-файл + наявні тести + список мутантів (рядок, колонка, `original → replacement`, тип).
3. **Запустити окремий Agent** — він дописує **мінімальні** тест-кейси, що вловлюють кожен мутант, **не чіпаючи** наявні тести, і сам проганяє `bun test <file>`.

### Крок 4 — перевірити всі тести

Запустити test-команду з Кроку 0b. Якщо щось падає — повернути відповідний Agent із текстом помилки на виправлення.

### Крок 5 — переміряти й вирішити щодо циклу

Повторно запустити coverage-команду, порахувати `newCount`:

| Умова                                     | Дія                              |
| ----------------------------------------- | -------------------------------- |
| `newCount < prevCount` **і** ітерацій < 3 | новий виток із Кроку 2           |
| `newCount >= prevCount`                   | ✓ **конвергенція** — стоп        |
| ітерацій == 3                             | ⚠️ ліміт — стоп із попередженням |

## Конвергенція — це норма, а не баг

Частину мутантів **неможливо** вбити: захищений зовнішній стан, недетермінована логіка, еквівалентні мутації (поведінка не змінюється), dead code. Коли після ітерації `newCount` не зменшився — скіл зупиняється, а не б'ється об стіну. Stryker `incremental` зберігає прогрес між запусками, тож crash ≠ старт із нуля.

## Запобіжники

- **Не запускати паралельно** — ні в різних Bash-задачах, ні в субагентах. Stryker пише в спільну директорію.
- **Тільки у worktree** — `meta.json → worktree: true`, preflight fail-fast.
- **Не комітить автоматично** — рішення, коли комітити, лишається за людиною.
- **Ліміт 3 ітерації** — захист від нескінченного циклу на стійких мутантах.

## Підсумок

| Компонент              | Роль                                             | Де живе                                  |
| ---------------------- | ------------------------------------------------ | ---------------------------------------- |
| `n-cursor coverage`    | генерує `COVERAGE.md` (Stryker + покриття)       | `npm/rules/test/coverage/coverage.mjs`   |
| `## Вцілілі мутанти`   | контракт-звіт: JSON-масив survived               | `COVERAGE.md` у корені проєкту           |
| Skill `n-coverage-fix` | автономний цикл «дописати тести до конвергенції» | `.cursor/skills/n-coverage-fix/SKILL.md` |
| Agent (на файл)        | пише мінімальні тести під конкретні мутанти      | диспатчиться скілом                      |

`n-coverage-fix` — точка, де детермінований інструмент (Stryker каже **ЩО** не перевірено) зустрічається з інтелектом LLM (агент вирішує **ЯК** це протестувати), замкнена у feedback-цикл із чесним критерієм зупинки.
