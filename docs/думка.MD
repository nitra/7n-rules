# Архітектура: Динамічний Самомодифікований Граф Задач

## Назва структури

**Рекурсивний складений ОАГ** (орієнтований ациклічний граф) — із динамічним розкладом вузлів та файловим сховищем стану.

---

## Вузол

Кожен вузол або є атомарним, або розкладається на підграф — рішення приймається **динамічно при запуску** на основі вхідних даних.

```
Вузол
  ├── реалізація:
  │     ├── Атомарний  — fn(вхідні) → вихідні
  │     └── Складений  — Граф{ вхід, вузли[], ребра[], виходи[] }
  ├── стан: очікує | виконується | вирішено | помилка | знедійснений
  ├── вхідні:  Map<portId, Value>
  └── вихідні: Map<portId, Value>   ← заповнюється при вирішено
```

Для батьківського вузла інтерфейс однаковий: він чекає `стан = вирішено` не знаючи що всередині (**інкапсуляція чорної скриньки**).

---

## Граф

- **ОАГ** — орієнтований, без циклів
- **Ребра** — потік даних: виходи одного вузла стають входами наступного
- **Вхідний вузол** — отримує вхідні дані батьківського складеного вузла
- **Вихідні вузли** — кілька, їх виходи зливаються у виходи батька
- Складений вузол може бути замінений атомарним і навпаки без змін у батьківському графі

```
СкладенийВузол
  └── Граф
        ├── вхід    → отримує вхідні дані батька
        ├── вузли[]: Вузол[]
        ├── ребра[]: { від: ВузолId+portId, до: ВузолId+portId }
        └── виходи[]: Вузол[]   ← кілька, злиття → виходи батька
```

**Топологія живе в `task.md` кожного дочірнього** — поле `deps:`. Жодного центрального файлу графу. Оркестратор відновлює повний граф скануванням усіх `task.md`.

---

## Вузол як агент

Кожен вузол — це запуск Claude. Оркестратор передає протокол через `system_prompt`, файли вузла — лише місія та дані. Агент самостійно вирішує атомарний/складений **тільки на старті** — тип не задається наперед у `task.md`.

**Запуск:**

```
Claude(
  system_prompt = <протокол оркестратора>,
  context       = [task.md] + run_*.md (всі попередні спроби)
)
```

**System prompt оркестратора** (агент ніколи не бачить у своїх файлах):

```
Твій файл: task.md (місія, inputs, бюджет).
Після виконання:
  -> успіх:    запиши outputs_NNN.md + run_NNN.md (result: success)
               виклич: n-cursor graph done <шлях-вузла>
  -> складно:  створи <child-id>/task.md для кожного дочірнього
               виклич: n-cursor graph spawn <шлях-вузла>
  -> помилка:  запиши run_NNN.md (result: failed)
               виклич: n-cursor graph failed <шлях-вузла>
```

**Агент може читати файли будь-яких вузлів** (батька, братів, дочірніх) без обмежень — для контексту.

**Всі агенти запускаються через скрипт-обгортку** — скрипт відстежує таймаут і може кілити процес.

---

## Naming convention вузлів

- Дозволені символи: `a-z`, `0-9`, `-`
- Роздільник: `-`
- Унікальність: в межах батька (серед сусідів в тій самій директорії)
- Приклади: `collect-data`, `analyze-results`, `synthesize`
- `id` вузла = назва директорії (не дублюється у фронтматері)
- Атрибути фронтматеру — англійські, snake_case
- **Всі імена файлів і директорій — англійська** (обробляються скриптами)
- Заголовки секцій що парсить скрипт — англійські; секції з довільними даними — будь-яка мова

## Файлова структура вузла

Дочірні вузли живуть **безпосередньо** в директорії батька. Якщо директорія містить `task.md` — це вузол.

```
tasks/
  <node-id>/
    task.md              <- місія + inputs + budget_sec (immutable після ворктрі)
    invalidated          <- sentinel: порожній файл, наявність = знедійснений
    run_001.md           <- перша спроба (immutable)
    run_002.md           <- друга спроба (immutable)
    outputs_001.md       <- вихідні першого успішного запуску (immutable)
    outputs_002.md       <- вихідні після re-run (immutable)
    <child-node-id>/     <- дочірній вузол (якщо складений)
      task.md
      ...                <- та сама структура рекурсивно
```

**Стан вузла визначається наявністю файлів** (не полем у фронтматері):

| Які файли існують | Стан |
|---|---|
| тільки `task.md` | `waiting` |
| є активний ворктрі | `running` |
| є `outputs_*.md` і немає `invalidated` | `resolved` |
| є `run_*.md` але немає `outputs_*.md` | `failed` |
| є `invalidated` | `invalidated` |

## Контракт файлів — інваріанти

**Формат:** Markdown з YAML-фронтматером.

- Атрибути фронтматеру → **англійські, snake_case**
- Заголовки секцій що парсить скрипт → **англійські**
- Секції з довільними даними → **будь-яка мова**

**Всі файли — immutable** (не змінюються після створення). Новий факт = новий файл.

`task.md`, `run_NNN.md`, `outputs_NNN.md`

**Межа:**

- До створення ворктрі — файли вузла можна вільно редагувати і видаляти
- Після створення ворктрі — лише новий вміст, нічого не змінюється

**Синтаксис посилань** (у `inputs.md` / `outputs.md`):

```
ref: ../collect-data/outputs.md              # весь файл (відносно поточного файлу)
ref: ../collect-data/outputs.md#results      # секція за заголовком
ref: ../collect-data/outputs.md lines 5-20  # діапазон рядків
```

Шляхи відносні від поточного файлу.

## Схеми файлів

### `task.md`

Єдиний файл вузла на вході — містить місію, критерій завершення і посилання на вхідні дані.

```markdown
---
created_at: 2026-06-06T10:00:00Z
budget_sec: 600                  # часовий ліміт на вузол
parent: research/collect-data    # відносно tasks/; відсутній у кореневого
deps:                             # відсутній якщо немає попередників
  - collect-data
  - fetch-sources
---
## Task
Що саме має виконати цей вузол.

## Done when
Чіткий критерій: що означає "вирішено".

## Inputs

### context
ref: ../collect-data/outputs_001.md#results

### raw-data
ref: ../fetch-sources/outputs_001.md

### instruction
Обробити лише перші 50 результатів, ігнорувати дублікати.
```

| Поле / секція | Обов'язкове | Примітка |
|---|---|---|
| `created_at` | так | ISO 8601, перше поле |
| `budget_sec` | так | секунди; wrapper-скрипт кілить процес після |
| `parent` | ні (лише якщо не кореневий) | відносно `tasks/` |
| `deps` | ні | лише siblings (id сусідів без шляху) |
| `## Task` | так | — |
| `## Done when` | так | — |
| `## Inputs` | ні | відсутній якщо немає deps і батько нічого не передає |
| підсекції `### name` | — | `ref:` якщо є файл, інакше inline |

Immutable після створення ворктрі. Якщо потрібно змінити — kill ворктрі, редагуй, restart.

Кореневий вузол — `## Inputs` заповнює людина вручну перед першим запуском.

---

### `outputs_NNN.md`

Immutable. Один файл на один успішний запуск (`outputs_001.md`, `outputs_002.md`, …). Скрипт читає файл з найбільшим номером як актуальний.

`## Summary` — обов'язкова перша секція для observability. Решта — довільні порти для наступників.

Правило даних: `ref:` якщо дані є у файлі, inline якщо немає деінде.

```markdown
---
created_at: 2026-06-06T10:05:00Z
---
## Summary
Проаналізовано 847 записів, виявлено 3 аномалії у секції payments.

## anomalies
ref: artifacts/anomalies.json

## full-report
ref: artifacts/report.md
```

| Секція | Обов'язкова | Правило |
|---|---|---|
| `## Summary` | так | одне речення, inline |
| довільні секції | ні | `ref:` якщо є файл, інакше inline |

---

### `run_NNN.md`

Immutable. Один файл на одну спробу (`run_001.md`, `run_002.md`, …) — будь-якого актора. Записується **після** завершення спроби.

```markdown
---
created_at: 2026-06-06T10:01:00Z
actor: agent
result: failed
---
## Reasoning
Спробував обробити весь датасет одразу — задача занадто велика.

## Script
exit_code: 1
stderr: Error: context length exceeded 200k tokens

## Ref
ref: outputs_001.md
```

| Поле / секція | Хто пише | Обов'язкове | Значення |
|---|---|---|---|
| `created_at` | агент/скрипт | так | ISO 8601 |
| `actor` | агент/скрипт | так | `agent` \| `engineer` \| `human` \| `auditor` |
| `result` | агент/скрипт | так | `success` \| `failed` |
| `## Reasoning` | агент/інженер | так | довільний текст |
| `## Script` | wrapper-скрипт | ні | лише якщо впав через скрипт |
| `## Ref` | агент/інженер | ні | `ref:` на `outputs_NNN.md`; відсутній якщо впав до запису |

Успішний агент: `## Ref` → `ref: outputs_NNN.md`. Інженер: `## Reasoning` містить і що вирішив, і що змінив — окремої секції змін немає. `## Ref` для інженера — відсутній або вказує на `outputs_NNN.md` якщо сам перезапустив вузол.

## Протокол spawn (розкладання вузла на підграф)

Агент вирішує "я Складений" **тільки на старті** після читання `task.md`.

**Структуру підграфу визначає виключно агент (LLM)** — оркестратор не втручається.

```
1. для кожного дочірнього:
   mkdir <node-id>/
   пише <node-id>/task.md              <- місія + deps (попередники)
2. виклич: n-cursor graph spawn <шлях-вузла>
```

**Динамічний spawn** (новий вузол під час виконання):

```
агент (будь-коли) ->
  mkdir <new-node-id>/
  пише <new-node-id>/task.md    <- deps: вказують на вже існуючі вузли
  виклич: n-cursor graph spawn <шлях-вузла>
```

Новий вузол підхоплюється оркестратором при наступному скануванні.

---

## Протокол патчу вузла що вже має залежних

Перед тим як патчити `collect-data` (вже вирішено) — інженер зобов'язаний:

```
1. знайти всі запущені ворктрі залежних
   collect-data -> analyze (running) -> synthesize (running)
2. кілл від листів до цілі (топологічний порядок):
   kill synthesize ворктрі
   kill analyze ворктрі
3. застосувати патч до collect-data у власному ворктрі
4. merge ворктрі
5. рестарт каскаду від collect-data
```

Інженер пише `run_NNN.md` (actor: engineer) з `## Reasoning` — це єдиний запис наміру і результату.

## Паралельне виконання: ворктрі

Незалежні вузли ОАГ виконуються паралельно — кожен агент у своєму git-ворктрі.

**Ворктрі-межа = межа атомарності.** Наступник стартує лише після merge попередника. Інженер оцінює стан при merge.

**Кожен патч інженера — у своєму ворктрі.** Ворктрі видаляється після завершення патчу.

### Щасливий шлях (без конфліктів)

Незалежні вузли пишуть у різні директорії — merge чисте, без конфліктів.

### Конфлікт при злитті

Якщо два ворктрі змінили один файл:

```
merge -> конфлікт -> spawn MediatorAgent(version_A, version_B, context)
                     результат = новий коміт -> продовжити виконання графу
```

**Симетрія:** конфлікт злиття = вузол що впав — той самий патерн відновлення.

### Ліміти ворктрі

Жорсткі мінімальні ліміти на старті (виконується на MacBook) — задаються при запуску системи. Черга очікування якщо ліміт вичерпано.

## Самовідновлення: агент-інженер

Інженер — **мета-рівень поза графом**, без власного стану. Викликається при `стан = помилка`.

```
EngineerAgent(run_NNN.md, full_path_from_root) ->
  GraphPatch:
    ├── замінити вузол (атомарний -> складений або навпаки)
    ├── вставити вузли (додати проміжні кроки)
    ├── перепідключити ребра
    └── змінити вхідні дані
```

Може змінювати **будь-який рівень** — від вузла що впав до кореня.

### Перевірка попередника перед стартом

```
predecessor/outputs_*.md існує AND NOT invalidated
  -> можна стартувати
  -> інакше — очікувати
```

### Каскад інвалідації

```
patch(analyze) ->
  analyze/outputs_*.md = застарілі ->
    synthesize, report = invalidated ->
      їх наступники = invalidated -> ...
```

## Збіжність: часовий бюджет

Замість ліміту спроб — **часовий бюджет**. Інженер адаптує стратегію до залишку:

- Багато часу — складна спроба
- Мало часу — швидке виправлення
- `залишок <= 0` — `стан = невирішуваний`, ескалація

---

## Ієрархія ескалації

Кожен рівень отримує **свіжий** часовий бюджет:

```
вузол впав
  -> EngineerAgent (budget: 10хв)
      -> timeout -> ескалація
          -> батько: EngineerAgent (свіжий budget: 10хв)
              -> ...
                  -> корінь: timeout
                      -> notify SeniorEngineer (людина)
```

**Максимальний час до людини:** `глибина x бюджет`

## Контракт для моніторингу

> **Ціль цього документу** — чіткий контракт по якому буде створено скрипт що в будь-який момент показує де і на якому етапі знаходиться робота.

Скрипт відновлює стан графу скануванням файлової структури:

```
для кожного tasks/**/task.md:
  визначити стан (які файли існують поруч)
  визначити залежності (deps: з task.md)
  зібрати run_NNN.md якщо є

вивести:
  - дерево вузлів зі станами
  - незавершені операції (потребують відновлення)
  - активні ворктрі
  - поточний часовий бюджет інженерів
```

## Відкриті питання (перед реалізацією)

### Рівень 1 — блокують усе інше

1. **Bootstrap** — хто і як створює кореневий `task.md`? Де живе директорія `tasks/`? Як виглядають перші `inputs.md` (від людини)?

2. **Тригер merge** — після завершення ворктрі хто ініціює злиття: wrapper-скрипт автоматично чи людина вручну?

3. **Dynamic spawn зсередини ворктрі** — вузол у ворктрі породжує підзадачі, невидимі до merge. Хто запускає їх worktrees?

### Рівень 2 — важливі дизайнерські рішення

4. **Контекстне вікно LLM** — якщо дані великі — `ref:`, а не inline.

5. **Bootstrap кореневого вузла** — `inputs.md` кореня створює людина вручну перед першим запуском.

6. **Ліміти ресурсів** — max паралельних ворктрі, max глибина графу, max розмір файлу.

## CLI контракт (`n-cursor graph`)

Конфіг: `NCURSOR_TASKS_DIR` env або `.n-cursor.json` → поле `tasks_dir`, дефолт `./tasks/`.

Всі команди підтримують `--json` для machine-readable виводу.

### Команди

```
graph status [path] [--json]
```
- без `path` → весь граф від `tasks/`
- з `path` → вузол і нащадки
- виводить: state, останній run_NNN, активний ворктрі
- exit: `0` ok | `1` вузол не знайдено

```
graph run <path>
```
1. Перевіряє що всі `deps` → `resolved`
2. Створює ворктрі
3. Запускає агента (читає `task.md`)
4. Стежить за `budget_sec` → kill при timeout
5. Пише `run_NNN.md` (агент + wrapper)
6. Завершується — merge вручну людиною

- exit: `0` agent success | `1` agent failed | `2` timeout | `3` system error

```
graph kill <path>
```
1. Визначає залежних у топологічному порядку (від листів)
2. Вбиває ворктрі залежних (kill process + delete worktree)

- exit: `0` all killed | `1` часткова помилка

```
graph invalidate <path>
```
- Створює `invalidated` sentinel
- exit: `0` | `1` вже invalidated

```
graph scan [--json]
```
- Сканує всі `task.md`, реконструює DAG
- Виводить стан кожного вузла
- exit: `0` граф чистий | `1` є вузли у стані `failed`

## Wrapper-скрипт

```
graph run <path> [--actor agent|engineer|human|auditor]
```

**Кроки:**
1. Читає `task.md` → `budget_sec`, `deps`
2. Перевіряє що всі `deps` → `resolved`
3. `git worktree add .worktrees/<sanitized-path>-<epoch> main`
4. Запускає агента (cwd = ворктрі):
   ```
   claude --system-prompt .n-cursor/system-prompt.md \
          --message "solve task at task.md"
   ```
5. Timeout monitor → SIGTERM при `budget_sec`, SIGKILL через 5s
6. Після виходу — пише `run_NNN.md`:
   - `result: success` якщо exit 0 і `outputs_NNN.md` існує
   - `result: failed` + `worktree: .worktrees/...` інакше (ворктрі залишається для debug)
7. Завершується — merge вручну людиною

**`run_NNN.md` схема (фінальна):**
```markdown
---
created_at: 2026-06-06T10:01:00Z
actor: agent
result: failed
worktree: .worktrees/research-subgraph-analyze-1749200400  # тільки якщо failed
---
## Script
exit_code: 124
stderr: Process killed after 600s

## Ref
```

## Bootstrap

```bash
# 1. Ініціалізація проекту (один раз)
n-cursor graph setup
# → .n-cursor.json, .n-cursor/system-prompt.md, tasks/

# 2. Створення кореневого вузла
n-cursor graph init my-project \
  --task "Розробити API для обробки платежів" \
  --budget-sec 3600
# → tasks/my-project/task.md

# 3. Людина доповнює ## Inputs в task.md

# 4. Запуск
n-cursor graph run tasks/my-project
```

Кореневий `task.md` — немає `parent:` і `deps:`, `## Inputs` заповнюється людиною вручну.

## Протокол агента-інженера

Інженер = той самий вузол, перезапущений з `--actor engineer`.

```
graph run <node-path> --actor engineer
```

1. Wrapper збирає контекст: `task.md` + всі `run_NNN.md` відсортовані за `created_at`
2. Запускає агента з `.n-cursor/engineer-prompt.md` (окремий system prompt)
3. Агент читає контекст, визначає причину і план дій
4. Якщо потрібен патч залежного вузла:
   - `graph kill <dep-node>`
   - Створює ворктрі, вносить зміни в `dep-node/task.md`
   - Мержить ворктрі
   - Перезапускає залежний вузол
5. Перезапускає себе (`graph run <node> --actor engineer`) якщо потрібно
6. Wrapper пише `run_NNN.md` з `actor: engineer`

Відмінності від звичайного агента:
- Інший system prompt (`.n-cursor/engineer-prompt.md`)
- Додатковий контекст: вся `run_NNN.md` історія вузла
- Дозволені операції: `graph kill`, зміни через ворктрі

## SWOT-аналіз

### Сильні сторони

- **Інкапсуляція:** батько не знає що всередині — замінюваність без змін у батьківському графі
- **Файловий стан:** безкоштовна персистентність, відновлення після збоїв, повний аудит через git history
- **Plan -> action -> fact:** будь-який збій відновлюється порівнянням plan і fact
- **LLM-first формат:** `run_NNN.md` — інженер читає і продовжує природно
- **Ворктрі = межа атомарності:** наступник стартує лише після merge — гонки неможливі за архітектурою
- **Git-native паралельність:** ворктрі — вже знайомий інструмент
- **Симетрія:** конфлікт злиття = вузол що впав — той самий патерн відновлення
- **Часовий бюджет** замість лічильника — реалістичне обмеження

### Слабкі сторони

- **Scan без індексу:** при великих графах сканування всіх `task.md` — дорого
- **Drift намірів:** після N патчів оригінальна місія розмивається (частково вирішено незмінним `## Task`)
- **MediatorAgent сам може помилитись** — хто його лагодить?
- **Масштаб ворктрі:** жорсткий ліміт на MacBook обмежує реальну паралельність

### Можливості

- `run_NNN.md` накопичує знання — можна дистилювати в кращі промпти майбутніх агентів
- Git history = безкоштовний time-travel debugging всього графу
- Файлова система може бути розподіленою (NFS, S3) — горизонтальне масштабування
- Природна інтеграція з CI/CD через git hooks

### Загрози

- **Cascade при зміні кореня:** інженер інвалідує весь граф і не вкладається у бюджет — система стоїть
- **LLM недетермінізм:** той самий вузол розкладається по-різному при перезапуску — ускладнює debugging
