# Архітектура: Динамічний Самомодифікований Граф Задач

## Назва структури

**Рекурсивний складений ОАГ** (орієнтований ациклічний граф) — із динамічним розкладом вузлів та файловим сховищем стану.

---

## Вузол

Кожен вузол або є атомарним, або розкладається на підграф — рішення приймається **динамічно при запуску** на основі вхідних даних.

```
Вузол
  ├── реалізація:
  │     ├── Атомарний  — fn(вхідні) → вихідні
  │     └── Складений  — Граф{ вхід, вузли[], ребра[], виходи[] }
  ├── стан: очікує | виконується | вирішено | помилка | невирішуваний
  ├── вхідні:  Map<портId, Значення>
  └── вихідні: Map<портId, Значення>   ← заповнюється при вирішено
```

Для батьківського вузла інтерфейс однаковий: він чекає `стан = вирішено` не знаючи що всередині (**інкапсуляція чорної скриньки**).

---

## Граф

- **ОАГ** — орієнтований, без циклів
- **Ребра** — потік даних: виходи одного вузла стають входами наступного
- **Вхідний вузол** — отримує вхідні дані батьківського складеного вузла
- **Вихідні вузли** — кілька, їх виходи зливаються у виходи батька
- Складений вузол може бути замінений атомарним і навпаки без змін у батьківському графі

```
СкладенийВузол
  └── Граф
        ├── вхід    → отримує вхідні дані батька
        ├── вузли[]: Вузол[]
        ├── ребра[]: { від: ВузолId+портId, до: ВузолId+портId }
        └── виходи[]: Вузол[]   ← кілька, злиття → виходи батька
```

## Вузол як агент

Кожен вузол — це запуск Claude. Оркестратор передає протокол через `system_prompt`, файли вузла — лише місія та дані. Агент самостійно вирішує атомарний/складений — тип не задається наперед у `місія.md`.

**Запуск:**
```
Claude(
  system_prompt = <протокол оркестратора>,
  context       = [місія.md, вхідні.md] + repair_history.md якщо є
)
```

**System prompt оркестратора** (агент ніколи не бачить у своїх файлах):
```
Твої файли: місія.md (що зробити), вхідні.md (дані).
Після виконання:
  → успіх:  запиши вихідні.md
            виклич: n-cursor graph done <шлях-вузла>
  → складно: створи підграф/<дочірній>/{місія.md, вхідні.md} для кожного
             виклич: n-cursor graph spawn <шлях-вузла>
  → помилка: запиши помилка.md
             виклич: n-cursor graph failed <шлях-вузла>
```

## Принцип план → дія → факт

Будь-яка операція в системі — агент, інженер, оркестратор — дотримується трьох кроків:

```
1. записати *-план.md   ← намір (до будь-якої дії)
2. виконати
3. записати *-факт.md   ← результат (після завершення)
```

При збої — оркестратор порівнює план і факт:
```
scan tasks/**/*-план.md:
  якщо відповідний *-факт.md відсутній → незавершена операція → відновити
```

Застосовується скрізь:

| Операція | план | факт |
|---|---|---|
| Виконання агента | `місія.md` | `вихідні.md` або `помилка.md` |
| Запуск підграфу | `операції/spawn-план.md` | `операції/spawn-факт.md` |
| Кілл залежних | `операції/kill-план.md` | `операції/kill-факт.md` |
| Патч інженера | `патчі/001-план.md` | `патчі/001-факт.md` |

## Формати файлів (LLM-first)

Всі файли — **Markdown з YAML-фронтматером**:
- фронтматер = машинозчитувані поля (ребра, мітки часу)
- тіло = вільний Markdown для LLM (міркування, контекст, опис)

**Стан вузла визначається наявністю файлів** (не полем у фронтматері):

| Які файли існують | Стан |
|---|---|
| тільки `місія.md` + `вхідні.md` | очікує |
| є активний ворктрі | виконується |
| є `вихідні.md` | вирішено |
| є `помилка.md` | помилка |
| є `знедійснений` | знедійснений |

### Файлова структура вузла

```
tasks/
  <ідВузла>/
    місія.md              ← план агента (незмінний після створення)
    вхідні.md             ← дані (незмінні після створення)
    вихідні.md            ← факт агента (з'являється при успіху)
    помилка.md            ← факт збою
    знедійснений          ← sentinel: порожній файл, наявність = знедійснений
    repair_history.md     ← append-only журнал спроб
    операції/
      spawn-план.md       ← намір перед запуском дочірніх
      spawn-факт.md       ← підтвердження
      kill-план.md        ← намір перед кілом залежних
      kill-факт.md        ← підтвердження
    патчі/
      001-план.md         ← намір інженера перед патчем
      001-факт.md         ← підтвердження патчу
    підграф/              ← якщо складений вузол
      <ідДочірнього>/
        ...
```

### `місія.md`
```markdown
---
id: вузол_7
батько: workflow_X/вузол_5
ребра:
  від: [вузол_5.вихід_a]
  до: [вузол_8.вхід_x]
---
## Оригінальна місія
Перетвори технічне завдання на покроковий план реалізації.
(ця секція незмінна — пишеться один раз)

## Критерій завершення
Структурований план з оціненими кроками.
```

### `вхідні.md` / `вихідні.md`
```markdown
---
створено_о: 2026-06-06T10:00:00Z
---
## запит
...

## контекст
...
```

### `помилка.md`
```markdown
---
час: 2026-06-06T10:05:00Z
тип: помилка_виконання
---
## Що пішло не так
...
```

### `repair_history.md` ← найважливіший для LLM
Append-only журнал — інженер читає як єдиний документ і продовжує природно.

```markdown
# Журнал відновлення

## Спроба 1 — 2026-06-06T10:01:00Z
**Міркування:** схоже проблема в параметрі X...
**Патч:** змінив вхідні дані
**Результат:** помилка — Y все ще виникає

## Спроба 2 — 2026-06-06T10:03:00Z
**Міркування:** спроба 1 не вийшла, спробую розбити вузол...
**Патч:** атомарний → складений з двох кроків
**Результат:** ...
```

Граф має **часовий вимір** — це послідовність знімків:
```
ОАГ_v1 → [ПатчІнженера] → ОАГ_v2 → [ПатчІнженера] → ОАГ_v3
```

## Паралельне виконання: ворктрі

Незалежні вузли ОАГ виконуються паралельно — кожен агент у своєму git-ворктрі.

```
основна гілка
  ├── .worktrees/вузол_5-виконання/   ← агент A
  └── .worktrees/вузол_6-виконання/   ← агент B
```

**Ворктрі-межа = межа атомарності.** Наступник стартує лише після merge попередника. Будь-який стан що агент записав у ворктрі — не потрапляє в основну гілку до merge. Інженер оцінює стан саме при merge.

### Щасливий шлях (без конфліктів)

Незалежні вузли пишуть у різні директорії — конфліктів немає:

```
агент A: tasks/вузол_5/вихідні.md  ← своя директорія
агент B: tasks/вузол_6/вихідні.md  ← своя директорія

злиття → чисте, без конфліктів
```

### Конфлікт при злитті

Якщо два ворктрі змінили один файл (наприклад інженер патчив батьківський вузол):

```
злиття → конфлікт → spawn АгентМедіатор(версія_A, версія_B, контекст)
                      результат = новий коміт
                      продовжити виконання графу
```

**Симетрія:** конфлікт злиття — це теж "вузол що впав", вирішується тим самим патерном.

## Самовідновлення: агент-інженер

Інженер — **мета-рівень поза графом**, без власного стану. Викликається при `стан = помилка`.

```
АгентІнженер(помилка, повний_шлях_від_кореня, repair_history) →
  ПатчГрафу:
    ├── замінити вузол (атомарний → складений або навпаки)
    ├── вставити вузли (додати проміжні кроки)
    ├── перепідключити ребра
    └── змінити вхідні дані
```

Може змінювати **будь-який рівень** — від вузла що впав до кореня.

### Протокол патчу вузла що вже має залежних

Перед тим як патчити вузол_1 (вже вирішено) — інженер зобов'язаний:

```
1. записати патчі/001-план.md у вузол_1   ← намір (спочатку)
2. знайти всі запущені ворктрі залежних
   вузол_1 → вузол_2 (виконується) → вузол_3 (виконується)
3. кілл від листів до цілі (топологічний порядок):
   kill вузол_3 ворктрі
   kill вузол_2 ворктрі
4. записати операції/kill-факт.md         ← підтвердження
5. застосувати патч до вузол_1
6. записати патчі/001-факт.md             ← факт
7. рестарт каскаду від вузол_1
```

Наявність `патчі/001-план.md` без `патчі/001-факт.md` при відновленні = незавершений патч → продовжити з кроку 3.

### Перевірка попередника перед стартом

Перед запуском кожного вузла — обов'язкова перевірка:
```
попередник.стан ∈ { вирішено } AND NOT знедійснений
  → можна стартувати
  → інакше — очікувати
```

### Каскад інвалідації

Зміна вузла інвалідує всіх наступників:
```
патч(вузол_6) →
  вузол_6.вихідні = застарілі →
    вузол_7, вузол_8 = знедійснені →
      їх наступники = знедійснені → ...
```

### Пам'ять

Пам'ять живе **на вузлі** де інженер вносить зміни. При зміні батьківського вузла через помилку дочірнього:
- `батько/repair_history.md` ← записується патч
- `дочірній/repair_history.md` ← посилання `triggered_parent_patch: 001`

## Збіжність: часовий бюджет

Замість ліміту спроб — **часовий бюджет**. Інженер адаптує стратегію до залишку:

```markdown
---
крайній_термін: 2026-06-06T10:10:00Z
розпочато_о: 2026-06-06T10:00:00Z
бюджет_сек: 600
---
```

- багато часу → складна спроба
- мало часу → швидке виправлення
- `залишок ≤ 0` → `стан = невирішуваний`, ескалація

---

## Ієрархія ескалації

Кожен рівень отримує **свіжий** часовий бюджет:

```
вузол впав
  → агент-інженер (бюджет: 10хв)
      → час вийшов → ескалація
          → батько: агент-інженер (свіжий бюджет: 10хв)
              → час вийшов → ескалація
                  → дідусь: агент-інженер (свіжий бюджет: 10хв)
                      → ...
                          → корінь: час вийшов
                              → сповіщення старшого інженера (людина)
```

**Максимальний час до людини:** `глибина × бюджет`

### Сповіщення старшого інженера (`звіт-старшому.md`)

```markdown
---
вузол_що_впав: root/проект_A/workflow_X/вузол_7
час_сповіщення: 2026-06-06T10:30:00Z
---
## Ланцюг ескалації
- вузол_7:    3 спроби, витрачено 10хв
- workflow_X: 2 спроби, витрачено 10хв
- проект_A:   1 спроба, витрачено 10хв

## Поточний стан графу
...

## Пропозиції наступних кроків
...
```

## SWOT-аналіз

### Сильні сторони

- **Інкапсуляція:** батько не знає що всередині — замінюваність без змін у батьківському графі
- **Файловий стан:** безкоштовна персистентність, відновлення після збоїв, повний аудит через git history
- **План → дія → факт:** будь-який збій відновлюється порівнянням плану і факту
- **LLM-first формат:** `repair_history.md` як append-only журнал — інженер читає і продовжує природно
- **Ворктрі = межа атомарності:** наступник стартує лише після merge — гонки неможливі за архітектурою
- **Git-native паралельність:** ворктрі — вже знайомий інструмент, naming convention = мютекс
- **Симетрія:** конфлікт злиття = вузол що впав → той самий патерн відновлення
- **Часовий бюджет** замість лічильника — реалістичне обмеження

### Слабкі сторони

- **Scan для відновлення стану:** немає індексу, при великих графах пошук усіх `*-план.md` → дорого
- **Drift намірів:** після N патчів оригінальна місія розмивається (частково вирішено незмінною секцією)
- **Медіатор сам може помилитись** — хто його лагодить?
- **Масштаб ворктрі:** 100+ паралельних вузлів = 100+ ворктрі на диску

### Можливості

- `repair_history.md` накопичує знання — можна дистилювати в кращі промпти майбутніх агентів
- Git history = безкоштовний time-travel debugging всього графу
- Файлова система може бути розподіленою (NFS, S3) → горизонтальне масштабування
- Природна інтеграція з CI/CD через git hooks

### Загрози

- **Cascade при зміні кореня:** інженер інвалідує весь граф і не вкладається у бюджет → система стоїть
- **LLM недетермінізм:** той самий вузол розкладається по-різному при перезапуску — ускладнює debugging
