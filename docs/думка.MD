# Архітектура: Динамічний Самомодифікований Граф Задач

## Назва структури

**Рекурсивний складений ОАГ** (орієнтований ациклічний граф) — із динамічним розкладом вузлів та файловим сховищем стану.

---

## Вузол

Кожен вузол або є атомарним, або розкладається на підграф — рішення приймається **динамічно при запуску** на основі вхідних даних.

```
Вузол
  ├── реалізація:
  │     ├── Атомарний  — fn(вхідні) → вихідні
  │     └── Складений  — Граф{ вхід, вузли[], ребра[], виходи[] }
  ├── стан: очікує | виконується | вирішено | помилка | знедійснений
  ├── вхідні:  Map<portId, Value>
  └── вихідні: Map<portId, Value>   ← заповнюється при вирішено
```

Для батьківського вузла інтерфейс однаковий: він чекає `стан = вирішено` не знаючи що всередині (**інкапсуляція чорної скриньки**).

---

## Граф

- **ОАГ** — орієнтований, без циклів
- **Ребра** — потік даних: виходи одного вузла стають входами наступного
- **Вхідний вузол** — отримує вхідні дані батьківського складеного вузла
- **Вихідні вузли** — кілька, їх виходи зливаються у виходи батька
- Складений вузол може бути замінений атомарним і навпаки без змін у батьківському графі

```
СкладенийВузол
  └── Граф
        ├── вхід    → отримує вхідні дані батька
        ├── вузли[]: Вузол[]
        ├── ребра[]: { від: ВузолId+portId, до: ВузолId+portId }
        └── виходи[]: Вузол[]   ← кілька, злиття → виходи батька
```

**Топологія живе в `task.md` кожного дочірнього** — поле `deps:`. Жодного центрального файлу графу. Оркестратор відновлює повний граф скануванням усіх `task.md`.

---

## Вузол як агент

Кожен вузол — це запуск Claude. Оркестратор передає протокол через `system_prompt`, файли вузла — лише місія та дані. Агент самостійно вирішує атомарний/складений **тільки на старті** — тип не задається наперед у `task.md`.

**Запуск:**

```
Claude(
  system_prompt = <протокол оркестратора>,
  context       = [task.md, inputs.md] + repair_history.md якщо є
)
```

**System prompt оркестратора** (агент ніколи не бачить у своїх файлах):

```
Твої файли: task.md (що зробити), inputs.md (посилання на дані).
Після виконання:
  -> успіх:    запиши outputs.md
               виклич: n-cursor graph done <шлях-вузла>
  -> складно:  запиши ops/spawn-plan-<ts>.md
               створи <дочірній>/{task.md, inputs.md} для кожного
               виклич: n-cursor graph spawn <шлях-вузла>
  -> помилка:  додай секцію у error.md
               виклич: n-cursor graph failed <шлях-вузла>
```

**Агент може читати файли будь-яких вузлів** (батька, братів, дочірніх) без обмежень — для контексту.

**Всі агенти запускаються через скрипт-обгортку** — скрипт відстежує таймаут і може кілити процес.

## Naming convention вузлів

- Дозволені символи: `a-z`, `0-9`, `-`
- Роздільник: `-`
- Унікальність: в межах батька (серед сусідів в тій самій директорії)
- Приклади: `collect-data`, `analyze-results`, `synthesize`
- `id` вузла = назва директорії (не дублюється у фронтматері)
- Атрибути фронтматеру — англійські, snake_case
- **Всі імена файлів і директорій — англійська** (обробляються скриптами)
- Заголовки секцій що парсить скрипт — англійські; секції з довільними даними — будь-яка мова

## Файлова структура вузла

Дочірні вузли живуть **безпосередньо** в директорії батька. Якщо директорія містить `task.md` — це вузол.

```
tasks/
  <node-id>/
    task.md              <- завдання (immutable після створення ворктрі)
    inputs.md            <- посилання на дані (immutable, опціональний)
    outputs.md           <- append-only: секції по успішних запусках
    outputs-v2.md        <- нова версія після re-spawn (якщо потрібно)
    error.md             <- append-only: секції Run N по кожному провалу
    invalidated          <- sentinel: порожній файл, наявність = знедійснений
    repair_history.md    <- append-only: журнал спроб інженера
    ops/
      spawn-plan-<ts>.md   <- намір перед spawn (immutable)
      spawn-fact-<ts>.md   <- підтвердження spawn (immutable)
      kill-plan-<ts>.md    <- намір перед кілом залежних (immutable)
      kill-fact-<ts>.md    <- підтвердження (immutable)
    patches/
      001-plan.md          <- намір інженера перед патчем (immutable)
      001-fact.md          <- підтвердження патчу (immutable)
      002-plan.md
      002-fact.md
    <child-node-id>/     <- дочірній вузол (якщо складений)
      task.md
      ...                <- та сама структура рекурсивно
```

**Стан вузла визначається наявністю файлів** (не полем у фронтматері):

| Які файли існують | Стан |
|---|---|
| тільки `task.md` (+ опц. `inputs.md`) | `waiting` |
| є активний ворктрі | `running` |
| є `outputs.md` і немає `invalidated` | `resolved` |
| є `error.md` | `failed` |
| є `invalidated` | `invalidated` |

## Контракт файлів — інваріанти

**Формат:** Markdown з YAML-фронтматером.

- Атрибути фронтматеру → **англійські, snake_case**
- Заголовки секцій що парсить скрипт → **англійські**
- Секції з довільними даними → **будь-яка мова**

**Immutable** (не змінюються і не доповнюються після створення):
`task.md`, `inputs.md`, `ops/*-plan.md`, `ops/*-fact.md`, `patches/*-plan.md`, `patches/*-fact.md`

**Append-only** (нові секції можна дописувати, існуючі не змінюються):
`error.md`, `repair_history.md`, `outputs.md`

> **Append-only = існуючі дані ніколи не змінюються. Нові секції можна дописувати. Послідовні записи (не паралельні) — допустимі.**

**Межа:**

- До створення ворктрі — файли вузла можна вільно редагувати і видаляти
- Після створення ворктрі — лише новий вміст, нічого не змінюється

**Синтаксис посилань** (у `inputs.md` / `outputs.md`):

```
ref: ../collect-data/outputs.md              # весь файл (відносно поточного файлу)
ref: ../collect-data/outputs.md#results      # секція за заголовком
ref: ../collect-data/outputs.md lines 5-20  # діапазон рядків
```

Шляхи відносні від поточного файлу.

## Схеми файлів

### `task.md`

```markdown
---
parent: research/collect-data   # відносно tasks/; відсутній у кореневого
deps:                            # відсутній якщо немає попередників
  - collect-data
  - fetch-sources
---
## Task
Що саме має виконати цей вузол.

## Done when
Чіткий критерій: що означає "вирішено".

## Контекст
(опціональна секція — довільна мова)
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `parent` | ні (лише якщо не кореневий) | відносно `tasks/` |
| `deps` | ні | лише siblings (id сусідів без шляху) |
| `## Task` | так | англійська |
| `## Done when` | так | англійська |

Immutable після створення ворктрі. Якщо потрібно змінити — видалити директорію і re-spawn.

Кореневий вузол без `inputs.md` — статус `waiting-for-input` — людина створює `inputs.md` вручну.

---

### `inputs.md`

Пишеться **батьком** при spawn, разом з `task.md`. Immutable.

Одна секція = одне посилання `ref:` або inline текст. Шляхи відносні від поточного файлу.

```markdown
---
created_at: 2026-06-06T10:00:00Z
---
## context
ref: ../collect-data/outputs.md#results

## raw-data
ref: ../fetch-sources/outputs.md

## instruction
Обробити лише перші 50 результатів, ігнорувати дублікати.
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `created_at` | так | ISO 8601 |
| секції | ні | довільні назви; один `ref:` або inline текст |

Опціональний файл — відсутній якщо немає `deps` і батько нічого не передає.

---

### `outputs.md`

Append-only. Секція = або inline дані, або `ref:` — агент вирішує по ситуації.

```markdown
---
created_at: 2026-06-06T10:05:00Z
---
## summary
Проаналізовано 47 джерел, виявлено 3 ключові патерни...

## report
ref: ./output/report.md

## structured-data
{"count": 42, "patterns": [...]}
```

| Поле | Обов'язкове | Примітка |
|---|---|---|
| `created_at` | так | ISO 8601 |
| секції | ні | inline або `ref:` на розсуд агента |

При re-spawn після патчу — новий файл `outputs-v2.md`. Скрипт читає файл з найбільшим номером як актуальний.

---

### `error.md`

Append-only. Одне `## Run N` = одна невдала спроба запуску агента.

**Порядок запису:** wrapper-скрипт пише технічну частину першим (після завершення агента), потім агент дописує аналіз (якщо встиг до краша).

```markdown
## Run 1 -- 2026-06-06T10:05:00Z
**exit_code:** 1
**type:** context-too-large
**Stderr:**
Error: context length exceeded...

**Analysis:**
Агент пише своїми словами що сталось і чому.

## Run 2 -- 2026-06-06T10:17:00Z
**exit_code:** 0
**type:** logical-error
**Analysis:**
Після патчу спробував X, але Y не відповів очікуваному формату...
```

| Поле | Хто пише | Обов'язкове |
|---|---|---|
| `exit_code` | wrapper | так |
| `type` | агент або wrapper | так (довільний рядок) |
| `Stderr` | wrapper | ні |
| `Analysis` | агент | ні (відсутнє при краші) |

---

### `repair_history.md`

Append-only журнал спроб інженера. Інженер читає як єдиний документ і продовжує природно.

```markdown
# Repair History

## Attempt 1 -- 2026-06-06T10:01:00Z
**Deadline:** 2026-06-06T10:11:00Z
**Reasoning:** вузол падає через некоректний формат вхідних даних...
**Patch:** patches/001-plan.md
**Result:** failed -- помилка залишилась, інша причина

## Attempt 2 -- 2026-06-06T10:13:00Z
**Reasoning:** спроба 1 не вийшла, тепер спробую розбити на підграф...
**Patch:** patches/002-plan.md
**Result:** escalated -- передав батьку
```

Якщо інженер змінює **батьківський вузол** через помилку дочірнього:

- `parent/repair_history.md` — записується патч
- `child/repair_history.md` — `**Triggered:** patch on parent <node-id>`

---

### `ops/spawn-plan-<ts>.md` / `spawn-fact-<ts>.md`

```markdown
---
created_at: 2026-06-06T10:00:00Z
nodes:
  - id: collect-data
    deps: []
  - id: analyze
    deps: [collect-data]
  - id: synthesize
    deps: [analyze]
---
## Why decomposed
Чому агент вирішив розбити на підграф.
```

```markdown
---
created_at: 2026-06-06T10:00:05Z
nodes_created: [collect-data, analyze, synthesize]
---
```

Recovery: `nodes` у плані != `nodes_created` у факті — spawn не завершився — продовжити.

---

### `ops/kill-plan-<ts>.md` / `kill-fact-<ts>.md`

Файли живуть у директорії **вузла-цілі патчу** (не вузлів що вбиваються).

```markdown
---
created_at: 2026-06-06T10:00:00Z
kill_order:
  - synthesize
  - analyze
reason: patch target collect-data
---
```

```markdown
---
created_at: 2026-06-06T10:00:30Z
nodes_killed: [synthesize, analyze]
---
```

`kill_order` != `nodes_killed` — kill не завершився — добити решту.
Kill = terminate process, then delete worktree (послідовно).

---

### `patches/001-plan.md` / `001-fact.md`

```markdown
---
created_at: 2026-06-06T10:00:00Z
target_node: collect-data
---
## Reason
Чому потрібен патч.

## Changes
Що буде змінено.
```

```markdown
---
created_at: 2026-06-06T10:02:00Z
result: success | failure
---
## Applied changes
Що фактично змінено.
```

## Принцип plan -> action -> fact

Будь-яка операція в системі — агент, інженер, оркестратор — дотримується трьох кроків:

```
1. записати *-plan.md   <- намір (до будь-якої дії)
2. виконати
3. записати *-fact.md   <- результат (після завершення)
```

При збої — оркестратор порівнює plan і fact:

```
scan tasks/**/*-plan.md:
  якщо відповідний *-fact.md відсутній -> незавершена операція -> відновити
```

Застосовується скрізь:

| Операція | plan | fact |
|---|---|---|
| Виконання агента | `task.md` | `outputs.md` або `error.md` |
| Запуск підграфу | `ops/spawn-plan-<ts>.md` | `ops/spawn-fact-<ts>.md` |
| Кілл залежних | `ops/kill-plan-<ts>.md` | `ops/kill-fact-<ts>.md` |
| Патч інженера | `patches/001-plan.md` | `patches/001-fact.md` |

---

## Протокол spawn (розкладання вузла на підграф)

Агент вирішує "я Складений" **тільки на старті** після читання `task.md` + `inputs.md`.

**Структуру підграфу визначає виключно агент (LLM)** — оркестратор не втручається.

```
1. агент пише ops/spawn-plan-<ts>.md   <- перелік дочірніх + їх залежності
2. для кожного дочірнього:
   mkdir <node-id>/
   пише <node-id>/task.md              <- місія + deps (попередники)
   пише <node-id>/inputs.md            <- ref: на outputs попередників
3. пише ops/spawn-fact-<ts>.md         <- підтвердження
```

**Динамічний spawn** (новий вузол під час виконання):

```
агент (будь-коли) ->
  mkdir <new-node-id>/
  пише <new-node-id>/task.md    <- deps: вказують на вже існуючі вузли
  пише <new-node-id>/inputs.md  <- ref: на outputs вже вирішених вузлів
  пише ops/spawn-plan-<ts>.md + spawn-fact-<ts>.md
  виклич: n-cursor graph spawn <шлях-вузла>
```

Новий вузол підхоплюється оркестратором при наступному скануванні.

---

## Протокол патчу вузла що вже має залежних

Перед тим як патчити `collect-data` (вже вирішено) — інженер зобов'язаний:

```
1. записати patches/001-plan.md у collect-data   <- намір (спочатку)
2. знайти всі запущені ворктрі залежних
   collect-data -> analyze (running) -> synthesize (running)
3. кілл від листів до цілі (топологічний порядок):
   kill synthesize ворктрі
   kill analyze ворктрі
4. записати ops/kill-fact-<ts>.md                <- підтвердження
5. застосувати патч до collect-data
6. записати patches/001-fact.md                  <- факт
7. рестарт каскаду від collect-data
```

Наявність `patches/001-plan.md` без `patches/001-fact.md` при відновленні = незавершений патч — продовжити з кроку 3.

## Паралельне виконання: ворктрі

Незалежні вузли ОАГ виконуються паралельно — кожен агент у своєму git-ворктрі.

**Ворктрі-межа = межа атомарності.** Наступник стартує лише після merge попередника. Інженер оцінює стан при merge.

**Кожен патч інженера — у своєму ворктрі.** Ворктрі видаляється після завершення патчу.

### Щасливий шлях (без конфліктів)

Незалежні вузли пишуть у різні директорії — merge чисте, без конфліктів.

### Конфлікт при злитті

Якщо два ворктрі змінили один файл:

```
merge -> конфлікт -> spawn MediatorAgent(version_A, version_B, context)
                     результат = новий коміт -> продовжити виконання графу
```

**Симетрія:** конфлікт злиття = вузол що впав — той самий патерн відновлення.

### Ліміти ворктрі

Жорсткі мінімальні ліміти на старті (виконується на MacBook) — задаються при запуску системи. Черга очікування якщо ліміт вичерпано.

## Самовідновлення: агент-інженер

Інженер — **мета-рівень поза графом**, без власного стану. Викликається при `стан = помилка`.

```
EngineerAgent(error.md, full_path_from_root, repair_history.md) ->
  GraphPatch:
    ├── замінити вузол (атомарний -> складений або навпаки)
    ├── вставити вузли (додати проміжні кроки)
    ├── перепідключити ребра
    └── змінити вхідні дані
```

Може змінювати **будь-який рівень** — від вузла що впав до кореня.

### Перевірка попередника перед стартом

```
predecessor/outputs.md існує AND NOT invalidated
  -> можна стартувати
  -> інакше — очікувати
```

### Каскад інвалідації

```
patch(analyze) ->
  analyze/outputs.md = застарілі ->
    synthesize, report = invalidated ->
      їх наступники = invalidated -> ...
```

## Збіжність: часовий бюджет

Замість ліміту спроб — **часовий бюджет**. Інженер адаптує стратегію до залишку:

- Багато часу — складна спроба
- Мало часу — швидке виправлення
- `залишок <= 0` — `стан = невирішуваний`, ескалація

---

## Ієрархія ескалації

Кожен рівень отримує **свіжий** часовий бюджет:

```
вузол впав
  -> EngineerAgent (budget: 10хв)
      -> timeout -> ескалація
          -> батько: EngineerAgent (свіжий budget: 10хв)
              -> ...
                  -> корінь: timeout
                      -> notify SeniorEngineer (людина)
```

**Максимальний час до людини:** `глибина x бюджет`

## Контракт для моніторингу

> **Ціль цього документу** — чіткий контракт по якому буде створено скрипт що в будь-який момент показує де і на якому етапі знаходиться робота.

Скрипт відновлює стан графу скануванням файлової структури:

```
для кожного tasks/**/task.md:
  визначити стан (які файли існують поруч)
  визначити залежності (deps: з task.md)
  визначити незавершені операції (*-plan.md без *-fact.md)
  зібрати repair_history.md якщо є

вивести:
  - дерево вузлів зі станами
  - незавершені операції (потребують відновлення)
  - активні ворктрі
  - поточний часовий бюджет інженерів
```

## Відкриті питання (перед реалізацією)

### Рівень 1 — блокують усе інше

1. **Bootstrap** — хто і як створює кореневий `task.md`? Де живе директорія `tasks/`? Як виглядають перші `inputs.md` (від людини)?

2. **Тригер merge** — після завершення ворктрі хто ініціює злиття: wrapper-скрипт автоматично чи людина вручну?

3. **Dynamic spawn зсередини ворктрі** — вузол у ворктрі породжує підзадачі, невидимі до merge. Хто запускає їх worktrees?

### Рівень 2 — важливі дизайнерські рішення

4. **Контекстне вікно LLM** — якщо дані великі — `ref:`, а не inline.

5. **Bootstrap кореневого вузла** — `inputs.md` кореня створює людина вручну перед першим запуском.

6. **Ліміти ресурсів** — max паралельних ворктрі, max глибина графу, max розмір файлу.

## SWOT-аналіз

### Сильні сторони

- **Інкапсуляція:** батько не знає що всередині — замінюваність без змін у батьківському графі
- **Файловий стан:** безкоштовна персистентність, відновлення після збоїв, повний аудит через git history
- **Plan -> action -> fact:** будь-який збій відновлюється порівнянням plan і fact
- **LLM-first формат:** `repair_history.md` — інженер читає і продовжує природно
- **Ворктрі = межа атомарності:** наступник стартує лише після merge — гонки неможливі за архітектурою
- **Git-native паралельність:** ворктрі — вже знайомий інструмент
- **Симетрія:** конфлікт злиття = вузол що впав — той самий патерн відновлення
- **Часовий бюджет** замість лічильника — реалістичне обмеження

### Слабкі сторони

- **Scan без індексу:** при великих графах пошук усіх `*-plan.md` — дорого
- **Drift намірів:** після N патчів оригінальна місія розмивається (частково вирішено незмінним `## Task`)
- **MediatorAgent сам може помилитись** — хто його лагодить?
- **Масштаб ворктрі:** жорсткий ліміт на MacBook обмежує реальну паралельність

### Можливості

- `repair_history.md` накопичує знання — можна дистилювати в кращі промпти майбутніх агентів
- Git history = безкоштовний time-travel debugging всього графу
- Файлова система може бути розподіленою (NFS, S3) — горизонтальне масштабування
- Природна інтеграція з CI/CD через git hooks

### Загрози

- **Cascade при зміні кореня:** інженер інвалідує весь граф і не вкладається у бюджет — система стоїть
- **LLM недетермінізм:** той самий вузол розкладається по-різному при перезапуску — ускладнює debugging

## Зведена таблиця рішень

| Аспект | Рішення |
|---|---|
| Структура | Рекурсивний складений ОАГ |
| Декомпозиція | Динамічна, тільки на старті агента |
| Хто вирішує структуру підграфу | Тільки агент (LLM) |
| Ребра | Потік даних (виходи -> входи); топологія в `deps:` кожного дочірнього `task.md` |
| Центральний файл графу | Відсутній — оркестратор сканує `task.md` дочірніх |
| Вихідні вузли | Кілька, злиття у виходи батька |
| Формат файлів | Markdown + YAML-фронтматер |
| Атрибути фронтматеру | Англійські, snake_case |
| Імена файлів і директорій | Англійська (обробляються скриптами) |
| Заголовки секцій скрипту | Англійські; довільний вміст — будь-яка мова |
| Immutable файли | `task.md`, `inputs.md`, `ops/*`, `patches/*` |
| Append-only файли | `error.md`, `repair_history.md`, `outputs.md` |
| `error.md` | Один файл, секції `## Run N`, wrapper+agent послідовно |
| Refs | `ref: path#section` відносно поточного файлу; без копій даних |
| Одна секція inputs | = один ref або inline текст |
| Стан вузла | Implicit — які файли існують |
| Протокол агента | System prompt оркестратора (не у файлах вузла) |
| Ізоляція агента | Може читати файли будь-яких вузлів без обмежень |
| Відновлення після збою | Порівняння `*-plan.md` і `*-fact.md` |
| Зберігання | Файли, кожен вузол пише свої; дані через refs (не копії) |
| Паралельність | Git-ворктрі, по одному на вузол |
| Атомарність | Ворктрі-межа: наступник стартує лише після merge |
| Злиття | Git merge; конфлікт -> MediatorAgent |
| Межа інваріантів | До ворктрі — вільно; після — тільки нові дані |
| Патч залежного вузла | Спочатку kill залежних (топологічний порядок), потім патч |
| Кожен патч інженера | У своєму ворктрі, видаляється після завершення |
| Самовідновлення | EngineerAgent — мета-рівень, необмежений доступ |
| Інвалідація | Каскад вниз по наступниках |
| Збіжність | Часовий бюджет (не кількість спроб) |
| Ескалація | Кожен рівень — свіжий budget |
| Зупинка кореня | Notify SeniorEngineer (людина) |
| Пам'ять інженера | `repair_history.md` на кожному вузлі де є відновлення |
| Оркестратор | Наразі людина вручну |
| Запуск агентів | Через скрипт-обгортку (відловлює таймаут, може кілити) |
| Іменування вузлів | Kebab-case, унікальні в межах батька; `id` = назва директорії |
| Ліміти ворктрі | Жорсткі мінімальні ліміти на старті (MacBook) |
| Моніторинг | Скрипт scan-ить файли і показує стан в будь-який момент |
