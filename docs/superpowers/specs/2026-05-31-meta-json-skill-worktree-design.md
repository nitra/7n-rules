# Дизайн: `auto.md` → `meta.json` + поле `worktree` для скілів

**Дата:** 2026-05-31  
**Статус:** Затверджено

---

## 1. Контекст

Кожен каталог скіла (`npm/skills/<id>/`) і правила (`npm/rules/<id>/`) містить файл `auto.md` з умовою автоактивації.  
Для скілів — це машинно-парситься в `auto-skills.mjs`, для правил — поки чиста проза-документація.

Виникла нова потреба: позначити, чи скіл має виконуватися в ізольованому git-worktree. Це **не можна** покласти в `auto.md` (вільний рядок без структури), а мати другий службовий файл — поганий прецедент.

Паралельно виявилася асиметрія форматів: rules і skills використовують один і той самий механізм файлу, але в різних форматах. Уніфікація настала.

---

## 2. Ціль

1. Замінити `auto.md` на структурований `meta.json` у кожному `npm/skills/<id>/` та `npm/rules/<id>/`.
2. Додати поле `worktree` для скілів — булевий прапорець, який декларує, що скіл виконується в окремому git-worktree.
3. При синку в `.cursor/skills/` вшивати/прибирати ідемпотентну worktree-інструкцію в копію `SKILL.md` (агент читатиме її при виконанні).
4. Забезпечити можливість запускати кілька `worktree: true`-скілів паралельно без взаємних перешкод: кожен скіл отримує власний ізольований checkout, а спільний `withLock` (git-common-dir) серіалізує лише операції в межах одного ключа — різні скіли не блокують одне одного.

---

## 3. Формат `meta.json`

### 3.1 Skills — `npm/skills/<id>/meta.json`

```json
{
  "auto": "завжди",
  "worktree": true
}
```

| Поле | Тип | Обовʼязкове | Значення |
|------|-----|:-----------:|---------|
| `auto` | `"завжди"` \| `string[]` | ні | Умова автоактивації (дзеркало нинішнього `auto.md`). Відсутнє поле = opt-in (скіл не додається автоматично). `string[]` — масив ідентифікаторів правил, при яких скіл включається. |
| `worktree` | `boolean` | **так** | `true` — скіл виконується в ізольованому git-worktree; **заборонено** запускати паралельно (один інстанс за раз). `false` — звичайний режим, worktree не потрібен. |

Валідується JSON-схемою `npm/schemas/skill-meta.schema.json`.

### 3.2 Rules — `npm/rules/<id>/meta.json`

```json
{
  "auto": "якщо в проекті є хоч один Cargo.toml"
}
```

| Поле | Тип | Обовʼязкове | Значення |
|------|-----|:-----------:|---------|
| `auto` | `string` | ні | Текстовий опис умови автоактивації (переноситься 1:1 з нинішнього `auto.md`). Наразі документація — `auto-rules.mjs` логіку не змінюємо (Spec 2). |

Валідується JSON-схемою `npm/schemas/rule-meta.schema.json`.

> **Примітка:** Дві окремі схеми (`skill-meta` / `rule-meta`) — навмисно. `rules` не мають `worktree`; `skills` мають обовʼязкове `worktree`. Спільна схема потребувала б умовних `if/then`, що ускладнює `check`-логіку без користі.

## 4. Worktree-прапорець: семантика та інструкція в `SKILL.md`

### 4.1 Принцип

`worktree: true` — **генеративні** скіли: створюють зміни з детермінованого джерела (правила, survived-мутанти, deps-latest, ADR-чернетки). Їм байдуже, що в робочому дереві головного checkout — worktree на чистій гілці ізолює результат.

`worktree: false` — **реактивні** скіли (лінтують незакомічені зміни поточного checkout) і **read-only** скіли. Worktree відрізав би їх від того, що вони мають перевіряти/читати.

### 4.2 D2-sync: ідемпотентна секція в копії `SKILL.md`

При синку (`n-cursor sync-skills` / `syncSkills`):

- якщо `worktree: true` — вставляє/оновлює блок між маркерами в `.cursor/skills/n-<id>/SKILL.md`:
  ```markdown
  <!-- n-cursor:worktree:start -->
  > **Worktree:** виконуй цей скіл в окремому git-worktree (`git worktree add`); **не** запускай паралельно — один інстанс за раз.
  <!-- n-cursor:worktree:end -->
  ```
- якщо `worktree: false` — видаляє блок між маркерами (якщо є).
- `meta.json` у `.cursor/skills/` **не копіюється** (залишається пакетно-внутрішнім).
- Операція ідемпотентна: повторний синк перезаписує блок, не дублює.

### 4.3 Заборона паралельного запуску

`worktree: true` → агент не запускає цей скіл паралельно. Механізм:
1. Інструкція в `SKILL.md` (D2-sync) — читається агентом.
2. `withLock` (наявний) із спільним `cacheDir` (git-common-dir) серіалізує важкі команди всередині скіла — незалежно від кількості worktree.

## 5. Міграція `auto.md` → `meta.json`

### 5.1 Skills (9 файлів)

| Скіл | `auto` | `worktree` | Причина |
|------|--------|:----------:|---------|
| `fix` | `"завжди"` | `true` | Генеративний: структурні правки (`@nitra/cursor fix`, `bun i`, `oxfmt`) не залежать від незакомічених змін. Worktree ізолює результат на гілку; кілька `fix`-агентів можуть паралельно розкладати різні скоупи. |
| `taze` | `["bun"]` | `true` | Ідеальний кандидат: потребує чистого дерева як передумову (worktree її задовольняє автоматично); dep-bump-и природно живуть на окремій гілці; не залежить від незакомічених змін головного checkout. |
| `coverage-fix` | `["js-lint"]` | `true` | Генеративний: пише нові тести на основі survived-мутантів (детермінований список). Worktree ізолює тестові правки; паралельно з іншими скілами не конфліктує по файлах. |
| `fix-tests` | `["js-lint"]` | `true` | Аналогічно `coverage-fix`: дописує тести за survived-мутантами з `COVERAGE.md`. Worktree дає чисту гілку для pull request з тестами. |
| `adr-normalize` | `["adr"]` | `true` | Генеративний: нормалізує `docs/adr/`-чернетки через LLM (джерело — існуючі файли, незалежно від незакомічених змін). `git diff` у worktree показує тільки ADR-правки без шуму. |
| `lint` | `"завжди"` | `false` | **Реактивний**: лінтить саме незакомічені зміни поточного checkout — ті, що агент щойно зробив. `git worktree add` створює checkout без цих змін, тобто worktree відрізає скіл від його предмету. Worktree тут шкодить, не допомагає. |
| `start-check` | `["bun"]` | `false` | Не пише в репо (відкочує side-effects); головний ризик у worktree — **конфлікт портів**: якщо паралельно біжить головний checkout, два сервери займають ті самі порти. Worktree не дає ізоляції по мережі. |
| `llm-patch` | `"завжди"` | `false` | **Read-only**: аналізує CWD і пише артефакт лише в `/tmp`. Нічого ізолювати в репо — worktree лише додає overhead (`bun i` в новому checkout). |
| `publish-telegram` | `"завжди"` | `false` | **Read-only**: формує текст із поточного контексту і надсилає в Telegram. Жодних змін у репо — worktree безглуздий. |

Після створення `meta.json` — видалити `auto.md` у кожному каталозі скіла.

### 5.2 Rules (29 файлів)

Переносимо текст `auto.md` в `meta.json.auto` (рядок-проза, 1:1). Для правил без `auto.md` — `meta.json` без поля `auto` (або з порожнім полем за рішенням `check`-правила).

Приклади:
- `rust/auto.md`: `"якщо в проекті є хоч один Cargo.toml"` → `{ "auto": "якщо в проекті є хоч один Cargo.toml" }`
- `abie/auto.md`: `"якщо в кореневому package.json..."` → `{ "auto": "якщо в кореневому package.json..." }`

`auto-rules.mjs` і його логіку **не змінюємо** (Spec 2).

Після створення `meta.json` — видалити `auto.md` у кожному каталозі правила.

## 6. Зміни в коді

### 6.1 `npm/scripts/auto-skills.mjs`

- `parseSkillAutoSpec`: читає `meta.json` замість `auto.md`; парсить `meta.json.auto` (ті самі три варіанти: `"завжди"` / `string[]` / відсутнє).
- `discoverSkillAutoActivation`: без змін логіки, лише джерело даних.
- Тести в `auto-skills.test.mjs`: фікстури переходять на `meta.json`.

### 6.2 `npm/bin/n-cursor.js` → `syncSkills`

- Замінити `if (file === 'auto.md') continue` на `if (file === 'meta.json') continue` (не копіюємо в `.cursor/skills/`).
- Після копіювання `SKILL.md` — викликати `injectWorktreeBlock(skillDir, metaJson.worktree)`:
  - читає `meta.json` скіла з пакету;
  - вставляє/видаляє D2-блок у щойно скопійованому `SKILL.md`.

### 6.3 `npm/schemas/`

Нові файли:
- `npm/schemas/skill-meta.schema.json` — схема skills `meta.json` (обовʼязкове `worktree: boolean`, опційне `auto`).
- `npm/schemas/rule-meta.schema.json` — схема rules `meta.json` (опційне `auto: string`).

### 6.4 Правило / `check`

Нова або розширена перевірка (`npm/rules/npm-module/` або окреме правило):

- Кожен `npm/skills/<id>/` має валідний `meta.json` (відповідає `skill-meta.schema.json`).
- Кожен `npm/rules/<id>/` має валідний `meta.json` (відповідає `rule-meta.schema.json`).
- Жодного `auto.md` не існує (`auto.md` заборонений після міграції).

### 6.5 Документація

- `scripts.mdc` рядки про структуру скіла: замінити `auto.md` на `meta.json`.
- `README.md` у `npm/`: оновити опис `skills/` структури.

---

## 7. Out of scope (Spec 2)

- Рефакторинг `auto-rules.mjs`: перехід з захардкодженої мапи на читання `rules/*/meta.json.auto` з машинно-парситься DSL-умовами. `auto-rules.mjs` залишається незмінним у цьому spec.
- Split `n-lint` на легкий (агентський) і важкий (CI + worktree) — окремий spec.

---

## 8. Change-файл

`npm/.changes/` — один `change --bump minor --section Changed` з описом переходу на `meta.json` + `worktree`.
