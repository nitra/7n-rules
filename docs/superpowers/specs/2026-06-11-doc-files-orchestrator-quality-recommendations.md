# Рекомендації по оркестратору doc-files: підвищення якості документації

**Дата:** 2026-06-11
**Статус:** аналіз пілота + рекомендації. **Tier 0 (R1–R2) і Tier 1 (R3–R7) реалізовано** 2026-06-11 (тести + лінт зелені; валідовано на пілотних доках — див. §5). Tier 2 (R8–R9) — на Інкремент 1.
**Базис:** рев'ю 10 пілотних замін (`git diff d6c23596..e2232d07`), читання коду оркестратора (`npm/skills/doc-files/js/*`), спек `docs/superpowers/specs/2026-06-10-doc-files-local-only-pipeline.md`, план `docs/superpowers/plans/2026-06-11-doc-files-regen-rollout.md`.

## 0. Підсумок

Пілот дав **10/10 OK з det-score=100** — і водночас **просів у фактичній точності**. Det-скор міряє форму, не правду (це чесно зафіксовано у спеку), тож «100» тут не сигнал якості. Найгірше: знайдено **два детерміновані баги екстракторів**, які перетворюють механізм «обов'язкових анкорів/маркерів» на генератор **впевнено хибних фактів** — і вони отруюватимуть так само Інкремент 1, бо той перевикористовує ті самі анкори й маркери.

**Вердикт Фази 1:** якість Кроку 0 **мілка** для добивання беклогу зараз → **Варіант 2b (чекати Інкремент 1)**. Але два баги екстракторів (R1, R2) треба полагодити **до будь-якого наступного прогону** — вони дешеві й корінь найдорожчого класу помилок.

## 1. Що показав пілот (докази)

10 перегенерованих док: `adr/js/hooks`, `bun/js/layout`, `capacitor/js/platforms`, `capacitor/fix`, `ci4/fix`, `docker/js/lint`, `adr/js/hooks`, abie-`env_dns`/`hc_pairing`/`ua_http_route`/`ua_node_selector`. Старі (хмарні) доки були по 250–400 рядків з сигнатурами; нові local-only — 40–90 рядків. **Скорочення — за дизайном** (style-промпт забороняє сигнатури/типи). Проблема не в довжині, а в трьох класах дефектів:

### 1.1 Впевнено хибні факти (найгірший клас)

| Дока | Хибне твердження | Корінь |
|---|---|---|
| `adr/js/hooks.md` Огляд | «використовує конфігурацію з файлу `.local.json`» | анкор-баг R1 |
| `capacitor/js/platforms.md` Огляд | «спирається на конфігурації, визначені у файлі `.config.json`» | анкор-баг R1 |
| `capacitor/js/platforms.md` Гарантії | «Кешує результати в межах одного прогону» | маркер-баг R2 |

Це **не випадковість моделі**. Перевірено детерміновано:

```
$ node -e "extractAnchors(hooks.mjs).configRefs"      → ['.local.json']
$ node -e "extractAnchors(platforms.mjs).configRefs"  → ['.config.json']
$ extractFacts(platforms.mjs).markers.caches          → true
```

- `hooks.mjs` має літерал `.claude/settings.local.json`; `CONFIG_REF_RE` (`docgen-extract-anchors.mjs:20`) через провідний `\.` вихоплює **частину** імені — `.local.json` (неіснуючий файл). Аналогічно `capacitor.config.json` → `.config.json`.
- Цей фейковий анкор підставляється у промпт блоком «АНКОРИ ДО ОБОВ'ЯЗКОВОГО ВКЛЮЧЕННЯ: Конфіги, на які спирається код: .local.json» — і модель **слухняно** пише його як факт. Тобто механізм анкорів **підсилює** баг екстрактора до впевненого брехливого твердження.
- `platforms.mjs` має `new Map()` (акумулятор `byPath`, не кеш); `CACHE_RE` (`docgen-extract.mjs:44`) матчить `new Map()` → `caches=true` → `guaranteesFromMarkers` детерміновано друкує фальшиву гарантію кешування.

### 1.2 Generic-Огляд, який проходить скоринг

- `bun/js/layout.md`: «Файл надає інструмент для перевірки відповідності даних визначеному контракту.»
- `capacitor/fix.md`/`ci4/fix.md` Public API: «check — виконує загальну перевірку.»

Style-промпт і критик мають **закритий список заборонених фраз** («забезпечує перевірку», «виконує валідацію», «застосовує логіку»), але **парафрази** («перевірка відповідності даних», «валідація даних», «інспектує та збирає дані») його обходять, а `scoreDoc` їх взагалі не штрафує → score=100 → best-of-2 не вмикається. Корінь — Огляд генерується **першим, з самого факт-листа без коду** (`docgen-prompts.mjs:67`), тож моделі нема з чого бути конкретною.

### 1.3 Витік внутрішніх символів у Поведінку/API

`capacitor/js/platforms.md` у Поведінці перелічує `extractNitraObjectBodySource`, `nitraObjectBodyStringAllowsCocoaPodsExempt`, `pathJsonShowsNitraCocoapodsExempt` — це **внутрішні** (не експортовані) функції. Поведінковий промпт для multi-export просить «для кожної публічної функції», але не обмежує список лише `facts.exports`, а `scoreDoc` ловить внутрішні імена тільки в Огляді/Гарантіях і лише в бектіках.

### 1.4 Суржик/русизми

`platforms.md`: «пропуская директорії» (рос., має бути «пропускаючи»). Критерій критика згадує англіцизми, але це не детермінована перевірка й до скору не входить.

## 2. Рекомендації

Три тири за «вартість↔ефект». **Tier 0 — обов'язково перед наступним прогоном.**

### Tier 0 — детерміновані баги (≈1 год, 0 токенів, лагодить впевнено-хибні факти)

**R1. `CONFIG_REF_RE` не має різати частину імені файлу.**
`docgen-extract-anchors.mjs:20`. Зараз `/\b(\.[a-z][\w.-]*\.json)\b/gi` на `settings.local.json` дає `.local.json`. Захоплювати **повне** ім'я файлу (без провідного обов'язкового `\.`), напр. `/(?<![\w/.])([\w][\w.-]*\.json)\b/g`, і додати **валідацію анкора**: лишати тільки токени, які присутні в `src` як цілісний шлях-літерал (`src.includes`). Анкор, що не є дослівним підрядком джерела, **відкидається** — це й загальний інваріант (див. R5).

**R2. `CACHE_RE` не має вважати `new Map()` кешем.**
`docgen-extract.mjs:44`: прибрати `new Map()` з патерна; вимагати, щоб «кеш» походив від **іменованого** маркера (ідентифікатор містить `cache`/`memo`, або патерн read-before-write по ключу), а не будь-якого `Map`/`cache`-підрядка. Усуває фальшиву гарантію «Кешує результати».

> R1+R2 чинні і для Інкремента 1 (§4.2 спека збагачує саме маркери/анкори) — без них AST-шар успадкує брехню.

### Tier 1 — оркестрація (середня вартість, в межах Кроку 0/раннього Інкремента 1)

**R3. «Огляд» — ОСТАННІМ, з готової Поведінки.** Це рішення вже зафіксоване у спеку (§3: «Огляд — узагальнення перевіреного»), але ще не реалізоване. Витягнути наперед дешево: переставити порядок у `orchestratedDoc` і подати текст Поведінки у вікно Огляду замість голого факт-листа. Лікує 1.2 у корені.

**R4. Generic-детектор на парафрази + штраф у scoreDoc.** Замінити exact-blocklist на детермінований детектор шаблону «<дієслово-загальник> <абстрактний-іменник>» (валідація/перевірка/обробка даних/контракту/логіки) і **штрафувати** його в `scoreDoc` (новий код `generic-overview`). Тоді generic-Огляд опускає score < порогу → best-of-2 реально вмикається.

**R5. Анкор-faithfulness як скорована перевірка (двобічна).** Додати у `scoreDoc`/майбутній `verify`: (а) **покриття** — кожен інжектований анкор присутній у відповідній секції; (б) **валідність** — кожен анкор є дослівним підрядком `src`. (б) ловить R1-клас навіть якщо екстрактор колись знову зрегресує. Обидва — у score.

**R6. Поведінка/API лише з `facts.exports`.** Обмежити список у поведінковому промпті строго експортованими іменами; додати det-перевірку: бектік-токен у секції «Публічний API» ∉ `internalSymbols`. Лікує 1.3.

**R7. Суржик/русизм-лінт.** Маленький словник частих русизмів (`пропуская`, `наступний раз`, `відповідаючий`, …) → штраф у score. Дешево, ловить 1.4 і латиницю-в-кирилиці.

### Tier 2 — структурний (Інкремент 1, уже в плані)

**R8. claims→render з evidence-фільтром — головний структурний лік** від втрати конкретики («які саме файли/конфіги читає») та від хибних фактів: рендерер фізично не має звідки взяти факт поза схваленими claims, а кожен claim звіряється дослівним `evidence`-підрядком + AST. Підтвердити пріоритет; **R1, R2 мають увійти раніше**, бо claims-вікно теж містить анкори/маркери.

**R9. Truth-проби у `bench` (регресійний запобіжник).** До det-метрик додати крихітний набір тверджень «must-contain / must-NOT-contain» на еталонних файлах: напр. дока `platforms.mjs` **не** містить «кеш»; дока `hooks.mjs` **не** містить «.local.json» і **містить** «settings.json». Це єдиний дешевий спосіб зловити регрес впевнено-хибних фактів, який score=100 пропускає.
