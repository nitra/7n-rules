# doc-files: local-only конвеєр документації будь-якої складності

**Дата:** 2026-06-10
**Статус:** затверджено (інтерактивна дизайн-сесія, всі рішення зафіксовано нижче)
**Зв'язані ADR:** `260610-0643` (Tier 1 через omlx), `260610-1349` (JS-owned tool-free loop), `260610-2228` (рішення цього спека)

## 1. Мета

Файлова документація для кодових файлів **будь-якої складності** генерується **виключно локальною моделлю** (`omlx/mlx-community--gemma-4-e2b-it-4bit`, MLX-сервер `http://127.0.0.1:8000`, Mac M2 8GB). Хмарних викликів у конвеєрі немає — cloud-код видаляється повністю.

Принцип: складність поглинає **JS-оркестрація** (AST-декомпозиція, детермінована верифікація, точкові виправлення, кілька викликів LLM на файл), а не потужність моделі.

Емпірична підстава (виміряно в цьому репо):

- та сама модель, ті самі файли: one-shot цілим файлом — 47% якості, секційна оркестрація — 92% → залежність «якість ↔ розмір вікна» порогова, не плавна;
- `sym`-кореляція з якістю −0.651 — лише на whole-file промптах; зріз окремої функції повертає задачу в уже розв'язаний клас (~89%);
- «факти в промпті» — головний важіль якості (+15 п.п. у транспортному A/B), і його можна застосовувати рекурсивно.

## 3. Конвеєр (цільовий стан)

```
файл → юніти + call-graph + анкори + маркери        (AST, 0 токенів)
для кожного юніта (bottom-up в інкременті 2):
  1. claims (1 виклик, few-shot) → evidence-фільтр + AST-звірка типізованих     [0 ткн]
  2. render best-of-2 зі схвалених claims (2 дрібні виклики) → det-вибір        [0 ткн]
  3. det-verify: анкор-покриття, identifier-whitelist, дублі, рендер-лінт       [0 ткн]
     → конкретні фейли → micro-refine (≤2 дрібні виклики, кожен про одне місце)
  4. сигнал (фейл після refine ∨ юніт понад поріг) → claims ×2 self-consistency → re-render
  5. усе ще фейл → degraded: score+issues у frontmatter, ідемо далі
секції файлу:
  «Гарантії поведінки» — детерміновано з AST-маркерів (0 викликів)
  «Публічний API»      — рендер із claims експортів (1 дрібний виклик)
  «Огляд»              — ОСТАННІМ, з уже готових описів поведінки (1 виклик)
```

Порядок секцій принциповий: «Огляд» — це узагальнення перевіреного, а не вгадування з голого факт-листа. Це лікує зафіксований дефект «хибний Огляд» на складних файлах.

## 4. Компоненти

### 4.1 Транспорт — `npm/lib/llm.mjs` (Крок 0)

- Єдина точка `callLlm(messages, model, opts)`:
  - роутинг **за префіксом model-id**: `isOmlxModel(model)` → `callOmlx` (прямий HTTP); інакше → `pi` CLI. Env-перемикач `N_CURSOR_DOCGEN_BACKEND` видаляється — конвенція з `npm/lib/models.mjs` стає єдиним механізмом;
  - дефолт doc-files: `N_CURSOR_DOCGEN_MODEL` → `resolveModel('min')` → `omlx/mlx-community--gemma-4-e2b-it-4bit` (щоб без env локальний шлях ішов напряму в omlx, а не через pi: виміряно +46% часу через pi CLI);
  - опційний wire-trace: `N_CURSOR_LLM_TRACE=<file>` → append JSONL `{ts, backend, model, ms, promptChars, outChars, ok}` (підстава: ADR 260610-1516/1524).
- `omlxHealthCheck(opts)`: мінімальний chat-виклик (`max_tokens: 1`). Розрізняє:
  - сервер недоступний (`curl` fail) → `{ok:false, reason:'down'}`;
  - **memory-guard** (`does not fit under the memory ceiling`) → `{ok:false, reason:'memory-guard'}` — модель 3.5GB не влазить у динамічну стелю зайнятої 8GB машини. Це «машина зайнята, прогін відклади», а не «модель погана»; без health-check масовий прогін дав би сотні хибних «✗».
- Бюджет викликів живе у `pipeline.mjs` (Інкремент 1), не в транспорті.

### 4.2 Юніт-шар — `units.mjs` + `units-js.mjs` / `units-vue.mjs` / `units-py.mjs` (Інкремент 1)

Мовно-агностична модель юніта:

```js
{ name, kind: 'function'|'class'|'const', exported: boolean,
  span: {start, end}, body: string, calls: string[],   // внутрішні виклики
  doc: string,            // JSDoc / docstring
  anchors: {...} }        // анкори в межах span
```

- **js/mjs/ts** — `oxc-parser` через наявний `npm/scripts/utils/ast-scan-utils.mjs` (вже пряма залежність, домашній патерн лінт-сканерів). Юніти = top-level функції/класи/const-функції; ребра call-graph — виклики внутрішніх імен у тілі.
- **vue** — детермінований витяг `<script>`/`<script setup>`-блока (без `@vue/compiler-sfc` як залежності) → той самий oxc; `defineProps`/`defineEmits` → факти «Інтерфейс компонента»; template дає анкори (імена компонентів/подій), не юніти.
- **py** — `spawnSync('python3', ['-c', PY_AST_SCRIPT, file])`: stdlib `ast` віддає JSON юнітів (def/class, діапазони, docstring, виклики). Нуль npm-залежностей; якщо `python3` відсутній — файл одразу degraded-шляхом whole-file (як міст).
- Юніт-шар замінює regex-екстрактори (`extractExports`, `precedingJsDoc`, `extractInternalSymbols`) і збагачує маркери: читані env-змінні, CLI-прапори, spawn-команди, тексти `throw` — це нові анкори.

### 4.3 Claims-фаза — `claims.mjs` (Інкремент 1)

Виклик на юніт: вікно = тіло юніта + його doc + анкори span + few-shot приклад. Вихід — JSON-масив атомарних тверджень:

```json
[
  {
    "text": "повертає false, якщо файл не існує",
    "type": "returns|reads-env|writes-fs|calls|condition|free",
    "evidence": "if (!existsSync(p)) return false"
  }
]
```

Валідація (0 токенів):

- `evidence` нормалізовано (пробіли) шукається в тілі юніта `includes`-ом → не знайдено → твердження **відкидається**;
- типізовані claims додатково звіряються з AST-фактами (`returns` ↔ return-вузли, `reads-env` ↔ env-сканер, `writes-fs` ↔ fs-маркери) → суперечність → відкидається;
- невалідний JSON → 1 retry (толерантний парсер: витяг першого `[...]`-блока);
- метрики фази (частка відкинутих, JSON-фейли) пишуться у bench-звіт — це det-проксі семантичної якості.

### 4.4 Render-фаза — `render.mjs` (Інкремент 1)

- Вікно = **лише** схвалені claims (без коду) + стиль + few-shot. Рендерер фізично не має звідки взяти новий «факт».
- **Best-of-2 завжди** (виклики дрібні): два семпли (temperature 0.2 / 0.5), det-вибір — менше лінт-порушень, повніше анкор-покриття.
- `assemble()` — фіксований порядок секцій; `stripSignatures`/`stripSection` лишаються як пост-обробка.

### 4.5 Det-verify — `verify.mjs` (Інкремент 1; спадкоємець `scoreDoc`)

Усі перевірки 0 токенів:

- **анкор-покриття**: кожен анкор E1 присутній у відповідній секції;
- **identifier-whitelist**: кожен \`бектік-токен\` у виході існує в src / claims / анкорах — ловить вигадані ідентифікатори і «протікання» few-shot прикладу (його foo/bar-імен немає в src);
- **дублікати між секціями**: n-gram overlap «Огляд» ↔ «Гарантії» (зафіксований дефект);
- **рендер-лінт фейлів gemma**: обірване останнє речення (без крапки), латиниця всередині кириличного слова (`моделіне`, `applys`), generic-блекліст («забезпечує перевірку», «виконує валідацію», «застосовує логіку»), `##`/`###` всередині секції, дослівні сигнатури;
- наявні перевірки `scoreDoc` (no-overview, short-behavior, cache-hallucination, internal-name) — мігрують сюди.

### 4.6 Micro-refine — `refine.mjs` (Інкремент 1)

На кожен конкретний det-фейл — один вузький виклик: цитується речення/місце + інструкція («вплети дослівно анкор X», «перепиши без англіцизму»). Не регенерація секції. Стеля 2 refine на юніт, далі — ярус 4 (self-consistency) або degraded.

### 4.7 Адаптивний self-consistency (Д2, Інкремент 2)

Сигнали: det-фейл після refine **або** юніт понад поріг розміру. Дія: друга незалежна claims-генерація (інша temperature) → перетин тверджень за збігом evidence (детерміновано); твердження поза перетином відкидаються → re-render. Платять лише проблемні юніти (~10–20%).

### 4.8 Few-shot (Е1, Інкремент 1)

По одному компактному прикладу на тип виклику (claims-JSON, render, мікро-резюме) у system-частині, статично в промпт-модулі. Домен прикладів — вигаданий (foo/bar), щоб identifier-whitelist автоматично ловив копіювання змісту прикладу. Ціна ~+300–600 ткн промпта ≈ +1–2.5 с/виклик на MLX.

### 4.9 Bottom-up мікро-резюме (А2, Інкремент 2)

- Топологічний обхід call-graph від листків; цикли схлопуються в одну групу.
- Мікро-резюме хелпера = claims→render у 1–2 речення (найменші вікна конвеєра).
- У вікні батьківського юніта тіла хелперів **замінюються** резюме: `ВІДОМО: parseJsDoc — …`. Рекурсивне застосування виміряного важеля «факти в промпті».
- Вмикається за порогом (тіло > N рядків ∨ викликів хелперів > M; пороги визначає bench інкремента 1; робоча гіпотеза N=80, M=3).
- Гігантська функція без хелперів → зріз по top-level блоках AST.

### 4.10 Degraded-механіка (Б1, Крок 0)

Розширений frontmatter (back-compat: старі доки без нових полів валідні):

```markdown
---
docgen:
  source: npm/lib/llm.mjs
  crc: a3f1c9e0
  score: 55
  issues: short-behavior,anchor-miss:N_CURSOR_LLM_TRACE
---
```

- `score` пишеться завжди, коли відомий; `issues` — лише коли `score < threshold` (70).
- **Гейт не блокує**: CRC свіжий → `check --git` проходить. Слабкість моделі не блокує завершення задач.
- `doc-files check --degraded` — список свіжих док зі `score < threshold` (інформаційний, exit 0).
- `doc-files gen --retry-degraded` — перегенерувати лише їх (після покращення конвеєра / на вільній машині).

### 4.11 Кеш юнітів (Інкремент 2)

`node_modules/.cache/nitra-doc-files/<crc-юніта>.json` → `{claims, microSummary}`. Гіт-ігнорований відновлюваний артефакт. Наслідки:

- повторний прогін після правки перегенерує лише змінені юніти (типово 1–2) — 30–60 с/файл замість хвилин; критично, бо doc-files — обов'язковий крок кожної задачі;
- `doc-aggregate` читає кеш як структурований вхід (claims експортів), за відсутності — перегенерує.

### 4.12 Bench — `doc-files bench` (Інкремент 1)

Відновлення загубленого при розбитті `docgen` A/B-скрипта як постійної команди.

- Сет: 8-сет попередніх бенчів + виміряно-проблемні `commands.mjs` (sym=15), `safety.mjs` (sym=17) + 1 vue + 1 py (~12 файлів).
- Метрики (всі det): score v2, % анкор-покриття, identifier-порушення, лінт-фейли, **% відкинутих claims**, **AST-суперечності**, % JSON-фейлів, викликів/файл, с/файл.
- Кожен механізм (зрізи, claims, refine, best-of, few-shot, bottom-up) — окремий прапор, щоб міряти внесок ізольовано (як E1–E4).
- Звіт JSON у `benchmarks/doc-files/results/` + порівняння з попереднім прогоном.

## 6. Інкременти

1. **Крок 0** (одразу): `npm/lib/llm.mjs` (префікс-роутинг, health-check, trace) + видалення cloud-шляхів + degraded-механіка в gen/check + дефолт моделі omlx напряму.
2. **Інкремент 1**: юніт-шар (4 мови) + плоскі per-export зрізи + claims→render + det-verify/micro-refine + render best-of-2 + few-shot + `bench` → **прогін бенча**.
3. **Інкремент 2**: bottom-up мікро-резюме + адаптивний self-consistency + кеш юнітів → бенч → фіксація порогів.
4. **Хвости**: `--retry-degraded` масовий прогін на вільній машині; тюнінг за бенчем.

## 7. Видаляється

- `DEFAULT_CLOUD_MODEL`, параметр `cloudModel`, pre-route `sym >= symThreshold`, обидві cloud-ескалації в `generateDoc`;
- `N_CURSOR_DOCGEN_BACKEND`, `callOmlxMessages` (роутинг тільки за префіксом);
- `critiqueRefineSection`, `criticMessages`, `refineMessages` у поточному вигляді (замінюються micro-refine);
- E4 best-of-2 цілого документа (замінюється render best-of-2 на юніт);
- regex-екстрактори після переходу юніт-шару на AST (Інкремент 1).
