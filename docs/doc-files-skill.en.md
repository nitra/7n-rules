# Skill `doc-files`: per-file behavioral documentation with a local-only pipeline

## Core Concept

This mechanism ensures automated, locally managed synchronization of documentation with source code, avoiding high costs associated with cloud LLM queries. It guarantees deterministic relevance checking using CRC32 and utilizes JavaScript orchestration to manage the complex generation process. If generation quality drops, instead of an error, the document is created with a marker, indicating a visible technical debt.

## Problem

Code documentation lies as soon as it falls behind the code—and it's impossible to manually keep hundreds of `md` files synchronized with the code. Generating everything with a cloud model is expensive: the first run of this repo $\approx 26M$ tokens ($\sim\$300–500$). Generating with a weak local model "straight up" is low quality: a one-shot whole file on `gemma-4-e2b` yields 47% quality compared to 92% in sectional orchestration (measured on the same files).

`doc-files` solves both problems simultaneously: **deterministic relevance checking** (CRC) + **local-only generation pipeline**, where complexity is managed by JS orchestration, not model size.

## Key Principles

1. **Orchestrates JS, not model.** Queueing, batching, LLM calls, CRC stamping—all in a CLI command. The agent does not dispatch sub-agents or keep hundreds of files in context: a massive run does not "freeze" the session.
2. **Local-only** (ADR `260610-2228`). Any file is generated by a local model via omlx; there are no cloud escalations. If the model doesn't meet the standard, the doc is written with a **degraded marker**, not delegated to the cloud.
3. **Determinism around LLM.** Everything that can be done without tokens is done without tokens: fact extraction, anchor identification, "Guarantees" section, post-lint, scoring, obsolescence checking.
4. **Doc next to code:** `<dir>/docs/<stem>.md` for every code file (`js`/`mjs`/`ts`/`vue`/`py`).

## While-Do States

The state is determined **deterministically** by the frontmatter against the current source:

```
                ┌─ No doc ──────────────► stale: missing
source ──CRC──►├─ crc $\neq$ frontmatter.crc ───► stale: crc-mismatch
                └─ crc matches ─┬─ score $\geq$ threshold or missing ─► fresh
                                   └─ score < threshold ───────────────► fresh, degraded
```

Frontmatter is machine metadata, the only exception to the "clean Markdown" rule:

```markdown
---
docgen:
  source: npm/lib/llm.mjs
  crc: a3f1c9e0
  score: 55
  issues: short-behavior,internal-name:callPi
---
```

- `crc` — CRC32 of the source bytes at the time of generation. Independent of git status (rebase, branches, uncommitted).
- `score`/`issues` — det-quality score (written when known; `issues` only for degraded). **Degraded $\neq$ stale**: CRC is fresh, the gate is not blocked, but the debt is visible separately.

## CLI Commands

| Command           | What it does                                                                                                                          | Exit                                     |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| `doc-files scan`  | JSON list of all code files with obsolescence status                                                                             | 0 / 1 (root does not exist)                  |
| `doc-files gen`   | Generate docs for stale; `--overwrite` — for all; `--retry-degraded` — only for degraded; `--from N --limit M` — in batches | 0 / 1 (file error or preflight fail) |
| `doc-files stamp` | (Re)stamp frontmatter in existing docs without LLM (migration; `score`/`issues` preserved)                                     | 0                                        |
| `doc-files check` | Obsolescence detector: `--hook` (one file from stdin), `--git` (Stop-gate), `--degraded` (report), explicit paths                         | 0 / 2 (stale found)                   |

## `gen` Algorithm (Batch)

```
scan tree ──► selection of targets (stale | all | degraded) ──► slice --from/--limit
      │
      ▼
preflight omlx (1 tiny call) ── fail ──► one clear stop, exit 1
      │ ok                                       (NOT hundreds of "✗" for files)
      ▼
for each file sequentially (parallelism 1 — roof 8GB):
      generateDoc(file) ──► Source CRC ──► stampDoc(md, source, crc, quality)
      │                                      └─ write <dir>/docs/<stem>.md
      ▼
summary: ✓ OK: N  ⚠ degraded: D  ✗ Err: E   (+hint --retry-degraded)
```

**Preflight** distinguishes three local server states requiring different actions:

- `down` — omlx not running → "run `omlx serve`";
- `memory-guard` — model (3.5GB) does not fit in the **dynamic** memory ceiling of the 8GB machine → "free memory or try later". This is a moment property, not a pipeline defect;
- `auth` — API key enabled on the server → "set `N_CURSOR_OMLX_KEY`" (usually unnecessary: key auto-resolves from `~/.omlx/settings.json`).

## Single File Generation Algorithm (`generateDoc`)

```
file
 │
 ├─ Stage 0  facts (0 tokens):  header-intent, exports+JSDoc, classified imports,
 │                           internalSymbols, behavior markers
 ├─ E1       anchors (0 tokens): URLs, exported constants, error markers,
 │                           config-links, examples from header comments
 │
 ├─ unsupported (vue/py)? ──► one-shot: 1 call for the whole document, score=null ──► done
 │
 ├─ Stage 1  sectional generation (1 call per section, see table)
 ├─ E2       critique→rewriting for vulnerable sections (+0–4 calls)
 ├─ Stage 2  post-lint (0 tokens): trim code-fences, random headers, signatures
 ├─ Stage 2.5 det-scoring (0 tokens)
 ├─ E4       score < threshold? ──► best-of-2: re-run with temperature 0.5,
 │                               the better one by score wins
 └─ result { md, score, issues, degraded } ──► batch makes the decision
```

### Sections: What the Model Sees

The principle of the minimal window — **code only enters where it cannot be without it**:

| Section               | Call Window                                                                                                           | Calls                                 | Critique E2                            |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- | ------------------------------------- |
| "Purpose"            | **protected, human-driven** — literally stored, never generated by LLM; provided to the rest as read-only context | **0**                                    | —                                     |
| "Behavior"           | **the only section with code** + facts + anchors; describes EXACTLY exported names                                              | 1                                        | no                                    |
| "Public API"         | Only a list of exports with descriptions                                                                                         | 0–1 (if >1 export)                   | if all descriptions are empty                |
| "Behavior Guarantees" | —                                                                                                                       | **0** (deterministic template with markers) | —                                     |
| "Overview"           | **LAST** — generalization of the already written "Behavior" (not just a fact list)                                              | 1                                        | always (the model tends to default to generic) |

The generation order is fundamental: "Overview" generalizes the verified, not a guess from the fact list (cures generic/false Overview). The section order in the source document is unchanged (Overview first — this is what `assemble` does).

**Protected "Purpose" Section (Variant B).** The handmade intent/contract (formerly — code header comment) lives in the doc under `## Purpose` (boundary — next `## `; `###`+ inside are allowed). `generateDoc` reads the existing doc, **literally** preserves this block upon regeneration, inserts it first (after H1), and feeds it to machine sections as read-only context ("do not duplicate, complement"). Scoring excludes it. Opt-in = mere presence of the header. Design: `docs/superpowers/specs/2026-06-11-doc-files-protected-intent-section.md`.

The fact list contains **negative markers** ("Caching: NONE — do not mention cache") — the cheapest way to filter out typical hallucinations even in the prompt. Anchors are inserted via a "mandatory inclusion" block so that the text does not drift into general phrases.

### E2: Critique → Rewriting

The Critic is the same model with a separate role and a **closed list of criteria** (generic phrases, missing anchors, grammar, subheadings within the section, copied signatures, invented facts). The response is a numbered list of issues or `NONE`. The Rewriter receives the draft + the list and returns only the updated text. One loop per section, no recursion.

### Stage 2.5: det-scoring

Scoring from 100 with penalties, each leaving code in `issues`:

| Check                                                                                                                    | Penalty | Code                   |
| ---------------------------------------------------------------------------------------------------------------------------- | :------ | :--------------------- |
| No "Overview" section                                                                                                         | −25     | `no-overview`         |
| Generic Overview: abstract formulas/paraphrases ("data verification/processing", "contract compliance", "applies logic")      | −35     | `generic-overview`    |
| "Behavior" < 60 characters                                                                                                    | −20     | `short-behavior`      |
| Mention of cache in "Guarantees" when the file doesn't cache (with negation recognition)                                                     | −20     | `cache-hallucination` |
| Backtick mention of internal/service symbol (imported-internal + non-exported top-level functions/classes) in visible sections | −10 per symbol          | `internal-name:<sym>` |
| Valid anchor (literal substring of src) missing in the document                                                                | −5 per anchor, max −20 | `anchor-miss:<token>` |
| Surzhik/Russism from curated list                                                                                           | −10     | `surzhik`             |

`score < 70` (env `N_CURSOR_DOC_FILES_THRESHOLD`) → **degraded**: the doc is written, marker in frontmatter, regeneration → `gen --retry-degraded` (after pipeline improvement or on a free machine). Transport error is **not** masked as degraded — it is an honest `✗`.

## Transport

The only `callLlm` point (`npm/lib/llm.mjs`); the backend determines **the model string**, no env switches:

```
model-id ── omlx/<m> ──► direct HTTP to omlx (127.0.0.1:8000, OpenAI-compatible, text-only)
         └─ otherwise ────► pi CLI (cloud providers; $\sim$46% slower for local)
```

- Default model: `N_CURSOR_DOCGEN_MODEL` → `resolveModel('min')` (cascade `N_LOCAL_*`/`N_CLOUD_*`) → direct to `omlx/mlx-community--gemma-4-e2b-it-4bit`.
- Omlx API key: call parameter → `N_CURSOR_OMLX_KEY` → `auth.api_key` from `~/.omlx/settings.json` (zero-config) → no header.
- Wire-trace: `N_CURSOR_LLM_TRACE=<file>` → JSONL line for each call (backend, model, duration, sizes, success/failure). Fail-safe: trace write error does not break the call.

## Gates and Hooks

- **PostToolUse** (`check --hook`): after editing a code file reminds that its evidence is missing (exit 2 → agent message).
- **Stop-gate** (`check --git`): before completing the task checks changes relative to the HEAD of the source. Outdated docs → block (exit 2). Safeguard: stale over `N_CURSOR_DOC_FILES_GATE_MAX` (default 50) → only a warning — a mass first run cannot permanently block work.
- **Degraded never blocks**: model weakness — not a reason to stop tasks. Visibility of debt → `check --degraded` (informational, exit 0).

## Scanner: What is documented

- Extensions: `.js .mjs .ts .vue .py`; tests (`*.test.*`, `*.spec.*`), `*.d.ts` are excluded.
- Ignore trees: `node_modules`, `dist`, `.git`, `coverage`, `__pycache__`, `.cursor`, `.claude`, all `docs/` folders, etc. — list in `docgen-ignore.mjs` (common for `doc-files` and `doc-aggregate` via re-export).
- Root with system-wide layout (`docs/adr` or `docs/explanation`) — file-level docs at the top level are not planned: the root `docs/` remains for system documentation (like this file).

## Known limitations (honestly)

- `vue`/`py` are currently one-shot as a whole file without scoring — the weakest path; cured by the unit layer (see "Evolution").
- Det-scorer measures **form**, not factual accuracy of the content: `score=100` does not guarantee truth. Semantic det-proxy (evidence filter) appears in the claims-pipeline.
- Parallelism is strictly 1: 8GB memory is shared between the model and the system; this is why there is a preflight memory-guard.
- Complex files (many internal symbols) perform poorly on whole-file prompts — this is the main motivation for the next stage.

## Evolution: where the pipeline goes next

The target architecture is specified in the spec [`docs/superpowers/specs/2026-06-10-doc-files-local-only-pipeline.md`](superpowers/specs/2026-06-10-doc-files-local-only-pipeline.md) and ADR `260610-2228`:

1. **Increment 1:** language-agnostic unit layer on AST (`oxc-parser`; vue — script-block, py — stdlib `ast`) → flat per-export slices; two-phase generation **claims→render** with a deterministic evidence filter; det-verify package (anchor-coverage, identifier-whitelist, duplicates, render-lint) + point micro-refining; render best-of-2; few-shot; `bench` command.
2. **Increment 2:** bottom-up micro-summaries by call-graph (any local complexity → construction), adaptive self-consistency, unit cache by hash (incremental re-runs).

Empirical basis of decisions: ADR `260610-0643` (omlx backend and self-critique choice directly), `260610-1344`/`260610-1349` (pi/omlx boundary, JS-owned loop), benchmarks of orchestrated pipeline v2.
