Графовий / файловий стан задачі

Дробимо роботу на блоки з DAG-залежностями, роздаємо людям і LLM — стан повністю у файлах git.

файл = стан вузла · claim атомарний · push = арбітр

01 Проблема

Велику задачу треба розбити на менші блоки й роздати: деякі — паралельно, деякі — послідовно; одні залежні, інші ні; виконавці — і люди, і LLM. Потрібно будь-якої миті зі стану зрозуміти: де процес у графі, на кому блок і який саме блок.

Залежності

граф, не список
  • hard — B потребує вихід A
  • resource — спільний файл
  • none — паралель

Виконавці

human + llm
  • механічне → LLM
  • судження/ризик → людина
  • змішано й паралельно

Вимога

стан у файлах
  • джерело істини — файли
  • придатне для git-рев'ю
  • без центральної БД

02 Модель: три файли на блок

Кожен вузол DAG має до трьох артефактів. Наявність файлу — це і є стан. Усе трекається git.

<id>.plan.md

намір · immutable
  • id, dependsOn[]
  • owner (кому)
  • isolation
  • Кроки · Контракт · Acceptance

<id>.claim.md

лок · поки in_progress
  • атомарний файл (wx)
  • by (хто взяв)
  • branch, started_at
  • heartbeat (для stale)

<id>.fact.md

результат · по завершенню
  • status done/failed
  • by, commits[]
  • gate, verifiedAt
  • claim знімається

03 Граф (DAG) — позиція процесу

Той самий граф читається зі скану файлів. Кольор вузла = похідний статус.

done — є fact in_progress — є claim ready — депенденси done blocked — є невиконаний dep

Фронт = in_progress ∪ ready (B02, B03 — можна паралелити). B04 чекає, бо dependsOn:[B02,B03]. Один скан → повна картина.

04 Атомарний claim → у git

Claim — звичайний файл (а не директорія), щоб потрапляти в git. Атомарність — двома рівнями.

  1. Локально — ексклюзивне створення: writeFileSync(path, …, {flag:'wx'}) (O_CREAT|O_EXCL). Один syscall: або створив, або EEXIST = зайнято. Без race.
  2. Розподілено — add claim → commit → push. Перший push у remote = власник.
  3. Конфлікт — чужий push був першим → твій відхиляється; fetch, бачиш claim → береш інший ready-вузол.
  4. Завершення — пишеш fact.md, видаляєш claim.md, commit, push.

Директорія-лок (mkdir) теж атомарна, але git її не несе — тому файл + push-арбітр. Завислий воркер: heartbeat старший за TTL → claim вважається вільним.

06 Життєвий цикл вузла

Залежний вузол сидить у blocked, доки всі його dependsOn не отримають fact: done — тоді стає ready.

07 Інтеграція з n-cursor

Граф не дублює рушій — він лягає поверх наявного лінійного flow.

блок ≈ flow

  • plan.md → flow plan
  • runtime — .flow.json
  • flow gate/release → fact.md

паралель = ізоляція

  • кожен блок — свій worktree/гілка
  • нема конфліктів за файли
  • фан-ін: merge + verify

trace = граф

  • nitra-block* + dependsOn
  • друкує позицію DAG
  • флагує осиротілі claim

Модель: лінійний flow × граф зверху. Один .flow.json = одна нитка; DAG паралельних задач = кілька flow, зшитих trace-лінками.
