# Архітектура: Динамічний Самомодифікований Граф Задач

## Назва структури

**Рекурсивний складений DAG** — рекурсивний складений орієнтований ациклічний граф із динамічним розкладом вузлів та файловим сховищем стану.

---

## Вузол

Кожен вузол або є атомарним, або розкладається на підграф — рішення приймається **динамічно при запуску** на основі вхідних даних.

```
Node
  ├── impl:
  │     ├── Atomic    — fn(inputs) → outputs
  │     └── Compound  — Graph{ entry, nodes[], edges[], exits[] }
  ├── state: pending | running | resolved | failed | unresolvable
  ├── inputs:  Map<portId, Value>
  └── outputs: Map<portId, Value>   ← заповнюється при resolved
```

Для батьківського вузла інтерфейс однаковий: він чекає `state = resolved` не знаючи що всередині (**інкапсуляція чорної скриньки**).

---

## Граф

- **DAG** — орієнтований, без циклів
- **Ребра** — потік даних: виходи одного вузла стають входами наступного
- **Вхідний вузол** — отримує вхідні дані батьківського складеного вузла
- **Вихідні вузли** — кілька, їх виходи зливаються у виходи батька
- Складений вузол може бути замінений атомарним і навпаки без змін у батьківському графі

```
CompoundNode
  └── Graph
        ├── entry → отримує inputs батька
        ├── nodes[]: Node[]
        ├── edges[]: { from: NodeId+portId, to: NodeId+portId }
        └── exits[]: Node[]   ← кілька, merge → parent.outputs
```

---

## Файлова структура (сховище стану)

Граф зберігається у файловій системі — кожен вузол пише свої файли сам. Граф відновлюється скануванням.

```
tasks/
  <nodeId>/
    meta.json            ← тип, стан, ребра
    inputs.json
    outputs.json         ← з'являється при resolved
    error.json           ← з'являється при failed
    repair_history.json  ← пам'ять інженера на цьому вузлі
    patches/
      patch_001.json     ← зміни від EngineerAgent
    graph/               ← якщо складений вузол
      <childId>/
        ...
```

Граф має **часовий вимір** — це послідовність знімків:
```
DAG_v1 → [EngineerPatch] → DAG_v2 → [EngineerPatch] → DAG_v3
```

---

## Самовідновлення: агент-інженер

Інженер — **мета-рівень поза графом**, без власного стану. Викликається при `node.state = failed`.

```
EngineerAgent(error, full_path_from_root, repair_history) →
  GraphPatch:
    ├── replace node (атомарний → складений або навпаки)
    ├── insert nodes (додати проміжні кроки)
    ├── rewire edges
    └── modify inputs
```

Може змінювати **будь-який рівень** — від вузла що впав до кореня.

### Каскад інвалідації

Зміна вузла інвалідує всіх наступників:
```
patch(node_6) →
  node_6.outputs = stale →
    node_7, node_8 = invalidated →
      їх successors = invalidated → ...
```

### Пам'ять

Пам'ять живе **на вузлі**, де інженер вносить зміни:
```json
repair_history.json
[
  {
    "attempt": 1,
    "engineer_reasoning": "...",
    "patch_applied": {},
    "result": "failed",
    "failure_reason": "..."
  }
]
```
При зміні батьківського вузла через помилку дочірнього:
- `parent.repair_history` ← записується патч
- `child.repair_history` ← записується `{ triggered_parent_patch: id }`

## Збіжність: часовий бюджет

Замість ліміту спроб — **часовий бюджет**:

```
repair_context.json
  ├── deadline: timestamp      ← встановлюється один раз
  ├── started_at: timestamp
  ├── time_budget_sec: 600     ← 10 хвилин
  └── attempts: []             ← необмежено в межах часу
```

Інженер адаптує стратегію до залишку часу:
- багато часу → складна спроба
- мало часу → швидкий виправлення
- `remaining ≤ 0` → `state = unresolvable`, ескалація

---

## Ієрархія ескалації

Кожен рівень отримує **свіжий** часовий бюджет:

```
вузол впав
  → агент-інженер (бюджет: 10хв)
      → час вийшов → ескалація
          → батько: агент-інженер (свіжий бюджет: 10хв)
              → час вийшов → ескалація
                  → дідусь: агент-інженер (свіжий бюджет: 10хв)
                      → ...
                          → корінь: час вийшов
                              → сповіщення старшого інженера (людина)
```

**Максимальний час до людини:** `глибина × бюджет`

### Сповіщення старшого інженера

```json
{
  "failed_node": "root/project_A/workflow_X/node_7",
  "escalation_chain": [
    { "level": "node_7",    "attempts": [], "time_spent": 600 },
    { "level": "workflow_X", "attempts": [], "time_spent": 600 },
    { "level": "project_A",  "attempts": [], "time_spent": 600 }
  ],
  "current_graph_snapshot": "...",
  "suggested_next_steps": []
}
```

## Зведена таблиця рішень

| Аспект | Рішення |
|---|---|
| Структура | Рекурсивний складений DAG |
| Декомпозиція | Динамічна, при запуску вузла |
| Ребра | Потік даних (виходи → входи) |
| Вихідні вузли | Кілька, злиття у виходи батька |
| Зберігання | Файли, кожен вузол пише свої |
| Самовідновлення | Агент-інженер — мета-рівень, необмежений доступ |
| Інвалідація | Каскад вниз по наступниках |
| Збіжність | Часовий бюджет (не кількість спроб) |
| Ескалація | Кожен рівень — свіжий бюджет |
| Зупинка кореня | Сповіщення старшого інженера (людина) |
| Пам'ять | `repair_history.json` на кожному вузлі де є зміни |
