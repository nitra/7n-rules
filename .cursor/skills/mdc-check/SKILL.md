# n-mdc-check — від правила до скрипта

Навичка для роботи з правилами пакета `@nitra/cursor`:

- **`npm/scripts/check-<id>.mjs`** — **канонічне місце** для перевірюваної логіки: умови, крайові випадки, повідомлення про помилки. Тут саме варто **документувати** реалізацію: багаторядковий верхній JSDoc файлу, JSDoc до функцій, за потреби **внутрішні коментарі**, де пояснюється _чому_ так перевіряємо (обмеження парсера, винятки, зв’язок із політикою). Людина, яка підтримує пакет, має змогу **читати скрипт як специфікацію з кодом**, а не вгадувати з `.mdc`.

- **`npm/mdc/<id>.mdc`** (і дзеркало **`.cursor/rules/n-<id>.mdc`**) — **людиноподібний шар**: навіщо правило, що вважається правильною поведінкою **словами**, як швидко виправити типові порушення, посилання на конвенції та команду перевірки. Уникай у тексті правила **дублювання** детального алгоритму, довгих списків умов і «програмних» конструкцій, які вже є в скрипті — їх важче синхронізувати й читати агенту й людині.

Канонічні вимоги до скриптів, тестів і AST — **`.cursor/rules/scripts.mdc`**.

## Коли застосовувати

- Додали або змінили вимоги в **`npm/mdc/<id>.mdc`** (або локальному **`n-<id>.mdc`**).
- У правилі багато «має містити», «заборонено», точних фрагментів файлів — і агенти повторюють перевірку вручну.
- Потрібно зменшити обсяг `.mdc` і уникнути роз’їзду тексту й фактичної перевірки.

## Ідентифікатор правила

- Файл правила в пакеті: **`npm/mdc/<id>.mdc`** → скрипт: **`npm/scripts/check-<id>.mjs`**, команда CLI: **`npx @nitra/cursor fix <id>`**.
- CLI підхоплює всі **`check-*.mjs`** з каталогу `scripts` пакета; окремої реєстрації не потрібно.
- Якщо правило згадане в **`AGENTS.md`**, для нього доречна programmatic перевірка — має існувати відповідний **`check-<id>.mjs`** (коли вимоги формалізовані).

## Workflow

### 1. Базова лінія

```bash
npx @nitra/cursor fix <id>
```

Зафіксуй поточний вивід (якщо скрипт уже є).

### 2. Розбір тексту правила

Прочитай **`npm/mdc/<id>.mdc`** повністю. Для кожної вимоги виріши:

| Тип                                                                                | Куди                                                                                     |
| ---------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Перевірювана структура (файли, поля JSON/YAML, наявність ключів, заборонені шляхи) | **`check-<id>.mjs`**                                                                     |
| Семантика JS/TS у файлах                                                           | **`check-<id>.mjs`** через **Oxc AST** (`oxc-parser`), не «сліпі» regex по всьому файлу  |
| Вміст у Vue SFC / MDX / Astro                                                      | Спочатку **виділи** валідний JS/TS-фрагмент, потім AST; regex — лише для ізоляції блоків |
| Не-JS конфіги, сирі workflow                                                       | Текстовий/структурний парсинг або обґрунтований regex; у JSDoc коротко **чому** не AST   |
| Смак, архітектура, «навіщо так», поради без однозначного критерію                  | Залиш у **`.mdc`**                                                                       |

### 3. Скрипт

- **Новий** `check-<id>.mjs`: орієнтуйся на наявні файли в **`npm/scripts/`** (наприклад **`check-bun.mjs`**) — стиль **`fail` / `pass`**, **`export async function check()`**, exit code **0 / 1**.
- **Документація в коді** — пріоритет для «як саме це перевіряється»: верхній JSDoc файлу, JSDoc функцій, коментарі біля нетривіальних гілок (винятки, евристики, обхід AST). У правилі **не** повторюй той самий рівень деталізації — залиш у `.mdc` лише те, що потрібно для **розуміння наміру** без читання коду.
- **На початку файлу** (перед `import`) — **багаторядковий** верхній JSDoc **українською**: що перевіряє файл; між смисловими блоками — рядок `*`; перші рядки без службових префіксів — одразу зрозуміло призначення.
- **JSDoc** до експортованих функцій і нетривіальних внутрішніх функцій.
- Повідомлення **`fail`**: конкретно, що не так і як виправити (як у інших check-скриптах).
- Залежність **`oxc-parser`** не додавай у корінь споживача — вона в **`dependencies`** пакета **`@nitra/cursor`** (див. **dev-dep** / **scripts.mdc**).

### 4. Тести

Якщо логіка нетривіальна або її легко зламати — додай **`npm/tests/check-<id>.test.mjs`** (або розшир наявні тести). Після змін:

```bash
cd npm && bun test
```

### 5. Спрощення `.mdc` (людинозрозумілий текст)

- Прибери з правила **дубль** детальних умов і покрокових критеріїв, які тепер у скрипті; залиш **короткий зміст**, мотивацію, посилання на файли/конвенції.
- Формулюй правило **природною мовою**: що очікується, чому це важливо, приклади «правильно / неправильно» там, де це допомагає людині й агенту, а не дублює код перевірки.
- Якщо для повноти потрібна технічна деталь — краще **одне речення-відсилання** («деталі перевірки — у `check-<id>.mjs`»), ніж копія логіки в markdown.
- Потрібно зберегти (або додати) секцію **«Перевірка»** з **`npx @nitra/cursor fix <id>`** (або повним ім’ям правила з `AGENTS.md`).

### 6. Верифікація та версія пакета

- Після змін у **`npm/`** (`.mdc`, `check-*.mjs`, тести) підвищ **patch**-версію в **`npm/package.json`** на **+1** від останнього збереженого в git стану (**n-npm-module**).
- Переконайся, що **`npx @nitra/cursor fix <id>`** і за потреби **`cd npm && bun test`** проходять без помилок.

```bash
npx @nitra/cursor fix <id>
cd npm && bun test
```

## Антипатерни

- Оновлювати лише текст правила без оновлення **`check-<id>.mjs`**, коли для правила вже є скрипт або з’явилась нова перевірювана умова.
- Тримати в **`.mdc`** довгі «специфікації» з переліком умов і крайових випадків, які можна однозначно перевірити в коді — це роз’їжджається зі скриптом і гірше підтримується.
- Скрипт без **достатніх коментарів/JSDoc**, коли логіка неочевидна — тоді людина змушена читати лише сухий код або застарілий `.mdc`.
- Дублювати однакову логіку в кількох check-скриптах замість спільної утиліти в **`npm/scripts/utils/`**.
- Реалізувати в скрипті **side effect** (автоміграція файлів, запис на диск, перейменування) без короткої згадки в **`.mdc`** — розробник читає правило і не знає, що `check` змінює файли у проєкті.

## Команда в чаті

`/n-mdc-check` або `/mdc-check` — застосуй цей workflow до правила з поточного контексту або до `<id>`, який вкаже користувач.
