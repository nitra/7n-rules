# n-lint — усі lint-скрипти та нуль зауважень

## Мета

Виконати **кожен** скрипт з префіксом `lint-` у `package.json` і довести стан до відсутності помилок лінтерів (після auto-fix — виправити вручну те, що інструменти не можуть).

## Які package.json враховувати

1. **Корінь репозиторію** — обов’язково: прочитай `scripts` і збери всі ключі, де ім’я починається з `lint-`.
2. **Workspaces** — якщо в кореневому `package.json` є поле `workspaces`, для кожного шляху з масиву відкрий відповідний `package.json` (якщо файл існує) і додай його `lint-*` скрипти. Запускай `bun run <скрипт>` з **поточної робочої директорії = корінь того пакета** (не з кореня монорепо), щоб залежності й шляхи були коректні.

Порядок виконання: спочатку корінь (у порядку ключів у `scripts`), потім workspace-пакети (наприклад, в алфавітному порядку шляхів).

## Перед запуском

- Якщо `node_modules` відсутні або лінтер «не знайдено» — виконай `bun i` у відповідному пакеті (корінь і/або workspace).
- Дотримуйся правил репозиторію: **Bun** для скриптів (`bun run …`), без `npm` / `yarn` / `pnpm` як менеджера пакетів.

## Запуск

Для кожного знайденого `lint-*`:

```bash
bun run <ім'я-скрипта>
```

Виконуй з каталогу того `package.json`, де визначено скрипт.

## Якщо скрипт падає або лишаються попередження

1. Прочитай вивід лінтера (файл, правило, рядок).
2. **Auto-fix**: якщо скрипт уже містить `--fix` / автовиправлення — перезапусти після змін у коді. Якщо інструмент підтримує fix, а скрипт його не викликає — можна один раз запустити еквівалент з fix **лише** якщо це не суперечить прийнятому в проєкті скрипту (пріоритет — оновити код під очікуваний `bun run lint-*`).
3. **Ручні правки** — для правил без auto-fix: зміни в коді/конфігах (наприклад `eslint.config.js`, `.cspell.json`) мінімально, щоб задовольнити лінтер.
4. Повтори `bun run <той самий lint-*>` доки exit code `0` і немає невирішених error-level повідомлень.

## Результат для користувача

Коротко підсумуй: які `lint-*` були запущені, у яких каталогах, що змінено (файли/тип правок). Якщо щось неможливо виправити без рішення архітектури — явно опиши блокер і варіанти.
