## Огляд

Єдина публічна точка входу для `rules/<id>/check.mjs`, через яку правило проходить стандартний оркестратор із вбудованою серіалізацією на рівні git-стану. Тут виконання загортається у `withLock`, тому паралельні запуски одного й того самого правила з `npx @nitra/cursor fix`, прямого `bun rules/<id>/check.mjs` або композиції `run` не дублюються, а різні правила можуть іти паралельно. Публічний API — `runStandardRule`. Поведінка read-only: без записів у ФС чи БД. Кешування діє в межах одного прогону.

## Поведінка

1. `runStandardRule` запускає стандартний прогін одного правила як єдину публічну точку входу для `rules/<id>/check.mjs`.
2. Вона визначає контекст конкретного правила та блокує паралельні запуски того самого правила, щоб повторні виклики не дублювали роботу в межах одного стану git-дерева; різні правила можуть іти одночасно.
3. Далі вона збирає правило для виконання і використовує кеш обходу файлів у межах поточного прогону, щоб повторні звернення під час цього ж запуску не перемелювали ті самі дані.
4. Після цього вона виконує стандартний ланцюг перевірок правила: застосовність, JS-логіку, policy та посилання на mdc.
5. Результат повертається як код завершення: `0`, якщо порушень немає, або `1`, якщо виявлено порушення.
6. `runStandardRule` не змінює ФС чи БД; це read-only оркестрація без локальної логіки в самому правилі, окрім розширення через `ctx`.

## Гарантії поведінки

- Read-only: не виконує операцій запису (ФС/БД).
- Кешує результати в межах одного прогону.
