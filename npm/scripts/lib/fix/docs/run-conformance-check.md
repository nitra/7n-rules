## Огляд

Цей модуль реалізує прямий механізм виконання конформності, оминаючи обгортки `subprocess`. Він визначає набір активних правил, спираючись на конфігурацію `.n-cursor.json`. Після селекції правил, він викликає конформність-фазу `lint` (read-only), движок (`orchestrator.mjs`, `t0.mjs`) та PostToolUse-хук. Кожне правило виконується окремим процесом `bun`, забезпечуючи ізоляцію на рівні кожного правила.

## Поведінка

resolveCheckRuleIds визначає набір ID правил для прогону, використовуючи `.n-cursor.json` як єдине джерело правди для селекції активних правил.
runConformanceCheck запускає перевірку конформності для визначених правил, викликаючи `check.mjs` кожного правила окремим процесом `bun` для ізоляції.

## Публічний API

resolveCheckRuleIds — Визначає ідентифікатори правил для виконання, використовуючи `.n-cursor.json` як основне джерело даних:
*   Якщо вказані `requestedRules`, вони фільтруються до активних правил, що є у конфігурації.
*   Якщо `requestedRules` відсутні, вибираються всі активні правила з конфігурації.
*   Якщо жодних правил не вказано і немає конфігурації, використовується резервний список правил, згенерованих з файлів `.cursor/rules/*.mdc`.

runConformanceCheck — Виконує перевірку відповідності кожного окремого правила без внесення змін.
