## Огляд

Цей модуль реалізує конформність-детект як пряму функцію, викликаючи конформність-фазу `lint`, движок (`orchestrator.mjs`, `t0.mjs`) та PostToolUse-хук. Селекція активних правил відбувається виключно тут, спираючись на конфігурацію `.n-cursor.json`. Для збору результатів конформності кожен перевірочний ендпоінт (`rules/<id>/main.mjs`) запускається окремим процесом `bun` для забезпечення per-rule ізоляції. Канон — єдиний `main.mjs` (ADR 2026-06-21), чий CLI-блок викликає `runRuleCli`.

## Поведінка

resolveCheckRuleIds визначає набір ID правил для прогону, використовуючи `.n-cursor.json` як єдине джерело правди для селекції. Якщо правила запитані явно, вони фільтруються за активністю. Якщо правила не запитані, вибираються активні правила з конфігу; у відсутності конфігу відбувається fallback на зматеріалізовані файли.
runConformanceCheck виконує перевірку конформності для кожного правила, визначеного через `resolveCheckRuleIds`, запускаючи кожен перевірочний ендпоінт окремим процесом `bun` та збираючи результати.

## Публічний API

resolveCheckRuleIds — Визначає, які правила будуть застосовані для перевірки, використовуючи `.n-cursor.json` як основне джерело:
*   Якщо вказані `requestedRules`, вони фільтруються до активних правил, доступних у конфігу.
*   Якщо не вказані `requestedRules`, беруться всі активні правила з конфігу.
*   Якщо не вказані жодні правила і в конфігу їх немає, використовується список правил, що були згенеровані з файлів `.cursor/rules/*.mdc` (режим налагодження).

runConformanceCheck — Виконує перевірку відповідності для кожного окремого правила, не вносячи змін.
