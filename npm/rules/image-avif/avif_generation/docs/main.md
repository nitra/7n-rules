## Огляд

Файл надає інструменти для перевірки стану імпортів зображень у `.vue`/`.html` файлах з метою забезпечення коректної роботи з `.avif`-двійниками. Визначає, чи потрібно переписати посилання, чи відсутні двійники, чи є сиріті, і використовується для генерації звітів.

Поведінка
packageHasAvifDisabled повертає `true`, якщо у `package.json` пакета визначено `"@nitra/minify-image": { "disable-avif": true }`.
resolveImageCandidates будує та повертає впорядкований список абсолютних шляхів, куди необхідно перевіряти наявність зображення для рерайту або збору існуючого `.avif`. Підтримуються відносні шляхи, шляхи у конвенції Vite/Quasar, абсолютні шляхи через `public/` або кореневий каталог пакета, а також "голі" шляхи, які можуть бути аліасами.
scanVueAvifInPackage виконує читання одного `.vue`/`.html` файлу з пакета, обчислює необхідні рерайти для растр-посилань на `.avif`-двійники, фіксує посилання, де двійника немає, та доповнює інформацію про використані `.avif`-двійники. Пропускає сканування файлів іншими пакетами, ігноруючи піддерева інших коренів, щоб уникнути дублювання звітів.
scanVueAvifImports сканує всі workspace-пакети, перевіряючи налаштування opt-out, і за потреби викликає scanVueAvifInPackage. Перевірка пропускається, якщо в репозиторі відсутні workspace або `.vue`-файли, що свідчить про те, що правило не застосовне. Повертає список абсолютних коренів пакетів, у яких активовано opt-out.
hasAnyVueRasterReference перевіряє наявність будь-якого растр-посилання в `.vue`/`.html`, яке потенційно потребує переписати на `.avif`-двійник, через використання регулярних виразів, що відповідають основному процесу рерайту. Перевірка пропускається, якщо не знайдено жодного посилання.
collectOrphanAvifs збирає список `.avif`-сиріт — двійників, на які вже не лишилося жодного живого посилання у `.vue`/`.html`. Ця функція ігнорує `.avif`-двійники з opt-out пакетів, оскільки їхні шаблони не сканувалися, інакше сиріт фактично не можна вважати.
scanAvif є єдиним read-only сканером усього етапу, який використовується як спільно для детектора порушень, детектора наявних `.avif`-двійників та детектора сиріт. Він збирає необхідні звіти про потрібні рерайти (`avif-needs-rewrite`), відсутність `.avif`-двійників (`avif-missing`) та наявність `.avif`-сиріт (`avif-orphan`).
lint використовує scanAvif для генерації результату, який включає звіт про потрібні рерайти, відсутність двійників та сиріти. Отриманий результат з lint має бути `0` для підтвердження чистого виходу після виправлення порушень.

## Публічний API

- AVIF_NEEDS_REWRITE — Стабільні reasons.
- AVIF_MISSING — Стабільний reason: для растрового зображення відсутній згенерований AVIF-двійник.
- AVIF_ORPHAN — Стабільний reason: AVIF-файл лишився без растрового джерела — кандидат на cleanup.
- MINIFY_PACKAGE_NAME — Імʼя CLI-пакета, який генерує AVIF (використовує T0-fix).
- CLEANUP_EXTRA_IGNORE_DIR_NAMES — Імена каталогів, які cleanup НЕ зачіпає, бо це артефакти збірки/нативні
платформи — `.avif` всередині — це продукт попереднього `bun run build`/Capacitor sync,
а не кандидати на видалення. `walkDir` уже скіпає `node_modules`, `.git`, `dist`,
`coverage`, `.turbo`, `.next` — додатково для cleanup ігноруємо ще ці.
- scanAvif — Чистий read-only скан усього AVIF-етапу (без npx, без запису, без unlink). Спільний
для detector-а (→ violations) і T0-fix (виконує генерацію, потім rescan + write/unlink).
- lint — Read-only detector AVIF-етапу: ЗВІТУЄ потрібні rewrite-и (`avif-needs-rewrite`),
відсутні `.avif`-двійники (`avif-missing`) і `.avif`-сироти (`avif-orphan`).
Не валідує image-compress cache/dependency policy — це окреме правило.
