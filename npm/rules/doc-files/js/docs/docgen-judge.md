## Огляд

Наданий файл відповідає за оцінку якості згенерованої документації. Він визначає модель для оцінки через `JUDGE_MODEL` та використовує функціональність для зіставлення документації з вихідним кодом. Механізм дозволяє викликати оцінку документації за допомогою `judgeDoc` та перевірити результат через `judgeFailsDoc`. Доступна функція `JUDGE_ENABLED` дозволяє контролювати активність цього механізму, тоді як `JUDGE_CONFIDENCE` надає рівень впевненості в оцінці.

## Поведінка

JUDGE_MODEL визначає модель, яка використовується суддею для оцінки якості документації.
JUDGE_ENABLED вказує, чи активний семантичний гейт для оцінки документації.
JUDGE_CONFIDENCE задає мінімальний поріг впевненості для відзначення документації як неточної.
parseDocVerdict витягує та перевіряє JSON-відповідь від судді, повертаючи структуровані дані про оцінку.
judgeDoc зважує згенеровану документацію проти вихідного коду за допомогою потужної моделі і визначає її семантичну точність.
judgeFailsDoc перевіряє, чи слід вважати документацію неякісною на підставі verdicts судді та заданого порогу.

## Публічний API

JUDGE_MODEL — Використовується модель-суддя `N_CLOUD_MIN_MODEL` (хмарний tier).
JUDGE_ENABLED — Функціонал судді ввімкнений автоматично, якщо вказана модель `N_CLOUD_MIN_MODEL`.
JUDGE_CONFIDENCE — Мінімальний рівень впевненості, який має мати висновок, щоб вважати документ занепалим (`degraded`).
parseDocVerdict — Витягує та перевіряє JSON-формат висновку з відповіді LLM.
judgeDoc — Сильна модель оцінює згенерований документ на відповідність вихідному джерелу.
judgeFailsDoc — Визначає, чи висновок позначив документ як занепалий (тільки якщо висновок `inaccurate` і впевненість достатня).
