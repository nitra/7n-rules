# firebase_hosting.mjs

## Огляд

Модуль `firebase_hosting.mjs` — це check-скрипт правила `abie` (див. `abie.mdc`), який забороняє наявність артефактів Firebase Hosting у **підкаталогах першого рівня** репозиторію. Перевірка реалізує політику: Firebase Hosting у проєктах abie заборонено, тому файли `.firebaserc`, `firebase.json` та директорія `.firebase/` не мають бути присутніми у директоріях першого рівня кореня репозиторію.

Особливості політики:
- Перевіряються **лише** підкаталоги першого рівня (тобто директорії безпосередньо в корені репозиторію).
- **Сам корінь** репозиторію не перевіряється — у корені ці імена дозволені, оскільки можуть належати до суміжних/зовнішніх проєктів.
- Зі сканування виключені директорії `.git` та `node_modules` (системні/залежності).

Модуль є ESM (`.mjs`), використовує синхронну перевірку існування файлу `existsSync` для конкретних шляхів і асинхронне читання вмісту директорії через `readdir`. Звітування про результат делеговано модулю `check-reporter.mjs` через утиліту `createCheckReporter`.

## Експорти / API

Модуль експортує одну іменовану асинхронну функцію:

- `check(cwd?: string): Promise<number>` — виконує перевірку та повертає числовий exit code (отриманий від репортера `createCheckReporter().getExitCode()`).

Default-export відсутній. Інших експортів немає.

## Функції

### `check(cwd = process.cwd())`

Сигнатура (JSDoc):

- `@param {string} [cwd]` — корінь репозиторію; за замовчуванням `process.cwd()`.
- `@returns {Promise<number>}` — числовий результат (exit code), який повертає `reporter.getExitCode()`.

Алгоритм:

1. Створює репортер через `createCheckReporter()` і деструктурує з нього методи `pass` та `fail`.
2. Зберігає корінь у локальну змінну `root = cwd`.
3. Намагається асинхронно прочитати вміст `root` через `readdir(root, { withFileTypes: true })`, отримуючи масив `Dirent`-записів.
   - У разі помилки читання (`try/catch`):
     - Витягує текст помилки: якщо це `Error` — `error.message`, інакше `String(error)`.
     - Викликає `fail("Не вдалося прочитати ${root} для перевірки Firebase Hosting: ${msg} (abie.mdc)")`.
     - Повертає `reporter.getExitCode()` достроково (не продовжує сканування).
4. Фільтрує `entries`, залишаючи тільки записи, для яких `e.isDirectory() === true` і чиє ім'я **не** входить до множини `SKIP_TOP_DIR_NAMES` (тобто не `.git` і не `node_modules`). Результат — `topDirs`.
5. Ініціалізує прапорець `hasViolation = false`.
6. Для кожного запису `e` у `topDirs`:
   - Для кожного імені з масиву `['.firebaserc', 'firebase.json']`:
     - Формує відносний шлях `rel = join(e.name, name).replaceAll('\\', '/')` — для уніфікованого виводу (Windows-роздільники замінюються на `/`).
     - Перевіряє існування абсолютного шляху `join(root, e.name, name)` через `existsSync(...)`.
     - Якщо файл існує — викликає `fail("Знайдено заборонений файл Firebase Hosting: ${rel} — видали його (abie.mdc)")` і виставляє `hasViolation = true`.
   - Окремо перевіряє наявність директорії `.firebase` у `e.name`: `existsSync(join(root, e.name, '.firebase'))`.
     - Якщо існує — викликає `fail("Знайдено заборонену директорію: ${e.name}/.firebase/ — видали її (abie.mdc)")` і виставляє `hasViolation = true`.
7. Якщо за весь обхід `hasViolation` залишився `false`, викликає `pass("Підкаталоги кореня (1-й рівень, без .git/node_modules): артефактів Firebase Hosting не знайдено (abie.mdc)")`.
8. Повертає `reporter.getExitCode()`.

Особливості реалізації:
- Перевірки `existsSync` виконуються синхронно для кожного кандидатного шляху всередині асинхронної функції.
- Цикл по підкаталогах **не перериває** перевірку після першого порушення — фіксує всі знахідки і повідомляє про них через `fail` накопичувально.
- Шлях у повідомленнях про файли (`.firebaserc`, `firebase.json`) виводиться як відносний з нормалізованими роздільниками; для директорії `.firebase` виводиться у форматі `${e.name}/.firebase/`.
- Тексти повідомлень закінчуються маркером `(abie.mdc)` для трасування правила.

## Залежності

Зовнішні (Node.js стандартна бібліотека):
- `node:fs` — імпортується `existsSync` (синхронна перевірка існування шляху).
- `node:fs/promises` — імпортується `readdir` (асинхронне читання директорії з опцією `withFileTypes: true`).
- `node:path` — імпортується `join` (побудова шляхів незалежно від ОС).

Внутрішні (модулі проєкту):
- `../../../scripts/lib/check-reporter.mjs` — імпортується `createCheckReporter`. Очікується, що ця фабрика повертає об'єкт із методами `pass(msg)`, `fail(msg)` та `getExitCode()`. Реалізація фабрики у цьому файлі не міститься.

Глобальні API:
- `process.cwd()` — використовується як значення параметра `cwd` за замовчуванням.

Константи модуля:
- `SKIP_TOP_DIR_NAMES = new Set(['.git', 'node_modules'])` — імена директорій, які пропускаються при скануванні підкаталогів першого рівня.

## Потік виконання / Використання

Типовий сценарій виклику:

1. Викликається `check(cwd)` (або без аргументу — тоді корінь = `process.cwd()`).
2. Створюється репортер.
3. Зчитується список елементів кореня.
4. Відфільтровуються директорії, окрім `.git` та `node_modules`.
5. Кожна така директорія перевіряється на наявність `.firebaserc`, `firebase.json` (як файлів/шляхів) та `.firebase` (як піддиректорії).
6. Усі порушення повідомляються через `fail(...)`; за відсутності порушень — одне повідомлення `pass(...)`.
7. Повертається `reporter.getExitCode()` — результат, який можна використати як код виходу процесу.

Приклади імпорту та запуску:

```js
import { check } from './firebase_hosting.mjs'

const code = await check()
process.exit(code)
```

Або з явним коренем:

```js
import { check } from './firebase_hosting.mjs'

const code = await check('/path/to/repo')
process.exit(code)
```

Сценарії результатів:
- **Помилка читання кореня** (наприклад, неіснуючий шлях): один `fail(...)` про неможливість прочитати `root`, дострокове повернення exit code.
- **Знайдено артефакти Firebase Hosting** у одному чи декількох підкаталогах: один або кілька `fail(...)` з конкретними відносними шляхами/директоріями; exit code відповідатиме помилковому стану згідно `createCheckReporter`.
- **Артефактів не знайдено**: один `pass(...)`; exit code відповідатиме успішному стану.

Гранично важливо для коректної роботи: `check` сканує тільки **перший рівень** підкаталогів — він не виконує рекурсивний обхід, тому файли Firebase Hosting у глибших підкаталогах цією перевіркою не виявляються. Це відповідає документованому контракту правила `abie.mdc` (див. JSDoc на початку файлу).
