## Огляд

Файл виконує T0-autofix для `python/doc_comments`: переносить суцільний авторський `#`-коментар безпосередньо над `def` або `class` у docstring одразу після заголовка. Він існує, щоб без LLM механічно виправляти лише безпечні випадки, не вигадуючи текст, не чіпаючи оголошення без придатного коментаря й не синтезуючи module-docstring. Публічні точки використання: `buildDocstring`, `patterns`.

## Поведінка

`patterns` запускає T0-виправлення лише для порушень, позначених як придатні до механічного перенесення коментаря. Дані приходять із lint-порушень, групуються за файлами, після чого кожен файл читається, змінюється в памʼяті та записується назад тільки якщо вміст справді змінився.

Для кожного виправлення `patterns` бере суцільний `#`-блок безпосередньо над `def` або `class`, зберігає авторський текст і передає його в `buildDocstring`. `buildDocstring` формує docstring у тому самому текстовому сенсі: короткий однорядковий варіант для одного рядка або багаторядковий блок для кількох рядків. Відступ узгоджується з тілом оголошення, щоб результат був валідною частиною Python-коду.

Після побудови docstring `patterns` вставляє його одразу після заголовка `def` або `class`, а початковий коментарний блок видаляє. Виправлення застосовуються знизу вгору у межах файлу, щоб попередні вставки й видалення не зміщували позиції ще не оброблених змін.

Спільне правило для `buildDocstring` і `patterns`: T0 не вигадує документацію, а лише переносить уже наявний авторський текст у форму docstring. Оголошення без придатного коментаря залишаються для LLM-ladder, а module-docstring не синтезується, бо провідний коментар файлу може бути shebang, ліцензією або іншим службовим текстом.

## Публічний API

- buildDocstring — Будує docstring-рядки з текстів `#`-коментарів (відступ — від першого рядка
тіла або 4 пробіли).
- patterns — повертає набір шаблонів файлів, які правило охоплює для подальшої обробки або перевірки.
