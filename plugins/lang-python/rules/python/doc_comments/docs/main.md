## Огляд

Файл відповідає за перевірку документувальних коментарів у лінтері для цілей, які визначає `isDocCommentTarget`, свідомо оминаючи шляхи в `node_modules`. `checkFileDocComments` збирає результати перевірки для окремого файлу, а `lint` запускає цю перевірку в межах лінтера.

Він існує, щоб під час одного прогону виявляти відсутню документацію без змін у коді та без повторного обчислення вже зібраних даних завдяки кешуванню в межах прогону.

## Поведінка

`lint` керує перевіркою Python-файлів у межах поточного запуску: бере список кандидатів із контексту лінту, відсіює нерелевантні шляхи через `isDocCommentTarget` і не сканує `node_modules`.

Для кожного прийнятого файла `lint` передає вміст у `checkFileDocComments`. Вона формує порушення для відсутнього module docstring та для публічних top-level функцій і класів без docstring одразу після заголовка. Результати всіх перевірених файлів збираються назад у відповідь `lint`.

`isDocCommentTarget` задає спільне правило охоплення для всієї перевірки, щоб detector працював лише з файлами, які мають підпадати під вимогу Python docstring-ів.

У межах одного прогону використовується кешування, щоб повторні звернення до тих самих даних не призводили до зайвої роботи. Файл не змінює перевірюваний код і повертає лише діагностичні результати.

## Публічний API

- isDocCommentTarget — Чи підпадає файл під вимогу docstring-ів.
- checkFileDocComments — Перевіряє один `.py`-файл: module-docstring + docstring над кожним top-level
публічним def/class.
- lint — Detector python/doc_comments: per-file (дельта) або повний обхід.

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Кешує результати в межах одного прогону.
- Свідомо пропускає шляхи: `node_modules`.
