## Огляд

Файл перевіряє, чи кодові елементи є ціллю для документаційного коментаря через `isDocCommentTarget`, аналізує документаційні порушення у файлі через `checkFileDocComments` і надає lint-вхід через `lint`. Він потрібен, щоб правило повертало зрозумілі діагностики для документації та свідомо не обробляло шляхи з `node_modules`.

Файл читає вміст Rust-файлів, відібраних для перевірки.

## Поведінка

`lint` отримує контекст лінту й формує список Rust-файлів для перевірки: у дельта-режимі працює з переданими змінами, у повному режимі сканує знайдені за glob `.rs`-цілі. Перед аналізом кожен шлях проходить через `isDocCommentTarget`, щоб у перевірку потрапляли лише доречні файли; шляхи в `node_modules` свідомо пропускаються.

Для кожного відібраного файлу `lint` передає вміст у `checkFileDocComments`. Перевірка застосовує спільне правило документації для Rust-коду: на початку файла очікується внутрішній doc-header, а кожен публічний top-level елемент має бути описаний rustdoc-коментарем безпосередньо над ним. Аналіз тестової частини зупиняється на межі test-конфігурації, бо такий блок за конвенцією розміщується наприкінці файла.

`checkFileDocComments` повертає знайдені порушення з прив’язкою до файла й рядків, а `lint` агрегує ці результати у фінальний lint-result для поверхні правил.

## Публічний API

- isDocCommentTarget — Чи підпадає файл під вимогу doc-коментарів.
- checkFileDocComments — Перевіряє один `.rs`-файл: `//!`-header + `///` над кожним top-level
pub-елементом. Сканування зупиняється на `#[cfg(test)]` (тест-модуль
конвенційно наприкінці файлу).
- lint — Detector rust/doc_comments: per-file (дельта) або повний обхід.

## Гарантії поведінки

- Читає вміст відібраних Rust-файлів для аналізу документаційних коментарів.
- Свідомо пропускає шляхи: `node_modules`.
