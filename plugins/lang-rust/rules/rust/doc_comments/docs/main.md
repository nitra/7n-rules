## Огляд

Файл перевіряє, чи цілі для документаційних коментарів мають опис наміру, і повертає lint-порушення для відсутньої документації без власних операцій запису.

`isDocCommentTarget` визначає, які елементи потребують doc-коментаря. `checkFileDocComments` виконує перевірку вмісту файла й формує знайдені порушення. `lint` застосовує перевірку як lint-концерн і свідомо пропускає шляхи в `node_modules`.

## Поведінка

`lint` формує набір Rust-файлів для перевірки: або бере переданий дельта-список, або обходить робочу директорію. На цьому етапі `isDocCommentTarget` відсікає все, що не належить до цільових Rust-файлів; під час повного обходу також свідомо пропускається `node_modules`.

Для кожного відібраного файла `lint` читає вміст і передає його в `checkFileDocComments`. Результати всіх перевірок збираються в єдиний список порушень, який повертається назовні як результат lint-концерну.

`checkFileDocComments` застосовує спільні правила rustdoc-документування: файл із публічними top-level елементами має містити провідний опис наміру файла, а кожен такий публічний елемент — власний опис для документації API. Якщо публічних елементів немає, порушення не створюються.

Порушення містять причину, повідомлення, шлях до файла й дані для подальшого автоматичного виправлення там, де звичайний коментар можна безпечно підвищити до rustdoc-коментаря без вигадування тексту.

## Публічний API

- isDocCommentTarget — Чи підпадає файл під вимогу doc-коментарів.
- checkFileDocComments — Перевіряє один `.rs`-файл: `//!`-header + `///` над кожним top-level
pub-елементом. Сканування зупиняється на `#[cfg(test)]` (тест-модуль
конвенційно наприкінці файлу).
- lint — Detector rust/doc_comments: per-file (дельта) або повний обхід.

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Свідомо пропускає шляхи: `node_modules`.
