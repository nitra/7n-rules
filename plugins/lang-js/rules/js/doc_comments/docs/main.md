## Огляд

`lint` перевіряє, чи мають публічні export-и в JS-файлах належні doc comments, і повертає перелік порушень для лінту. Він окремо визначає, чи можна підняти коментарі `//` до придатного doc comment, щоб зафіксувати відсутню або неповну документацію над export-ом або на початку файла. Звіт формується для відносних POSIX-шляхів, щоб збігатися з дельта-перевіркою.

## Поведінка

lint запускає перевірку для js/doc_comments або по одному файлу з delta-контексту, або через повний обхід, і повертає список порушень поверх вихідного контексту лінту. Для кожного кандидата спершу застосовується isDocCommentTarget, щоб відсіяти позаобсягні шляхи; node_modules свідомо не перевіряється. Далі перевірка файла зводиться до зіставлення export-оголошень із наявним header-JSDoc та JSDoc над кожним експортом: checkFileDocComments збирає ці порушення в єдиний результат файла.

У межах цього потоку promotableHeaderBlock і promotableLineBlockBefore дають поведінкове визначення прийнятних `//`-блоків, які можна підняти до JSDoc: перший — для початку файла, другий — для декларації безпосередньо над позицією експорту. Такі блоки використовуються як резервний шлях, коли явного JSDoc ще немає, але структура коментарів відповідає вимогам до документації. Результати всіх перевірок нормалізуються у порушення файла з прив’язкою до relPosix-шляху й передаються назад у lint для підсумкового звіту.

## Публічний API

- isDocCommentTarget — Чи підпадає файл під вимогу doc-коментарів.
- promotableLineBlockBefore — Суцільний блок `//`-коментарів, що стоїть ВПРИТУЛ над позицією `pos`
(між блоком і декларацією — лише пробіли/переводи рядка; всередині блоку —
теж). Це кандидат на T0-підвищення до JSDoc.
- promotableHeaderBlock — Провідний суцільний `//`-блок на самому початку файлу (перед будь-яким кодом)
— кандидат на T0-підвищення до header-JSDoc.
- checkFileDocComments — Перевіряє один файл: header + JSDoc над кожним експортом.
- lint — Detector js/doc_comments: per-file (дельта) або повний обхід.

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Свідомо пропускає шляхи: `node_modules`.
