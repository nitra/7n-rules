## Огляд

Надає публічну функцію `lint` для запуску перевірки правила й повернення результатів lint-аналізу. Це потрібно, щоб правило мало єдину зовнішню точку входу для використання в lint-оркестраторі.

## Поведінка

1. `lint` обходить JS/TS-джерела проєкту з урахуванням виключених шляхів і перевіряє їх на небезпечні сценарії використання Bun SQL.

2. Для кожного знайденого порушення формує діагностику з маркером правила ``, щоб розробник міг швидко пов’язати проблему з відповідною політикою.

3. Перевіряє, що SQL-запити не виконуються небезпечними способами: без неконтрольованих викликів, динамічних списків, незахищених порожніх `IN`-списків, некоректної роботи з JSONB та масивами.

4. Окремо збирає ознаки використання `pg` і LISTEN/NOTIFY, щоб відрізнити дозволений сценарій подій бази даних від прихованого використання `pg` для звичайних запитів.

5. Зіставляє залежність `pg` у `package.json` із фактичними LISTEN/NOTIFY-сценаріями: залежність вважається обґрунтованою лише тоді, коли в проєкті є відповідне використання.

6. Вимагає, щоб файл з імпортом `pg` сам містив LISTEN/NOTIFY-поведінку; інакше повідомляє про ризик обходу правила через легітимну залежність.

7. Повертає результат лінту з переліком знайдених порушень або успішних перевірок без змін у файловій системі чи базі даних.
