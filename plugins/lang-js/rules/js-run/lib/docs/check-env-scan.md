## Огляд

AST-сканер правила `process.env / CheckEnv` (js-run.mdc): шукає прямі звернення до `process.env.X` і `process.env['X']`, а також деструктуризацію `const { X, Y } = process.env`, щоб вимагати або `env` з `@nitra/check-env` із закриттям через `checkEnv`, або `env` з `node:process` для опційних значень. Якщо у файл уже імпортовано `env` з `@nitra/check-env`, кожне `env.X`, `env['X']` і `const { X } = env` має бути закрите літеральним `checkEnv` у тому ж файлі; кілька викликів зводяться до одного набору змінних. Порушення можна точково приглушити коментарем `// n-rules:ignore-next-line checkEnv` на рядку безпосередньо перед ним. Якщо файл не парситься через `parseProgramOrNull`, сканер повертає порожній результат.

## Поведінка

`findUncheckedProcessEnvInText` спершу розбирає текст файлу через AST і лише для успішно розібраного коду продовжує перевірку. Далі вона збирає всі закриті імена зі статичних викликів `checkEnv`, перевіряє, чи є у файлі імпорт `env` саме з `@nitra/check-env`, і після цього проходить по дереву, шукаючи доступи до `process.env` та, за наявності цього імпорту, до `env`. Для кожного порушення формується запис із типом, іменем змінної та рядком; результати повертаються в порядку зустрічі в AST.

Правило охоплює прямі доступи, деструктуризацію та статичні члени, але свідомо пропускає обчислювані ключі, коли неможливо однозначно встановити назву ENV. Для `env` перевірка застосовується лише тоді, коли джерелом є `@nitra/check-env`, щоб не плутати його з локальними змінними чи `env` з `node:process`. Кожне знайдене порушення можна точково приглушити маркером `// n-rules:ignore-next-line checkEnv` у рядку безпосередньо перед ним; повідомлення для цього правила позначаються як ``.

`isCheckEnvScanSourceFile` виконує лише фільтрацію за шляхом: до сканування допускаються JS/TS-файли відповідної сім’ї, але не `.d.ts`. Це дає змогу запускати AST-перевірку тільки там, де правило реально має зміст, і не витрачати час на нерелевантні файли.

## Публічний API

- findUncheckedProcessEnvInText — Знаходить порушення правила «process.env / CheckEnv» у файлі.
- isCheckEnvScanSourceFile — Чи сканувати цей файл за розширенням (JS/TS-сім'я, без `.d.ts`).
