## Огляд

`lint` перевіряє опубліковані файли npm-пакета й допомагає не включати тести або fixtures до публічного артефакту. `findTestFrameworkImport` виявляє ознаки тестового коду, а `classifyPublishedFileAsTest` класифікує файл як тестовий для рішення lint-перевірки.

Шляхи `.github` і `.git` свідомо пропускаються, щоб службові директорії репозиторію не впливали на результат. Файл не має власних операцій запису у ФС/БД.

Поведінка fail-safe: помилки перехоплюються й не викидаються назовні; у частині помилкових сценаріїв повертається порожнє значення, зокрема `null`.

## Поведінка

`lint` запускає перевірку проєкту як npm-модуля: бере дані з робочого дерева, конфігурацій `package.json` і `tsconfig.emit-types.json`, GitHub workflow та hk-конфігурації, після чого перетворює знайдені невідповідності на pass/fail повідомлення з маркером (npm-module.mdc).

У потоці перевірки `lint` спершу підтверджує базову npm-структуру, потім визначає, чи потрібні додаткові вимоги для layout із `npm/src`, перевіряє наявність типів, конфігурації генерації типів, publish workflow і потрібних hk-кроків. Окремо він аналізує майбутній вміст npm-tarball, щоб пакет не публікував тести або fixtures.

`classifyPublishedFileAsTest` використовується в цьому tarball-контролі як єдина точка класифікації опублікованих файлів: рішення приймається за шляхом, назвою файла та, для JS/TS-коду, за ознаками імпорту тестового фреймворку. Для цього вона делегує розпізнавання імпортів у `findTestFrameworkImport`.

`findTestFrameworkImport` читає текст файла як код і повертає назву тестового модуля, якщо файл явно залежить від тестового фреймворку. Якщо код неможливо коректно розібрати, результат вважається невизначеним, а не помилкою цього правила.

Перевірки працюють fail-safe: очікувані збої читання чи розбору не пробиваються винятками назовні, а перетворюються на порожній результат або lint-повідомлення. Під час обходу свідомо пропускаються службові шляхи `.github` і `.git`, щоб не змішувати перевірку npm-вмісту з інфраструктурою репозиторію.

## Публічний API

- findTestFrameworkImport — Чи є у файлі імпорт/require/dynamic-import з модуля тест-фреймворку.
Парсимо через oxc-parser (як `bunyan-imports`/`redis-imports`). При помилці
парсингу повертаємо `null` — це не наш checker для синтаксису.
- classifyPublishedFileAsTest — Класифікує опублікований файл як test/fixture, якщо хоча б одна з ознак:
(1) у шляху є каталог із `TEST_DIR_NAMES`; (2) basename відповідає
`TEST_FILE_PATTERNS`; (3) для JS/TS-розширень — імпорт test-фреймворку.

Carve-out: для шляху `rules/<rule-name>/...` сегмент `<rule-name>` (індекс 1)
— це ім'я правила, а не каталог. Зокрема, правило з id `test` (або `tests`)
описує конвенцію розміщення тестів і саме по собі не є test-fixture'ом.
Подальші сегменти (наприклад, `rules/<r>/js/<c>/tests/`) продовжують перевірятись.
- lint — Перевіряє відповідність проєкту правилам npm-module.mdc

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
- Свідомо пропускає шляхи: `.github`, `.git`.
