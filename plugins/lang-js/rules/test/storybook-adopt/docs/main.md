## Огляд

Adopt-режим канону Storybook для пакетів: діагностує секційний diff між наявним `.storybook/` і канонічними `template/`-секціями (`main.js`, `preview.js`, `empty-vite.config.js` — лише для бібліотек, `mocks/gql-sse.js`, `package.json#scripts.storybook`, `vitest test.projects`, `vitest.stryker.config`, `.storybook/fixtures/` — лише для app-пакетів) і не перезаписує розбіжні файли сліпо. `--fix-missing` добудовує тільки секції, яких немає взагалі, з тим самим рендером, що й T0-фікс concern-ів `scaffold`/`vitest-config`; `.storybook/fixtures/` не генерується цим шляхом, бо це app-специфічний вміст без канонічного рендера. Збій діагностики або фіксу одного пакета дає `status: 'broken'` лише для цього пакета й не зупиняє обробку інших.

## Поведінка

STATUS і SECTION задають спільну модель результату для всієї діагностики: перший тримає стан пакета та прогону, другий — стан окремої секції. diagnosePackage збирає пакетний зріз із канонічних перевірок, розгалужує його за типом пакета й не зупиняє весь пакет на локальній помилці: збій однієї секції перетворюється на broken лише для цього пакета. Для бібліотек перевіряються спільні storybook-артефакти, а для app-пакетів додається секція fixtures і свідомо пропускається empty-vite.config.js, бо ці пакети не залежать від цього обхідного файлу. Усі секції порівнюються з канонічними шаблонами або маркерами, а повідомлення про розбіжності збираються в людський detail без сліпого перезапису; це ж стосується сигналів із (mocking.mdc), (storybook.mdc) і (vitest-config.mdc). fixMissingSections працює лише з відсутніми секціями одного пакета: вона створює канонічний вміст там, де файлу ще немає, і навмисно не чіпає розбіжні файли; fixtures для app-пакетів тут не генеруються, бо це пакетно-специфічний вміст без канонічного рендера. runAdopt оркеструє прогін по всіх або вибраних пакетах зі скоупу, застосовує діагностику, за потреби добудовує пропуски та ізолює збої на рівні одного пакета, щоб решта прогону завершувалась нормально. formatReport перетворює зведення runAdopt на читабельний текст для CLI, використовуючи стан пакетів, секцій і список записаних файлів. Уся логіка спирається на package.json як джерело сценарію storybook та пов’язаних точок входу.

## Публічний API

- STATUS — Статуси однієї секції діагностики (діагностика ≠ lint-violation — тут завжди 4 значення).
- SECTION — Канонічні назви секцій (стабільні — на них зав'язаний `--fix-missing`-switch).
- diagnosePackage — Діагностика одного пакета в скоупі по секціях. Розгалужена за `entry.type` (хвиля 2a):
  app-пакети діагностуються за app-канонічними маркерами й БЕЗ `empty-vite.config.js`
  (свідома асиметрія — app-и не використовують `viteConfigPath`-обхід), з додатковою
  секцією `.storybook/fixtures/`. Ніколи не кидає — збій окремої секції (парсинг/IO)
  відображається як секція `differ`, а не виняток, що впав би на весь пакет; лишається
  на розсуд `diagnosePackage` (circuit breaker рівня пакета — тут не потрібен, бо секції
  вже ізольовані одна від одної).
- fixMissingSections — Генерує канонічний вміст лише для секцій зі статусом `missing` одного пакета
  (adopt-автофікс НІКОЛИ не чіпає секції зі статусом `differ` — інструкція
  для агента/людини, не сліпий перезапис). Кожна секція фіксується незалежно;
  збій однієї не блокує решту (той самий circuit-breaker принцип, лише на дрібнішому рівні).
  `.storybook/fixtures/` (app-пакети) свідомо не генерується тут — вміст app-специфічний,
  канонічного рендера немає ({@link diagnoseFixturesSection}).
- runAdopt — Adopt-прогін усіх (чи обраних) пакетів у скоупі. Circuit breaker (ADR Кластер 8):
  збій діагностики/фіксу ОДНОГО пакета деградує до `status: 'broken'` для нього —
  решта пакетів обробляються далі, весь прогін ніколи не падає через один зламаний.
  відсутні секції; `rootDirs` — звузити прогін до цих коренів пакетів (порожньо/відсутнє → усі в скоупі)
- formatReport — Форматує людський звіт по результатах `runAdopt` (українською, для виводу скіла в CLI).

## Гарантії поведінки
