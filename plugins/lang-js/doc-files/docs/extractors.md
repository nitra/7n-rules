## Огляд

`extractFacts` збирає факти про JavaScript/TypeScript-файл і використовує їх для подальшого опису. Під час аналізу він звертається до мережі, а для вже опрацьованих даних тримає кеш у межах одного прогону, щоб не повторювати однакову роботу. За окремих помилок повертає порожнє значення, зокрема `null`, замість винятку, щоб не зупиняти обробку. Свідомо пропускає `.github`, `.git`, `node_modules`, `base/`, `ua/` і `.firebase`, щоб не витрачати зусилля на нерелевантні або службові частини дерева.

## Поведінка

1. Приймає файл як джерело фактів і формує для нього структурований профіль: мову, провідний коментар, експорти, імпорти, внутрішні та локальні символи, а також поведінкові маркери.
2. Спершу відсікає шляхи, які не входять у зону аналізу: `.github`, `.git`, `node_modules`, `base/`, `ua/`, `.firebase`.
3. Використовує кеш у межах одного прогону, щоб повторний аналіз однакових файлів не дублював роботу.
4. Переважно спирається на реальний AST-парсинг, щоб пов’язати коментарі й елементи коду без хибних спрацьовувань на текст, схожий на JSDoc, усередині коментарів або рядків.
5. Якщо парсинг не вдається, повертається до більш простого текстового аналізу й продовжує збір фактів замість падіння з винятком.
6. Для файлів, де доступний коректний розбір, визначає головний намір файлу через провідний коментар, а для публічних елементів — через коментарі, що стоять безпосередньо перед ними.
7. Окремо класифікує імпорти, щоб відрізнити стандартні залежності, зовнішні пакети та внутрішні модулі.
8. Витягає імена внутрішніх символів, щоб модель не подавала їх як частину публічного API.
9. Витягає локальні top-level сутності, які не експортуються, щоб відокремити службовий код від зовнішньої поведінки.
10. Додає поведінкові маркери на основі характерних ознак коду, щоб краще описати наміри файлу в узагальненому вигляді.
11. Повертає порожній результат замість помилки, якщо аналіз неможливий для окремого сценарію, зберігаючи стабільність потоку обробки.

## Публічний API

- extractFacts — Головний екстрактор: код файлу → факт-лист.
Коментарі беруться з реального AST-парсера (`parseProgramAndCommentsOrNull`),
не regex по сирому тексту — усуває клас false positive, де "/**"-подібний
текст усередині `//`-коментаря чи рядкового літералу (напр. glob-патерн)
помилково читається як відкриття JSDoc. Парсинг не вдався (синтаксична
помилка) → `comments: null`, `extractFileHeader`/`extractExports` падають
назад на свій regex-шлях (той самий, що й до цієї зміни).

## Гарантії поведінки

- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
- Кешує результати в межах одного прогону.
- Свідомо пропускає шляхи: `.github`, `.git`, `node_modules`, `base/`, `ua/`, `.firebase`.
