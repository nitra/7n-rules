## Огляд

`isCommentOnlyChange` відокремлює JS/TS-зміни, що торкнулися лише коментарів або форматування, від реальних змін коду відносно `merge-base` делти. Файл потрібен, щоб delta gate покриття не блокував rollout документаційних правок.

Порівняння виконується через `oxc` AST без коментарів із JSON-серіалізацією програм, тому рішення спирається на структуру коду, а не на текстові евристики. За будь-якої невизначеності, зокрема відсутності base-версії або синтаксичної помилки, зміна вважається не comment-only і файл консервативно лишається у gate.

## Поведінка

1. `isCommentOnlyChange` визначає, чи зміни у JS/TS-файлі відносно базової версії є лише коментарями або форматуванням.

2. Функція знаходить базову версію для порівняння; якщо її неможливо визначити, зміна вважається зміною коду й не пропускається.

3. Функція читає версію файлу з Git та поточний файл у робочому дереві; якщо будь-яке читання неуспішне, результат консервативно означає, що файл має лишитися в coverage-гейті.

4. Для обох версій файл перетворюється на канонічне представлення коду без урахування коментарів і позицій у тексті, щоб відрізняти реальні зміни поведінки від суто текстових правок.

5. Якщо будь-яка версія не може бути коректно проаналізована, `isCommentOnlyChange` не вважає зміну comment-only, щоб не приховати потенційно значущу зміну.

6. Якщо канонічні представлення збігаються, функція повертає ознаку, що файл можна пропустити в делта-гейті покриття; якщо відрізняються — файл лишається у гейті.

## Публічний API

- isCommentOnlyChange — Чи зміна файлу vs base — лише коментарі/пробіли (код ідентичний).

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
