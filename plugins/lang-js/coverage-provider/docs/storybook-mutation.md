## Огляд

Mutation testing для Storybook-покритого коду у vitest browser mode: замість Stryker `vitest-runner` тут працює власний mutate→run→restore executor, бо browser mode на базі Playwright ним не підтримується. Мутанти генеруються AST-based через `parseAst` у формі ESTree, а набір операторів упорядкований у 5 tier-ів за ймовірністю виживання: boundary → logical → equality → literals/return → arithmetic. У роботу беруться лише рядки, позначені як покриті в lcov `DA:`, із бюджетом `maxPerFile`/`maxTotal`; для `.vue` мутується тільки вміст `<script>` і `<script setup>`, а `lang="ts"`-скрипти, які `parseAst` не розбирає, тихо пропускаються. Кожен мутант проходить реальний прогін `vitest run --project=storybook` зі сторі-фільтром: `KILLED` при ненульовому exit, `SURVIVED` при exit `0`, `TIMEOUT` при примусовому завершенні за таймаутом, після чого оригінал відновлюється у `finally`. Помилки обробляються fail-safe, назовні не кидаються.

## Поведінка

parseLcovCoveredLines перетворює lcov-звіт на карту покритих рядків для файлів відносно робочого кореня.

extractMutableCode вирізає лише той код, який можна безпечно мутувати: для Vue береться лише script-частина, а TypeScript-скрипти тихо пропускаються.

generateMutants бере вихідний файл і карту покритих рядків, обмежує пошук лише покритими місцями та генерує детерміновані мутанти за пріоритетом від найбільш поведінково значущих до менш значущих змін. Для `.vue` це працює тільки в межах script-блоку, тож template лишається поза мутаціями.

runStorybookMutation зводить усе в один цикл: для кожного файлу отримує покриття, будує набір мутантів, по черзі підставляє кожен у файл, запускає реальний Storybook-специфічний прогін тестів і за результатом фіксує, чи мутант убито, чи він вижив, чи завис прогін. Після кожної спроби оригінальний вміст відновлюється, тому наступний мутант завжди стартує з чистого стану. Додаткові LLM-запропоновані мутанти, якщо їх підключено, живуть за тими самими правилами відбору й оцінюються лише реальним прогоном, але мають окремий ліміт і не дублюють уже згенеровані ділянки.

## Публічний API

- parseLcovCoveredLines — Парс lcov.info у мапу «файл → покриті рядки» з `SF:`/`DA:<line>,<hits>` records.
  Шляхи рібейзяться відносно `baseDir` (lcov типово містить абсолютні шляхи).
- extractMutableCode — Витягує JS-вміст для мутації: для `.vue` — перший `<script>`-блок (без TS),
  для решти — увесь файл. Експортовано для LLM-джерела мутантів
  (storybook-mutation-llm.mjs) — той самий скоуп і синтакс-валідація.
- generateMutants — Генерує детерміновані мутанти для одного файлу (AST-based, лише покриті рядки).
  Тіри (пріоритет відбору): 1 boundary (`<`↔`<=`…), 2 logical (`&&`↔`||`, зняття `!`),
  3 equality (`===`↔`!==`…), 4 literals/return (`true`↔`false`, `return X`→`return null`),
  5 arithmetic (`-`↔`+`, `*`↔`/`). `+` свідомо не мутується (string-concat дає шумні
  псевдо-еквівалентні мутанти).
- runStorybookMutation — Виконує mutate→run→restore цикл для набору файлів одного Storybook-root.

Гарантія відновлення: оригінальний вміст тримається в пам'яті і записується назад
у `finally` після КОЖНОГО мутанта — жоден наступний прогін не бачить попередню
мутацію, а креш процесу лишає щонайбільше один мутований файл (відновлюється
`git checkout`, файл завжди у git — ми мутуємо лише tracked production-код).

Класифікація: exit 0 → survived; exit ≠ 0 → killed; null status (kill за
таймаутом spawnSync) → timeout, рахується як caught (мутант завісив виконання).

`proposeExtraMutants` (опційно) — друге джерело мутантів (LLM, Mutahunter/ACH-патерн):
ПРОПОНУЄ додаткові bug-like мутанти, але вбиває/милує так само лише реальний прогін.
Додаткові мутанти мають ВЛАСНУ стелю (`maxExtraPerFile`) поверх детермінованої і
дедуплікуються проти детермінованих за діапазоном; помилки хука (нема API-ключа
тощо) — відповідальність самого хука (тут не ловляться).
