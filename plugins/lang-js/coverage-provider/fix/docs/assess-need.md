## Огляд

`assessNeed` допомагає fix-шляху coverage у правилі test для команди `npx @7n/rules lint test` вирішити, для яких непокритих файлів справді треба додавати unit-тести. Спершу `quickClassify` локально відсіює очевидні випадки, а LLM залучається лише для неоднозначних файлів. Файл не має власних операцій запису, перехоплює помилки й не кидає винятків назовні, щоб повертати безпечний вердикт навіть за збоїв.

## Поведінка

1. `assessNeed` приймає перелік файлів із низьким покриттям і формує для кожного бізнес-вердикт: чи варто додавати unit-тести та коротку причину українською.

2. Для недоступного файлу повертає безпечний негативний вердикт: тести не потрібні, бо файл неможливо прочитати.

3. Спершу застосовує швидку локальну класифікацію, щоб без звернення до LLM відсіяти очевидні випадки на кшталт файлів без поведінкової логіки або файлів із явними розгалуженнями.

4. Для неоднозначних файлів передає моделі обмежений вміст файлу й контекст поточного покриття, щоб отримати рішення про доцільність unit-тестів.

5. Якщо відповідь моделі коректна, використовує її рішення та пояснення; якщо модель не дала явної відмови від тестів, файл вважається таким, що потребує тестів.

6. Якщо оцінка через модель не вдалась або відповідь неможливо розібрати, працює fail-safe: не кидає помилку назовні й позначає файл як такий, що потребує тестів.

7. Повертає вердикти для всіх переданих файлів, не змінюючи файлову систему чи інші сховища.

## Публічний API

- assessNeed — Оцінює список непокритих файлів: чи потрібні їм тести.
  Очевидні випадки (реекспорти, функції з розгалуженнями) вирішуються локально,
  LLM викликається лише для неоднозначних.

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
