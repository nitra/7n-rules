## Огляд

Файл автоматизує fix-шлях для падаючих unit-тестів у правилі `test`: знаходить актуальні падіння через `bunx vitest run --reporter=json`, батчами читає поточний test/source-код, отримує від LLM виправлений ```js code block через `callText`і напряму записує результат через`writeFileSync`із`recordWrite`. Він повторює виправлення до `MAX_FIX_ATTEMPTS`, щоб стабілізувати coverage-перевірку без крихких Edit-операцій на складних рядках із URL, backtick-рядками та спецсимволами.

## Поведінка

`getFailingTests` запускає локальний для проєкту vitest у JSON-режимі та перетворює його звіт на список тестових файлів, які впали, разом із помилками. Дані беруться з фактичного прогону тестів, зокрема через `results.json`, тому подальші кроки працюють із поточним станом репозиторію.

`buildFixTestsPrompt` формує самодостатній prompt для LLM: додає помилки, поточний код тестів і, коли шлях можна вивести за конвенцією, відповідний source-код. Це потрібно, щоб модель могла запропонувати виправлення без окремого читання файлів інструментами агента.

`buildFixTestsBatch` керує розміром одного звернення до LLM: включає стільки падаючих файлів, скільки вміщується в prompt-бюджет, а решту відкладає на наступний прохід. Якщо окремий файл завеликий, він усе одно потрапляє в окремий prompt із обрізанням допоміжного контексту, щоб тестовий код залишався доступним для виправлення.

`fixFailingTests` є основним потоком: отримує або приймає список падінь, збирає batch, надсилає prompt у text mode, очікує відповідь із виправленим JS-кодом і записує зміни напряму в наявні тестові файли. Після запису він повторно перевіряє стан падінь і продовжує ітерації до зеленого результату, вичерпання спроб або дедлайну. Такий text+write шлях потрібен, щоб уникати крихкості edit-операцій на рядках із URL, backtick-рядками та спецсимволами.

Результат потоку — кількість застосованих виправлень, залишок падаючих файлів і список змінених тестів. Якщо на старті падінь немає, `fixFailingTests` завершується без записів.

## Публічний API

- getFailingTests — Жене vitest у JSON-режимі й повертає падаючі тест-файли з помилками.
  Project-local vitest споживача через `bunx` (патерн runJsCoverage) —
  bundled-vitest shim колишнього `\@7n/test` не переносився.
- buildFixTestsPrompt — Будує промпт для виправлення падаючих тест-файлів.
  Включає поточний вміст файлів — LLM не потребує file-read інструментів.
- buildFixTestsBatch — Складає промпт під бюджет `budgetFor('fix')`: скільки файлів влазить —
  стільки в батч (найменші першими), решта — у `deferred` на наступний
  прохід. Файл, що сам-один перевищує бюджет, отримує соло-промпт із
  внутрішнім обрізанням (спершу ріжеться source-довідка, потім
  vitest-помилки; сам тест-код захищений) — ніколи не скипається.
- fixFailingTests — Виявляє й виправляє падаючі тести через LLM text mode + прямий writeFileSync.
  Повертається одразу з count=0, якщо всі тести вже зелені.
