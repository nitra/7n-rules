## Огляд

`batchSurvived`, `fixSurvivedMutants` і `buildFixPrompt` запускають fix-шлях для survived-мутантів правила `test`, щоб `runAgentFix` з `@7n/llm-lib/agent-fix` міг цілеспрямовано писати тести, які вбивають вцілілі мутації Stryker під час `npx @7n/rules lint test`.

Модуль працює з survivors, що вже є в пам’яті з `violations`: для кожного мутанта бере контекст `file`, `line`, original code, survived variant і mutation type, а не читає `COVERAGE.md`, який більше не є джерелом цих даних.

Вибір моделі йде через `ctx.model` ladder-а з fallback на `CLOUD_MAX` або `N_CURSOR_COVERAGE_FIX_MODEL`, а всі записи проходять через `recordWrite` у central rollback ladder.

Обробка fail-safe: помилки не виходять назовні, щоб fix-сесії не ламали наступні кроки пайплайна.

## Поведінка

`batchSurvived` формує черги роботи так, щоб кожна batch зберігала цілісність файлів і не дробила контекст між прогоном.

`fixSurvivedMutants` бере вцілілі мутанти з пам’яті, розбиває їх на batch за доступним budget, для кожної batch готує rich-промпт через `buildFixPrompt` і запускає окрему agent fix-сесію. Помилка однієї batch не зупиняє інші: прогін переходить далі, а результати успішних і невдалих batch накопичуються окремо. Фактично змінені файли повертаються окремим списком, щоб ladder міг вирішити подальший rollback і наступні rung-и без цього модуля.

`buildFixPrompt` збирає для агента компактний робочий контекст по кожному файлу: список вцілілих мутантів, локальний кодовий контекст навколо них і достатньо інформації, щоб агент міг сам знайти або створити потрібні тести. Якщо джерело контексту недоступне, файл не підміняє його вигаданими даними, а передає лише те, що реально є у вхідних survived-даних.

## Публічний API

- batchSurvived — Ділить групи вцілілих мутантів на batches, кожен у межах `budget` мутантів сумарно —
  жадібне пакування у порядку вхідного масиву. Файл ніколи не ріжеться навпіл (мутанти
  одного файлу завжди в одному batch, навіть якщо сам файл перевищує budget) — узгодженість
  контексту для агента важливіша за точне дотримання стелі.
- fixSurvivedMutants — Запускає агентні fix-сесії для написання тестів по вцілілих мутантах — по batches,
  кожен своїм `runAgentFix`-викликом. Помилка одного batch (напр. timeout) не зупиняє
  решту: логується з переліком файлів batch (частковий прогрес реєструється через
  recordWrite — rollback вирішує ladder, не цей модуль), і прогін продовжується.
  Власних retry-циклів немає — конвергенцію жене ladder ядра повторними rung-ами.
- buildFixPrompt — Формує rich-промпт для агента: список вцілілих мутантів згрупований по файлах,
  з контекстом ±3 рядки навколо кожного мутанта з source-файлу.

## Гарантії поведінки

- Власних операцій запису (ФС/БД) у файлі немає; виклики імпортованих модулів можуть писати.
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
