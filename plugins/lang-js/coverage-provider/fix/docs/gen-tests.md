## Огляд

Файл генерує unit-тести для coverage-гілки правила `test` у `npx @7n/rules lint test`. `generateTests` будує тест-файл із cloud-згенерованого header-а та per-export `describe`-блоків: trivial/simple exports спершу спрямовуються в локальну модель з `opts.localModel` або `N_LOCAL_MIN_MODEL`, complex exports — у cloud.

Усі LLM-виклики проходять через `lib/llm.mjs`. Локально згенеровані блоки перевіряються через project-local `bunx vitest run`; за анти-патернів використовується fallback на cloud. Якщо локальна модель недоступна або exports не знайдено, файл генерується single-file cloud-шляхом. Результат записується через `recordWrite`.

`findTestRules` знаходить релевантні правила для тестів, `buildGenTestsPrompt` формує prompt для генерації, `generateTests` керує маршрутизацією, валідацією та записом результату.

## Поведінка

`generateTests` є основним входом fix-шляху coverage: отримує перелік source-файлів із причиною недостатнього покриття, визначає доступність локальної моделі та повертає абсолютні шляхи реально записаних тестів для контракту `touchedFiles`.

Для кожного файлу генерація читає фрагмент source, визначає майбутній шлях тесту за проєктною конвенцією `tests/`, підтягує найближчі правила `n-test.mdc` через `findTestRules` і формує контекст для LLM. Якщо доступна локальна модель і з файлу можна отримати export-и, використовується per-export маршрут: cloud-модель готує спільний header, простіші export-и спершу пробуються локально, складніші одразу йдуть у cloud, а неприйнятні або падаючі локальні блоки ескалюються в cloud із діагностикою помилок.

Згенеровані describe-блоки перевіряються запуском project-local `vitest` у проєкті споживача. Повторні спроби отримують фідбек із помилок тестового запуску, щоб LLM виправляла саме поведінкову невідповідність. Після успішної збірки спільний header і валідні блоки об’єднуються в один тест-файл; кінцевий запис проходить через `recordWrite`, щоб rollback ladder міг відстежити зміни.

Якщо локальної моделі немає або per-export маршрут непридатний, `generateTests` переходить на single-file cloud-генерацію: один prompt описує source, очікуваний тестовий файл, правила проєкту та coverage-причину, після чого витягнутий JS-код записується як тест.

`findTestRules` централізує пошук тестових правил для prompt-ів: піднімається від кореня проєкту обмежену кількість рівнів і повертає текст правил або відсутність правил. Це дає генерації однаковий контекст незалежно від обраного маршруту.

`buildGenTestsPrompt` створює лише display-only summary для тестів і діагностики. Він агрегує перелік файлів, coverage-відсотки, причини та короткі source-фрагменти, але не запускає LLM і не записує файли.

## Публічний API

- findTestRules — Знаходить n-test.mdc правила проєкту, піднімаючись від dir (максимум 4 рівні).
- buildGenTestsPrompt — Будує display-only summary-промпт (використовується в тестах).
- generateTests — Генерує тести для всіх переданих файлів.
  Per-export tiered-маршрутизація коли доступна локальна модель;
  інакше — single-file cloud-генерація. Повертає записані файли для
  `touchedFiles`-контракту fix-worker-а.
