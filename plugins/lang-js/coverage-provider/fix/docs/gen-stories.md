## Огляд

Генерує окремий `.stories.*` файл для кожного непокритого Vue-компонента, щоб для однієї SFC природно існував один story-файл замість складання stories у `tests/`-структурі.  
Коли споживач декларує `vitest` у `package.json`, валідовує згенерований story через `bunx vitest run --project=storybook <story>` у корені споживача; якщо `vitest` немає, все одно записує story best-effort і лише попереджає про пропуск перевірки.

## Поведінка

1. Приймає список непокритих Vue-компонентів і для кожного окремо запускає генерацію `.stories.*` поруч із компонентом, дотримуючись конвенції Storybook, а не `tests/`.
2. Для кожного компонента формує цільовий story-файл, визначає мову story за самим SFC і підбирає текст запиту з коротким фрагментом source та проєктними правилами з `package.json`.
3. Створює story як окрему одиницю роботи: один компонент — один story-файл, без спільного проходу на весь список.
4. Подає запит у LLM, витягує з відповіді код story і перевіряє, чи він придатний до запису.
5. Якщо код не проходить валідацію, використовує feedback-loop: уточнює запит з урахуванням попереднього коду та помилок, і пробує ще раз у межах дозволених спроб.
6. Після успішної генерації записує story у файл і запускає валідацію через Storybook vitest-проєкт споживача.
7. Валідацію виконує лише тоді, коли споживач оголошує vitest; якщо ні, story все одно може бути збережений як best-effort, але без browser-mode перевірки.
8. Якщо валідація не проходить, прибирає невдалий файл, щоб у репозиторії не залишалась поламана story.
9. Для кожного успішно створеного story повертає список змінених файлів як результат роботи.

## Публічний API

- generateStories — Генерує `.stories.*` для списку непокритих Vue-компонентів.
  Кожен файл — окрема chain-одиниця (tracing/бюджет, паритет з generateTests).
